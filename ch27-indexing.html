01 — What Are Indexes and Why They Matter

Understanding B-Tree Indexes and Full Table Scans

At their core, database indexes are data structures that enable faster lookups. Without indexes, a query must perform a full table scan: reading every single row in the table and checking if it matches your WHERE clause. With millions of rows, this becomes prohibitively slow. Indexes—typically implemented as B-Tree structures—organize data in a way that allows the database to jump directly to the rows you want.

A B-Tree is a balanced tree structure where each node can contain multiple keys and pointers to child nodes. The key insight is that the tree stays balanced, meaning lookups always take roughly O(log N) time instead of O(N). With a million rows, a table scan might require checking 1,000,000 rows; a B-Tree index might require checking only 20 nodes to find the exact row you need.

SQL — Without Index (Full Table Scan) -- This query scans every row in Orders table -- With 1 million rows, this is expensive SELECT * FROM Orders WHERE CustomerId = 5; -- Without index, SQL Server must check 1,000,000 rows -- With index on CustomerId, SQL Server can: -- 1. Use B-Tree to find first matching row -- 2. Follow pointer to retrieve full row -- Result: 20-30 page reads vs 5000+ page reads
C# — How EF Core Benefits from Indexing // Without index: slow query var orders = dbContext.Orders .Where(o => o.CustomerId == 5) .ToList(); // Slow! Scans 1M rows // With index: fast query (same code, better performance) var orders = dbContext.Orders .Where(o => o.CustomerId == 5) .ToList(); // Fast! Uses index seek, ~20 rows read // The C# code is identical. The difference is entirely // on the database side. When you add an index, EF's // execution time drops dramatically without any changes // to your application code.
Key Insight: Indexes Don't Change Your Code

The remarkable thing about indexes is that they require no changes to your C# application code. The same LINQ query that was slow becomes fast once you add the appropriate index on the database side. This is why understanding indexing is crucial—a single well-placed index can multiply query performance by 100x or more.

To see the difference, you can use SET STATISTICS IO in SQL Server to count logical page reads:

SQL — Measuring Impact with STATISTICS IO -- Enable detailed statistics SET STATISTICS IO ON; -- Query WITHOUT index (or before creating it) SELECT * FROM Orders WHERE CustomerId = 5; -- Table 'Orders'. Scan count 1, logical reads 5000 -- Now create index CREATE INDEX IX_Orders_CustomerId ON Orders(CustomerId); -- Same query WITH index SELECT * FROM Orders WHERE CustomerId = 5; -- Table 'Orders'. Scan count 1, logical reads 15 -- That's a 333x improvement in I/O operations!
C# — Observing Performance in EF Core // Enable logging to see SQL generated and timings var options = new DbContextOptionsBuilder() .UseSqlServer(connectionString) .LogTo(Console.WriteLine, LogLevel.Information) .Build(); using var context = new OrderContext(options); // Before index: query takes ~2.5 seconds // "Executed DbCommand (2500ms) [Parameters=...]" var orders = context.Orders .Where(o => o.CustomerId == 5) .ToList(); // After adding index on database: same code, ~10ms // "Executed DbCommand (10ms) [Parameters=...]" // Your C# code doesn't change—only DB performance changes
02 — Creating Indexes: SQL vs EF Fluent API

Building Indexes in Both SQL and C#

There are two primary ways to create indexes: writing raw SQL CREATE INDEX statements and using Entity Framework's Fluent API during model configuration. Both achieve the same result on the database—applied indexes are identical whether created via SQL or EF migrations—but the approach differs depending on your workflow.

SQL — CREATE INDEX Statement -- Basic index on a single column CREATE INDEX IX_Orders_CustomerId ON Orders(CustomerId); -- Best practice: name indexes with IX_ prefix -- IX = Index -- Table name = Orders -- Column(s) = CustomerId -- Querying with this index is now fast: SELECT * FROM Orders WHERE CustomerId = 5; -- You can also create UNIQUE indexes to enforce -- uniqueness at the database level CREATE UNIQUE INDEX IX_Users_Email ON Users(Email); -- This prevents two users from having the same email, -- enforced by the database itself
C# — EF Core Fluent API Configuration // In your DbContext OnModelCreating method protected override void OnModelCreating( ModelBuilder modelBuilder) { // Simple non-unique index modelBuilder.Entity() .HasIndex(o => o.CustomerId) .HasName("IX_Orders_CustomerId"); // UNIQUE index on User email modelBuilder.Entity() .HasIndex(u => u.Email) .HasName("IX_Users_Email") .IsUnique(); // Add IsUnique() for UNIQUE constraint base.OnModelCreating(modelBuilder); } // When you run 'dotnet ef migrations add AddIndexes', // EF generates a migration with these CREATE INDEX statements: // migrationBuilder.CreateIndex( // name: "IX_Orders_CustomerId", // table: "Orders", // column: "CustomerId"); // // The generated migration contains exact SQL that will // be executed when you run 'dotnet ef database update'

The Fluent API approach is generally preferred in modern applications because it keeps your index definitions with your model definitions, and EF automatically generates the correct SQL and migration scripts. However, you can also define indexes using Data Annotations:

SQL — For Reference: The Generated Migration -- When EF generates the migration, it creates this SQL: CREATE TABLE Orders ( Id INT PRIMARY KEY, CustomerId INT, OrderDate DATETIME, Total DECIMAL(10,2) ); -- Then adds indexes separately: CREATE INDEX IX_Orders_CustomerId ON Orders(CustomerId); -- Indexes are created AFTER the table exists. -- You cannot index a non-existent column. -- Create the table first, then add indexes.
C# — Data Annotations Alternative using System.ComponentModel.DataAnnotations.Schema; public class Order { public int Id { get; set; } // Using IndexAttribute (available in EF Core 5.0+) [Index(nameof(CustomerId))] public int CustomerId { get; set; } public DateTime OrderDate { get; set; } public decimal Total { get; set; } } public class User { public int Id { get; set; } // Unique index using IsUnique = true [Index(nameof(Email), IsUnique = true)] public string Email { get; set; } } // Both approaches (Fluent and Annotations) work. // Fluent API is more flexible for complex scenarios. // Annotations are more concise for simple cases.
Key Insight: Indexes and Migrations Go Hand in Hand

When you define indexes in EF's Fluent API or with IndexAttribute, those definitions automatically appear in your migration files. The migration generates the exact SQL CREATE INDEX statements needed. When you deploy to production, running 'dotnet ef database update' applies both the schema and indexes. This is safer and more maintainable than trying to manually add indexes to production databases.

03 — Composite Indexes and Column Order

Multiple Columns in a Single Index

Sometimes you have queries that filter on multiple columns. A composite index—an index on multiple columns—can be dramatically more efficient than separate indexes on each column. However, the order of columns in the index matters greatly. This is called the leftmost prefix rule: the database can use an index to efficiently search for values in the leftmost columns, but adding columns on the right becomes progressively less useful.

SQL — Composite Index on CustomerId, OrderDate -- Create composite index: CustomerId first, OrderDate second CREATE INDEX IX_Orders_CustDate ON Orders(CustomerId, OrderDate); -- This index works efficiently for queries like: SELECT * FROM Orders WHERE CustomerId = 5; -- Uses leftmost column (fast) SELECT * FROM Orders WHERE CustomerId = 5 AND OrderDate > '2024-01-01'; -- Uses both columns (very fast) -- This index is LESS useful for: SELECT * FROM Orders WHERE OrderDate > '2024-01-01'; -- Must scan the whole index because OrderDate isn't first -- The leftmost prefix rule: columns must be in order -- An index on (A, B, C) can be used as (A), (A,B), or (A,B,C) -- But NOT as (B), (B,C), (C), or (A,C) -- If you need to query on OrderDate alone, create: CREATE INDEX IX_Orders_OrderDate ON Orders(OrderDate);
C# — Composite Index in Fluent API protected override void OnModelCreating( ModelBuilder modelBuilder) { // Composite index: CustomerId first, OrderDate second modelBuilder.Entity() .HasIndex(o => new { o.CustomerId, o.OrderDate }) .HasName("IX_Orders_CustDate"); // This enables efficient queries like: var orders = dbContext.Orders .Where(o => o.CustomerId == 5) .ToList(); // Uses IX_Orders_CustDate, fast var recentOrders = dbContext.Orders .Where(o => o.CustomerId == 5 && o.OrderDate > DateTime.Parse("2024-01-01")) .ToList(); // Uses both columns, very fast // If you also query by OrderDate alone, add separate: modelBuilder.Entity() .HasIndex(o => o.OrderDate) .HasName("IX_Orders_OrderDate"); base.OnModelCreating(modelBuilder); } // Real-world scenario: you have 5 indexes but typically // only 1-2 are used per query. SQL Server optimizer // analyzes the WHERE clause and chooses the best index. // Your job: make sure the right indexes exist.

The leftmost prefix rule is critical to understand. Consider a real-world example: an e-commerce database with millions of orders. You want fast lookup by customer, and within each customer's orders, you want fast filtering by date.

SQL — Understanding Leftmost Prefix -- Composite index on (CustomerId, OrderDate) CREATE INDEX IX_Orders_CustDate ON Orders(CustomerId, OrderDate); -- Structure visualization: -- Level 1: CustomerId values (1, 2, 3, 4, 5, ...) -- Level 2: Within each CustomerId, OrderDate values -- CustId 1: [2023-01-15, 2023-02-20, 2024-01-10] -- CustId 2: [2023-03-05, 2024-02-14] -- CustId 5: [2024-01-01, 2024-03-15, 2024-03-20] -- Fast: CustomerId is the leftmost column SELECT COUNT(*) FROM Orders WHERE CustomerId = 5; -- Index search: Jump to CustId=5 section, count rows -- Fast: Both columns in order SELECT OrderId FROM Orders WHERE CustomerId = 5 AND OrderDate >= '2024-01-01'; -- Index search: Jump to CustId=5, then find OrderDate>=2024-01-01 -- Slow: Skipping the leftmost column SELECT COUNT(*) FROM Orders WHERE OrderDate >= '2024-01-01'; -- Cannot use this index effectively -- Must scan the entire index (still slow)
C# — Querying with Composite Indexes public class OrderRepository { private readonly OrderContext _context; public OrderRepository(OrderContext context) { _context = context; } // Query 1: Filters on leftmost column only (CustomerId) public List GetOrdersByCustomer(int customerId) { return _context.Orders .Where(o => o.CustomerId == customerId) .ToList(); // Uses IX_Orders_CustDate, very fast } // Query 2: Filters on both columns in order public List GetRecentOrdersByCustomer( int customerId, DateTime since) { return _context.Orders .Where(o => o.CustomerId == customerId && o.OrderDate >= since) .ToList(); // Uses IX_Orders_CustDate, very fast } // Query 3: Filters on OrderDate only (rightmost column) public List GetOrdersSince(DateTime since) { return _context.Orders .Where(o => o.OrderDate >= since) .ToList(); // Cannot use IX_Orders_CustDate effectively // Need separate index: IX_Orders_OrderDate } } // In OnModelCreating: // modelBuilder.Entity() // .HasIndex(o => new { o.CustomerId, o.OrderDate }); // modelBuilder.Entity() // .HasIndex(o => o.OrderDate);
04 — Covering Indexes and the INCLUDE Clause

Adding Extra Columns for Needle-Free Lookups

Normally, an index tells the database where to find rows (the row's location or cluster key). The database still must access the actual table to retrieve all the columns you SELECT. A covering index includes extra columns in the index itself, so the database can answer your query entirely from the index without touching the table. This is called a covering index because the index "covers" all the columns needed for the query.

SQL — Covering Index with INCLUDE -- Standard index: only contains CustomerId CREATE INDEX IX_Orders_CustomerId ON Orders(CustomerId); -- Query needs CustomerId, Total, and OrderDate SELECT CustomerId, Total, OrderDate FROM Orders WHERE CustomerId = 5; -- Database: -- 1. Uses index to find rows with CustomerId = 5 -- 2. Follows pointer to actual row in table (KEY LOOKUP) -- 3. Retrieves Total and OrderDate from table -- Result: index seeks + key lookups -- Better: Covering index that INCLUDES the extra columns CREATE INDEX IX_Orders_Cust ON Orders(CustomerId) -- Search column INCLUDE (Total, OrderDate); -- Extra columns included in index -- Same query now: SELECT CustomerId, Total, OrderDate FROM Orders WHERE CustomerId = 5; -- Database: -- 1. Uses index to find rows with CustomerId = 5 -- 2. Retrieves Total and OrderDate FROM THE INDEX -- 3. No need to access the table! -- Result: index seeks, no key lookups (faster) -- Covering indexes are powerful for read-heavy scenarios -- where you SELECT a specific set of columns repeatedly
C# — Covering Indexes via Raw SQL Migration // EF Core doesn't have direct syntax for INCLUDE, // so you add it via raw SQL in a migration // In Package Manager Console or .NET CLI: // dotnet ef migrations add AddCoveringIndex // Then edit the migration file: public partial class AddCoveringIndex : Migration { protected override void Up(MigrationBuilder migrationBuilder) { // Raw SQL: create covering index with INCLUDE migrationBuilder.Sql(@" CREATE INDEX IX_Orders_Cust ON Orders(CustomerId) INCLUDE (Total, OrderDate) "); } protected override void Down(MigrationBuilder migrationBuilder) { migrationBuilder.Sql("DROP INDEX IX_Orders_Cust ON Orders"); } } // Now run: dotnet ef database update // The covering index is created, and queries benefit: var orders = dbContext.Orders .Where(o => o.CustomerId == customerId) .Select(o => new { o.CustomerId, o.Total, o.OrderDate }) .ToList(); // This query uses the covering index // All data comes from index, no table access needed
Key Insight: Balance Coverage with Maintenance Overhead

Covering indexes are powerful for specific queries, but they increase the size of the index and slow down write operations (every INSERT/UPDATE must maintain all included columns). Use them selectively for frequent, critical queries. Don't add every column to every index—focus on your actual access patterns.

05 — When Indexes Hurt: The Write Performance Penalty

Understanding the INSERT/UPDATE/DELETE Cost

Indexes dramatically improve read performance, but they come at a cost: every INSERT, UPDATE, and DELETE operation must update all indexes on that table. If you have 10 indexes on a table and insert a row, the database must insert that row into the table, then update 10 indexes. This cost is invisible when you're doing a few inserts but becomes painfully obvious during bulk operations.

SQL — Insert Performance with Multiple Indexes -- Table with 10 indexes CREATE TABLE Products ( Id INT PRIMARY KEY, Name NVARCHAR(100), Sku NVARCHAR(50), Price DECIMAL(10,2), CategoryId INT, SupplierId INT, CreatedDate DATETIME, ModifiedDate DATETIME, IsActive BIT ); -- Indexes for common queries: CREATE INDEX IX_Products_CategoryId ON Products(CategoryId); CREATE INDEX IX_Products_SupplierId ON Products(SupplierId); CREATE INDEX IX_Products_Sku ON Products(Sku); CREATE INDEX IX_Products_IsActive ON Products(IsActive); CREATE INDEX IX_Products_CreatedDate ON Products(CreatedDate); -- ... plus 5 more indexes -- Insert 100,000 products: -- Naive approach: 100,000 individual inserts INSERT INTO Products (Name, Sku, Price, CategoryId, SupplierId) VALUES ('Product 1', 'SKU-0001', 29.99, 1, 1); -- Cost per insert: 1 table write + 10 index writes = 11 operations -- 100,000 inserts * 11 operations = 1.1 million page writes -- Execution time: 2-3 minutes for 100K rows -- Better approach: BULK INSERT (disables some indexes) -- Or use a transaction and batch: BEGIN TRANSACTION; INSERT INTO Products SELECT * FROM TempProducts; COMMIT; -- Still slow with all indexes active
C# — Bulk Insert Strategies public class ProductRepository { private readonly ProductContext _context; // Naive approach: Add each product individually public void InsertProductsSlowly(List products) { foreach (var product in products) { _context.Products.Add(product); _context.SaveChanges(); // Persists immediately, hits all indexes } // For 100K products: 100K database round trips // Each SaveChanges flushes to DB // Extremely slow! } // Better: Batch add-then-save public void InsertProductsFaster(List products) { foreach (var product in products) { _context.Products.Add(product); } _context.SaveChanges(); // Single batch insert // Still updates all indexes, but fewer round trips // For 100K: 1 database round trip with large insert // Faster than naive, but still slow with many indexes } // Best: Disable indexes, insert, rebuild indexes public async Task InsertProductsFastest(List products) { var connection = _context.Database.GetDbConnection(); // Disable all indexes on Products table await connection.ExecuteAsync( "ALTER INDEX ALL ON Products DISABLE"); try { foreach (var product in products) { _context.Products.Add(product); } await _context.SaveChangesAsync(); // Inserts are fast: no indexes to maintain } finally { // Rebuild all indexes await connection.ExecuteAsync( "ALTER INDEX ALL ON Products REBUILD"); } // For 100K: Insert takes ~10 seconds // Index rebuild takes ~20 seconds // Total: ~30 seconds vs 2-3 minutes with active indexes } }

The tradeoff is clear: indexes speed up SELECT but slow down INSERT/UPDATE/DELETE. In a typical OLTP (Online Transaction Processing) system, this tradeoff is favorable because most operations are reads. In a data warehouse doing nightly bulk loads, you might disable indexes during load and rebuild them afterward.

SQL — Measuring Index Maintenance Cost -- Example: Table with index, track insert performance CREATE TABLE TestTable ( Id INT PRIMARY KEY, Value NVARCHAR(100), CreatedDate DATETIME ); CREATE INDEX IX_TestTable_CreatedDate ON TestTable(CreatedDate); -- Insert 10K rows and measure: SET STATISTICS IO ON; INSERT INTO TestTable (Id, Value, CreatedDate) SELECT TOP 10000 ROW_NUMBER() OVER (ORDER BY (SELECT 1)), 'Value' + CAST(ROW_NUMBER() OVER (ORDER BY (SELECT 1)) AS NVARCHAR(10)), GETDATE() FROM master.dbo.spt_values; -- Table 'TestTable'. Scan count 0, logical reads 35 -- Index 'IX_TestTable_CreatedDate'. Scan count 0, logical reads 45 -- Total: 80 logical reads for inserts + index updates -- Now disable index and retry: ALTER INDEX IX_TestTable_CreatedDate ON TestTable DISABLE; INSERT INTO TestTable (Id, Value, CreatedDate) SELECT TOP 10000 ... -- Table 'TestTable'. Scan count 0, logical reads 35 -- No index reads! Significantly faster. -- Rebuild index: ALTER INDEX IX_TestTable_CreatedDate ON TestTable REBUILD;
C# — Practical Bulk Insert Example public class BulkDataLoader { private readonly IDbConnection _connection; public async Task LoadLargeDataset( List products, string tableName = "Products") { // Step 1: Disable indexes var disableCmd = $"ALTER INDEX ALL ON {tableName} DISABLE"; await _connection.ExecuteAsync(disableCmd); try { // Step 2: Use SqlBulkCopy for ultra-fast inserts using (var bulkCopy = new SqlBulkCopy(_connection as SqlConnection)) { bulkCopy.DestinationTableName = tableName; bulkCopy.BatchSize = 5000; bulkCopy.NotifyAfter = 10000; // Convert List to DataTable DataTable dt = ConvertToDataTable(products); await bulkCopy.WriteToServerAsync(dt); Console.WriteLine($"Inserted {products.Count} rows"); } } finally { // Step 3: Rebuild indexes var rebuildCmd = $"ALTER INDEX ALL ON {tableName} REBUILD"; await _connection.ExecuteAsync(rebuildCmd); } } // With indexes disabled: 100K products in ~5 seconds // With indexes enabled: 100K products in ~90 seconds // That's an 18x improvement by managing indexes strategically! }
06 — Reading Execution Plans and Identifying Bottlenecks

Analyzing Query Performance with Execution Plans

An execution plan is a visual representation of how the database executes your query. It shows every step: which tables are scanned, which indexes are used, and how much work each step costs. Learning to read execution plans is essential for identifying performance bottlenecks. The three key operations are Table Scan (slow), Index Scan (medium), and Index Seek (fast).

SQL — Viewing Execution Plans in SSMS -- In SQL Server Management Studio, enable actual execution plan: -- Query → Include Actual Execution Plan (Ctrl+Shift+A) -- Example query to analyze: SELECT * FROM Orders WHERE CustomerId = 5; -- Execution Plan will show: -- 1. SELECT (top of tree) -- └── Table Scan [Orders] -- └── Predicate: [CustomerId] = 5 -- Analysis: -- - "Table Scan" means every row was checked -- - This is inefficient if table has millions of rows -- - Solution: Add index on CustomerId -- After creating index: CREATE INDEX IX_Orders_CustomerId ON Orders(CustomerId); -- Re-run query with execution plan: SELECT * FROM Orders WHERE CustomerId = 5; -- New Execution Plan: -- 1. SELECT -- └── Index Seek [IX_Orders_CustomerId] -- └── Predicate: [CustomerId] = 5 -- └── Key Lookup [Orders] -- Analysis: // "Index Seek" = efficient (directly finds rows) // "Key Lookup" = small cost to fetch full row data // Estimated subtree cost: 0.08 (was 2.5+ with scan) // That's 30x faster!
C# — Analyzing EF Core Queries public class QueryAnalyzer { private readonly OrderContext _context; public QueryAnalyzer(OrderContext context) { _context = context; } // Enable query logging to see execution time public void ConfigureLogging() { // In DbContext constructor: var options = new DbContextOptionsBuilder() .UseSqlServer(connectionString) .LogTo(Console.WriteLine, new[] { RelationalEventId.CommandExecuted }) .EnableSensitiveDataLogging() .Build(); } // Example slow query public List GetOrdersSlowly() { return _context.Orders .Where(o => o.CustomerId == 5) .ToList(); // Console output: // "Executed DbCommand (2500ms)" // Slow! Probably a table scan. // Check execution plan in SQL Server: // 1. Enable "Include Actual Execution Plan" in SSMS // 2. Copy the SQL from EF logs // 3. Paste into SSMS and execute // 4. View the execution plan } // After adding index, same code is fast: public List GetOrdersFast() { return _context.Orders .Where(o => o.CustomerId == 5) .ToList(); // Console output: // "Executed DbCommand (10ms)" // Index seek is being used! } // Complex query analysis public List AnalyzeComplexQuery() { // Multi-join query: check if it uses indexes var results = _context.Orders .Include(o => o.Customer) .Include(o => o.LineItems) .Where(o => o.OrderDate >= DateTime.Today.AddMonths(-1)) .ToList(); // To analyze: // 1. Enable logging and copy the generated SQL // 2. Run in SSMS with Include Actual Execution Plan // 3. Look for Table Scans (bad) vs Index Seeks (good) // 4. Add indexes for frequently scanned columns return null; } }

Understanding the three core query operations helps you quickly diagnose problems:

SQL — Three Core Query Operations -- OPERATION 1: Table Scan -- Reads every row in the table sequentially -- Cost: O(N) where N = number of rows -- Example: SELECT * FROM Orders WHERE Status = 'Pending'; -- Without index on Status, must scan all 1M rows -- OPERATION 2: Index Scan -- Reads every row in an index sequentially -- Cost: Still O(N), but smaller dataset than table -- Example: Index exists but can't be used efficiently SELECT * FROM Orders WHERE OrderDate > '2024-01-01'; -- Index on (CustomerId, OrderDate) cannot be used -- because CustomerId isn't in WHERE clause // Must scan the entire index (slow, but less slow than table scan) -- OPERATION 3: Index Seek -- Uses index to jump directly to matching rows -- Cost: O(log N) + O(K) where K = result rows -- Example: SELECT * FROM Orders WHERE CustomerId = 5; -- With index on CustomerId: // Jump to CustomerId=5 section (log N steps) // Retrieve matching rows (K steps) // Total: 20 + 100 = 120 steps vs 1M steps (table scan) -- Rule of thumb: // Seek = good (what you want) // Scan = acceptable (if necessary) // Table Scan = investigate why there's no useful index
C# — Using Azure Data Studio for Analysis // Azure Data Studio (open source, free) is excellent // for analyzing EF Core-generated queries // Step 1: Configure EF to log SQL var options = new DbContextOptionsBuilder() .UseSqlServer(connectionString) .LogTo(Console.WriteLine, new[] { RelationalEventId.CommandExecuted }) .Build(); using var context = new OrderContext(options); // Step 2: Run your LINQ query var orders = context.Orders .Where(o => o.CustomerId == 5) .ToList(); // Step 3: Copy the logged SQL from console output: // SELECT "o"."OrderId", "o"."CustomerId", ... // FROM "Orders" AS "o" // WHERE "o"."CustomerId" = 5 // Step 4: Open Azure Data Studio // 1. Paste the SQL into a new query // 2. Click "Explain Query Plan" (or Ctrl+Alt+L) // 3. Visual tree shows: // - Index Seek on CustomerId (good!) // - Estimated cost: 0.005 // - Rows returned: 47 // Step 5: If you see Table Scan instead: // 1. Add the index: HasIndex(o => o.CustomerId) // 2. Create migration: dotnet ef migrations add AddCustomerIdIndex // 3. Run: dotnet ef database update // 4. Re-run the query and re-check the plan // 5. Now shows Index Seek (problem solved!) // Query before index: 2500ms // Query after index: 10ms (250x faster)
Key Insight: The Execution Plan is Your Best Debugging Tool

When a query is slow, the execution plan tells you exactly why. Is it a table scan? Add an index. Is it a missing statistic? Update statistics. Is it a complex join creating a hash match? Rewrite the query or add indexes. The execution plan is objective data—it shows exactly what the database is doing. Always check it before guessing.

Key Takeaways

  • Indexes use B-Tree structures to enable O(log N) lookups instead of O(N) table scans, often providing 100x+ performance improvements
  • Create indexes using SQL CREATE INDEX or EF Core's Fluent API; both approaches generate identical database structures
  • Composite indexes follow the leftmost prefix rule: columns must be used in index order for the index to be effective
  • Covering indexes with INCLUDE clauses allow the database to answer queries entirely from the index, avoiding table lookups
  • Every index slows down INSERT/UPDATE/DELETE operations, so index strategically: focus on frequently-queried columns
  • Execution plans (Table Scan vs Index Scan vs Index Seek) reveal exactly how the database executes your query and identify bottlenecks
  • Use Azure Data Studio or SQL Server Management Studio to visualize execution plans and verify that your indexes are being used
