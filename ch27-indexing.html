Database Indexing

Speed up queries with proper indexing. Understand B-trees, execution plans, and when to index.

In This Chapter

01

What is an Index?

An index is a sorted data structure that speeds up lookups. Without an index, the database scans every row. With an index, it jumps directly to matches.

OperationWithout IndexWith Index
Find customer by IDTable scan (1M rows)B-tree lookup (log time)
Find orders by date rangeScan all rowsIndex seek + range scan
JOIN on customer IDNested loop (slow)Index join (fast)
Rule of Thumb:

Index columns you filter on (WHERE), join on (ON), and sort on (ORDER BY).

02

Index Types

Different index types optimize different queries.

SQL -- Single-column index CREATE INDEX IX_Customers_Email ON Customers(Email); -- Composite index (multiple columns) CREATE INDEX IX_Orders_CustomerDate ON Orders(CustomerId, CreatedAt); -- Unique index (prevents duplicates) CREATE UNIQUE INDEX IX_Customers_EmailUnique ON Customers(Email); -- Filtered index (only active rows) CREATE INDEX IX_Customers_Active ON Customers(Name) WHERE IsActive = 1;
EF Core // Single-column e.HasIndex(c => c.Email); // Composite e.HasIndex(o => new { o.CustomerId, o.CreatedAt }); // Unique e.HasIndex(c => c.Email).IsUnique(); // Filtered e.HasIndex(c => c.Name) .HasFilter("[IsActive] = 1");
Composite Order Matters:

Index on (CustomerId, CreatedAt) helps queries filtering on CustomerId, but not on CreatedAt alone.

03

Clustered vs Non-Clustered

Clustered index is the table's primary sort order. Non-clustered indexes are separate structures.

AspectClusteredNon-Clustered
Count per tableOne (usually PK)Many
StorageIS the tableSeparate structure
PerformanceFast lookupsFast lookups + extra hop
TypicalPrimary key (Id)Foreign keys, search columns
SQL Server -- Primary key = clustered by default CREATE TABLE Customers ( Id INT PRIMARY KEY, -- Clustered Email NVARCHAR(255), Name NVARCHAR(100) ); -- Add non-clustered index CREATE NONCLUSTERED INDEX IX_Customers_Email ON Customers(Email); -- Query uses Email index, then jumps to clustered key SELECT * FROM Customers WHERE Email = 'alice@example.com';
EF Core modelBuilder.Entity<Customer>(e => { e.HasKey(c => c.Id); // Clustered by default // Non-clustered index e.HasIndex(c => c.Email) .IsUnique() .IsClustered(false); // For SQL Server: override with NONCLUSTERED e.ToTable("Customers", t => t.HasIndex("IX_Customers_Email") .IsClustered(false)); });
04

Query Execution Plans

Use execution plans to see if your query uses indexes or scans the table.

SQL Server: Check Execution Plan -- Enable actual execution plan (CTRL+L in SSMS) SET STATISTICS IO ON; -- Query with index SELECT * FROM Customers WHERE Email = 'alice@example.com'; -- Output: Scan count 1, logical reads: 2 (index seek = good) -- Query without index SELECT * FROM Customers WHERE Name LIKE '%Alice%'; -- Output: Scan count 1, logical reads: 5000 (table scan = bad)
Good Signs:

Index Seek = efficient. Table Scan = missing index or wrong WHERE clause.

05

Indexing Strategy

Don't index everything. Balance read performance against write cost.

Column UsageIndex?Why
WHERE Email = '...'YesDirect lookup
JOIN Orders ON CustomerIdYesJoin optimization
WHERE CreatedAt > '2024-01-01'YesRange scan
ORDER BY NameMaybeSort optimization (if large)
SELECT in projectionNoNo filter benefit
WHERE rare_column = 1NoLow selectivity
Cost of Indexes:

Every INSERT/UPDATE/DELETE must maintain all indexes. Each index adds disk and memory overhead.

06

EF Core Migrations with Indexes

Define indexes in entity configuration; migrations create them automatically.

Migration Code modelBuilder.Entity<Customer>(e => { e.HasKey(c => c.Id); e.Property(c => c.Email).IsRequired(); // Create index e.HasIndex(c => c.Email).IsUnique(); e.HasIndex(c => c.CreatedAt); // Composite index for common queries e.HasIndex(c => new { c.IsActive, c.CreatedAt }) .HasFilter("[IsActive] = 1") .HasDatabaseName("IX_Customers_ActiveCreated"); }); // Then: dotnet ef migrations add AddCustomerIndexes // This generates: // CREATE INDEX IX_Customers_Email ON Customers(Email) // CREATE INDEX IX_Customers_CreatedAt ON Customers(CreatedAt) // CREATE INDEX IX_Customers_ActiveCreated ON Customers(IsActive, CreatedAt) WHERE IsActive = 1
