Chapter 27

Indexing & Performance

Your query scans a million rows when it should jump to 10. Indexes fix this. Learn how to create them and when they help.

01

Requirement: "Make Searches Faster"

Business Requirement
Users search products by name. Takes 5 seconds. Need under 100ms. Index Name column.
SQL Developer Thinks
Without index, query scans all rows (table scan). With index on Name, database jumps to matching rows (index seek). Huge difference.
C# Developer Thinks
In EF Core, define indexes in entity configuration. EF generates migrations that create them in SQL.
Query TypeWithout IndexWith Index
Find by IDTable scan (1M rows)Index seek
WHERE Name = XScan all rowsIndex seek
WHERE Date GT XScan all rowsRange scan
SQL
CREATE INDEX IX_Products_Name ON Products(Name);
CREATE UNIQUE INDEX IX_Products_Sku ON Products(Sku);
CREATE INDEX IX_Orders_Date ON Orders(CreatedAt);
C# EF Core
e.HasIndex(p => p.Name);
e.HasIndex(p => p.Sku).IsUnique();
e.HasIndex(o => o.CreatedAt);
02

Composite & Filtered Indexes

Business Requirement
We filter by Category AND Price together. Only index active products.
SQL Developer Thinks
For Category + Price, create composite index on (CategoryId, Price). For active-only, use WHERE clause (filtered index).
C# Developer Thinks
In HasIndex, pass new {} for multiple columns. Add HasFilter for WHERE condition.
SQL
CREATE INDEX IX_Cat_Price
    ON Products(CategoryId, Price DESC);

CREATE INDEX IX_Products_Active
    ON Products(Name)
    WHERE IsActive = 1;
C# EF Core
e.HasIndex(p => new { p.CategoryId, p.Price })
    .IsDescending(false, true);

e.HasIndex(p => p.Name)
    .HasFilter("[IsActive] = 1");
03

Query Execution & Plans

Business Requirement
How do I know if my query uses an index? Check the execution plan.
SQL Developer Thinks
Enable actual execution plan (Ctrl+L in SSMS). Look for Index Seek (good) vs Table Scan (bad).
C# Developer Thinks
LINQ and Dapper generate parameterized SQL. Dapper is faster for reads because it skips ORM overhead.
SQL Server Execution Plan
SET STATISTICS IO ON;
SELECT * FROM Products WHERE Name = @Name;
-- Good: Index Seek (2 logical reads)
-- Bad: Table Scan (5000 logical reads)
04

When to Index & When to Skip

Business Requirement
Every index slows down inserts/updates. When is indexing worth it?
SQL Developer Thinks
Index columns in WHERE, JOIN, ORDER BY. Skip low-cardinality columns and columns only in projections.
C# Developer Thinks
Same logic applies. Use sp_helpindex to see what indexes exist. Remove unused ones.
Column UsageIndex?Why
WHERE Email = XYesDirect lookup
JOIN ON FKYesJoin optimization
ORDER BY DateMaybeIf large result set
SELECT in projection onlyNoNo filter benefit
