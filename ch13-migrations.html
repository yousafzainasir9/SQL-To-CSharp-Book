01 — ALTER TABLE VS ADD-MIGRATION

Parallel Thinking: SQL vs EF Migrations

In SQL, you manually write ALTER TABLE statements to modify schema: ADD COLUMN, DROP COLUMN, CREATE INDEX, etc. These are one-off scripts that modify the database directly. There's no version history unless you manage it yourself. In Entity Framework, migrations are version-controlled changesets generated from your model changes. Each migration is a timestamped C# file that tracks what changed and how to revert it.

The power of migrations: they're reversible, version-controlled in git, and integrated with your codebase. Rolling back to a previous schema is as simple as reverting a migration. For SQL, you'd need manual rollback scripts—error-prone and easily forgotten.

SQL: Manual ALTER TABLE 
-- Manual script (no version tracking)
ALTER TABLE Customers 
  ADD PhoneNumber NVARCHAR(20);

-- To undo, you must write reverse script:
-- ALTER TABLE Customers 
--   DROP COLUMN PhoneNumber;
-- No automatic tracking of what changed
EF Migration: Generated from Model Change 
// Step 1: Change model
public class Customer
{
  public int Id { get; set; }
  public string Name { get; set; }
  public string Email { get; set; }
  public string PhoneNumber { get; set; } // NEW
}

// Step 2: dotnet ef migrations add AddPhoneNumber
// EF generates migration file automatically

// Step 3: Migration file (auto-generated)
public partial class AddPhoneNumber 
  : Migration
{
  protected override void Up(
    MigrationBuilder mb)
  {
    mb.AddColumn(
      "PhoneNumber",
      "Customers",
      maxLength: 20,
      nullable: true);
  }

  protected override void Down(
    MigrationBuilder mb)
  {
    mb.DropColumn("PhoneNumber", "Customers");
  }
}
Migrations are Declarative Diffs

EF analyzes your current model against the previous snapshot and generates ONLY the changes needed. If you add a column, it generates AddColumn. If you rename a column (via a specific fluent API call), it generates a RenameColumn operation. The migration system tracks every state transition, making schema evolution trackable and reversible.

Every migration has an Up() method (apply changes) and a Down() method (revert changes). EF generates both automatically for standard operations. The workflow is: modify your model, run dotnet ef migrations add MigrationName, review the generated code, then run dotnet ef database update to apply it to your database.

02 — THE MIGRATION WORKFLOW

From Model Change to Database Update

The migration workflow is a four-step loop: (1) Change your model (add properties, change constraints), (2) Add migration (EF introspects your changes and generates a migration file), (3) Review the migration (ensure EF understood your intent correctly), (4) Update the database (execute the Up() method against your database). This cycle repeats every time you evolve your schema.

Each migration is a C# file with a timestamp in its name (e.g., 20240401120530_AddPhoneNumber.cs) stored in a Migrations folder. This folder is part of your source code repository, creating an audit trail of schema changes.

Complete Migration Workflow Example 
// STEP 1: Modify the model (Customer.cs)
public class Customer
{
  public int Id { get; set; }
  public string Name { get; set; }
  public string Email { get; set; }
  // NEW: Add PhoneNumber and LastOrderDate
  public string PhoneNumber { get; set; }
  public DateTime? LastOrderDate { get; set; }
}

// STEP 2: Add migration
// Terminal: dotnet ef migrations add AddPhoneAndLastOrder

// STEP 3: Review generated migration
// File: Migrations/20240401120530_AddPhoneAndLastOrder.cs
public partial class AddPhoneAndLastOrder 
  : Migration
{
  protected override void Up(
    MigrationBuilder migrationBuilder)
  {
    migrationBuilder.AddColumn(
      name: "PhoneNumber",
      table: "Customers",
      type: "nvarchar(max)",
      nullable: true);

    migrationBuilder.AddColumn(
      name: "LastOrderDate",
      table: "Customers",
      type: "datetime2",
      nullable: true);
  }

  protected override void Down(
    MigrationBuilder migrationBuilder)
  {
    migrationBuilder.DropColumn(
      name: "PhoneNumber",
      table: "Customers");

    migrationBuilder.DropColumn(
      name: "LastOrderDate",
      table: "Customers");
  }
}

// STEP 4: Apply migration to database
// Terminal: dotnet ef database update
// EF executes Up() against the database
Idempotency: The Key to Safe Deployments

Migrations are not idempotent by default (running them twice causes errors). In production, you use idempotent scripts (with IF NOT EXISTS checks) to ensure running the same script multiple times is safe. EF Core supports this via dotnet ef migrations script --idempotent, which wraps operations in SQL checks that prevent duplicate execution.

The __EFMigrationsHistory table tracks which migrations have been applied. When you run dotnet ef database update, EF queries this table, determines which migrations haven't run, and executes them in order. This ensures migrations run exactly once, in sequence, preventing conflicts.

03 — MIGRATION OPERATIONS AND THEIR SQL

Mapping Operations to SQL Statements

EF migration operations translate directly to SQL DDL statements. CreateTable maps to CREATE TABLE, AddColumn to ALTER TABLE ADD, CreateIndex to CREATE INDEX, etc. Understanding this mapping helps you debug migrations and predict their impact. Each operation has SQL consequences: some are safe (AddColumn with default), some require table rewrites (AddColumn without nullable).

The migration builder exposes these operations: migrationBuilder.CreateTable(), AddColumn(), DropColumn(), RenameColumn(), CreateIndex(), AddForeignKey(), etc. Each generates the corresponding SQL DDL.

8+ Migration Operations and Their SQL 
protected override void Up(
  MigrationBuilder mb)
{
  // 1. CreateTable → CREATE TABLE
  mb.CreateTable(
    name: "Customers",
    columns: table => new
    {
      Id = table.Column(nullable: false)
        .Annotation("SqlServer:Identity", 
          "1, 1"),
      Name = table.Column(
        maxLength: 100, nullable: false),
      Email = table.Column(
        maxLength: 100, nullable: false)
    },
    constraints: table =>
    {
      table.PrimaryKey("PK_Customers", 
        x => x.Id);
      table.UniqueConstraint("UX_Email", 
        x => x.Email);
    });
  // SQL: CREATE TABLE Customers (...)

  // 2. AddColumn → ALTER TABLE ADD
  mb.AddColumn(
    name: "PhoneNumber",
    table: "Customers",
    maxLength: 20,
    nullable: true);
  // SQL: ALTER TABLE Customers 
  //   ADD PhoneNumber NVARCHAR(20) NULL;

  // 3. DropColumn → ALTER TABLE DROP
  mb.DropColumn(
    name: "LegacyField",
    table: "Customers");
  // SQL: ALTER TABLE Customers 
  //   DROP COLUMN LegacyField;

  // 4. RenameColumn → sp_rename
  mb.RenameColumn(
    name: "CompanyName",
    table: "Customers",
    newName: "OrganizationName");
  // SQL: EXEC sp_rename 'Customers.CompanyName',
  //   'OrganizationName';

  // 5. CreateIndex → CREATE INDEX
  mb.CreateIndex(
    name: "IX_Email",
    table: "Customers",
    column: "Email");
  // SQL: CREATE INDEX IX_Email 
  //   ON Customers(Email);

  // 6. DropIndex → DROP INDEX
  mb.DropIndex(
    name: "IX_Email",
    table: "Customers");
  // SQL: DROP INDEX IX_Email ON Customers;

  // 7. AddForeignKey → ALTER TABLE ADD CONSTRAINT
  mb.AddForeignKey(
    name: "FK_Orders_Customers",
    table: "Orders",
    column: "CustomerId",
    principalTable: "Customers",
    principalColumn: "Id",
    onDelete: ReferentialAction.Cascade);
  // SQL: ALTER TABLE Orders 
  //   ADD CONSTRAINT FK_Orders_Customers
  //   FOREIGN KEY (CustomerId) 
  //   REFERENCES Customers(Id) 
  //   ON DELETE CASCADE;

  // 8. DropForeignKey → ALTER TABLE DROP CONSTRAINT
  mb.DropForeignKey(
    name: "FK_Orders_Customers",
    table: "Orders");
  // SQL: ALTER TABLE Orders 
  //   DROP CONSTRAINT FK_Orders_Customers;

  // 9. AlterColumn → ALTER TABLE ALTER COLUMN
  mb.AlterColumn(
    name: "Name",
    table: "Customers",
    type: "nvarchar(200)",
    nullable: false);
  // SQL: ALTER TABLE Customers 
  //   ALTER COLUMN Name NVARCHAR(200) NOT NULL;

  // 10. CreateSequence (rarely used)
  mb.CreateSequence(
    name: "OrderNumberSequence",
    startValue: 1000L,
    incrementBy: 1);
  // SQL: CREATE SEQUENCE OrderNumberSequence 
  //   START WITH 1000 INCREMENT BY 1;
}
Performance Impact: RenameColumn is Dangerous

Some operations cause table rewrites: AddColumn on a NOT NULL column without a default, DropColumn, AlterColumn to narrow the type. These lock the table during execution. RenameColumn via sp_rename is deceptive: it works, but downstream code using old column names will break. Always verify the generated SQL for large tables.

When you run dotnet ef migrations add, EF generates the Up() and Down() methods. For standard operations, both are generated automatically. For complex scenarios (like RenameColumn where EF can't determine intent), you must manually edit the migration to specify the operation correctly. Always review migrations before applying them.

04 — DATA SEEDING WITH HASDATA

HasData: Seed Data as Migrations

HasData allows you to seed initial data in your model configuration. This data is tracked by migrations and generates INSERT statements automatically. Unlike manual INSERT scripts, seed data is version-controlled and reproducible: running migrations applies all seed data atomically. This is ideal for reference data: roles, status values, system users.

HasData is configured in OnModelCreating, and EF generates InsertData() calls in your migration. The seed data becomes part of the schema version, not a separate concern.

SQL: Manual INSERT 
-- Separate script (no version tracking)
INSERT INTO Roles (Id, Name) 
  VALUES (1, 'Admin');
INSERT INTO Roles (Id, Name) 
  VALUES (2, 'User');
INSERT INTO Roles (Id, Name) 
  VALUES (3, 'Guest');

-- To undo, manually write DELETE
EF: HasData with Migration 
protected override void OnModelCreating(
  ModelBuilder mb)
{
  mb.Entity().HasData(
    new Role { Id = 1, Name = "Admin" },
    new Role { Id = 2, Name = "User" },
    new Role { Id = 3, Name = "Guest" }
  );
}

// Migration generated automatically:
// migrationBuilder.InsertData(
//   table: "Roles",
//   columns: new[] { "Id", "Name" },
//   values: new object[,] {
//     { 1, "Admin" },
//     { 2, "User" },
//     { 3, "Guest" }
//   });
HasData Tracking: Immutable Reference Data

EF tracks HasData entities and generates UpdateData() or DeleteData() operations if you remove or modify them in subsequent migrations. This is powerful for reference data that changes rarely, but dangerous if misused: modifying HasData in production migrations requires careful planning to avoid data loss or corruption on existing databases where data has been manually modified.

HasData Workflow and Migration Generation 
// OnModelCreating: Define seed data
protected override void OnModelCreating(
  ModelBuilder mb)
{
  mb.Entity().HasData(
    new OrderStatus 
    { 
      Id = 1, 
      Name = "Pending",
      Description = "Order received" 
    },
    new OrderStatus 
    { 
      Id = 2, 
      Name = "Shipped",
      Description = "Order sent" 
    },
    new OrderStatus 
    { 
      Id = 3, 
      Name = "Delivered",
      Description = "Order delivered" 
    }
  );
}

// Terminal: dotnet ef migrations add AddOrderStatuses

// Generated Migration
public partial class AddOrderStatuses : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.CreateTable(
      name: "OrderStatuses",
      columns: table => new
      {
        Id = table.Column(nullable: false),
        Name = table.Column(
          maxLength: 50, nullable: false),
        Description = table.Column(
          nullable: true)
      },
      constraints: table =>
      {
        table.PrimaryKey("PK_OrderStatuses", x => x.Id);
      });

    mb.InsertData(
      table: "OrderStatuses",
      columns: new[] { "Id", "Name", "Description" },
      values: new object[,]
      {
        { 1, "Pending", "Order received" },
        { 2, "Shipped", "Order sent" },
        { 3, "Delivered", "Order delivered" }
      });
  }

  protected override void Down(MigrationBuilder mb)
  {
    mb.DropTable(name: "OrderStatuses");
  }
}

HasData is excellent for static reference data that comes with your application. It's less suitable for user-generated data or data that changes frequently. For that, use separate seeding scripts or a database seeding service that runs during application startup.

05 — ROLLING BACK: DOWN MIGRATIONS

Reverting to Previous Schemas

Every migration has a Down() method that reverses the Up() changes. If you applied a migration and realized it's incorrect, you can revert to a previous migration. The terminal command is: dotnet ef database update PreviousMigrationName. EF executes Down() methods in reverse order until it reaches the target migration. This is powerful for development and testing, but must be used carefully in production because Down() may result in data loss.

The __EFMigrationsHistory table records which migrations have been applied. Running dotnet ef database update targets a specific migration by name or removes all migrations if no name is provided.

Rollback Scenario and Down() Methods 
// Scenario: You added a migration, but realized it's wrong
// Migrations applied: InitialCreate, AddPhoneNumber, AddPhoneIndex

// Terminal: dotnet ef database update AddPhoneNumber
// This reverts AddPhoneIndex (runs its Down() method)

// Migration: AddPhoneIndex.cs
public partial class AddPhoneIndex : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.CreateIndex(
      name: "IX_Customers_PhoneNumber",
      table: "Customers",
      column: "PhoneNumber");
  }

  protected override void Down(MigrationBuilder mb)
  {
    mb.DropIndex(
      name: "IX_Customers_PhoneNumber",
      table: "Customers");
    // Reverts to state before index was created
  }
}

// Key points:
// 1. Down() must undo exactly what Up() did
// 2. Down() may cause data loss if Up() added NOT NULL columns
// 3. In production, rollbacks are rare; prevention is better
// 4. Always backup before running migrations on prod
// 5. Test migrations against production schema first

// Check current migration status
// Terminal: dotnet ef migrations list
// Shows all migrations and which are applied
Rollbacks Can Fail or Cause Data Loss

If your Up() method adds a NOT NULL column without a default, Down() removes it—data in that column is lost permanently. Down() is auto-generated for standard operations but may need manual editing for complex scenarios. Always understand what your Down() method does before running it on a database you care about.

In development and testing, rollbacks are common and safe. In production, they're rare and dangerous. Instead of rolling back, you typically create a new migration that fixes the problem. This way, the migration history stays linear and auditable.

06 — CUSTOM SQL IN MIGRATIONS

Data Migrations and Stored Procedures

Sometimes migrations need more than standard table changes. You might need to populate data, rename columns safely, or deploy stored procedures. For these, you write custom SQL directly in the migration using migrationBuilder.Sql(). This allows arbitrary SQL execution within the migration framework, tracked and versioned with your schema changes.

Custom SQL is powerful but requires care: unlike auto-generated operations, you must write both Up() and Down() to be fully reversible. Down() should undo what Up() did.

3+ Real-World Custom SQL Examples 
// Example 1: Data migration with computed field
public partial class PopulateFullName : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    // Add FullName column
    mb.AddColumn(
      name: "FullName",
      table: "Customers",
      type: "nvarchar(200)",
      nullable: true);

    // Populate FullName from FirstName + LastName
    mb.Sql(
      @"UPDATE Customers 
        SET FullName = FirstName + ' ' + LastName
        WHERE FirstName IS NOT NULL 
          AND LastName IS NOT NULL");
  }

  protected override void Down(MigrationBuilder mb)
  {
    mb.DropColumn("FullName", "Customers");
  }
}

// Example 2: Safe column rename with custom SQL
public partial class RenameCompanyName : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    // Add new column
    mb.AddColumn(
      name: "OrganizationName",
      table: "Customers",
      maxLength: 200,
      nullable: true);

    // Copy data from old to new
    mb.Sql(
      @"UPDATE Customers 
        SET OrganizationName = CompanyName");

    // Drop old column
    mb.DropColumn("CompanyName", "Customers");
  }

  protected override void Down(MigrationBuilder mb)
  {
    mb.AddColumn(
      name: "CompanyName",
      table: "Customers",
      maxLength: 200,
      nullable: true);

    mb.Sql(
      @"UPDATE Customers 
        SET CompanyName = OrganizationName");

    mb.DropColumn("OrganizationName", "Customers");
  }
}

// Example 3: Deploy stored procedure
public partial class CreateOrderReportProcedure 
  : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.Sql(@"
      CREATE PROCEDURE sp_GetMonthlyOrderReport
        @Month INT,
        @Year INT
      AS
      BEGIN
        SELECT 
          DATETRUNC(DAY, OrderDate) AS OrderDay,
          COUNT(*) AS OrderCount,
          SUM(Total) AS DailyTotal
        FROM Orders
        WHERE MONTH(OrderDate) = @Month
          AND YEAR(OrderDate) = @Year
        GROUP BY DATETRUNC(DAY, OrderDate)
        ORDER BY OrderDay;
      END
    ");
  }

  protected override void Down(MigrationBuilder mb)
  {
    mb.Sql(
      @"DROP PROCEDURE IF EXISTS 
        sp_GetMonthlyOrderReport");
  }
}
Custom SQL is Unvalidated: Test Thoroughly

EF doesn't analyze or validate custom SQL. If your Sql() call has a syntax error, it won't be caught until you run the migration against the database. Always test migrations against a test database before applying to production.

Custom SQL migrations are essential for real-world schema evolution, especially when migrating data or deploying database objects like stored procedures, views, or functions that EF can't express via Fluent API.

07 — PRODUCTION DEPLOYMENT STRATEGIES

Bundles, Idempotent Scripts, and CI/CD Integration

In production, migrations must be reliable and safe. Standard migrations require EF runtime and tools, which you may not want on production servers. EF Core offers two production patterns: (1) Bundle: dotnet ef migrations bundle generates a self-contained executable that applies migrations without EF Core tools, (2) Idempotent script: dotnet ef migrations script --idempotent generates SQL with IF NOT EXISTS checks that safely run multiple times. Both integrate with CI/CD pipelines for automated, safe deployments.

The canonical production pattern: generate a bundle or idempotent script in your CI/CD pipeline, review it, then execute it during deployment. No EF tools needed on production; migrations are pure SQL.

Production Deployment Strategies 
// STRATEGY 1: Migrations Bundle
// Terminal (in build pipeline):
// dotnet ef migrations bundle --output /tmp/migrationBundle.exe

// On production:
// .\migrationBundle.exe --connection "Server=prod-db;..."
// Self-contained, no EF tools needed, idempotent

// STRATEGY 2: Idempotent SQL Script
// Terminal (in build pipeline):
// dotnet ef migrations script --idempotent --output /tmp/migrations.sql

// Generated SQL includes IF NOT EXISTS checks:
// IF NOT EXISTS (SELECT 1 FROM __EFMigrationsHistory ...)
// BEGIN
//   CREATE TABLE ...
// END

// On production:
// sqlcmd -S prod-server -d prod-db -i /tmp/migrations.sql
// Pure SQL, no .NET runtime needed

// STRATEGY 3: CI/CD Integration (GitHub Actions Example)
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2

      - name: Setup .NET
        uses: actions/setup-dotnet@v1
        with:
          dotnet-version: '6.0'

      - name: Generate Migration Bundle
        run: |
          dotnet ef migrations bundle \
            --output ./migrationBundle \
            --self-contained

      - name: Upload to Deployment Server
        run: scp ./migrationBundle deploy@prod:/home/deploy/

      - name: Apply Migrations
        run: |
          ssh deploy@prod '/home/deploy/migrationBundle \
            --connection "Server=prod-db;Database=MyApp;..."'

      - name: Verify Deployment
        run: |
          # Query database to verify schema is up-to-date
          sqlcmd -S prod-db -d MyApp \
            -Q "SELECT COUNT(*) FROM __EFMigrationsHistory" 

// STRATEGY 4: Blue-Green Deployment
// 1. Maintain two identical database schemas (blue, green)
// 2. Apply migrations to green database first
// 3. Run integration tests against green
// 4. If tests pass, switch application traffic to green
// 5. If tests fail, keep traffic on blue; debug green
// 6. Next deployment swaps roles

// STRATEGY 5: Monitoring and Rollback Plan
// Track migration application:
-- Query migration history
SELECT * FROM __EFMigrationsHistory ORDER BY Id DESC;

// If a migration causes issues:
// 1. Immediately switch traffic back to previous schema (blue-green)
// 2. Run Down() migration to revert
// 3. Create new migration to fix the issue
// 4. Test thoroughly before re-deploying
Never Run Migrations Directly on Production

In production, always use bundles or idempotent scripts generated in your CI/CD pipeline. Never log into production and run dotnet ef database update manually. This prevents inconsistencies, creates audit trails, and allows review before execution.

Best practice: test migrations against a production-like schema first. Many teams run migrations against a staging database that mirrors production, verify everything works, then run the same migration on production. This eliminates surprises and validates that Down() works correctly if you need to rollback.

08 — SCHEMA EVOLUTION THEORY

SQL ALTER TABLE History vs Migrations as Version Control

In traditional SQL development, schema changes happen via ad-hoc ALTER TABLE statements, manually executed by DBAs. The schema history is implicit: you have to query system tables to understand what changed and when. Migrations flip this model. Instead of ad-hoc SQL, you write C# code that generates SQL. The migration history is explicit and version-controlled in Git alongside your application code. Every schema change is a versioned artifact, reproducible and auditable.

This shift has profound implications. With migrations, you can recreate your entire database schema from scratch by replaying all migrations in order. If you want to understand why a column exists, you look at the migration that created it. If you need to know who added a column and when, you check Git blame on that migration file. Migrations make schema history transparent and testable—you can run migrations against a test database to validate they work before applying to production.

Traditional Ad-Hoc SQL Approach 
-- Changes made manually by DBA
-- No version control, implicit history

-- Monday, 10 AM (John did this)
ALTER TABLE Customers ADD Email NVARCHAR(255);

-- Wednesday, 2 PM (Sarah did this)
ALTER TABLE Customers ADD PhoneNumber VARCHAR(20);

-- History is lost: no audit trail, unclear why

-- To understand current schema:
-- Query INFORMATION_SCHEMA or sp_help
SELECT COLUMN_NAME, DATA_TYPE, IS_NULLABLE
FROM INFORMATION_SCHEMA.COLUMNS
WHERE TABLE_NAME = 'Customers';

-- Problem: No way to rollback
-- Problem: No reproducibility
-- Problem: Unclear who changed what
EF Migrations Approach (Version Controlled) 
// Git history shows exact schema evolution
// Each migration is a versioned file

// Migrations/20240101_InitialCreate.cs
public partial class InitialCreate : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.CreateTable("Customers", c => new
    {
      Id = c.Column(),
      Name = c.Column()
    });
  }
}

// Migrations/20240115_AddEmail.cs
public partial class AddEmail : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.AddColumn(
      name: "Email",
      table: "Customers");
  }
}

// Migrations/20240117_AddPhone.cs
public partial class AddPhone : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.AddColumn(
      name: "PhoneNumber",
      table: "Customers");
  }
}

// Replay history to recreate schema
// dotnet ef database update
// Executes all migrations in order
// Result: identical schema on dev, test, prod

// Audit trail visible via Git
// git log --oneline Migrations/
// Shows who changed what and when

// Rollback capability
// dotnet ef database update AddEmail
// Reverts AddPhone migration safely

// Reproducible: new developer runs
// dotnet ef database update
// Gets identical schema, no manual steps
Migrations = Schema as Code

Treating migrations as code—versioned in Git, reviewed in PRs, tested before merge—brings software engineering rigor to database schema. You can't accidentally modify production schema outside of migrations. All changes flow through the same pipeline as application code changes, ensuring traceability and auditability.

The fundamental benefit: a new developer can clone the repository and run one command to have the entire database schema set up locally, with all seed data. This is impossible with ad-hoc SQL. Every developer, every test environment, and every production instance has identical schema because they all replay the same migrations.

09 — DATA MIGRATIONS DURING SCHEMA CHANGES

Migrating Data While Evolving Schema

Adding a column is simple. Renaming a column safely is complex. You need to create a new column, copy data, drop the old, all while the application might be reading the old column. Migrations handle this via custom SQL. The pattern: (1) Add new column, (2) Populate from old column, (3) Deploy application code that uses new column, (4) Drop old column in next migration. For zero-downtime deployads, you deploy application code between migrations, allowing old and new code to coexist during transition.

Data migrations are also essential when normalizing denormalized data or applying business logic transformations. If you're splitting a FullName column into FirstName and LastName, you write a migration that parses the full name and populates the new columns. The migration framework ensures this transformation is atomic and versioned.

Data Migration Pattern: Safe Column Rename 
// Migration 1: AddNewEmailField
// Goal: Safely migrate from Email to EmailAddress
public partial class AddNewEmailField : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    // Create new column (nullable for existing rows)
    mb.AddColumn(
      name: "EmailAddress",
      table: "Customers",
      type: "nvarchar(255)",
      nullable: true);

    // Copy data from old to new column
    mb.Sql(
      @"UPDATE Customers
        SET EmailAddress = Email
        WHERE Email IS NOT NULL");
  }

  protected override void Down(MigrationBuilder mb)
  {
    mb.DropColumn("EmailAddress", "Customers");
  }
}

// Application code deployed here
// Update application to use EmailAddress
// Old Email column still readable for safety

// Migration 2: DropOldEmailField
// After application change is deployed everywhere
public partial class DropOldEmailField : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.DropColumn("Email", "Customers");
  }

  protected override void Down(MigrationBuilder mb)
  {
    mb.AddColumn(
      name: "Email",
      table: "Customers",
      type: "nvarchar(255)",
      nullable: true);

    // Restore data if needed
    mb.Sql(
      @"UPDATE Customers
        SET Email = EmailAddress
        WHERE EmailAddress IS NOT NULL");
  }
}

// Timeline:
// Day 1: Deploy migration 1 (add EmailAddress)
// Day 1-2: Deploy app code change (use EmailAddress)
// Day 2: Deploy migration 2 (drop Email)
// Result: Zero downtime, safe transition

// Data transformation example
public partial class NormalizeNames : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    // Add columns for normalized data
    mb.AddColumn(
      name: "FirstName",
      table: "Customers",
      nullable: true);

    mb.AddColumn(
      name: "LastName",
      table: "Customers",
      nullable: true);

    // Split FullName into components

    mb.Sql(
      @"UPDATE Customers
        SET FirstName = LTRIM(SUBSTRING(
          FullName, 1,
          CHARINDEX(' ', FullName) - 1)),
            LastName = RTRIM(SUBSTRING(

public partial class MakeEmailRequired : Migration

  protected override void Up(MigrationBuilder mb)
  {
    // First, populate any NULL emails with placeholder
    mb.Sql(
      @"UPDATE Customers
        SET Email = 'unknown@example.com'
        WHERE Email IS NULL");

      name: "Email",

Separate Schema Changes From Data Migrations

Complex data migrations should be tested separately. Run the data-heavy Sql() against a production-like dataset to ensure it completes in acceptable time. Adding an index to a 1-billion-row table takes hours; migrations should account for this (use ONLINE index creation in SQL Server, or background migration strategies).

For large tables, consider splitting data migrations into separate deployment windows. Migrate data in batches outside of peak hours. Migrations are powerful, but data transformation at scale requires planning. Monitor migration performance; if a migration takes longer than acceptable, create new strategies like bulk copy utilities alongside the migration.

10 — MIGRATION BUNDLES FOR DEPLOYMENT

Self-Contained Executables for Safe Production Deployments

Bundles are self-contained executables that apply migrations without EF Core tools installed on the production server. You generate a bundle in your CI/CD pipeline, ship it with your application, and run it as part of deployment. This is safer and more reliable than relying on dotnet ef tools on production. Bundles handle connection retries, idempotency, and rollback logging automatically.

The bundle contains compiled migration code. When executed, it reads the __EFMigrationsHistory table, determines which migrations need applying, and applies them atomically. Bundles work offline; they don't need an internet connection or access to your .NET project files. For regulated environments, bundles provide an audit trail: you know exactly which migration bundle was deployed when.

Migration Bundle Generation and Deployment 
// In CI/CD pipeline (GitHub Actions example)
name: Create Migration Bundle

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2

      - name: Setup .NET
        uses: actions/setup-dotnet@v1
        with:
          dotnet-version: '6.0'

      - name: Restore
        run: dotnet restore

      - name: Generate Migration Bundle
        run: |
          dotnet ef migrations bundle \
            --output ./publish/migrationBundle \
            --project ./src/Data/Data.csproj \
            --self-contained

      - name: Upload bundle as artifact
        uses: actions/upload-artifact@v2
        with:
          name: migration-bundle
          path: ./publish/migrationBundle

// On production server
// Download bundle from artifact
// Run it:

// Windows
.\migrationBundle --connection "Server=prod-db;Database=MyApp;User=admin;Password=secret;"

// Linux
./migrationBundle --connection "Server=prod-db;Database=MyApp;User=admin;Password=secret;"

// Exit codes:
// 0: Success
// 1: Failure (check logs)
// Bundle output includes SQL executed and timestamps

// Bundle features:
// - Self-contained: no .NET SDK needed
// - Idempotent: safe to run multiple times
// - Connection retry: handles transient failures
// - Detailed logging: shows what ran
// - No dependency on application being deployed

// Versioning bundles
// Timestamp bundle to track which was deployed
// Mv migrationBundle migrationBundle_2024-01-20_15-30-45.exe
// Deploy and track which version was used

// Rollback with bundles
// Bundles can revert migrations if needed:
./migrationBundle --connection "..." --targetMigration "PreviousMigrationName"

// This is safer than manual rollback:
// - Recorded in deployment logs
// - Same code path as forward migrations
// - Tested before deployment
Bundles Are the Production Standard

For production, always use bundles. Never use dotnet ef database update on production servers. Bundles are self-contained, auditable, and require no EF Core tools. They're the safest way to apply migrations in regulated environments where you need to prove what ran and when.

Bundles also support connection strings via command-line arguments, allowing the same bundle to run against multiple databases. This is crucial for multi-environment deployments: build once, deploy to dev/staging/prod with different connection strings.

11 — SEED DATA WITH HASDATA()

Embedding Reference Data in Migrations

HasData() in OnModelCreating seeds data into migrations. When you run dotnet ef migrations add, EF detects HasData() calls and generates migration code to insert that data. This ensures reference data (lookup tables, static enums, configuration values) exists on every database instance without manual seeding scripts. Seed data is version-controlled alongside your schema, making it reproducible.

HasData() is perfect for reference data but unsuitable for user-generated data. Use it for status enums, default configurations, and lookup tables that your application depends on. For large datasets or data that changes, use separate seeding services or database snapshots.

HasData() Seed Data Patterns 
public class AppDbContext : DbContext
{
  public DbSet OrderStatuses { get; set; }
  public DbSet Countries { get; set; }
  public DbSet PaymentMethods { get; set; }

  protected override void OnModelCreating(
    ModelBuilder modelBuilder)
  {
    // Seed OrderStatus lookup table
    modelBuilder.Entity()
      .HasData(
        new OrderStatus { Id = 1, Name = "Pending" },
        new OrderStatus { Id = 2, Name = "Processing" },
        new OrderStatus { Id = 3, Name = "Shipped" },
        new OrderStatus { Id = 4, Name = "Delivered" },
        new OrderStatus { Id = 5, Name = "Cancelled" }
      );

    // Seed Countries with common ones
    modelBuilder.Entity()
      .HasData(
        new Country { Id = 1, Code = "US", Name = "United States" },
        new Country { Id = 2, Code = "CA", Name = "Canada" },
        new Country { Id = 3, Code = "UK", Name = "United Kingdom" },
        new Country { Id = 4, Code = "DE", Name = "Germany" },
        new Country { Id = 5, Code = "FR", Name = "France" }
      );

    // Seed PaymentMethods
    modelBuilder.Entity()
      .HasData(
        new PaymentMethod
        {
          Id = 1,
          Name = "Credit Card",
          IsActive = true
        },
        new PaymentMethod
        {
          Id = 2,
          Name = "PayPal",
          IsActive = true
        },
        new PaymentMethod
        {
          Id = 3,
          Name = "Bank Transfer",
          IsActive = true
        },
        new PaymentMethod
        {
          Id = 4,
          Name = "Cheque",
          IsActive = false
        }
      );
  }
}

// Generated migration includes:
public partial class SeedInitialData : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.InsertData(
      table: "OrderStatus",
      columns: new[] { "Id", "Name" },

        // ... more countries
      });

    // ... PaymentMethods too
  }

  {
    mb.DeleteData(
      table: "OrderStatus",
      keyColumn: "Id",
      keyValues: new object[] { 1, 2, 3, 4, 5 });

      table: "Countries",
      keyColumn: "Id",

      table: "PaymentMethods",
      keyColumn: "Id",

// Updating seed data in subsequent migration
public partial class UpdatePaymentMethods : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.UpdateData(
      table: "PaymentMethods",
      keyColumn: "Id",
      keyValue: 4,
      column: "IsActive",
      value: true); // Cheque is now active again
  }

  protected override void Down(MigrationBuilder mb)

      table: "PaymentMethods",
      keyColumn: "Id",

      column: "IsActive",
      value: false);
  }
}
HasData() is for Small, Static Datasets

HasData() is ideal for enums, status tables, and configuration. Don't use it for datasets larger than ~1000 rows or data that changes frequently. For those, maintain separate seed scripts or use SQL Server's bulk insert mechanisms. HasData() keeps seed data visible in code and version-controlled, but large datasets are unwieldy in migrations.

A practical pattern: use HasData() for schema-critical reference data that your application requires to function. Use separate seeding services for demo data or test data that's useful but not required.

12 — TEAM MIGRATION CONFLICTS

Handling Concurrent Migration Development

In teams, multiple developers create migrations simultaneously. If developer A creates "AddPhoneNumber" and developer B creates "AddEmail", and both migrations are added before being merged, Git will have two branches with conflicting migration histories. Migrations are named by timestamp, making conflicts rare but possible. The solution: coordinate migration creation or use a naming convention that prevents conflicts (e.g., developer initials + feature).

EF provides dotnet ef migrations add with --configuration flag to handle this. If a conflict is detected, you can regenerate migrations to combine them or manually resolve the conflict by editing migration classes. The __EFMigrationsHistory table prevents accidentally running migrations twice—it tracks which migrations have been applied, so conflicting migrations can coexist as long as their SQL is compatible.

Migration Conflict Resolution Patterns 
// Scenario: Two developers create migrations simultaneously

// Developer A: AddPhoneNumber migration
// Migrations/20240120_AddPhoneNumber.cs
public partial class AddPhoneNumber : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.AddColumn(
      name: "PhoneNumber",
      table: "Customers",
      maxLength: 20);
  }

  protected override void Down(MigrationBuilder mb)
  {
    mb.DropColumn("PhoneNumber", "Customers");
  }
}

// Developer B: AddEmail migration
// Migrations/20240120_AddEmail.cs (same timestamp!)
public partial class AddEmail : Migration
{
  protected override void Up(MigrationBuilder mb)
  {
    mb.AddColumn(
      name: "Email",
      table: "Customers",
      maxLength: 255);
  }

  protected override void Down(MigrationBuilder mb)
  {
    mb.DropColumn("Email", "Customers");
  }
}

// Both migrations committed to main branch
// Result: Git conflict, unclear migration order

// Solution 1: Regenerate to combine migrations
// Terminal: dotnet ef migrations remove
// This removes the last migration added
// Then: dotnet ef migrations add CombinedChanges
// EF generates one migration with both changes

// Solution 2: Manual resolution
// Edit migration names to have unique timestamps:
// 20240120_120000_AddPhoneNumber.cs
// 20240120_120001_AddEmail.cs
// Then update __EFMigrationsHistory if needed

// Solution 3: Establish naming convention
// AddMigrationName --schema [InitialName]
// Example:
// Alice_AddPhoneNumber
// Bob_AddEmail
// Prevents timestamp collision

// Best practice in teams:
// 1. Pull latest before creating migration
// 2. Coordinate large schema changes
// 3. Use feature branches until migration is final
// 4. Review migrations in pull requests before merge
// 5. Run integration tests against merged migrations

// Checking for migration conflicts programmatically
public class MigrationService
{
  private readonly AppDbContext _context;

  public MigrationService(AppDbContext context)
  {
    _context = context;
  }

  public async Task>
    GetPendingMigrationsAsync()
  {
    // Returns migrations not yet applied
    var migrations = await _context.Database
      .GetPendingMigrationsAsync();

    return migrations.ToList();
  }

  public async Task>
    GetAppliedMigrationsAsync()
  {
    // Returns migrations already applied
    var migrations = await _context.Database
      .GetAppliedMigrationsAsync();

    return migrations.ToList();
  }

  // Use this to detect unmerged migrations
  public bool HasConflictingMigrations()
  {
    var pending = _context.Database
      .GetPendingMigrations()
      .ToList();

    // If multiple pending migrations exist
    // and they modify the same table,
    // log a warning
    return pending.Count > 1;
  }
}
Merge Migrations in Pull Request Reviews

Always review migrations in pull requests. Ensure migrations from different developers don't conflict or cause unexpected schema. If two developers both modify Customers table, the second migration might fail. Catch these issues in review, not in production.

A team practice: designate one person to merge long-running branches to prevent migration conflicts. Once migrations are finalized, they're immutable in production—changing applied migrations causes data corruption. Prevent this by enforcing strict migration review processes.

13 — REVERSE ENGINEERING (SCAFFOLDING) EXISTING DATABASES

Generating Models From Existing Database Schemas

If you inherit an existing database, you can scaffold DbContext and models from it. The dotnet ef dbcontext scaffold command reverse-engineers the schema, generating C# classes that match existing tables. This is a one-time operation; afterward, you manage the schema via migrations as usual. Scaffolding is also useful for exploring unfamiliar databases or integrating with legacy systems.

Scaffolding generates complete OnModelCreating configurations, including foreign keys, indexes, and constraints. The generated code often needs cleanup—removing unwanted tables, adjusting naming conventions—but it's a great starting point. After scaffolding, move the generated files into your project structure and create an initial migration to version control the existing schema.

Scaffolding Existing Database 
// Terminal: Scaffold an existing database
dotnet ef dbcontext scaffold \
  "Server=legacy-db;Database=OldApp;User=sa;Password=secret;" \
  Microsoft.EntityFrameworkCore.SqlServer \
  --output-dir Models \
  --context OldAppContext \
  --data-annotations \
  --use-database-names

// Generated OldAppContext.cs:
public partial class OldAppContext : DbContext
{
  public OldAppContext()
  {
  }

  public OldAppContext(
    DbContextOptions options)
    : base(options)
  {
  }

  public virtual DbSet Customers { get; set; }
  public virtual DbSet Orders { get; set; }
  public virtual DbSet Products { get; set; }

  protected override void OnConfiguring(
    DbContextOptionsBuilder optionsBuilder)
  {
    if (!optionsBuilder.IsConfigured)
    {
      optionsBuilder.UseSqlServer(
        "Server=legacy-db;Database=OldApp;...");
    }
  }

  protected override void OnModelCreating(
    ModelBuilder modelBuilder)
  {
    // All tables, constraints, indexes mapped
    modelBuilder.Entity(entity =>
    {
      entity.HasKey(e => e.CustomerId)
        .HasName("PK_Customer");

      entity.Property(e => e.Name)
        .IsRequired()
        .HasMaxLength(100);

      entity.Property(e => e.CreatedDate)
        .HasDefaultValueSql("GETUTCDATE()");
    });

    modelBuilder.Entity(entity =>
    {
      entity.HasKey(e => e.OrderId);

      entity.HasOne(d => d.Customer)
        .WithMany(p => p.Orders)
        .HasForeignKey(d => d.CustomerId);
    });

    OnModelCreatingPartial(modelBuilder);
  }

  partial void OnModelCreatingPartial(
    ModelBuilder modelBuilder);
}

dotnet ef migrations add InitialScaffold

// exists), but it establishes the migration baseline

// future schema changes

// 1. Use --data-annotations for attribute-based config
// 2. Use --context-dir for custom path
// 3. Use --models-dir for custom model path
// 4. Generate to separate branch first, review

Scaffolding is a Starting Point, Not Permanent

Use scaffolding to bootstrap models from existing databases. After scaffolding, treat the generated code like any other code: review it, test it, refactor as needed. Don't expect scaffolding to generate perfect code; use it to save initial typing and mapping work.

Scaffolding is also useful during development when you want to understand a complex schema. Generate models, examine the OnModelCreating configuration, and learn how EF mapped the database. This is especially valuable when dealing with legacy systems or unfamiliar schemas.

14 — SQL SCRIPT GENERATION

Generating SQL Scripts From Migrations

The dotnet ef migrations script command generates pure SQL from your migrations. You can generate idempotent scripts (safe to run multiple times) or targeted scripts (from one migration to another). This is essential for production deployments where you want to review SQL before executing and maintain complete SQL audit trails. Generated scripts include all migration operations translated to SQL DDL and DML.

Script generation is also useful for documentation. You can generate the complete database creation script and share it with DBAs, include it in deployment documentation, or archive it for compliance. The script shows exactly what will be applied, allowing for review and approval before deployment.

SQL Script Generation From Migrations 
// Terminal: Generate all pending migrations as SQL
dotnet ef migrations script --output migrations.sql

// Generated SQL (example):
BEGIN TRANSACTION;
GO

CREATE TABLE [Customers] (
  [Id] int NOT NULL IDENTITY,
  [Name] nvarchar(max) NOT NULL,
  [Email] nvarchar(255),
  [CreatedDate] datetime2 NOT NULL
  CONSTRAINT [PK_Customers] PRIMARY KEY ([Id])
);
GO

CREATE TABLE [Orders] (
  [Id] int NOT NULL IDENTITY,
  [CustomerId] int NOT NULL,
  [OrderDate] datetime2 NOT NULL,
  [Total] decimal(18, 2),
  CONSTRAINT [PK_Orders] PRIMARY KEY ([Id]),
  CONSTRAINT [FK_Orders_Customers_CustomerId]
    FOREIGN KEY ([CustomerId])
    REFERENCES [Customers] ([Id])
    ON DELETE CASCADE
);
GO

-- Insert seed data
INSERT INTO [OrderStatuses] ([Id], [Name])
VALUES (1, 'Pending');
INSERT INTO [OrderStatuses] ([Id], [Name])
VALUES (2, 'Shipped');
GO

INSERT INTO [__EFMigrationsHistory]
  ([MigrationId], [ProductVersion])
VALUES
  ('20240120000000_InitialCreate', '6.0.0');
GO

COMMIT;
GO

// Terminal: Idempotent script (safe for rerun)
dotnet ef migrations script --idempotent --output idempotent.sql

// Generated idempotent SQL includes IF NOT EXISTS:
IF NOT EXISTS(
  SELECT * FROM [__EFMigrationsHistory]
  WHERE [MigrationId] = '20240120000000_InitialCreate')
BEGIN
  CREATE TABLE [Customers] (
    [Id] int NOT NULL IDENTITY,
    [Name] nvarchar(max) NOT NULL
  );
  GO

  INSERT INTO [__EFMigrationsHistory]
    ([MigrationId], [ProductVersion])
  VALUES
    ('20240120000000_InitialCreate', '6.0.0');
  GO
END

// Idempotent scripts allow safe reruns:
// sqlcmd -S server -d database -i idempotent.sql
// Run multiple times: same result, no errors

// Terminal: Generate specific migration range
dotnet ef migrations script InitialCreate AddPhoneNumber \
  --output upgrade_to_phone.sql

// Generates SQL from InitialCreate to AddPhoneNumber

// Terminal: Generate for rollback
dotnet ef migrations script AddPhoneNumber InitialCreate \
  --output rollback_to_initial.sql

// Generates DOWN() SQL to revert AddPhoneNumber

// Usage in CI/CD pipeline
name: Generate Migration Script
on:
  pull_request:

jobs:
  script:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - uses: actions/setup-dotnet@v1

      - name: Generate migration script
        run: |
          dotnet ef migrations script \
            --idempotent \
            --output migrations.sql

      - name: Comment script in PR
        uses: actions/github-script@v6
        with:
          script: |
            const fs = require('fs');
            const script = fs.readFileSync('migrations.sql');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: '```sql\n' + script + '\n```'
            });

// This allows reviewers to see SQL changes
// before approving the PR
Script Generation Enables SQL Review and Auditing

For regulated environments, generating SQL scripts is essential. You can review the exact SQL that will be applied, get DBA approval, and archive it for audit trails. Idempotent scripts are safe for automation: they won't fail if run multiple times.

A deployment best practice: generate the script in your CI/CD pipeline, upload it as an artifact, and include it in deployment documentation. This creates a record of exactly what was deployed and when. If something goes wrong, you have the SQL to investigate and potentially fix manually.

Key Takeaways

  • Migrations are version-controlled, reversible schema changes generated from model modifications
  • Workflow: Change model, add migration, review, update database
  • Each migration maps to SQL DDL: CreateTable→CREATE TABLE, AddColumn→ALTER TABLE ADD, etc.
  • HasData embeds seed data in migrations, tracked and reproducible across databases
  • Every migration has Up() (apply) and Down() (revert) methods for full reversibility
  • Custom SQL via migrationBuilder.Sql() allows data migrations, stored procedures, and complex transformations
  • Production deployments use bundles (self-contained executable) or idempotent scripts (pure SQL with IF NOT EXISTS)
  • Blue-green deployment strategies test migrations on green database before switching production traffic
