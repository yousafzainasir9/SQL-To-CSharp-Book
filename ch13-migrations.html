Chapter 13

Migrations

A SQL developer writes ALTER TABLE statements to change the schema. A C# developer adds a property to a class and runs a migration. Both change the database structure — one manually, one with version control. Migrations are how Entity Framework tracks and applies schema changes over time.

01

Requirement: "Add a Phone Number Field"

Business Requirement
The customer support team needs to store phone numbers. Add a PhoneNumber column to the Customers table.
SQL Developer Thinks Simple. I’ll run ALTER TABLE Customers ADD PhoneNumber NVARCHAR(20) NULL. The schema changes immediately. If I need to undo it, I write another ALTER TABLE to DROP the column. Manual process, manual record.
C# Developer Thinks I add a string? property to the Customer class. Then I run dotnet ef migrations add AddPhone. Entity Framework generates a migration file that tracks this change. Running dotnet ef database update applies it to the database.
SQL — ALTER TABLE
ALTER TABLE Customers
ADD PhoneNumber NVARCHAR(20) NULL;
C# — Add Property
public class Customer
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string? PhoneNumber { get; set; }
}
C# — CLI Command
dotnet ef migrations add AddPhoneNumberToCustomers
dotnet ef database update
Generated Migration — Up() Method
public partial class AddPhoneNumberToCustomers : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.AddColumn<string>(
            name: "PhoneNumber",
            table: "Customers",
            type: "nvarchar(max)",
            nullable: true);
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropColumn(
            name: "PhoneNumber",
            table: "Customers");
    }
}
02

Requirement: "Deploy Schema Changes to Production"

Business Requirement
New columns are ready. They need to go to staging, then production. We must be able to deploy safely and know which migrations have been applied.
SQL Developer Thinks I run SQL scripts manually or use a deployment tool. I have to track what’s been applied myself. Scripts are version controlled, but I have to manage them. For prod, I might use a tool like Flyway or hand off to DevOps.
C# Developer Thinks I can run dotnet ef database update in a CI/CD pipeline. Or I call db.Database.MigrateAsync() at app startup in code. Entity Framework tracks applied migrations in the __EFMigrationsHistory table automatically.
C# — CLI Deploy
# In dev
dotnet ef database update

# In CI/CD (bundle migrations)
dotnet ef migrations bundle --self-contained

# Offline bundle for prod
./efbundle.exe

# Or from code
await db.Database.MigrateAsync();
Program.cs — Run at Startup
var app = builder.Build();

if (app.Environment.IsProduction())
{
    // Auto-apply migrations on app start
    using (var scope = app.Services
        .CreateScope())
    {
        var db = scope.ServiceProvider
            .GetRequiredService<AppDbContext>();
        await db.Database.MigrateAsync();
    }
}

app.Run();
__EFMigrationsHistory Table — Entity Framework’s Record
SELECT MigrationId, ProductVersion
FROM __EFMigrationsHistory
ORDER BY MigrationId DESC;

-- Output:
-- MigrationId                               | ProductVersion
-- 20240401150000_AddPhoneNumberToCustomers | 8.0.0
-- 20240331120000_CreateCustomersTable      | 8.0.0
03

Requirement: "Roll Back a Bad Migration"

Business Requirement
The last migration deleted data by mistake or caused a constraint conflict. Undo it. The migration before that one worked fine.
SQL Developer Thinks I write a new ALTER TABLE script to undo the change. Or I restore a backup. It’s manual and error-prone. If the original script dropped a column, I have to write the reverse by hand.
C# Developer Thinks I run dotnet ef database update PreviousMigrationName. Entity Framework executes the Down() method of the current migration, automatically rolling back the change. The migration code has both Up() and Down() defined.
C# — Rollback Command
# List all migrations
dotnet ef migrations list

# Roll back to a previous migration
dotnet ef database update 20240331120000_CreateCustomersTable

# Or remove the last migration entirely
dotnet ef migrations remove
Migration Down() — Automatic Reversal
public partial class AddPhoneNumberToCustomers
    : Migration
{
    protected override void Up(MigrationBuilder m)
    {
        // Applies the change
        m.AddColumn<string>(
            name: "PhoneNumber",
            table: "Customers",
            type: "nvarchar(max)",
            nullable: true);
    }

    protected override void Down(MigrationBuilder m)
    {
        // Reverts the change
        m.DropColumn(
            name: "PhoneNumber",
            table: "Customers");
    }
}
04

Requirement: "Run Custom SQL During Migration"

Business Requirement
We’re adding a new Status column. During migration, set all existing customers to status “Active”. Also create an index on this column for fast filtering.
SQL Developer Thinks I write a SQL script: ALTER TABLE ADD column, UPDATE to seed data, CREATE INDEX. All in one script. Straightforward.
C# Developer Thinks I add the property to the class. In the migration, I use migrationBuilder.Sql() to run raw SQL alongside the EF-generated commands. I can add column, seed data, and create indexes all in the Up() method.
Migration — AddColumn + SQL + Index
public partial class AddStatusToCustomers : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        // Add the column
        migrationBuilder.AddColumn<string>(
            name: "Status",
            table: "Customers",
            type: "nvarchar(20)",
            nullable: false,
            defaultValue: "Active");

        // Run raw SQL to update existing rows
        migrationBuilder.Sql(
            @"UPDATE Customers
              SET Status = 'Active'
              WHERE Status IS NULL");

        // Create an index for fast filtering
        migrationBuilder.CreateIndex(
            name: "IX_Customers_Status",
            table: "Customers",
            column: "Status");
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropIndex(
            name: "IX_Customers_Status",
            table: "Customers");

        migrationBuilder.DropColumn(
            name: "Status",
            table: "Customers");
    }
}
Seeding Data — More Complex Example
protected override void Up(MigrationBuilder migrationBuilder)
{
    migrationBuilder.AddColumn<int>(
        name: "CategoryId",
        table: "Products",
        nullable: false,
        defaultValue: 0);

    // Seed the Categories table
    migrationBuilder.Sql(
        @"INSERT INTO Categories (Name, IsActive)
          VALUES ('Electronics', 1),
                 ('Clothing', 1),
                 ('Books', 1)");

    // Assign default category
    migrationBuilder.Sql(
        @"UPDATE Products
          SET CategoryId = (
              SELECT Id FROM Categories
              WHERE Name = 'Electronics'
          )");

    // Create foreign key
    migrationBuilder.CreateIndex(
        name: "IX_Products_CategoryId",
        table: "Products",
        column: "CategoryId");
}
05

Requirement: "Clean Up 50 Old Migrations"

Business Requirement
The project has accumulated 50 migrations over 2 years. It’s hard to manage and slow to replay. Consolidate them into a fresh baseline.
SQL Developer Thinks I would just keep all the scripts. There’s nothing to clean up — they’re just files. But tracking them all is messy.
C# Developer Thinks Entity Framework lets me squash migrations. I can delete old migration files, combine them into one, and reset the baseline. This keeps history clean and deployment faster. The database sees the same end result.
Strategy 1: Squash Migrations
# Remove old migration files from the Migrations folder
# Keep only the most recent one

# Generate a new "baseline" migration that represents the current state
dotnet ef migrations add InitialCreateFresh

# Or, remove and re-scaffold from the current model
dotnet ef migrations remove
dotnet ef migrations add InitialCreate
Strategy 2: Reset Baseline for Existing Database
// When database already exists in production
protected override void Up(MigrationBuilder migrationBuilder)
{
    // This migration does nothing — it’s a snapshot
    // The database already has all the old schema
}

protected override void Down(MigrationBuilder migrationBuilder)
{
    throw new NotSupportedException(
        "This baseline migration cannot be rolled back.");
}
Strategy 3: Snapshot + Reset
# Export all current tables/indexes as a fresh migration
dotnet ef migrations add ResetBaseline

# In production, mark as applied without running
dotnet ef database update --connection "YourConnectionString" --no-build

# Going forward, all new migrations start from this baseline
06

Requirement: "Two Developers Changed the Same Model"

Business Requirement
Alice added a Phone column in branch A. Bob added an Email2 column in branch B. Both branches create a migration with the same timestamp. Merging causes a conflict.
SQL Developer Thinks Two SQL scripts, same namespace? Doesn’t apply here. But I’d merge them manually and test. Conflicts are a natural part of source control.
C# Developer Thinks Entity Framework migrations are files too — they can have merge conflicts. Both developers generated migrations with similar names. I need to rename one, regenerate the snapshot, and make sure they apply in sequence.
Conflict Resolution Steps
# Branch A: 20240401_AddPhone.cs
# Branch B: 20240401_AddEmail2.cs

# After merging, conflicts exist in:
# - Migrations/20240401_AddPhone.cs
# - Migrations/20240401_AddEmail2.cs
# - Migrations/AppDbContextModelSnapshot.cs

# Step 1: Manually resolve the migration file conflicts
#         (keep both Up() and Down() methods)

# Step 2: Rename one migration to apply after the other
#         20240401_AddPhone.cs → 20240401_AddPhoneNumber.cs
#         20240402_AddEmail2.cs

# Step 3: Regenerate the snapshot to include both changes
dotnet ef migrations add ResolveConflict

# Step 4: Delete the helper migration (we created it just to sync)
dotnet ef migrations remove

# Step 5: Apply in order
dotnet ef database update
Merged Migration File — Both Changes Combined
public partial class AddPhoneAndEmail : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        // From Alice’s change
        migrationBuilder.AddColumn<string>(
            name: "PhoneNumber",
            table: "Customers",
            type: "nvarchar(max)",
            nullable: true);

        // From Bob’s change
        migrationBuilder.AddColumn<string>(
            name: "Email2",
            table: "Customers",
            type: "nvarchar(255)",
            nullable: true);
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropColumn(
            name: "PhoneNumber",
            table: "Customers");

        migrationBuilder.DropColumn(
            name: "Email2",
            table: "Customers");
    }
}
Regenerate ModelSnapshot After Merge
# The snapshot becomes out of sync after merging
# Regenerate it to match the combined entity model
dotnet ef migrations script

# Or, create a dummy migration to sync
dotnet ef migrations add _SyncSnapshot
dotnet ef migrations remove

# Now the snapshot reflects both Alice’s and Bob’s changes
