Chapter 25

CQRS

Separate Command (write) and Query (read) models for scalability and clarity.

01

Command Query Responsibility Segregation

CQRS separates read operations from write operations. One model handles inserts/updates/deletes; another handles queries and reporting.

Unified Model (CRUD)
// Same entity for reads and writes
public interface IRepository<T>
{
    Task<T> GetByIdAsync(int id);
    Task<IEnumerable<T>> GetAllAsync();
    Task AddAsync(T entity);
    Task UpdateAsync(T entity);
}

var customer = await repo.GetByIdAsync(1);
await repo.UpdateAsync(customer);

// Problem: Read model has all data;
// write model needs validation
CQRS Pattern
// Separate command and query interfaces
public interface ICommandHandler<TCommand>
{
    Task ExecuteAsync(TCommand command);
}

public interface IQueryHandler<TQuery, TResult>
{
    Task<TResult> HandleAsync(TQuery query);
}

// Lean write model
var cmd = new UpdateCustomerCommand { Id = 1, Name = "Bob" };
await handler.ExecuteAsync(cmd);

// Rich read model
var query = new GetCustomerDetailsQuery { Id = 1 };
var details = await handler.HandleAsync(query);
Key Difference:

Write model validates and persists. Read model optimizes for queries (can include denormalized views).

02

Commands (Writes)

Commands are explicit operations that mutate state. They return nothing or minimal acknowledgment.

Command Pattern
// Define commands as classes
public class CreateOrderCommand
{
    public int CustomerId { get; set; }
    public List<OrderItemRequest> Items { get; set; }
}

public class UpdateOrderStatusCommand
{
    public int OrderId { get; set; }
    public OrderStatus Status { get; set; }
}

// Command handler
public class CreateOrderCommandHandler
    : ICommandHandler<CreateOrderCommand>
{
    private readonly IUnitOfWork _uow;

    public CreateOrderCommandHandler(IUnitOfWork uow) => _uow = uow;

    public async Task ExecuteAsync(CreateOrderCommand cmd, CancellationToken cancellationToken = default)
    {
        // Validate
        var customer = await _uow.Customers.GetByIdAsync(cmd.CustomerId, cancellationToken);
        if (customer is null)
            throw new InvalidOperationException("Customer not found");

        // Create and persist
        var order = new Order { CustomerId = cmd.CustomerId };
        foreach (var item in cmd.Items)
        {
            order.Items.Add(new OrderItem
            {
                ProductId = item.ProductId,
                Quantity = item.Quantity
            });
        }

        await _uow.Orders.AddAsync(order, cancellationToken);
        await _uow.SaveAsync(cancellationToken);
    }
}

// Usage
var cmd = new CreateOrderCommand
{
    CustomerId = 5,
    Items = new() { new OrderItemRequest { ProductId = 10, Quantity = 2 } }
};
await handler.ExecuteAsync(cmd);
03

Queries (Reads)

Queries return data without side effects. They can query denormalized read models optimized for specific views.

Query Pattern
// Define queries as classes
public class GetCustomerDetailsQuery
{
    public int CustomerId { get; set; }
}

public class GetCustomerDetailsResult
{
    public int Id { get; set; }
    public string Name { get; set; }
    public string Email { get; set; }
    public int OrderCount { get; set; }
    public decimal TotalSpent { get; set; }
}

// Query handler
public class GetCustomerDetailsQueryHandler
    : IQueryHandler<GetCustomerDetailsQuery, GetCustomerDetailsResult>
{
    private readonly DbContext _db;

    public GetCustomerDetailsQueryHandler(DbContext db) => _db = db;

    public async Task<GetCustomerDetailsResult> HandleAsync(
        GetCustomerDetailsQuery query, CancellationToken cancellationToken = default)
    {
        return await _db.Customers
            .AsNoTracking()
            .Where(c => c.Id == query.CustomerId)
            .Select(c => new GetCustomerDetailsResult
            {
                Id = c.Id,
                Name = c.Name,
                Email = c.Email,
                OrderCount = c.Orders.Count(),
                TotalSpent = c.Orders.SelectMany(o => o.Items)
                    .Sum(i => i.Price * i.Quantity)
            })
            .FirstOrDefaultAsync(cancellationToken);
    }
}

// Usage
var query = new GetCustomerDetailsQuery { CustomerId = 5 };
var result = await handler.HandleAsync(query);
Console.WriteLine($"{result.Name}: {result.OrderCount} orders, ${result.TotalSpent}");
Optimization:

Queries can hit a read-optimized database or cache layer completely separate from writes.

04

Mediator Pattern (Bus)

A mediator routes commands and queries to their handlers automatically, decoupling callers from handler implementations.

Simple Mediator
public interface IMediator
{
    Task<TResult> HandleAsync<TResult>(IQuery<TResult> query);
    Task ExecuteAsync<TCommand>(TCommand command);
}

public class Mediator : IMediator
{
    private readonly IServiceProvider _services;

    public Mediator(IServiceProvider services) => _services = services;

    public async Task<TResult> HandleAsync<TResult>(IQuery<TResult> query, CancellationToken cancellationToken = default)
    {
        var handlerType = typeof(IQueryHandler<,>)
            .MakeGenericType(query.GetType(), typeof(TResult));
        var handler = _services.GetService(handlerType);

        var method = handlerType.GetMethod("HandleAsync");
        return await (Task<TResult>)method.Invoke(handler, new object[] { query, cancellationToken });
    }

    public async Task ExecuteAsync<TCommand>(TCommand command, CancellationToken cancellationToken = default)
    {
        var handlerType = typeof(ICommandHandler<>)
            .MakeGenericType(command.GetType());
        var handler = _services.GetService(handlerType);

        var method = handlerType.GetMethod("ExecuteAsync");
        await (Task)method.Invoke(handler, new object[] { command, cancellationToken });
    }
}

// In controller
public class OrderController
{
    private readonly IMediator _mediator;

    [HttpPost]
    public async Task<IActionResult> CreateOrder(CreateOrderCommand cmd, CancellationToken cancellationToken)
    {
        await _mediator.ExecuteAsync(cmd, cancellationToken);
        return Ok();
    }

    [HttpGet("{id}")]
    public async Task<IActionResult> GetOrder(int id, CancellationToken cancellationToken)
    {
        var result = await _mediator.HandleAsync(new GetOrderDetailsQuery { Id = id }, cancellationToken);
        return Ok(result);
    }
}
05

CQRS with Event Sourcing

Commands generate events. Event store stores immutable events; read model builds from them.

Event-Sourced CQRS
// Events
public abstract class Event
{
    public int AggregateId { get; set; }
    public DateTime Timestamp { get; set; }
}

public class OrderCreatedEvent : Event
{
    public int CustomerId { get; set; }
    public decimal Total { get; set; }
}

public class OrderShippedEvent : Event
{
    public DateTime ShippedDate { get; set; }
}

// Command handler persists events
public class CreateOrderCommandHandler
{
    private readonly IEventStore _eventStore;

    public async Task ExecuteAsync(CreateOrderCommand cmd, CancellationToken cancellationToken = default)
    {
        // Validate and create event
        var @event = new OrderCreatedEvent
        {
            AggregateId = cmd.OrderId,
            CustomerId = cmd.CustomerId,
            Total = cmd.Total,
            Timestamp = DateTime.Now
        };

        // Persist to event store
        await _eventStore.AppendAsync(@event, cancellationToken);
    }
}

// Read model built from events
public class OrderReadModel
{
    private readonly IEventStore _eventStore;

    public async Task<OrderDetailsDto> GetOrderAsync(int orderId, CancellationToken cancellationToken = default)
    {
        var events = await _eventStore.GetEventsAsync(orderId, cancellationToken);

        var order = new OrderDetailsDto { Id = orderId };
        foreach (var e in events)
        {
            if (e is OrderCreatedEvent created)
                order.CustomerId = created.CustomerId;
            else if (e is OrderShippedEvent shipped)
                order.ShippedDate = shipped.ShippedDate;
        }

        return order;
    }
}
Benefit:

Event sourcing provides complete audit trail. Read models are eventually consistent but highly optimized.

Key Takeaways

  • CQRS separates command (write) and query (read) models.
  • Commands are explicit, validated operations; queries are read-only projections.
  • Mediator pattern routes commands and queries to appropriate handlers.
  • Read models can be denormalized or cached for performance.
  • Event sourcing pairs well with CQRS for audit trails and eventual consistency.
  • Use CQRS when reads and writes have different scaling or optimization needs.
