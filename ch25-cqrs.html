01

The CQRS Principle: Commands vs Queries

CQRS stands for Command Query Responsibility Segregation. The principle is deceptively simple: separate the code path that writes data (commands) from the code path that reads data (queries). They have fundamentally different optimization needs, and conflating them forces compromise.

  • Commands (Writes): Change state. Need validation, business rule enforcement, referential integrity, change tracking, proper error handling. EF Core with its entity tracking and relationship management is ideal.
  • Queries (Reads): Don't change state. Need speed, flexible shape, denormalization, minimal data transfer. Dapper with raw SQL is ideal.

CQRS recognizes that your best read architecture might not be your best write architecture. In SQL, this is obvious: you normalize tables for inserts and updates, but create denormalized views for reports and reads. CQRS applies that same wisdom to C# code.

SQL: Database-Level CQRS
-- NORMALIZED WRITE TABLES
-- Minimize data redundancy
CREATE TABLE Orders (
    Id INT PRIMARY KEY,
    CustomerId INT,
    OrderDate DATETIME,
    FOREIGN KEY (CustomerId)
        REFERENCES Customers(Id)
);

CREATE TABLE OrderItems (
    Id INT PRIMARY KEY,
    OrderId INT,
    ProductId INT,
    Quantity INT
);

-- DENORMALIZED READ VIEW
-- Pre-joined, pre-aggregated
CREATE VIEW OrderSummaryView AS
SELECT
    o.Id,
    o.CustomerId,
    c.Name AS CustomerName,
    c.Email,
    o.OrderDate,
    COUNT(oi.Id) AS ItemCount,
    SUM(oi.Quantity) AS TotalQty
FROM Orders o
JOIN Customers c ON
    c.Id = o.CustomerId
LEFT JOIN OrderItems oi ON
    oi.OrderId = o.Id
GROUP BY o.Id, o.CustomerId,
         c.Name, c.Email, o.OrderDate;

CQRS is Not a Database Pattern, It's an Architecture Pattern
CQRS doesn't require separate databases. You can use the same SQL Server instance with normalized tables (for writes via EF) and denormalized views (for reads via Dapper). The separation is in your code architecture, not in your database topology.
02

Read Models vs Write Models

In traditional layered architecture, a single entity class (like Order) is used for both reading and writing. This is convenient but problematic: a write operation needs validation, relationships, and business logic; a read operation needs denormalization, pre-aggregation, and minimal data transfer.

CQRS separates these concerns with two distinct models:

  • Write Model (Command Model): Normalized entities with validation, constraints, and relationships. Maps to normalized tables. Used by EF Core.
  • Read Model (Query Model): Denormalized DTOs with only needed fields, pre-joined and pre-aggregated. Maps to views or result sets. Used by Dapper.
Write Model (EF Entity)
// Normalized for WRITES
public class Order
{
    public int Id { get; set; }
    public int CustomerId { get; set; }
    public DateTime
        OrderDate { get; set; }

    // Navigation properties
    // for relationships
    public Customer Customer
        { get; set; }
    public List<OrderItem>
        Items { get; set; }

    // Methods for business logic
    public void AddItem(
        OrderItem item)
    {
        if (item.Quantity <= 0)
            throw new
                ArgumentException();
        Items.Add(item);
    }
}
Read Model (DTO)
// Denormalized for READS
// Flat structure, no joins
public class OrderSummaryDto
{
    public int Id { get; set; }
    public int CustomerId { get; set; }

    // Denormalized fields
    public string CustomerName
        { get; set; }
    public string CustomerEmail
        { get; set; }

    // Pre-aggregated values
    public int ItemCount
        { get; set; }
    public int TotalQuantity
        { get; set; }
    public DateTime
        OrderDate { get; set; }

    // No navigation properties
    // No business logic
    // Just data, optimized for reads
}
Different Models, Different Concerns
The write model (your EF entity) enforces constraints and business rules. The read model (a DTO) is shaped for the UI or API response. They're not fighting for optimization: the write model optimizes for correctness, the read model optimizes for speed. Both win.

CQRS with Separate DbContexts

The simplest CQRS implementation uses two DbContext classes: one for writes (EF Core with change tracking), one for reads (AsNoTracking queries). Both point to the same database, but they have different optimization strategies.

Write DbContext (EF Core)
// WriteContext: With change tracking
// For commands that modify state
public class WriteDbContext
    : DbContext
{
    public DbSet<Order>
        Orders { get; set; }
    public DbSet<Customer>
        Customers { get; set; }

    protected override void
        OnConfiguring(
            DbContextOptionsBuilder
            builder)
    {
        builder.UseSqlServer(
            "Server=.;...");
    }

    protected override void
        OnModelCreating(
            ModelBuilder builder)
    {
        // Relationships, constraints
        builder.Entity<Order>()
            .HasMany(o =>
                o.Items)
            .WithOne(oi =>
                oi.Order);
    }
}

Write Command Handler
// IRequestHandler handles writes
public class
    CreateOrderCommandHandler
    : IRequestHandler<
        CreateOrderCommand,
        CreateOrderResult>
{
    private readonly
        WriteDbContext _context;

    public CreateOrderCommandHandler(
        WriteDbContext context)
    {
        _context = context;
    }

    public async Task<
        CreateOrderResult>
        Handle(
            CreateOrderCommand cmd,
            CancellationToken ct)
    {
        var order = new
            Order
        {
            CustomerId =
                cmd.CustomerId,
            OrderDate = DateTime
                .UtcNow
        };

        foreach (var item
            in cmd.Items)
        {
            order.AddItem(
                new OrderItem {
                    ProductId =
                        item.ProductId,
                    Quantity =
                        item.Quantity
                });
        }

        _context.Orders.Add(order);
        await
            _context.SaveChangesAsync(

        {
            OrderId = order.Id
        };
    }
}
Read Query Handler 
// IRequestHandler handles reads
public class
    GetOrderSummaryQueryHandler
    : IRequestHandler<
        GetOrderSummaryQuery,
        OrderSummaryDto>
{
    private readonly
        ReadDbContext _context;

    public
        GetOrderSummaryQueryHandler(
            ReadDbContext context)
    {
        _context = context;
    }

    public async Task<
        OrderSummaryDto>
        Handle(
            GetOrderSummaryQuery q,
            CancellationToken ct)
    {
        return await
            _context.Orders
            .AsNoTracking()
            .Where(o =>
                o.Id == q.OrderId)
            .Select(o => new
                OrderSummaryDto
            {
                Id = o.Id,
                CustomerId =
                    o.CustomerId,
                CustomerName =
                    o.Customer.Name,
                CustomerEmail =
                    o.Customer.Email,
                ItemCount =
                    o.Items.Count(),
                TotalQuantity =
                    o.Items
                    .Sum(i =>
                        i.Quantity),
                OrderDate = o.OrderDate
            })
            .FirstOrDefaultAsync(
                ct);
    }
}
Both Contexts Share the Same Schema
The WriteDbContext and ReadDbContext point to the same database and tables. The difference is how they use them: WriteDbContext enables change tracking and enforces relationships; ReadDbContext disables tracking and optimizes for speed. You only need two contexts if you want to force different defaults and prevent accidental mutation on reads.
04

CQRS with Dapper for Reads

For maximum read performance, replace EF entirely on the read side with Dapper. Write with EF Core (normalized, validated), read with Dapper (raw SQL, denormalized views). This is the classic CQRS pattern: two tools, each optimized for its job.

SQL: Denormalized View for Reads
CREATE VIEW OrderSummaryView AS
SELECT
    o.Id,
    o.CustomerId,
    c.Name AS CustomerName,
    c.Email AS CustomerEmail,
    o.OrderDate,
    COUNT(oi.Id) AS
        ItemCount,
    SUM(oi.Quantity) AS
        TotalQuantity,
    SUM(oi.Quantity * p.Price)
        AS TotalPrice
FROM Orders o
JOIN Customers c
    ON c.Id = o.CustomerId
LEFT JOIN OrderItems oi
    ON oi.OrderId = o.Id
LEFT JOIN Products p
    ON p.Id = oi.ProductId
GROUP BY o.Id, o.CustomerId,
         c.Name, c.Email,
         o.OrderDate;
C#: Dapper Query Handler 
// Using Dapper for reads
public class
    GetOrderSummaryQueryHandler
    : IRequestHandler<
        GetOrderSummaryQuery,
        OrderSummaryDto>
{
    private readonly
        IDbConnection _db;

    public
        GetOrderSummaryQueryHandler(
            IDbConnection db)
    {
        _db = db;
    }

    public async Task<
        OrderSummaryDto>
        Handle(
            GetOrderSummaryQuery q,
            CancellationToken ct)
    {
        var sql = @"
            SELECT Id, CustomerId,
                   CustomerName,
                   CustomerEmail,
                   OrderDate,
                   ItemCount,
                   TotalQuantity,
                   TotalPrice
            FROM OrderSummaryView
            WHERE Id = @OrderId";

        return await _db
            .QuerySingleOrDefaultAsync<
                OrderSummaryDto>(
                sql,
                new { q.OrderId });
    }
}
Query Syntax: LINQ 
// LINQ query syntax
// (if using EF for reads)
public async Task<
    List<OrderSummaryDto>>
    GetRecentOrdersSyntax()
{
    var result = await (
        from o in
            _context.Orders
        where o.OrderDate >=
            DateTime.UtcNow
            .AddDays(-7)
        orderby o.OrderDate
            descending
        select new
            OrderSummaryDto
        {
            Id = o.Id,
            CustomerName =
                o.Customer.Name,
            ItemCount =
                o.Items.Count()
        })

        .ToListAsync();

    List<OrderSummaryDto>>
    GetRecentOrdersMethod()
{

        _context.Orders
        .AsNoTracking()
        .Where(o =>
            o.OrderDate >=
            DateTime.UtcNow
            .AddDays(-7))
        .OrderByDescending(o =>
            o.OrderDate)
        .Select(o => new
            OrderSummaryDto
        {
            Id = o.Id,
            CustomerName =
                o.Customer.Name,
            ItemCount =
                o.Items.Count()

Dapper: When You Control the SQL
Dapper is not an ORM. It's a lightweight mapper that executes raw SQL and maps results to objects. There's no query translation, no lazy loading, no surprises. For read-heavy workloads with complex denormalized queries, Dapper is faster and more predictable than EF Core. You write the SQL you know is efficient.

MediatR Integration: Clean Command and Query Handlers

MediatR is a mediator pattern library that elegantly bridges CQRS and your application code. Instead of calling handlers directly, you send requests to the mediator, which routes them to the appropriate handler. This decouples your application logic from infrastructure.

MediatR Request and Handler Interface

// Command: A request with a result
public class CreateOrderCommand
    : IRequest<int>
{
    public int CustomerId { get; set; }
    public List<OrderItemDto>
        Items { get; set; }
}

// Handler: Implements IRequestHandler
public class
    CreateOrderCommandHandler
    : IRequestHandler<
        CreateOrderCommand, int>
{

        WriteDbContext _context;

    public
        CreateOrderCommandHandler(
            WriteDbContext context)
    {
        _context = context;
    }

        Handle(
            CreateOrderCommand cmd,
            CancellationToken ct)
    {
        // Build entity, validate,
        // save
        var order = new
            Order
        {
            CustomerId = cmd.CustomerId
        };

        foreach (var item
            in cmd.Items)
        {
            order.AddItem(
                new OrderItem
                {
                    ProductId =
                        item.ProductId,
                    Quantity =
                        item.Quantity
                });
        }

        _context.Orders.Add(order);
        await
            _context

    }
}

Event Sourcing and Eventual Consistency

Advanced CQRS often includes event sourcing: instead of storing current state, store every change as an immutable event. From these events, you can rebuild the current state (write model) and project any number of denormalized read models. This provides an audit trail and enables complex queries.

SQL: Event Store (Append-Only)
-- Immutable event store
-- Events never updated/deleted
CREATE TABLE Events (
    EventId BIGINT
        PRIMARY KEY
        IDENTITY,
    AggregateId INT NOT NULL,
    AggregateType
        VARCHAR(100),
    EventType VARCHAR(100),
    EventData NVARCHAR(MAX),
    -- JSON payload
    Timestamp DATETIME
        DEFAULT
        GETUTCDATE(),
    INDEX IX_AggregateId
        (AggregateId)
);

-- Sample events for Order #123:
-- 1. OrderCreatedEvent
{
--   "customerId": 456,
--   "orderDate": "2024-01-15"
-- }

-- 2. OrderItemAddedEvent
{
--   "productId": 789,
--   "quantity": 2
-- }

-- 3. OrderConfirmedEvent
{
--   "confirmationTime": "2024-01-15T10:30:00"
-- }
C#: Event Sourcing Handler 
// Domain event (immutable)
public abstract class
    DomainEvent
{
    public Guid AggregateId
        { get; set; }
    public DateTime
        Timestamp { get; set; }
}

public class
    OrderCreatedEvent
    : DomainEvent
{
    public int CustomerId
        { get; set; }
    public DateTime
        OrderDate { get; set; }
}

// Rebuild aggregate from
// event stream
public class Order
{
    public int Id { get; set; }
    public int CustomerId
        { get; set; }
    public List<
        DomainEvent> Events
        { get; set; }

    public static Order
        LoadFromEvents(
            List<DomainEvent>
            events)
    {
        var order = new
            Order();

        foreach (var @event
            in events)
        {
            if (@event is
                OrderCreatedEvent
                created)
            {
                order.Id =
                    created.AggregateId
                    .GetHashCode();
                order.CustomerId =
                    created.CustomerId;
            }
        }

        order.Events = events;
        return order;
    }
}
Event Sourcing is Complex and Not Always Necessary
Event sourcing is powerful: you have a complete audit trail, you can replay events to rebuild state, you can project multiple read models from the same events. But it's also complex: you need to handle event versioning, deal with eventual consistency, and manage event replay. Start with simple CQRS (EF writes, Dapper reads) before moving to event sourcing.

Command Validation and Handlers

Commands must be validated before execution. Use FluentValidation or MediatR behaviors to validate commands before they reach the handler. This keeps your handlers clean and your validation testable.

FluentValidation for Commands
// Install: dotnet add package
//   FluentValidation.DependencyInjectionExtensions

// Validator for CreateOrderCommand
public class
    CreateOrderCommandValidator
    : AbstractValidator<
        CreateOrderCommand>
{
    private readonly
        ReadDbContext _context;

    public
        CreateOrderCommandValidator(
            ReadDbContext context)
    {
        _context = context;

        // Customer must exist
        RuleFor(cmd =>
            cmd.CustomerId)
            .MustAsync(
                async (customerId,
                ct) =>
                {
                    return await
                        _context.Customers
                        .AsNoTracking()
                        .AnyAsync(c =>
                            c.Id ==
                            customerId,
                            ct);
                })
            .WithMessage(
                "Customer not found");

        // Items required
        RuleFor(cmd =>
            cmd.Items)
            .NotEmpty()
            .WithMessage(
                "Order must have items");

        // Each item valid
        RuleForEach(cmd =>
            cmd.Items)
            .SetValidator(
                new
                OrderItemValidator());
    }
}

public class
    OrderItemValidator
    : AbstractValidator<
        OrderItemDto>
{
    public
        OrderItemValidator()
    {
        RuleFor(item =>
            item.Quantity)
            .GreaterThan(0)
            .WithMessage(
                
                "Quantity must be > 0");
    }
}
MediatR Behavior: Auto-Validate 
// Validation behavior (runs
// before handler)
public class
    ValidationBehavior<TRequest,
        TResponse>
    : IPipelineBehavior<
        TRequest,
        TResponse>
    where TRequest
        : IRequest<
            TResponse>
{
    private readonly
        IValidator<TRequest>
            _validator;

            IValidator<

            validator)

        _validator = validator;
    }

        TResponse>
        Handle(
            TRequest request,
            RequestHandlerDelegate<
                TResponse>
            next,
            CancellationToken ct)

            _validator
            .ValidateAsync(
                request, ct);

        if (!result.IsValid)
        {
            throw new

                    result.Errors);
        }

        // Valid, call handler
        return await next();
    }
}

Query Optimization with Projections

On the read side, use projections extensively. Instead of loading full entities and filtering in memory, project only needed fields directly in the database query. This reduces network traffic and memory usage dramatically.

Inefficient: Load Full Entities 
// SLOW: Loads full Orders
// and Customers
public async Task<
    List<OrderDto>>
    GetOrdersInefficient()
{
    var orders = await
        _context.Orders
        .Include(o =>
            o.Customer)
        .ToListAsync();

    // Filter in memory
    return orders
        .Where(o =>
            o.Customer.IsActive)
        .OrderByDescending(o =>
            o.OrderDate)
        .Take(10)
        .Select(o => new
            OrderDto
        {
            Id = o.Id,
            CustomerName =
                o.Customer.Name
        })
        .ToList();
}

// Loads ALL orders,
// all customers,
// filters in memory
Efficient: Project in Database 
// FAST: Database does
// filtering, sorting, limit
public async Task<
    List<OrderDto>>
    GetOrdersEfficient()
{
    return await
        _context.Orders
        .AsNoTracking()
        .Where(o =>
            o.Customer.IsActive)
        .OrderByDescending(o =>
            o.OrderDate)
        .Take(10)
        .Select(o => new
            OrderDto
        {
            Id = o.Id,
            CustomerName =
                o.Customer.Name
        })
        .ToListAsync();
}

// Database returns
// only 10 rows with
// 2 columns each
Generated SQL: Inefficient 
-- Loads all orders and customers
SELECT o.*, c.*
FROM Orders o
JOIN Customers c
ON c.Id = o.CustomerId;

-- Then in-memory filtering
-- for IsActive, ordering, limit
Generated SQL: Efficient 
-- Filtering, ordering, limit
-- all in SQL
SELECT TOP 10
    o.Id, c.Name
FROM Orders o
JOIN Customers c
ON c.Id = o.CustomerId
WHERE c.IsActive = 1
ORDER BY o.OrderDate DESC;

-- Returns exactly 10 rows
-- with 2 columns each
Projection First, Then Filtering
When using EF, always apply Where and OrderBy before Select. This way, EF translates the entire operation to SQL. If you Select first and then Where, EF loads all data into memory and filters there. This is the opposite of what you want.
09

When NOT to Use CQRS

CQRS is a powerful pattern, but it adds complexity. Two DbContexts, separate command and query handlers, more code to maintain. Sometimes the simplest approach wins.

  • Early-stage applications: You don't know your read patterns yet. Start with a simple ORM, optimize later if needed.
  • Simple CRUD apps: If every read is just "get by ID" and writes are simple inserts, CQRS overhead outweighs benefits.
  • Low traffic systems: Performance differences matter when you have 1000s of requests/sec. For 10 requests/sec, simpler code wins.
  • Team unfamiliarity: If your team doesn't understand CQRS, the complexity cost is high. Build expertise first.
When Simple CRUD Works 
// A simple blog app:
// - ~100 posts
// - ~1000 views/day
// - Simple read: get by ID

Start Simple, Evolve if Needed
Begin with a simple repository pattern or minimal CQRS. If performance analysis shows reads are slow, add CQRS: separate the query code, introduce Dapper, create denormalized views. If writes are slow, optimize EF. You can evolve incrementally without a complete rewrite.

public class
    SearchProductsQueryHandler
    : IRequestHandler<
        SearchProductsQuery,
        List<
            ProductListItemDto>>
{
    private readonly
        IDbConnection _db;

    public
        SearchProductsQueryHandler(
            IDbConnection db)
    {
        _db = db;
    }

    public async Task<
        List<
            ProductListItemDto>>
        Handle(
            SearchProductsQuery q,
            CancellationToken ct)
    {
        var sql = @"
            SELECT p.Id, p.Name,
                   p.Price, c.Name
                   AS CategoryName,
                   (CASE
                       WHEN p.StockQuantity
                           > 0
                       THEN 1
                       ELSE 0
                   END) AS InStock
            FROM Products p
            JOIN Categories c
                ON c.Id = p.CategoryId
            WHERE p.Name LIKE
                @Search
            ORDER BY p.Name
            OFFSET (@Page - 1) *
                @PageSize
            ROWS FETCH NEXT
                @PageSize ROWS ONLY";

        var search = $"%{q.SearchTerm}%";
        var offset =
            (q.PageNumber - 1) *
            q.PageSize;

        return (await
            _db
            .QueryAsync<
                ProductListItemDto>(
                sql,
                new
                {
                    Search = search,
                    Page = q.PageNumber,
                    PageSize = q.PageSize
                }))
            .ToList();
    }
}

Done! Your First CQRS Implementation
You now have a working CQRS system: commands route through EF Core with validation and change tracking, queries route through Dapper with raw SQL for speed. The mediator pattern keeps everything decoupled. Start here, add complexity only when profiling shows you need it.

Key Takeaways

  • CQRS separates read and write code paths, mirroring how databases use normalized tables for writes and denormalized views for reads.
  • Use EF Core for commands (validated writes with change tracking) and Dapper for queries (fast reads with raw SQL).
  • Separate write models (entities with relationships) from read models (flattened DTOs) without guilt—they solve different problems.
  • MediatR elegantly routes commands and queries to handlers, keeping your controllers thin and logic testable.
  • FluentValidation with MediatR behaviors ensures commands are validated before persistence, keeping handlers focused on business logic.
  • Always project in the database (Where → OrderBy → Select) rather than loading full entities and filtering in memory.
  • Event sourcing is powerful but complex; start with simple CQRS before graduating to event-based systems.
  • CQRS adds complexity. For simple CRUD apps or early-stage projects, simpler patterns (basic repositories) often win.
  • Evolve incrementally: start simple, profile, add CQRS where performance analysis shows reads or writes are bottlenecked.
  • The biggest win: reads and writes are now optimized independently. Your team can scale the read pipeline without touching write logic.
