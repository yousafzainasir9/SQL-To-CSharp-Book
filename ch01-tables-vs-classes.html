Tables vs Classes — The Fundamental Parallel

Every SQL table has a parallel structure in object-oriented programming. Understanding this fundamental relationship is the key to building effective data access layers. This chapter explores how relational database concepts map to C# classes, and why both perspectives matter when working with modern data access frameworks like Entity Framework Core.

01

The Relational Model vs Object-Oriented Model

To understand why tables map to classes, we need to step back and examine two fundamentally different ways of thinking about data.

In 1970, Edgar Codd published a groundbreaking paper introducing the relational model. This model treats data as mathematical relations — sets of tuples. A table is a relation, each row is a tuple, and each column is an attribute. The relational model is declarative and set-based: you describe WHAT you want, and the database figures out HOW to retrieve it. SQL queries operate on entire sets of rows at once. A SELECT statement that returns 1000 rows is conceptually the same as one that returns one row — you're describing a set.

Object-oriented programming (OOP), which emerged in the 1960s and 80s, treats data differently. In OOP, data is organized into objects that combine state (properties) and behavior (methods). A class is a blueprint; an object is an instance of that class. Thinking in OOP is imperative: you describe HOW to retrieve and manipulate data by calling methods on objects. You iterate through collections, call methods, and mutate state.

This fundamental tension is at the heart of the "impedance mismatch" problem. Relational thinking is SET-based and declarative. OOP thinking is OBJECT-based and imperative. Yet in modern applications, we must bridge both worlds: we have relational databases (SQL Server, PostgreSQL, MySQL) and object-oriented code (C#, Java, Python).

This book teaches you how to navigate both mental models effectively. You'll learn to think in sets when writing SQL queries, and in objects when writing C# code. More importantly, you'll understand when each model is appropriate and how to translate between them using modern tools like Entity Framework Core.

Key Insight: Understanding both mental models is essential to effective data access. SQL thinks in sets of rows. C# thinks in collections of objects. They solve the same problems differently. The better you understand both perspectives, the better your code will be.
02

Columns Are Properties: The Type Mapping

Every SQL column has a data type. Every C# property has a CLR (Common Language Runtime) type. The mapping between them is the foundation of all ORM (Object-Relational Mapping) work. Entity Framework Core handles most of this mapping automatically, but understanding it is crucial for diagnosing issues and making good design choices.

Let's start with a concrete example. Below, we define a Customers table in SQL, and then the corresponding Customer class in C#.

SQL
CREATE TABLE Customers (
    Id INT NOT NULL PRIMARY KEY,
    Name NVARCHAR(100) NOT NULL,
    Email NVARCHAR(255) NOT NULL,
    Age INT NULL,
    Balance DECIMAL(10, 2) NOT NULL DEFAULT 0,
    IsActive BIT NOT NULL DEFAULT 1,
    CreatedAt DATETIME2 NOT NULL DEFAULT GETDATE(),
    ProfilePhoto VARBINARY(MAX) NULL,
    ExternalId UNIQUEIDENTIFIER NULL
);
C#
public class Customer {
    public int Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public string Email { get; set; } = string.Empty;
    public int? Age { get; set; }
    public decimal Balance { get; set; }
    public bool IsActive { get; set; } = true;
    public DateTime CreatedAt { get; set; }
    public byte[]? ProfilePhoto { get; set; }
    public Guid? ExternalId { get; set; }
}

Notice the careful mapping: INT columns become int properties. NVARCHAR becomes string. But there's a critical pattern here: look at the Age column in SQL — it's INT NULL, meaning it can contain NULL. In C#, this maps to int? (nullable int). The question mark is not arbitrary; it signals that the value can be absent.

Similarly, Name in SQL is NOT NULL, so in C# it's a non-nullable string. Modern C# with nullable reference types (C# 8.0+) enforces this distinction: a string property cannot be null, while a string? property can be.

SQL to C# Type Mapping Reference
SQL Data Type C# CLR Type Notes
INT int 32-bit signed integer
BIGINT long 64-bit signed integer; use for large numbers or Unix timestamps
SMALLINT short 16-bit signed integer; rarely used
TINYINT byte 8-bit unsigned integer; 0–255
BIT bool Boolean true/false; stored as 0 or 1
NVARCHAR(n) string Unicode text; n is max length (in characters). EF doesn't enforce n directly.
NVARCHAR(MAX) string Unlimited Unicode text; use sparingly for large documents
CHAR(n) string Fixed-length text; padded with spaces. Use NVARCHAR instead.
DATETIME2 DateTime Preferred over DATETIME; 100-nanosecond precision
DATE DateOnly Date without time; requires .NET 6+
TIME TimeOnly Time without date; requires .NET 6+
DATETIMEOFFSET DateTimeOffset DateTime with timezone offset; use for globally distributed systems
DECIMAL(p,s) decimal Fixed-point decimal; p=precision, s=scale. Essential for money.
FLOAT double 64-bit floating-point; approximate. Don't use for money.
REAL float 32-bit floating-point; rarely used. Avoid.
MONEY decimal Fixed to 4 decimal places; use DECIMAL(10,2) instead for flexibility
UNIQUEIDENTIFIER Guid UUID/GUID; 128-bit unique identifier. Useful for distributed keys.
VARBINARY(MAX) byte[] Binary data (images, files, etc.); consider storing file paths instead
XML string XML as text; use JSON or structured data instead
NTEXT string Deprecated; use NVARCHAR(MAX) instead

The pattern to notice: SQL nullable columns (marked with NULL) map to C# nullable types (marked with ?). Non-nullable SQL columns map to non-nullable C# types. This alignment between database nullability and C# nullability is how the two type systems prevent mismatches.

Entity Framework Core knows these mappings by convention. When you define a DbContext and use fluent API or data annotations, EF generates the correct SQL schema. It also handles the reverse: when you load data from the database, EF converts SQL types to C# types automatically.

Type Mapping Insight: Notice the NULL pattern: SQL nullable columns map to C# nullable types (int? for INT NULL). Non-nullable SQL columns map to non-nullable C# types. This is how the two type systems align. When EF Core generates SQL migrations, it respects this mapping in both directions. Breaking this pattern introduces bugs.
03

Constraints → Validation

SQL constraints enforce data integrity at the DATABASE level. They are the last line of defense: if bad data somehow gets past your application code, the database rejects it. C# validation attributes (Data Annotations) and fluent API configuration enforce rules at the APPLICATION level, providing user-friendly error messages.

The key insight: use BOTH. Database constraints protect you from race conditions, rogue API clients, and direct SQL updates. Application validation provides fast feedback to users and prevents unnecessary database round trips.

Here's how common SQL constraints map to C# equivalents:

SQL Constraints → C# Validation
SQL Constraint C# Data Annotations C# Fluent API Purpose
NOT NULL [Required] .IsRequired() Value cannot be null or empty
PRIMARY KEY [Key] .HasKey() Uniquely identifies each row; no duplicates or nulls
UNIQUE (No annotation) .HasIndex().IsUnique() Column values must be unique across all rows
FOREIGN KEY [ForeignKey("NavProp")] .HasForeignKey() References primary key in another table
CHECK (Age >= 0 AND Age <= 150) [Range(0, 150)] .HasCheckConstraint("name", "condition") Value must satisfy a condition
DEFAULT 0 (No annotation) .HasDefaultValue(0) Use this value if no value provided on INSERT
MAXLENGTH (column width) [MaxLength(255)] .HasMaxLength(255) String values cannot exceed specified length
Custom validation [EmailAddress], [Range], [RegularExpression] .HasCheckConstraint() (SQL-only) Domain-specific validation (email format, date range, etc.)

Let's see a practical example. Suppose we need to enforce that users must be between 0 and 150 years old, and email must be provided:

SQL
CREATE TABLE Users (
    Id INT IDENTITY(1,1) PRIMARY KEY,
    Email NVARCHAR(255) NOT NULL UNIQUE,
    Age INT NULL CHECK (Age >= 0 AND Age <= 150),
    Name NVARCHAR(100) NOT NULL,
    CreatedAt DATETIME2 NOT NULL DEFAULT GETDATE()
);
C# with Data Annotations
public class User {
    [Key]
    public int Id { get; set; }

    [Required]
    [EmailAddress]
    [MaxLength(255)]
    public string Email { get; set; } = string.Empty;

    [Range(0, 150)]
    public int? Age { get; set; }

    [Required]
    [MaxLength(100)]
    public string Name { get; set; } = string.Empty;

    public DateTime CreatedAt { get; set; }
}

And the same configuration using Fluent API:

C# with Fluent API Configuration
modelBuilder.Entity<User>(entity =>
{
    entity.HasKey(u => u.Id);

    entity.Property(u => u.Email)
        .IsRequired()
        .HasMaxLength(255);

    entity.Property(u => u.Age)
        .IsRequired(false);

    entity.HasCheckConstraint("CK_User_Age", "[Age] >= 0 AND [Age] <= 150");

    entity.Property(u => u.Name)
        .IsRequired()
        .HasMaxLength(100);

    entity.Property(u => u.CreatedAt)
        .HasDefaultValueSql("GETDATE()");

    entity.HasIndex(u => u.Email).IsUnique();
});
Constraint Strategy Insight: Database constraints catch bugs that slip through application validation. Think of scenarios like: a race condition between two simultaneous requests, a rogue API client bypassing your code, or a direct SQL update from an admin script. Constraints protect you when application code can't. Use database constraints as your safety net and application validation for user-friendly feedback.
04

Identity Columns → Auto-Generated Keys

Almost every table needs a primary key — a column (or set of columns) that uniquely identifies each row. The simplest approach is an auto-incrementing integer, which SQL Server provides via the IDENTITY keyword. When you insert a row without specifying a value for an IDENTITY column, SQL Server automatically assigns the next integer.

Entity Framework Core detects IDENTITY columns by convention. If you name a property Id or {EntityName}Id (like CustomerId), EF assumes it's an auto-generated key. You don't need annotations or fluent API configuration in most cases.

However, there are scenarios where IDENTITY isn't appropriate. Consider a globally distributed system where you can't rely on a central database to assign IDs. GUIDs (Globally Unique Identifiers) solve this: each application instance can generate a unique ID locally, without coordinating with the database.

SQL: IDENTITY (Auto-increment)
CREATE TABLE Orders (
    Id INT IDENTITY(1,1) PRIMARY KEY,
    CustomerId INT NOT NULL,
    OrderDate DATETIME2 NOT NULL,
    Total DECIMAL(10, 2) NOT NULL
);
C#: IDENTITY (Convention)
public class Order {
    public int Id { get; set; }
    public int CustomerId { get; set; }
    public DateTime OrderDate { get; set; }
    public decimal Total { get; set; }
}

For the IDENTITY approach, that's all you need. EF Core infers the key from the naming convention. If you prefer to be explicit:

C#: IDENTITY (Explicit with [Key])
public class Order {
    [Key]
    public int Id { get; set; }

    public int CustomerId { get; set; }
    public DateTime OrderDate { get; set; }
    public decimal Total { get; set; }
}
C#: IDENTITY (Fluent API)
modelBuilder.Entity<Order>(entity =>
{
    entity.HasKey(o => o.Id);

    entity.Property(o => o.Id)
        .ValueGeneratedOnAdd();
});

Now, let's look at alternatives. GUIDs are useful when you need to generate IDs without database coordination:

SQL: GUID with NEWSEQUENTIALID()
CREATE TABLE Products (
    Id UNIQUEIDENTIFIER PRIMARY KEY DEFAULT NEWSEQUENTIALID(),
    Name NVARCHAR(100) NOT NULL,
    Price DECIMAL(10, 2) NOT NULL
);
C#: GUID Keys
public class Product {
    public Guid Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public decimal Price { get; set; }
}

When using GUIDs, you can either generate them on the client side (via Guid.NewGuid()) or let SQL Server generate them. NEWSEQUENTIALID() generates sequential GUIDs, which are better for index performance than random GUIDs.

In Entity Framework Core, you configure this in OnModelCreating:

C#: Fluent API for GUID Generation
// Client-side GUID generation
modelBuilder.Entity<Product>(entity =>
{
    entity.Property(p => p.Id)
        .ValueGeneratedOnAdd()
        .HasDefaultValueSql("NEWSEQUENTIALID()");
});

// Or generate on client before insert
public class Product {
    public Guid Id { get; set; } = Guid.NewGuid();
    public string Name { get; set; } = string.Empty;
    public decimal Price { get; set; }
}

There's a third approach: the Hi-Lo algorithm. This hybrid strategy uses the database to allocate RANGES of IDs, then generates IDs on the application side. This is efficient and works in distributed scenarios:

C#: Hi-Lo Algorithm for Efficient ID Generation
modelBuilder.Entity<Order>(entity =>
{
    entity.Property(o => o.Id)
        .ValueGeneratedOnAdd()
        .UseHiLo("order_hi_lo_sequence");
});

// The database creates a SEQUENCE behind the scenes:
// CREATE SEQUENCE order_hi_lo_sequence START WITH 1 INCREMENT BY 10;
Key Generation Insight: IDENTITY is the simplest choice for single-database applications where IDs are generated sequentially. GUIDs are better for distributed systems where you need to generate IDs in multiple places. The Hi-Lo algorithm combines the benefits of both — it's database-efficient (allocates IDs in batches) while still allowing application-side generation. Choose IDENTITY unless you have a specific reason to use something else.

Schema Definition vs Class Configuration (Complete Example)

Now let's put it all together with a realistic example. We'll define a Products table with a full schema including primary key, foreign key, constraints, indexes, and defaults. Then we'll show the exact same schema configured in Entity Framework Core.

This is the complete picture: how SQL DDL (Data Definition Language) maps to C# entity configuration. Notice how every SQL constraint, index, and default has a parallel in the C# code.

SQL: Complete Products Table with All Constraints
CREATE TABLE Categories (
    Id INT IDENTITY(1,1) PRIMARY KEY,
    Name NVARCHAR(100) NOT NULL UNIQUE,
    Description NVARCHAR(500) NULL,
    CreatedAt DATETIME2 NOT NULL DEFAULT GETDATE()
);

CREATE TABLE Products (
    Id INT IDENTITY(1,1) PRIMARY KEY,
    Name NVARCHAR(200) NOT NULL,
    Description NVARCHAR(MAX) NULL,
    CategoryId INT NOT NULL,
    Price DECIMAL(10, 2) NOT NULL CHECK (Price > 0),
    Stock INT NOT NULL CHECK (Stock >= 0) DEFAULT 0,
    IsActive BIT NOT NULL DEFAULT 1,
    Sku NVARCHAR(50) NOT NULL UNIQUE,
    CreatedAt DATETIME2 NOT NULL DEFAULT GETDATE(),
    UpdatedAt DATETIME2 NOT NULL DEFAULT GETDATE(),

    CONSTRAINT FK_Products_Categories
        FOREIGN KEY (CategoryId) REFERENCES Categories(Id) ON DELETE CASCADE
);

-- Indexes for query performance
CREATE INDEX IX_Products_CategoryId ON Products(CategoryId);
CREATE INDEX IX_Products_Sku ON Products(Sku);
CREATE INDEX IX_Products_IsActive ON Products(IsActive);
CREATE INDEX IX_Products_CreatedAt ON Products(CreatedAt);
CREATE UNIQUE INDEX IX_Products_SkuUnique ON Products(Sku);
C#: Entity Classes
public class Category {
    [Key]
    public int Id { get; set; }

    [Required]
    [MaxLength(100)]
    public string Name { get; set; } = string.Empty;

    [MaxLength(500)]
    public string? Description { get; set; }

    public DateTime CreatedAt { get; set; }

    // Navigation property
    public ICollection<Product> Products { get; set; } = new List<Product>();
}

public class Product {
    [Key]
    public int Id { get; set; }

    [Required]
    [MaxLength(200)]
    public string Name { get; set; } = string.Empty;

    public string? Description { get; set; }

    [Required]
    public int CategoryId { get; set; }

    [Required]
    [Range(0.01, double.MaxValue)]
    public decimal Price { get; set; }

    [Required]
    [Range(0, int.MaxValue)]
    public int Stock { get; set; }

    [Required]
    public bool IsActive { get; set; } = true;

    [Required]
    [MaxLength(50)]
    public string Sku { get; set; } = string.Empty;

    public DateTime CreatedAt { get; set; }
    public DateTime UpdatedAt { get; set; }

    // Navigation property
    public Category? Category { get; set; }
}

And here's the Fluent API configuration in OnModelCreating that defines all constraints, indexes, and relationships:

C#: Fluent API Configuration in DbContext
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure Category entity
    modelBuilder.Entity<Category>(entity =>
    {
        entity.HasKey(c => c.Id);

        entity.Property(c => c.Id)
            .ValueGeneratedOnAdd();

        entity.Property(c => c.Name)
            .IsRequired()
            .HasMaxLength(100);

        entity.Property(c => c.Description)
            .HasMaxLength(500);

        entity.Property(c => c.CreatedAt)
            .HasDefaultValueSql("GETDATE()");

        entity.HasIndex(c => c.Name)
            .IsUnique();

        // One-to-many relationship
        entity.HasMany(c => c.Products)
            .WithOne(p => p.Category)
            .HasForeignKey(p => p.CategoryId)
            .OnDelete(DeleteBehavior.Cascade);
    });

    // Configure Product entity
    modelBuilder.Entity<Product>(entity =>
    {
        entity.HasKey(p => p.Id);

        entity.Property(p => p.Id)
            .ValueGeneratedOnAdd();

        entity.Property(p => p.Name)
            .IsRequired()
            .HasMaxLength(200);

        entity.Property(p => p.Description);

        entity.Property(p => p.CategoryId)
            .IsRequired();

        entity.Property(p => p.Price)
            .IsRequired()
            .HasPrecision(10, 2);

        entity.HasCheckConstraint("CK_Product_Price", "[Price] > 0");

        entity.Property(p => p.Stock)
            .IsRequired()
            .HasDefaultValue(0);

        entity.HasCheckConstraint("CK_Product_Stock", "[Stock] >= 0");

        entity.Property(p => p.IsActive)
            .IsRequired()
            .HasDefaultValue(true);

        entity.Property(p => p.Sku)
            .IsRequired()
            .HasMaxLength(50);

        entity.Property(p => p.CreatedAt)
            .HasDefaultValueSql("GETDATE()");

        entity.Property(p => p.UpdatedAt)
            .HasDefaultValueSql("GETDATE()");

        // Indexes
        entity.HasIndex(p => p.CategoryId)
            .HasName("IX_Products_CategoryId");

        entity.HasIndex(p => p.Sku)
            .HasName("IX_Products_Sku");

        entity.HasIndex(p => p.IsActive)
            .HasName("IX_Products_IsActive");

        entity.HasIndex(p => p.CreatedAt)
            .HasName("IX_Products_CreatedAt");

        entity.HasIndex(p => p.Sku)
            .IsUnique()
            .HasName("IX_Products_SkuUnique");

        // Foreign key relationship (can also be configured from Category side)
        entity.HasOne(p => p.Category)
            .WithMany(c => c.Products)
            .HasForeignKey(p => p.CategoryId)
            .OnDelete(DeleteBehavior.Cascade);
    });
}
Complete Schema Insight: This example shows the full mapping: every SQL column becomes a C# property with the same type and nullability. Every SQL constraint (NOT NULL, PRIMARY KEY, UNIQUE, CHECK, FOREIGN KEY) gets a C# equivalent via attributes or fluent API. Indexes are configured via HasIndex(). When you run migrations, EF generates the exact SQL DDL shown above from this C# configuration. This is the power of Entity Framework Core — one source of truth in C#, one generated schema in SQL.

Data Annotations vs Fluent API

Entity Framework Core supports two ways to configure entities: Data Annotations and Fluent API. Both accomplish the same goal — telling EF how your classes map to database tables — but they differ in style, power, and maintainability.

Data Annotations are attributes you place directly on C# class properties. They're concise and keep configuration close to the property they describe. However, they're limited: some advanced features (like table splitting, shadow properties, and composite keys) aren't supported by annotations.

Fluent API is code you write in OnModelCreating that configures entities programmatically. It's more verbose but far more powerful. Every annotation has a fluent equivalent, and fluent API can do things annotations can't.

Let's compare the same entity configured both ways:

Data Annotations
using System.ComponentModel.DataAnnotations;

public class User {
    [Key]
    public int Id { get; set; }

    [Required]
    [MaxLength(100)]
    public string Email { get; set; } = string.Empty;

    [Required]
    [MaxLength(50)]
    public string Username { get; set; } = string.Empty;

    [MaxLength(500)]
    public string? Bio { get; set; }

    [Range(18, 120)]
    public int? Age { get; set; }

    [EmailAddress]
    public string? SecondaryEmail { get; set; }
}
Fluent API
public class User {
    public int Id { get; set; }
    public string Email { get; set; } = string.Empty;
    public string Username { get; set; } = string.Empty;
    public string? Bio { get; set; }
    public int? Age { get; set; }
    public string? SecondaryEmail { get; set; }
}

// In OnModelCreating:
modelBuilder.Entity<User>(entity =>
{
    entity.HasKey(u => u.Id);

    entity.Property(u => u.Email)
        .IsRequired()
        .HasMaxLength(100);

    entity.Property(u => u.Username)
        .IsRequired()
        .HasMaxLength(50);

    entity.Property(u => u.Bio)
        .HasMaxLength(500);

    entity.Property(u => u.Age)
        .IsRequired(false);

    entity.HasCheckConstraint("CK_User_Age", "[Age] >= 18 AND [Age] <= 120");

    entity.Property(u => u.SecondaryEmail);
});

Pros and Cons:

Data Annotations Pros: Configuration lives close to properties; minimal boilerplate for simple cases; immediately visible at a glance. Data Annotations Cons: Can't express complex relationships or advanced features; can make classes harder to read if too many attributes; limited to what the annotation library supports.

Fluent API Pros: Extremely powerful — can express any EF Core configuration; keeps entity classes clean and focused on domain logic; more readable for complex scenarios; separates infrastructure concerns from domain model. Fluent API Cons: More verbose; configuration is separate from the properties it describes; requires understanding the fluent API.

In practice, many teams use a hybrid approach: Data Annotations for simple, common rules (Required, MaxLength, Range) and Fluent API in OnModelCreating for complex configurations. Here's an example:

C#: Hybrid Approach (Annotations + Fluent API)
// Entity class with simple annotations
public class BlogPost {
    [Key]
    public int Id { get; set; }

    [Required]
    [MaxLength(500)]
    public string Title { get; set; } = string.Empty;

    public string? Content { get; set; }

    [Required]
    public int AuthorId { get; set; }

    public DateTime CreatedAt { get; set; }
    public DateTime? UpdatedAt { get; set; }

    // Navigation properties (no annotations)
    public Author? Author { get; set; }
    public ICollection<Comment> Comments { get; set; } = new List<Comment>();
    public ICollection<Tag> Tags { get; set; } = new List<Tag>();
}

// Complex configuration in OnModelCreating
modelBuilder.Entity<BlogPost>(entity =>
{
    // Basic properties are handled by annotations above

    // Complex configurations go here
    entity.HasOne(p => p.Author)
        .WithMany(a => a.BlogPosts)
        .HasForeignKey(p => p.AuthorId)
        .OnDelete(DeleteBehavior.Restrict);

    // Many-to-many relationship (can't be done with annotations)
    entity.HasMany(p => p.Tags)
        .WithMany(t => t.BlogPosts)
        .UsingEntity("BlogPostTags");

    // Indexes for performance
    entity.HasIndex(p => p.CreatedAt)
        .HasName("IX_BlogPost_CreatedAt");

    entity.HasIndex(p => p.AuthorId)
        .HasName("IX_BlogPost_AuthorId");

    // Default values
    entity.Property(p => p.CreatedAt)
        .HasDefaultValueSql("GETDATE()");
});
Configuration Pattern Insight: Data Annotations work well for straightforward scenarios and provide documentation at the property level. Fluent API becomes essential for complex relationships, many-to-many mappings, and advanced features. The best approach is to use annotations for basic constraints and fluent API in OnModelCreating for relationship configuration. This keeps your entity classes readable while putting sophisticated logic where it belongs: in your DbContext configuration.
07

Indexes: SQL CREATE INDEX vs EF HasIndex

Indexes are a critical performance tool in databases. An index on a column allows the database to quickly find rows without scanning the entire table. However, indexes come with a cost: every INSERT, UPDATE, and DELETE must maintain every index on that table. Over-indexing can actually slow down writes more than it speeds up reads.

The general rule: index columns that appear frequently in WHERE clauses, JOIN conditions, and ORDER BY clauses. Don't index every column. Focus on the query patterns you actually see in your application.

Let's look at how to create indexes in SQL and how to configure them in Entity Framework Core:

SQL: Simple Index
CREATE INDEX IX_Products_Category
    ON Products(Category);
C#: Simple Index with Fluent API
modelBuilder.Entity<Product>(entity =>
{
    entity.HasIndex(p => p.Category)
        .HasName("IX_Products_Category");
});
SQL: Unique Index (Constraint)
CREATE UNIQUE INDEX IX_Users_Email
    ON Users(Email);
C#: Unique Index with Fluent API
modelBuilder.Entity<User>(entity =>
{
    entity.HasIndex(u => u.Email)
        .IsUnique()
        .HasName("IX_Users_Email");
});
SQL: Composite Index
CREATE INDEX IX_Orders_Composite
    ON Orders(CustomerId, OrderDate)
    INCLUDE (Total);
C#: Composite Index with Fluent API
modelBuilder.Entity<Order>(entity =>
{
    // Note: INCLUDE is not directly supported
    // by fluent API; it requires EF migration SQL
    entity.HasIndex(o => new {
        o.CustomerId,
        o.OrderDate
    })
    .HasName("IX_Orders_Composite");
});

Here's a more complete example showing how to define multiple indexes on an entity:

C#: Complete Index Configuration
modelBuilder.Entity<Order>(entity =>
{
    entity.HasKey(o => o.Id);

    // Single-column index for foreign key lookups
    entity.HasIndex(o => o.CustomerId)
        .HasName("IX_Orders_CustomerId");

    // Single-column index for date range queries
    entity.HasIndex(o => o.OrderDate)
        .HasName("IX_Orders_OrderDate");

    // Composite index for queries filtering by customer and date
    entity.HasIndex(o => new { o.CustomerId, o.OrderDate })
        .HasName("IX_Orders_Customer_Date");

    // Unique index ensures no duplicate orders
    entity.HasIndex(o => o.OrderNumber)
        .IsUnique()
        .HasName("IX_Orders_OrderNumber_Unique");

    // Index for status filtering
    entity.HasIndex(o => o.Status)
        .HasName("IX_Orders_Status");

    // Properties
    entity.Property(o => o.Id).ValueGeneratedOnAdd();
    entity.Property(o => o.CustomerId).IsRequired();
    entity.Property(o => o.OrderDate).HasDefaultValueSql("GETDATE()");
    entity.Property(o => o.OrderNumber).IsRequired().HasMaxLength(50);
    entity.Property(o => o.Status).IsRequired().HasMaxLength(20);
    entity.Property(o => o.Total).HasPrecision(10, 2);
});

When to Index: Index a column if you frequently query by that column (WHERE Status = 'Active'), join on it (JOIN Orders ON Orders.CustomerId = Customers.Id), or sort by it (ORDER BY OrderDate DESC). Common candidates include foreign keys, status fields, and date columns.

When NOT to Index: Don't index boolean columns if they have poor selectivity (like IsDeleted where 99% of rows are false). Don't index columns that are rarely used in queries. Don't index small tables (less than 1000 rows) — a full table scan is faster than index lookup overhead. Every index has maintenance cost on writes.

Covering Indexes (INCLUDE): In SQL, you can create a "covering index" that includes non-key columns, allowing the database to satisfy entire queries from the index without touching the main table. EF Core doesn't expose INCLUDE directly in fluent API, so you need to write raw SQL migrations if you want covering indexes.

Index Strategy Insight: Indexes speed up reads but slow down writes. Every INSERT, UPDATE, and DELETE must maintain every index on a table. The key is to index strategically: focus on the columns in your WHERE clauses, JOIN conditions, and ORDER BY operations. Monitor query performance and only add indexes when you see problems. A database with too many indexes is as bad as one with too few.

Key Takeaways

  • SQL tables and C# classes solve the same problem differently: tables use set-based relational thinking; classes use object-oriented imperative thinking.
  • Every SQL column maps to a C# property with a corresponding CLR type. Nullable SQL columns map to nullable C# types (e.g., int? for INT NULL).
  • SQL constraints enforce data integrity at the database level; C# validation attributes enforce rules at the application level. Use both.
  • Identity columns (IDENTITY) are the simplest primary key strategy for single-database systems. GUIDs are better for distributed systems. Hi-Lo is a good hybrid.
  • Entity Framework Core automatically detects conventions (like Id property names) but also supports explicit configuration via Data Annotations or Fluent API.
  • Data Annotations are concise for simple configurations; Fluent API is more powerful for complex relationships and advanced features. Many teams use a hybrid approach.
  • Indexes accelerate reads but add overhead to writes. Index strategically on columns that appear in WHERE, JOIN, and ORDER BY clauses. Monitor performance to avoid over-indexing.
  • The goal is a single source of truth: your C# code. Entity Framework Core generates the SQL schema from your configuration via migrations.
