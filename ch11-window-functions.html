Chapter 11

Window Functions

You get a requirement: "Number each order per customer" or "Show running total of sales." SQL uses window functions with the OVER clause to solve these elegantly. LINQ has limited support — sometimes you need raw SQL. Here's when each approach works.

01

Requirement: "Number Each Row in the Result"

Business Requirement
A paginated API needs a sequential row number for each result, ordered by creation date (newest first). The UI uses this for display purposes.
SQL Developer Thinks ROW_NUMBER() OVER (ORDER BY CreatedAt DESC) gives me 1, 2, 3, 4... for each row in the result set. Simple and database-efficient.
C# Developer Thinks LINQ has no native window function. I can use raw SQL via EF Core, or load the results in memory and use .Select() with the index overload.
SQL — ROW_NUMBER()
SELECT
    ROW_NUMBER() OVER (ORDER BY CreatedAt DESC) AS RowNum,
    Id,
    OrderTotal,
    CreatedAt
FROM Orders
ORDER BY RowNum;
C# — Raw SQL
var orders = await db.Orders
    .FromSqlInterpolated($@"
        SELECT
            ROW_NUMBER() OVER (
                ORDER BY CreatedAt DESC
            ) AS RowNum,
            Id, OrderTotal, CreatedAt
        FROM Orders
    ")
    .AsNoTracking()
    .ToListAsync(ct);
Dapper — Direct SQL
await using var conn = new SqlConnection(connectionString);
var orders = await conn.QueryAsync<OrderDto>(@"
    SELECT
        ROW_NUMBER() OVER (ORDER BY CreatedAt DESC) AS RowNum,
        Id, OrderTotal, CreatedAt
    FROM Orders
");
C# — In-Memory Fallback
var orders = await db.Orders
    .AsNoTracking()
    .OrderByDescending(o => o.CreatedAt)
    .ToListAsync(ct);

var result = orders
    .Select((order, index) => new
    {
        RowNum = index + 1,
        order.Id,
        order.OrderTotal,
        order.CreatedAt
    })
    .ToList();
02

Requirement: "Number Orders Within Each Customer"

Business Requirement
For each customer, number their orders 1, 2, 3, ... by order date. This helps the reporting system identify "Customer's 3rd order".
SQL Developer Thinks ROW_NUMBER() with PARTITION BY CustomerId resets the counter for each customer. Perfect for grouping logic within groups.
C# Developer Thinks Raw SQL is cleanest. If in-memory: GroupBy by customer, then Select with index inside each group.
SQL — PARTITION BY
SELECT
    ROW_NUMBER() OVER (
        PARTITION BY CustomerId
        ORDER BY OrderDate ASC
    ) AS OrderNum,
    CustomerId,
    Id AS OrderId,
    OrderDate,
    OrderTotal
FROM Orders
ORDER BY CustomerId, OrderDate;
C# — Raw SQL
var orders = await db.Orders
    .FromSqlInterpolated($@"
        SELECT
            ROW_NUMBER() OVER (
                PARTITION BY CustomerId
                ORDER BY OrderDate
            ) AS OrderNum,
            CustomerId, Id, OrderDate, OrderTotal
        FROM Orders
    ")
    .AsNoTracking()
    .ToListAsync(ct);
C# — In-Memory GroupBy Approach
var orders = await db.Orders
    .AsNoTracking()
    .ToListAsync(ct);

var result = orders
    .GroupBy(o => o.CustomerId)
    .SelectMany(g => g
        .OrderBy(o => o.OrderDate)
        .Select((order, index) => new
        {
            OrderNum = index + 1,
            order.CustomerId,
            order.Id,
            order.OrderDate,
            order.OrderTotal
        })
    )
    .ToList();
03

Requirement: "Rank Products by Sales"

Business Requirement
Show product rankings by total revenue. When two products have the same revenue, they get the same rank. RANK skips the next number; DENSE_RANK doesn't.
SQL Developer Thinks RANK() OVER (ORDER BY TotalRevenue DESC) for "1, 1, 3, 4". DENSE_RANK() for "1, 1, 2, 3". Use RANK unless ties must stay consecutive.
C# Developer Thinks Raw SQL is best. In-memory: sort by revenue, then iterate and assign ranks manually, checking for ties.
SQL — RANK vs DENSE_RANK
SELECT
    RANK() OVER (
        ORDER BY TotalRevenue DESC
    ) AS Rank,
    DENSE_RANK() OVER (
        ORDER BY TotalRevenue DESC
    ) AS DenseRank,
    ProductName,
    TotalRevenue
FROM (
    SELECT
        p.Id,
        p.Name AS ProductName,
        SUM(oi.Quantity * oi.Price)
            AS TotalRevenue
    FROM Products p
    LEFT JOIN OrderItems oi ON p.Id = oi.ProductId
    GROUP BY p.Id, p.Name
) summary
ORDER BY TotalRevenue DESC;
C# — Raw SQL
var rankings = await db.Products
    .FromSqlInterpolated($@"
        SELECT
            RANK() OVER (ORDER BY TotalRevenue DESC)
                AS Rank,
            DENSE_RANK() OVER (ORDER BY TotalRevenue DESC)
                AS DenseRank,
            Name AS ProductName,
            TotalRevenue
        FROM (
            SELECT
                p.Id, p.Name,
                SUM(oi.Quantity * oi.Price)
                    AS TotalRevenue
            FROM Products p
            LEFT JOIN OrderItems oi
                ON p.Id = oi.ProductId
            GROUP BY p.Id, p.Name
        ) summary
    ")
    .AsNoTracking()
    .ToListAsync(ct);
C# — In-Memory Ranking
var productRevenue = await db.Products
    .AsNoTracking()
    .Include(p => p.OrderItems)
    .ToListAsync(ct);

var summaries = productRevenue
    .Select(p => new
    {
        p.Id,
        p.Name,
        TotalRevenue = p.OrderItems
            .Sum(oi => oi.Quantity * oi.Price)
    })
    .OrderByDescending(x => x.TotalRevenue)
    .ToList();

// Compute DenseRank by iterating
var ranked = new List<dynamic>();
decimal? prevRevenue = null;
int rank = 0;
foreach (var item in summaries)
{
    if (item.TotalRevenue != prevRevenue) rank++;
    ranked.Add(new
    {
        item.Id,
        item.Name,
        item.TotalRevenue,
        DenseRank = rank
    });
    prevRevenue = item.TotalRevenue;
}
04

Requirement: "Compare Each Order to the Previous One"

Business Requirement
Show each order alongside the previous order's total, so analysts can see trends: "This order is $50 more than the last one."
SQL Developer Thinks LAG(Total, 1) OVER (ORDER BY OrderDate) gives me the previous row. LEAD() gives the next row. Perfect for comparisons.
C# Developer Thinks No LINQ equivalent. Raw SQL or load the list and iterate with index or Zip to compare consecutive rows.
SQL — LAG / LEAD
SELECT
    Id,
    CustomerId,
    OrderDate,
    OrderTotal,
    LAG(OrderTotal, 1) OVER (
        PARTITION BY CustomerId
        ORDER BY OrderDate ASC
    ) AS PreviousTotal,
    LEAD(OrderTotal, 1) OVER (
        PARTITION BY CustomerId
        ORDER BY OrderDate ASC
    ) AS NextTotal,
    OrderTotal - LAG(OrderTotal, 1) OVER (
        PARTITION BY CustomerId
        ORDER BY OrderDate ASC
    ) AS DiffFromPrevious
FROM Orders
ORDER BY CustomerId, OrderDate;
C# — Raw SQL
var orders = await db.Orders
    .FromSqlInterpolated($@"
        SELECT
            Id, CustomerId, OrderDate,
            OrderTotal,
            LAG(OrderTotal, 1) OVER (
                PARTITION BY CustomerId
                ORDER BY OrderDate
            ) AS PreviousTotal,
            LEAD(OrderTotal, 1) OVER (
                PARTITION BY CustomerId
                ORDER BY OrderDate
            ) AS NextTotal
        FROM Orders
        ORDER BY CustomerId, OrderDate
    ")
    .AsNoTracking()
    .ToListAsync(ct);
C# — In-Memory Comparison with Zip
var orders = await db.Orders
    .AsNoTracking()
    .OrderBy(o => o.CustomerId)
    .ThenBy(o => o.OrderDate)
    .ToListAsync(ct);

var result = new List<dynamic>();

var grouped = orders.GroupBy(o => o.CustomerId);
foreach (var customerOrders in grouped)
{
    var sorted = customerOrders
        .OrderBy(o => o.OrderDate)
        .ToList();

    // Use Zip to compare consecutive orders
    var withPrevious = sorted
        .Zip(new[] { (Order?)null }
             .Concat(sorted.SkipLast(1)),
             (current, previous) => new
             {
                 current.Id,
                 current.CustomerId,
                 current.OrderDate,
                 current.OrderTotal,
                 PreviousTotal = previous?.OrderTotal,
                 DiffFromPrevious =
                     previous != null
                         ? current.OrderTotal - previous.OrderTotal
                         : (decimal?)null
             })
        .ToList();

    result.AddRange(withPrevious);
}
05

Requirement: "Show Running Total of Sales"

Business Requirement
Finance needs a chart showing cumulative revenue over time: Jan = $10k, Feb = $24k (10 + 14), Mar = $35k (10 + 14 + 11).
SQL Developer Thinks SUM(Total) OVER (ORDER BY Date ROWS UNBOUNDED PRECEDING) computes the running sum from the first row to the current row. Simple and fast.
C# Developer Thinks Raw SQL is best. In-memory: sort by date, then use a loop or Aggregate() with a running variable.
SQL — Running SUM()
SELECT
    OrderDate,
    OrderTotal,
    SUM(OrderTotal) OVER (
        ORDER BY OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING
             AND CURRENT ROW
    ) AS CumulativeTotal
FROM Orders
ORDER BY OrderDate;
C# — Raw SQL
var running = await db.Orders
    .FromSqlInterpolated($@"
        SELECT
            OrderDate,
            OrderTotal,
            SUM(OrderTotal) OVER (
                ORDER BY OrderDate
                ROWS BETWEEN UNBOUNDED PRECEDING
                     AND CURRENT ROW
            ) AS CumulativeTotal
        FROM Orders
        ORDER BY OrderDate
    ")
    .AsNoTracking()
    .ToListAsync(ct);
C# — In-Memory Running Total
var orders = await db.Orders
    .AsNoTracking()
    .OrderBy(o => o.OrderDate)
    .ToListAsync(ct);

var result = new List<dynamic>();
decimal runningTotal = 0;

foreach (var order in orders)
{
    runningTotal += order.OrderTotal;
    result.Add(new
    {
        order.OrderDate,
        order.OrderTotal,
        CumulativeTotal = runningTotal
    });
}

// Alternative: using Aggregate (functional approach)
var resultAggregate = orders
    .Aggregate(
        seed: (list: new List<dynamic>(), cumulative: 0m),
        (acc, order) =>
        {
            acc.cumulative += order.OrderTotal;
            acc.list.Add(new
            {
                order.OrderDate,
                order.OrderTotal,
                CumulativeTotal = acc.cumulative
            });
            return acc;
        }
    )
    .list;
06

When to Use Raw SQL vs LINQ

The Question
I need window function behavior. Should I use LINQ, raw SQL, or in-memory processing?
Window FunctionLINQ SupportRecommendation
ROW_NUMBER()NoRaw SQL (fast) or in-memory .Select() with index
RANK() / DENSE_RANK()NoRaw SQL preferred; in-memory needs manual logic
LAG() / LEAD()NoRaw SQL preferred; in-memory requires Zip or loop
Running Aggregates (SUM...ROWS)NoRaw SQL for efficiency; in-memory with loop or Aggregate()
PARTITION BYPartialUse with GroupBy; complex cases need raw SQL
SQL Dev Says Use window functions directly in SQL. They're designed for this. Let the database do the work — it's optimized for it. EF's FromSqlInterpolated() is your friend.
C# Dev Says Use raw SQL for window functions. LINQ can't express them. For simple cases (small datasets, in-memory), GroupBy + Select is fine. For production, FromSqlInterpolated with proper parameterization.
Best Practice: Parameterized Raw SQL
// Safe from SQL injection, readable
var minDate = DateTime.UtcNow.AddMonths(-6);

var results = await db.Orders
    .FromSqlInterpolated($@"
        SELECT
            ROW_NUMBER() OVER (
                PARTITION BY CustomerId
                ORDER BY OrderDate DESC
            ) AS OrderNum,
            CustomerId, Id, OrderDate, OrderTotal
        FROM Orders
        WHERE OrderDate >= {minDate}
    ")
    .AsNoTracking()
    .ToListAsync(ct);

// Parameters are automatically safe due to
// string interpolation handling in EF Core
