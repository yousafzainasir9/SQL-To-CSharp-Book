01 — WINDOW FUNCTIONS FUNDAMENTALS

What Are Window Functions: Theory & OVER Clause

Window functions compute values over a set of rows related to the current row, without collapsing the result set. They execute a function (SUM, COUNT, RANK, ROW_NUMBER) over a "window" of rows defined by PARTITION BY and ORDER BY clauses. This is fundamentally different from GROUP BY, which collapses rows into groups.

The syntax is: FUNCTION() OVER (PARTITION BY column ORDER BY column). The PARTITION BY defines logical groups (like GROUP BY), and ORDER BY orders within each partition. The window function computes across rows that remain in the result set—no collapsing.

SQL Window Function Anatomy 
-- Core syntax: function OVER (PARTITION BY ... ORDER BY ...)
SELECT
  EmployeeId,
  Name,
  Salary,
  Department,
  SUM(Salary) OVER (PARTITION BY Department) AS DepartmentTotal,
  SUM(Salary) OVER (PARTITION BY Department ORDER BY Salary) AS CumulativeSum,
  ROW_NUMBER() OVER (PARTITION BY Department ORDER BY Salary DESC) AS RankInDept,
  LAG(Salary) OVER (PARTITION BY Department ORDER BY HireDate) AS PreviousSalary
FROM Employees;

-- Example output:
-- EmployeeId | Name   | Salary | Department | DepartmentTotal | CumulativeSum | RankInDept | PreviousSalary
-- 1          | Alice  | 60000  | Sales      | 180000          | 60000         | 1          | NULL
-- 2          | Bob    | 70000  | Sales      | 180000          | 130000        | 2          | 60000
-- 3          | Carol  | 50000  | Sales      | 180000          | 180000        | 3          | 70000
-- 4          | Dave   | 65000  | IT         | 140000          | 65000         | 1          | NULL
-- 5          | Eve    | 75000  | IT         | 140000          | 140000        | 2          | 65000
Critical Limitation: LINQ Has No Native Window Function Support

LINQ to SQL and Entity Framework do NOT have native equivalents for window functions. There is no .RowNumber(), .Rank(), .LAG(), or .LEAD() method. When you need window functions, you must either use raw SQL with FromSqlRaw or Dapper, or implement approximations in C# (LINQ to Objects). This is a hard boundary of LINQ's expressiveness.

Window functions are powerful because they allow side-by-side ranking, running totals, and neighbor access without the complexity of self-joins. Without them, you'd need multiple JOINs, GROUP BY with subqueries, or post-processing in memory.

The PARTITION BY clause creates logical groups without collapsing rows. The ORDER BY defines the sequence within each partition. Frame specifications (ROWS BETWEEN ...) further refine which rows are included in the function calculation.

Window Function Concepts Illustrated 
-- PARTITION BY creates logical groups
-- ORDER BY orders rows within partitions
-- ROWS BETWEEN defines the window frame

SELECT
  OrderDate,
  Amount,
  -- Partition by month, compute running total within that month
  SUM(Amount) OVER (
    PARTITION BY MONTH(OrderDate)
    ORDER BY OrderDate
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS MonthlyRunningTotal,

  -- No partition, company-wide running total
  SUM(Amount) OVER (
    ORDER BY OrderDate
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS CompanyRunningTotal,

  -- Rank within region
  RANK() OVER (
    PARTITION BY Region
    ORDER BY Amount DESC
  ) AS RegionalRank
FROM Orders;

Understanding window functions is essential because many business requirements naturally map to them: "rank products by sales within each category", "show cumulative expenses by month", "compare each employee's salary to their predecessor's", "find the first and last order dates". Without window functions or their workarounds, these queries become complex and inefficient.

02 — ROW_NUMBER

ROW_NUMBER: Sequential Numbering Without LINQ

ROW_NUMBER() assigns a unique sequential number to each row within a partition, starting at 1. It's used for pagination, deduplication, and identifying "top N per group" scenarios. Unlike RANK or DENSE_RANK, ROW_NUMBER generates no ties—each row gets a unique number.

LINQ has no direct equivalent. You must use raw SQL (FromSqlRaw) or post-process in memory with Select(x => x, index) patterns.

SQL ROW_NUMBER() 
SELECT
  CustomerId,
  Name,
  OrderDate,
  Amount,
  ROW_NUMBER() OVER (
    ORDER BY OrderDate DESC
  ) AS OrderSequence
FROM Orders;
Raw SQL (EF Core) 
var result = context.Orders
  .FromSqlRaw(@"
    SELECT CustomerId, Name, OrderDate, Amount,
      ROW_NUMBER() OVER (ORDER BY OrderDate DESC) AS OrderSequence
    FROM Orders
  ")
  .ToList();
Why No LINQ Equivalent?

LINQ's .Select(x => x, index) provides index-based numbering in LINQ to Objects, but it runs client-side and requires loading all data into memory first. This defeats the purpose of window functions, which compute efficiently at the server level with indexing. Raw SQL is the correct approach here.

ROW_NUMBER Approximation in LINQ to Objects 
// INEFFICIENT: Loads all orders into memory, then numbers them
var orders = context.Orders.ToList(); // All data loaded
var result = orders
  .OrderByDescending(o => o.OrderDate)
  .Select((o, index) => new {
    o.CustomerId,
    o.Name,
    o.OrderDate,
    o.Amount,
    OrderSequence = index + 1
  })
  .ToList();

// BETTER: Use raw SQL if available
var result = context.Orders
  .FromSqlRaw(@"
    SELECT CustomerId, Name, OrderDate, Amount,
      ROW_NUMBER() OVER (ORDER BY OrderDate DESC) AS OrderSequence
    FROM Orders
  ")
  .ToList();

// WORKAROUND with Dapper (if not using EF):
using (var conn = new SqlConnection(connectionString)) {
  var sql = @"
    SELECT CustomerId, Name, OrderDate, Amount,
      ROW_NUMBER() OVER (ORDER BY OrderDate DESC) AS OrderSequence
    FROM Orders
  ";
  var result = conn.Query(sql).ToList();
}

ROW_NUMBER is commonly used for pagination ("get rows 1-10 per category") or finding duplicates ("delete all but the first occurrence"). These use cases are better solved at the database layer where ROW_NUMBER is natively efficient.

03 — ROW_NUMBER WITH PARTITION BY

ROW_NUMBER with PARTITION BY: Grouping & Numbering

Adding PARTITION BY to ROW_NUMBER resets the sequence number for each partition. This pattern is common for "top N per group" queries: "the 3 highest-paid employees per department", "the top 5 products by category", "the first order for each customer".

In SQL, this is a single efficient query. In LINQ, you'd need to post-process with GroupBy and Select with index, which is memory-intensive for large datasets.

SQL PARTITION BY 
SELECT
  DepartmentId,
  Name,
  Salary,
  ROW_NUMBER() OVER (
    PARTITION BY DepartmentId
    ORDER BY Salary DESC
  ) AS RankInDept
FROM Employees
WHERE RankInDept <= 3;  -- Use CTE or subquery
Raw SQL with CTE 
var result = context.Employees
  .FromSqlRaw(@"
    WITH RankedEmps AS (
      SELECT DepartmentId, Name, Salary,
        ROW_NUMBER() OVER (
          PARTITION BY DepartmentId
          ORDER BY Salary DESC
        ) AS RankInDept
      FROM Employees
    )
    SELECT * FROM RankedEmps WHERE RankInDept <= 3
  ")
  .ToList();
Top N Per Group

PARTITION BY enables "top N per group" patterns efficiently. Without window functions, you'd need a self-join or a complex GROUP BY with HAVING, which is slower and harder to read. This is a classic use case where raw SQL is the right tool.

LINQ Approximation: Memory-Based Solution 
// Load all employees into memory and group
var result = context.Employees
  .ToList()  // Load all data
  .GroupBy(e => e.DepartmentId)
  .SelectMany(dept => dept
    .OrderByDescending(e => e.Salary)
    .Select((e, index) => new {
      e.DepartmentId,
      e.Name,
      e.Salary,
      RankInDept = index + 1
    })
    .Where(x => x.RankInDept <= 3)
  )
  .ToList();

// This works but requires loading ALL employees before filtering
// Raw SQL filters at the database, which is much more efficient

This illustrates the fundamental limitation: without database-level window functions, you must load data into memory and process it client-side. For large datasets, this is impractical and slow.

04 — RANK AND DENSE_RANK

RANK vs DENSE_RANK: Handling Ties & Gaps

RANK() and DENSE_RANK() assign ranks to rows within partitions. The difference: RANK produces gaps when values tie, while DENSE_RANK produces no gaps. For example, if two employees tie for first place, RANK gives them both 1 and the next rank is 3, but DENSE_RANK gives them 1 and the next is 2.

Choosing between them depends on your business logic. Leaderboards often use DENSE_RANK (no gaps), while competition rankings use RANK (with gaps for ties).

SQL RANK vs DENSE_RANK 
SELECT
  Name,
  Score,
  RANK() OVER (ORDER BY Score DESC) AS RankWithGaps,
  DENSE_RANK() OVER (ORDER BY Score DESC) AS RankNoDenseGaps
FROM Competitors;

-- Output with ties:
-- Name  | Score | RankWithGaps | RankNoDenseGaps
-- Alice | 100   | 1            | 1
-- Bob   | 100   | 1            | 1
-- Carol | 90    | 3            | 2  (gap in RANK, no gap in DENSE_RANK)
-- Dave  | 85    | 4            | 3
No LINQ Equivalent

LINQ has no .Rank() or .DenseRank() methods. You must use raw SQL. The gap/no-gap logic is part of the window function specification, not something you can easily replicate in C#.

Raw SQL for RANK/DENSE_RANK 
// Using RANK (with gaps for ties)
var result = context.Competitors
  .FromSqlRaw(@"
    SELECT Name, Score,
      RANK() OVER (ORDER BY Score DESC) AS RankWithGaps
    FROM Competitors
  ")
  .ToList();

// Using DENSE_RANK (no gaps)
var result = context.Competitors
  .FromSqlRaw(@"
    SELECT Name, Score,
      DENSE_RANK() OVER (ORDER BY Score DESC) AS RankNoDenseGaps
    FROM Competitors
  ")
  .ToList();

// With Dapper
using (var conn = new SqlConnection(connectionString)) {
  var sql = @"
    SELECT Name, Score,
      RANK() OVER (ORDER BY Score DESC) AS RankWithGaps,
      DENSE_RANK() OVER (ORDER BY Score DESC) AS RankNoDenseGaps
    FROM Competitors
  ";
  var result = conn.Query(sql).ToList();
}

RANK and DENSE_RANK are critical for any ranking requirement. The choice between them is a business decision. For public-facing leaderboards, DENSE_RANK preserves sequential positions. For official records with ties, RANK reflects that multiple entities occupy the same position.

Practical RANK/DENSE_RANK Examples 
// Leaderboard with DENSE_RANK
var leaderboard = context.Players
  .FromSqlRaw(@"
    SELECT PlayerId, Name, Score,
      DENSE_RANK() OVER (ORDER BY Score DESC) AS Position
    FROM Players WHERE IsActive = 1
    ORDER BY Position
  ")
  .ToList();

// Sales ranking per region with RANK
var regional = context.SalesReps
  .FromSqlRaw(@"
    SELECT SalesRepId, Name, Region, TotalSales,
      RANK() OVER (PARTITION BY Region ORDER BY TotalSales DESC)
        AS RegionalRank
    FROM SalesReps
  ")
  .ToList();

// Find all gold medalists (tied for first)
var gold = context.Athletes
  .FromSqlRaw(@"
    SELECT AthleteId, Name, Time,
      DENSE_RANK() OVER (ORDER BY Time ASC) AS Rank
    FROM RaceResults WHERE RaceId = @raceId
  ", new SqlParameter("@raceId", raceId))
  .Where(a => a.Rank == 1)
  .ToList();
05 — LAG AND LEAD

LAG and LEAD: Accessing Previous/Next Row Values

LAG() and LEAD() access values from previous or following rows within a partition, without requiring self-joins. LAG returns the previous row's value; LEAD returns the next row's value. Common uses: "show each employee's salary change", "compare daily sales to prior day", "identify sequence gaps".

These functions have NO direct LINQ equivalent. You must use raw SQL or Dapper without compromise.

SQL LAG and LEAD 
SELECT
  EmployeeId,
  Name,
  Salary,
  LAG(Salary) OVER (ORDER BY HireDate) AS PreviousSalary,
  LEAD(Salary) OVER (ORDER BY HireDate) AS NextSalary,
  Salary - LAG(Salary) OVER (ORDER BY HireDate)
    AS SalaryIncrease
FROM Employees;
Raw SQL (No LINQ Alternative) 
var result = context.Employees
  .FromSqlRaw(@"
    SELECT EmployeeId, Name, Salary,
      LAG(Salary) OVER (ORDER BY HireDate) AS PreviousSalary,
      LEAD(Salary) OVER (ORDER BY HireDate) AS NextSalary,
      Salary - LAG(Salary) OVER (ORDER BY HireDate)
        AS SalaryIncrease
    FROM Employees
  ")
  .ToList();
LAG/LEAD: Impossible in LINQ

There is no way to access "the previous row" in LINQ without raw SQL. This is a fundamental limitation of the LINQ model. You must use raw SQL (FromSqlRaw) or Dapper. No workarounds, no approximations.

LAG/LEAD with Dapper & Business Logic 
using (var conn = new SqlConnection(connectionString)) {
  var sql = @"
    SELECT EmployeeId, Name, Salary, HireDate,
      LAG(Salary) OVER (ORDER BY HireDate) AS PreviousSalary,
      LEAD(Salary) OVER (ORDER BY HireDate) AS NextSalary,
      Salary - LAG(Salary) OVER (ORDER BY HireDate) AS SalaryIncrease
    FROM Employees
    ORDER BY HireDate
  ";

  var employees = conn.Query(sql).ToList();

  // Business logic on the results
  var promotions = employees
    .Where(e => e.SalaryIncrease.HasValue && e.SalaryIncrease > 0)
    .OrderByDescending(e => e.SalaryIncrease)
    .ToList();

  var largePay Raises = promotions
    .Where(e => e.SalaryIncrease.Value > 5000)
    .ToList();
}

LAG and LEAD are essential for time-series analysis, change tracking, and sequential logic. They eliminate expensive self-joins and make comparisons trivial. Whenever you need to compare a row to its neighbors, LAG/LEAD is the answer, and that answer requires raw SQL.

06 — RUNNING TOTALS & CUMULATIVE SUMS

Running Totals: SUM with ROWS BETWEEN

Running totals use SUM with a ROWS BETWEEN frame specification. SUM(amount) OVER (ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) computes cumulative sums from the first row to current. Essential for accounting reports, inventory tracking, and financial statements.

You can approximate in LINQ by loading data and iterating, but server-side computation is far more efficient for large datasets.

SQL Running Total 
SELECT
  TransactionDate,
  Amount,
  SUM(Amount) OVER (
    ORDER BY TransactionDate, TransactionId
    ROWS BETWEEN UNBOUNDED PRECEDING
      AND CURRENT ROW
  ) AS RunningTotal
FROM Transactions
ORDER BY TransactionDate;
Raw SQL (Efficient) 
var result = context.Transactions
  .FromSqlRaw(@"
    SELECT TransactionDate, Amount,
      SUM(Amount) OVER (
        ORDER BY TransactionDate, TransactionId
        ROWS BETWEEN UNBOUNDED PRECEDING
          AND CURRENT ROW
      ) AS RunningTotal
    FROM Transactions
    ORDER BY TransactionDate
  ")
  .ToList();
LINQ Approximation: Running Total in Memory 
// Less efficient: load all data, compute in memory
var transactions = context.Transactions
  .OrderBy(t => t.TransactionDate)
  .ThenBy(t => t.TransactionId)
  .ToList();

var result = new List();
decimal runningTotal = 0;

foreach (var t in transactions) {
  runningTotal += t.Amount;
  result.Add(new {
    t.TransactionDate,
    t.Amount,
    RunningTotal = runningTotal
  });
}

// This works but requires loading ALL transactions first
// Raw SQL scales efficiently with database indexes

// Monthly running total example
var monthlyRunning = context.Expenses
  .FromSqlRaw(@"
    SELECT ExpenseDate, Amount, Category,
      SUM(Amount) OVER (
        PARTITION BY MONTH(ExpenseDate), YEAR(ExpenseDate)
        ORDER BY ExpenseDate
        ROWS BETWEEN UNBOUNDED PRECEDING
          AND CURRENT ROW
      ) AS MonthlyRunningTotal,
      SUM(Amount) OVER (
        ORDER BY ExpenseDate
        ROWS BETWEEN UNBOUNDED PRECEDING
          AND CURRENT ROW
      ) AS CompanyRunningTotal
    FROM Expenses
    WHERE ExpenseDate >= DATEADD(YEAR, -1, GETDATE())
    ORDER BY ExpenseDate
  ")
  .ToList();

// Department budget tracking with running totals
var budgetStatus = context.DepartmentBudget
  .FromSqlRaw(@"
    SELECT DepartmentId, Department, Amount, TransactionDate,
      SUM(Amount) OVER (
        PARTITION BY DepartmentId
        ORDER BY TransactionDate
        ROWS BETWEEN UNBOUNDED PRECEDING
          AND CURRENT ROW
      ) AS DeptRunningTotal,
      1000000 - SUM(Amount) OVER (
        PARTITION BY DepartmentId
        ORDER BY TransactionDate
        ROWS BETWEEN UNBOUNDED PRECEDING
          AND CURRENT ROW
      ) AS RemainingBudget
    FROM DepartmentBudget
    WHERE YEAR(TransactionDate) = YEAR(GETDATE())
  ")
  .ToList();

Running totals are fundamental to financial reporting, inventory management, and progress tracking. They allow you to show cumulative progress alongside individual transactions. Window functions make this trivial; without them, you'd need complex GROUP BY logic or repeated loops in C#.

When to Drop to Raw SQL: Decision Framework

Window functions have no native LINQ equivalent. You must use raw SQL (FromSqlRaw in EF Core) or Dapper. The decision is simple: if your query requires window functions, use raw SQL. Period. Don't try to approximate in LINQ—it's inefficient and unreliable for large datasets.

Window Functions = Raw SQL

This is a hard rule. LINQ to SQL and Entity Framework have no support for ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, or framed aggregates. Attempting to approximate them in C# requires loading data into memory, which defeats the purpose of window functions' efficiency.

Decision Framework: Examples 
// Scenario 1: Pagination with ROW_NUMBER
// Decision: Raw SQL (FromSqlRaw)
var result = context.Orders
  .FromSqlRaw(@"
    WITH Ranked AS (
      SELECT *,
        ROW_NUMBER() OVER (ORDER BY OrderDate DESC) AS RowNum
      FROM Orders
    )
    SELECT * FROM Ranked WHERE RowNum BETWEEN @skip AND @skip + @take
  ", new SqlParameter("@skip", 20), new SqlParameter("@take", 10))
  .ToList();

// Scenario 2: Top 5 products per category
// Decision: Raw SQL with PARTITION BY
var result = context.Products
  .FromSqlRaw(@"
    WITH Ranked AS (
      SELECT *,
        ROW_NUMBER() OVER (PARTITION BY CategoryId ORDER BY Sales DESC) AS Rank
      FROM Products
    )
    SELECT * FROM Ranked WHERE Rank <= 5
  ")
  .ToList();

// Scenario 3: Year-over-year comparison (LAG)
// Decision: Raw SQL (no LINQ workaround)
var result = context.MonthlySales
  .FromSqlRaw(@"
    SELECT *,
      LAG(Amount) OVER (PARTITION BY Region ORDER BY YearMonth) AS PriorMonthAmount,
      Amount - LAG(Amount) OVER (PARTITION BY Region ORDER BY YearMonth) AS MoMChange
    FROM MonthlySales
  ")
  .ToList();

// Scenario 4: Cumulative expense tracking
// Decision: Raw SQL with SUM OVER
var result = context.Expenses
  .FromSqlRaw(@"
    SELECT *,
      SUM(Amount) OVER (
        PARTITION BY DepartmentId
        ORDER BY ExpenseDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
      ) AS CumulativeExpense
    FROM Expenses
    ORDER BY DepartmentId, ExpenseDate
  ")
  .ToList();

// Scenario 5: Duplicate detection
// Decision: Raw SQL with ROW_NUMBER
var result = context.Customers
  .FromSqlRaw(@"
    WITH Duplicates AS (
      SELECT *,
        ROW_NUMBER() OVER (PARTITION BY Email ORDER BY CreatedDate) AS DupNum
      FROM Customers
    )
    SELECT * FROM Duplicates WHERE DupNum > 1
  ")
  .ToList();

When you encounter a requirement that screams "window function" (pagination, ranking, running totals, lag/lead comparisons), immediately write the SQL directly. Using FromSqlRaw or Dapper is the correct approach. Don't waste time trying to approximate it in LINQ—you'll either produce inefficient code or give up and use raw SQL anyway.

Advanced: Complex Window Function Example with EF Core 
// Multiple window functions in a single query
var result = context.Orders
  .FromSqlRaw(@"
    SELECT
      o.OrderId,
      o.CustomerId,
      c.Name,
      o.OrderDate,
      o.Total,
      -- Cumulative spending per customer
      SUM(o.Total) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
      ) AS CustomerRunningTotal,

      -- Rank by amount within customer
      RANK() OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.Total DESC
      ) AS OrderRankByAmount,

      -- Previous order date for customer
      LAG(o.OrderDate) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
      ) AS PreviousOrderDate,

      -- Days since last order
      DATEDIFF(DAY,
        LAG(o.OrderDate) OVER (PARTITION BY o.CustomerId ORDER BY o.OrderDate),
        o.OrderDate
      ) AS DaysSinceLast,

      -- Row number for pagination
      ROW_NUMBER() OVER (ORDER BY o.OrderDate DESC) AS GlobalRowNum

    FROM Orders o
    INNER JOIN Customers c ON o.CustomerId = c.CustomerId
  ")
  .ToList();

// Process results in C#
var topOrdersPerCustomer = result
  .Where(r => r.OrderRankByAmount <= 3)
  .GroupBy(r => r.CustomerId)
  .ToList();

// Identify dormant customers
var dormantCustomers = result
  .Where(r => r.DaysSinceLast.HasValue && r.DaysSinceLast > 90)
  .DistinctBy(r => r.CustomerId)
  .ToList();

This example uses multiple window functions simultaneously, computing cumulative totals, rankings, neighbor access, and row numbering in a single efficient server-side query. In pure LINQ, you'd need multiple round trips to the database or load massive datasets into memory. Window functions excel at these scenarios.

Dapper Alternative for Window Functions 
public class OrderAnalysis {
  public int OrderId { get; set; }
  public int CustomerId { get; set; }
  public string Name { get; set; }
  public DateTime OrderDate { get; set; }
  public decimal Total { get; set; }
  public decimal CustomerRunningTotal { get; set; }
  public int OrderRankByAmount { get; set; }
  public DateTime? PreviousOrderDate { get; set; }
  public int? DaysSinceLast { get; set; }
  public int GlobalRowNum { get; set; }
}

using (var conn = new SqlConnection(connectionString)) {
  var sql = @"
    SELECT o.OrderId, o.CustomerId, c.Name, o.OrderDate, o.Total,
      SUM(o.Total) OVER (
        PARTITION BY o.CustomerId
        ORDER BY o.OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
      ) AS CustomerRunningTotal,
      RANK() OVER (PARTITION BY o.CustomerId ORDER BY o.Total DESC)
        AS OrderRankByAmount,
      LAG(o.OrderDate) OVER (PARTITION BY o.CustomerId ORDER BY o.OrderDate)
        AS PreviousOrderDate,
      DATEDIFF(DAY,
        LAG(o.OrderDate) OVER (PARTITION BY o.CustomerId ORDER BY o.OrderDate),
        o.OrderDate
      ) AS DaysSinceLast,
      ROW_NUMBER() OVER (ORDER BY o.OrderDate DESC) AS GlobalRowNum
    FROM Orders o
    INNER JOIN Customers c ON o.CustomerId = c.CustomerId
  ";

  var orders = conn.Query(sql).ToList();
  var highValueDormant = orders
    .Where(o => o.CustomerRunningTotal > 5000 && o.DaysSinceLast > 90)
    .ToList();
}

Dapper provides a clean mapping layer while maintaining the efficiency of raw SQL. For applications that heavily use window functions, Dapper often becomes preferable to Entity Framework because it eliminates the translation layer and gives you direct SQL control.

Key Takeaways

  • Window functions compute values over sets of rows without collapsing results—fundamentally different from GROUP BY
  • LINQ has NO native window function support: no .RowNumber(), .Rank(), .LAG(), .LEAD(), or framed aggregates
  • ROW_NUMBER() is used for pagination and duplicate detection—use raw SQL with FromSqlRaw
  • PARTITION BY resets functions within groups—critical for "top N per group" patterns
  • RANK and DENSE_RANK handle ties differently—choose based on your semantics (gaps vs no gaps)
  • LAG and LEAD access previous/next rows—have no LINQ equivalent, use raw SQL or Dapper
  • Running totals with SUM OVER are more efficient at the database than LINQ approximations
  • Always use raw SQL (FromSqlRaw or Dapper) for window function requirements—don't try to approximate in C#
  • Measure: window functions in SQL are far more efficient than loading data into memory and processing in C#

Window functions are a cornerstone of SQL that have no equivalent in LINQ. Recognizing when your query needs them and switching to raw SQL is a critical skill. The performance benefits are dramatic—often 10x faster with 10x less memory than approximations in C#. Master window functions at the SQL level, use raw SQL or Dapper to access them, and your applications will be faster and more scalable. This chapter establishes a firm boundary: when you need window functions, step outside LINQ and into raw SQL. That's the correct choice, and it's what professional database developers do daily. From here, we move on to Entity Framework Core configuration, migrations, and best practices for managing database connections and state.
