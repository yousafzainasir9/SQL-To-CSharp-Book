Chapter 06

WHERE & Filtering

You get a requirement: "Show only active customers in New York." A SQL developer writes WHERE IsActive = 1 AND City = 'New York'. A C# developer writes .Where(c => c.IsActive && c.City == "New York"). Same logic, different syntax.

01

Requirement: "Show Only Active Customers"

Business Requirement
Dashboard shows only active customers. We need to filter out inactive accounts.
SQL Developer Thinks I need a WHERE clause. Filter where IsActive equals 1 (true). This runs in the database, so it filters before sending data back.
C# Developer Thinks I need a .Where() method on the query. Filter where c.IsActive is true. LINQ translates this to SQL WHERE for me automatically.
SQL
SELECT Id, Name, Email
FROM Customers
WHERE IsActive = 1;
LINQ Method
var customers = await db.Customers
    .AsNoTracking()
    .Where(c => c.IsActive)
    .Select(c => new { c.Id, c.Name, c.Email })
    .ToListAsync(ct);
LINQ Query
var customers = await (
    from c in db.Customers.AsNoTracking()
    where c.IsActive
    select new { c.Id, c.Name, c.Email }
).ToListAsync(ct);
02

Requirement: "Find Premium Customers in Specific Cities"

Business Requirement
Marketing needs active customers in New York or Los Angeles with account balance greater than $1000. These are our high-value prospects.
SQL Developer Thinks I'll use AND, OR, and NOT. AND binds tighter than OR, so I need parentheses: WHERE IsActive = 1 AND (City = 'NY' OR City = 'LA') AND Balance > 1000.
C# Developer Thinks I'll use && (AND), || (OR), and ! (NOT). Same precedence rules apply: && binds tighter than ||, so parentheses matter.
SQL
SELECT Id, Name, City, Balance
FROM Customers
WHERE IsActive = 1
  AND (City = 'NY' OR City = 'LA')
  AND Balance > 1000
ORDER BY Balance DESC;
C# — LINQ Method
var premiumCustomers = await db.Customers
    .AsNoTracking()
    .Where(c => c.IsActive
        && (c.City == "NY" || c.City == "LA")
        && c.Balance > 1000)
    .OrderByDescending(c => c.Balance)
    .Select(c => new { c.Id, c.Name, c.City, c.Balance })
    .ToListAsync(ct);
C# — LINQ Query Syntax
var premiumCustomers = await (
    from c in db.Customers.AsNoTracking()
    where c.IsActive
       && (c.City == "NY" || c.City == "LA")
       && c.Balance > 1000
    orderby c.Balance descending
    select new { c.Id, c.Name, c.City, c.Balance }
).ToListAsync(ct);
03

Requirement: "Search Products by Name"

Business Requirement
Search bar that finds products containing user's search text. Users type partial names. Must be fast.
SQL Developer Thinks LIKE '%search%' finds text anywhere. LIKE 'search%' finds text at the start. LIKE '%search' finds text at the end. %search% can be slow without indexes.
C# Developer Thinks .Contains() for "anywhere", .StartsWith() for "begins with", .EndsWith() for "ends with". EF translates these to SQL LIKE. For special LIKE patterns, use EF.Functions.Like().
SQL — LIKE Patterns
-- Find "Laptop" anywhere in name
SELECT Id, Name, Price
FROM Products
WHERE Name LIKE '%Laptop%';

-- Find products starting with "Dell"
SELECT Id, Name, Price
FROM Products
WHERE Name LIKE 'Dell%';

-- Find products ending with "Pro"
SELECT Id, Name, Price
FROM Products
WHERE Name LIKE '%Pro';
C# — String Methods
// Anywhere in name (LIKE '%Laptop%')
var results = await db.Products
    .AsNoTracking()
    .Where(p => p.Name.Contains("Laptop"))
    .ToListAsync(ct);

// Starts with "Dell" (LIKE 'Dell%')
var results = await db.Products
    .AsNoTracking()
    .Where(p => p.Name.StartsWith("Dell"))
    .ToListAsync(ct);

// Ends with "Pro" (LIKE '%Pro')
var results = await db.Products
    .AsNoTracking()
    .Where(p => p.Name.EndsWith("Pro"))
    .ToListAsync(ct);
C# — Advanced: EF.Functions.Like() for Case-Insensitive
using Microsoft.EntityFrameworkCore;

// Case-insensitive search (SQL collation-safe)
var search = "laptop";
var results = await db.Products
    .AsNoTracking()
    .Where(p => EF.Functions.Like(p.Name, $"%{search}%"))
    .ToListAsync(ct);
Dapper — Raw SQL with LIKE
await using var conn = new SqlConnection(connectionString);
var search = "laptop";
var results = await conn.QueryAsync<ProductDto>(
    @"SELECT Id, Name, Price
      FROM Products
      WHERE Name LIKE @Pattern",
    new { Pattern = $"%{search}%" },
    commandTimeout: 30);
04

Requirement: "Handle Missing Data"

Business Requirement
Some customers don't have a phone number on file. Find customers with missing phone numbers AND find customers who DO have a phone number.
SQL Developer Thinks NULL is not a value—it's the absence of a value. I can't use WHERE Phone = NULL (always false). I must use IS NULL or IS NOT NULL.
C# Developer Thinks I can use .Where(c => c.Phone == null). EF translates this to SQL IS NULL automatically. For combining null checks, I can use the ?? operator or nested conditions.
SQL — IS NULL / IS NOT NULL
-- Find customers with no phone
SELECT Id, Name, Phone
FROM Customers
WHERE Phone IS NULL;

-- Find customers WITH a phone
SELECT Id, Name, Phone
FROM Customers
WHERE Phone IS NOT NULL;

-- Customers without email OR phone
SELECT Id, Name
FROM Customers
WHERE Email IS NULL OR Phone IS NULL;
C# — == null / != null
// Find customers with no phone
var customers = await db.Customers
    .AsNoTracking()
    .Where(c => c.Phone == null)
    .Select(c => new { c.Id, c.Name, c.Phone })
    .ToListAsync(ct);

// Find customers WITH a phone
var customers = await db.Customers
    .AsNoTracking()
    .Where(c => c.Phone != null)
    .Select(c => new { c.Id, c.Name, c.Phone })
    .ToListAsync(ct);

// No email OR no phone
var customers = await db.Customers
    .AsNoTracking()
    .Where(c => c.Email == null || c.Phone == null)
    .ToListAsync(ct);
C# — Using COALESCE vs ?? Operator
using Microsoft.EntityFrameworkCore;

// SQL COALESCE — use EF.Functions
// Get phone or email, prefer phone
var contacts = await db.Customers
    .AsNoTracking()
    .Select(c => new {
        c.Id,
        c.Name,
        PreferredContact = EF.Functions.Coalesce(c.Phone, c.Email)
    })
    .ToListAsync(ct);

// Client-side: use ?? operator (after data loads)
var custList = await db.Customers.AsNoTracking().ToListAsync(ct);
var contacts = custList.Select(c => new {
    c.Id,
    c.Name,
    PreferredContact = c.Phone ?? c.Email ?? "No contact"
}).ToList();
05

Requirement: "Filter by Multiple Categories"

Business Requirement
Show products in Electronics, Books, or Clothing categories. Must be efficient—SQL IN clause is perfect for this.
SQL Developer Thinks WHERE CategoryId IN (1, 2, 3) is efficient. The database checks if the value matches any item in the list quickly using an index.
C# Developer Thinks I'll use .Where(p => ids.Contains(p.CategoryId)). EF is smart: it generates IN in SQL, not multiple ORs.
SQL — IN Clause
-- Show products in 3 categories
SELECT Id, Name, CategoryId, Price
FROM Products
WHERE CategoryId IN (1, 2, 3)
ORDER BY Name;
C# — .Contains()
var categoryIds = new[] { 1, 2, 3 };
var products = await db.Products
    .AsNoTracking()
    .Where(p => categoryIds.Contains(p.CategoryId))
    .OrderBy(p => p.Name)
    .Select(p => new { p.Id, p.Name, p.CategoryId, p.Price })
    .ToListAsync(ct);

// Also works with List<T>
var cats = new List<int> { 1, 2, 3 };
var products = await db.Products
    .AsNoTracking()
    .Where(p => cats.Contains(p.CategoryId))
    .ToListAsync(ct);
Dapper — IN with Parameters
await using var conn = new SqlConnection(connectionString);
var categoryIds = new[] { 1, 2, 3 };
var products = await conn.QueryAsync<ProductDto>(
    @"SELECT Id, Name, CategoryId, Price
      FROM Products
      WHERE CategoryId IN @CatIds
      ORDER BY Name",
    new { CatIds = categoryIds.AsEnumerable() });
06

Requirement: "Find Customers with High-Value Orders"

Business Requirement
Find customers who have placed at least one order over $500. This requires checking related data—orders.
SQL Developer Thinks I need a subquery with EXISTS or IN: WHERE EXISTS (SELECT 1 FROM Orders WHERE CustomerId = c.Id AND Total > 500). This checks if ANY order is over $500.
C# Developer Thinks I'll use .Any() on the navigation property: .Where(c => c.Orders.Any(o => o.Total > 500)). EF converts this to an EXISTS subquery automatically.
SQL — EXISTS
SELECT Id, Name, Email
FROM Customers c
WHERE EXISTS (
    SELECT 1
    FROM Orders o
    WHERE o.CustomerId = c.Id
      AND o.Total > 500
)
ORDER BY Name;
C# — .Any()
var bigSpenders = await db.Customers
    .AsNoTracking()
    .Where(c => c.Orders
        .Any(o => o.Total > 500))
    .OrderBy(c => c.Name)
    .Select(c => new { c.Id, c.Name, c.Email })
    .ToListAsync(ct);
C# — .All() for "Every Order Must Meet Condition"
// Find customers where ALL orders are over $50 (high-value customers)
var allHighValue = await db.Customers
    .AsNoTracking()
    .Where(c => c.Orders.Count > 0
        && c.Orders.All(o => o.Total > 50))
    .ToListAsync(ct);

// Find customers where NO orders are cancelled
var activeCustomers = await db.Customers
    .AsNoTracking()
    .Where(c => !c.Orders.Any(o => o.Status == "Cancelled"))
    .ToListAsync(ct);
SQL — Alternative: IN with Subquery
SELECT Id, Name, Email
FROM Customers
WHERE Id IN (
    SELECT DISTINCT CustomerId
    FROM Orders
    WHERE Total > 500
)
ORDER BY Name;
07

Requirement: "Make Filters Faster"

Business Requirement
Which WHERE patterns are index-friendly vs which force table scans? Some filters kill performance.
WHERE Pattern Index-Friendly? Why?
WHERE City = 'NY' Yes Direct equality on indexed column is fast
WHERE Price > 100 Yes Range queries use index seek
WHERE Name LIKE 'A%' Yes Prefix match can use index (start-bound)
WHERE Name LIKE '%ABC%' No Wildcard on both sides forces scan
WHERE YEAR(CreatedAt) = 2024 No Function on column prevents index use
WHERE CreatedAt >= '2024-01-01' Yes No function—index works
WHERE Name = @name COLLATE Latin1_General_CS_AS Maybe Collation can prevent index if mismatched
WHERE Price * 2 > 1000 No Expression on column prevents index
WHERE CategoryId IN (1, 2, 3) Yes IN is optimized; translates to multiple seeks
WHERE IsActive = 1 AND City = 'NY' Yes Composite index on (IsActive, City)
Best Practices: How to Keep Filters Fast
// FAST: Direct column comparison
var active = await db.Customers
    .AsNoTracking()
    .Where(c => c.IsActive && c.City == "NY")
    .ToListAsync(ct);

// SLOW: Function on column
var active = await db.Customers
    .AsNoTracking()
    .Where(c => c.Name.ToUpper() == "JOHN")
    .ToListAsync(ct);

// FAST: Date range (no function)
var recent = await db.Customers
    .AsNoTracking()
    .Where(c => c.CreatedAt >= new DateTime(2024, 1, 1))
    .ToListAsync(ct);

// SLOW: Function on date column
var recent = await db.Customers
    .AsNoTracking()
    .Where(c => EF.Functions.DateDiffDay(c.CreatedAt, DateTime.Now) < 30)
    .ToListAsync(ct);

// FAST: LIKE with prefix
var search = await db.Products
    .AsNoTracking()
    .Where(p => p.Name.StartsWith("Dell"))
    .ToListAsync(ct);

// SLOW: LIKE with both wildcards
var search = await db.Products
    .AsNoTracking()
    .Where(p => p.Name.Contains("laptop"))
    .ToListAsync(ct);

// BETTER: Full-text search for complex patterns
var search = await db.Products
    .AsNoTracking()
    .FromSqlInterpolated(
        $@"SELECT * FROM Products
           WHERE CONTAINS(Name, {searchTerm})")
    .ToListAsync(ct);
