Chapter 06

WHERE → .Where() — Filtering Data

Master SQL WHERE filtering and LINQ .Where() methods. Learn boolean logic, NULL semantics across SQL and C#, string matching patterns, and the subtle differences in three-valued logic that cause bugs.

In This Chapter

  • Comparison operators and their behavior across SQL and C#

  • NULL handling: three-valued logic vs two-valued logic

01

Simple Conditions and Comparison Operators

WHERE filtering is the foundation of data retrieval. A WHERE clause contains one or more boolean expressions that evaluate to true or false for each row. Only rows where the expression evaluates to true are included in the result set. LINQ's .Where() method works identically: you provide a predicate (a function returning bool), and only elements satisfying the predicate are retained.

The key principle: filter at the database, not in your application. The database is optimized for filtering millions of rows in parallel. Your C# application is optimized for logic and presentation. Mixing these responsibilities is a common source of performance problems.

All comparison operators map directly:

Comparison Operators 
SQL    C#     Meaning
=      ==     Equal to
!=     !=     Not equal to
<>     !=     Not equal to (SQL alternative)
<      <      Less than
>      >      Greater than
<=     <=     Less than or equal to
>=     >=     Greater than or equal to

Basic filtering:

SQL 
SELECT *
FROM Products
WHERE Price > 100;
Method Syntax 
var result = products
  .Where(p => p.Price > 100)
  .ToList();
Query Syntax 
var result = (from p in products
              where p.Price > 100
              select p)
  .ToList();

Multiple conditions (all must be true):

SQL Multiple Conditions 
SELECT Id, Name, Price
FROM Products
WHERE Price >= 50 AND Price <= 200
  AND Category = 'Electronics';
C# Multiple Conditions 
var result = context.Products
  .Where(p => p.Price >= 50
    && p.Price <= 200
    && p.Category == "Electronics")
  .Select(p => new {
    p.Id,
    p.Name,
    p.Price
  })
  .ToList();
Filter at the Database Level

A critical principle: fetch only the rows you need. Never call .ToList() and then .Where() in C#. This fetches every row from the database and filters in memory—catastrophically inefficient. Always chain .Where() before .ToList() so the filter executes at the database. The database has indexes, query optimization, and parallelization. Your C# code does not.

02

AND / OR / NOT Logic and Precedence

Complex WHERE clauses combine conditions with AND (all must be true), OR (at least one must be true), and NOT (negate the result). Both SQL and C# have the same logical operators (though with different syntax), and both follow the same operator precedence: AND binds tighter than OR. This means AND conditions are evaluated first.

AND logic (all conditions required):

SQL AND 
SELECT *
FROM Customers
WHERE Country = 'USA'
  AND Status = 'Active'
  AND CreatedDate >= '2023-01-01';
Method Syntax 
var result = customers
  .Where(c => c.Country == "USA"
    && c.Status == "Active"
    && c.CreatedDate >=
      new DateTime(2023, 1, 1))
  .ToList();
Query Syntax 
var result = (from c in customers
              where c.Country == "USA"
                && c.Status == "Active"
                && c.CreatedDate >=
                  new DateTime(2023, 1, 1)
              select c)
  .ToList();

OR logic (at least one condition required):

SQL OR 
SELECT *
FROM Customers
WHERE Country = 'USA'
  OR Country = 'Canada'
  OR Country = 'Mexico';
C# OR (Chained) 
var result = customers
  .Where(c => c.Country == "USA"
    || c.Country == "Canada"
    || c.Country == "Mexico")
  .ToList();

// For cleaner OR with lists, use IN pattern:
var countries = new[] { "USA", "Canada", "Mexico" };
var result2 = customers
  .Where(c => countries.Contains(c.Country))
  .ToList();

NOT logic and negation:

SQL NOT 
-- Find products that are NOT food
-- and NOT cheap
SELECT *
FROM Products
WHERE NOT (Category = 'Food'
  AND Price < 10);

-- Equivalent using De Morgan's Law:
WHERE Category != 'Food'
  OR Price >= 10;
C# NOT 
// Direct negation with !
var result = products
  .Where(p => !(p.Category == "Food"
    && p.Price < 10))
  .ToList();

// Equivalent (De Morgan's Law):
var result2 = products
  .Where(p => p.Category != "Food"
    || p.Price >= 10)
  .ToList();

Operator precedence: AND evaluated before OR. Always use parentheses for clarity:

Precedence and Parentheses 
// This expression:
var result = customers
  .Where(c => c.Status == "Active" && c.Country == "USA"
    || c.Email.Contains("@gmail"))
  .ToList();

// Is parsed as: (Active AND USA) OR gmail
// Because && binds tighter than ||

// If you intended: Active AND (USA OR gmail):
var result2 = customers
  .Where(c => c.Status == "Active"
    && (c.Country == "USA" || c.Email.Contains("@gmail")))
  .ToList();

// ALWAYS use parentheses to remove ambiguity
// Even when not strictly required by precedence
Parentheses = Clarity

Explicit parentheses make your intent unmistakable. Someone reading your code (maybe you, six months later) can instantly understand the logic without remembering operator precedence. The performance impact is zero, but the readability impact is enormous.

03

IN Clause → .Contains()

SQL's IN operator checks whether a value appears in a list. Rather than writing WHERE Status = 'A' OR Status = 'B' OR Status = 'C', you write WHERE Status IN ('A', 'B', 'C'). It's syntactic sugar for multiple OR conditions, but far more readable. LINQ provides .Contains(), which generates the same SQL but with cleaner C# syntax.

SQL IN 
SELECT *
FROM Customers
WHERE City IN ('London', 'Paris',
  'Berlin', 'Amsterdam');
Method Syntax 
var cities = new[] {
  "London", "Paris",
  "Berlin", "Amsterdam"
};

var result = customers
  .Where(c => cities.Contains(c.City))
  .ToList();
Query Syntax 
var cities = new[] {
  "London", "Paris",
  "Berlin", "Amsterdam"
};

var result = (from c in customers
              where cities.Contains(c.City)
              select c)
  .ToList();

NOT IN (exclude matches):

SQL NOT IN 
SELECT *
FROM Customers
WHERE City NOT IN ('London', 'Paris', 'Berlin');
C# NOT IN 
var excluded = new[] { "London", "Paris", "Berlin" };

var result = customers
  .Where(c => !excluded.Contains(c.City))
  .ToList();

// Read as: where city is NOT in the excluded list

IN with subqueries (finding orders containing specific products):

SQL IN with Subquery 
SELECT *
FROM Orders o
WHERE OrderId IN (
  SELECT OrderId FROM OrderItems
  WHERE ProductId IN (42, 99, 157)
);
C# IN with Subquery 
var productIds = new[] { 42, 99, 157 };

var orderIds = context.OrderItems
  .Where(oi => productIds.Contains(oi.ProductId))
  .Select(oi => oi.OrderId)
  .Distinct()
  .ToList();

var result = context.Orders
  .Where(o => orderIds.Contains(o.OrderId))
  .ToList();

// Or more elegantly with single query:
var result2 = context.Orders
  .Where(o => context.OrderItems
    .Where(oi => productIds.Contains(oi.ProductId))
    .Select(oi => oi.OrderId)
    .Contains(o.OrderId))
  .ToList();
IN Performance: Both Good and Bad

IN with a small list (tens of values) is efficient; the database uses hash matching. IN with millions of values can slow down. For very large lists, consider loading into a temp table in the database. Entity Framework handles reasonable list sizes well. Avoid thousands of values in a single IN clause.

04

BETWEEN → Range Comparisons

SQL's BETWEEN operator checks if a value falls within a range, inclusive on both boundaries. LINQ has no direct BETWEEN equivalent; instead, use >= and <= comparisons. The generated SQL is identical, and the semantics are the same. BETWEEN is purely syntactic convenience in SQL; C# is equally expressive using basic comparisons.

SQL BETWEEN 
SELECT *
FROM Products
WHERE Price BETWEEN 50 AND 200;
Method Syntax 
var result = products
  .Where(p => p.Price >= 50
    && p.Price <= 200)
  .ToList();
Query Syntax 
var result = (from p in products
              where p.Price >= 50
                && p.Price <= 200
              select p)
  .ToList();

Date ranges (extremely common):

SQL Date BETWEEN 
SELECT *
FROM Orders
WHERE OrderDate BETWEEN '2024-01-01'
  AND '2024-12-31';
C# Date Range 
var startDate = new DateTime(2024, 1, 1);
var endDate = new DateTime(2024, 12, 31);

var result = orders
  .Where(o => o.OrderDate >= startDate
    && o.OrderDate <= endDate)
  .ToList();

// Or using DateOnly (modern C#):
var result2 = orders
  .Where(o => o.OrderDate.Date >= startDate.Date
    && o.OrderDate.Date <= endDate.Date)
  .ToList();

NOT BETWEEN (outside a range):

SQL NOT BETWEEN 
SELECT *
FROM Products
WHERE Price NOT BETWEEN 50 AND 200;

-- Equivalent to:
WHERE Price < 50 OR Price > 200;
C# NOT BETWEEN 
// Direct approach (De Morgan's Law):
var result = products
  .Where(p => p.Price < 50 || p.Price > 200)
  .ToList();

// Or negate the range:
var result2 = products
  .Where(p => !(p.Price >= 50 && p.Price <= 200))
  .ToList();
BETWEEN Includes Both Boundaries

This is critical: BETWEEN is inclusive on both ends. Price BETWEEN 50 AND 200 includes both 50 and 200. In C#, use >= and <=, never > and <. An off-by-one error here can silently exclude edge cases you didn't notice in testing.

05

NULL Handling: Three-Valued Logic vs Two-Valued Logic

This is one of the most important sections: NULL semantics differ fundamentally between SQL and C#, and this difference causes subtle bugs that are maddening to debug. Understanding this is essential for writing correct queries.

SQL uses three-valued logic: true, false, and unknown. When you compare any value to NULL, the result is unknown—not false. C# uses two-valued logic: true or false. NULL in C# is a reference (object is null) or a nullable type value (int? is null). This difference means the same comparison can behave differently in SQL versus C#.

The infamous NULL = NULL problem:

SQL NULL Comparisons 
-- This returns 0 rows
SELECT * FROM Customers WHERE Phone = NULL;
-- Because NULL = NULL evaluates to unknown,
-- not true. Unknown rows are filtered out.

-- Correct way to find NULL values:
SELECT * FROM Customers WHERE Phone IS NULL;

-- And to exclude NULLs:
SELECT * FROM Customers WHERE Phone IS NOT NULL;
C# NULL Comparisons 
// C# == null checks for null correctly
var result = customers
  .Where(c => c.Phone == null)
  .ToList();

var result2 = customers
  .Where(c => c.Phone != null)
  .ToList();

// EF translates these to IS NULL / IS NOT NULL
// But complex logic with NULL is still tricky

Why this matters: complex boolean logic with NULLs behaves counterintuitively in SQL:

SQL Three-Valued Logic Problem 
-- This finds customers with specific phone
-- or no phone at all
SELECT * FROM Customers
WHERE Phone = '555-1234' OR Phone IS NULL;

-- Now, what does this return?
SELECT * FROM Customers
WHERE NOT (Phone = '555-1234');

-- WRONG EXPECTATION: All except 555-1234
-- REALITY: Only non-NULL values that aren't 555-1234
-- NULL rows are excluded!
-- Because NOT unknown = unknown (filtered out)

-- To include NULLs:
WHERE Phone != '555-1234' OR Phone IS NULL;
C# Two-Valued Logic 
// C# logic is simpler: null is null or not
var result = customers
  .Where(c => c.Phone == "555-1234" || c.Phone == null)
  .ToList();

// NOT works as you'd expect:
var result2 = customers
  .Where(c => c.Phone != "555-1234")
  .ToList();

// But EF translates C# to SQL with its
// three-valued semantics, so edge cases
// with NULL can still surprise you

Using COALESCE and ?? to handle NULL safely:

SQL COALESCE 
-- Convert NULL to a known value
SELECT *
FROM Customers
-- Treat NULL phone as "Unknown"
WHERE COALESCE(Phone, 'Unknown') = '555-1234'
  OR COALESCE(Phone, 'Unknown') = 'Unknown';

-- ISNULL (SQL Server):
WHERE ISNULL(Phone, 'Unknown') != '';
C# Null Coalescing (??) 
// Use ?? to provide defaults
var result = customers
  .Where(c => (c.Phone ?? "Unknown") == "555-1234"
    || (c.Phone ?? "Unknown") == "Unknown")
  .ToList();

// EF translates ?? to COALESCE in SQL
// This makes NULL behavior predictable
Always Test NULL Cases Explicitly

Three-valued logic is counterintuitive. If your WHERE clause uses NOT or complex boolean combinations with nullable columns, write specific tests with NULL values. Use .ToQueryString() to verify the generated SQL. When uncertain, use COALESCE (SQL) or ?? (C#) to convert NULL to a known value first, eliminating three-valued logic.

LIKE Patterns → String Methods

SQL's LIKE operator matches string patterns using wildcard characters. LINQ has no LIKE operator; instead, use string methods (.Contains(), .StartsWith(), .EndsWith()). Entity Framework translates these to LIKE patterns in the generated SQL, making them equivalent. For complex patterns not easily expressed with these methods, use EF.Functions.Like() for direct control.

SQL LIKE 
-- Contains 'smith'
SELECT * FROM Customers
WHERE Name LIKE '%smith%';

-- Starts with 'John'
SELECT * FROM Customers
WHERE Name LIKE 'John%';

-- Ends with 'son'
SELECT * FROM Customers
WHERE Name LIKE '%son';
Method Syntax 
// Contains 'smith'
var r1 = customers
  .Where(c => c.Name.Contains("smith"))
  .ToList();

// Starts with 'John'
var r2 = customers
  .Where(c => c.Name.StartsWith("John"))
  .ToList();

// Ends with 'son'
var r3 = customers
  .Where(c => c.Name.EndsWith("son"))
  .ToList();
Query Syntax 
// Contains
var r1 = (from c in customers
          where c.Name.Contains("smith")
          select c).ToList();

// Starts with
var r2 = (from c in customers
          where c.Name.StartsWith("John")
          select c).ToList();

// Ends with
var r3 = (from c in customers
          where c.Name.EndsWith("son")
          select c).ToList();

Case sensitivity varies by database:

SQL Case Sensitivity 
-- SQL Server: case-insensitive by default
SELECT * FROM Customers WHERE Name LIKE 'john%';
-- Matches 'John', 'john', 'JOHN'

-- PostgreSQL: case-sensitive by default
SELECT * FROM Customers WHERE Name LIKE 'john%';
-- Only matches exact case 'john'

-- PostgreSQL case-insensitive:
SELECT * FROM Customers WHERE Name ILIKE 'john%';
C# Case Handling 
// C# methods are case-sensitive
var result = customers
  .Where(c => c.Name.StartsWith("John"))
  .ToList();

// For case-insensitive, use StringComparison
var result2 = customers
  .Where(c => c.Name.StartsWith("john",
    StringComparison.OrdinalIgnoreCase))
  .ToList();

// EF translates to COLLATE or UPPER/LOWER
// depending on the database

Complex patterns with EF.Functions.Like():

Complex Patterns with EF.Functions 
using Microsoft.EntityFrameworkCore;

// Pattern: contains 'a' then 'b'
var result = context.Customers
  .Where(c => EF.Functions.Like(c.Name, "%a%b%"))
  .ToList();

// Pattern: starts with 'J', ends with 'n'
var result2 = context.Customers
  .Where(c => EF.Functions.Like(c.Name, "J%n"))
  .ToList();

// Escape special characters if needed:
var result3 = context.Customers
  .Where(c => EF.Functions.Like(c.Name, "%@%%", "@"))
  // This finds names containing literal @ or %
  .ToList();
Index Usage and Performance

LIKE patterns beginning with % (like %smith) cannot use an index on the column. The database must perform a full table scan. Patterns without leading % (like smith%) can use an index efficiently. For full-text search on large text columns, use full-text indexes instead of LIKE. Use LIKE only for short strings or when you can start the pattern without a wildcard.

07

EXISTS → .Any() and ALL → .All()

SQL's EXISTS checks if a subquery returns any rows. ALL checks if a condition is true for all rows returned by a subquery. LINQ provides .Any() and .All() methods on sequences, making these operations cleaner and more intuitive in C# than in SQL. Both are powerful for filtering based on related data without needing to fetch and compare explicitly.

SQL EXISTS 
SELECT *
FROM Customers c
WHERE EXISTS (
  SELECT 1 FROM Orders o
  WHERE o.CustomerId = c.Id
);
Method Syntax 
var result = customers
  .Where(c => c.Orders.Any())
  .ToList();

// Read: "customers that have at least one order"
Query Syntax 
var result = (from c in customers
              where c.Orders.Any()
              select c)
  .ToList();

EXISTS with conditions (customers with recent orders):

SQL EXISTS with Condition 
SELECT *
FROM Customers c
WHERE EXISTS (
  SELECT 1 FROM Orders o
  WHERE o.CustomerId = c.Id
    AND o.OrderDate >= '2024-01-01'
);
C# .Any() with Predicate 
var result = customers
  .Where(c => c.Orders.Any(o =>
    o.OrderDate >= new DateTime(2024, 1, 1)))
  .ToList();

// Read: "customers with at least one recent order"

NOT EXISTS (finding records with no related data):

SQL NOT EXISTS 
SELECT *
FROM Customers c
WHERE NOT EXISTS (
  SELECT 1 FROM Orders o
  WHERE o.CustomerId = c.Id
);
C# !.Any() 
var result = customers
  .Where(c => !c.Orders.Any())
  .ToList();

// Read: "customers with no orders"

ALL (verify condition on all related rows):

SQL ALL 
-- Find customers where ALL orders exceed $100
SELECT *
FROM Customers c
WHERE 100 < ALL (
  SELECT Total FROM Orders o
  WHERE o.CustomerId = c.Id
);
C# .All() 
var result = customers
  .Where(c => c.Orders.All(o => o.Total > 100))
  .ToList();

// Read: "customers where every order exceeds $100"
.Any() vs .All(): Know the Difference

.Any() can short-circuit: the database returns as soon as one matching row is found. .All() must check every related row. On navigation properties, EF generates SQL EXISTS for .Any() and NOT EXISTS for the negation. Use .Any() to check if a relation exists; use .All() to verify all related items satisfy a condition. Performance-wise, .Any() is faster when the collection is large.

08

Dynamic Query Building with Chained .Where()

A powerful LINQ feature is building queries dynamically by chaining multiple .Where() calls. Each .Where() applies an AND condition. This is essential for search forms where users select different filter criteria, applying only the filters they choose. The beauty is that LINQ doesn't execute the query until .ToList(), so all conditions are compiled into a single database round-trip.

Building queries incrementally:

Chained .Where() for Dynamic Filters 
// Start with base query (IQueryable, not executed yet)
IQueryable query = context.Products;

// Apply filters conditionally based on user input
if (!string.IsNullOrEmpty(searchTerm))
{
  query = query.Where(p => p.Name.Contains(searchTerm));
}

if (minPrice.HasValue)
{
  query = query.Where(p => p.Price >= minPrice);
}

if (maxPrice.HasValue)
{
  query = query.Where(p => p.Price <= maxPrice);
}

if (!string.IsNullOrEmpty(category))
{
  query = query.Where(p => p.Category == category);
}

// Execute query ONCE, after all conditions are set
var result = query.ToList();

// Generated SQL includes all WHERE clauses
// combined with AND operators, exactly as if
// you'd written them in a single WHERE clause

Real-world example: customer search form:

Real-World: Search Form Implementation 
public class CustomerSearchRequest
{
  public string Name { get; set; }
  public string Country { get; set; }
  public decimal? MinSpent { get; set; }
  public decimal? MaxSpent { get; set; }
}

public List SearchCustomers(
  CustomerSearchRequest request)
{
  // Start with base query
  var query = context.Customers.AsQueryable();

  // Apply name filter (searches name AND email)
  if (!string.IsNullOrWhiteSpace(request.Name))
  {
    query = query.Where(c =>
      c.Name.Contains(request.Name)
      || c.Email.Contains(request.Name));
  }

  // Apply country filter
  if (!string.IsNullOrWhiteSpace(request.Country))
  {
    query = query.Where(c => c.Country == request.Country);
  }

  // Apply spending filters
  if (request.MinSpent.HasValue)
  {
    query = query.Where(c =>
      c.Orders.Sum(o => o.Total ?? 0)
        >= request.MinSpent);
  }

  if (request.MaxSpent.HasValue)
  {
    query = query.Where(c =>
      c.Orders.Sum(o => o.Total ?? 0)
        <= request.MaxSpent);
  }

  // Project and execute
  return query
    .Select(c => new CustomerDto {
      Id = c.Id,
      Name = c.Name,
      Country = c.Country,
      TotalSpent = c.Orders.Sum(o => o.Total ?? 0)
    })
    .OrderBy(c => c.Name)
    .ToList();
}

For OR logic across multiple fields, build more carefully:

Dynamic OR Logic 
// Building OR conditions dynamically requires
// Expression composition or predicate builder
var statusList = new List { "Active", "Pending" };

// Simple: chain the OR inside a single Where
var result = context.Orders
  .Where(o =>
    (o.Status == "Active" || o.Status == "Pending")
    && o.OrderDate >= startDate)
  .ToList();

// For complex dynamic OR, consider using
// System.Linq.Dynamic.Core library:
// var dynamicQuery = orders.AsQueryable()
//   .Where("Status == @0 || Status == @1", "Active", "Pending")
//   .ToList();

// Or build predicates manually using Expression trees
// But this is complex; keep filters simple when possible
Chained .Where() = Multiple AND Conditions

Each .Where() adds an AND condition. This is perfect for search filters where users select multiple criteria independently. Keep queries as IQueryable (don't call .ToList()) until ready to execute. Entity Framework compiles all chained operations into a single SQL WHERE clause before hitting the database, so performance is equivalent to writing everything in one WHERE.

Key Takeaways

  • WHERE filtering must happen at the database level using .Where() before .ToList(). Fetching all rows and filtering in C# is catastrophically inefficient.
  • Comparison operators map directly: = to ==, <> to !=, < and >, <= and >=. All six operators work identically across SQL and C#.
  • Boolean logic uses AND (&&), OR (||), and NOT (!). AND binds tighter than OR. Always use parentheses to make logic explicit, even when precedence would be clear.
  • IN clause translates to .Contains() in C#. Both are syntactic sugar for multiple OR conditions but far more readable and performant.
  • BETWEEN has no direct LINQ equivalent; use >= and <= instead. Remember: BETWEEN is inclusive on both boundaries—a common source of off-by-one errors.
  • NULL semantics differ critically: SQL uses three-valued logic (true/false/unknown), C# uses two-valued. Use IS NULL / IS NOT NULL in SQL, == null / != null in C#. For complex logic, use COALESCE (??) to convert NULL to known values.
  • LIKE patterns map to .Contains(), .StartsWith(), .EndsWith(). Patterns without leading % can use indexes; leading % wildcards cannot. Use EF.Functions.Like() for complex patterns.
  • EXISTS becomes .Any(), ALL becomes .All(). Both are powerful for filtering on related data. Use .Any() to check existence (can short-circuit); use .All() to verify a condition on all related items.
  • Chain multiple .Where() calls to build dynamic queries. Each adds an AND condition. Keep as IQueryable until .ToList() so EF compiles everything into one SQL statement.
  • Always verify generated SQL with .ToQueryString() to ensure filters translate correctly and efficiently. Never assume—verify your expectations.
