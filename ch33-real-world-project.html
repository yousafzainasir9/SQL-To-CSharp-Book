01 — Project Overview: The E-Commerce Domain

Entities, Relationships, and Database Schema

Our e-commerce system manages customers, products organized by category, orders with line items, customer reviews, and shipping addresses. The domain is complex enough to showcase EF Core and Dapper patterns, but focused enough to understand completely. Let's start with the database schema.

Key Insight: Value Objects vs Entities

Money is a value object (immutable record) because it's defined by its values, not by an identity. Money(100) equals Money(100). Product and Order are entities because they have unique identities (Id). This distinction drives how you map to the database and how EF tracks changes.

03 — Write Side: EF Core for Commands

Creating Orders with Change Tracking and Transactions

The write side handles business operations. EF Core excels here because it tracks changes, manages relationships, and enforces domain constraints. A CreateOrderHandler demonstrates validation, change tracking, and transaction handling.

Create Order Command with EF Core using MediatR; using Ecommerce.Domain; using Ecommerce.Application.Repositories; namespace Ecommerce.Application.Commands { public class CreateOrderCommand : IRequest { public int CustomerId { get; set; } public int DeliveryAddressId { get; set; } public List Items { get; set; } } public class CreateOrderItemDto { public int ProductId { get; set; } public int Quantity { get; set; } } public class OrderResult { public int Id { get; set; } public string OrderNumber { get; set; } public decimal TotalAmount { get; set; } } // ════════════════════════════════════════ // HANDLER: Orchestrates Write Operations // ════════════════════════════════════════ public class CreateOrderHandler : IRequestHandler { private readonly IUnitOfWork _unitOfWork; private readonly ILogger _logger; public CreateOrderHandler( IUnitOfWork unitOfWork, ILogger logger) { _unitOfWork = unitOfWork; _logger = logger; } public async Task Handle( CreateOrderCommand request, CancellationToken cancellationToken) { try { // ───────────────────────────────── // STEP 1: Validate prerequisites // ───────────────────────────────── var customer = await _unitOfWork.Customers .GetByIdAsync(request.CustomerId); if (customer == null) throw new InvalidOperationException( $"Customer {request.CustomerId} not found"); if (customer.Status == "Suspended") throw new InvalidOperationException( "Customer account is suspended"); // Verify delivery address belongs to customer var address = customer.Addresses .FirstOrDefault(a => a.Id == request.DeliveryAddressId); if (address == null) throw new InvalidOperationException( "Invalid delivery address"); // ───────────────────────────────── // STEP 2: Validate and reserve inventory // ───────────────────────────────── var items = new List(); decimal totalAmount = 0; foreach (var itemRequest in request.Items) { var product = await _unitOfWork.Products .GetByIdAsync(itemRequest.ProductId); items.Add(item); totalAmount += item.Subtotal.Amount; } // ───────────────────────────────── // STEP 3: Create order // ───────────────────────────────── var order = new Order { CustomerId = customer.Id, OrderNumber = GenerateOrderNumber(), OrderDate = DateTime.UtcNow, DeliveryAddressId = address.Id, Status = "Pending", TotalAmount = new Money(totalAmount) }; // Add items to order // (EF tracks this relationship) foreach (var item in items) { order.Items.Add(item); } // ───────────────────────────────── // STEP 4: Reduce product stock // ───────────────────────────────── foreach (var item in items) { var product = await _unitOfWork.Products .GetByIdAsync(item.ProductId); product.ReduceStock(item.Quantity); _unitOfWork.Products.Update(product); } // ───────────────────────────────── // STEP 5: Persist with transaction // ───────────────────────────────── await _unitOfWork.BeginTransactionAsync(); try { await _unitOfWork.Orders.AddAsync(order); var rowsAffected = await _unitOfWork.SaveChangesAsync(); if (rowsAffected == 0) throw new InvalidOperationException( "Failed to save order"); await _unitOfWork.CommitAsync(); Key Insight: EF Change Tracking Manages Complexity

When you add items to order.Items, EF tracks that relationship. When you call SaveChangesAsync(), EF generates appropriate INSERT statements for Order, OrderItem, and UPDATE for Products. You don't write SQL; EF manages the transaction and relationships automatically.

04 — Read Side: Dapper for Queries

Complex Joins and Performance-Optimized Reads

The read side doesn't need change tracking. Dapper excels here because it's lightweight and you control the SQL. A GetOrdersSummaryHandler shows how to fetch orders with related data, project to DTOs, and handle pagination efficiently.

Dapper Query Handler with Multi-Mapping using System.Data; using Dapper; namespace Ecommerce.Infrastructure.Persistence.Queries { public class OrderSummaryDto { public int Id { get; set; } public string OrderNumber { get; 05 — Dependency Injection and Service Configuration

Registering Repositories, DbContext, and Handlers

Clean architecture requires proper dependency injection. The Composition Root (Program.cs) wires up all the components: DbContext, repositories, handlers, and business logic. This is where the magic of abstraction becomes reality.

Key Insight: Scoped Lifetime for DbContext

Register DbContext with AddScoped, not Singleton. Each HTTP request gets its own DbContext instance, ensuring isolated change tracking. If you used Singleton, concurrent requests would interfere with each other's change tracking.

Key Insight: Clean Request Routing

Notice how the controller is thin. GET requests delegate to Dapper handlers for speed. POST/PUT requests dispatch to MediatR handlers, which use EF Core's Unit of Work for complex business logic. This separation of concerns—reads via Dapper, writes via EF—is the essence of CQRS in practice.

Key Takeaways

  • The domain model (entities, value objects) is independent of data access tools; it encodes business logic and constraints
  • Clean architecture separates Domain, Application, Infrastructure, and API layers; each can be developed and tested independently
  • Repository pattern abstracts data access; interface definitions live in Application, implementations in Infrastructure
  • Unit of Work pattern ensures atomic transactions across multiple repositories; EF Core's DbContext provides built-in implementation
  • EF Core excels for writes: change tracking, relationships, transactions, and validation all work together seamlessly
  • Dapper excels for reads: you control the SQL, get lean objects, and can optimize complex queries with JOINs and aggregations
  • CQRS (Command Query Responsibility Segregation) maps naturally: EF Core for commands, Dapper for queries; they don't compete, they complement
  • MediatR provides a dispatch mechanism for CQRS; handlers encapsulate business logic and keep controllers thin and testable
