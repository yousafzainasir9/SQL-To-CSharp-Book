Unit of Work

Save multiple changes in one atomic transaction. All succeed or all fail together. DbContext IS Unit of Work — changes batch until SaveChangesAsync() commits them.

01

Requirement: "Atomic Multi-Entity Saves"

Business Requirement
Create order, add line items, decrement inventory — all in one transaction. If any step fails, roll back everything.
SQL Developer Thinks BEGIN TRANSACTION, execute statements, COMMIT or ROLLBACK. Atomic.
C# Developer Thinks Track changes with DbContext. SaveChangesAsync() commits one transaction. Failures roll back automatically.
SQL — Explicit Transaction
BEGIN TRANSACTION
INSERT INTO Orders (CustomerId, Total)
VALUES (@CustId, @Total)

INSERT INTO OrderItems (OrderId, ProductId, Qty)
VALUES (@OrderId, @ProdId, @Qty)

UPDATE Products
SET Stock = Stock - @Qty
WHERE Id = @ProdId

COMMIT TRANSACTION

02

Generic Repository (CRUD for Any Entity)

Business Requirement
Create, read, update, delete. Every entity needs these operations. Don't repeat the same code for each table.
SQL Developer Thinks Basic CRUD — SELECT, INSERT, UPDATE, DELETE. Same pattern applies to Orders, Products, Customers.
C# Developer Thinks Generic<T> handles any entity type. One implementation covers all. For specific business queries, create a specialized repository.
C# — EF Core Implementation
public interface IRepository<T> where T : class
{
    Task<T?> GetByIdAsync(int id, CancellationToken ct = default);
    Task<IEnumerable<T>> GetAllAsync(CancellationToken ct = default);
    Task AddAsync(T entity, CancellationToken ct = default);
    Task UpdateAsync(T entity, CancellationToken ct = default);
    Task RemoveAsync(int id, CancellationToken ct = default);
}

public class Repository<T> : IRepository<T> where T : class
{
    private readonly DbContext _db;

    public Repository(DbContext db) => _db = db;

    public async Task<T?> GetByIdAsync(int id, CancellationToken ct = default)
        => await _db.Set<T>().FindAsync(new object[] { id }, ct);

    public async Task<IEnumerable<T>> GetAllAsync(CancellationToken ct = default)
        => await _db.Set<T>().AsNoTracking().ToListAsync(ct);

    public async Task AddAsync(T entity, CancellationToken ct = default)
    {
        await _db.Set<T>().AddAsync(entity, ct);
        await _db.SaveChangesAsync(ct);
    }

    public async Task UpdateAsync(T entity, CancellationToken ct = default)
    {
        _db.Set<T>().Update(entity);
        await _db.SaveChangesAsync(ct);
    }

    public async Task RemoveAsync(int id, CancellationToken ct = default)
    {
        var entity = await GetByIdAsync(id, ct);
        if (entity != null)
        {
            _db.Set<T>().Remove(entity);
            await _db.SaveChangesAsync(ct);
        }
    }
}
03

Domain-Specific Repository (Encapsulate Business Logic)

Business Requirement
"Get active customers" is a business concept. Map it directly to a repository method, not a scattered LINQ query.
SQL Developer Thinks Stored procedures map to domain concepts. Instead of SELECT...WHERE...ORDER, you have usp_GetActiveCustomers.
C# Developer Thinks Repository methods map to business actions. GetActiveAsync(), GetByEmailAsync(), GetWithRecentOrdersAsync(). Clear intent.
C# — Domain-Focused Interface
public interface ICustomerRepository
{
    Task<Customer?> GetByIdAsync(int id, CancellationToken ct = default);
    Task<IEnumerable<Customer>> GetActiveAsync(CancellationToken ct = default);
    Task<Customer?> GetByEmailAsync(string email, CancellationToken ct = default);
    Task<IEnumerable<Customer>> GetWithBalanceAboveAsync(
        decimal minBalance, CancellationToken ct = default);
    Task AddAsync(Customer customer, CancellationToken ct = default);
    Task UpdateAsync(Customer customer, CancellationToken ct = default);
}

public class CustomerRepository : ICustomerRepository
{
    private readonly DbContext _db;

    public async Task<IEnumerable<Customer>> GetActiveAsync(CancellationToken ct = default)
        => await _db.Customers
            .AsNoTracking()
            .Where(c => c.IsActive)
            .OrderBy(c => c.Name)
            .ToListAsync(ct);

    public async Task<Customer?> GetByEmailAsync(string email, CancellationToken ct = default)
        => await _db.Customers
            .AsNoTracking()
            .FirstOrDefaultAsync(c => c.Email == email, ct);

    public async Task<IEnumerable<Customer>> GetWithBalanceAboveAsync(
        decimal minBalance, CancellationToken ct = default)
        => await _db.Customers
            .AsNoTracking()
            .Where(c => c.Balance >= minBalance)
            .OrderByDescending(c => c.Balance)
            .ToListAsync(ct);
}
04

Pagination (Handle Large Datasets)

Business Requirement
Return all 50,000 customers at once? Never. Implement pagination to fetch 10 at a time, with total count for UI.
SQL Developer Thinks Use OFFSET and FETCH. SELECT * FROM Customers ORDER BY Name OFFSET 0 ROWS FETCH NEXT 10 ROWS ONLY.
C# Developer Thinks Skip() and Take(). Also count the total to show "Page 2 of 500" in the UI.
C# — Paged Query
public class PagedResult<T>
{
    public IEnumerable<T> Items { get; set; } = new List<T>();
    public int PageNumber { get; set; }
    public int PageSize { get; set; }
    public int TotalCount { get; set; }
    public int TotalPages => (TotalCount + PageSize - 1) / PageSize;
}

public class CustomerRepository : ICustomerRepository
{
    public async Task<PagedResult<Customer>> GetPagedAsync(
        int pageNumber, int pageSize, CancellationToken ct = default)
    {
        var query = _db.Customers.AsNoTracking().Where(c => c.IsActive);
        var totalCount = await query.CountAsync(ct);

        var items = await query
            .OrderBy(c => c.Name)
            .Skip((pageNumber - 1) * pageSize)
            .Take(pageSize)
            .ToListAsync(ct);

        return new PagedResult<Customer>
        {
            Items = items,
            PageNumber = pageNumber,
            PageSize = pageSize,
            TotalCount = totalCount
        };
    }
}
05

Testing (Mock Repositories)

Business Requirement
Unit tests should not hit the database. Swap in a mock repository that returns test data.
SQL Developer Thinks Rollback transactions to avoid polluting test databases. Or use separate test data.
C# Developer Thinks Interface allows dependency injection. Inject a fake repository in tests; inject the real one in production.
C# — Mock Repository for Testing
public class MockCustomerRepository : ICustomerRepository
{
    private readonly List<Customer> _customers = new();

    public async Task<IEnumerable<Customer>> GetActiveAsync(CancellationToken ct = default)
    {
        await Task.Delay(0, ct); // Simulate async
        return _customers.Where(c => c.IsActive).OrderBy(c => c.Name);
    }

    public async Task AddAsync(Customer customer, CancellationToken ct = default)
    {
        _customers.Add(customer);
        await Task.CompletedTask;
    }
}

// Unit test
[Fact]
public async Task GetActive_ReturnsOnlyActiveCustomers()
{
    var repo = new MockCustomerRepository();
    await repo.AddAsync(new Customer { Id = 1, Name = "Alice", IsActive = true });
    await repo.AddAsync(new Customer { Id = 2, Name = "Bob", IsActive = false });

    var result = await repo.GetActiveAsync();

    Assert.Single(result);
    Assert.Equal("Alice", result.First().Name);
}
