01

Unit of Work: From SQL to C#

Unit of Work is an architectural pattern that groups multiple database operations into a single atomic unit—either all operations succeed, or all fail and are rolled back. The concept originated in Martin Fowler's work on enterprise application architecture, but its roots go much deeper: to SQL transactions themselves.

Think of a Unit of Work as a boundary around a set of operations that must succeed together. In e-commerce, when a customer places an order, you might:

  • Insert a new Order record
  • Insert multiple OrderItem records
  • Update the Inventory table
  • Insert a Payment record

If any of these steps fails, the entire operation must roll back. You cannot have an order without items, or a deducted inventory without a corresponding order. That's the Unit of Work.

Fowler's Definition
Martin Fowler defines Unit of Work as: "Maintains a list of objects affected by a business transaction and coordinates the writing of changes out to the database and the resolution of concurrency problems."

In practice, this means:

  • Track: The UoW knows about all objects that have been created, modified, or deleted
  • Defer writes: Changes are held in memory until explicitly committed
  • Commit atomically: All changes are written to the database in a single transaction
  • Rollback on failure: If anything goes wrong, the entire transaction rolls back
SQL: BEGIN TRANSACTION
BEGIN TRANSACTION; INSERT INTO Orders (CustomerId, OrderDate, Total) VALUES (42, GETDATE(), 99.99); DECLARE @OrderId INT = SCOPE_IDENTITY(); INSERT INTO OrderItems (OrderId, ProductId, Qty) VALUES (@OrderId, 1, 2); UPDATE Products SET StockLevel = StockLevel - 2 WHERE ProductId = 1; -- If all statements succeed: COMMIT TRANSACTION; -- If any statement fails: -- (automatically rolled back)
C#: DbContext as UoW
using var context = new OrderDbContext(); // Begin implicit transaction var order = new Order { CustomerId = 42, OrderDate = DateTime.UtcNow, Total = 99.99m }; context.Orders.Add(order); order.Items.Add(new OrderItem { ProductId = 1, Quantity = 2 }); var product = await context.Products .FindAsync(1); product.StockLevel -= 2; // Atomic commit await context.SaveChangesAsync(); // If SaveChanges fails: // (transaction rolls back automatically)
Key Insight
DbContext IS a Unit of Work. When you call SaveChangesAsync(), it wraps all your changes in a database transaction. You don't need to create a separate abstraction unless you're coordinating multiple repositories or data access layers.
02

ACID Properties: SQL to C#

ACID is the database guarantee that transactions are reliable. Understanding ACID from the SQL perspective helps you understand why Unit of Work matters in C#.

ACID Explained
Atomicity: All-or-nothing. Either all operations in the transaction commit, or all rollback. No partial updates.

Consistency: The database moves from one valid state to another. Constraints, triggers, and application logic maintain data integrity.

Isolation: Concurrent transactions don't interfere with each other. SQL defines isolation levels (READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE).

Durability: Once committed, data is permanent, even in case of system failure.

In C#, DbContext respects these guarantees:

SQL: Isolation Levels
-- Dirty read allowed SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED; BEGIN TRANSACTION; SELECT * FROM Accounts; COMMIT; -- Default: prevents dirty reads SET TRANSACTION ISOLATION LEVEL READ COMMITTED; -- Strict consistency SET TRANSACTION ISOLATION LEVEL SERIALIZABLE; -- Slowest but safest
C#: Controlling Isolation
// Explicit isolation level using var tx = await context .Database .BeginTransactionAsync( IsolationLevel .Serializable); try { context.Accounts.Add(acct); await context .SaveChangesAsync(); await tx.CommitAsync(); } catch { await tx.RollbackAsync(); throw; }
Isolation Level Trade-off
Higher isolation (SERIALIZABLE) provides stronger guarantees but reduces concurrency. READ COMMITTED is the SQL Server default and balances safety with performance.
03

DbContext as Built-In Unit of Work

Entity Framework Core's DbContext IS a Unit of Work. It tracks entity changes and coordinates a single SaveChanges() call that commits everything atomically. You don't need to create a separate abstraction for simple scenarios.

When you add, modify, or delete entities, DbContext holds these changes in memory. Only when you call SaveChanges() or SaveChangesAsync() does it execute a SQL transaction.

C#: DbContext Tracking
using var ctx = new OrderDbContext(); // Track: Create var order = new Order { Id = 1 }; ctx.Orders.Add(order); // Track: Modify order.Total = 50m; // Track: Delete var old = await ctx .Orders.FindAsync(999); if (old != null) ctx.Orders.Remove(old); // Atomic commit await ctx .SaveChangesAsync();
SQL: What EF Executes
BEGIN TRANSACTION; INSERT INTO Orders (Id, Total) VALUES (1, 50.00); DELETE FROM Orders WHERE Id = 999; COMMIT TRANSACTION; -- Single atomic batch
C#: Inspecting DbContext Changes
// See all tracked entries var entries = context.ChangeTracker .Entries(); foreach (var entry in entries) { Console.WriteLine( $"Entity: {entry.Entity.GetType()}"); Console.WriteLine( $"State: {entry.State}"); } // Get only changes var changes = context.ChangeTracker .Entries() .Where(e => e.State != EntityState .Unchanged) .ToList();
When DbContext Alone Is Enough
For most applications, DbContext's built-in Unit of Work behavior is sufficient. You only need a custom IUnitOfWork abstraction when coordinating multiple DbContexts, mixing EF with Dapper, or enforcing explicit boundaries in a strict architectural style.

Custom IUnitOfWork Interface Design

When you need explicit UoW boundaries—especially in larger systems—you can create a custom IUnitOfWork interface that wraps DbContexts and repositories. This clarifies intent and makes dependency injection easier.

C#: Custom IUnitOfWork Interface
public interface IUnitOfWork { IOrderRepository Orders { get; } ICustomerRepository Customers { get; } IProductRepository Products { get; } async Task SaveChangesAsync( CancellationToken ct = default); async Task<IUnitOfWorkTransaction> BeginTransactionAsync( IsolationLevel level = IsolationLevel .ReadCommitted); void Rollback(); } public interface IUnitOfWorkTransaction { async Task CommitAsync(); async Task RollbackAsync(); }
C#: Custom IUnitOfWork Implementation
public class EfCoreUnitOfWork : IUnitOfWork { private readonly OrderDbContext _context; private readonly Lazy<IOrderRepository> _orders; public EfCoreUnitOfWork(OrderDbContext ctx) { _context = ctx; _orders = new Lazy<IOrderRepository>( () => new OrderRepository(_context)); } public IOrderRepository Orders => _orders.Value; public async Task SaveChangesAsync( CancellationToken ct = default) { await _context.SaveChangesAsync(ct); } public async Task<IUnitOfWorkTransaction> BeginTransactionAsync( IsolationLevel level = IsolationLevel.ReadCommitted) { var tx = await _context.Database .BeginTransactionAsync(level); return new UoWTransaction(tx, _context); } public void Rollback() { _context.ChangeTracker.Clear(); } } public class UoWTransaction : IUnitOfWorkTransaction { private readonly IDbContextTransaction _tx; private readonly DbContext _context; public async Task CommitAsync() { try { await _context.SaveChangesAsync(); await _tx.CommitAsync(); } catch { await RollbackAsync(); throw; } } public async Task RollbackAsync() { await _tx.RollbackAsync(); } }
C#: Using Custom IUnitOfWork
public class OrderService { private readonly IUnitOfWork _uow; public async Task<Order> CreateOrderAsync( CreateOrderRequest request) { var order = new Order { CustomerId = request.CustomerId, Total = request.Total }; _uow.Orders.Add(order); foreach (var item in request.Items) { order.Items.Add(new OrderItem { ProductId = item.ProductId }); } await _uow.SaveChangesAsync(); return order; } }
Lazy Loading Repositories
Notice the Lazy<T> pattern used in the custom UoW implementation. This defers repository creation until first use, reducing unnecessary instantiation. This is useful when your system has many repositories but only uses a few per transaction.
05

LINQ Query and Method Syntax

Within a Unit of Work, you use LINQ to query entities. LINQ supports both method syntax and query syntax. Understanding both helps you write flexible, efficient code.

Query syntax is more SQL-like and clearer for complex scenarios. Method syntax is more functional and flexible for advanced operators.

C#: LINQ Query Syntax
// Query syntax (more SQL-like) var highValue = from order in uow.Orders.GetAll() where order.Total > 1000 orderby order.Total descending select new { order.Id, order.Total }; var results = await highValue .ToListAsync(); // With joins var joined = from order in uow.Orders.GetAll() join customer in uow.Customers.GetAll() on order.CustomerId equals customer.Id where customer.Country == "USA" select new { order.Id, customer.Name };
C#: LINQ Method Syntax
// Method syntax (more functional) var highValue = uow.Orders .GetAll() .Where(o => o.Total > 1000) .OrderByDescending(o => o.Total) .Select(o => new { o.Id, o.Total }); var results = await highValue .ToListAsync(); // Joins (more flexible) var joined = uow.Orders .GetAll() .Join( uow.Customers.GetAll(), o => o.CustomerId, c => c.Id, (o, c) => new { o, c }) .Where(x => x.c.Country == "USA") .Select(x => new { x.o.Id, x.c.Name }); // GroupBy var grouped = uow.Orders .GetAll() .GroupBy(o => o.CustomerId) .Select(g => new { CustomerId = g.Key, Count = g.Count(), Total = g.Sum(o => o.Total) });
When to Use Each
Use query syntax for complex scenarios with multiple joins and where clauses. Use method syntax for advanced operators and when chaining many filters. Both compile to identical IL, so performance is the same.
06

Savepoints in SQL vs DbContext

A savepoint is a marker within a transaction that lets you rollback to that point without rolling back the entire transaction. This is useful for partial rollback.

SQL supports savepoints directly. DbContext has support through EF Core 6+.

SQL: SAVEPOINT
BEGIN TRANSACTION; INSERT INTO Orders (CustomerId) VALUES (1); -- Mark savepoint SAVE TRANSACTION sp1; INSERT INTO OrderItems (OrderId) VALUES (1); -- Rollback only to sp1 ROLLBACK TRANSACTION sp1; -- OrderItems undone, Order remains COMMIT TRANSACTION;
C#: EF Core 6+ Savepoint
using (var tx = await context .Database .BeginTransactionAsync()) { try { var order = new Order(); context.Orders.Add(order); await context .SaveChangesAsync(); // Savepoint await tx .CreateSavepointAsync( "sp1"); var item = new OrderItem(); context.OrderItems.Add(item); await context .SaveChangesAsync(); await tx.CommitAsync(); } catch { await tx .RollbackToSavepointAsync( "sp1"); await tx.CommitAsync(); } }
Savepoint Limitations
Savepoints add complexity but are powerful for partial rollback. Most applications don't need them. Use savepoints when you have multiple independent operations within one UoW and want to recover from failure in one without losing all work.
07

Unit of Work with Dapper

Unlike Entity Framework, Dapper doesn't provide a built-in Unit of Work. You manage transactions explicitly through ADO.NET's IDbTransaction. This gives you more control but requires more code.

In Dapper, you create an IDbConnection, begin a transaction, pass it to Execute/Query methods, and manage commit/rollback yourself.

C#: Basic Dapper Transaction
public async Task<Order> CreateOrderAsync( Order order, List<OrderItem> items) { using (var conn = new SqlConnection( _connStr)) { await conn.OpenAsync(); using (var tx = conn .BeginTransaction()) { try { // Insert order var orderId = await conn.ExecuteAsync( @"INSERT INTO Orders (CustomerId, Total) VALUES (@CustomerId, @Total); SELECT SCOPE_IDENTITY();", order, tx); order.Id = (int)orderId; // Insert items foreach (var item in items) { item.OrderId = order.Id; await conn .ExecuteAsync( @"INSERT INTO OrderItems (OrderId, ProductId, Qty) VALUES (@OrderId, @ProductId, @Qty);", item, tx); } // Commit tx.Commit(); return order; } catch { tx.Rollback(); throw; } } } }
SQL: What Dapper Executes
BEGIN TRANSACTION; INSERT INTO Orders (CustomerId, Total) VALUES (42, 99.99); DECLARE @OrderId INT = SCOPE_IDENTITY(); INSERT INTO OrderItems (OrderId, ProductId, Qty) VALUES (@OrderId, 1, 2); INSERT INTO OrderItems (OrderId, ProductId, Qty) VALUES (@OrderId, 3, 1); COMMIT TRANSACTION;
Dapper vs DbContext
With Dapper, you have explicit control over every SQL statement and transaction. DbContext abstracts these details. For performance-critical code with precise SQL control, Dapper is superior. For rapid development, DbContext's UoW is simpler. Many applications use both: DbContext for general CRUD, Dapper for complex queries.
08

TransactionScope for Cross-Connection Transactions

When you need to coordinate transactions across multiple database connections—or mix DbContext with Dapper—use TransactionScope. It's an ambient transaction API that automatically enlists database operations.

TransactionScope is part of System.Transactions and coordinates MSDTC (Distributed Transaction Coordinator) for transactions spanning multiple servers.

C#: TransactionScope Basics
using (var scope = new TransactionScope( TransactionScopeAsyncFlowOption .Enabled)) { try { // EF Core operation var order = new Order { Total = 100 }; efContext.Orders.Add(order); await efContext .SaveChangesAsync(); // Dapper operation // in SAME transaction await dapperConn .ExecuteAsync( @"UPDATE Inventory SET StockLevel -= 1 WHERE ProductId = 1"); // Both commit together scope.Complete(); } catch { // Both rollback together throw; } }
C#: TransactionScope Options
// Fine-grained control var options = new TransactionOptions { // Isolation level IsolationLevel = IsolationLevel .Serializable, // Timeout (5 seconds) Timeout = TimeSpan .FromSeconds(5) }; using (var scope = new TransactionScope( TransactionScopeOption .RequiresNew, options, TransactionScopeAsyncFlowOption .Enabled)) { // Your operations scope.Complete(); }

Important: TransactionScope with distributed transactions has overhead. Use it only when necessary:

  • Same database: DbContext's implicit transaction is sufficient
  • Multiple connections to same server: TransactionScope can optimize to a local transaction
  • Multiple servers: TransactionScope uses MSDTC (significant overhead)
  • Microservices: Use the Saga pattern instead
Ambient Transaction Context
TransactionScope creates an ambient transaction context. Any connection opened within the using block automatically joins the transaction—you don't explicitly pass the transaction. This is convenient but can hide transaction semantics in deep call stacks.

Error Handling and Rollback Patterns

Proper error handling in a Unit of Work ensures that failed operations are safely rolled back and that retry logic can be implemented. There are several patterns for handling errors.

C#: Comprehensive Error Handling
public async Task<Result<Order>> CreateOrderSafelyAsync( CreateOrderRequest request) { using (var ctx = new OrderDbContext()) { var retries = 0; const int maxRetries = 3; while (retries < maxRetries) { try { var order = new Order { CustomerId = request.CustomerId, Total = request.Total }; ctx.Orders.Add(order); foreach (var item in request.Items) { order.Items.Add(new OrderItem { ProductId = item.ProductId, Quantity = item.Qty }); } await ctx .SaveChangesAsync(); return Result<Order> .Success(order); } catch ( DbUpdateConcurrencyException) { // Retry on concurrency ctx.ChangeTracker.Clear(); retries++; } catch ( DbUpdateException ex) { // Constraint violation ctx.ChangeTracker.Clear(); return Result<Order> .Failure( "Invalid data"); } } return Result<Order> .Failure("Max retries exceeded"); } }
C#: Compensating Transactions
public class OrderWorkflow { public async Task<bool> ProcessOrderAsync(Order order) { var orderId = -1; var paymentId = -1; try { // Step 1 orderId = await _orderService .CreateAsync(order); // Step 2 paymentId = await _paymentService .ChargeAsync( order.CustomerId, order.Total); // Step 3 await _emailService .SendAsync( order.CustomerId); return true; } catch { // Undo in reverse order if (paymentId > 0) await _paymentService .RefundAsync(paymentId); if (orderId > 0) await _orderService .CancelAsync(orderId); return false; } } }
Idempotency in Compensation
When implementing compensating transactions, ensure each compensation step is idempotent (safe to call multiple times). Use status flags to avoid double-refunds or duplicate cancellations.

Unit of Work Anti-Patterns

Understanding what NOT to do is as important as knowing best practices. Here are common Unit of Work anti-patterns and why they fail.

C#: Anti-Pattern 1 — Nested Units of Work
// WRONG: Nested UoW public async Task CreateAsync() { using (var uow1 = new EfCoreUnitOfWork(ctx1)) { var order = new Order(); uow1.Orders.Add(order); // Nested: another UoW // Both commit independently! using (var uow2 = new EfCoreUnitOfWork(ctx2)) { var item = new OrderItem(); uow2.Items.Add(item); await uow2 .SaveChangesAsync(); // Inner commits! } await uow1 .SaveChangesAsync(); // If this fails, inner // already committed! } } // RIGHT: Single UoW public async Task CreateAsync() { using (var uow = new EfCoreUnitOfWork(ctx)) { var order = new Order(); uow.Orders.Add(order); var item = new OrderItem(); uow.Items.Add(item); await uow .SaveChangesAsync(); // Single atomic commit } }
C#: Anti-Pattern 2 — Long-Running Transactions
// WRONG: UoW spans entire request public async Task<IActionResult> CreateAsync() { // Transaction begins here var order = new Order(); _uow.Orders.Add(order); // User-visible delay: Send email await _emailService .SendAsync(order.Id); // Transaction still open! // Locks held! await _uow .SaveChangesAsync(); return Ok(order); } // RIGHT: Short transaction public async Task<IActionResult> CreateAsync() { int orderId; using (var uow = new EfCoreUnitOfWork(ctx)) { // Minimal transaction var order = new Order(); uow.Orders.Add(order); await uow .SaveChangesAsync(); orderId = order.Id; } // Outside UoW: Send email await _emailService .SendAsync(orderId); return Ok(new { orderId }); }
Request Scope vs Transaction Scope
In web applications, it's common to inject DbContext/IUnitOfWork per HTTP request. But transactions should be explicit and short-lived. The HTTP request is a business boundary, not a transaction boundary. A transaction should be the duration of a single atomic operation.

Async Patterns and Cancellation

Modern Unit of Work implementations must support async/await and cancellation tokens. Cancellation tokens allow graceful shutdown and timeouts.

C#: Async UoW with Cancellation
public async Task CreateOrderAsync( CreateOrderRequest request, CancellationToken ct) { using (var uow = new EfCoreUnitOfWork(ctx)) { var order = new Order { CustomerId = request.CustomerId, Total = request.Total }; uow.Orders.Add(order); // Pass token through await uow .SaveChangesAsync(ct); return order; } } // In controller: public async Task<IActionResult> CreateAsync() { var ct = HttpContext .RequestAborted; try { var order = await _service .CreateOrderAsync( request, ct); return Ok(order); } catch ( OperationCanceledException) { // Client disconnected return StatusCode(499); } }
C#: Timeout Pattern
public async Task<Order> CreateWithTimeoutAsync( CreateOrderRequest request) { // 5-second timeout using (var cts = new CancellationTokenSource( TimeSpan .FromSeconds(5))) { try { return await _service .CreateOrderAsync( request, cts.Token); } catch ( OperationCanceledException) { throw new TimeoutException( "Timeout"); } } }
Async Best Practices
Always pass CancellationToken through your async call chains. Use ConfigureAwait(false) in library code. DbContext is not thread-safe; each async context should have its own instance.

Distributed Systems: Saga Pattern

In microservices, you can't use traditional ACID transactions. The Saga pattern replaces distributed transactions with orchestrated sequences of local transactions and compensating actions.

A saga is orchestration-based (a central orchestrator directs steps) or choreography-based (services emit events and react).

C#: Saga Orchestration
public class OrderSaga { private readonly IOrderService _orders; private readonly IInventoryService _inventory; private readonly IPaymentService _payment; public async Task<bool> ExecuteAsync(OrderRequest req) { var resId = null as string; var payId = null as string; var ordId = -1; try { // Step 1 resId = await _inventory.ReserveAsync( req.Items); // Step 2 payId = await _payment.ChargeAsync( req.CustomerId, req.Total); // Step 3 ordId = await _orders.CreateAsync(req); return true; } catch { // Undo in reverse if (ordId > 0) await _orders .CancelAsync(ordId); if (payId != null) await _payment .RefundAsync(payId); if (resId != null) await _inventory .ReleaseAsync(resId); return false; } } }
C#: Saga Choreography
// Order service publishes public class OrderService { public async Task CreateAsync( OrderRequest req) { var order = new Order(); await _db.SaveAsync(order); // Emit event await _bus.PublishAsync( new OrderCreatedEvent { OrderId = order.Id, Items = req.Items, Total = req.Total }); } } // Inventory subscribes public class InventoryService { public async Task OnOrderCreated( OrderCreatedEvent evt) { try { await ReserveAsync( evt.Items); // Emit success await _bus .PublishAsync( new InventoryReservedEvent { OrderId = evt.OrderId }); } catch { // Emit failure await _bus .PublishAsync( new OrderFailedEvent { OrderId = evt.OrderId }); } } }
Saga vs Unit of Work
Unit of Work is for single-database consistency (ACID). Saga is for cross-service consistency (eventual). In a saga, if Step 2 fails, you explicitly run compensating transactions to undo Step 1. This is slower and more complex than a simple database transaction, but necessary in distributed systems where you can't use traditional locks.

Key Takeaways

  • Unit of Work groups operations into atomic units—either all succeed or all fail.
  • DbContext IS a Unit of Work—it tracks changes and commits them atomically via SaveChangesAsync().
  • ACID properties (Atomicity, Consistency, Isolation, Durability) originate in SQL and are enforced by DbContext.
  • Custom IUnitOfWork abstractions are useful for coordinating multiple repositories or enforcing explicit boundaries.
  • Savepoints enable partial rollback within a transaction.
  • Dapper requires manual transaction management via IDbConnection and IDbTransaction.
  • TransactionScope coordinates transactions across multiple connections (EF Core + Dapper, or multiple databases).
  • Keep transaction boundaries short—don't hold locks during I/O operations.
  • In web applications, use scoped DbContext lifetime (one instance per HTTP request).
  • Proper error handling includes retry logic and compensating transactions.
  • In microservices, replace distributed transactions with the Saga pattern (orchestration or choreography).
  • Anti-patterns: nested UoW, long-running transactions, mixing explicit and implicit transactions.
  • Always pass CancellationToken through async call chains for graceful shutdown and timeout support.
