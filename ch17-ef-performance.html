Chapter 17

EF Performance

Requirement: “The page loads in 5 seconds — make it faster.” EF Core can be fast or slow depending on how you use it. Here are the common performance pitfalls and how to fix them.

01

Requirement: “Customer List Makes 100 Database Calls”

Business Requirement
Page shows 100 customers with order counts. Takes 5 seconds. Each customer triggers a separate query.
SQL Developer Thinks One query with JOIN and COUNT gets everything in a single round-trip. The database is made for this.
C# Developer Thinks Use Include() to load related data in one query, or Select() projection to get exactly what I need. One query instead of N+1.
Bad Code — N+1 Problem (100+ queries)
var customers = await db.Customers.ToListAsync(ct);

foreach (var c in customers)
{
    // This runs 100 times! One query per customer
    c.OrderCount = await db.Orders
        .CountAsync(o => o.CustomerId == c.Id, ct);
}

// Result: 1 query to load customers + 100 queries for counts
// Total: 101 queries, 5 seconds
Fixed with Include()
var customers = await db.Customers
    .Include(c => c.Orders)
    .AsNoTracking()
    .ToListAsync(ct);

// Loads customers + all related orders
// Then count in memory
var result = customers.Select(c => new
{
    c.Id,
    c.Name,
    OrderCount = c.Orders.Count
});

// Result: 1 query + join
// Pros: Simple, eager loading
// Cons: Loads all order data (wasteful)
Fixed with Select() Projection
var customers = await db.Customers
    .AsNoTracking()
    .Select(c => new
    {
        c.Id,
        c.Name,
        OrderCount = c.Orders.Count()
    })
    .ToListAsync(ct);

// EF generates subquery:
// SELECT c.Id, c.Name,
//    (SELECT COUNT(*) FROM Orders
//     WHERE CustomerId = c.Id)
// FROM Customers

// Result: 1 query, perfect shape
// Pros: Minimal data, single query
// Cons: Subquery pattern
02

Requirement: “Insert 10,000 Records Faster”

Business Requirement
CSV import needs to create 10,000 products. Individual SaveChanges is too slow.
SQL Developer Thinks BULK INSERT or INSERT with multiple VALUE rows. One statement, one round-trip, minimal logging.
C# Developer Thinks AddRange() batches adds in memory, then SaveChanges() once. Or EF.BulkExtensions for BULK INSERT speed.
Slow Code — One SaveChanges Per Record
var products = LoadProductsFromCsv("data.csv");

foreach (var p in products)
{
    db.Products.Add(p);
    await db.SaveChangesAsync(ct); // Saves 1 record
}

// Result: 10,000 database round-trips
// Each INSERT logged, flushed, committed
// Time: 30+ seconds
Fast — AddRange + SaveChanges
var products = LoadProductsFromCsv("data.csv");

// Add all at once (in memory)
db.Products.AddRange(products);

// SaveChanges batches them
// Default: ~1000 per batch
await db.SaveChangesAsync(ct);

// EF generates:
// INSERT INTO Products (...)
//   VALUES (...), (...), ...

// Result: 10 batches, 10 round-trips
// Time: 2 seconds
Fastest — EF.BulkExtensions
var products = LoadProductsFromCsv("data.csv");

// NuGet: EFCore.BulkExtensions
await db.BulkInsertAsync(products, ct);

// Uses native BULK INSERT
// Or: INSERT with TABLOCK for speed

// Result: 1 statement, 1 round-trip
// SQL Server streams directly
// Time: 0.5 seconds

// Tradeoff: Less validation,
// no triggers, no change tracking
03

Requirement: “Read-Only Queries Are Slow”

Business Requirement
Dashboard loads 5,000 records for display. Memory spikes.
SQL Developer Thinks SELECT is lightweight by nature. The database doesn’t track anything — it just returns rows.
C# Developer Thinks By default, EF tracks every loaded entity to detect changes. For reads, skip tracking with AsNoTracking(). 30–50% faster, less memory.
With Tracking (Slow) — Default Behavior
var products = await db.Products
    .Where(p => p.IsActive)
    .ToListAsync(ct);

// EF creates ChangeTracker snapshot
// for each entity: original, current, state
// Uses memory ∝ record count
// For display only: wasteful overhead
// 5,000 records with 20 properties
// = 100K snapshots = bloat
AsNoTracking() — Fast Read
var products = await db.Products
    .Where(p => p.IsActive)
    .AsNoTracking()
    .ToListAsync(ct);

// EF skips ChangeTracker snapshot
// Just materializes objects
// No memory overhead
// 30–50% faster, 50% less memory

// Tradeoff: Cannot call SaveChanges
// on these objects. Read-only data.
AsNoTrackingWithIdentityResolution
var results = await db.Customers
    .AsNoTrackingWithIdentityResolution()
    .Include(c => c.Orders)
    .ToListAsync(ct);

// No change tracking, but prevents
// duplicate entity instances
// Same Customer appears once in
// memory even if in multiple Orders

// Useful: Reads with relationships,
// no modifications needed

Requirement: “Find Out Why This Query Is Slow”

Business Requirement
One API endpoint takes 3 seconds. Need to find the bottleneck.
SQL Developer Thinks EXPLAIN PLAN or SET STATISTICS IO ON. Look at execution plan, row counts, index usage.
C# Developer Thinks ToQueryString() shows generated SQL. Log queries. Use execution plan in SQL Server Management Studio. Then optimize.
Step 1: ToQueryString() — See Generated SQL
var query = db.Orders
    .Where(o => o.Total > 100)
    .Include(o => o.Customer)
    .Select(o => new { o.Id, o.Total,
                         o.Customer.Name });

var sql = query.ToQueryString();
Console.WriteLine(sql);

// Output:
// SELECT [o].[Id], [o].[Total], [c].[Name]
// FROM [Orders] AS [o]
// INNER JOIN [Customers] AS [c]
//   ON [o].[CustomerId] = [c].[Id]
// WHERE [o].[Total] > 100
Step 2: Enable Query Logging
var optionsBuilder = new DbContextOptionsBuilder
    <AppDbContext>();
optionsBuilder.UseSqlServer(connStr)
    .LogTo(Console.WriteLine,
           LogLevel.Information)
    .EnableDetailedErrors();

// Logs every query EF sends
// Shows parameters, execution time
// Warning: Logs sensitive data
// Use only in development
Step 3: Check Execution Plan
-- In SQL Server Management Studio:
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

SELECT [o].[Id], [o].[Total], [c].[Name]
FROM [Orders] AS [o]
INNER JOIN [Customers] AS [c]
  ON [o].[CustomerId] = [c].[Id]
WHERE [o].[Total] > 100;

-- Output shows:
-- Logical reads, scans vs seeks
-- CPU time, elapsed time
-- Join type (hash, loop, merge)
Common Bottlenecks & Fixes
ProblemSymptomFix
N+1 queriesRepeated same queryInclude() or Select()
Full table scanLogical reads = all rowsAdd index on WHERE column
Inefficient joinNested loop with large rowsEnsure FK indexed
Wrong join typeHash join (slow) vs seekCheck stats, rebuild index
Implicit conversionCONVERT in WHERE clauseMatch column type to param

Requirement: “Same Query Runs 1000 Times Per Second”

Business Requirement
Hot path query called on every request. Minimize overhead.
SQL Developer Thinks Stored procedure or parameterized query plan caching. Prepare once, execute many times.
C# Developer Thinks EF.CompileAsyncQuery() pre-compiles the LINQ expression tree. Skips compilation on every call. Only for hot paths.
Normal Query (Slow at Scale)
public async Task<Customer?> GetCustomer(
    int id, CancellationToken ct)
{
    // Called 1000 times/sec
    return await db.Customers
        .AsNoTracking()
        .FirstOrDefaultAsync(
            c => c.Id == id, ct);
}

// EF compiles LINQ to SQL every call
// Tree traversal: 0.5ms per call
// 1000 calls × 0.5ms = 500ms wasted
// Result: Only 50% CPU for work
Compiled Query (Fast)
private static readonly Func<
    AppDbContext, int, CancellationToken,
    Task<Customer?>> _getCustomer =
  EF.CompileAsyncQuery(
    (AppDbContext db, int id,
     CancellationToken ct) =>
      db.Customers
        .AsNoTracking()
        .FirstOrDefault(c => c.Id == id)
);

public async Task<Customer?> GetCustomer(
    int id, CancellationToken ct)
{
    return await _getCustomer
        .Invoke(db, id, ct);
}

// Result: 0 compilation overhead
// 10–20x faster for hot paths
When to Use Compiled Queries
Use CaseBenefitCaution
High-frequency (>100/sec)Saves compilation overheadMicro-optimization
Hot path endpointsEvery millisecond mattersProfile first
Same query, different paramsCompiled once, many executionsParameters only, not structure
Dashboard reportsRarely neededCalls infrequent

Requirement: “Page Through 1 Million Products”

Business Requirement
Product catalog has 1M items. Users browse pages of 20. Performance degrades on deep pages (page 50,000).
SQL Developer Thinks OFFSET/FETCH for simplicity, but skipping 999,980 rows is expensive. Keyset pagination (seek) is faster for large datasets.
C# Developer Thinks Skip()/Take() generates OFFSET/FETCH. For big datasets, use keyset pagination: WHERE Id > lastId. Always fast, regardless of page depth.
Offset Pagination (Simple but Slow at Deep Pages)
var pageNumber = 1;
var pageSize = 20;

var products = await db.Products
    .AsNoTracking()
    .OrderBy(p => p.Id)
    .Skip((pageNumber - 1) * pageSize)
    .Take(pageSize)
    .ToListAsync(ct);

// Page 1: Skip 0, Take 20 = fast
// Page 100: Skip 1980, Take 20 = slower
// Page 50,000: Skip 999,980 = very slow

// Generated SQL:
// SELECT ... FROM Products
//   ORDER BY Id
//   OFFSET 999980 ROWS FETCH NEXT 20 ROWS ONLY
Keyset Pagination (Always Fast)
int pageSize = 20;
int? lastSeenId = null;

var products = await db.Products
    .AsNoTracking()
    .Where(p => lastSeenId == null ||
               p.Id > lastSeenId)
    .OrderBy(p => p.Id)
    .Take(pageSize + 1)
    .ToListAsync(ct);

// Check if more rows exist
var hasMore = products.Count > pageSize;
if (hasMore)
    products = products.Take(pageSize).ToList();

// Last item for next page
lastSeenId = products.Last().Id;

// Always: Seek on indexed Id
// Every page: same speed
Cursor-Based API Pattern
public class PageResult<T>
{
    public List<T> Items { get; set; }
    public string? NextCursor { get; set; }
}

public async Task<PageResult<Product>>
  GetProductsPage(
    string? cursor = null,
    int pageSize = 20,
    CancellationToken ct = default)
{
    var lastId = cursor == null ? 0
      : int.Parse(cursor);
    var items = await db.Products
        .Where(p => p.Id > lastId)
        .OrderBy(p => p.Id)
        .Take(pageSize + 1)
        .AsNoTracking()
        .ToListAsync(ct);

    var hasMore = items.Count > pageSize;
    if (hasMore) items = items.Take(pageSize)
        .ToList();

    return new PageResult<Product>
    {
        Items = items,
        NextCursor = hasMore
            ? items.Last().Id.ToString()
            : null
    };
}

Best Practices Summary
  • Always measure. Use SET STATISTICS, execution plans, and profilers. Intuition is wrong.
  • AsNoTracking() by default. Add change tracking only when you need SaveChanges().
  • Use projections (Select()). Load only the columns you need. Let EF optimize.
  • Batch operations. AddRange() + one SaveChanges() is 10x faster than loops.
  • Index aggressively. WHERE columns, JOIN columns, ORDER BY columns. Cardinality matters.
  • Avoid OFFSET for pagination. Use keyset pagination for large datasets. Always O(1).
  • Compile hot queries. Only if profiling shows compilation is the bottleneck.
  • Log in development only. LogTo() with ILogger exposes sensitive data. Turn off in production.
  • Use async/await. Never block threads. Always pass CancellationToken to async methods.
