01

Logging the SQL EF Generates

You can't optimize what you can't see. EF can log every SQL statement it generates, allowing you to inspect queries and spot performance problems.

Enable logging in OnConfiguring() to see exactly what SQL is being executed.

Enable SQL Logging
// In DbContext protected override void OnConfiguring( DbContextOptionsBuilder optionsBuilder) { optionsBuilder .UseSqlServer(connString) .LogTo( Console.WriteLine, LogLevel.Information); } // Alternative: Log to a file var logFile = new StreamWriter( "sql.log", append: true); optionsBuilder .LogTo(logFile.WriteLine, LogLevel.Information); // Now every query is logged: // info: Microsoft.EntityFrameworkCore // .Database.Command[20100] // SELECT [c].[Id], [c].[Name] // FROM [Customers] AS [c]
SQL Output (What You'll See)
-- Query 1 (from logging) SELECT [c].[Id], [c].[Name], [c].[Email] FROM [Customers] AS [c] -- Query 2 (if N+1 problem) SELECT [o].[Id], [o].[CustomerId], [o].[Amount] FROM [Orders] AS [o] WHERE [o].[CustomerId] = 1 -- Query 3, 4, 5... (repeat) -- This pattern shows N+1 problem

Using Tags to Label Queries:

TagWith for Query Labels
// Tag queries for easy searching var customers = await context.Customers .TagWith( "GetActiveCustomers") .Where(c => c.IsActive) .ToListAsync(); // Log output shows: -- GetActiveCustomers SELECT [c].[Id], [c].[Name] FROM [Customers] AS [c] WHERE [c].[IsActive] = 1 // Makes finding specific queries // in large log files easy
Log Analysis Pattern
-- Sample log file with tags -- GetActiveCustomers SELECT * FROM Customers WHERE IsActive = 1; -- LoadCustomerOrders SELECT * FROM Orders WHERE CustomerId = 1; -- LoadCustomerOrders SELECT * FROM Orders WHERE CustomerId = 2; -- Repeated query = N+1 problem!
Always Log in Development
Enable SQL logging in development. It's the first tool for diagnosing performance issues. When you see the same query repeated 100 times in a row, you've found your N+1 problem. Tags make it easy to correlate application logic with database queries.
02

The N+1 Problem Revisited with Benchmarks

The N+1 problem is the most common EF performance issue. You load N entities, then in a loop, you access a navigation property that triggers a query for each one. Total: 1 + N queries.

Here's the problem, fixes, and real query counts.

N+1 Problem: 101 Queries
// Load 100 customers var customers = await context.Customers .ToListAsync(); // Query 1: SELECT * FROM Customers // Loop: triggers N more queries foreach (var cust in customers) { // This accesses the navigation // property (lazy loading) var orderCount = cust.Orders.Count(); // Queries 2-101: 100 separate // SELECT * FROM Orders // WHERE CustomerId = @id } // Total: 1 + 100 = 101 queries // Performance: ~2 seconds (network)
SQL: 101 Queries in Log
-- Query 1 SELECT * FROM Customers; -- Queries 2-101 SELECT * FROM Orders WHERE CustomerId = 1; SELECT * FROM Orders WHERE CustomerId = 2; SELECT * FROM Orders WHERE CustomerId = 3; -- ... (97 more times) -- Problem: network roundtrips, -- database overhead

Fix 1: Eager Loading with Include() — 1 Query

Fix: Use Include()
// Load customers with orders // in a single query var customers = await context.Customers .Include(c => c.Orders) .ToListAsync(); // Query 1: SELECT c.*, o.* FROM // Customers c // LEFT JOIN Orders o // ON c.Id = o.CustomerId // Loop: no additional queries! foreach (var cust in customers) { var orderCount = cust.Orders.Count(); // Orders are already loaded // No query needed } // Total: 1 query // Performance: ~50ms // 40x faster!
SQL: Single JOIN Query
SELECT c.Id, c.Name, c.Email, o.Id, o.CustomerId, o.Amount FROM Customers c LEFT JOIN Orders o ON c.Id = o.CustomerId; -- All data fetched in one query -- Single network roundtrip -- Database processes once

Fix 2: Projection with Select() — 1 Query, Only Needed Data

Fix: Use Projection
// Load only what you need var result = await context.Customers .Select(c => new { c.Id, c.Name, OrderCount = c.Orders .Count() }) .ToListAsync(); // Query 1: Projection executes // in database, not memory // Loop: no entities, just DTOs foreach (var item in result) { // OrderCount already calculated Console.WriteLine( item.OrderCount); } // Total: 1 query // Data transferred: minimal // Performance: ~30ms (best)
SQL: Aggregation in Database
SELECT c.Id, c.Name, COUNT(o.Id) AS OrderCount FROM Customers c LEFT JOIN Orders o ON c.Id = o.CustomerId GROUP BY c.Id, c.Name; -- Aggregation happens in SQL -- Only summary data returned -- Most efficient solution
Projection is Often the Best Fix
When building reports or API responses, projection (using .Select()) is often better than eager loading. It returns only the data you need, the database does the aggregation, and you avoid loading unnecessary entities. Use eager loading when you need full entity objects for modification.
03

AsNoTracking and AsNoTrackingWithIdentityResolution

Change tracking has overhead. For read-only queries, disabling tracking saves memory and CPU. EF Core offers two options: AsNoTracking() for simple reads, and AsNoTrackingWithIdentityResolution() for complex graphs with duplicate entities.

AsNoTracking Performance
// WITH tracking (default) var sw = Stopwatch .StartNew(); var customers = await context.Customers .ToListAsync(); sw.Stop(); // Time: 145ms for 10,000 rows // WITHOUT tracking sw = Stopwatch .StartNew(); customers = await context.Customers .AsNoTracking() .ToListAsync(); sw.Stop(); // Time: 120ms for 10,000 rows // ~17% faster! // Memory usage: 30% lower
Benchmark Results
-- Same SQL query in both -- cases, but EF behavior -- differs: -- With Tracking: -- - Maintain snapshot -- - Allocate memory -- - 145ms -- Without Tracking: -- - No snapshot -- - Lower memory -- - 120ms -- Same SQL, different overhead

AsNoTrackingWithIdentityResolution: Use when querying complex graphs with Include() but the same entity appears multiple times in the result.

Complex Graphs with Duplicates
// Query: customers with orders // with products // With tracking (normal): var customers = await context.Customers .Include(c => c.Orders) .ThenInclude( o => o.Products) .ToListAsync(); // Problem: cartesian explosion // Same customer duplicated // for each order // With AsNoTrackingWithIdentity: customers = await context.Customers .AsNoTrackingWithIdentityResolution() .Include(c => c.Orders) .ThenInclude( o => o.Products) .ToListAsync(); // EF deduplicates: one // customer instance, // multiple order references
Cartesian Explosion Problem
-- SQL returns all combinations -- If customer 1 has 5 orders, -- and each order has 3 products, -- 1 customer appears 15 times SELECT c.*, o.*, p.* FROM Customers c LEFT JOIN Orders o ON c.Id = o.CustomerId LEFT JOIN OrderProducts op ON o.Id = op.OrderId LEFT JOIN Products p ON op.ProductId = p.Id; -- 1 customer * 5 orders -- * 3 products = 15 rows -- Customer data duplicated
Choose the Right Tracking Mode
For simple read-only queries, use AsNoTracking(). For complex graphs with potential duplicates, use AsNoTrackingWithIdentityResolution(). For modification scenarios, use the default (with tracking).
04

Compiled Queries for Hot Paths

LINQ queries are compiled to SQL at runtime. If you execute the same query thousands of times, you pay that compilation cost every time.

Compiled queries pre-compile the LINQ expression. The first execution is cached; subsequent executions reuse the compiled form. Great for hot paths like web request handlers executed thousands of times.

Normal Query (Recompiled Each Time)
// This query recompiles every // time it executes public class CustomerService { public async Task<Customer> GetById(int id) { var customer = await context.Customers .FirstOrDefaultAsync( c => c.Id == id); // Query compiled every call return customer; } } // Benchmark: 10,000 calls // Time: 1200ms // Includes 10,000 compilations
Query Compilation Overhead
-- SQL is the same each time: SELECT * FROM Customers WHERE Id = @id; -- But EF has to: -- 1. Parse LINQ expression -- 2. Generate SQL -- 3. Prepare parameter -- 4. Execute -- Steps 1-3 happen every time! -- Compilation adds ~100us

Compiled Query Version:

Using Compiled Queries
// Pre-compiled query private static readonly Func<AppDbContext, int, Task<Customer>> _getCustomerById = EF.CompileAsyncQuery( (AppDbContext ctx, int id) => ctx.Customers .FirstOrDefault( c => c.Id == id)); public async Task<Customer> GetById(int id) { // Reuses pre-compiled form return await _getCustomerById .Invoke(context, id); } // Benchmark: 10,000 calls // Time: 850ms // No compilation overhead // ~30% faster!
Execution Plan Caching
-- Database also caches -- execution plans SELECT * FROM Customers WHERE Id = @id; -- First execution: -- Database parses and plans -- Subsequent executions: -- Database reuses plan -- With compiled queries: -- EF reuses compiled form -- Database reuses plan -- Double efficiency!
Use Compiled Queries Selectively
Only compile queries in hot paths: request handlers, background jobs, queries executed thousands of times. For one-off queries or rarely-used code paths, the overhead of compiling isn't worth the code complexity. Profile first; optimize based on evidence.
05

Bulk Operations: ExecuteUpdate and ExecuteDelete

EF Core 7+ introduced ExecuteUpdate() and ExecuteDelete(). These execute a single UPDATE or DELETE statement without loading entities into memory.

This is massively more efficient than the old pattern: load all entities, modify in memory, save.

OLD (Bad): Load, Modify, Save
// OLD way: Load ALL products var products = await context.Products .Where(p => p.Category == "Electronics") .ToListAsync(); // Modify in memory foreach (var p in products) { p.Price *= 1.1m; } // Save all await context.SaveChangesAsync(); // Queries executed: // 1. SELECT * FROM Products // WHERE Category = 'Electronics' // 2-N. Individual UPDATE for // each product // Result: N+1 queries, slow! // 1000 products = 1001 queries
OLD: Multiple Statements
SELECT * FROM Products WHERE Category = 'Electronics'; -- 1000 individual updates: UPDATE Products SET Price = 110.00 WHERE Id = 1; UPDATE Products SET Price = 121.00 WHERE Id = 2; -- ... (998 more times) -- Horrible performance!

NEW (Good): ExecuteUpdate in Single Statement

NEW: ExecuteUpdate (EF 7+)
// NEW way: Single UPDATE // statement var rowsAffected = await context.Products .Where(p => p.Category == "Electronics") .ExecuteUpdateAsync( s => s .SetProperty( p => p.Price, p => p.Price * 1.1m)); Console.WriteLine( $"{rowsAffected} products " + "updated"); // Query executed: // 1. Single UPDATE statement // Result: 1 query, super fast! // 1000 products = 1 query // 1000x faster than loading!
NEW: Single UPDATE Statement
-- Generated by EF: UPDATE Products SET Price = Price * 1.1 WHERE Category = 'Electronics'; -- Database handles in one -- operation -- No entities loaded -- No round-trips -- Extremely fast

ExecuteDelete for bulk deletes:

ExecuteDelete: Remove Old Records
// Delete orders older than // 2 years var rowsDeleted = await context.Orders .Where(o => o.OrderDate < DateTime.UtcNow .AddYears(-2)) .ExecuteDeleteAsync(); Console.WriteLine( $"{rowsDeleted} old orders " + "deleted"); // Query executed: // DELETE FROM Orders // WHERE OrderDate < @date // Single statement, no loading
SQL: DELETE Generated
DELETE FROM Orders WHERE OrderDate < DATEADD(YEAR, -2, GETUTCDATE()); -- Executed in database -- No entities involved -- Perfect for cleanup tasks
ExecuteUpdate/ExecuteDelete are Essential for Bulk Operations
Never load entities just to bulk update or delete them. Use ExecuteUpdate() and ExecuteDelete() to execute a single database statement. For 1000 products, the difference is 1 query vs 1001 queries. Use these for bulk operations, cleanup tasks, and performance-critical code.
06

Split Queries vs Single Queries

When you eager load with multiple Include() calls, EF can generate either a single query with multiple JOINs (cartesian explosion risk) or multiple separate queries (more roundtrips).

Use .AsSplitQuery() or .AsSingleQuery() to control which strategy EF uses.

Problem: Cartesian Explosion
// Default: Single query with // multiple JOINs var customers = await context.Customers .Include(c => c.Orders) .Include(c => c.Addresses) .ToListAsync(); // Generated SQL: // SELECT c.*, o.*, a.* // FROM Customers c // LEFT JOIN Orders o // LEFT JOIN Addresses a // Problem: If customer has // 5 orders and 3 addresses, // they appear 15 times! // Result set is 15x larger // Network transfer bloated
Cartesian Product Issue
-- Rows = Customers × Orders -- × Addresses SELECT c.*, o.*, a.* FROM Customers c LEFT JOIN Orders o ON c.Id = o.CustomerId LEFT JOIN Addresses a ON c.Id = a.CustomerId; -- Customer 1: 5 orders, -- 3 addresses -- Result rows: 5 × 3 = 15 -- Customer data duplicated 15x -- Wasteful!

Solution: AsSplitQuery() — Multiple Roundtrips, Smaller Result Sets

AsSplitQuery: Better for Complex Graphs
// Multiple queries, each // smaller and focused var customers = await context.Customers .AsSplitQuery() .Include(c => c.Orders) .Include(c => c.Addresses) .ToListAsync(); // Generated SQL: // Query 1: SELECT * FROM // Customers // Query 2: SELECT * FROM // Orders WHERE CustomerId IN // Query 3: SELECT * FROM // Addresses WHERE CustomerId IN // Result: 3 smaller queries, // no cartesian explosion // Total data transferred: less // Preferred for complex graphs
Split Query Approach
-- Query 1: Customers SELECT * FROM Customers; -- Query 2: Orders for those -- customers SELECT * FROM Orders WHERE CustomerId IN (1, 2, 3, ...); -- Query 3: Addresses for those -- customers SELECT * FROM Addresses WHERE CustomerId IN (1, 2, 3, ...); -- More queries, but no -- cartesian explosion -- Smaller result sets

Decision Table: When to use each strategy:

Scenario Use This Why
Single Include (e.g. Customers + Orders) AsSingleQuery One JOIN is efficient; minimal cartesian explosion
Multiple Includes (complex graph) AsSplitQuery Avoids cartesian explosion; multiple smaller queries
Uncertain; need safety AsSplitQuery (default in 8.0) More roundtrips but safer; no data duplication risk
AsSplitQuery is the Safe Default for Complex Graphs
For simple eager loading with one or two includes, AsSingleQuery() is fine. For complex graphs with multiple includes, AsSplitQuery() is safer. It generates more queries but avoids cartesian explosion. In EF Core 8.0+, AsSplitQuery() became the default for safety.

Key Takeaways

  • Enable SQL logging in development with optionsBuilder.LogTo(). You can't optimize what you can't see. Log to console or file to inspect generated SQL.
  • Tag queries with .TagWith() for easy identification in logs. Makes it simple to correlate application code with generated SQL.
  • The N+1 problem: load N entities, then load related data for each = N+1 queries. Fix with .Include() (1 JOIN query), .Select() projection, or .AsSplitQuery().
  • Projection (.Select()) is often the best N+1 fix. Return only the data you need, let the database do aggregations. Most efficient for reporting.
  • AsNoTracking() disables change tracking for read-only queries, improving performance 10-20%. Use for API responses, exports, reporting. AsNoTrackingWithIdentityResolution() for complex graphs with duplicates.
  • Compiled queries cache the LINQ compilation step. Use for hot paths executed thousands of times (web handlers, scheduled jobs). Avoid for one-off queries.
  • ExecuteUpdate() and ExecuteDelete() execute single SQL statements without loading entities. For 1000 products, this is 1 query vs 1001 queries. Essential for bulk operations.
  • AsSplitQuery() avoids cartesian explosion with multiple includes. Generates more queries but smaller result sets. Safe for complex graphs. AsSplitQuery is the EF 8.0+ default.
