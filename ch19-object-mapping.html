Chapter 19

Object Mapping with Dapper

Dapper automatically maps SQL result columns to C# properties — no configuration needed when names match. When they don’t, you have options: SQL aliases, custom mappers, or attribute-based mapping. Learn when to use each approach and how to handle edge cases like records, dynamic results, and JSON columns.

01

Requirement: “Map Query Results to a C# Class”

Business Requirement
SELECT returns Id, Name, Email columns. Map them automatically to a Customer class with the same property names.
SQL Developer Thinks Column names are the contract. I’ll return Id, Name, Email from the SELECT. Whatever reads this needs to match those column names.
C# Developer Thinks Dapper matches column names to property names case-insensitively. If I have properties Id, Name, Email on my Customer class, Dapper automatically assigns the values. Zero configuration.
SQL — SELECT the Data
SELECT Id, Name, Email
FROM Customers
WHERE Id = @CustomerId;
C# — Define the Class
public class Customer
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string Email { get; set; } = "";
}
C# — Dapper QueryAsync Mapping
var customer = await conn.QueryFirstOrDefaultAsync<Customer>(
    @"SELECT Id, Name, Email FROM Customers WHERE Id = @Id",
    new { Id = customerId },
    commandTimeout: 30);

Dapper inspects the Customer class, sees properties Id, Name, Email, and assigns the SQL columns to them automatically. Column names are case-insensitive.

02

Requirement: “Database Column Names Don’t Match C# Properties”

Business Requirement
The DBA uses snake_case (first_name, last_name, email_address). Our C# classes use PascalCase (FirstName, LastName, EmailAddress). Dapper can’t map them automatically.
SQL Developer Thinks Column names are fixed. If the application wants to use different property names, it needs to alias the columns in the SELECT.
C# Developer Thinks Option 1: Use SQL aliases (SELECT first_name AS FirstName). Option 2: Use DefaultTypeMap with MatchNamesWithUnderscores. Option 3: Custom column-to-property mapping.
Approach 1: SQL Aliases (Recommended)
SELECT
    id AS Id,
    first_name AS FirstName,
    last_name AS LastName,
    email_address AS EmailAddress
FROM users
WHERE id = @UserId;
C# — Simple Class (Aliases Handle the Mapping)
public class User
{
    public int Id { get; set; }
    public string FirstName { get; set; } = "";
    public string LastName { get; set; } = "";
    public string EmailAddress { get; set; } = "";
}
C# — Query with Aliases
var user = await conn.QueryFirstOrDefaultAsync<User>(
    @"SELECT id AS Id, first_name AS FirstName,
             last_name AS LastName, email_address AS EmailAddress
      FROM users WHERE id = @UserId",
    new { UserId = userId });
Approach 2: DefaultTypeMap.MatchNamesWithUnderscores
// Configure Dapper to match snake_case columns to PascalCase properties
SqlMapper.SetTypeMap(typeof(User),
    new CustomPropertyTypeMap(typeof(User),
        (type, columnName) => type.GetProperty(
            columnName.Replace("_", ""),
            System.Reflection.BindingFlags.IgnoreCase
            | System.Reflection.BindingFlags.Public
            | System.Reflection.BindingFlags.Instance)));

// Now this works — no aliases needed
var user = await conn.QueryFirstOrDefaultAsync<User>(
    @"SELECT id, first_name, last_name, email_address
      FROM users WHERE id = @UserId",
    new { UserId = userId });
Approach 3: Custom Column Mapping with Attributes
public class User
{
    [Column("id")]
    public int Id { get; set; }

    [Column("first_name")]
    public string FirstName { get; set; } = "";

    [Column("last_name")]
    public string LastName { get; set; } = "";

    [Column("email_address")]
    public string EmailAddress { get; set; } = "";
}

// Custom TypeMap that reads the [Column] attribute
public class ColumnAttributeTypeMap : SqlMapper.ITypeMap
{
    private readonly Type _type;

    public ColumnAttributeTypeMap(Type type) => _type = type;

    public SqlMapper.IDataReader GetReader(IDataReader reader)
    {
        return reader;
    }

    public ConstructorInfo FindConstructor(string[] names)
    {
        return _type.GetConstructor(Type.EmptyTypes);
    }

    public ConstructorInfo FindExplicitConstructor()
    {
        return _type.GetConstructor(Type.EmptyTypes);
    }

    public SqlMapper.IMemberMap GetMember(string name)
    {
        var property = _type.GetProperties()
            .FirstOrDefault(p =>
            {
                var attr = p.GetCustomAttribute<ColumnAttribute>();
                return attr?.Name.Equals(name, StringComparison.OrdinalIgnoreCase) ?? false;
            });

        if (property == null)
            property = _type.GetProperty(name,
                System.Reflection.BindingFlags.IgnoreCase
                | System.Reflection.BindingFlags.Public
                | System.Reflection.BindingFlags.Instance);

        return property == null ? null : new SimpleMemberMap(name, property);
    }
}
03

Requirement: “Use Immutable Records for Query Results”

Business Requirement
Our API responses must be immutable. Use C# records instead of mutable classes for safety and clarity.
SQL Developer Thinks A query returns rows. How the client structures them is up to the client.
C# Developer Thinks C# 9+ records are perfect for read-only result objects. Dapper maps to record constructors automatically. I can also use readonly structs for value-type efficiency.
C# — Record Definition
// C# record — immutable by default
public record Customer(
    int Id,
    string Name,
    string Email);

// Or with a primary constructor and properties
public record CustomerRecord
{
    public int Id { get; init; }
    public string Name { get; init; } = "";
    public string Email { get; init; } = "";
}
C# — Dapper Maps to Records (Constructor-Based)
// Dapper passes constructor arguments by position
var customers = await conn.QueryAsync<Customer>(
    @"SELECT Id, Name, Email FROM Customers
      WHERE Id IN @Ids",
    new { Ids = new[] { 1, 2, 3 } },
    commandTimeout: 30);
C# — Readonly Struct (Value Type)
// Readonly struct — even more efficient for small results
public readonly struct CustomerStruct
{
    public int Id { get; init; }
    public string Name { get; init; } = "";
    public string Email { get; init; } = "";
}

Records prevent accidental mutation and make immutable intent explicit. Dapper maps by position first, then by property name if position fails.

04

Requirement: “I Don’t Know the Column Names at Runtime”

Business Requirement
An ad-hoc reporting tool lets users pick columns dynamically. I can’t define a static class for every possible column combination. Use dynamic results.
SQL Developer Thinks SELECT * or a dynamic column list. The result structure depends on the query at runtime.
C# Developer Thinks Use QueryAsync<dynamic>. Dapper returns ExpandoObject — access columns by name as object properties. Dictionary-like but accessed with dot notation.
SQL — Dynamic Query
-- User selected: Id, Name, Email, CreatedAt
SELECT Id, Name, Email, CreatedAt
FROM Customers
WHERE IsActive = 1
ORDER BY CreatedAt DESC;
C# — QueryAsync<dynamic>
// Dapper returns IEnumerable<dynamic>
// where each row is an ExpandoObject
var results = await conn.QueryAsync<dynamic>(
    @"SELECT Id, Name, Email, CreatedAt
      FROM Customers
      WHERE IsActive = 1
      ORDER BY CreatedAt DESC",
    commandTimeout: 30);

// Access columns by name
foreach (var row in results)
{
    int id = row.Id;
    string name = row.Name;
    string email = row.Email;
    DateTime created = row.CreatedAt;
}
C# — Accessing Dynamic Results Safely
var results = await conn.QueryAsync<dynamic>(sql);

foreach (var row in results)
{
    // Cast to int
    var id = (int)row.Id;

    // Handle missing columns
    var email = row.Email as string ?? "N/A";

    // Check if a property exists
    if (((IDictionary<string, object>)row).ContainsKey("PhoneNumber"))
    {
        var phone = row.PhoneNumber;
    }
}

When to use dynamic: Ad-hoc reports, user-defined queries, export tools. When NOT to use: Regular CRUD operations (use typed classes), performance-critical code, APIs with strict response contracts.

05

Requirement: “One Query Returns Two Result Sets”

Business Requirement
A stored procedure returns customers in the first result set and their orders in the second. Execute one query, get both collections.
SQL Developer Thinks Multiple SELECT statements in a batch or stored procedure. SQL Server sends all results together.
C# Developer Thinks Use QueryMultipleAsync. It returns a GridReader object. Call ReadAsync<T> for each result set in order.
SQL — Multiple SELECT Results
-- Stored procedure returning two result sets
CREATE PROCEDURE GetCustomerWithOrders
    @CustomerId INT
AS
BEGIN
    -- First result set: Customer
    SELECT Id, Name, Email
    FROM Customers
    WHERE Id = @CustomerId;

    -- Second result set: Orders
    SELECT OrderId, OrderDate, Total
    FROM Orders
    WHERE CustomerId = @CustomerId
    ORDER BY OrderDate DESC;
END;
C# — QueryMultipleAsync Pattern
public record Customer(int Id, string Name, string Email);
public record Order(int OrderId, DateTime OrderDate, decimal Total);

// Use QueryMultipleAsync to get GridReader
using var reader = await conn.QueryMultipleAsync(
    "GetCustomerWithOrders",
    new { CustomerId = customerId },
    commandType: CommandType.StoredProcedure,
    commandTimeout: 30);

// Read each result set in order
var customer = await reader.ReadSingleOrDefaultAsync<Customer>();
var orders = await reader.ReadAsync<Order>().ToListAsync();

// Use the results
if (customer != null)
{
    Console.WriteLine($"Customer: {customer.Name}");
    Console.WriteLine($"Orders: {orders.Count}");
    foreach (var order in orders)
    {
        Console.WriteLine($"  {order.OrderDate:yyyy-MM-dd}: ${order.Total}");
    }
}

Key: Call ReadAsync<T> in the same order as the SELECT statements in your SQL. If your SQL has 3 SELECTs, call Read three times.

06

Requirement: “Map a JSON Column to a C# Object”

Business Requirement
A Settings column stores JSON (e.g., {"theme": "dark", "language": "en"}). Deserialize it automatically to a C# object.
SQL Developer Thinks The column is NVARCHAR(MAX) with JSON data. I’ll return it as a string or use JSON_VALUE to extract specific fields.
C# Developer Thinks Dapper supports custom type handlers. Implement SqlMapper.TypeHandler<T> to deserialize the JSON string and serialize it back on writes.
C# — Define the Settings Class and TypeHandler
public class AppSettings
{
    public string Theme { get; set; } = "light";
    public string Language { get; set; } = "en";
    public bool NotificationsEnabled { get; set; } = true;
}

// Custom TypeHandler for JSON serialization/deserialization
public class JsonTypeHandler<T> : SqlMapper.TypeHandler<T>
{
    private static readonly JsonSerializerOptions Options = new()
    {
        PropertyNameCaseInsensitive = true
    };

    public override T Parse(object value)
    {
        if (value == null || value == DBNull.Value)
            return default;

        var json = value.ToString();
        return JsonSerializer.Deserialize<T>(json, Options)
            ?? throw new InvalidOperationException($"Failed to deserialize JSON for {typeof(T).Name}");
    }

    public override void SetValue(IDbDataParameter parameter, T value)
    {
        if (value == null)
        {
            parameter.Value = DBNull.Value;
        }
        else
        {
            parameter.Value = JsonSerializer.Serialize(value);
        }
    }
}
C# — Register the TypeHandler and Use It
// Register once at application startup
SqlMapper.AddTypeHandler(new JsonTypeHandler<AppSettings>());

// Define the entity
public class User
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public AppSettings Settings { get; set; } = new();
}

// Query as normal — Dapper uses the handler automatically
var user = await conn.QueryFirstOrDefaultAsync<User>(
    @"SELECT Id, Name, Settings
      FROM Users
      WHERE Id = @UserId",
    new { UserId = userId });

// Settings is already deserialized
Console.WriteLine($"Theme: {user.Settings.Theme}");
Console.WriteLine($"Language: {user.Settings.Language}");
SQL — JSON Column Definition
CREATE TABLE Users
(
    Id       INT IDENTITY(1,1) PRIMARY KEY,
    Name     NVARCHAR(100) NOT NULL,
    Settings NVARCHAR(MAX) NOT NULL
             DEFAULT N'{}'
);

-- Example data
INSERT INTO Users (Name, Settings)
VALUES (
    'Alice',
    N'{"theme": "dark", "language": "en", "notificationsEnabled": true}'
);

Custom type handlers work for any complex type: JSON, XML, encrypted values, enums stored as strings. Register them once; use them everywhere.
