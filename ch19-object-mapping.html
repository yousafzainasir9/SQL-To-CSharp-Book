Part 5: Dapper and Micro-ORMs

Object Mapping — SQL Results to C# Objects
Conventions, Handlers, and Advanced Techniques

Dapper's greatest strength is its ability to map SQL query results directly to C# objects with minimal configuration. Understanding how Dapper maps columns to properties, how to handle type conversions, and when to use dynamic results vs typed objects will make you a proficient Dapper developer. This chapter covers convention-based mapping, type handlers, records, QueryMultiple, and practical patterns for every mapping scenario.

In This Chapter

  • Convention-Based Mapping — Case-insensitive column-to-property matching

01

How Dapper Maps Results

Dapper uses reflection to map SQL result columns to C# object properties. When you call connection.Query("SELECT * FROM Customers"), Dapper examines the Customer class, finds its public properties, then matches them against the SQL result columns by name (case-insensitive). Each matching column value is assigned to the corresponding property via its setter. For performance, Dapper caches this mapping after the first execution, so subsequent calls are extremely fast.

The algorithm is straightforward: for each column in the result set, find a property on the target class with the same name (ignoring case). If found, assign the value. If a column doesn't have a matching property, it's silently ignored. If a property doesn't have a matching column, it remains uninitialized (null for reference types, default for value types). This convention is so common that it requires almost no configuration in typical scenarios.

The caching mechanism is crucial for Dapper's performance. The first time you execute a query mapping to Customer, Dapper builds and caches the mapping strategy using IL code generation. Subsequent executions reuse the cached strategy, avoiding reflection overhead entirely. This is why Dapper can be faster than Entity Framework for simple queries—there's minimal overhead once the cache is warm. The cache is per-type per-query result shape, so different queries mapping to the same class still benefit from caching.

SQL: Result Columns -- SQL returns columns with specific names SELECT Id, FirstName, LastName, Email, CreatedAt FROM Customers WHERE Id = 5; -- Result columns (metadata): -- Column 0: Id (INT) -- Column 1: FirstName (NVARCHAR) -- Column 2: LastName (NVARCHAR) -- Column 3: Email (NVARCHAR) -- Column 4: CreatedAt (DATETIME2)
C#: Property Mapping // C# class with properties matching SQL columns public class Customer { public int Id { get; set; } public string FirstName { get; set; } public string LastName { get; set; } public string Email { get; set; } public DateTime CreatedAt { get; set; } } // Dapper's mapping algorithm: // 1. Read result column metadata (names, types) // 2. For each column, find property by name (case-insensitive) // 3. Convert SQL value to CLR type if needed // 4. Assign to property via setter // 5. Cache the mapping strategy for future queries // Usage: var customers = connection.Query( "SELECT Id, FirstName, LastName, Email, CreatedAt FROM Customers" ).ToList(); // Dapper internally: // SqlDataReader reader = cmd.ExecuteReader(); // var mapping = GetOrBuildMapping(reader, typeof(Customer)); // while (reader.Read()) // { // var customer = new Customer(); // customer.Id = (int)reader[0]; // customer.FirstName = (string)reader[1]; // customer.LastName = (string)reader[2]; // customer.Email = (string)reader[3]; // customer.CreatedAt = (DateTime)reader[4]; // yield return customer; // }
C#: Property Mapping and Caching public class Customer { public int Id { get; set; } public string FirstName { get; set; } public string LastName { get; set; } public string Email { get; set; } public DateTime CreatedAt { get; set; } } // Usage with Dapper var customers = connection.Query( "SELECT Id, FirstName, LastName, Email, CreatedAt FROM Customers" ).ToList(); // First execution: Dapper builds mapping via reflection and IL // Second execution: Dapper reuses cached mapping (very fast) // Third+ executions: Cache hit every time
Key Insight:

Dapper caches mappings for performance. First execution builds the mapping via reflection. Subsequent executions reuse the cached strategy. This makes Dapper very fast once warmed up. The cache is per-type-per-query-result-shape, so different queries to the same class benefit from caching.

02

Convention-Based Mapping

Dapper's convention is simple: column names (case-insensitive) are matched to property names. FirstName matches firstname, FIRSTNAME, or FirstName. But first_name does not match FirstName. This is Dapper's straightforward approach—no complex name transformations.

C#: Mapping Scenarios // Scenario 1: Perfect PascalCase match var perfect = connection.Query( "SELECT Id, FirstName, LastName, Email FROM Customers" ).ToList(); // All columns match perfectly // Scenario 2: Case-insensitive (all work the same) var lowercase = connection.Query( "SELECT id, firstname, lastname, email FROM customers" ).ToList(); // Still matches (case is ignored) // Scenario 3: snake_case DOES NOT match PascalCase var snakecase = connection.Query( "SELECT id, first_name, last_name, email FROM customers" ).ToList(); // first_name does NOT match FirstName! // FirstName property will be null (mistake!) // Solution: Use aliases for snake_case var withAlias = connection.Query( "SELECT id AS Id, first_name AS FirstName, " + "last_name AS LastName, email AS Email FROM customers" ).ToList(); // Now it works perfectly
Key Insight:

Column naming must match property names exactly (case-insensitive). Snake_case doesn't automatically convert to PascalCase. Use SQL aliases to bridge naming mismatches. This is the most common source of mapping bugs.

03

Column Aliases for Name Mismatches

SQL aliases solve naming mismatches when you can't change the C# class or the database schema. SELECT customer_id AS CustomerId creates an alias that Dapper sees as the column name. This is the standard solution for legacy databases with snake_case columns and modern C# classes with PascalCase properties.

SQL: Using Aliases SELECT customer_id AS CustomerId, full_name AS FullName, email_address AS Email, created_at AS CreatedAt FROM legacy_customers WHERE customer_id = 5;
C#: Mapping with Aliases public class Customer { public int CustomerId { get; set; } public string FullName { get; set; } public string Email { get; set; } public DateTime CreatedAt { get; set; } } // With aliases: perfect mapping var customers = connection.Query( "SELECT customer_id AS CustomerId, full_name AS FullName, " + "email_address AS Email, created_at AS CreatedAt FROM customers" ).ToList();
Key Insight:

Aliases are the standard solution for schema naming mismatches. Use them liberally when you can't change C# classes or database columns. This is professional, production-grade code—never assume names will match.

In real projects, you'll encounter legacy databases with inconsistent naming. Some tables use snake_case, others camelCase, some abbreviate column names cryptically. SQL aliases bridge this gap elegantly. As a best practice, your SELECT statements should always use clear, meaningful aliases that match your C# properties, regardless of how the database names columns.

C#: Comprehensive Aliasing Example // Legacy messy database // Tables: tblCustomer, tblOrder // Columns: cust_id, cust_name, ord_date, ord_amt public class CustomerSummary { public int CustomerId { get; set; } public string CustomerName { get; set; } public int OrderCount { get; set; } public decimal TotalAmount { get; set; } } // Clean SQL with aliases var query = @" SELECT c.cust_id AS CustomerId, c.cust_name AS CustomerName, COUNT(o.ord_id) AS OrderCount, COALESCE(SUM(o.ord_amt), 0) AS TotalAmount FROM tblCustomer c LEFT JOIN tblOrder o ON c.cust_id = o.cust_id GROUP BY c.cust_id, c.cust_name "; var results = connection.Query(query).ToList(); // Perfect mapping despite messy database names
Key Insight:

Aliases bridge naming gaps between database and C#. Use them liberally for snake_case or abbreviated column names. Your SELECT statement is where the mapping happens—make it explicit and clear.

04

Custom Type Handlers

Sometimes Dapper can't automatically convert a SQL type to a C# type. DateOnly (C# 6.0+) doesn't have a direct SQL Server equivalent. JSON columns are stored as NVARCHAR but need to map to objects. Custom type handlers register converters for specific types. Register them once at startup, and Dapper uses them automatically.

C#: Custom Type Handlers // Handler for DateOnly public class DateOnlyHandler : SqlMapper.TypeHandler { public override DateOnly Parse(object value) { return DateOnly.FromDateTime((DateTime)value); } public override void SetValue(IDbDataParameter parameter, DateOnly value) { parameter.Value = value.ToDateTime(TimeOnly.MinValue); } } // Register at startup SqlMapper.AddTypeHandler(new DateOnlyHandler()); // Now this works: public class Event { public int Id { get; set; } public DateOnly EventDate { get; set; } } var events = connection.Query( "SELECT Id, EventDate FROM Events" ).ToList(); // EventDate (DATETIME in SQL) auto-converts to DateOnly // JSON handler example public class JsonHandler : SqlMapper.TypeHandler where T : class { public override T Parse(object value) { var json = (string)value; return JsonConvert.DeserializeObject(json); } public override void SetValue(IDbDataParameter parameter, T value) { parameter.Value = JsonConvert.SerializeObject(value); } } // Register JSON handler for specific types SqlMapper.AddTypeHandler( new JsonHandler>() ); public class Product { public int Id { get; set; } public string Name { get; set; } public List Tags { get; set; } // Maps from JSON column } var products = connection.Query( "SELECT Id, Name, Tags FROM Products" ).ToList(); // Tags (JSON_NVARCHAR in SQL) auto-deserializes to List
Key Insight:

Custom type handlers extend Dapper's mapping capabilities. Register them once at startup, and Dapper uses them automatically for all queries. This is the mechanism for supporting modern C# types (DateOnly, TimeOnly) and custom scenarios (JSON columns, enums).

Type handlers are powerful for bridging SQL and C# type systems. DateOnly is a C# 6.0+ type with no direct SQL equivalent—you store it as DATETIME2, then convert on read/write. JSON columns in SQL Server (JSON datatype or NVARCHAR with JSON content) need deserialization. Custom value types or domain objects need handlers. The handler pattern is extensible and clean.

C#: More Type Handlers // Enum handler public class EnumHandler : SqlMapper.TypeHandler where T : struct, Enum { public override T Parse(object value) { if (value == null) return default; if (value is T) return (T)value; if (value is string str) return Enum.Parse(str, ignoreCase: true); return (T)Enum.ToObject(typeof(T), value); } public override void SetValue(IDbDataParameter parameter, T value) { parameter.Value = value.ToString(); } } // TimeOnly handler public class TimeOnlyHandler : SqlMapper.TypeHandler { public override TimeOnly Parse(object value) { var ts = (TimeSpan)value; return TimeOnly.FromTimeSpan(ts); } public override void SetValue(IDbDataParameter parameter, TimeOnly value) { parameter.Value = value.ToTimeSpan(); } } // Register at startup SqlMapper.AddTypeHandler(new TimeOnlyHandler()); SqlMapper.AddTypeHandler(new EnumHandler()); // Now use them: public enum OrderStatus { Pending, Shipped, Delivered } public class Order { public int Id { get; set; } public TimeOnly DeliveryTime { get; set; } public OrderStatus Status { get; set; } } var orders = connection.Query( "SELECT Id, DeliveryTime, Status FROM Orders" ).ToList(); // TimeOnlyHandler and EnumHandler are used automatically
Key Insight:

Custom type handlers register converters for types Dapper doesn't natively support. Register once at startup, use everywhere. This is clean, reusable, and follows the handler pattern.

05

Mapping to Records, Structs, and Value Objects

Dapper works with more than just classes. C# 9 records provide immutable mapping targets. Structs map similarly to classes. This flexibility lets you use the right C# type for each situation.

C#: Records, Structs, and Value Objects // Record: immutable mapping target public record Customer(int Id, string FirstName, string LastName, string Email); var customers = connection.Query( "SELECT Id, FirstName, LastName, Email FROM Customers" ).ToList(); // Dapper constructs records via positional parameters // Struct: value type mapping public struct CustomerId { public int Value { get; set; } } // Readonly struct (immutable) public readonly struct Money { public decimal Amount { get; init; } public string Currency { get; init; } } // Init-only properties (C# 9) public class Order { public int Id { get; init; } public DateTime CreatedAt { get; init; } public decimal Total { get; init; } } var orders = connection.Query( "SELECT Id, CreatedAt, Total FROM Orders" ).ToList(); // Dapper assigns to init properties
Key Insight:

Records provide immutable, concise mapping targets. Structs are value types useful for small objects. Init-only properties create read-only semantics. Choose based on your domain model needs, not just convenience.

Records (C# 9+) are ideal for DTOs and immutable value objects. They support positional construction, which Dapper uses automatically. Structs are value types, useful for small, frequently-copied objects. Init-only properties (init accessor, C# 9+) create immutability after construction. All work seamlessly with Dapper's mapping.

C#: Advanced Record and Value Object Patterns // Record with positional constructor public record CustomerRecord( int Id, string FirstName, string LastName, string Email); var customers = connection.Query( "SELECT Id, FirstName, LastName, Email FROM Customers" ).ToList(); // Dapper constructs records via positional parameters // Record with named parameters (alternative syntax) public record Product { public int Id { get; init; } public string Name { get; init; } public decimal Price { get; init; } } var products = connection.Query( "SELECT Id, Name, Price FROM Products" ).ToList(); // Init properties provide immutability // Nested record public record OrderData( int OrderId, CustomerRecord Customer, decimal Total) { // Note: nested objects don't auto-populate in Dapper // You'd need to handle this with projection or multi-mapping } // Value type for IDs (strong typing) public readonly record struct CustomerId(int Value); public readonly record struct OrderId(int Value); // Prevents mixing Customer IDs and Order IDs by mistake public class OrderWithStrongTypes { public OrderId Id { get; init; } public CustomerId CustomerId { get; init; } public DateTime CreatedAt { get; init; } } // Mapping with value record structs var orders = connection.Query( "SELECT Id, CustomerId, CreatedAt FROM Orders" ).ToList(); // Dapper creates CustomerId(value) and OrderId(value) automatically

QueryMultiple: Multiple Result Sets

QueryMultiple executes multiple SQL queries and returns their results together in a single round trip to the database. This is useful when you need multiple datasets: customers and their orders, or product list and categories. You read results sequentially via Read().

C#: QueryMultiple Example var sql = @" SELECT Id, FirstName, Email FROM Customers; SELECT Id, CustomerId, OrderDate, Total FROM Orders; SELECT COUNT(*) FROM Products; "; using (var multi = connection.QueryMultiple(sql)) { var customers = multi.Read().ToList(); var orders = multi.Read().ToList(); var productCount = multi.ReadSingle(); Console.WriteLine($"Customers: {customers.Count}"); Console.WriteLine($"Orders: {orders.Count}"); Console.WriteLine($"Products: {productCount}"); } // All three queries execute in one round trip // More efficient than three separate connections
Key Insight:

QueryMultiple batches multiple SQL queries in one round trip. No connection overhead between queries. Ideal for fetching related data: a customer and their orders, a product and its reviews. More efficient than multiple separate Query() calls.

Network round trips are expensive. Each Query() call opens a connection, executes, reads results, and closes. QueryMultiple executes all queries in one connection. The results are read sequentially via Read(), ReadSingle(), or ReadSingleOrDefault(). This is essential for performance-conscious code.

C#: Advanced QueryMultiple Patterns // Complex multi-query scenario var sql = @" -- Query 1: Get customer SELECT Id, FirstName, Email FROM Customers WHERE Id = @CustomerId; -- Query 2: Get their orders SELECT Id, OrderDate, Total FROM Orders WHERE CustomerId = @CustomerId ORDER BY OrderDate DESC; -- Query 3: Get product recommendations SELECT p.Id, p.Name, p.Price FROM Products p WHERE p.CategoryId IN ( SELECT DISTINCT CategoryId FROM OrderItems oi JOIN Orders o ON oi.OrderId = o.Id WHERE o.CustomerId = @CustomerId ) LIMIT 5; -- Query 4: Get summary stats SELECT COUNT(*) as TotalOrders, SUM(Total) as LifetimeValue FROM Orders WHERE CustomerId = @CustomerId; "; using (var multi = connection.QueryMultiple(sql, new { CustomerId = 5 })) { var customer = multi.ReadSingle(); var orders = multi.Read().ToList(); var recommendations = multi.Read().ToList(); var stats = multi.ReadSingle(); Console.WriteLine($\"{customer.FirstName} ({orders.Count} orders)\"); Console.WriteLine($\"Lifetime value: {stats.LifetimeValue:C}\"); } // Best practice: reusable method public (Customer, List, List, OrderStats) GetCustomerData(int customerId) { var sql = @"[multi-query SQL above]"; using (var multi = connection.QueryMultiple(sql, new { customerId })) { var customer = multi.ReadSingle(); var orders = multi.Read().ToList(); var products = multi.Read().ToList(); var stats = multi.ReadSingle(); return (customer, orders, products, stats); } }
Key Insight:

QueryMultiple reduces network round trips and improves performance. Batch related queries together. Ideal for fetching a parent with related children, or fetching summary data alongside detailed data.

07

Dynamic Results and When to Use Them

Dapper can return dynamic objects instead of typed classes. No class definition needed. Dynamic is flexible but loses IntelliSense and type safety. Use it for reporting queries, exploratory data work, or legacy adapters where you don't know the schema in advance. For production APIs and domain code, always use typed objects.

When you call Query without a generic type (or use Query), Dapper returns dynamic objects. You access columns by property name: row.FirstName, row.Email. If you misspell a column name, you get a runtime error—there's no compile-time checking. This flexibility comes at a cost: slower due to runtime reflection and harder to debug and maintain.

C#: Dynamic Mapping // Dynamic: no class needed var customers = connection.Query("SELECT * FROM Customers").ToList(); dynamic first = customers[0]; Console.WriteLine(first.FirstName); // No intellisense // ExpandoObject: dict-like dynamic var customers2 = connection.Query( "SELECT * FROM Customers" ).ToList(); IDictionary row = customers2[0]; Console.WriteLine(row["FirstName"]); // Dictionary: explicit key-value var customers3 = connection.Query>( "SELECT * FROM Customers" ).ToList(); Console.WriteLine(customers3[0]["FirstName"]); // Pros: flexible, no class needed // Cons: no type checking, slower, no intellisense // Use: exploratory queries, dynamic reports, legacy adapters
Key Insight:

Dynamic objects are flexible but sacrifice type safety and performance. Use typed objects for production code. Use dynamic only for truly dynamic scenarios (reporting, exploratory queries, legacy adapters).

Dynamic results are appropriate in a few scenarios: building a generic data export tool (where schema is unknown), ad-hoc reporting (users query arbitrary tables), or adapting legacy systems. For all permanent business code, use typed objects. Types provide compile-time checking, IntelliSense, performance, and maintainability.

C#: Typed vs Dynamic Comparison // TYPED: Best for production code public class Customer { public int Id { get; set; } public string FirstName { get; set; } public string Email { get; set; } } var typedCustomers = connection.Query( "SELECT Id, FirstName, Email FROM Customers" ).ToList(); foreach (var c in typedCustomers) { Console.WriteLine(c.FirstName); // IntelliSense ✓ // Compile-time checking ✓ // Fast ✓ } // DYNAMIC: Only for special cases var dynamicCustomers = connection.Query( "SELECT Id, FirstName, Email FROM Customers" ).ToList(); foreach (var c in dynamicCustomers) { Console.WriteLine(c.FirstName); // No IntelliSense // Runtime errors if column misspelled // Slower due to reflection } // RULE: Use typed for 95% of code, dynamic for 5% (special cases)

Key Takeaways

  • Dapper's mapping uses reflection to match SQL column names to C# property names, case-insensitively. The mapping is cached for performance.
  • Column names must match property names exactly (case-insensitive). FirstName matches firstname but NOT first_name.
  • Use SQL aliases to bridge naming mismatches. SELECT column_name AS PropertyName converts snake_case to PascalCase.
  • Custom type handlers register converters for types Dapper doesn't natively support (DateOnly, JSON columns, etc.).
  • Map to records (immutable), structs, classes, or even dictionaries. Choose based on your needs.
  • QueryMultiple executes multiple SQL queries in a single round trip, efficient for fetching related datasets.
  • Dynamic objects offer flexibility but lose type safety and performance. Use typed objects in production code.
  • Always verify your column names match your property names. This is the most common mapping bug.
  • Dapper's simplicity and speed come from its straightforward mapping algorithm. No magic, just convention.
  • For complex scenarios (many relationships, nested objects), consider Entity Framework. For simple CRUD with performance, Dapper wins.
  • Register custom type handlers at startup (in Startup.cs or Program.cs). They apply to all subsequent queries automatically.
  • Records provide concise, immutable mapping targets. Prefer records for DTOs and read-only data.
  • QueryMultiple is essential for performance. Batch multiple related queries in one round trip instead of separate calls.
  • Understand the impedance mismatch: SQL returns flat rows; C# has objects with properties. Dapper bridges this gap with mapping.
  • Caching is automatic and transparent. Don't worry about performance of the first query—subsequent queries use the cached mapping.
  • Test your mappings! Verify that all SQL columns map to properties correctly. A missing column becomes a null property with no warning.
  • Document unusual mappings or complex type handlers. Future maintainers need to understand why the mapping exists.
  • Dapper is fastest when you know your schema and can write precise, simple queries. The fewer columns you select, the faster the mapping.
