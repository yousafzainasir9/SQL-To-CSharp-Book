01 — THE OBJECT-RELATIONAL IMPEDANCE MISMATCH

Why Loading Strategy Matters: SQL vs ORM Perspectives

In SQL, you explicitly write every query and control exactly what data is fetched. You decide: do I need a single LEFT JOIN that loads customers and orders together, or do I execute two separate queries? The decision is deliberate and visible in your code. Entity Framework abstracts this decision into a loading strategy abstraction. When you query context.Customers, related Orders might not load automatically. You declare your intent through Include() (eager loading), virtual properties (lazy loading), or Entry().Collection() (explicit loading). This abstraction solves a deep problem: the object-relational impedance mismatch—the fundamental difference between how relational databases represent data (normalized tables with foreign keys) and how object-oriented languages represent data (objects with references to other objects).

The mismatch manifests when loading related data. In SQL, you write joins. In C#, you access properties. An ORM bridges this gap by providing loading strategies that translate property access into appropriate SQL queries. Wrong strategy choices cause severe performance degradation: N+1 problems from lazy loading in loops, cartesian explosion from too many eager joins, or unnecessary memory usage from loading data you don't need. Understanding these strategies is essential for building performant applications because the performance impact can be dramatic—the difference between a 50ms query and a 10-second application freeze.

SQL Explicit Control 
-- You decide the strategy
-- Option 1: Single query with JOIN
SELECT c.Id, c.Name, o.Id, o.Total
FROM Customers c
LEFT JOIN Orders o ON c.Id = o.CustomerId
WHERE c.IsActive = 1;

-- Option 2: Multiple queries
SELECT * FROM Customers WHERE IsActive = 1;
SELECT * FROM Orders
  WHERE CustomerId IN (...);

-- Option 3: Separate targeted query
SELECT * FROM Customers WHERE IsActive = 1;
-- Then later:
SELECT * FROM Orders WHERE CustomerId = 5;
EF Declares Intent via Strategy 
// Eager: "Load all related data upfront"
var customers = context.Customers
  .Include(c => c.Orders)
  .Where(c => c.IsActive)
  .ToList();

// Lazy: "Load on access automatically"
var customers = context.Customers
  .Where(c => c.IsActive)
  .ToList();
var orders = customers[0].Orders; // Trigger!

// Explicit: "Load when I say so"
var customer = context.Customers.Find(1);
await context.Entry(customer)
  .Collection(c => c.Orders)
  .LoadAsync();
LINQ Query Syntax Alternative 
// Eager with query syntax
var customers = from c in context.Customers
                  .Include(c => c.Orders)
                where c.IsActive
                select c;

// Query syntax doesn't hide the Include call
// Include() is a method that must be called
// even in query syntax
var result = customers.ToList();
Impedance Mismatch Resolved Through Abstraction

SQL requires explicit query writing; EF requires declaring loading intent. The mismatch is that databases optimize for JOINs across tables, but objects are designed for navigable references. EF's loading strategies bridge this by mapping property access patterns to appropriate SQL. Understand your access patterns first, then choose your strategy: always need Orders with Customers? Use eager loading. Sometimes access Orders? Use lazy loading (carefully) or explicit loading. Never need Orders? Use projection to select only what's needed.

The three core strategies map to fundamentally different SQL patterns and performance characteristics. Eager loading generates a single query with JOINs (AsSingleQuery, default) or multiple queries (AsSplitQuery). Lazy loading generates the initial query, then additional queries on-demand when navigation properties are accessed. Explicit loading defers loading until you explicitly call LoadAsync() or Load(). Each has profound tradeoffs in terms of database roundtrips, memory usage, data duplication, and overall query execution efficiency. Choosing wisely can be the difference between a responsive application and one that struggles under load.

02 — EAGER LOADING WITH INCLUDE

Loading All Related Data Upfront via Include and ThenInclude

Eager loading requests all related data in the initial query via the Include() method. ThenInclude() chains further to load nested relationships multiple levels deep. EF generates a single query with LEFT JOINs for each Include level (default), or multiple queries if AsSplitQuery is enabled. The result is a single roundtrip to the database and complete object graphs available in memory immediately. Eager loading is optimal when you know you will access the related data because it prevents N+1 problems by loading everything upfront in the initial query, eliminating hidden additional queries later.

Include() is fluent and chainable: call it multiple times to include different navigation properties at the same level. ThenInclude() goes deeper into nested relationships: after Include(c => c.Orders), you can call ThenInclude(o => o.OrderItems). The generated SQL contains LEFT JOINs that combine parent and child data in a single result set that EF then materializes into object graphs with all relationships intact.

SQL Generated JOINs 
-- Eager loading generates JOINs
SELECT c.Id, c.Name, c.Email,
       o.Id, o.OrderDate, o.Total,
       oi.Id, oi.ProductId, oi.Quantity
FROM Customers c
LEFT JOIN Orders o
  ON c.Id = o.CustomerId
LEFT JOIN OrderItems oi
  ON o.Id = oi.OrderId
WHERE c.IsActive = 1
ORDER BY c.Id, o.Id, oi.Id;
C# Include with ThenInclude 
// Method syntax for nested includes
var customers = context.Customers
  .Include(c => c.Orders)
    .ThenInclude(o => o.OrderItems)
  .Include(c => c.Contacts)
  .Where(c => c.IsActive)
  .ToList();

// Now access data without additional queries
foreach (var customer in customers)
{
  Console.WriteLine(customer.Name);
  foreach (var order in customer.Orders)
  {
    Console.WriteLine($"  {order.OrderDate}: {order.Total}");
    foreach (var item in order.OrderItems)
    {
      Console.WriteLine($"    Item: {item.ProductId}");
    }
  }
}
LINQ Query Syntax Alternative 
// Method syntax is more intuitive for Include
// but can be mixed with query syntax
var customers = from c in context.Customers
                  .Include(c => c.Orders)
                    .ThenInclude(o => o.OrderItems)
                  .Include(c => c.Contacts)
                where c.IsActive
                select c;

// Or apply Include after the query
var customers = (from c in context.Customers
                 where c.IsActive
                 select c)
  .Include(c => c.Orders)
  .ThenInclude(o => o.OrderItems)
  .ToList();
Include Chains Translate to Multiple JOINs

Each Include() becomes a LEFT JOIN in SQL. Multiple includes on the same parent create multiple JOINs. Critical: cartesian explosion can occur when you include multiple collections. If Customers has 10 Orders and each Order has 5 OrderItems, a single-query join returns 50 rows per customer with heavy duplication of Customer and Order data. For 100 customers this is 5000 rows. AsSplitQuery eliminates this by executing 3 separate queries instead of 1 large JOIN.

Eager loading is powerful because it prevents N+1 query problems that commonly occur with lazy loading. However, you must be careful about loading too much data or causing cartesian explosion with multiple levels of includes. Always profile to ensure your eager loaded query is efficient and uses the appropriate split-query strategy for your data volumes. Monitor the SQL generated by logging to ensure you understand what's being executed.

03 — LAZY LOADING SETUP AND RISKS

Virtual Properties, Proxies, and Auto-Loading Dangers

Lazy loading defers loading related data until a navigation property is accessed. It requires two things: the virtual keyword on all navigation properties, and the Microsoft.EntityFrameworkCore.Proxies package installed. When you access customer.Orders, if Orders aren't already loaded, EF creates and configures a dynamic proxy that intercepts property access and triggers a separate query automatically. This is convenient for development but dangerous in production code because it hides queries and can cause N+1 problems silently in loops where you don't expect additional queries.

Lazy loading setup is deceptively simple: mark navigation properties as virtual and call UseLazyLoadingProxies() on the DbContextOptions builder. But the danger is equally simple: a loop iterating 100 customers accessing customer.Orders triggers 100 hidden queries you might not notice. The application works fine during development with small datasets, then experiences severe performance degradation in production under load. This is why many teams disable lazy loading by default and opt for explicit loading or eager loading strategies.

Lazy Loading Configuration 
// Step 1: Install Microsoft.EntityFrameworkCore.Proxies
// dotnet add package Microsoft.EntityFrameworkCore.Proxies

// Step 2: Mark navigation properties as virtual
public class Customer
{
  public int Id { get; set; }
  public string Name { get; set; }
  // Virtual allows lazy loading via proxy
  public virtual ICollection Orders { get; set; }
  public virtual Address Address { get; set; }
}

public class Order
{
  public int Id { get; set; }
  public int CustomerId { get; set; }
  public DateTime OrderDate { get; set; }
  // Must be virtual for lazy loading
  public virtual ICollection OrderItems { get; set; }
}
Enable Lazy Loading in Context 
public class AppDbContext : DbContext
{
  protected override void OnConfiguring(
    DbContextOptionsBuilder options)
  {
    options
      .UseSqlServer("connection-string")
      // Enable lazy loading proxies
      .UseLazyLoadingProxies();
  }
}

// Usage - lazy loading happens automatically
var customer = context.Customers.Find(1);
// At this point, Orders are NOT loaded

// Accessing Orders triggers a query
var orders = customer.Orders;
// Query executes: SELECT * FROM Orders
//   WHERE CustomerId = 1
// Now Orders are loaded
LINQ Query Syntax With Lazy Loading 
// Query syntax works with lazy loading
var customer = (from c in context.Customers
                where c.Id == 1
                select c).FirstOrDefault();

// Lazy loading still works after query execution
// Accessing Orders triggers automatic query
var orders = customer.Orders;

// But in loops, lazy loading creates N+1
var customers = (from c in context.Customers
                 select c).ToList();

foreach (var c in customers)
{
  // Each access triggers a query!
  var count = c.Orders.Count();
}
Lazy Loading is Insidious in Loops: N+1 Problem

Lazy loading in loops creates N+1 problems silently without any obvious indication that multiple queries are being executed. Fetch 100 customers with one query, then access Orders for each in a loop = 100 additional queries = 101 total. This can turn a millisecond operation into a multi-second bottleneck. The code looks innocent: you're just accessing a property. But under the hood, EF is executing queries you never explicitly wrote. Disable lazy loading by default and use Include() or Explicit() instead. If you must use lazy loading, ensure you enable query logging during development to spot N+1 problems immediately.

Lazy loading can be useful in scenarios with unpredictable access patterns, such as a generic repository or a framework that doesn't know what data callers need ahead of time. However, the performance cost is usually not worth it. Most teams find explicit loading or eager loading with Include() provides better visibility and control. At minimum, always enable query logging in development to see how many queries are executing and catch lazy loading surprises early.

04 — EXPLICIT LOADING FOR CONTROL

On-Demand Loading with Entry Collection and Reference

Explicit loading gives you fine-grained, predictable control over when related data is loaded. Use context.Entry(entity).Collection() for loading collections (ICollection) or .Reference() for single navigation properties (Foreign key relationships). You can filter related data before loading or just load the entire relationship. Explicit loading is visible, predictable, and avoids lazy loading surprises. You control exactly when queries execute by explicitly calling LoadAsync() or Load(), making query execution visible in code and easy to understand.

Explicit loading is particularly useful when you want to load related data selectively based on application logic: load Orders only if the customer is in a certain status, load filtered OrderItems (only items totaling more than $100), or load a reference only when you need it. The cost is slightly more verbose code, but the benefit is complete visibility and control over database operations.

Explicit Loading Patterns and Techniques 
var customer = context.Customers.Find(1);

// Pattern 1: Load all orders for this customer
await context.Entry(customer)
  .Collection(c => c.Orders)
  .LoadAsync();

// Pattern 2: Load with filter applied during load
// (only orders over $100)
await context.Entry(customer)
  .Collection(c => c.Orders)
  .Query()
  .Where(o => o.Total > 100)
  .LoadAsync();

// Pattern 3: Load single reference (Foreign key relation)
await context.Entry(customer)
  .Reference(c => c.Address)
  .LoadAsync();

// Pattern 4: Check if already loaded (avoid reload)
var entry = context.Entry(customer);
bool ordersLoaded = entry.Collection(c => c.Orders).IsLoaded;
if (!ordersLoaded)
{
  await entry.Collection(c => c.Orders).LoadAsync();
}

// Pattern 5: Load with Include on the loaded data
await context.Entry(customer)
  .Collection(c => c.Orders)
  .Query()
  .Include(o => o.OrderItems)
  .LoadAsync();

// Synchronous version (blocking, not recommended for async APIs)
context.Entry(customer)
  .Collection(c => c.Orders)
  .Load();

// Pattern 6: Load with select/projection
var orders = await context.Entry(customer)
  .Collection(c => c.Orders)
  .Query()
  .Select(o => new { o.Id, o.Total })
  .ToListAsync();
Explicit Loading Best Practices

Explicit loading shines when you need conditional loading or filtered data. Always check IsLoaded before loading to avoid unnecessary queries. Use async/await (LoadAsync) in web applications. Combine with Query() to filter or project data before materializing into memory. This pattern is ideal for APIs where you conditionally load related data based on query parameters or user permissions.

Explicit loading is ideal for web APIs and services where access patterns are unpredictable or conditional. Instead of always eagerly loading all related data, you load it on-demand based on the specific request. This provides flexibility, good performance, and clear code that shows exactly what's being loaded and when.

05 — THE N+1 QUERY PROBLEM

Anatomy of One Parent Query Plus N Child Queries

The N+1 problem is the primary performance pitfall with ORMs like Entity Framework. You execute 1 query to fetch N parent entities, then N additional queries to fetch children for each parent (or access a property that triggers loading). This equals N+1 total queries, which causes severe performance degradation. For 100 customers, instead of 1-2 efficient queries, you run 101 queries. The database server spends most time on connection setup, network roundtrips, and query parsing rather than actual data transfer. The application becomes unresponsive as the number of roundtrips becomes the bottleneck.

The N+1 problem happens with lazy loading or when you load parents without including children, then iterate and access the navigation property. It's insidious because the code looks correct and works fine during development with small datasets (10 customers, 10 queries feels fast). Performance degradation becomes apparent only under load with large datasets (1000 customers, 1000 queries = catastrophic slowdown). Always use query logging during development to detect N+1 problems early. Profile before deploying to production.

The N+1 Problem Pattern 
// PROBLEM: N Plus One Queries
var customers = context.Customers
  .ToList(); // 1 query

foreach (var c in customers)
{
  // Each iteration triggers a query!
  var orderCount = c.Orders.Count();
  Console.WriteLine(c.Name + ": " + orderCount);
}
// Total: 1 + N queries
// For 100 customers: 101 queries!

// SQL Generated:
// Query 1: SELECT * FROM Customers
// Query 2: SELECT COUNT(*) FROM Orders
//   WHERE CustomerId = 1
// Query 3: SELECT COUNT(*) FROM Orders
//   WHERE CustomerId = 2
// ... (100 more similar queries)
Three Solutions to N+1 
// SOLUTION 1: Eager Loading Include
var customers = context.Customers
  .Include(c => c.Orders)
  .ToList(); // 1 query with JOIN

// SOLUTION 2: Projection Select
var stats = context.Customers
  .Select(c => new
  {
    c.Name,
    OrderCount = c.Orders.Count()
  })
  .ToList(); // 1 query with subquery

// SOLUTION 3: Explicit Loading
var customers = context.Customers.ToList();
await context.Entry(customers[0])
  .Collection(c => c.Orders)
  .LoadAsync(); // Load specific customer only

// SOLUTION 4: Split Query with GroupBy
var stats = context.Orders
  .GroupBy(o => o.CustomerId)
  .Select(g => new
  {
    CustomerId = g.Key,
    OrderCount = g.Count()
  })
  .ToList();
// Then join with customers client-side
Real-World Impact Example

N+1 with 100 customers: 101 queries × 10ms per roundtrip = 1010ms minimum (1 second). Include() with JOIN: 1 query × 100ms for larger result set = 100ms. Projection: 1 query × 15ms with subquery = 15ms. The difference compounds under load: at 100 concurrent users, 1 second becomes 100 seconds total database time. The server becomes resource-starved and becomes unresponsive to all users.

The N+1 problem is prevented by using eager loading with Include() or projection with Select(). Always enable query logging during development to see the actual queries being executed. Write integration tests that assert on query counts: ensure your repository methods execute the expected number of queries. If you see N+1 patterns in logs, refactor immediately before committing code.

06 — DETECTING N+1 WITH QUERY LOGGING

Enable Logging to Spot Hidden Queries

The N+1 problem is silent—there's no error, just performance degradation. You discover it through query logging: enabling EF Core to output executed queries to console, file, or structured logs. You can also set breakpoints and inspect context.ChangeTracker to see what queries have been executed. Profiling tools can capture database metrics showing unexpected query counts and slow requests.

Enable Console Query Logging 
var options = new DbContextOptionsBuilder()
  .UseSqlServer("connection-string")
  .LogTo(
    Console.WriteLine,
    LogLevel.Information)
  .EnableSensitiveDataLogging()
  .Options;

using var context = new AppDbContext(options);

// Now every query prints to console
var customers = context.Customers.ToList();
// Output: SELECT [c].[Id], [c].[Name] FROM [Customers]

// And lazy loading queries appear:
var orders = customers[0].Orders;
// Output: SELECT [o].[Id], [o].[OrderDate]...
//   FROM [Orders] WHERE [o].[CustomerId] = 1
Integration Test to Assert Query Count 
// NUnit/xUnit test
[Test]
public async Task GetCustomers_ShouldExecuteOnlyOneQuery()
{
  var listener = new DiagnosticListener("Microsoft.EntityFrameworkCore.Database.Command");
  int queryCount = 0;

  listener.Subscribe(new QueryCountDiagnosticObserver(
    () => queryCount++));

  using var context = CreateTestContext();
  var customers = await context.Customers
    .Include(c => c.Orders)
    .ToListAsync();

  Assert.That(queryCount, Is.EqualTo(1),
    "Expected 1 query but got " + queryCount);
  Assert.That(customers, Has.Count.GreaterThan(0));
}

// Test fails if queries > expected
// Catches N+1 problems in CI/CD
Make Query Logging Part of Development Workflow

Enable query logging in development, write tests that assert on query counts, and review log output regularly. This catches N+1 problems early before they reach production. Many teams add query-count assertions to integration tests: if a method is supposed to execute 2 queries and it executes 10, the test fails immediately.

07 — ASSPLITQUERY VS ASSINGLQUERY

Tradeoffs: Single Roundtrip with Cartesian Explosion vs Multiple Roundtrips with Clean Data

Include() with multiple levels causes cartesian explosion: data duplication in the result set when one parent has multiple children. AsSingleQuery (default) executes one query with all JOINs, minimizing roundtrips but returning large result sets with duplication. AsSplitQuery executes separate queries per relationship, avoiding duplication but requiring more roundtrips. Choose based on your data volume, network latency, and database server load. For small result sets, AsSingleQuery is faster. For large result sets with many includes, AsSplitQuery is often more efficient.

The tradeoff is fundamental: single queries with multiple JOINs reduce roundtrips (good for high-latency networks) but increase result set size with duplication (bad for memory and bandwidth). Multiple queries eliminate duplication but increase roundtrips (bad for high-latency networks, good for low-latency local connections). Profile both strategies with your actual data and network conditions to determine which is faster.

AsSingleQuery Default Strategy 
// Default: Single query with all JOINs
var customers = context.Customers
  .Include(c => c.Orders)
  .Include(c => c.Contacts)
  .AsSingleQuery()
  .ToList();

// SQL Generated (simplified):
// SELECT c.*, o.*, ct.* FROM Customers c
//   LEFT JOIN Orders o ON c.Id = o.CustomerId
//   LEFT JOIN Contacts ct ON c.Id = ct.CustomerId

// Result: 100 customers × 5 orders × 3 contacts
//   = 1500 rows (same Customer repeated 15 times!)
// Result set size: ~150KB
// Memory usage: High (duplication)
// Roundtrips: 1 (fast for high-latency networks)
// Processing: EF must deduplicate rows to reconstruct graphs
AsSplitQuery Split Strategy 
// Alternative: Multiple queries, no duplication
var customers = context.Customers
  .Include(c => c.Orders)
  .Include(c => c.Contacts)
  .AsSplitQuery()
  .ToList();

// SQL Generated (3 queries):
// Query 1: SELECT * FROM Customers
// Query 2: SELECT * FROM Orders
//   WHERE CustomerId IN (1,2,3,...100)
// Query 3: SELECT * FROM Contacts
//   WHERE CustomerId IN (1,2,3,...100)

// Result: 100 + 500 + 300 = 900 rows (no duplication!)
// Result set size: ~90KB
// Memory usage: Low (clean data, no duplication)
// Roundtrips: 3 (slower for high-latency networks)
// Processing: EF simply matches IDs, fast deduplication
LINQ Query Syntax with Split Query 
// Query syntax with AsSplitQuery
var customers = (from c in context.Customers
                 .Include(c => c.Orders)
                 .Include(c => c.Contacts)
                 .AsSplitQuery()
                 select c)
  .ToList();

// Same result as method syntax
// AsSplitQuery can be called on IQueryable
// before or after select
When to Use Each Strategy

Use AsSingleQuery for small result sets with few includes (avoids roundtrips). Use AsSplitQuery for large result sets with many includes (avoids data duplication and memory waste). Always profile both to determine which is faster for your specific data volume and network conditions. For Azure SQL, AsSingleQuery is often preferred (low latency). For distant servers or APIs, AsSplitQuery may be better (result set size matters). Monitor query execution time and result set sizes in production to validate your choice.

The choice between strategies is data-dependent and requires profiling. Some rules of thumb: if your result set is under 1000 rows, AsSingleQuery is usually faster. If your result set is over 10,000 rows with heavy duplication, AsSplitQuery is usually better. Always measure, don't guess. Use SQL Server profiler or EF Core logging to see query times and result set sizes. Then make an informed decision based on your actual workload.

08 — PROJECTION AS ADVANCED LOADING STRATEGY

Select for Optimal Data Shapes and Client-Side Processing

Projection with Select() is often more efficient than Include() because it generates targeted SQL that fetches exactly what you need. Instead of loading entire entity graphs with all properties, you select specific properties and computed values. This reduces data transfer and memory usage, and forces you to think about what data you actually need. The generated SQL is often more efficient because EF can push aggregates and filters into the database query, avoiding client-side LINQ to Objects processing.

Projection is particularly useful for APIs, reports, and read-only scenarios where you don't need full entity objects, just specific data for the response. The smaller result sets and more efficient SQL often make projection faster than Include() approaches. You pay a small cost in code verbosity (defining DTOs), but gain significant performance benefits and memory savings.

Include vs Projection Performance 
// Include: Loads entire objects
var customers = context.Customers
  .Include(c => c.Orders)
  .ToList();

// Memory: Full Customer and Order entities,
//   all properties, change tracking overhead
// SQL: SELECT c.*, o.* (all columns)
// Usage: When you need mutable entities

// Projection: Loads only needed data
var summaries = context.Customers
  .Select(c => new CustomerSummary
  {
    Id = c.Id,
    Name = c.Name,
    OrderCount = c.Orders.Count(),
    TotalSpent = c.Orders.Sum(o => o.Total),
    RecentOrders = c.Orders
      .Where(o => o.OrderDate >
        DateTime.Now.AddYears(-1))
      .Select(o => new { o.Id, o.Total })
      .ToList()
  })
  .ToList();

// Memory: Only DTO properties
// SQL: SELECT c.Id, c.Name, (SELECT COUNT(*)...),
//        (SELECT SUM(...)...)
// Usage: API responses, reports, read-only scenarios
Complex Projection with Nested Data 
var results = context.Customers
  .Select(c => new
  {
    c.Id,
    c.Name,
    OrderStats = new
    {
      Total = c.Orders.Count(),
      Revenue = c.Orders.Sum(o => o.Total),
      Average = c.Orders.Average(o => o.Total)
    },
    RecentOrders = c.Orders
      .Where(o => o.OrderDate >=
        DateTime.Now.AddMonths(-1))
      .OrderByDescending(o => o.OrderDate)
      .Take(5)
      .Select(o => new
      {
        o.Id,
        o.OrderDate,
        o.Total,
        ItemCount = o.OrderItems.Count()
      })
      .ToList()
  })
  .ToList();

// SQL: Single query with subqueries
// Result: Only needed data, lightweight DTOs
// Memory: Minimal, no change tracking overhead
LINQ Query Syntax Projection 
var summaries = from c in context.Customers
                select new CustomerSummary
                {
                  Id = c.Id,
                  Name = c.Name,
                  OrderCount = c.Orders.Count(),
                  TotalSpent = c.Orders.Sum(o => o.Total)
                };

// Lazy - not executed until ToList()
var list = summaries.ToList();

// Or compose projections
var filtered = from s in summaries
               where s.TotalSpent > 1000
               select s;

var final = filtered.ToList();
Projection Generates Better SQL Than Include

Projection often generates better SQL than Include() because EF pushes aggregates and filters to the database via SQL subqueries and WHERE clauses, not client-side LINQ to Objects processing. The result is smaller result sets, less data transfer, less memory usage, and no change tracking overhead. Perfect for APIs and read-only scenarios.

Projection is ideal for web APIs, reports, and any scenario where you don't need mutable entity objects. Define simple DTOs containing only the properties you need, then use Select() to project entities to DTOs. This pattern is highly efficient and produces clean separation between your persistence layer (entities) and your API layer (DTOs).

09 — FILTERED INCLUDES (EF CORE 5+)

Selective Loading of Related Data Based on Predicates

EF Core 5.0 introduced filtered includes: you can now filter related data during the Include() call using Where(). This allows loading only relevant child records instead of all children. For example, load only recent orders instead of all orders, or only comments with positive ratings. The filter is applied during the query execution, so only filtered data is retrieved from the database and loaded into memory. This is more efficient than loading all children then filtering client-side in LINQ to Objects.

Filtered Includes with Where 
// EF Core 5+ only
// Load customers with only recent orders
var customers = context.Customers
  .Include(c => c.Orders
    .Where(o => o.OrderDate >= DateTime.Now.AddMonths(-3))
    .OrderByDescending(o => o.OrderDate))
  .ToList();

// Load orders with only expensive items
var orders = context.Orders
  .Include(o => o.OrderItems
    .Where(oi => oi.Price > 100))
  .ToList();

// Nested filtered includes
var customers = context.Customers
  .Include(c => c.Orders
    .Where(o => o.Total > 500))
    .ThenInclude(o => o.OrderItems
      .Where(oi => oi.Quantity > 1))
  .ToList();

// Compare to manual filtering (inefficient)
var customers = context.Customers
  .Include(c => c.Orders)
  .ToList();
// Then filter client-side (loads ALL orders first)
var filtered = customers
  .Select(c => new
  {
    c.Id,
    c.Name,
    Orders = c.Orders
      .Where(o => o.OrderDate >= DateTime.Now.AddMonths(-3))
      .ToList()
  })
  .ToList();
// This loads all orders then filters, inefficient!

// SQL generated by filtered include:
// SELECT c.*, o.* FROM Customers c
//   LEFT JOIN Orders o ON c.Id = o.CustomerId
//     AND o.OrderDate >= '2024-01-01'
//   ORDER BY c.Id, o.OrderDate DESC
Filtered Includes Push Filters to SQL

Filtered includes eliminate the inefficiency of loading all related data then filtering client-side. The filter is applied in the SQL WHERE clause, so only relevant data is returned from the database. This can significantly reduce result set size and memory usage, especially when you're filtering out large amounts of data.

10 — LOADING STRATEGIES COMPARISON TABLE

Choose Your Strategy: Eager vs Lazy vs Explicit vs Projection

Each loading strategy has different performance characteristics, tradeoffs, and use cases. This table summarizes the key differences to help you choose the right strategy for your scenario.

Strategy Queries Roundtrips Memory Best For Risks
Eager Loading (Include) 1 with JOINs 1 High (duplication) Small-medium result sets, always need related data Cartesian explosion with many includes
Eager + AsSplitQuery Multiple (no JOIN) N+1 Low (clean data) Large result sets, many includes, low cartesian product High-latency networks suffer from roundtrips
Lazy Loading 1 + N (hidden) N+1 Low initially Unpredictable access patterns N+1 in loops (DANGER!)
Explicit Loading 1 + selective 1+ Low (load only needed) Conditional loading, selective filtering More verbose code
Projection (Select) 1 optimized 1 Very low (DTOs only) APIs, reports, read-only, specific columns No entities (read-only), requires DTOs
Decision Framework: Choose Your Strategy

Ask these questions: (1) Do I always need related data? Use Include(). (2) Will result set be large with many includes? Use AsSplitQuery. (3) Do I need only specific data for API response? Use Projection. (4) Is access pattern unpredictable but I need control? Use Explicit Loading. (5) Do I need mutable entities with change tracking? Avoid Projection. (6) Is performance critical? Avoid Lazy Loading. Default to Explicit or Projection, use Include() only when you know you always need related data.

12 — DEEP THEORY: OBJECT-RELATIONAL MAPPING AND MATERIALIZATION

How ORMs Solve the Impedance Mismatch Through Loading Strategies

The object-relational impedance mismatch is the fundamental problem ORMs exist to solve. Databases organize data as normalized relational tables with foreign keys. Objects organize data as interconnected graphs with references. Mapping from relational to object requires solving several problems: (1) How to reconstruct object graphs from normalized table rows? (2) When to load related objects? (3) How to track changes to objects and map them back to relational updates? Loading strategies are the answer to #2.

When EF executes a query like context.Customers.Include(c => c.Orders), several things happen: (1) EF generates SQL with a JOIN. (2) The database returns rows: each row contains Customer columns repeated for each Order. (3) EF's materialization engine reads rows and reconstructs object graphs: creates Customer objects, creates Order objects, links them through the Orders collection. (4) The change tracker registers each object for potential updates. The choice of loading strategy directly affects this process: eager loading materializes all relationships in one query, lazy loading defers materialization until access, explicit loading lets you control when materialization happens.

How EF Materializes Object Graphs 
// Simple query with Include
var customers = context.Customers
  .Include(c => c.Orders)
  .Where(c => c.IsActive)
  .ToList();

// Behind the scenes:
// 1. EF translates to SQL with JOIN
// 2. Database returns rows (Customer data repeated per Order)
// 3. EF's materialization engine processes rows:
//    - Reads Customer row 1, creates Customer object 1
//    - Reads Order row 1, creates Order object 1,
//      links to Customer 1
//    - Reads Order row 2, creates Order object 2,
//      links to Customer 1
//    - Reads Customer row 2, creates Customer object 2
//    - Reads Order row 3 (CustomerId=2), links to Customer 2
// 4. Change tracker registers each object
// 5. Returns List with fully populated Orders collections

// The key insight: EF deduplicates Customer objects
// It sees Customer ID 1 twice (repeated with orders)
// But creates only one Customer object
// Both orders reference the same Customer 1 object

// This is why lazy loading with loops is dangerous:
// Loop doesn't have Orders loaded
// Accessing customer.Orders triggers a new query
// But that query is hidden from your code

// And why projection is efficient:
// No object graph materialization needed
// No change tracking overhead
// Just simple DTO construction
Materialization, Deduplication, and Change Tracking

EF's materialization engine is sophisticated: it deduplicates parent objects (avoids creating the same Customer twice), reconstructs relationships correctly, and registers each object with the change tracker. This complexity is hidden from you, but understanding it helps you choose better loading strategies. Lazy loading works because proxies intercept property access and trigger materialization on-demand. Include() works because EF materializes everything upfront. Projection is efficient because it skips materialization entirely. Each strategy has different performance characteristics because materialization, deduplication, and change tracking have costs.

Understanding how ORMs work internally helps you make better choices about loading strategies. You're not just choosing between Include() and Projection arbitrarily; you're choosing between different materialization strategies with different performance tradeoffs. Include() materializes everything immediately but deduplicates well and provides mutable objects. Projection skips materialization but gives you read-only DTOs. Lazy loading defers materialization but hides queries. Each choice has consequences for performance, memory, and code clarity.

13 — BEST PRACTICES AND ANTI-PATTERNS

What to Do and What to Avoid for Optimal Loading Strategy Usage

Loading strategies are powerful but easy to use incorrectly. Follow these best practices to avoid common pitfalls and build performant applications.

Best Practices and Common Pitfalls 
// BEST PRACTICE 1: Default to explicit loading or projection
// Only use Include() when you know you always need related data
var customers = context.Customers
  .Select(c => new CustomerDTO
  {
    Id = c.Id,
    Name = c.Name,
    OrderCount = c.Orders.Count()
  })
  .ToList(); // Efficient projection

// ANTI-PATTERN 1: Lazy loading in loops
// NEVER do this (generates N+1 queries)
var customers = context.Customers.ToList();
foreach (var c in customers)
{
  var orders = c.Orders; // DANGER: query per customer!
}

// BEST PRACTICE 2: Disable lazy loading globally
public class AppDbContext : DbContext
{
  protected override void OnConfiguring(DbContextOptionsBuilder options)
  {
    options
      .UseSqlServer("connection-string")
      // Don't enable lazy loading
      // .UseLazyLoadingProxies()
      ;
  }
}
// Force explicit loading or Include()

// BEST PRACTICE 3: Enable query logging in development
// Catch N+1 problems immediately
var options = new DbContextOptionsBuilder()
  .UseSqlServer("connection-string")
  .LogTo(Console.WriteLine)
  .Options;

// ANTI-PATTERN 2: Include unnecessary data
var customers = context.Customers
  .Include(c => c.Orders)
  .Include(c => c.Contacts)
  .Include(c => c.Addresses)
  .Include(c => c.PaymentMethods)
  .Include(c => c.Preferences)
  .ToList(); // Loading 100 things you don't need!

// BEST PRACTICE 4: Load only what you need
// Use filtered includes or projection
var customers = context.Customers
  .Select(c => new
  {
    c.Id,
    c.Name,
    RecentOrders = c.Orders
      .Where(o => o.OrderDate >= DateTime.Now.AddMonths(-3))
      .Take(5)
  })
  .ToList();

// ANTI-PATTERN 3: Include with AsSingleQuery on large result sets
// Cartesian explosion, huge memory usage
var customers = context.Customers
  .Include(c => c.Orders)
  .Include(c => c.Contacts)
  .AsSingleQuery()
  .ToList(); // 100 customers × 10 orders × 5 contacts = 5000 rows!

// BEST PRACTICE 5: Use AsSplitQuery for large result sets
var customers = context.Customers
  .Include(c => c.Orders)
  .Include(c => c.Contacts)
  .AsSplitQuery()
  .ToList(); // 3 queries, 615 rows, no duplication

// BEST PRACTICE 6: Profile before and after changes
var sw = Stopwatch.StartNew();
var result = LoadCustomersOptimized();
sw.Stop();
Console.WriteLine($"Optimized: {sw.ElapsedMilliseconds}ms");

// ANTI-PATTERN 4: Lazy loading in library/framework code
// Don't assume callers will set up lazy loading proxies
// Always include data explicitly or use explicit loading

// BEST PRACTICE 7: Use entry.IsLoaded to avoid redundant loading
var customer = context.Customers.Find(1);
var entry = context.Entry(customer);

if (!entry.Collection(c => c.Orders).IsLoaded)
{
  await entry.Collection(c => c.Orders).LoadAsync();
}

// BEST PRACTICE 8: Compose queries intentionally
// Use queryable methods (Include, Where, Select) before ToList()
var query = context.Customers.AsQueryable();

if (needOrders)
{
  query = query.Include(c => c.Orders);
}

var customers = await query.ToListAsync();
Core Principles for Loading Strategy Success

First: Know your access patterns (what data will be needed?). Second: Measure (don't guess about performance). Third: Avoid lazy loading in production code (hides queries). Fourth: Use Include() only when you always need data. Fifth: Prefer explicit loading or projection for flexibility. Sixth: Enable logging in development to spot N+1 problems. Seventh: Profile with realistic data volumes. Eighth: Monitor production performance and adjust based on real-world metrics.

Common Loading Strategy Mistakes

Even experienced developers fall into loading strategy traps. Understanding these anti-patterns and their solutions will save hours of debugging and database optimization.

Mistake 1: Cartesian Explosion with Multiple Includes

When you Include() multiple one-to-many relationships, the result set multiplies. If Customer has 10 Orders and each Order has 5 OrderItems, you get 50 rows per customer.

// WRONG: Results in cartesian explosion
var customers = context.Customers
  .Include(c => c.Orders)
  .Include(c => c.Invoices)
  .Include(c => c.SalesNotes)
  .ToList(); // If each customer has 10 orders, 5 invoices, 2 notes = 100 rows per customer

Solution: Use AsSplitQuery() to execute separate queries, or use projection to fetch only needed data.

// CORRECT: AsSplitQuery avoids cartesian multiplication
var customers = context.Customers
  .Include(c => c.Orders)
  .Include(c => c.Invoices)
  .AsSplitQuery()
  .ToList(); // 3 separate queries, no cartesian explosion

// BETTER: Projection fetches exactly what you need
var customerDtos = context.Customers
  .Select(c => new CustomerDTO
  {
    Id = c.Id,
    Name = c.Name,
    OrderCount = c.Orders.Count(),
    InvoiceTotal = c.Invoices.Sum(i => i.Amount)
  })
  .ToList(); // Single query, minimal data transfer

Mistake 2: Accessing Navigation Properties After Materialization

If you call ToList() before accessing related data, lazy loading triggers queries. In a loop, this becomes an N+1 nightmare.

// WRONG: N+1 problem
var customers = context.Customers.ToList(); // 1 query
foreach (var customer in customers)
{
  var orderCount = customer.Orders.Count(); // N queries (1 per customer)
}

Solution: Include related data before calling ToList().

// CORRECT: Single query with join
var customers = context.Customers
  .Include(c => c.Orders)
  .ToList();

foreach (var customer in customers)
{
  var orderCount = customer.Orders.Count(); // No queries—data already loaded
}

Mistake 3: Using Include() with Where() on Collections

You cannot use Include() to filter related collections in a single query. This is a common misconception that leads to logic errors.

// WRONG: This doesn't filter orders—it loads all orders and the condition is ignored
var customers = context.Customers
  .Include(c => c.Orders.Where(o => o.Status == "Completed"))
  .ToList();
// EF loads ALL orders anyway; the Where() clause is evaluated in-memory

Solution: Use filtered includes (EF 5+), or project and filter explicitly.

// CORRECT: Filtered includes work with collections
var customers = context.Customers
  .Include(c => c.Orders.Where(o => o.Status == "Completed"))
  .AsSplitQuery()
  .ToList();

// ALTERNATIVE: Project to DTO with explicit filtering
var customerDtos = context.Customers
  .Select(c => new
  {
    c.Id,
    c.Name,
    CompletedOrders = c.Orders.Where(o => o.Status == "Completed").ToList()
  })
  .ToList();

Mistake 4: Lazy Loading in Multi-Tier Applications

In a web API, entities are often materialized and serialized outside the DbContext lifetime. Lazy loading proxies fail when the context is disposed.

// WRONG: Lazy loading fails after context disposal
public OrderDTO GetOrderWithCustomer(int orderId)
{
  using (var context = new AppDbContext())
  {
    var order = context.Orders.Find(orderId); // Customer not loaded
    return new OrderDTO
    {
      Id = order.Id,
      CustomerName = order.Customer.Name // Context disposed, lazy load fails!
    };
  }
}

Solution: Always explicitly load related data before exiting the context scope.

// CORRECT: Include or explicit load before disposal
public OrderDTO GetOrderWithCustomer(int orderId)
{
  using (var context = new AppDbContext())
  {
    var order = context.Orders
      .Include(o => o.Customer)
      .First(o => o.Id == orderId);

    return new OrderDTO
    {
      Id = order.Id,
      CustomerName = order.Customer.Name // Data loaded while context is alive
    };
  }
}

Mistake 5: Including Same Relation Multiple Times

Redundant includes waste resources and duplicate data in memory.

// WRONG: Orders included twice
var customers = context.Customers
  .Include(c => c.Orders)
  .Include(c => c.Orders)
  .ToList();
// Second Include() is wasted

Solution: Include each relation once, or use chained includes for deep navigation.

// CORRECT: Each relation included once
var customers = context.Customers
  .Include(c => c.Orders)
  .ThenInclude(o => o.OrderItems)
  .ToList();

// Verify no duplicates in your Include structure
// One Include per relation path

Mistake 6: Ignoring Query Logs During Development

Without query logging enabled, N+1 problems and inefficient loads are invisible. Developers guess at solutions instead of measuring.

// WRONG: No visibility into actual queries
var customers = context.Customers
  .Where(c => c.Status == "Active")
  .ToList();

foreach (var customer in customers)
{
  var orders = customer.Orders.ToList(); // Hidden queries—developer doesn't see the N+1 problem
}

Solution: Enable query logging in development to see actual SQL executed.

// CORRECT: Enable logging to identify problems
var loggerFactory = LoggerFactory.Create(builder => builder.AddConsole());
var options = new DbContextOptionsBuilder<AppDbContext>()
  .UseLoggerFactory(loggerFactory)
  .UseSqlServer(connectionString)
  .Build();

var context = new AppDbContext(options);

// Now you see every query executed
// Count queries: if N+1 exists, you'll see "N+1" queries
// Adjust strategy based on actual SQL logged

Key Takeaways

  • Loading strategy directly translates to SQL queries, roundtrips, memory usage, and overall performance
  • Eager loading with Include() executes single query with JOINs; prevents N+1 problems but can cause cartesian explosion
  • Lazy loading with virtual properties triggers queries on access; convenient but extremely dangerous in loops (N+1 problem)
  • Explicit loading gives fine-grained control via Entry().Collection().Load(); visible, predictable, no hidden queries
  • N+1 problem: 1 parent query plus N child queries; typically caused by lazy loading in loops or accessing navigation properties without eager loading
  • Detect N+1 with query logging during development; add query-count assertions to integration tests to catch regressions early
  • AsSingleQuery joins everything (1 roundtrip but cartesian explosion); AsSplitQuery uses multiple queries (more roundtrips but no duplication)
  • Projection with Select() often beats Include() by generating targeted SQL and avoiding change tracking overhead; ideal for APIs and reports
  • Filtered includes (EF 5+) allow loading selective related data with Where(); reduces result set size and memory usage vs loading all children
  • Choose strategy based on access patterns, data volume, network latency, and whether you need mutable entities with change tracking
  • Always profile with realistic data volumes; small differences are noise, look for order-of-magnitude improvements before refactoring
  • Deep principle: ORMs solve impedance mismatch by providing loading strategies; each strategy has different materialization, deduplication, and tracking costs
