Chapter 14

Loading Strategies

Requirement: "Show customer details with their orders." How do you load related data? SQL developers write JOINs. C# developers choose between eager loading (Include), lazy loading (on-access), and explicit loading (manual). Each has trade-offs.

01

Requirement: "Show Customer with All Their Orders"

Business Requirement
Customer detail page needs customer info and all orders in one query. No delays. Load everything up front.
SQL Developer Thinks SELECT with JOIN — one query gets everything. The database combines customer and order rows. I get all the data I need in a single round trip.
C# Developer Thinks .Include(c => c.Orders) — EF generates the JOIN for me. I just tell it which navigation property to load, and it creates the SQL.
SQL — JOIN
SELECT
    c.Id, c.Name, c.Email,
    o.Id, o.Total, o.OrderDate
FROM Customers c
LEFT JOIN Orders o ON c.Id = o.CustomerId
WHERE c.Id = @CustomerId
ORDER BY o.OrderDate DESC;
C# — Include
var customer = await db.Customers
    .Include(c => c.Orders)
    .FirstOrDefaultAsync(c => c.Id == id, ct);

// customer.Orders is now loaded
var orders = customer?.Orders ?? new();
C# — ThenInclude for Deep Loading
var customer = await db.Customers
    .Include(c => c.Orders)
        .ThenInclude(o => o.Items) // Order items
    .Include(c => c.Addresses) // Multiple includes
    .FirstOrDefaultAsync(c => c.Id == id, ct);

// Three levels deep: Customer > Orders > Items
var items = customer?.Orders
    .FirstOrDefault()?.Items ?? new();
02

Requirement: "Load Orders Only If User Clicks Expand"

Business Requirement
Customer list loads fast. Orders load only when user clicks to expand. Avoid loading 1000 customers with 10,000 orders if most are never viewed.
SQL Developer Thinks Run a second query when needed. First query: SELECT * FROM Customers. If user expands, second query: SELECT * FROM Orders WHERE CustomerId = @Id.
C# Developer Thinks Lazy loading proxies — navigation property triggers a query on first access. Customer loads. When code accesses customer.Orders, EF fires a second query automatically.
SQL — Two Queries
-- Query 1: Get all customers
SELECT Id, Name, Email FROM Customers;

-- Query 2: Load orders when needed
SELECT Id, Total, OrderDate
FROM Orders
WHERE CustomerId = @CustomerId;
C# — Lazy Loading
// Query 1: Get customer (no orders loaded)
var customer = await db.Customers
    .FirstOrDefaultAsync(c => c.Id == id, ct);

// Query 2: Fires automatically on access
var orders = customer?.Orders; // EF queries here

// Navigation property must be virtual
public virtual ICollection<Order> Orders { get; }
Enable Lazy Loading in DbContext
services.AddDbContext<AppDb>(options =>
{
    options.UseSqlServer(connectionString);
    // Enable lazy loading
    options.UseLazyLoadingProxies();
});
The N+1 Danger
Load 1000 customers without Include, then loop: foreach (var c in customers) var o = c.Orders; Triggers 1001 queries total (1 for customers + 1000 for each customer's orders). Lazy loading can silently cause this.
03

Requirement: "Load Related Data Conditionally"

Business Requirement
After fetching a customer, decide at runtime whether to load orders based on business logic. "Show detailed view? Load orders. Show list view? Skip orders."
SQL Developer Thinks Run a separate SELECT when needed. First: get customer. Then, if business logic says yes: run a second SELECT for orders.
C# Developer Thinks context.Entry(customer).Collection(c => c.Orders).LoadAsync() — Manual load after the fact. Fetch customer, check a flag, then explicitly load orders if needed.
SQL — Separate Query
-- Query 1: Get customer
SELECT * FROM Customers WHERE Id = @CustomerId;

-- Query 2 (conditional): Load orders only if needed
SELECT * FROM Orders
WHERE CustomerId = @CustomerId
ORDER BY OrderDate DESC;
C# — Explicit Load
var customer = await db.Customers
    .FirstOrDefaultAsync(c => c.Id == id, ct);

if (customer != null && showDetails)
{
    // Load orders explicitly only if needed
    await db.Entry(customer)
        .Collection(c => c.Orders)
        .LoadAsync(ct);
}

var orders = customer?.Orders ?? new();
Explicit Load with Filter
var customer = await db.Customers
    .FirstOrDefaultAsync(c => c.Id == id, ct);

if (customer != null)
{
    // Load only orders from the last 30 days
    await db.Entry(customer)
        .Collection(c => c.Orders)
        .Query() // Access the IQueryable
        .Where(o => o.OrderDate >= DateTime.Now.AddDays(-30))
        .LoadAsync(ct);
}

var recentOrders = customer?.Orders ?? new();
04

Requirement: "API Returns Only Specific Fields"

Business Requirement
Mobile API needs customer name and order count only. Don't load full entities. Keep payload small.
SQL Developer Thinks SELECT c.Name, COUNT(o.Id) AS OrderCount FROM Customers c LEFT JOIN Orders o ON c.Id = o.CustomerId — projection is natural. Get exactly what you need.
C# Developer Thinks .Select(c => new { c.Name, OrderCount = c.Orders.Count() }) — Projection is better than Include for APIs. You load only what you need, not full entities.
SQL — Projection
SELECT
    c.Name,
    COUNT(o.Id) AS OrderCount
FROM Customers c
LEFT JOIN Orders o ON c.Id = o.CustomerId
GROUP BY c.Id, c.Name;
C# — Include (Wrong)
// BAD: Loads full Customer and Order entities
var customers = await db.Customers
    .Include(c => c.Orders)
    .ToListAsync(ct);

var dto = customers.Select(c => new {
    c.Name,
    OrderCount = c.Orders.Count
}).ToList();
// Inefficient: returns many entities
C# — Select (Correct)
// GOOD: Loads only what you need
var customers = await db.Customers
    .AsNoTracking() // Read-only, better performance
    .Select(c => new
    {
        c.Name,
        OrderCount = c.Orders.Count()
    })
    .ToListAsync(ct);

// Returns exactly what SQL query would return
// EF generates the JOIN and COUNT in SQL, not in memory
05

Requirement: "Page Load Is Slow — Too Much Data"

Business Requirement
Customer detail page includes orders and addresses. Customer has 100 orders and 50 addresses. Include both and the query returns 5000 rows (100 × 50). Cartesian explosion.
SQL Developer Thinks JOINing two collections causes explosion. 100 orders × 50 addresses = 5000 rows, with customer data duplicated 5000 times. Use separate queries or SELECT with aggregation.
C# Developer Thinks .Include() with multiple collections hits the same problem. Use AsSplitQuery() to run separate SELECTs instead of one big JOIN.
Problem: Cartesian Explosion
-- BAD: Returns 5000 rows
SELECT
    c.Id, c.Name,
    o.Id, o.Total,
    a.Id, a.Street
FROM Customers c
LEFT JOIN Orders o ON c.Id = o.CustomerId
LEFT JOIN Addresses a ON c.Id = a.CustomerId
WHERE c.Id = @CustomerId;

-- Result: 100 orders × 50 addresses = 5000 rows
-- Customer data repeated 5000 times
C# — Split Query
var customer = await db.Customers
    .AsSplitQuery() // Run separate queries
    .Include(c => c.Orders)
    .Include(c => c.Addresses)
    .FirstOrDefaultAsync(c => c.Id == id, ct);

// EF generates:
// Query 1: SELECT * FROM Customers WHERE Id = @id
// Query 2: SELECT * FROM Orders WHERE CustomerId = @id
// Query 3: SELECT * FROM Addresses WHERE CustomerId = @id
// No duplication, 151 total rows returned
06

Which Loading Strategy When?

Scenario Strategy Why
Detail page: customer + orders + addresses Eager: .Include().AsSplitQuery() Load everything needed; split query avoids cartesian
List page: customer names only Select projection Load minimal data; efficient network payload
API endpoint: customer + order count Select { Name, Count } Exact columns needed; aggregation in SQL
Load orders conditionally (if flag = true) Explicit: Entry().Collection().LoadAsync() Manual control; load only when needed
Lazy loading: access on demand Virtual nav props + UseLazyLoadingProxies() Transparent; but watch for N+1
Read-only query: no modifications AsNoTracking() Skip change tracking; faster execution
