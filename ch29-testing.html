Testing Data Access

Test data access with in-memory databases, SQLite, and TestContainers. Mock repositories for unit tests.

In This Chapter

01

Testing Strategies

Unit tests use mocks/in-memory. Integration tests use real databases. Start with unit; add integration for critical paths.

TypeSpeedReliabilityCostBest For
Unit (Mock)FastGoodNoneBusiness logic, edge cases
Integration (In-Memory)FastGoodNoneEF Core LINQ queries
Integration (SQLite)FastBetterNoneSQL syntax verification
E2E (Real DB)SlowBestHighProduction-like scenarios
Rule:

Test database queries with actual SQL engines (SQLite, TestContainers). Never use in-memory-only for query logic.

02

In-Memory Testing

Fast unit tests for business logic that doesn't depend on SQL specifics.

In-Memory Contextvar options = new DbContextOptionsBuilder<AppDbContext>() .UseInMemoryDatabase("TestDb_" + Guid.NewGuid()) .Options; var db = new AppDbContext(options); // Seed test data db.Customers.Add(new Customer { Id = 1, Name = "Alice", Balance = 100 }); db.SaveChanges(); // Test var customer = db.Customers.FirstOrDefault(c => c.Name == "Alice"); Assert.NotNull(customer); Assert.Equal(100, customer.Balance);
Caution:

In-memory DB doesn't execute real SQL. LIKE queries, date functions, and complex filters may not work.

03

SQLite for Testing

Real SQL engine in a file. Great for testing EF queries without setting up SQL Server.

SQLite Test Setupvar connection = new SqliteConnection("DataSource=:memory:"); connection.Open(); var options = new DbContextOptionsBuilder<AppDbContext>() .UseSqlite(connection) .Options; using var db = new AppDbContext(options); db.Database.EnsureCreated(); // Seed and test db.Customers.Add(new Customer { Id = 1, Name = "Alice" }); db.SaveChanges(); var result = await db.Customers .Where(c => EF.Functions.Like(c.Name, "%ice")) // SQLite supports LIKE .ToListAsync(); Assert.Single(result);
Connection Lifetime:

Use :memory: for in-process tests. Use file path for persistent databases between test runs.

04

Mock Repositories

Unit test business logic without touching databases.

Mock Repository Patternpublic class MockCustomerRepository : ICustomerRepository { private readonly List<Customer> _customers = new(); public async Task<Customer?> GetByIdAsync(int id, CancellationToken cancellationToken = default) => await Task.FromResult(_customers.FirstOrDefault(c => c.Id == id)); public async Task<IEnumerable<Customer>> GetActiveAsync(CancellationToken cancellationToken = default) => await Task.FromResult(_customers.Where(c => c.IsActive)); public async Task AddAsync(Customer customer, CancellationToken cancellationToken = default) { _customers.Add(customer); await Task.CompletedTask; } } // Test [Fact] public async Task OrderService_CreatesOrderForActiveCustomer() { var repo = new MockCustomerRepository(); repo.AddAsync(new Customer { Id = 1, IsActive = true }); var service = new OrderService(repo); var order = await service.CreateOrderAsync(1, ...); Assert.NotNull(order); }
05

TestContainers

Spin up real Docker containers (SQL Server, PostgreSQL) for each test run.

TestContainers Setup// NuGet: Testcontainers, Testcontainers.MsSql [Collection("Database collection")] public class CustomerRepositoryTests : IAsyncLifetime { private readonly MsSqlContainer _container; private AppDbContext _db; public CustomerRepositoryTests() { _container = new MsSqlBuilder() .WithPassword("YourStrong@Password1") .Build(); } public async Task InitializeAsync() { await _container.StartAsync(); var connStr = _container.GetConnectionString(); var options = new DbContextOptionsBuilder<AppDbContext>() .UseSqlServer(connStr) .Options; _db = new AppDbContext(options); await _db.Database.MigrateAsync(); } public async Task DisposeAsync() => await _container.StopAsync(); [Fact] public async Task GetCustomersByRegion_ReturnsMatching() { _db.Customers.Add(new Customer { Id = 1, Region = "West" }); await _db.SaveChangesAsync(); var result = await _db.Customers .AsNoTracking() .Where(c => c.Region == "West") .ToListAsync(); Assert.Single(result); } }
When to Use:

TestContainers for CI/CD pipelines. Start with SQLite for local dev. Use containers for production-like testing.
