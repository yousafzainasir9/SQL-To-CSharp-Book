Chapter 29

Testing Data Access

Test database code without a database. Mock it, use in-memory, use SQLite, or spin up Docker containers. Learn when and how for each approach.

01

Requirement: "Test Without a Real Database"

Business Requirement
Unit tests run fast on every developer machine. No SQL Server required. But tests must catch real bugs, not false positives.
SQL Developer Thinks
Mocking is for business logic. Real SQL queries need a real SQL engine.
C# Developer Thinks
Use three tiers: mocks for logic, in-memory/SQLite for queries, TestContainers for integration tests.
ApproachSpeedAccuracyBest For
Mock RepositoryVery FastLogic onlyBusiness logic
In-Memory DBFastLINQ queriesSimple LINQ
SQLite In-MemoryFastReal SQLSQL migrations
TestContainersSlowerExactIntegration
02

In-Memory DB Testing

Business Requirement
Fast LINQ query tests. In-memory DB doesn't execute real SQL. When is it right?
SQL Developer Thinks
In-memory has no SQL engine. LIKE, date functions, complex logic may fail.
C# Developer Thinks
For simple WHERE/ORDER BY use in-memory. For SQL functions use SQLite.
In-Memory DbContext
var opts = new DbContextOptionsBuilder()
    .UseInMemoryDatabase("TestDb").Options;

var db = new AppDbContext(opts);
db.Customers.Add(new Customer { Name = "Alice" });
await db.SaveChangesAsync();

var c = await db.Customers
    .FirstOrDefaultAsync(c => c.Name == "Alice");
03

SQLite for Real SQL Testing

Business Requirement
Test real SQL. Database migrations. Date functions. Complex WHERE clauses.
SQL Developer Thinks
SQLite is a real SQL engine. It's lightweight, in-memory or file-based.
C# Developer Thinks
Use SqliteConnection with in-memory database. Run migrations. Execute real SQL syntax.
SQLite In-Memory
var conn = new SqliteConnection("DataSource=:memory:");
await conn.OpenAsync();

var opts = new DbContextOptionsBuilder()
    .UseSqlite(conn).Options;

var db = new AppDbContext(opts);
await db.Database.MigrateAsync();

db.Customers.Add(new Customer { CreatedAt = DateTime.Now });
await db.SaveChangesAsync();
04

Mock Repositories for Unit Tests

Business Requirement
Test business logic without any database. Fast unit tests for OrderService, PaymentService, etc.
SQL Developer Thinks
Mocks return fake data. No SQL executed.
C# Developer Thinks
Use Moq or create fake IRepository implementations. Fast, isolated unit tests.
Mock Repository Pattern
public class MockCustomerRepository : ICustomerRepository {
    private List<Customer> customers = new();

    public async Task<Customer> GetByIdAsync(int id, CancellationToken ct = default) {
        return await Task.FromResult(customers.FirstOrDefault(c => c.Id == id));
    }
}

[Fact]
public async Task OrderService_ProcessesOrder() {
    var repo = new MockCustomerRepository();
    repo.Add(new Customer { Id = 1, IsActive = true });

    var svc = new OrderService(repo);
    var order = await svc.CreateOrderAsync(1);

    Assert.NotNull(order);
}
