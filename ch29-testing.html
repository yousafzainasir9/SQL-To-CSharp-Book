01 — The Testing Challenge: Database Dependencies

Understanding Why Data Access Tests Are Different

Testing data access code is uniquely challenging. Unlike business logic, which is purely functional, data access involves external state (the database), persistence, constraints, and concurrent access. A unit test for a pricing algorithm can run in isolation with no dependencies. A test for a data repository has dependencies: it needs a real or simulated database, test data, schema setup, and cleanup.

The testing pyramid applies here: many fast unit tests at the bottom, fewer integration tests in the middle, and few slow end-to-end tests at the top. For data access, the pyramid inverts slightly. You can't meaningfully unit test EF Core or Dapper code without a database—you're testing ORM mapping, SQL generation, constraint enforcement, and transaction behavior. These require at least a test database.

SQL — Example: What Are We Testing? -- A data access repository would execute this SQL SELECT o.*, c.*, li.* FROM Orders o INNER JOIN Customers c ON o.CustomerId = c.Id LEFT JOIN LineItems li ON o.Id = li.OrderId WHERE o.OrderDate >= @StartDate AND o.Status = 'Completed' ORDER BY o.OrderDate DESC -- Tests need to verify: -- 1. INNER JOIN works (missing customer fails) -- 2. LEFT JOIN works (missing line items OK) -- 3. WHERE filtering works (correct records returned) // 4. ORDER BY works (correct sequence) // 5. EF tracking works (change tracking enabled) // 6. Constraints enforced (FK violations caught) // 7. Isolation level correct (concurrency works) -- These can't be tested in pure unit test // (no SQL engine without database)
C# — The Unit Test Dilemma // This class is hard to unit test: public class OrderRepository { private readonly OrderContext _context; public async Task> GetRecentCompletedOrders( DateTime since) { return await _context.Orders .Include(o => o.Customer) .Include(o => o.LineItems) .Where(o => o.OrderDate >= since && o.Status == "Completed") .OrderByDescending(o => o.OrderDate) .ToListAsync(); // Must be tested against real DB } } // Naive mocking approach (wrong): var mockContext = new Mock(); mockContext.Setup(c => c.Orders) .Returns(new[] { new Order { OrderDate = DateTime.Today, Status = "Completed" } }.AsQueryable()); // Problem: This doesn't test EF Core's LINQ translation // Doesn't test SQL generation // Doesn't test Include() behavior // Doesn't test database constraints // You're testing your mock, not the real code! // Better: Use real database in test (integration test) public async Task GetRecentCompletedOrders_ReturnsCorrectData() { // Use real or test database var context = new OrderContext(testDbOptions); // Arrange: Create test data var order1 = new Order { OrderDate = DateTime.Today }; var order2 = new Order { OrderDate = DateTime.Today }; context.Orders.AddRange(order1, order2); await context.SaveChangesAsync(); // Act: Call the method var result = await repository.GetRecentCompletedOrders( DateTime.Today.AddDays(-7)); // Assert: Verify results Assert.Equal(2, result.Count); }
Key Insight: You're Testing the ORM, Not Just Your Code

Data access tests inevitably test the ORM library (EF Core or Dapper), the SQL Server, constraint enforcement, and transaction semantics. You can't isolate your code from these concerns. This is actually good—you want to catch bugs in SQL generation, constraint violations, and concurrency issues. The tradeoff is that these tests must run against real databases (or faithful simulators like SQLite).

02 — In-Memory Database for EF Core Tests

Fast But Limited: When to Use EF Core's InMemoryDatabase

Entity Framework Core includes an in-memory database provider for testing. It's fast, requires no setup, and works great for simple scenarios. However, it doesn't implement actual SQL—it uses LINQ to Objects semantics. This means some queries work in the in-memory database but fail against real SQL, and vice versa. Use it for fast feedback during development, but move critical tests to SQLite or a real database.

SQL — Queries That Work in SQL But Not In-Memory -- In-memory database uses LINQ semantics, not SQL -- This works in SQL Server: SELECT DATEDIFF(day, OrderDate, GETDATE()) AS DaysSince FROM Orders WHERE DATEDIFF(day, OrderDate, GETDATE()) <= 30 -- In-memory database doesn't support DATEDIFF // EF would throw: NotSupportedException // DATEDIFF is SQL Server specific -- Another example: String functions SELECT * FROM Customers WHERE SUBSTRING(Name, 1, 1) = 'A' -- In-memory supports (maps to string methods) -- But complex case: SELECT * FROM Orders WHERE CAST(OrderDate AS DATE) = CAST(GETDATE() AS DATE) -- In-memory has limited CAST support

In-memory databases have significant limitations:

SQL — In-Memory Database Limitations -- Limitation 1: No actual SQL translation -- This works in SQL Server but may fail in-memory: SELECT * FROM Orders WHERE Name LIKE 'A%' -- In-memory may not support LIKE (depends on EF version) -- Limitation 2: No transactions/isolation -- In-memory database doesn't enforce isolation levels // Transaction tests won't catch real issues -- Limitation 3: No complex queries -- Recursive CTEs don't work -- Window functions limited -- Temporal tables not supported -- Better to test these against real database
C# — When NOT to Use In-Memory Database // DON'T use in-memory for these tests: // 1. Complex SQL generation (use SQLite instead) var complexResults = await context.Orders .Where(o => EF.Functions.Like(o.Name, "A%")) .ToListAsync(); // 2. Transaction/isolation level tests (use real DB) using var transaction = await context.Database.BeginTransactionAsync( IsolationLevel.Serializable); // In-memory doesn't actually enforce isolation // 3. Constraint violation tests var order = new Order { CustomerId = 999 }; // In-memory supports FK constraints, but not all DB constraints // Use SQLite for realistic constraint behavior // 4. Concurrency tests // In-memory is single-threaded effectively // Concurrency issues won't surface // Better approach for critical tests: private OrderContext CreateSQLiteContext() { var options = new DbContextOptionsBuilder() .UseSqlite("DataSource=:memory:") .Options; var context = new OrderContext(options); context.Database.EnsureCreated(); // Create schema return context; }
03 — SQLite In-Memory for Better Tests

Real SQL Without Infrastructure Complexity

SQLite in-memory mode provides the best compromise for testing: it uses real SQL (so queries work exactly like they would in production), runs fast (in-memory), requires no external setup, and still enforces constraints. The only limitations are SQLite-specific (like lack of some advanced SQL Server features), but for most tests, SQLite matches production SQL Server behavior closely enough.

SQL — SQLite vs SQL Server: Compatibility -- SQLite supports standard SQL operations well: CREATE TABLE Orders ( Id INTEGER PRIMARY KEY, CustomerId INTEGER NOT NULL, OrderDate DATETIME, FOREIGN KEY (CustomerId) REFERENCES Customers(Id) ); -- SQLite limitations vs SQL Server: -- 1. No GETDATE() (use datetime('now')) -- 2. No DATEDIFF() (use julianday() or similar) -- 3. No OFFSET/FETCH in all versions (but newer has it) -- 4. No window functions in older SQLite (newer has them) -- 5. No temporal tables, JSON functions limited -- SQLite strengths: -- 1. Real SQL (unlike in-memory EF provider) -- 2. ACID transactions work // 3. Foreign keys enforced (if enabled) // 4. Quick test execution (in-memory) // 5. Simple setup (no database server needed)

SQLite makes a good default for most unit and integration tests. Use it until you need specific SQL Server features. Here's a complete test fixture pattern:

04 — Testing Dapper Code and Repository Patterns

Interface-Based Testing and Real Database Scenarios

Dapper code is inherently more testable via mocking because you're working with lower-level abstractions (IDbConnection, SQL strings). You can mock the connection and verify SQL generation without hitting a database. However, for realistic testing, running against a real or test database is still recommended to catch SQL syntax errors and query logic bugs.

SQL — Example Dapper Query to Test -- Dapper would execute this SQL: SELECT o.*, c.*, li.* FROM Orders o INNER JOIN Customers c ON o.CustomerId = c.Id LEFT JOIN LineItems li ON o.Id = li.OrderId WHERE o.OrderDate >= @StartDate AND c.Status = 'Active' ORDER BY o.OrderDate DESC -- Tests need to verify: // 1. SQL syntax is correct // 2. JOIN conditions work // 3. WHERE filters work // 4. ORDER BY works // 5. Parameter binding works
