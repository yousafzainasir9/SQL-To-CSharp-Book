Chapter 22

Bulk Operations with Dapper

Requirement: “Import 50,000 products from a CSV file.” One-by-one inserts take minutes. Bulk operations take seconds. Here’s how to handle large datasets with Dapper and SqlBulkCopy.

01

Requirement: “Insert 1,000 Records from a Form Submission”

Business Requirement
Admin uploads a list of 1,000 new products. Each product has a name, SKU, price, and category ID. We need to insert them all quickly.
SQL Developer Thinks Multi-row INSERT: all 1,000 rows in one statement. Or use a loop with sp_executesql. But a single multi-row INSERT is fastest — one network round-trip, one transaction.
C# Developer Thinks Dapper ExecuteAsync with a list. Dapper loops the SQL for each item internally, but keeps everything in one execute call. Faster than one-by-one, simpler than building a mega-SQL statement.
SQL — Multi-Row INSERT
INSERT INTO Products (Name, Sku, Price, CategoryId)
VALUES
    ('Product 1', 'SKU-001', 29.99, 5),
    ('Product 2', 'SKU-002', 49.99, 5),
    ('Product 3', 'SKU-003', 19.99, 3),
    -- ... 997 more rows
    ('Product 1000', 'SKU-1000', 99.99, 7);
C# — Dapper Batch Execute
var products = new List<Product>
{
    new { Name = "Prod 1", Sku = "SKU-001",
          Price = 29.99m, CategoryId = 5 },
    new { Name = "Prod 2", Sku = "SKU-002",
          Price = 49.99m, CategoryId = 5 },
    // ... 998 more items
};

await using var conn = new SqlConnection(cs);
var rowsInserted = await conn.ExecuteAsync(
    @"INSERT INTO Products (Name, Sku, Price, CategoryId)
      VALUES (@Name, @Sku, @Price, @CategoryId)",
    products,
    commandTimeout: 120);
Dapper — ExecuteAsync Loops Internally
// Dapper executes the command once per item in the list
// Behind the scenes, it's roughly:
// for each product in products:
//     cmd.Parameters[@Name] = product.Name
//     cmd.Parameters[@Sku] = product.Sku
//     cmd.Parameters[@Price] = product.Price
//     cmd.Parameters[@CategoryId] = product.CategoryId
//     cmd.ExecuteNonQuery()
//
// Result: 1,000 inserts sent to SQL Server
// + PERFORMANCE NOTE: Still faster than one-by-one from C#
//   because it uses connection pooling and batched execution
// - Consider SqlBulkCopy for 50K+ rows
02

Requirement: “Update Prices for 5,000 Products”

Business Requirement
Supplier sent new prices. We have a list of 5,000 product IDs with their new prices. Update all of them.
SQL Developer Thinks Option 1: Loop 5,000 UPDATE statements. Option 2: Use a temp table with all the new prices, then JOIN and update in one batch. Option 2 is much faster.
C# Developer Thinks Option 1: Dapper ExecuteAsync with a loop (sends 5,000 separate commands). Option 2: Load data into a temp table with SqlBulkCopy, then execute one UPDATE with a JOIN. Option 2 scales better.
Dapper — ExecuteAsync Loop
var priceUpdates = new List<(int productId, decimal newPrice)>
{
    (123, 19.99m),
    (456, 24.99m),
    (789, 14.99m),
    // ... 4,997 more
};

await using var conn = new SqlConnection(cs);
foreach (var (id, price) in priceUpdates)
{
    await conn.ExecuteAsync(
        @"UPDATE Products
          SET Price = @Price
          WHERE Id = @Id",
        new { Id = id, Price = price });
}
C# — Temp Table (Faster)
var priceUpdates = new List<(int ProductId, decimal NewPrice)>
{
    (123, 19.99m), (456, 24.99m), // ... 4,998 more
};

await using var conn = new SqlConnection(cs);

// Create temp table
await conn.ExecuteAsync(
    @"CREATE TABLE #TempPrices (
        ProductId INT, NewPrice DECIMAL(10,2))");

// Bulk insert price data
using var bulkCopy = new SqlBulkCopy(conn)
{
    DestinationTableName = "#TempPrices"
};
var dt = new DataTable();
dt.Columns.Add("ProductId", typeof(int));
dt.Columns.Add("NewPrice", typeof(decimal));
foreach (var (id, price) in priceUpdates)
    dt.Rows.Add(id, price);
await bulkCopy.WriteToServerAsync(dt);

// Update from temp table
await conn.ExecuteAsync(
    @"UPDATE Products p
      SET p.Price = t.NewPrice
      FROM #TempPrices t
      WHERE p.Id = t.ProductId");

// Cleanup
await conn.ExecuteAsync("DROP TABLE #TempPrices");
03

Requirement: “Import 100,000 Records in Under 5 Seconds”

Business Requirement
Nightly data import of 100,000+ records from a partner system. Must complete in under 5 seconds. One-by-one or even batch ExecuteAsync is too slow.
SQL Developer Thinks BULK INSERT from file, or load data via BCP utility. This bypasses most query parsing and logging — pure data stream to disk.
C# Developer Thinks SqlBulkCopy streams data directly to SQL Server in bulk. It’s the .NET equivalent of BULK INSERT. Works with DataTable, DataReader, or IEnumerable.
C# — SqlBulkCopy with DataTable
public async Task BulkInsertProductsAsync(
    List<Product> products,
    CancellationToken ct = default)
{
    // Create a DataTable matching the Products schema
    var dt = new DataTable("Products");
    dt.Columns.Add("Id", typeof(int));
    dt.Columns.Add("Name", typeof(string));
    dt.Columns.Add("Sku", typeof(string));
    dt.Columns.Add("Price", typeof(decimal));
    dt.Columns.Add("Stock", typeof(int));
    dt.Columns.Add("CategoryId", typeof(int));
    dt.Columns.Add("CreatedAt", typeof(DateTime));

    // Populate DataTable from products list
    foreach (var p in products)
    {
        dt.Rows.Add(
            p.Id, p.Name, p.Sku, p.Price,
            p.Stock, p.CategoryId, DateTime.UtcNow);
    }

    // Stream to SQL Server
    await using var conn = new SqlConnection(_connectionString);
    await conn.OpenAsync(ct);

    using var bulkCopy = new SqlBulkCopy(conn)
    {
        DestinationTableName = "Products",
        BatchSize = 5000,              // Rows per batch
        BulkCopyTimeout = 300,         // Seconds
        EnableStreaming = true         // Stream mode
    };

    await bulkCopy.WriteToServerAsync(dt, ct);
}
C# — SqlBulkCopy with IDataReader (Memory-Efficient)
public async Task BulkInsertFromCsvAsync(
    string csvFilePath,
    CancellationToken ct = default)
{
    // Use CsvHelper or similar to stream from file
    using var reader = new StreamReader(csvFilePath);
    using var csv = new CsvReader(reader, CultureInfo.InvariantCulture);

    // Convert CSV to DataReader
    using var dt = csv.GetDataTable();

    await using var conn = new SqlConnection(_connectionString);
    await conn.OpenAsync(ct);

    using var bulkCopy = new SqlBulkCopy(conn)
    {
        DestinationTableName = "Products",
        BatchSize = 5000,
        BulkCopyTimeout = 300
    };

    // Column mapping: CSV column name -> Table column name
    bulkCopy.ColumnMappings.Add("ProductId", "Id");
    bulkCopy.ColumnMappings.Add("ProductName", "Name");
    bulkCopy.ColumnMappings.Add("Sku", "Sku");
    bulkCopy.ColumnMappings.Add("ListPrice", "Price");
    bulkCopy.ColumnMappings.Add("QtyInStock", "Stock");

    await bulkCopy.WriteToServerAsync(dt, ct);
}
C# — Column Mapping (Explicit)
// If source and target column names don't match:
var dt = new DataTable();
dt.Columns.Add("ProductId");       // CSV column name
dt.Columns.Add("ProductName");     // CSV column name
dt.Columns.Add("Sku");
// ... populate dt from CSV ...

using var bulkCopy = new SqlBulkCopy(conn)
{
    DestinationTableName = "Products"
};

// Map source column to destination table column
bulkCopy.ColumnMappings.Add("ProductId", "Id");
bulkCopy.ColumnMappings.Add("ProductName", "Name");
bulkCopy.ColumnMappings.Add("Sku", "Sku");

await bulkCopy.WriteToServerAsync(dt, ct);

// Performance: 100,000 rows in < 5 seconds on modern hardware

Requirement: “Delete All Expired Records”

Business Requirement
Remove expired sessions older than 30 days. Could be millions of rows. A single DELETE could lock the table for too long.
SQL Developer Thinks Simple: DELETE FROM Sessions WHERE ExpiresAt < @Cutoff. But for millions of rows, this locks the whole table. Better: DELETE TOP 10,000 in a loop with a short delay to let readers access the table between batches.
C# Developer Thinks Simple delete: ExecuteAsync once. Large delete: loop ExecuteAsync with DELETE TOP to avoid lock escalation.
Simple Delete (Small Dataset)
await using var conn = new SqlConnection(cs);
var cutoffDate = DateTime.UtcNow.AddDays(-30);

var rowsDeleted = await conn.ExecuteAsync(
    @"DELETE FROM Sessions
      WHERE ExpiresAt < @CutoffDate",
    new { CutoffDate = cutoffDate },
    commandTimeout: 300);
Chunked Delete (Millions of Rows)
public async Task DeleteExpiredSessionsAsync(
    CancellationToken ct = default)
{
    var cutoffDate = DateTime.UtcNow.AddDays(-30);
    var batchSize = 10000;
    var totalDeleted = 0;

    await using var conn = new SqlConnection(_cs);
    await conn.OpenAsync(ct);

    while (true)
    {
        // Delete top 10,000 rows at a time
        var deleted = await conn.ExecuteAsync(
            @"DELETE TOP (@BatchSize) FROM Sessions
              WHERE ExpiresAt < @CutoffDate",
            new { BatchSize = batchSize,
                  CutoffDate = cutoffDate },
            commandTimeout: 60);

        if (deleted == 0) break;
        totalDeleted += deleted;

        // Pause 100ms to allow reads between batches
        await Task.Delay(100, ct);
    }

    return totalDeleted;
}
05

Requirement: “Import Must Be All-or-Nothing”

Business Requirement
If any insert fails during the 100,000-row import, roll back everything. Partial data is worse than no data.
SQL Developer Thinks BEGIN TRAN at the start, COMMIT at the end, ROLLBACK if anything fails. One atomic operation.
C# Developer Thinks Open transaction with conn.BeginTransactionAsync(). Run SqlBulkCopy or ExecuteAsync. Commit or rollback on success/failure.
C# — Transactional Bulk Insert
public async Task BulkInsertWithTransactionAsync(
    List<Product> products,
    CancellationToken ct = default)
{
    await using var conn = new SqlConnection(_connectionString);
    await conn.OpenAsync(ct);

    // Begin transaction
    using var tran = await conn.BeginTransactionAsync(
        IsolationLevel.ReadCommitted, ct);

    try
    {
        // Create DataTable
        var dt = new DataTable("Products");
        dt.Columns.Add("Name", typeof(string));
        dt.Columns.Add("Sku", typeof(string));
        dt.Columns.Add("Price", typeof(decimal));
        dt.Columns.Add("CategoryId", typeof(int));

        foreach (var p in products)
        {
            dt.Rows.Add(p.Name, p.Sku, p.Price, p.CategoryId);
        }

        // Bulk insert within transaction
        using var bulkCopy = new SqlBulkCopy(conn,
            SqlBulkCopyOptions.Default, tran)
        {
            DestinationTableName = "Products",
            BatchSize = 5000
        };

        await bulkCopy.WriteToServerAsync(dt, ct);

        // If we got here, commit
        await tran.CommitAsync(ct);
        return true;
    }
    catch (Exception ex)
    {
        // Any error rolls back the entire import
        await tran.RollbackAsync(ct);
        Console.WriteLine($"Import rolled back: {ex.Message}");
        return false;
    }
}
C# — Transaction with Dapper ExecuteAsync
public async Task InsertWithTransactionAsync(
    List<Product> products,
    CancellationToken ct = default)
{
    await using var conn = new SqlConnection(_connectionString);
    await conn.OpenAsync(ct);

    using var tran = await conn.BeginTransactionAsync(ct);

    try
    {
        // Execute within transaction
        await conn.ExecuteAsync(
            @"INSERT INTO Products (Name, Sku, Price, CategoryId)
              VALUES (@Name, @Sku, @Price, @CategoryId)",
            products,
            transaction: tran,
            commandTimeout: 120);

        await tran.CommitAsync(ct);
    }
    catch (Exception)
    {
        await tran.RollbackAsync(ct);
        throw;
    }
}

Performance Comparison: Bulk Methods

Method 100 Rows 10K Rows 100K Rows Best For
ExecuteAsync (1-by-1) ~100ms ~10s 100s+ (infeasible) Small batch, simple logic
ExecuteAsync (list loop) ~50ms ~5s 50s+ (slow) Small-medium batches (<5K)
Temp table + JOIN ~80ms ~2s ~15s Updates, complex merges
SqlBulkCopy ~30ms ~300ms ~3s Fast inserts, large batches (100K+)
EF AddRange ~60ms ~8s 80s+ (infeasible) ORM convenience only, not performance
Guidelines
— < 1,000 rows: ExecuteAsync loop is fine.
— 1,000 – 50,000 rows: Temp table or ExecuteAsync loop, depending on operation (insert vs update).
— 50,000+ rows: SqlBulkCopy for inserts, temp table for updates/merges.
