01

What is Dapper? The Micro-ORM Philosophy

Dapper is not a full-featured ORM like Entity Framework Core. It's a micro-ORM—a thin, high-performance wrapper around IDbConnection that does one job exceptionally well: map SQL query results to C# objects. Nothing more, nothing less.

The Philosophy: You Write SQL, Dapper Maps It

With Entity Framework, you write LINQ and the framework generates SQL for you. With Dapper, you write the SQL yourself, and Dapper handles the object mapping. This approach has profound trade-offs:

SQL (You Write This) 
SELECT Id, FirstName, Email,
       CreatedAt, IsActive
FROM Customers
WHERE Department = @Department
ORDER BY CreatedAt DESC
C# (Dapper Maps It) 
var customers = connection
    .Query<Customer>(sql,
        new { Department = "Sales" })
    .ToList();
Why Choose Dapper?

Speed: Minimal overhead. Direct mapping from DataReader to objects. Control: You see and write every query. No surprises, no N+1 queries unless you write them. Performance: Often faster than EF for bulk operations. No change tracking overhead. Simplicity: No DbContext, no migrations, no configuration. Just SQL and parameters.

Installation via NuGet

Dapper is a single NuGet package with zero dependencies (except the database driver you're using).

Package Manager Console 
Install-Package Dapper
Or with .NET CLI 
dotnet add package Dapper

That's it. No DbContext scaffolding, no migrations, no config files. Just add the package and start writing SQL.

Dapper vs Entity Framework: The Mindset Shift

EF Core encourages you to think in terms of objects and relationships. Dapper encourages you to think in terms of SQL queries and result sets. Which is better? It depends on your use case:

Aspect Dapper Entity Framework
Learning Curve You know SQL? You're done. Concepts, configurations, quirks.
Performance Faster. Minimal overhead. Good. Change tracking cost.
Query Control Complete. You write it. Translates LINQ to SQL.
CRUD Operations Manual SQL for each. Built-in SaveChanges.
Relationships Manual joins, manual mapping. Automatic navigation props.
02

Your First Dapper Query

Let's query a list of active customers from the database. Here's the complete, working code.

Step 1: Define Your Model

C# - Customer Class 
public class Customer
{
    public int Id { get; set; }
    public string FirstName { get; set; }
    public string LastName { get; set; }
    public string Email { get; set; }
    public bool IsActive { get; set; }
    public DateTime CreatedAt { get; set; }
}

Step 2: Write the Query

SQL 
SELECT Id, FirstName, LastName,
       Email, IsActive, CreatedAt
FROM Customers
WHERE IsActive = 1
C# with Dapper 
const string connString =
    "Data Source=localhost;Database=Shop;Trusted_Connection=true;";

using var conn =
    new SqlConnection(connString);

var customers = conn
    .Query<Customer>(
        "SELECT Id, FirstName, LastName, Email, IsActive, CreatedAt " +
        "FROM Customers " +
        "WHERE IsActive = 1"
    )
    .ToList();

That's it. Three steps: connection, query, map to objects. Dapper does the heavy lifting of reading the DataReader and creating Customer instances.

Comparison: Dapper vs EF Core

The same query with Entity Framework Core:

C# with EF Core 
using var context = new ShopContext();

var customers = context.Customers
    .Where(c => c.IsActive)
    .ToList();

EF is more concise and type-safe (LINQ), but Dapper gives you explicit control over the SQL. If the query runs slowly, you'll know exactly why—it's your SQL, not EF's query translation.

Dapper Magic: Convention-Based Mapping

Dapper maps SQL column names to C# property names by name matching (case-insensitive). The column FirstName in your query automatically maps to the property FirstName on your Customer class. No configuration needed.

03

Parameterized Queries (Preventing SQL Injection)

Never concatenate user input directly into SQL. Always use parameters. Dapper makes this trivial with anonymous objects.

The Wrong Way (Don't Do This!)

C# - VULNERABLE TO SQL INJECTION 
// NEVER DO THIS
string userCity = GetInputFromUser(); // "London' OR '1'='1"
string sql = $"SELECT * FROM Customers WHERE City = '{userCity}'";
var customers = conn.Query<Customer>(sql).ToList();

If a user enters London' OR '1'='1, your query becomes:

SQL (Injected!) 
SELECT * FROM Customers WHERE City = 'London' OR '1'='1'

This returns all customers, not just those in London. Catastrophic security breach.

The Right Way: Use Parameters

SQL with Parameter Placeholder 
SELECT Id, FirstName, Email, City
FROM Customers
WHERE City = @City
ORDER BY FirstName
C# - Safe Parameter Passing 
string userCity = GetInputFromUser();

var customers = conn
    .Query<Customer>(
        "SELECT Id, FirstName, Email, City " +
        "FROM Customers " +
        "WHERE City = @City " +
        "ORDER BY FirstName",
        new { City = userCity }
    )
    .ToList();

The @City parameter in SQL is replaced safely by the database driver. User input is never interpreted as SQL code.

Multiple Parameters

C# - Multiple Parameters with Anonymous Object 
var customers = conn
    .Query<Customer>(
        "SELECT Id, FirstName, Email, City, CreatedAt " +
        "FROM Customers " +
        "WHERE City = @City AND IsActive = @IsActive " +
        "AND CreatedAt > @JoinDate",
        new
        {
            City = "London",
            IsActive = true,
            JoinDate = new DateTime(2020, 1, 1)
        }
    )
    .ToList();
Anonymous Objects for Parameters

The parameter names in the anonymous object must match the SQL parameter placeholders. new { City = "London" } satisfies WHERE City = @City. Dapper matches them by name.

Complex Scenarios: DynamicParameters

For advanced scenarios (output parameters, special types), use DynamicParameters:

C# - Using DynamicParameters 
var parameters = new DynamicParameters();
parameters.Add("@City", "London");
parameters.Add("@IsActive", true);
parameters.Add("@CreatedAfter", new DateTime(2020, 1, 1));

var customers = conn
    .Query<Customer>(
        "SELECT * FROM Customers " +
        "WHERE City = @City AND IsActive = @IsActive " +
        "AND CreatedAt > @CreatedAfter",
        parameters
    )
    .ToList();

This provides the same safety as anonymous objects but with more explicit control over parameter types and directions (we'll see this in the stored procedures chapter).

04

Execute for INSERT, UPDATE, DELETE

For queries that don't return results (INSERT, UPDATE, DELETE), use Execute instead of Query. It returns the number of rows affected.

INSERT: Add a New Customer

SQL 
INSERT INTO Customers
    (FirstName, LastName, Email, City, IsActive, CreatedAt)
VALUES
    (@FirstName, @LastName, @Email, @City, @IsActive, @CreatedAt)
C# 
int rowsInserted = conn
    .Execute(
        "INSERT INTO Customers " +
        "(FirstName, LastName, Email, City, IsActive, CreatedAt) " +
        "VALUES (@FirstName, @LastName, @Email, @City, @IsActive, @CreatedAt)",
        new
        {
            FirstName = "John",
            LastName = "Doe",
            Email = "john@example.com",
            City = "London",
            IsActive = true,
            CreatedAt = DateTime.Now
        }
    );

Console.WriteLine($"Inserted {rowsInserted} row(s)");

UPDATE: Modify an Existing Customer

SQL 
UPDATE Customers
SET FirstName = @FirstName,
    LastName = @LastName,
    Email = @Email
WHERE Id = @Id
C# 
int rowsUpdated = conn
    .Execute(
        "UPDATE Customers " +
        "SET FirstName = @FirstName, " +
        "    LastName = @LastName, " +
        "    Email = @Email " +
        "WHERE Id = @Id",
        new
        {
            Id = 5,
            FirstName = "Jane",
            LastName = "Smith",
            Email = "jane@example.com"
        }
    );

Console.WriteLine($"Updated {rowsUpdated} row(s)");

DELETE: Remove a Customer

SQL 
DELETE FROM Customers
WHERE Id = @Id
C# 
int rowsDeleted = conn
    .Execute(
        "DELETE FROM Customers WHERE Id = @Id",
        new { Id = 5 }
    );

Console.WriteLine($"Deleted {rowsDeleted} row(s)");

Batch Operations: Insert Multiple Records

One of Dapper's strengths is batching. Pass a collection and Dapper executes the query once per item:

C# - Batch Insert 
var newCustomers = new List<Customer>
{
    new { FirstName = "Alice", LastName = "Brown", Email = "alice@example.com", City = "Paris", IsActive = true, CreatedAt = DateTime.Now },
    new { FirstName = "Bob", LastName = "Green", Email = "bob@example.com", City = "Berlin", IsActive = true, CreatedAt = DateTime.Now },
    new { FirstName = "Carol", LastName = "White", Email = "carol@example.com", City = "Amsterdam", IsActive = true, CreatedAt = DateTime.Now }
};

int totalInserted = conn
    .Execute(
        "INSERT INTO Customers (FirstName, LastName, Email, City, IsActive, CreatedAt) " +
        "VALUES (@FirstName, @LastName, @Email, @City, @IsActive, @CreatedAt)",
        newCustomers
    );

Console.WriteLine($"Inserted {totalInserted} rows total");
Return Value: Rows Affected

Execute returns an int representing the number of rows affected. For batches, it returns the total across all iterations. Always check this to verify your operation succeeded.

05

QueryFirst, QuerySingle, QueryFirstOrDefault

So far we've used Query<T> which returns an IEnumerable. But what if you expect zero or one result? Dapper provides specialized methods with different exception behaviors.

QueryFirst: The First Result (Throws if None)

C# 
// Get the most recently created customer
var customer = conn
    .QueryFirst<Customer>(
        "SELECT TOP 1 Id, FirstName, Email, CreatedAt " +
        "FROM Customers " +
        "ORDER BY CreatedAt DESC"
    );

Console.WriteLine($"Latest: {customer.FirstName}, {customer.CreatedAt}");

If the query returns no rows, QueryFirst throws an InvalidOperationException. Use this when you're certain a result exists.

QueryFirstOrDefault: The First Result (Returns Null if None)

C# 
// Get the most recently created customer, or null
var customer = conn
    .QueryFirstOrDefault<Customer>(
        "SELECT TOP 1 Id, FirstName, Email, CreatedAt " +
        "FROM Customers " +
        "ORDER BY CreatedAt DESC"
    );

if (customer != null)
{
    Console.WriteLine($"Latest: {customer.FirstName}");
}
else
{
    Console.WriteLine("No customers found");
}

This is safer when results are optional. No exception thrown; you just get null.

QuerySingle: Exactly One Result (Throws Otherwise)

C# 
// Get a specific customer by ID (expect exactly one)
var customer = conn
    .QuerySingle<Customer>(
        "SELECT Id, FirstName, Email, CreatedAt " +
        "FROM Customers " +
        "WHERE Id = @Id",
        new { Id = 42 }
    );

Console.WriteLine($"Customer: {customer.FirstName}");

If no rows are found, or more than one row is found, QuerySingle throws. Use this when you expect exactly one result (like fetching by primary key).

QuerySingleOrDefault: Exactly One or None (Returns Null Otherwise)

C# 
// Try to get a customer by ID, or null
var customer = conn
    .QuerySingleOrDefault<Customer>(
        "SELECT Id, FirstName, Email, CreatedAt " +
        "FROM Customers " +
        "WHERE Id = @Id",
        new { Id = 42 }
    );

if (customer != null)
{
    Console.WriteLine($"Found: {customer.FirstName}");
}
else
{
    Console.WriteLine("Customer not found");
}

Summary: When to Use Each

Method Throws if No Results? Throws if Multiple? Returns Collection?
Query<T> No (empty) No (all returned) Yes
QueryFirst<T> Yes No (takes first) No
QueryFirstOrDefault<T> No (null) No (takes first) No
QuerySingle<T> Yes Yes No
QuerySingleOrDefault<T> No (null) No (throws on 2+) No
Design Pattern: Match Your Expectations

Choose the method that matches what you expect. If you expect exactly one customer by ID, use QuerySingleOrDefault. If you're fetching a list, use Query. If you want the most recent but it might not exist, use QueryFirstOrDefault.

06

Async Queries

For I/O-bound operations, use async/await to free up threads while waiting for the database. Dapper provides fully async versions of all query methods.

Basic Async Query

C# - Synchronous 
public List<Customer> GetCustomers()
{
    using var conn =
        new SqlConnection(connString);

    var customers = conn
        .Query<Customer>(
            "SELECT * FROM Customers " +
            "WHERE IsActive = 1"
        )
        .ToList();

    return customers;
}
C# - Asynchronous 
public async Task<List<Customer>>
    GetCustomersAsync()
{
    using var conn =
        new SqlConnection(connString);

    var customers = await conn
        .QueryAsync<Customer>(
            "SELECT * FROM Customers " +
            "WHERE IsActive = 1"
        );

    return customers.ToList();
}

The async version uses QueryAsync instead of Query, and returns a Task<IEnumerable<T>> instead of IEnumerable<T>.

Async Execute (INSERT, UPDATE, DELETE)

C# - Async Insert 
public async Task InsertCustomerAsync(Customer customer)
{
    using var conn = new SqlConnection(connString);

    int rowsInserted = await conn
        .ExecuteAsync(
            "INSERT INTO Customers " +
            "(FirstName, LastName, Email, City, IsActive, CreatedAt) " +
            "VALUES (@FirstName, @LastName, @Email, @City, @IsActive, @CreatedAt)",
            new
            {
                customer.FirstName,
                customer.LastName,
                customer.Email,
                customer.City,
                customer.IsActive,
                customer.CreatedAt
            }
        );

    if (rowsInserted == 0)
        throw new InvalidOperationException("Insert failed");
}

Async Single/First Variants

C# - Async QuerySingleOrDefault 
public async Task<Customer?> GetCustomerByIdAsync(int id)
{
    using var conn = new SqlConnection(connString);

    var customer = await conn
        .QuerySingleOrDefaultAsync<Customer>(
            "SELECT Id, FirstName, Email, CreatedAt " +
            "FROM Customers " +
            "WHERE Id = @Id",
            new { Id = id }
        );

    return customer;
}

ConfigureAwait: A Best Practice

When writing library or reusable code, add .ConfigureAwait(false) to async calls. This tells the runtime not to capture the current synchronization context, improving performance in high-throughput scenarios.

C# - With ConfigureAwait 
public async Task<List<Customer>> GetCustomersAsync()
{
    using var conn = new SqlConnection(connString);

    var customers = await conn
        .QueryAsync<Customer>(
            "SELECT * FROM Customers WHERE IsActive = 1"
        )
        .ConfigureAwait(false);

    return customers.ToList();
}

Full Example: Async Repository Method

C# - Production-Ready Async Method 
public class CustomerRepository
{
    private readonly string _connectionString;

    public CustomerRepository(string connectionString)
    {
        _connectionString = connectionString;
    }

    public async Task<Customer?> GetByIdAsync(int customerId)
    {
        const string sql =
            "SELECT Id, FirstName, LastName, Email, City, " +
            "IsActive, CreatedAt FROM Customers WHERE Id = @Id";

        using var conn = new SqlConnection(_connectionString);

        var customer = await conn
            .QuerySingleOrDefaultAsync<Customer>(sql,
                new { Id = customerId })
            .ConfigureAwait(false);

        return customer;
    }

    public async Task<IEnumerable<Customer>> GetAllActiveAsync()
    {
        const string sql =
            "SELECT Id, FirstName, LastName, Email, City, " +
            "IsActive, CreatedAt FROM Customers " +
            "WHERE IsActive = 1 ORDER BY FirstName";

        using var conn = new SqlConnection(_connectionString);

        var customers = await conn
            .QueryAsync<Customer>(sql)
            .ConfigureAwait(false);

        return customers;
    }

    public async Task<int> InsertAsync(Customer customer)
    {
        const string sql =
            "INSERT INTO Customers " +
            "(FirstName, LastName, Email, City, IsActive, CreatedAt) " +
            "VALUES (@FirstName, @LastName, @Email, @City, @IsActive, @CreatedAt)";

        using var conn = new SqlConnection(_connectionString);

        return await conn
            .ExecuteAsync(sql, customer)
            .ConfigureAwait(false);
    }
}
Async: Not Just for Concurrency

Using async/await in ASP.NET and other frameworks allows a small thread pool to handle many concurrent requests. While your code is waiting for the database, other requests can use the freed thread. This is critical for scalable web applications.

Key Takeaways

  • Dapper is a micro-ORM: you write SQL, it maps results to objects.
  • Install via NuGet (Install-Package Dapper) and start querying immediately.
  • Always use parameterized queries (@Parameter) to prevent SQL injection.
  • Use Query<T> for lists, Execute for INSERT/UPDATE/DELETE operations.
  • Choose the right variant: QueryFirst, QuerySingle, QueryFirstOrDefault, QuerySingleOrDefault based on expected results.
  • Use async methods (QueryAsync, ExecuteAsync) in web applications for better scalability.
  • Add .ConfigureAwait(false) in library code to avoid unnecessary context switching.
