Chapter 18

Dapper: Connection & Queries

EF Core generates SQL for you. Dapper lets you write the SQL yourself and maps results to C# objects. It’s a thin layer — you get speed and full SQL power. Requirement: “I need direct SQL control with minimal overhead.”

01

Requirement: “Set Up a Lightweight Database Layer”

Business Requirement
Microservice needs fast, simple database access. EF is too heavy. We need to write raw SQL and get objects back with minimal framework overhead.
SQL Developer Thinks I write my own SELECT, INSERT, UPDATE, DELETE statements. I know exactly what query is running. No magic ORM, no LINQ translation surprises. I pass parameters to avoid injection.
C# Developer Thinks Dapper extends IDbConnection. One NuGet package. I call QueryAsync<T>, pass my SQL and parameters. It maps rows to objects. That’s it.
1. Install NuGet Package
dotnet add package Dapper
2. Set Up Connection (SqlClient)
using System.Data.SqlClient;
using Dapper;

public class CustomerRepository
{
    private readonly string _connectionString;

    public CustomerRepository(string connStr)
    {
        _connectionString = connStr;
    }

    private IDbConnection GetConnection()
        => new SqlConnection(_connectionString);
}
3. First Query: Get All Customers
public class Customer
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string Email { get; set; } = "";
    public bool IsActive { get; set; }
}

public async Task<IEnumerable<Customer>> GetAllAsync(
    CancellationToken ct = default)
{
    using var connection = GetConnection();
    var sql = "SELECT Id, Name, Email, IsActive FROM Customers";
    return await connection.QueryAsync<Customer>(sql);
}
02

Requirement: “Query Must Be Safe from SQL Injection”

Business Requirement
Search feature takes user input. Must be injection-proof. No concatenating user strings into SQL.
SQL Developer Thinks Use @parameters. Never concatenate strings. Even if it looks safe, parameterized queries are the only safe way.
C# Developer Thinks Pass an anonymous object to Dapper. Property names match the @parameter names. Dapper handles parameterization automatically.
SQL — BAD (Vulnerable)
-- NEVER DO THIS
var email = userInput;
var sql = "SELECT * FROM Customers " +
          "WHERE Email = '" + email + "'";
-- User enters: ' OR 1=1 --
-- Result: dumps entire table!
Dapper — GOOD (Safe)
public async Task<Customer?> SearchByEmailAsync(
    string email, CancellationToken ct = default)
{
    using var connection = GetConnection();
    var sql = "SELECT * FROM Customers " +
              "WHERE Email = @Email";
    return await connection.QueryFirstOrDefaultAsync<Customer>(
        sql, new { Email = email });
}

How Dapper Parameterizes: Property names in the anonymous object become SQL parameters. new { Email = email } creates @Email. The parameter value is passed separately from the SQL, so SQL injection is impossible.

03

Requirement: “Fetch a Single Customer by ID”

Business Requirement
API endpoint: GET /customers/42 returns one customer. Must handle “not found” gracefully.
SQL Developer Thinks SELECT * FROM Customers WHERE Id = @Id returns zero or one row.
C# Developer Thinks QueryFirstOrDefaultAsync<T> returns T? — null if no row found. QuerySingleOrDefaultAsync<T> throws if more than one row. Choose carefully.
Fetch List (All Matching Rows)
public async Task<IEnumerable<Customer>> GetByIsActiveAsync(
    bool isActive, CancellationToken ct = default)
{
    using var connection = GetConnection();
    var sql = "SELECT Id, Name, Email, IsActive " +
              "FROM Customers WHERE IsActive = @IsActive";
    return await connection.QueryAsync<Customer>(
        sql, new { IsActive = isActive });
}
Fetch One or None (First Match)
public async Task<Customer?> GetByIdAsync(
    int id, CancellationToken ct = default)
{
    using var connection = GetConnection();
    var sql = "SELECT Id, Name, Email, IsActive " +
              "FROM Customers WHERE Id = @Id";
    return await connection.QueryFirstOrDefaultAsync<Customer>(
        sql, new { Id = id });
}
Fetch Exactly One (Throws If Missing or Multiple)
public async Task<Customer> GetByIdRequiredAsync(
    int id, CancellationToken ct = default)
{
    using var connection = GetConnection();
    var sql = "SELECT Id, Name, Email, IsActive " +
              "FROM Customers WHERE Id = @Id";
    return await connection.QuerySingleOrDefaultAsync<Customer>(
        sql, new { Id = id })
        ?? throw new InvalidOperationException($"Customer {id} not found");
}
Method Returns Throws If
QueryAsync<T> IEnumerable<T> Never
QueryFirstOrDefaultAsync<T> T? Never (returns null if no match)
QuerySingleOrDefaultAsync<T> T? More than 1 row
QuerySingleAsync<T> T Zero rows or more than 1
ExecuteScalarAsync<T> T (single value) Never (returns default if null)
04

Requirement: “Create, Update, Delete with Dapper”

Business Requirement
Full CRUD without EF — insert new customer, update email, delete inactive accounts. Return affected row count or new ID.
SQL Developer Thinks INSERT returns nothing by default. Use OUTPUT INSERTED.Id or SCOPE_IDENTITY() to get the new ID. UPDATE and DELETE return affected row count.
C# Developer Thinks ExecuteAsync returns int — affected rows. For INSERT, query the identity in the same command or use ExecuteScalarAsync.
INSERT — Get New ID with OUTPUT
public async Task<int> CreateAsync(
    Customer customer, CancellationToken ct = default)
{
    using var connection = GetConnection();
    var sql = @"
        INSERT INTO Customers (Name, Email, IsActive)
        OUTPUT INSERTED.Id
        VALUES (@Name, @Email, @IsActive)";

    var newId = await connection.ExecuteScalarAsync<int>(
        sql, customer);
    return newId;
}
UPDATE — Return Affected Rows
public async Task<int> UpdateEmailAsync(
    int id, string newEmail, CancellationToken ct = default)
{
    using var connection = GetConnection();
    var sql = @"
        UPDATE Customers
        SET Email = @Email
        WHERE Id = @Id";

    return await connection.ExecuteAsync(
        sql, new { Id = id, Email = newEmail });
}
DELETE — Soft or Hard Delete
public async Task<int> DeleteAsync(
    int id, CancellationToken ct = default)
{
    using var connection = GetConnection();
    // Soft delete: just mark IsActive = false
    var sql = "UPDATE Customers SET IsActive = 0 WHERE Id = @Id";

    return await connection.ExecuteAsync(
        sql, new { Id = id });
}
05

Requirement: “Multiple Operations Must Succeed or All Fail”

Business Requirement
Create order + order items — both must succeed atomically. If item insert fails, roll back the order insert too.
SQL Developer Thinks BEGIN TRAN ... COMMIT or ROLLBACK. If any statement fails, the whole transaction rolls back.
C# Developer Thinks connection.BeginTransactionAsync() returns an IDbTransaction. Pass it to each Execute/Query call. On success, commit. On exception, rollback.
Dapper Transaction Pattern
public async Task CreateOrderWithItemsAsync(
    Order order, List<OrderItem> items,
    CancellationToken ct = default)
{
    using var connection = GetConnection();
    await connection.OpenAsync(ct);

    using var transaction = connection.BeginTransaction();
    try
    {
        // Insert order
        var orderSql = @"
            INSERT INTO Orders (CustomerId, Total)
            OUTPUT INSERTED.Id
            VALUES (@CustomerId, @Total)";
        var orderId = await connection.ExecuteScalarAsync<int>(
            orderSql, order, transaction);

        // Insert items
        foreach (var item in items)
        {
            var itemSql = @"
                INSERT INTO OrderItems (OrderId, ProductId, Qty)
                VALUES (@OrderId, @ProductId, @Qty)";
            await connection.ExecuteAsync(itemSql,
                new { OrderId = orderId,
                      ProductId = item.ProductId,
                      Qty = item.Qty },
                transaction);
        }

        // All succeeded
        transaction.Commit();
    }
    catch (Exception ex)
    {
        transaction.Rollback();
        throw;
    }
    finally
    {
        await connection.CloseAsync();
    }
}

Key Points: Explicit OpenAsync() required. Each Execute/Query method accepts transaction parameter. If exception occurs, Rollback() is called. Otherwise, Commit() persists changes.

06

“Dapper vs EF Core: When to Choose Dapper”

Scenario Choose Dapper Choose EF Core
Raw SQL control Yes — full power Limited (LINQ translation)
Max query performance Yes — no overhead Good (change tracking cost)
Complex reporting queries Yes — hand-tuned SQL Difficult (LINQ to SQL)
Microservices, read-heavy Yes — lightweight Overkill
CRUD with change tracking No (manual tracking) Yes — automatic
Migrations & schema management No (write SQL migrations) Yes — built-in
Complex includes/relationships No (join manually) Yes — .Include()
Team already knows ORM No — steeper learning Yes — faster onboard

Rule of Thumb: Use Dapper if you write SQL anyway and want speed. Use EF Core if you want framework convenience and automatic migrations. Many teams use both — EF Core for business logic CRUD, Dapper for reporting and high-performance queries.
