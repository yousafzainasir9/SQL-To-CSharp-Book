Chapter 21

Stored Procedures with Dapper

Call the DBA’s stored procedure from C#. Stored procs live in the database. Dapper calls them with CommandType.StoredProcedure and maps results to C# objects — input params, output params, return values, and multiple result sets.

01

Requirement: "Get Customer Report from a Stored Procedure"

Business Requirement
The DBA wrote sp_GetCustomerReport. Call it and map the results to a C# class. No parameters needed for this first call.
SQL Developer Thinks I execute the stored procedure by name. EXEC sp_GetCustomerReport returns a result set. The DBA wrote the logic — you just call it.
C# Developer Thinks I use QueryAsync with commandType: CommandType.StoredProcedure. Tell Dapper the procedure name and the type to map to. Dapper handles the rest.
SQL — Stored Procedure Definition
CREATE PROCEDURE sp_GetCustomerReport
AS
BEGIN
    SELECT
        Id,
        Name,
        Email,
        TotalOrders,
        TotalSpent,
        LastOrderDate
    FROM Customers
    WHERE IsActive = 1
    ORDER BY TotalSpent DESC;
END;
C# — Call with Dapper
public class CustomerReportDto
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string Email { get; set; } = "";
    public int TotalOrders { get; set; }
    public decimal TotalSpent { get; set; }
    public DateTime? LastOrderDate { get; set; }
}

var customers = await connection.QueryAsync<CustomerReportDto>(
    "sp_GetCustomerReport",
    commandType: CommandType.StoredProcedure,
    commandTimeout: 30);

foreach (var customer in customers)
{
    Console.WriteLine($"{customer.Name}: {customer.TotalSpent:C}");
}
C# — Same with EF Core (for comparison)
var customers = await context.Database
    .SqlQueryRaw<CustomerReportDto>(
        "EXEC sp_GetCustomerReport")
    .ToListAsync(cancellationToken);

// Note: EF requires DbContext + manually mapped DTOs
// Dapper is simpler for proc results
02

Requirement: "Pass Search Filters to the Procedure"

Business Requirement
The stored procedure accepts @StartDate, @EndDate, and @Status filters. Pass them from C# to narrow the results.
SQL Developer Thinks CREATE PROCEDURE with @Parameters. Use them in the WHERE clause. If null, ignore them (or make them optional with defaults).
C# Developer Thinks Create an anonymous object or named class with properties that match parameter names. Dapper maps them automatically. Names must match exactly (case-insensitive by default).
SQL — Stored Procedure with Parameters
CREATE PROCEDURE sp_GetOrdersByFilter
    @StartDate DATETIME2 = NULL,
    @EndDate DATETIME2 = NULL,
    @Status NVARCHAR(50) = NULL
AS
BEGIN
    SELECT
        Id, OrderNumber, OrderDate,
        CustomerId, TotalAmount, Status
    FROM Orders
    WHERE (OrderDate >= @StartDate
           OR @StartDate IS NULL)
      AND (OrderDate <= @EndDate
           OR @EndDate IS NULL)
      AND (Status = @Status
           OR @Status IS NULL)
    ORDER BY OrderDate DESC;
END;
C# — Anonymous Object Parameters
var orders = await connection.QueryAsync<OrderDto>(
    "sp_GetOrdersByFilter",
    new
    {
        StartDate = new DateTime(2024, 1, 1),
        EndDate = new DateTime(2024, 12, 31),
        Status = "Completed"
    },
    commandType: CommandType.StoredProcedure);

// Parameters map by name (case-insensitive)
// Only required params must be provided
var pendingOrders = await connection.QueryAsync<OrderDto>(
    "sp_GetOrdersByFilter",
    new { Status = "Pending" },
    commandType: CommandType.StoredProcedure);
C# — Using DynamicParameters for clarity
var parameters = new DynamicParameters();
parameters.Add("@StartDate", new DateTime(2024, 1, 1));
parameters.Add("@EndDate", new DateTime(2024, 12, 31));
parameters.Add("@Status", "Completed");

var orders = await connection.QueryAsync<OrderDto>(
    "sp_GetOrdersByFilter",
    parameters,
    commandType: CommandType.StoredProcedure);

// DynamicParameters is also required for OUTPUT and RETURN values
03

Requirement: "Get the New Customer ID After Insert"

Business Requirement
The stored procedure inserts a new customer and returns the new ID via an OUTPUT parameter so we can use it immediately.
SQL Developer Thinks Add @NewId INT OUTPUT to the parameter list. After the INSERT, assign the new ID to it: SET @NewId = SCOPE_IDENTITY();
C# Developer Thinks Use DynamicParameters with direction: ParameterDirection.Output. After ExecuteAsync, read the value back: parameters.Get<int>("@NewId")
SQL — OUTPUT Parameter
CREATE PROCEDURE sp_InsertCustomer
    @Name NVARCHAR(100),
    @Email NVARCHAR(255),
    @NewId INT OUTPUT
AS
BEGIN
    INSERT INTO Customers (Name, Email, IsActive)
    VALUES (@Name, @Email, 1);

    -- Return the new ID
    SET @NewId = SCOPE_IDENTITY();
END;
C# — Read OUTPUT Parameter
var parameters = new DynamicParameters();
parameters.Add("@Name", "Alice Johnson");
parameters.Add("@Email", "alice@example.com");
parameters.Add("@NewId",
    dbType: DbType.Int32,
    direction: ParameterDirection.Output);

await connection.ExecuteAsync(
    "sp_InsertCustomer",
    parameters,
    commandType: CommandType.StoredProcedure);

int newCustomerId = parameters.Get<int>("@NewId");
Console.WriteLine($"New customer ID: {newCustomerId}");
04

Requirement: "Check If the Operation Succeeded"

Business Requirement
The stored procedure returns 0 for success or −1 for failure (e.g., email already exists). Check it in C# to handle errors.
SQL Developer Thinks Use RETURN statement. RETURN 0 means success; RETURN -1 means failure. The DBA controls the error codes.
C# Developer Thinks RETURN statements are captured in a DynamicParameter with direction: ParameterDirection.ReturnValue. Read it after execution.
SQL — RETURN Statement
CREATE PROCEDURE sp_RegisterUser
    @Email NVARCHAR(255),
    @Password NVARCHAR(255)
AS
BEGIN
    -- Check if email exists
    IF EXISTS (SELECT 1 FROM Users
               WHERE Email = @Email)
    BEGIN
        RETURN -1; -- Email already registered
    END

    INSERT INTO Users (Email, Password, CreatedAt)
    VALUES (@Email, @Password, GETDATE());

    RETURN 0; -- Success
END;
C# — Read RETURN Value
var parameters = new DynamicParameters();
parameters.Add("@Email", "bob@example.com");
parameters.Add("@Password", hashedPassword);
parameters.Add("@ReturnCode",
    dbType: DbType.Int32,
    direction: ParameterDirection.ReturnValue);

await connection.ExecuteAsync(
    "sp_RegisterUser",
    parameters,
    commandType: CommandType.StoredProcedure);

int returnCode = parameters.Get<int>("@ReturnCode");

if (returnCode == 0)
{
    Console.WriteLine("User registered successfully");
}
else if (returnCode == -1)
{
    Console.WriteLine("Email already registered");
}
05

Requirement: "One Procedure Returns Customers AND Their Orders"

Business Requirement
Dashboard needs a summary of one customer and all their orders. Instead of two separate calls, one stored proc returns both result sets.
SQL Developer Thinks Two SELECT statements in the proc body. First SELECT returns the customer summary. Second SELECT returns their orders. Dapper reads both.
C# Developer Thinks Use QueryMultipleAsync instead of QueryAsync. It returns an SqlMapper.GridReader. Call ReadAsync<T> for each result set.
SQL — Multiple SELECTs
CREATE PROCEDURE sp_GetCustomerDashboard
    @CustomerId INT
AS
BEGIN
    -- First result set: customer summary
    SELECT
        Id, Name, Email, Phone,
        TotalOrders, TotalSpent,
        LastOrderDate
    FROM Customers
    WHERE Id = @CustomerId;

    -- Second result set: their orders
    SELECT
        Id, OrderNumber, OrderDate,
        TotalAmount, Status
    FROM Orders
    WHERE CustomerId = @CustomerId
    ORDER BY OrderDate DESC;
END;
C# — QueryMultipleAsync
public class CustomerSummary
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string Email { get; set; } = "";
    public string Phone { get; set; } = "";
    public int TotalOrders { get; set; }
    public decimal TotalSpent { get; set; }
    public DateTime? LastOrderDate { get; set; }
}

public class OrderDto
{
    public int Id { get; set; }
    public string OrderNumber { get; set; } = "";
    public DateTime OrderDate { get; set; }
    public decimal TotalAmount { get; set; }
    public string Status { get; set; } = "";
}

// Call the procedure
var reader = await connection.QueryMultipleAsync(
    "sp_GetCustomerDashboard",
    new { CustomerId = 42 },
    commandType: CommandType.StoredProcedure);

// Read the first result set (customer)
var customer = await reader.ReadSingleAsync<CustomerSummary>();

// Read the second result set (orders)
var orders = await reader.ReadAsync<OrderDto>();

Console.WriteLine($"{customer.Name}: {orders.Count()} orders");
C# — Using the GridReader pattern
// If the proc returns 3+ result sets:
var reader = await connection.QueryMultipleAsync(
    "sp_GetDashboard",
    commandType: CommandType.StoredProcedure);

var summaryList = await reader.ReadAsync<Summary>();
var orderList = await reader.ReadAsync<Order>();
var invoiceList = await reader.ReadAsync<Invoice>();

// ReadAsync returns IEnumerable<T>
// ReadSingle returns T (throws if not exactly 1)
// ReadSingleOrDefault returns T? (null if not found)
// ReadFirst returns T (takes the first)

When to Use Stored Procedures

The Question
Should I use a stored procedure or inline SQL?
Use CaseStored ProcedureInline SQL
Complex business logic (many conditions)Proc (DBA-owned)C# (Dev-owned)
Simple CRUD (select, insert, update, delete)—Inline SQL or ORM
Multiple dependent queriesProc (atomic)Multiple calls
Performance-critical (batching, temp tables)Proc (optimized)Inline (simple)
Report generation (multiple result sets)Proc (clean)Multiple calls
DBA manages updatesProc (versioned)—
Reused by multiple apps (Java, C++, etc.)Proc (shared)App-specific
Developer team owns all code—Inline (trackable)
SQL Developer Thinks Stored procs live in the database. I version them, test them, and deploy them separately. Multiple apps call the same proc — no duplication.
C# Developer Thinks Inline SQL stays in C# with my code. Everything is in version control, in the same commit. No surprises from secret proc changes.
07

Complete Example: Customer Reporting Proc

Business Requirement
Report all orders for a customer in a date range. Return the customer name and all matching orders. Use a stored procedure because the DBA wrote the complex filtering logic.
SQL — Stored Procedure
CREATE PROCEDURE sp_GetCustomerOrderReport
    @CustomerId INT,
    @StartDate DATETIME2 = NULL,
    @EndDate DATETIME2 = NULL,
    @TotalAmount DECIMAL(10,2) OUTPUT
AS
BEGIN
    -- Get customer name (first result set)
    SELECT Name
    FROM Customers
    WHERE Id = @CustomerId;

    -- Get orders (second result set)
    SELECT
        Id, OrderNumber, OrderDate,
        TotalAmount, Status, ItemCount
    FROM Orders
    WHERE CustomerId = @CustomerId
      AND (OrderDate >= @StartDate OR @StartDate IS NULL)
      AND (OrderDate <= @EndDate OR @EndDate IS NULL)
    ORDER BY OrderDate DESC;

    -- Calculate total (OUTPUT param)
    SELECT @TotalAmount = SUM(TotalAmount)
    FROM Orders
    WHERE CustomerId = @CustomerId
      AND (OrderDate >= @StartDate OR @StartDate IS NULL)
      AND (OrderDate <= @EndDate OR @EndDate IS NULL);
END;
C# — Call and Process Results
public class OrderReportDto
{
    public int Id { get; set; }
    public string OrderNumber { get; set; } = "";
    public DateTime OrderDate { get; set; }
    public decimal TotalAmount { get; set; }
    public string Status { get; set; } = "";
    public int ItemCount { get; set; }
}

public async Task<(string CustomerName, List<OrderReportDto> Orders, decimal TotalSpent)>
    GetCustomerOrderReportAsync(
        int customerId,
        DateTime? startDate = null,
        DateTime? endDate = null,
        CancellationToken ct = default)
{
    var parameters = new DynamicParameters();
    parameters.Add("@CustomerId", customerId);
    parameters.Add("@StartDate", startDate);
    parameters.Add("@EndDate", endDate);
    parameters.Add("@TotalAmount",
        dbType: DbType.Decimal,
        direction: ParameterDirection.Output,
        precision: 10,
        scale: 2);

    var reader = await connection.QueryMultipleAsync(
        "sp_GetCustomerOrderReport",
        parameters,
        commandType: CommandType.StoredProcedure,
        commandTimeout: 60,
        cancellationToken: ct);

    // Read customer name
    var customerName = await reader.ReadSingleAsync<string>();

    // Read orders
    var orders = (await reader.ReadAsync<OrderReportDto>()).ToList();

    // Read output parameter
    decimal totalAmount = parameters.Get<decimal>("@TotalAmount");

    return (customerName, orders, totalAmount);
}

// Usage
var (name, orders, total) = await GetCustomerOrderReportAsync(
    customerId: 42,
    startDate: new DateTime(2024, 1, 1),
    endDate: new DateTime(2024, 12, 31));

Console.WriteLine($"{name} spent {total:C} on {orders.Count} orders");
