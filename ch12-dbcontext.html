Chapter 12

DbContext

You get a requirement: "Connect the app to the database." A SQL developer opens a connection, runs queries, closes it. A C# developer creates a DbContext — it IS the connection, the query builder, and the transaction manager all in one.

01

Requirement: "Connect My App to the Database"

Business Requirement
New project needs database access. Set up the connection.
SQL Developer Thinks Open a SqlConnection, pass the connection string, execute commands, manage open/close lifecycle. If the connection is left open too long, bad things happen.
C# Developer Thinks Create a DbContext class. It wraps the connection, builds queries from LINQ, tracks changes, and generates SQL automatically. One DbContext per operation, then dispose.
SQL — Manual Connection
using (SqlConnection conn = new SqlConnection(
    "Server=localhost;Database=Shop;..."))
{
    conn.Open();
    SqlCommand cmd = new SqlCommand(
        "SELECT * FROM Customers", conn);
    SqlDataReader reader = cmd.ExecuteReader();
    while (reader.Read())
    {
        // Process rows manually
    }
    conn.Close(); // Must close explicitly
}
C# — DbContext
using var db = new ShopContext();
var customers = await db.Customers
    .ToListAsync(cancellationToken);

// DbContext handles:
// - Connection pool management
// - Query generation from LINQ
// - Connection open/close lifecycle
// - Automatic disposal via using
C# — Define Your DbContext Class
public class ShopContext : DbContext
{
    public ShopContext(DbContextOptions<ShopContext> options)
        : base(options) { }

    // Each DbSet represents a table
    public DbSet<Customer> Customers { get; set; }
    public DbSet<Order> Orders { get; set; }
    public DbSet<Product> Products { get; set; }

    protected override void OnModelCreating(
        ModelBuilder modelBuilder)
    {
        // Configure tables, keys, relationships, constraints
    }
}
02

Requirement: "Define Which Tables My App Uses"

Business Requirement
App needs access to Customers, Orders, and Products tables. Make them queryable.
SQL Developer Thinks Tables exist in the database. I query them directly: SELECT * FROM Customers. The database schema is separate from my query logic.
C# Developer Thinks Each DbSet<T> maps to a table. DbSet<Customer> = Customers table. DbSet<Order> = Orders table. I query DbSets with LINQ, not raw SQL.
SQL — Direct Table Access
-- Tables exist in the database
SELECT * FROM Customers;
SELECT * FROM Orders;
SELECT * FROM Products;

-- Join them in queries
SELECT c.Name, o.Total
FROM Customers c
JOIN Orders o ON c.Id = o.CustomerId
WHERE o.Total > 100;
C# — DbSets & LINQ Queries
public class ShopContext : DbContext
{
    public DbSet<Customer> Customers { get; set; }
    public DbSet<Order> Orders { get; set; }
    public DbSet<Product> Products { get; set; }
}

// Query with LINQ
var expensive = await db.Customers
    .Where(c => c.Orders.Any(o => o.Total > 100))
    .ToListAsync(ct);
C# — Full DbContext with Multiple DbSets
public class ShopContext : DbContext
{
    public ShopContext(DbContextOptions<ShopContext> options)
        : base(options) { }

    public DbSet<Customer> Customers { get; set; }
    public DbSet<Order> Orders { get; set; }
    public DbSet<Product> Products { get; set; }
    public DbSet<OrderItem> OrderItems { get; set; }
    public DbSet<Category> Categories { get; set; }

    protected override void OnModelCreating(
        ModelBuilder modelBuilder)
    {
        // Configure relationships, keys, constraints
    }
}
03

Requirement: "Store Connection Details Securely"

Business Requirement
Connection string varies per environment (dev/staging/prod). Don't hardcode it.
SQL Developer Thinks Connection string is a runtime detail. Dev uses localhost, prod uses the cloud. Store it in config, not in code. Credentials must never be in source control.
C# Developer Thinks Load connection from appsettings.json or environment variables. Register DbContext in dependency injection. Inject it into services. Different config per environment.
appsettings.json — Store Connection String
{
  "ConnectionStrings": {
    "Default": "Server=localhost;Database=Shop;Integrated Security=true;TrustServerCertificate=true;"
  },
  "Logging": { "LogLevel": { "Default": "Information" } }
}
Program.cs — Register DbContext in DI
var builder = WebApplicationBuilder.CreateBuilder(args);

// Load connection string from appsettings.json
var connectionString = builder.Configuration
    .GetConnectionString("Default")
    ?? throw new InvalidOperationException(
        "Connection string 'Default' not found.");

// Register DbContext in the DI container
builder.Services.AddDbContext<ShopContext>(options =>
    options.UseSqlServer(connectionString));

var app = builder.Build();
app.MapControllers();
app.Run();
Using the DbContext in a Service
public class CustomerService
{
    private readonly ShopContext _db;

    // DbContext is injected automatically by DI
    public CustomerService(ShopContext db)
    {
        _db = db;
    }

    public async Task<Customer?> GetCustomerAsync(
        int id, CancellationToken ct = default)
    {
        return await _db.Customers
            .FirstOrDefaultAsync(c => c.Id == id, ct);
    }
}
04

Requirement: "Configure Table Names, Keys, and Constraints in Code"

Business Requirement
DBA named the table "tbl_Customers" but C# class is Customer. Map them. Also configure primary keys, foreign keys, and constraints via code.
SQL Developer Thinks Table names, column types, indexes, constraints — all in DDL (CREATE TABLE). The schema is the source of truth.
C# Developer Thinks Use OnModelCreating with Fluent API to define schema as code. Map class names to table names, configure keys, relationships, and constraints. Schema follows the code.
SQL — CREATE TABLE with Constraints
CREATE TABLE tbl_Customers (
    customer_id INT IDENTITY(1,1) PRIMARY KEY,
    customer_name NVARCHAR(100) NOT NULL,
    email NVARCHAR(255) NOT NULL UNIQUE,
    is_active BIT NOT NULL DEFAULT 1,
    created_at DATETIME2 NOT NULL
        DEFAULT GETDATE()
);
C# — Fluent API Configuration
protected override void OnModelCreating(
    ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Customer>(e =>
    {
        // Map to table named tbl_Customers
        e.ToTable("tbl_Customers");

        // Primary key (maps to customer_id)
        e.HasKey(c => c.Id)
         .HasName("PK_Customers");

        e.Property(c => c.Id)
         .HasColumnName("customer_id")
         .ValueGeneratedOnAdd();

        e.Property(c => c.Name)
         .HasColumnName("customer_name")
         .IsRequired()
         .HasMaxLength(100);

        e.Property(c => c.Email)
         .HasColumnName("email")
         .IsRequired()
         .HasMaxLength(255);

        e.HasIndex(c => c.Email).IsUnique();

        e.Property(c => c.IsActive)
         .HasColumnName("is_active")
         .HasDefaultValue(true);

        e.Property(c => c.CreatedAt)
         .HasColumnName("created_at")
         .HasDefaultValueSql("GETDATE()");
    });
}
05

Requirement: "Save Multiple Changes as One Transaction"

Business Requirement
Creating an order also creates order items. Both must succeed or both fail. No partial orders in the database.
SQL Developer Thinks BEGIN TRANSACTION, then multiple INSERTs, then COMMIT. If any INSERT fails, ROLLBACK everything. That is atomicity.
C# Developer Thinks Add order, add items to the order, call SaveChangesAsync() once. EF wraps it in a transaction automatically. If anything fails, the whole thing rolls back.
SQL — Transaction Control
BEGIN TRANSACTION;

INSERT INTO Orders (CustomerId, Total, CreatedAt)
VALUES (5, 250.00, GETDATE());

DECLARE @OrderId INT = SCOPE_IDENTITY();

INSERT INTO OrderItems (OrderId, ProductId, Qty, Price)
VALUES (@OrderId, 10, 2, 50.00);
INSERT INTO OrderItems (OrderId, ProductId, Qty, Price)
VALUES (@OrderId, 11, 1, 150.00);

COMMIT; -- Or ROLLBACK if error occurs
C# — SaveChangesAsync
public async Task CreateOrderAsync(
    Order order, List<OrderItem> items,
    CancellationToken ct = default)
{
    db.Orders.Add(order);
    db.OrderItems.AddRange(items);

    // One call, wrapped in a transaction
    // Both succeed or both fail
    await db.SaveChangesAsync(ct);
}
C# — Control Transaction Explicitly
public async Task CreateOrderAsync(
    Order order, List<OrderItem> items,
    CancellationToken ct = default)
{
    using (var transaction = await db.Database
        .BeginTransactionAsync(ct))
    {
        try
        {
            db.Orders.Add(order);
            db.OrderItems.AddRange(items);
            await db.SaveChangesAsync(ct);

            // Do business logic (e.g., email, logging)

            await transaction.CommitAsync(ct);
        }
        catch
        {
            await transaction.RollbackAsync(ct);
            throw;
        }
    }
}
06

Requirement: "How Long Should a DbContext Live?"

Business Requirement
Web API handles thousands of requests. Each request needs database access. How do we manage DbContext lifetime without memory leaks or stale data?
SQL Developer Thinks Connection pooling. Open a connection, use it, close it quickly. The pool reuses connections. This scales to thousands of requests.
C# Developer Thinks Scoped lifetime. One DbContext per HTTP request via dependency injection. Request starts, DbContext created. Request ends, DbContext disposed. EF handles the connection pool underneath.
LifetimeScopeUse CaseRisk
Transient New instance every time Stateless operations, unit tests High memory overhead
Scoped One per HTTP request Web APIs, web apps (RECOMMENDED) None — sweet spot for scalability
Singleton One instance for app lifetime N/A — NEVER use for DbContext Memory leak, stale data, thread-safety issues
Program.cs — Register DbContext with Scoped Lifetime
var builder = WebApplicationBuilder.CreateBuilder(args);

// Scoped: one DbContext per HTTP request
builder.Services.AddDbContext<ShopContext>(options =>
    options.UseSqlServer(connectionString));

// Transient: new instance every time (use carefully)
// builder.Services.AddDbContext<ShopContext>
//     .Lifetime = ServiceLifetime.Transient;

// NEVER: Singleton will cause memory leaks and data stale issues
// builder.Services.AddSingleton<ShopContext>();

var app = builder.Build();
app.MapControllers();
app.Run();
C# — DbContext Pooling for High-Throughput Apps
// For very high throughput (>1000 requests/sec),
// use DbContext pooling to reuse instances
builder.Services.AddDbContextPool<ShopContext>(
    options => options.UseSqlServer(connectionString),
    poolSize: 128); // Reuse up to 128 DbContext instances

// This reduces allocation overhead when combined with
// AddQuotaPooling for connection pooling
