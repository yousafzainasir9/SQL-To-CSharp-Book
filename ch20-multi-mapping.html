Chapter 20

Multi-Mapping with Dapper

Map JOIN results to multiple C# objects, split result sets by key, and populate related objects in a single query.

01

One-to-One Joins

Multi-mapping tells Dapper which columns belong to which object. Use Query<Parent, Child, Result> with a splitOn parameter.

SQL
SELECT c.Id, c.FirstName, c.Email,
       a.Id, a.Street, a.City
FROM Customers c
INNER JOIN Addresses a ON c.Id = a.CustomerId
C#
public class Customer
{
    public int Id { get; set; }
    public string FirstName { get; set; }
    public string Email { get; set; }
    public Address Address { get; set; }
}

public class Address
{
    public int Id { get; set; }
    public string Street { get; set; }
    public string City { get; set; }
}

var sql = @"SELECT c.Id, c.FirstName, c.Email,
                  a.Id, a.Street, a.City
           FROM Customers c
           INNER JOIN Addresses a ON c.Id = a.CustomerId";

var results = connection.Query<Customer, Address, Customer>(sql,
    (customer, address) =>
    {
        customer.Address = address;
        return customer;
    },
    splitOn: "Id").ToList();
splitOn parameter: Tells Dapper where one object ends and the next begins. Usually "Id" (the key of the second object).
02

One-to-Many: Grouping Children

When a parent has many children, Dapper returns duplicate parent rows. Group them in C# using ToDictionary or GroupBy.

SQL
SELECT c.Id, c.FirstName,
       o.Id, o.OrderDate, o.Total
FROM Customers c
LEFT JOIN Orders o ON c.Id = o.CustomerId
ORDER BY c.Id
C# — Using Dictionary
public class Customer
{
    public int Id { get; set; }
    public string FirstName { get; set; }
    public List<Order> Orders { get; set; } = new();
}

var sql = @"SELECT c.Id, c.FirstName,
                  o.Id, o.OrderDate, o.Total
           FROM Customers c
           LEFT JOIN Orders o ON c.Id = o.CustomerId
           ORDER BY c.Id";

var customerDict = new Dictionary<int, Customer>();

connection.Query<Customer, Order, Customer>(sql,
    (customer, order) =>
    {
        if (!customerDict.TryGetValue(customer.Id, out var existing))
        {
            existing = customer;
            customerDict.Add(customer.Id, existing);
        }
        if (order != null)
            existing.Orders.Add(order);
        return existing;
    },
    splitOn: "Id").ToList();

var results = customerDict.Values.ToList();
Key to avoid duplicates: Use a dictionary keyed on the parent ID to consolidate rows. Only add each parent once.
03

Three or More Tables

Join more than two tables by specifying multiple splitOn columns and using Query<T1, T2, T3, TResult>.

C# — Three-Way Join
var sql = @"SELECT c.Id, c.FirstName,
                  o.Id, o.OrderDate,
                  p.Id, p.ProductName, p.Price
           FROM Customers c
           JOIN Orders o ON c.Id = o.CustomerId
           JOIN OrderItems p ON o.Id = p.OrderId";

var results = await connection.QueryAsync<Customer, Order, Product, (Customer, Order, Product)>(sql,
    (customer, order, product) => (customer, order, product),
    splitOn: "Id,Id");  // Two splits: after first Id, after second Id

foreach (var (customer, order, product) in results)
{
    Console.WriteLine($"{customer.FirstName}: {order.OrderDate} - {product.ProductName}");
}
splitOn syntax: Comma-separated column names. "Id,Id" means split after the 1st Id and 2nd Id.
04

Alternative: Separate Queries with QueryMultiple

Instead of complex JOINs, use QueryMultiple to fetch parents and children separately, then join in C#.

C#
var sql = @"SELECT Id, FirstName FROM Customers;
            SELECT Id, CustomerId, OrderDate FROM Orders;";

using var multi = await connection.QueryMultipleAsync(sql);
var customers = (await multi.ReadAsync<Customer>()).ToList();
var orders = (await multi.ReadAsync<Order>()).ToList();

// Join in C#
var result = customers
    .GroupJoin(orders, c => c.Id, o => o.CustomerId,
        (customer, customerOrders) =>
        {
            customer.Orders = customerOrders.ToList();
            return customer;
        })
    .ToList();
Trade-off: Two queries but simpler logic. Better for complex one-to-many relationships.
05

Dynamic Objects

Use Query<dynamic> to avoid defining classes, but you lose type safety and must cast values.

C#
var results = await connection.QueryAsync<dynamic>(sql);

foreach (dynamic row in results)
{
    int customerId = (int)row.CustomerId;
    string firstName = (string)row.FirstName;
    decimal total = (decimal)row.Total;
    Console.WriteLine($"{firstName}: {total}");
}
Use sparingly: Dynamic is flexible but requires casting and loses IntelliSense. Define classes for production code.
06

Performance Considerations

Multi-mapping with large one-to-many sets can cause significant data duplication across the network. Choose your approach wisely.

Comparison
// Approach 1: Single JOIN Query
// Pro: One round trip
// Con: Duplicate parent data for every child row
var singleQuery = await connection.QueryAsync<Customer, Order, Customer>(
    @"SELECT c.Id, c.FirstName, o.Id, o.Total
      FROM Customers c JOIN Orders o ON c.Id = o.CustomerId");

// Approach 2: Two Separate Queries
// Pro: No duplication, cleaner queries
// Con: Two round trips
var parents = await connection.QueryAsync<Customer>("SELECT * FROM Customers");
var children = await connection.QueryAsync<Order>("SELECT * FROM Orders");
Choose based on: Small result sets use JOIN. Large one-to-many use separate queries.

Key Takeaways

  • One-to-one: Use Query<Parent, Child, Result> with splitOn.
  • One-to-many: Use a dictionary to group duplicate parent rows.
  • Multiple tables: Specify multiple columns in splitOn.
  • QueryMultiple is cleaner for complex relationships than single JOIN.
  • Dynamic objects work but lose compile-time safety.
  • Large one-to-many: Two queries often better than one JOIN with duplication.
