Chapter 02

Rows vs Objects

You get requirements: "Add a customer," "Find a customer," "Update their email," "Remove inactive accounts." A SQL developer writes INSERT, SELECT, UPDATE, DELETE. A C# developer creates, reads, modifies, and removes objects. Same operations, different mental models.

01

Requirement: "Add a New Customer"

Business Requirement
When someone signs up, store their name, email, and mark them active with today's date. Return the new customer ID.
SQL Developer Thinks I need an INSERT statement with VALUES. The database auto-generates the ID with IDENTITY, so I use SCOPE_IDENTITY() to grab it back. One statement, one result.
C# Developer Thinks I new up a Customer object, set its properties, call Add() to stage it, SaveChangesAsync() to persist it. EF automatically populates the Id after the save. I have an object in memory, not just raw values.
SQL — INSERT with SCOPE_IDENTITY
INSERT INTO Customers (Name, Email, IsActive, CreatedAt)
VALUES ('Alice Brown', 'alice@example.com', 1, GETDATE());

SELECT SCOPE_IDENTITY() AS NewId;
C# — EF Core Create
var customer = new Customer
{
    Name = "Alice Brown",
    Email = "alice@example.com",
    IsActive = true,
    CreatedAt = DateTime.UtcNow
};

db.Customers.Add(customer);
await db.SaveChangesAsync(ct);

// customer.Id is now populated
var newId = customer.Id;
Dapper — ExecuteScalarAsync
const string sql = @"
    INSERT INTO Customers (Name, Email, IsActive, CreatedAt)
    VALUES (@Name, @Email, @IsActive, @CreatedAt);
    SELECT CAST(SCOPE_IDENTITY() as int);";

var newId = await connection.ExecuteScalarAsync<int>(sql, new
{
    Name = "Alice Brown",
    Email = "alice@example.com",
    IsActive = true,
    CreatedAt = DateTime.UtcNow
}, commandTimeout: null, commandType: CommandType.Text);
02

Requirement: "Find Customer by ID and List Active Customers"

Business Requirement
User clicks a profile — show their details by ID. Also, need to display all active customers in the dashboard. Both are read-only operations.
SQL Developer Thinks Single customer: SELECT * WHERE Id = @id. Multiple customers: SELECT * WHERE IsActive = 1. Simple filters, simple results.
C# Developer Thinks Single: use FindAsync(id) — EF knows the primary key. Multiple: Where() for filtering, ToListAsync() to fetch. Always use AsNoTracking() for read-only queries — no change tracking overhead.
SQL — Find by ID and List Active
-- Find by ID
SELECT Id, Name, Email, IsActive, CreatedAt
FROM Customers
WHERE Id = @CustomerId;

-- List all active customers
SELECT Id, Name, Email, IsActive, CreatedAt
FROM Customers
WHERE IsActive = 1
ORDER BY CreatedAt DESC;
LINQ — Method Syntax
// Find by ID
var customer = await db.Customers
    .AsNoTracking()
    .FirstOrDefaultAsync(
        c => c.Id == customerId, ct);

// List all active customers
var activeCustomers = await db.Customers
    .AsNoTracking()
    .Where(c => c.IsActive)
    .OrderByDescending(c => c.CreatedAt)
    .ToListAsync(ct);
LINQ — Query Syntax
// Find by ID
var customer = await db.Customers
    .AsNoTracking()
    .FirstOrDefaultAsync(
        c => c.Id == customerId, ct);

// List all active customers
var activeCustomers = await (
    from c in db.Customers.AsNoTracking()
    where c.IsActive
    orderby c.CreatedAt descending
    select c
).ToListAsync(ct);
Dapper — Direct SQL
// Find by ID
const string sql1 = @"
    SELECT Id, Name, Email, IsActive, CreatedAt
    FROM Customers
    WHERE Id = @CustomerId";

var customer = await connection
    .QueryFirstOrDefaultAsync<CustomerDto>(
        sql1, new { CustomerId = customerId });

// List all active
var activeCustomers = await connection
    .QueryAsync<CustomerDto>(
        "SELECT ... FROM Customers WHERE IsActive = 1
         ORDER BY CreatedAt DESC");
03

Requirement: "Customer Changed Their Email"

Business Requirement
Customer submits a profile update form. Change their email and record when it was modified. Also support bulk updates: admin marks dozens of customers as inactive.
SQL Developer Thinks UPDATE SET the columns I want to change in the WHERE clause. For one customer, use WHERE Id = @id. For bulk, use WHERE with a condition. Simple mutations, fast.
C# Developer Thinks Single: Fetch the entity, modify its properties, SaveChangesAsync(). EF tracks what changed. Bulk: use ExecuteUpdateAsync() to avoid fetching everything into memory — it's a direct database mutation.
SQL — Update Single Customer
UPDATE Customers
SET Email = @NewEmail,
    UpdatedAt = GETDATE()
WHERE Id = @CustomerId;
C# — Fetch-Modify-Save Pattern (EF Core)
var customer = await db.Customers.FindAsync(
    new object[] { customerId }, cancellationToken: ct);
if (customer == null)
    throw new InvalidOperationException("Customer not found");

customer.Email = "newemail@example.com";
customer.UpdatedAt = DateTime.UtcNow;

await db.SaveChangesAsync(ct);
// EF only sent UPDATE for changed columns
C# — Service Layer with DTO (Production Pattern)
public record UpdateCustomerRequest(int Id, string Email);

public class CustomerService
{
    private readonly AppDbContext _db;

    public async Task UpdateAsync(
        UpdateCustomerRequest request, CancellationToken ct)
    {
        var customer = await _db.Customers.FindAsync(
            new object[] { request.Id }, cancellationToken: ct);
        if (customer == null)
            throw new InvalidOperationException("Customer not found");

        customer.Email = request.Email;
        customer.UpdatedAt = DateTime.UtcNow;

        await _db.SaveChangesAsync(ct);
    }
}
SQL — Bulk Update (Conditional)
UPDATE Customers
SET IsActive = 0
WHERE LastLoginAt < DATEADD(YEAR, -2, GETDATE());
C# — EF7+ ExecuteUpdateAsync (No Load)
// Mark inactive: customers who haven't logged in for 2 years
var twoYearsAgo = DateTime.UtcNow.AddYears(-2);

var affected = await db.Customers
    .Where(c => c.LastLoginAt < twoYearsAgo)
    .ExecuteUpdateAsync(s => s
        .SetProperty(c => c.IsActive, false)
        .SetProperty(c => c.UpdatedAt, DateTime.UtcNow),
        cancellationToken: ct);

// Returns count of affected rows, database handles the mutation
Dapper — ExecuteAsync
// Single update
const string sql = @"
    UPDATE Customers
    SET Email = @Email, UpdatedAt = GETDATE()
    WHERE Id = @CustomerId";

var affected = await connection.ExecuteAsync(sql, new
{
    CustomerId = customerId,
    Email = "newemail@example.com"
});

// Bulk update
const string bulkSql = @"
    UPDATE Customers
    SET IsActive = 0
    WHERE LastLoginAt < DATEADD(YEAR, -2, GETDATE())";

var bulkAffected = await connection.ExecuteAsync(bulkSql);
04

Requirement: "Remove Inactive Accounts"

Business Requirement
Delete customers who haven't logged in for 2 years. Also, let admins delete individual accounts on demand. We want audit trails, so soft-delete is an option.
SQL Developer Thinks DELETE WHERE for individual or bulk. Or, for audit: SET IsDeleted = 1 instead of actual deletion. Direct, permanent, or soft.
C# Developer Thinks Remove() + SaveChangesAsync() for single entities. ExecuteDeleteAsync() for bulk (no fetch). Soft-delete: update a flag instead of removing. Track who deleted and when.
SQL — Hard Delete (Single and Bulk)
-- Delete single customer
DELETE FROM Customers WHERE Id = @CustomerId;

-- Delete inactive (2+ years no login)
DELETE FROM Customers
WHERE LastLoginAt < DATEADD(YEAR, -2, GETDATE());
C# — Remove Single Entity
var customer = await db.Customers.FindAsync(
    new object[] { customerId }, cancellationToken: ct);
if (customer != null)
{
    db.Customers.Remove(customer);
    await db.SaveChangesAsync(ct);
}
C# — EF7+ ExecuteDeleteAsync (Bulk)
// Delete customers inactive for 2+ years
var twoYearsAgo = DateTime.UtcNow.AddYears(-2);

var deletedCount = await db.Customers
    .Where(c => c.LastLoginAt < twoYearsAgo)
    .ExecuteDeleteAsync(cancellationToken: ct);

// Returns count of deleted rows, database executes the DELETE
SQL — Soft Delete (Audit Trail)
-- Add soft-delete columns to table
ALTER TABLE Customers ADD DeletedAt DATETIME2 NULL;
ALTER TABLE Customers ADD DeletedBy NVARCHAR(255) NULL;

-- Soft-delete single customer
UPDATE Customers
SET DeletedAt = GETDATE(), DeletedBy = @AdminId
WHERE Id = @CustomerId;

-- When querying, exclude soft-deleted
SELECT * FROM Customers WHERE DeletedAt IS NULL;
C# — Soft Delete Pattern
public class Customer
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string Email { get; set; } = "";
    public DateTime CreatedAt { get; set; }

    // Soft-delete audit columns
    public DateTime? DeletedAt { get; set; }
    public string? DeletedBy { get; set; }

    public bool IsDeleted => DeletedAt.HasValue;
}

// In queries, always filter
var activeCustomers = await db.Customers
    .AsNoTracking()
    .Where(c => !c.IsDeleted)
    .ToListAsync(ct);

// Soft-delete a customer
customer.DeletedAt = DateTime.UtcNow;
customer.DeletedBy = adminId;
await db.SaveChangesAsync(ct);
Dapper — ExecuteAsync for Delete
// Hard delete single
const string sql1 = "DELETE FROM Customers WHERE Id = @CustomerId";
var affected = await connection.ExecuteAsync(
    sql1, new { CustomerId = customerId });

// Hard delete bulk
const string sql2 = @"
    DELETE FROM Customers
    WHERE LastLoginAt < DATEADD(YEAR, -2, GETDATE())";
var bulkAffected = await connection.ExecuteAsync(sql2);

// Soft delete
const string softSql = @"
    UPDATE Customers
    SET DeletedAt = GETDATE(), DeletedBy = @AdminId
    WHERE Id = @CustomerId";
var softAffected = await connection.ExecuteAsync(softSql, new
{
    CustomerId = customerId,
    AdminId = adminId
});
05

Requirement: "Handle Concurrent Edits"

Business Requirement
Two admins edit the same customer simultaneously. If both try to save, we should detect the conflict and ask one of them to re-fetch and try again — we don't want to silently lose changes.
SQL Developer Thinks I need a ROWVERSION column (timestamp). Include it in the WHERE clause. If it doesn't match, no rows updated — I know there was a conflict. Client detects zero affected rows.
C# Developer Thinks EF has [Timestamp] attribute — maps to ROWVERSION. On save, if the version changed, EF throws DbUpdateConcurrencyException. I catch it, handle the conflict (merge, re-fetch, or reject).
SQL — ROWVERSION Concurrency
-- Add ROWVERSION to table
ALTER TABLE Customers ADD RowVersion ROWVERSION;

-- Update includes the version check
UPDATE Customers
SET Email = @NewEmail, UpdatedAt = GETDATE()
WHERE Id = @CustomerId
  AND RowVersion = @OriginalRowVersion;

-- If no rows affected, there was a conflict
IF @@ROWCOUNT = 0
    THROW 50001, 'Concurrency conflict: Customer was modified', 1;
C# — Entity with [Timestamp]
public class Customer
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string Email { get; set; } = "";
    public DateTime CreatedAt { get; set; }
    public DateTime? UpdatedAt { get; set; }

    // EF maps [Timestamp] to ROWVERSION in SQL Server
    [Timestamp]
    public byte[] RowVersion { get; set; } = Array.Empty<byte>();
}
C# — Concurrency Exception Handling
try
{
    var customer = await db.Customers.FindAsync(
        new object[] { customerId }, cancellationToken: ct);
    if (customer == null)
        throw new InvalidOperationException("Customer not found");

    customer.Email = "newemail@example.com";
    customer.UpdatedAt = DateTime.UtcNow;

    await db.SaveChangesAsync(ct);
}
catch (DbUpdateConcurrencyException ex)
{
    // Concurrency conflict detected
    var entry = ex.Entries.Single();
    var dbValues = await entry.GetDatabaseValuesAsync(ct);

    if (dbValues == null)
        throw new InvalidOperationException("Customer was deleted");

    // Merge strategy: keep database values, notify user
    entry.OriginalValues.SetValues(dbValues);
    throw new InvalidOperationException(
        "Customer was modified by someone else. Please refresh and try again.");
}
Dapper — Manual Version Check
const string sql = @"
    UPDATE Customers
    SET Email = @Email, UpdatedAt = GETDATE()
    WHERE Id = @CustomerId
      AND RowVersion = @OriginalVersion;

    SELECT CAST(@@ROWCOUNT AS INT);";

var affected = await connection.ExecuteScalarAsync<int>(sql, new
{
    CustomerId = customerId,
    Email = "newemail@example.com",
    OriginalVersion = originalRowVersion
});

if (affected == 0)
    throw new InvalidOperationException(
        "Concurrency conflict: Customer was modified");
06

Which Tool for Which Job? — EF Core vs Dapper CRUD

Quick Decision Guide
Both handle CRUD. EF abstracts the SQL; Dapper is direct SQL. Choose based on your needs.
Operation EF Core Strength Dapper Strength Use EF When Use Dapper When
CREATE new Customer() { } → Add() → SaveChanges() INSERT → ExecuteScalarAsync() Simple inserts, navigation auto-handled Bulk inserts, custom ID generation
READ FindAsync(id) / Where().ToList() QueryFirstOrDefaultAsync() / QueryAsync() Complex filters, eager-load relationships Simple queries, projection to DTO
UPDATE Fetch → Mutate → SaveChanges() UPDATE → ExecuteAsync() Single entity, change tracking Bulk updates, ExecuteUpdateAsync() alternative
DELETE Fetch → Remove() → SaveChanges() DELETE → ExecuteAsync() Single entity, cascades auto-handled Bulk deletes, soft-delete direct SQL
Transactions Built-in, SaveChanges() is one unit Explicit Begin/Commit Multi-entity changes, automatic rollback Complex logic across multiple statements
Concurrency [Timestamp], DbUpdateConcurrencyException Manual RowVersion in WHERE Automatic versioning, built-in conflict handling Custom concurrency logic
A SQL Developer's Take CRUD is just 4 basic operations. Both EF and Dapper do them fine. EF hides complexity; Dapper shows you the SQL. Neither is wrong — it's about what your app needs.
A C# Developer's Take EF is objects and LINQ — composable, safe, refactor-friendly. Dapper is strings and parameters — faster to write for trivial queries, but you own the SQL maintenance. Use both: EF for your core domain, Dapper for reporting.
