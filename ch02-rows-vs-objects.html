Part 1: The Two Worlds — SQL Meets OOP

Rows vs Objects
CRUD Through Both Lenses

A single row in a table is an instance of that entity. When you work with data in C#, you're working with objects—instances of classes. CRUD operations (Create, Read, Update, Delete) exist in both SQL and C#, but they're expressed differently. Learn how INSERT becomes construction, UPDATE becomes mutation, and how Entity Framework and Dapper translate between these two worlds. This chapter is foundational to understanding the impedance mismatch between relational databases and object-oriented programming.

Row Instances: Understanding SELECT and Result Sets

The fundamental parallel between SQL and C# is this: a row in a table is a single instance of an entity type. When you SELECT a customer by ID from the database, you get back one row with columns containing values. That row is the manifestation of one customer instance. In C#, when you query for that same customer using LINQ or Entity Framework, you get back a Customer object—an instance of the Customer class with properties populated from the database columns.

This mapping happens automatically in Object-Relational Mapping (ORM) frameworks like Entity Framework. The framework retrieves the row data and constructs a C# object with matching properties. You never explicitly map columns to properties—the framework handles this based on your configuration and property names. Understanding this parallelism is crucial because it means SQL thinking and C# thinking describe the same reality from different angles. A row is a unit of data; an object is a unit of behavior and state. They represent the same domain entity.

SQL: Fetch a Row from Table -- SELECT returns a result set -- A result set is a collection of rows -- Each row is one instance SELECT Id, Name, Email, CreatedAt, Status FROM Customers WHERE Id = 42; -- Result set (one row): -- | Id | Name | Email | CreatedAt | Status | -- | 42 | John Doe | john@example.com | 2024-01-15 | Active |
C#: Fetch an Object Instance // LINQ Method Syntax var customer = context.Customers .FirstOrDefault(c => c.Id == 42); // LINQ Query Syntax (equivalent) var customer = (from c in context.Customers where c.Id == 42 select c) .FirstOrDefault(); // Result: A Customer object instance // customer.Id = 42 // customer.Name = "John Doe" // customer.Email = "john@example.com" // customer.CreatedAt = DateTime(2024,1,15) // customer.Status = "Active" if (customer != null) { Console.WriteLine($"Customer: {customer.Name}"); }
Key Insight: Result Sets as Collections

SQL's SELECT always returns a result set, which is a collection of rows—even if it's a collection of one. In C#, you use methods like FirstOrDefault(), SingleOrDefault(), or ToList() to extract instances from the IQueryable result. FirstOrDefault() returns one row as one object, or null if no rows match. ToList() returns all rows as a collection of objects. The mapping between rows and objects happens transparently.

SQL: Multiple Rows (Result Set) -- SELECT can return multiple rows SELECT Id, Name, Email FROM Customers WHERE Status = 'Active' ORDER BY Name; -- Result set (multiple rows): -- | Id | Name | Email | -- | 1 | Alice | alice@example.com | -- | 5 | Bob | bob@example.com | -- | 42 | John Doe | john@example.com |
C#: Multiple Objects as Collection // LINQ Method Syntax var customers = context.Customers .Where(c => c.Status == "Active") .OrderBy(c => c.Name) .ToList(); // LINQ Query Syntax (equivalent) var customers = (from c in context.Customers where c.Status == "Active" orderby c.Name select c) .ToList(); // Result: List of Customer objects // customers[0].Name = "Alice" // customers[1].Name = "Bob" // customers[2].Name = "John Doe" foreach (var customer in customers) { Console.WriteLine(customer.Name); }
Key Insight: LINQ Query vs Method Syntax

LINQ offers two syntaxes: query syntax (SQL-like with from/where/select keywords) and method syntax (fluent, chainable methods). They're equivalent—query syntax is syntactic sugar that compiles to method syntax. Choose based on readability and team preference. Query syntax feels more SQL-like; method syntax is more functional. Both operate on IQueryable, which translates to SQL before executing.

CREATE: SQL INSERT vs C# Object Construction

In SQL, you use INSERT to create a new row in a table. You provide values for each column (or accept defaults for columns with default values). The database writes the row to disk and assigns an IDENTITY if the table has an identity column. In C#, you instantiate a new object, set its properties, and tell Entity Framework to save it. EF translates this into an INSERT statement, sends it to the database, and the new row is created.

The power of Entity Framework is that after SaveChanges() completes, EF automatically populates the ID property on your object from the database-generated identity. This keeps your in-memory object synchronized with what the database created. You can immediately use the ID for further operations without needing a separate query.

SQL: INSERT a New Row -- Insert with explicit columns INSERT INTO Customers (Name, Email, Status, CreatedAt) VALUES ('Jane Smith', 'jane@example.com', 'Active', GETUTCDATE()); -- Get the newly created ID SELECT SCOPE_IDENTITY() AS NewId; -- Returns: 43 (the new identity value) -- Verify: SELECT * FROM Customers WHERE Id = 43;
C#: Create and Save Object // Create a new object instance var customer = new Customer { Name = "Jane Smith", Email = "jane@example.com", Status = "Active", CreatedAt = DateTime.UtcNow // Note: Id is NOT set; database will generate it }; // Add to context context.Customers.Add(customer); // Save to database context.SaveChanges(); // After SaveChanges(), EF populated the Id // automatically from the database identity Console.WriteLine($"New customer ID: {customer.Id}"); // 43 // The object is now synchronized with the database // You can use customer.Id for further operations
Key Insight: Automatic ID Population

Entity Framework's SaveChanges() method returns the number of rows affected, but more importantly, it populates identity columns on your objects after the INSERT completes. This is done using SCOPE_IDENTITY() or similar mechanisms. You don't need a separate query to fetch the generated ID—EF handles it automatically. This keeps your object model in sync with the database state.

SQL: Bulk INSERT -- Insert multiple rows INSERT INTO Customers (Name, Email, Status, CreatedAt) VALUES ('Alice', 'alice@example.com', 'Active', GETUTCDATE()), ('Bob', 'bob@example.com', 'Active', GETUTCDATE()), ('Charlie', 'charlie@example.com', 'Inactive', GETUTCDATE());
C#: Bulk Create (Multiple Objects) // Create multiple objects var customers = new List { new Customer { Name = "Alice", Email = "alice@example.com", Status = "Active", CreatedAt = DateTime.UtcNow }, new Customer { Name = "Bob", Email = "bob@example.com", Status = "Active", CreatedAt = DateTime.UtcNow }, new Customer { Name = "Charlie", Email = "charlie@example.com", Status = "Inactive", CreatedAt = DateTime.UtcNow } }; // Add all to context context.Customers.AddRange(customers); // One SaveChanges() for all context.SaveChanges(); // All objects now have IDs populated foreach (var c in customers) { Console.WriteLine($"Created: {c.Name} with ID {c.Id}"); }
Key Insight: Batch Inserts with SaveChanges

You can add multiple objects before calling SaveChanges(). EF batches them into efficient SQL, often using multi-value INSERT syntax: INSERT INTO ... VALUES (...), (...), (...). One SaveChanges() call handles all of them, and all objects get their IDs populated. This is much faster than calling SaveChanges() for each object individually.

For comparison, here's how Dapper (a micro-ORM) approaches INSERT—more explicit, more control:

SQL: Manual INSERT with Parameters INSERT INTO Customers (Name, Email, Status, CreatedAt) VALUES (@Name, @Email, @Status, @CreatedAt); SELECT SCOPE_IDENTITY() AS Id;
C#: Dapper INSERT Approach var customer = new Customer { Name = "Jane Smith", Email = "jane@example.com", Status = "Active", CreatedAt = DateTime.UtcNow }; var sql = @" INSERT INTO Customers (Name, Email, Status, CreatedAt) VALUES (@Name, @Email, @Status, @CreatedAt); SELECT SCOPE_IDENTITY() AS Id; "; // Execute and get the new ID var newId = connection .QuerySingle(sql, customer); // Manually set the ID on the object customer.Id = newId;
Key Insight: EF vs Dapper Philosophy

Entity Framework abstracts away SQL generation and ID retrieval. You never write SQL; you construct objects and call SaveChanges(). Dapper requires you to write SQL explicitly, but you have full control. For simple CRUD, EF is faster to develop. For complex operations or performance-critical code, Dapper's explicitness can be valuable. Both strategies work; they represent different tradeoffs.

READ: SELECT Queries and LINQ Query Syntax

SELECT in SQL retrieves rows from tables. The WHERE clause filters rows, ORDER BY sorts them, and you can aggregate with COUNT, SUM, AVG, etc. In C#, LINQ queries do the same thing, but with C# syntax. The LINQ provider (Entity Framework's provider) translates your C# code into SQL and executes it at the database level. You can write LINQ using method syntax or query syntax—both translate to the same SQL.

One crucial distinction: LINQ queries are lazy. They don't execute immediately. You call ToList(), FirstOrDefault(), Single(), Count(), or another materialization method to force execution. Until you materialize, the LINQ query is just an expression tree waiting to be translated to SQL. This allows you to build queries incrementally—adding Where() clauses conditionally—before executing.

SQL: SELECT with WHERE and ORDER BY -- Select customers with specific criteria SELECT Id, Name, Email, Status, CreatedAt FROM Customers WHERE Status = 'Active' AND CreatedAt > '2024-01-01' ORDER BY CreatedAt DESC LIMIT 10; -- Translates to: fetch 10 rows matching criteria
C#: LINQ Method Syntax (Equivalent) // This doesn't execute until you call ToList()! var query = context.Customers .Where(c => c.Status == "Active") .Where(c => c.CreatedAt > new DateTime(2024, 1, 1)) .OrderByDescending(c => c.CreatedAt) .Take(10); // query is still IQueryable - no DB hit yet! // This materializes the query - NOW it executes var results = query.ToList(); // EF translates to SQL and fetches 10 rows foreach (var customer in results) { Console.WriteLine($"{customer.Name}: {customer.CreatedAt}"); }
Key Insight: Lazy Evaluation and Deferred Execution

LINQ queries don't execute when you call Where(), OrderBy(), or Take(). They're lazy—they wait until you call a materialization method like ToList(), FirstOrDefault(), Count(), or iterate in a foreach. This allows conditional query building: you can add Where() clauses based on user input without worrying about performance. The final query, however constructed, only executes once—at the database level—not row-by-row in C#.

SQL: COUNT Aggregation -- Count rows matching criteria SELECT COUNT(*) AS TotalCount FROM Customers WHERE Status = 'Active';
C#: LINQ Query Syntax with COUNT // Query Syntax (more SQL-like) var activeCount = (from c in context.Customers where c.Status == "Active" select c) .Count(); // Method Syntax (equivalent) var activeCount = context.Customers .Where(c => c.Status == "Active") .Count(); // This executes: SELECT COUNT(*) WHERE Status='Active' Console.WriteLine($"Active customers: {activeCount}");
Key Insight: Projections and Select()

SQL's SELECT clause specifies which columns to retrieve. LINQ's Select() method does the same—it projects each row into a specific shape. You can project entire objects, or specific properties, or anonymous types. Projecting only needed columns is a performance optimization: you reduce data transfer and memory usage. This is client-side filtering vs server-side filtering.

SQL: Select Specific Columns -- Projection: only fetch needed columns SELECT Id, Name, Email FROM Customers WHERE Status = 'Active';
C#: Projection with Select() // Project to an anonymous type var customers = context.Customers .Where(c => c.Status == "Active") .Select(c => new { c.Id, c.Name, c.Email // Only these three properties are fetched! }) .ToList(); // Query Syntax (equivalent) var customers = (from c in context.Customers where c.Status == "Active" select new { c.Id, c.Name, c.Email }) .ToList(); foreach (var c in customers) { Console.WriteLine($"{c.Name}: {c.Email}"); }
Key Insight: Client-Side vs Server-Side Operations

Operations before a materialization method (ToList()) execute at the database level. Operations after ToList() execute in C# memory. Never do Where() or complex operations after ToList()—always push filtering and aggregation to the database. Database operations are typically orders of magnitude faster. After ToList(), you're working with objects in memory; the database is out of the picture.

UPDATE: SQL SET and Object Mutation with Change Tracking

SQL's UPDATE statement changes values in columns for rows matching a WHERE clause. It's a set-based operation—you can update thousands of rows in one statement. In C#, the traditional approach is to fetch an object, modify its properties, and call SaveChanges(). Entity Framework tracks the changes you make and generates an UPDATE statement for only the columns that changed. This is powerful but different from SQL's set-based mindset.

Entity Framework's change tracking system is one of its most important features. When you fetch an object, EF remembers its original state. When you modify a property (e.g., customer.Name = "New Name"), EF detects this change. When you call SaveChanges(), EF compares the current state to the original state and generates UPDATE statements for only the changed columns. This means EF generates efficient SQL—it doesn't rewrite all columns, just the ones you actually changed. This is more sophisticated than raw UPDATE SQL from a developer experience perspective.

SQL: UPDATE Statement -- Update specific columns for matching rows UPDATE Customers SET Name = 'Jane Doe', Email = 'jane@example.com', UpdatedAt = GETUTCDATE() WHERE Id = 42; -- Verify the update SELECT * FROM Customers WHERE Id = 42;
C#: Fetch, Mutate, Save (Traditional) // Fetch the object var customer = context.Customers .FirstOrDefault(c => c.Id == 42); if (customer != null) { // Mutate the object's properties customer.Name = "Jane Doe"; customer.Email = "jane@example.com"; customer.UpdatedAt = DateTime.UtcNow; // SaveChanges detects changes via change tracking // Generates: UPDATE Customers SET Name=@p0, Email=@p1, UpdatedAt=@p2 WHERE Id=@p3 context.SaveChanges(); } // Query Syntax equivalent (for fetching): var customer = (from c in context.Customers where c.Id == 42 select c) .FirstOrDefault(); // Rest is the same: mutate and SaveChanges()
Key Insight: Change Tracking Efficiency

Entity Framework's change tracking is sophisticated: it only generates UPDATE statements for columns that actually changed. If you fetch a customer and change their Name but not their Email, the generated SQL only updates Name. This is efficient because your UPDATE statement is minimal. It also prevents accidental overwrites of concurrent changes (to some degree—it depends on concurrency strategy, which is covered in later chapters).

However, for large-scale operations (updating thousands of rows), the traditional fetch-mutate-save approach is inefficient. You'd need to fetch all rows into memory, loop through them, mutate each one, and call SaveChanges(). That's thousands of database round trips. Entity Framework Core 5+ introduced ExecuteUpdate() for set-based updates that execute at the database level.

SQL: Batch UPDATE at Database -- Update many rows in one statement -- This is efficient: one round trip, all rows updated UPDATE Customers SET Status = 'Inactive' WHERE LastLoginDate < DATEADD(YEAR, -1, GETUTCDATE()) AND Status = 'Active';
C#: Batch UPDATE with ExecuteUpdate (EF 5+) // Set-based update: executes at database level var cutoffDate = DateTime.UtcNow.AddYears(-1); var rowsUpdated = context.Customers .Where(c => c.LastLoginDate < cutoffDate && c.Status == "Active") .ExecuteUpdate(s => s .SetProperty(c => c.Status, "Inactive") ); Console.WriteLine($"Updated {rowsUpdated} customers"); // This generates one UPDATE statement and executes it at the DB // Much faster than: fetch all, loop, mutate, SaveChanges()
Key Insight: ExecuteUpdate for Batch Operations

When you need to update many rows, use ExecuteUpdate(), not the traditional fetch-mutate-save pattern. ExecuteUpdate() executes at the database level in a single statement. The traditional pattern fetches all matching rows into memory—inefficient for large datasets. ExecuteUpdate() is O(1) database round trips; the traditional approach is O(n) where n is the number of rows to update.

SQL: What fetch-mutate-save generates -- Step 1: Fetch all matching rows SELECT Id, Name, Status, LastLoginDate FROM Customers WHERE LastLoginDate < DATEADD(YEAR, -1, GETUTCDATE()); -- Step 2: Then for EACH customer: UPDATE Customers SET Status='Inactive' WHERE Id=@id; -- This repeats 10,000 times! -- Total: 1 SELECT + 10,000 UPDATEs = 10,001 round trips!
C#: INEFFICIENT - Avoid This Pattern // SLOW: Do NOT do this for bulk updates! var cutoffDate = DateTime.UtcNow.AddYears(-1); var customersToUpdate = context.Customers .Where(c => c.LastLoginDate < cutoffDate && c.Status == "Active") .ToList(); // ← Fetches all matching rows into memory! foreach (var customer in customersToUpdate) { customer.Status = "Inactive"; context.SaveChanges(); // ← One SaveChanges per customer! } // This generates 1 SELECT + N UPDATEs // Each SaveChanges() is a round trip to the database // For 10,000 customers: 10,001 round trips!
Key Insight: Think in Sets, Not Loops

SQL is a set-oriented language. When you update many rows, think like SQL: generate one UPDATE statement that changes all matching rows at once. Don't think like C#: loop through individual objects. This is the impedance mismatch in action. When you need batch operations, use ExecuteUpdate(). When you need to update a few specific objects, use the traditional fetch-mutate-save pattern.

DELETE: SQL Removal and Object Disposal

In SQL, DELETE removes rows from a table. You specify which rows with a WHERE clause—even if it's WHERE Id = 42, which deletes one row. The row is gone; any foreign keys referencing it may cascade delete related rows depending on the ON DELETE rule configured on the foreign key.

In C#, you fetch an object, call Remove() on the DbSet, and call SaveChanges(). Entity Framework marks the object for deletion. When you call SaveChanges(), EF generates a DELETE statement using the object's primary key as the WHERE clause. The row is removed from the database. The object still exists in memory, but it's no longer in the database and its EntityState is changed to Deleted.

SQL: DELETE a Row -- Delete a specific row by ID DELETE FROM Customers WHERE Id = 42; -- Verify deletion SELECT COUNT(*) AS RemainingCount FROM Customers WHERE Id = 42; -- Should return: 0
C#: Remove and Save Object // Fetch the object var customer = context.Customers .FirstOrDefault(c => c.Id == 42); if (customer != null) { // Remove from the context context.Customers.Remove(customer); // SaveChanges generates DELETE and executes it context.SaveChanges(); // The object still exists in memory but is // deleted from the database Console.WriteLine(customer.Name); // Still accessible in memory // But if you query the DB, it won't be there } // Equivalent in Query Syntax: var customer = (from c in context.Customers where c.Id == 42 select c) .FirstOrDefault();
Key Insight: Remove() Uses Primary Key

Entity Framework uses the object's primary key (Id) to generate the DELETE statement. You don't write a WHERE clause—EF generates it from the object's key. This is safer than raw SQL because you can only delete objects you have in memory. You can't accidentally delete the wrong row because the object's identity is what drives the deletion. This is one of EF's safety features.

SQL: Batch DELETE -- Delete multiple rows in one statement DELETE FROM Customers WHERE Status = 'Inactive' AND CreatedAt < DATEADD(YEAR, -2, GETUTCDATE());
C#: Batch DELETE with ExecuteDelete // EF Core 6+ ExecuteDelete for batch deletion var cutoffDate = DateTime.UtcNow.AddYears(-2); var rowsDeleted = context.Customers .Where(c => c.Status == "Inactive" && c.CreatedAt < cutoffDate) .ExecuteDelete(); Console.WriteLine($"Deleted {rowsDeleted} customers"); // This executes one DELETE statement at the database level // Much faster than fetching all rows and removing individually
Key Insight: ExecuteDelete for Batch Deletions

Just like ExecuteUpdate(), ExecuteDelete() handles batch deletions efficiently. Use it when deleting many rows. The traditional fetch-remove-save approach for batch deletes is inefficient. ExecuteDelete() executes at the database level in one statement. One round trip to delete thousands of rows is vastly faster than fetching them all into memory and removing them one-by-one.

SQL: Dapper DELETE DELETE FROM Customers WHERE Id = @Id;
C#: Dapper DELETE Approach var customerId = 42; var sql = "DELETE FROM Customers WHERE Id = @Id;"; var rowsAffected = connection.Execute(sql, new { Id = customerId }); Console.WriteLine($"Deleted {rowsAffected} rows");
Key Insight: Soft Deletes vs Hard Deletes

Hard delete removes data from the database permanently. Soft delete (not shown here but important in practice) marks rows as deleted without removing them—you add an IsDeleted flag or DeletedAt timestamp. Soft deletes are useful for auditing, recovery, and maintaining referential integrity. They require filtering out soft-deleted rows in all queries, so design this carefully. Hard deletes (what we've covered) are simpler but permanent.

Batch Operations: Sets vs Collections (ExecuteUpdate/ExecuteDelete)

This is where SQL and C# thinking diverge most clearly. SQL is set-oriented: you write one statement that modifies or deletes thousands of rows at once. The database engine optimizes this at a low level. C# is object-oriented: you think about individual objects, their properties, their state. When applied naively to batch operations, C# thinking leads to loops and multiple round trips, which is inefficient.

Entity Framework 5+ bridged this gap with ExecuteUpdate() and ExecuteDelete() (6+). These methods execute at the database level, generating set-based SQL that processes all matching rows in one statement. When you need batch operations, think like SQL. Use LINQ to specify criteria, but use ExecuteUpdate() or ExecuteDelete() to process all matching rows at once, not in a loop.

SQL: Set-Based Operations (Efficient) -- One statement: updates all matching rows -- Database processes the entire set at once -- One round trip, extremely fast UPDATE Orders SET Status = 'Shipped' WHERE OrderDate < DATEADD(DAY, -7, GETUTCDATE()) AND Status = 'Processing'; -- Delete all expired sessions DELETE FROM Sessions WHERE ExpiresAt < GETUTCDATE();
C#: Set-Based with ExecuteUpdate (Efficient) // EF 5+: ExecuteUpdate executes at database level var weekAgo = DateTime.UtcNow.AddDays(-7); var rowsUpdated = context.Orders .Where(o => o.OrderDate < weekAgo && o.Status == "Processing") .ExecuteUpdate(s => s.SetProperty(o => o.Status, "Shipped")); // EF 6+: ExecuteDelete executes at database level var rowsDeleted = context.Sessions .Where(s => s.ExpiresAt < DateTime.UtcNow) .ExecuteDelete(); // Both execute one statement at the database level // One round trip, extremely fast Console.WriteLine($"Updated {rowsUpdated}, Deleted {rowsDeleted}");
Key Insight: Set-Based Operations Require SQL Thinking

Set-based operations are paradigmatically different from object-oriented programming. But they're crucial for performance at scale. When processing 100,000 rows, the difference between one set-based operation and 100,000 individual operations is the difference between milliseconds and minutes. Always use set-based operations when the semantics allow it. Batch by criteria using Where(), then use ExecuteUpdate() or ExecuteDelete(), not individual object mutations.

SQL: What Client-Side Loop Generates -- Step 1: Fetch all matching rows SELECT Id, Status, OrderDate FROM Orders WHERE OrderDate < DATEADD(DAY, -7, GETUTCDATE()); -- Step 2: Then for EACH order: UPDATE Orders SET Status='Shipped' WHERE Id=@id; -- This repeats 50,000 times! -- Total: 1 SELECT + 50,000 UPDATEs = 50,001 round trips -- Each round trip adds latency -- Total time: potentially minutes!-- One round trip, extremely fast UPDATE Orders SET Status = 'Shipped' WHERE OrderDate < DATEADD(DAY, -7, GETUTCDATE()) AND Status = 'Processing'; -- Delete all expired sessions DELETE FROM Sessions WHERE ExpiresAt < GETUTCDATE();
C#: Set-Based with ExecuteUpdate (Efficient) // EF 5+: ExecuteUpdate executes at database level var weekAgo = DateTime.UtcNow.AddDays(-7); var rowsUpdated = context.Orders .Where(o => o.OrderDate < weekAgo && o.Status == "Processing") .ExecuteUpdate(s => s.SetProperty(o => o.Status, "Shipped")); // EF 6+: ExecuteDelete executes at database level var rowsDeleted = context.Sessions .Where(s => s.ExpiresAt < DateTime.UtcNow) .ExecuteDelete(); // Both execute one statement at the database level // One round trip, extremely fast Console.WriteLine($"Updated {rowsUpdated}, Deleted {rowsDeleted}");
