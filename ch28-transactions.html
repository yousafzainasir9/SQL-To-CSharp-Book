01

ACID Properties

ACID ensures reliable database operations: Atomicity (all or nothing), Consistency (valid state), Isolation (no conflicts), Durability (survives failure).

PropertyMeaningExample
AtomicityAll changes or noneTransfer: debit + credit both succeed or both fail
ConsistencyValid state maintainedConstraints always satisfied (no negative balance)
IsolationTransactions don't interfereTwo simultaneous updates don't corrupt data
DurabilityCommitted data persistsAfter commit, data survives power loss
DefaultBehavior:

EF Core SaveChangesAsync() wraps changes in a transaction automatically.

02

Explicit Transactions

Control transaction boundaries explicitly for multi-step operations.

Without Explicit Transaction var customer = await db.Customers.FindAsync(1); customer.Balance -= 100; await db.SaveChangesAsync(); // Transaction 1 commits var account = await db.Accounts.FindAsync(1); account.Balance += 100; await db.SaveChangesAsync(); // Transaction 2 commits // PROBLEM: If 2nd fails, customer's money already deducted
With Explicit Transaction await using var tx = await db.Database.BeginTransactionAsync(); try { var customer = await db.Customers.FindAsync(1); customer.Balance -= 100; await db.SaveChangesAsync(); var account = await db.Accounts.FindAsync(1); account.Balance += 100; await db.SaveChangesAsync(); await tx.CommitAsync(); } catch { await tx.RollbackAsync(); throw; }
Rule:

Use explicit transactions for multi-table updates where all-or-nothing is critical.

03

Isolation Levels

Control how strictly transactions are isolated from each other.

LevelDirty Read?Non-Repeatable Read?Phantom?Performance
Read UncommittedYesYesYesFastest (avoid)
Read CommittedNoYesYesDefault (good)
Repeatable ReadNoNoYesSlower
SerializableNoNoNoSlowest (strict)
Set Isolation Level await using var tx = await db.Database.BeginTransactionAsync( IsolationLevel.RepeatableRead); // Your queries here are protected from phantom reads // Multiple reads of same row return identical data await tx.CommitAsync();
Default (ReadCommitted):

Safe enough for most apps. Use Serializable only if you need absolute guarantees and can accept slower performance.

04

Optimistic Concurrency

Detect conflicts instead of locking. Multiple users can read; updates detect if data changed.

Optimistic Concurrency with RowVersion public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } [Timestamp] // SQL ROWVERSION/version column public byte[] RowVersion { get; set; } } // Entity config modelBuilder.Entity<Product>(e => { e.Property(p => p.RowVersion) .IsRowVersion(); // Auto-increment on each update }); // Usage try { var product = await db.Products.FindAsync(1); product.Price = 99.99; // If another user updated product between read and write, // SaveChangesAsync throws DbUpdateConcurrencyException await db.SaveChangesAsync(); } catch (DbUpdateConcurrencyException ex) { // Handle conflict: reload and retry await ex.Entries[0].ReloadAsync(); }
When to Use:

High read volume, low conflict rate. Good for web apps. Avoids lock contention.

05

Deadlock Prevention

Deadlock occurs when two transactions wait for each other's locks. Prevent with proper ordering.

Risk: Deadlock // Transaction 1: Lock Customer, then Order var cust = await db.Customers.FindAsync(1); // ... other code ... var order = await db.Orders.FindAsync(10); // Transaction 2: Lock Order, then Customer (opposite order!) var ord = await db.Orders.FindAsync(10); // ... other code ... var cust = await db.Customers.FindAsync(1); // DEADLOCK: T1 holds Cust, waits for Order // T2 holds Order, waits for Cust
Safe: Consistent Order // Always lock in same order // Rule: Lock by table name alphabetical order // Transaction 1 var cust = await db.Customers.FindAsync(1); var order = await db.Orders.FindAsync(10); await db.SaveChangesAsync(); // Transaction 2 (same order) var cust = await db.Customers.FindAsync(1); var order = await db.Orders.FindAsync(10); await db.SaveChangesAsync(); // No deadlock: both follow same locking sequence
Prevention:

Always lock multiple tables in the same order. Keep transactions short. Avoid user think-time inside transactions.
