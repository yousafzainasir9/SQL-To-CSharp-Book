01 — ACID Properties: Database Guarantees

Understanding Atomic, Consistent, Isolated, Durable Operations

ACID is an acronym describing four properties that guarantee reliable database transactions. Atomicity means a transaction either completes entirely or not at all—there's no partial success. Consistency means the database moves from one valid state to another; constraints are always enforced. Isolation means concurrent transactions don't interfere with each other. Durability means once committed, data persists even if the system crashes.

Consider a bank transfer: moving $100 from Account A to Account B. Without ACID guarantees, the system might debit Account A, then crash before crediting Account B. The customer loses money. ACID prevents this: the debit and credit are atomic—both succeed or both fail, never one without the other.

SQL — Transaction Basics -- ACID in action: bank transfer BEGIN TRANSACTION; -- Atomicity: Both operations succeed or both fail UPDATE Accounts SET Balance = Balance - 100 WHERE AccountId = 1; UPDATE Accounts SET Balance = Balance + 100 WHERE AccountId = 2; -- If we get here, both updates succeeded -- If an error occurs, automatic rollback COMMIT; -- Persist changes permanently -- If error before COMMIT: ROLLBACK; -- Undo all changes as if nothing happened -- ACID GUARANTEES: -- Atomicity: Both UPDATEs happen or neither happens -- Consistency: Total balance before = total balance after -- Isolation: Other users don't see intermediate state // Durability: After COMMIT, even power loss preserves data
C# — Transaction Semantics in EF Core public class BankingService { private readonly BankContext _context; public async Task TransferFunds( int fromAccountId, int toAccountId, decimal amount) { // EF Core automatically wraps SaveChanges in a transaction var fromAccount = await _context.Accounts .FirstAsync(a => a.Id == fromAccountId); var toAccount = await _context.Accounts .FirstAsync(a => a.Id == toAccountId); // Update both accounts fromAccount.Balance -= amount; toAccount.Balance += amount; // SaveChanges automatically: // 1. Executes both UPDATEs // 2. If no errors, commits (Atomicity, Durability) // 3. If error during SaveChanges, rolls back (Atomicity) await _context.SaveChangesAsync(); // Database ACID guarantees ensure: // - Both updates happen or neither (Atomicity) // - Balance constraints are enforced (Consistency) // - Other transactions see complete transfer (Isolation) // - After this method returns, data is permanent (Durability) } }
Key Insight: SaveChanges is Already a Transaction

In Entity Framework, a single call to SaveChanges() is automatically wrapped in a transaction. All pending changes—all pending Adds, Updates, and Deletes—are included in that atomic transaction. If any single operation fails, all pending changes are rolled back. You don't need to explicitly create a transaction for single SaveChanges calls.

02 — Transactions in Entity Framework Core

Explicit Transaction Control with DbContext

While SaveChanges is already atomic, sometimes you need multiple SaveChanges calls to be atomic as a unit. For example, you might want to save changes to Orders and LineItems in a single atomic operation, or you need to execute raw SQL commands as part of the same transaction. This is where explicit transactions come in.

SQL — Multi-Statement Transaction -- Explicit transaction wrapping multiple statements BEGIN TRANSACTION; -- Create order INSERT INTO Orders (CustomerId, OrderDate, Total) VALUES (5, GETDATE(), 149.99); DECLARE @OrderId INT = SCOPE_IDENTITY(); -- Add line items INSERT INTO LineItems (OrderId, ProductId, Quantity, Price) VALUES (@OrderId, 42, 3, 49.99); INSERT INTO LineItems (OrderId, ProductId, Quantity, Price) VALUES (@OrderId, 89, 1, 0.00); -- Update product stock UPDATE Products SET StockQuantity = StockQuantity - 3 WHERE ProductId = 42; UPDATE Products SET StockQuantity = StockQuantity - 1 WHERE ProductId = 89; -- All succeed or all fail (Atomic) COMMIT; -- If error: ROLLBACK;
C# — Explicit Transaction in EF Core public class OrderService { private readonly OrderContext _context; public async Task CreateOrderWithItems( int customerId, List lines) { // Method 1: Use explicit transaction await using var transaction = await _context.Database.BeginTransactionAsync(); try { // Create the order var order = new Order { CustomerId = customerId, OrderDate = DateTime.UtcNow, Total = lines.Sum(l => l.Quantity * l.UnitPrice) }; _context.Orders.Add(order); await _context.SaveChangesAsync(); // Add line items foreach (var lineDto in lines) { var lineItem = new LineItem { OrderId = order.Id, ProductId = lineDto.ProductId, Quantity = lineDto.Quantity, UnitPrice = lineDto.UnitPrice }; _context.LineItems.Add(lineItem); // Update product stock var product = await _context.Products .FirstAsync(p => p.Id == lineDto.ProductId); product.StockQuantity -= lineDto.Quantity; } await _context.SaveChangesAsync(); // Explicitly commit await transaction.CommitAsync(); } catch (Exception ex) { // Explicit rollback (though using statement does this) await transaction.RollbackAsync(); throw; } } // Method 2: Simpler pattern - let using statement handle rollback public async Task CreateOrderSimpler( int customerId, List lines) { await using var transaction = await _context.Database.BeginTransactionAsync(); try { var order = new Order { CustomerId = customerId }; _context.Orders.Add(order); await _context.SaveChangesAsync(); foreach (var line in lines) { _context.LineItems.Add(new LineItem { OrderId = order.Id, ProductId = line.ProductId, Quantity = line.Quantity }); } await _context.SaveChangesAsync(); await transaction.CommitAsync(); // If CommitAsync throws, the using statement disposes // the transaction without committing (rollback effect) } catch { // Rollback happens automatically when transaction // is disposed without CommitAsync being called throw; } } }

You can also include raw SQL commands in the same transaction:

SQL — Mixing Raw SQL in Transaction -- All statements in one atomic unit BEGIN TRANSACTION; INSERT INTO Orders (CustomerId, OrderDate) VALUES (5, GETDATE()); DECLARE @OrderId INT = SCOPE_IDENTITY(); INSERT INTO LineItems (OrderId, ProductId, Quantity) VALUES (@OrderId, 42, 3); UPDATE Products SET StockQuantity = StockQuantity - 3 WHERE ProductId = 42; COMMIT;
C# — Mixing SaveChanges and Raw SQL public async Task CreateOrderWithRawSQL( int customerId, int productId, int quantity) { await using var transaction = await _context.Database.BeginTransactionAsync(); try { // EF-tracked change var order = new Order { CustomerId = customerId }; _context.Orders.Add(order); await _context.SaveChangesAsync(); // Raw SQL - included in same transaction await _context.Database.ExecuteSqlAsync( $"INSERT INTO LineItems (OrderId, ProductId, Quantity) " + $"VALUES ({order.Id}, {productId}, {quantity})"); // Another raw SQL command await _context.Database.ExecuteSqlAsync( $"UPDATE Products SET StockQuantity = " + $"StockQuantity - {quantity} WHERE Id = {productId}"); await transaction.CommitAsync(); } catch { throw; } } // WARNING: Use parameterization! // WRONG: ExecuteSqlAsync($"... WHERE Id = {id}") // RIGHT: ExecuteSqlAsync("... WHERE Id = {0}", id) // or: ExecuteSqlAsync("... WHERE Id = @id", // new SqlParameter("@id", id))
03 — Transactions in Dapper

Managing Explicit Transactions with Raw SQL

Dapper is a lightweight micro-ORM that gives you direct control over SQL, including explicit transaction management. Unlike Entity Framework's automatic transaction wrapping, you must explicitly manage transactions in Dapper. This gives you fine-grained control but requires more code.

SQL — Bank Transfer Logic (Reference) BEGIN TRANSACTION; -- Transfer $100 from Account 1 to Account 2 UPDATE Accounts SET Balance = Balance - 100 WHERE AccountId = 1; UPDATE Accounts SET Balance = Balance + 100 WHERE AccountId = 2; COMMIT;
C# — Dapper Transaction Pattern public class DapperBankingRepository { private readonly string _connectionString; public void TransferFunds( int fromAccountId, int toAccountId, decimal amount) { using (var connection = new SqlConnection(_connectionString)) { connection.Open(); // Explicitly begin transaction using (var transaction = connection.BeginTransaction()) { try { // First operation connection.Execute( @"UPDATE Accounts SET Balance = Balance - @Amount WHERE AccountId = @AccountId", new { Amount = amount, AccountId = fromAccountId }, transaction); // Second operation connection.Execute( @"UPDATE Accounts SET Balance = Balance + @Amount WHERE AccountId = @AccountId", new { Amount = amount, AccountId = toAccountId }, transaction); // Commit both operations atomically transaction.Commit(); } catch { // If any operation fails, rollback both transaction.Rollback(); throw; } } } } // Async version public async Task TransferFundsAsync( int fromAccountId, int toAccountId, decimal amount) { using (var connection = new SqlConnection(_connectionString)) { await connection.OpenAsync(); using (var transaction = connection.BeginTransaction()) { try { await connection.ExecuteAsync( @"UPDATE Accounts SET Balance = Balance - @Amount WHERE AccountId = @AccountId", new { Amount = amount, AccountId = fromAccountId }, transaction); await connection.ExecuteAsync( @"UPDATE Accounts SET Balance = Balance + @Amount WHERE AccountId = @AccountId", new { Amount = amount, AccountId = toAccountId }, transaction); transaction.Commit(); } catch { transaction.Rollback(); throw; } } } } }
Key Insight: Dapper vs EF Core Transaction Patterns

EF Core's SaveChanges is automatically transactional, so you only need explicit transaction calls when you have multiple SaveChanges or need to mix SaveChanges with raw SQL. Dapper requires explicit transaction management for every multi-operation sequence. This is a tradeoff: EF automates the common case (single SaveChanges), while Dapper gives you explicit control and requires more ceremony.

04 — Isolation Levels Explained

Controlling How Concurrent Transactions See Each Other

Isolation is the "I" in ACID. When multiple transactions run concurrently, isolation determines what each transaction can see. The strictest isolation level (SERIALIZABLE) is safest but slowest—it's like running transactions one-at-a-time. The loosest level (READ UNCOMMITTED) is fastest but allows dirty reads. Most applications use READ COMMITTED, a middle ground.

SQL — Isolation Levels and Anomalies -- Set isolation level for current connection SET TRANSACTION ISOLATION LEVEL READ COMMITTED; -- Other isolation levels: -- READ UNCOMMITTED: Allows dirty reads (fastest, risky) -- READ COMMITTED: Doesn't allow dirty reads (default) -- REPEATABLE READ: Prevents dirty & non-repeatable reads -- SERIALIZABLE: Strictest, prevents all anomalies (slowest) -- SNAPSHOT: MVCC approach, alternative to SERIALIZABLE -- Example: Dirty read anomaly (READ UNCOMMITTED only) -- Transaction A inserts data but doesn't commit -- Transaction B reads uncommitted data // If Transaction A rolls back, Transaction B has invalid data -- Example: Non-repeatable read (READ UNCOMMITTED/COMMITTED) -- Transaction A reads a row -- Transaction B updates that row and commits -- Transaction A reads the same row again, gets different value // Violates consistency assumption (same query = same result) -- Example: Phantom read -- Transaction A runs SELECT WHERE City = 'Seattle' // Transaction B inserts a new row with City = 'Seattle' // Transaction A re-runs the same SELECT, gets different rows -- ISOLATION LEVEL CHART: // READ UNCOMMITTED: ✓ dirty reads // READ COMMITTED: ✗ dirty, ✓ non-repeatable, ✓ phantom // REPEATABLE READ: ✗ dirty, ✗ non-repeatable, ✓ phantom // SERIALIZABLE: ✗ dirty, ✗ non-repeatable, ✗ phantom
C# — Setting Isolation Level in EF Core public class AccountRepository { private readonly AccountContext _context; // Default isolation level (READ COMMITTED) public async Task GetAccountAsync(int id) { return await _context.Accounts .FirstOrDefaultAsync(a => a.Id == id); } // Custom isolation level for critical operation public async Task GetAccountBalanceStrictAsync(int id) { // Use SERIALIZABLE for critical financial reads await using var transaction = await _context.Database.BeginTransactionAsync( IsolationLevel.Serializable); try { var account = await _context.Accounts .FirstAsync(a => a.Id == id); await transaction.CommitAsync(); return account.Balance; } catch { throw; } } // Loose isolation for reporting/non-critical reads public async Task GetAccountDirtyAsync(int id) { // READ UNCOMMITTED: allows dirty reads but very fast await using var transaction = await _context.Database.BeginTransactionAsync( IsolationLevel.ReadUncommitted); try { var account = await _context.Accounts .FirstAsync(a => a.Id == id); await transaction.CommitAsync(); return account; } catch { throw; } } } // In Dapper: public async Task GetAccountDapperAsync(int id) { using (var connection = new SqlConnection(_connString)) { await connection.OpenAsync(); using (var transaction = connection.BeginTransaction(IsolationLevel.ReadUncommitted)) { var account = await connection.QueryFirstAsync( "SELECT * FROM Accounts WHERE Id = @Id", new { Id = id }, transaction); transaction.Commit(); return account; } } }

The choice of isolation level is a critical performance vs safety decision. READ COMMITTED is the default for good reason: it prevents the most dangerous anomalies (dirty reads) without the performance cost of SERIALIZABLE.

05 — Optimistic Concurrency Control

Detecting Conflicts Without Locking Rows

Optimistic concurrency assumes conflicts are rare. Instead of locking rows, you include a version column in your WHERE clause. If another user modified the row, the version will be different, and your UPDATE will fail. This is efficient for read-heavy workloads where conflicts are uncommon. Entity Framework has built-in support for this pattern.

SQL — Optimistic Concurrency with RowVersion -- Add a RowVersion column (SQL Server timestamp) CREATE TABLE Products ( Id INT PRIMARY KEY, Name NVARCHAR(100), Price DECIMAL(10,2), RowVersion ROWVERSION -- Auto-incrementing, database-managed ); -- Initial state: -- Id=1, Name='Widget', Price=10.00, RowVersion=1 -- User A reads product: RowVersion=1 -- User B reads product: RowVersion=1 -- User B updates: Price=12.00, RowVersion becomes 2 -- User A tries to update with old RowVersion: UPDATE Products SET Price = 11.00 WHERE Id = 1 AND RowVersion = 1; -- Result: 0 rows affected (User B's change won) // User A must: // 1. Refresh their copy (get new RowVersion=2) // 2. Reapply their changes // 3. Try again with RowVersion=2 -- User B's update succeeded because: UPDATE Products SET Price = 12.00 WHERE Id = 1 AND RowVersion = 1; -- 1 row affected, RowVersion auto-incremented to 2
C# — Optimistic Concurrency in EF Core public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } // EF Core recognizes [Timestamp] for optimistic concurrency [Timestamp] public byte[] RowVersion { get; set; } } public class ProductService { private readonly ProductContext _context; public async Task UpdateProductPriceAsync(int productId, decimal newPrice) { var product = await _context.Products .FirstAsync(p => p.Id == productId); // User gets: Id=1, Price=10.00, RowVersion=[0,0,0,1] product.Price = newPrice; try { // SaveChanges includes RowVersion in WHERE clause: // UPDATE Products // SET Price = @Price // WHERE Id = @Id AND RowVersion = @OldRowVersion // // If another user changed the row: // - RowVersion will be different // - UPDATE affects 0 rows // - DbUpdateConcurrencyException is thrown await _context.SaveChangesAsync(); } catch (DbUpdateConcurrencyException ex) { // Conflict detected! // RowVersion mismatch = someone else modified the row // Option 1: Lose changes (take database version) await ex.Entries[0].ReloadAsync(); var current = ex.Entries[0].Entity as Product; Console.WriteLine($"Price is now {current.Price}"); // Option 2: Retry with conflict resolution product.Price = newPrice; await _context.SaveChangesAsync(); // This time it succeeds because we loaded fresh RowVersion } } // Typical conflict resolution pattern public async Task TryUpdateWithRetryAsync( int productId, decimal newPrice, int maxRetries = 3) { for (int attempt = 0; attempt < maxRetries; attempt++) { try { var product = await _context.Products .AsNoTracking() .FirstAsync(p => p.Id == productId); product.Price = newPrice; _context.Products.Update(product); await _context.SaveChangesAsync(); return true; } catch (DbUpdateConcurrencyException) { if (attempt == maxRetries - 1) throw; // Retry with fresh data _context.ChangeTracker.Clear(); } } return false; } }
Key Insight: Optimistic vs Pessimistic Locking

Optimistic concurrency (version checking) is efficient when conflicts are rare. Pessimistic concurrency (explicit locking) is safer when conflicts are common but creates contention. Choose optimistic for read-heavy scenarios, pessimistic for write-heavy scenarios. EF Core makes optimistic easy with the [Timestamp] attribute; SQL Server's ROWVERSION auto-increments on every modification.

06 — Deadlocks: Detection and Prevention

Understanding and Recovering from Circular Lock Waits

A deadlock occurs when two transactions are waiting on each other: Transaction A holds a lock that Transaction B needs, and Transaction B holds a lock that Transaction A needs. Neither can proceed. The database detects this and kills one transaction (the "deadlock victim"), rolling back its changes. The other transaction proceeds.

SQL — Deadlock Example -- TRANSACTION A (Session 1) BEGIN TRANSACTION; UPDATE Accounts SET Balance = Balance - 100 WHERE Id = 1; -- A now holds lock on Accounts.Id=1 -- A wants lock on Accounts.Id=2 WAITFOR DELAY '00:00:05'; -- Simulate work UPDATE Accounts SET Balance = Balance + 100 WHERE Id = 2; COMMIT; -- TRANSACTION B (Session 2) BEGIN TRANSACTION; UPDATE Accounts SET Balance = Balance - 100 WHERE Id = 2; -- B now holds lock on Accounts.Id=2 -- B wants lock on Accounts.Id=1 UPDATE Accounts SET Balance = Balance + 100 WHERE Id = 1; COMMIT; -- What happens: // A updates Account 1 (locks it) // B updates Account 2 (locks it) // A tries to update Account 2 (waits for B's lock) // B tries to update Account 1 (waits for A's lock) // DEADLOCK: Both waiting, neither can proceed // SQL Server kills victim (randomly chosen) // Victim's transaction rolls back completely // Winner's transaction proceeds -- Error in victim transaction: // Msg 1205, Level 13 // Transaction (Process ID 57) was deadlocked on {lock} // resources with another process and has been chosen // as the deadlock victim. Rerun the transaction.
C# — Handling Deadlock with Polly Retry public class ResilientAccountService { private readonly AccountContext _context; private readonly IAsyncPolicy _retryPolicy; public ResilientAccountService(AccountContext context) { _context = context; // Polly policy: retry on deadlock (SQL error 1205) _retryPolicy = Policy .Handle(ex => ex.Number == 1205) .Or(ex => ex.InnerException is SqlException se && se.Number == 1205) .WaitAndRetryAsync( retryCount: 3, sleepDurationProvider: attempt => TimeSpan.FromMilliseconds(100 * (int)Math.Pow(2, attempt)), onRetry: (outcome, timespan, retryCount, context) => { Console.WriteLine( $"Deadlock detected, retry {retryCount} after {timespan.TotalMilliseconds}ms"); }); } // Automatic retry on deadlock public async Task TransferFundsWithRetryAsync( int fromId, int toId, decimal amount) { return await _retryPolicy.ExecuteAsync(async () => { await using var transaction = await _context.Database.BeginTransactionAsync(); try { var from = await _context.Accounts .FirstAsync(a => a.Id == fromId); var to = await _context.Accounts .FirstAsync(a => a.Id == toId); from.Balance -= amount; to.Balance += amount; await _context.SaveChangesAsync(); await transaction.CommitAsync(); return true; } catch { await transaction.RollbackAsync(); throw; } }); } // Prevent deadlocks: acquire locks in consistent order public async Task TransferFundsOrderedAsync( int fromId, int toId, decimal amount) { // Lock accounts in order of ID to prevent circular waits int firstId = Math.Min(fromId, toId); int secondId = Math.Max(fromId, toId); await using var transaction = await _context.Database.BeginTransactionAsync(); try { // Always acquire locks in same order across all transactions var first = await _context.Accounts .FromSqlInterpolated($"SELECT * FROM Accounts WHERE Id = {firstId}") .ToListAsync(); // Forces lock var second = await _context.Accounts .FromSqlInterpolated($"SELECT * FROM Accounts WHERE Id = {secondId}") .ToListAsync(); // Forces lock in order // Now safe to proceed; no circular wait possible var from = _context.Accounts.Local.First(a => a.Id == fromId); var to = _context.Accounts.Local.First(a => a.Id == toId); from.Balance -= amount; to.Balance += amount; await _context.SaveChangesAsync(); await transaction.CommitAsync(); } catch { throw; } } }

The two strategies for deadlock prevention are:

SQL — Deadlock Prevention Strategies -- STRATEGY 1: Lock ordering -- Always acquire locks in the same order across all transactions -- Transaction A (correct): BEGIN TRANSACTION; UPDATE Accounts SET Balance = Balance - 100 WHERE Id = 1; -- Lock 1 UPDATE Accounts SET Balance = Balance + 100 WHERE Id = 2; -- Lock 2 COMMIT; -- Transaction B (must also lock in order): BEGIN TRANSACTION; UPDATE Accounts SET Balance = Balance - 100 WHERE Id = 1; -- Lock 1 UPDATE Accounts SET Balance = Balance + 100 WHERE Id = 2; -- Lock 2 (waits for Trans A) COMMIT; -- No deadlock: B waits for A, then proceeds -- STRATEGY 2: Timeout SET LOCK_TIMEOUT 5000; -- Timeout after 5 seconds BEGIN TRANSACTION; UPDATE Accounts SET ... -- If can't acquire lock within 5 seconds, fail with error -- STRATEGY 3: Deadlock priority (rare) SET DEADLOCK_PRIORITY HIGH; -- This transaction is less likely to be victim SET DEADLOCK_PRIORITY LOW; -- This transaction is more likely to be victim
C# — Deadlock Prevention Best Practices public class DeadlockPreventionService { private readonly AccountContext _context; // Best practice: Always lock in same order public async Task SafeTransferAsync( int fromId, int toId, decimal amount) { // Normalize order var (first, second) = fromId < toId ? (fromId, toId) : (toId, fromId); await using var transaction = await _context.Database.BeginTransactionAsync(); try { // Load in locked order var account1 = await _context.Accounts .FirstAsync(a => a.Id == first); var account2 = await _context.Accounts .FirstAsync(a => a.Id == second); // Now update in original order if (fromId == first) { account1.Balance -= amount; account2.Balance += amount; } else { account2.Balance -= amount; account1.Balance += amount; } await _context.SaveChangesAsync(); await transaction.CommitAsync(); } catch (SqlException ex) when (ex.Number == 1205) { // Deadlock victim - this is rare if lock ordering is correct throw new InvalidOperationException( "Deadlock occurred despite lock ordering. Retry later.", ex); } } // Use short transactions (smaller window for conflicts) public async Task QuickTransferAsync( int fromId, int toId, decimal amount) { // Minimize transaction scope var fromAccount = await _context.Accounts .AsNoTracking() .FirstAsync(a => a.Id == fromId); var toAccount = await _context.Accounts .AsNoTracking() .FirstAsync(a => a.Id == toId); // Calculate outside transaction var newFromBalance = fromAccount.Balance - amount; var newToBalance = toAccount.Balance + amount; // Short transaction with just the updates await using var transaction = await _context.Database.BeginTransactionAsync(); try { await _context.Database.ExecuteSqlAsync( $"UPDATE Accounts SET Balance = {newFromBalance} WHERE Id = {fromId}"); await _context.Database.ExecuteSqlAsync( $"UPDATE Accounts SET Balance = {newToBalance} WHERE Id = {toId}"); await transaction.CommitAsync(); } catch { throw; } } }

Key Takeaways

  • ACID properties (Atomicity, Consistency, Isolation, Durability) guarantee reliable database transactions even during failures
  • In Entity Framework, SaveChanges() is automatically atomic; use explicit transactions only when you need multiple SaveChanges or mixing SaveChanges with raw SQL
  • Dapper requires explicit transaction management for every multi-operation sequence via IDbTransaction
  • Isolation levels control how concurrent transactions see each other; READ COMMITTED is the default and usually optimal
  • Optimistic concurrency using row version columns ([Timestamp] in EF) detects conflicts efficiently without locking
  • Deadlocks occur when transactions wait on each other in circular fashion; prevent them by acquiring locks in consistent order and keeping transactions short
  • Use Polly retry policies to automatically recover from deadlock victims with exponential backoff
