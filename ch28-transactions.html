Chapter 28

Transactions

Transfer money between accounts. Both must succeed or both fail. Learn ACID properties, explicit transactions, isolation levels, and deadlock prevention.

01

Requirement: "Transfer Money Atomically"

Business Requirement
Debit one account and credit another. If credit fails after debit, we're broken. Both changes must commit together or both roll back.
SQL Developer Thinks
Wrap both operations in BEGIN TRANSACTION...COMMIT. If anything fails, ROLLBACK automatically. This is ACID atomicity.
C# Developer Thinks
SaveChangesAsync() wraps changes in a transaction automatically. For multi-step operations, use explicit transaction with try/catch.
ACID PropertyMeaningExample
AtomicityAll or nothingBoth debit and credit succeed or both fail
ConsistencyValid state maintainedBalances never negative
IsolationNo interferenceTwo transfers don't corrupt data
DurabilityPersists after commitAfter COMMIT, survives power loss
SQL
BEGIN TRANSACTION;
UPDATE Accounts SET Balance = Balance - 100 WHERE Id = 1;
UPDATE Accounts SET Balance = Balance + 100 WHERE Id = 2;
COMMIT TRANSACTION;
C# EF Core
await using var tx = await db.Database.BeginTransactionAsync(ct);
try {
  var from = await db.Accounts.FindAsync(new object[] { 1 }, cancellationToken: ct);
  from.Balance -= 100;
  var to = await db.Accounts.FindAsync(new object[] { 2 }, cancellationToken: ct);
  to.Balance += 100;
  await db.SaveChangesAsync(ct);
  await tx.CommitAsync(ct);
} catch {
  await tx.RollbackAsync(ct); throw;
}
02

Isolation Levels & Dirty Reads

Business Requirement
Two users update the same record simultaneously. How strict should isolation be?
SQL Developer Thinks
Set ISOLATION LEVEL. Default is READ COMMITTED. SERIALIZABLE is strictest but slowest.
C# Developer Thinks
Pass IsolationLevel to BeginTransactionAsync(). Most apps use default READ COMMITTED.
LevelDirty Read?Non-Rep Read?Phantom?Speed
Read UncommittedYesYesYesFast (avoid)
Read CommittedNoYesYesDefault
Repeatable ReadNoNoYesSlower
SerializableNoNoNoSlowest
C# Set Isolation Level
await using var tx = await db.Database.BeginTransactionAsync(IsolationLevel.RepeatableRead, ct);
// Your queries here protected from phantom reads
await tx.CommitAsync(ct);
03

Optimistic Concurrency

Business Requirement
Multiple users can read; detect if data changed between read and write.
SQL Developer Thinks
Use a RowVersion column that auto-increments on each update. Include it in WHERE clause of UPDATE.
C# Developer Thinks
Add byte[] RowVersion property with [Timestamp]. EF compares before updating.
C# with RowVersion
public class Product {
  public int Id { get; set; }
  public string Name { get; set; }
  [Timestamp] public byte[] RowVersion { get; set; }
}

var product = await db.Products.FindAsync(1);
product.Name = "New Name";
await db.SaveChangesAsync(ct);  // Throws DbUpdateConcurrencyException if changed
04

Deadlock Prevention

Business Requirement
Two transactions wait for each other's locks. How do we prevent deadlock?
SQL Developer Thinks
Always lock tables in the same order. Transaction 1 locks Customer then Order. Transaction 2 must also lock Customer then Order.
C# Developer Thinks
Same principle. Load entities in alphabetical table order.
C# Safe Order
// Always lock: Accounts, then Orders
var acct = await db.Accounts.FindAsync(1, ct);
var order = await db.Orders.FindAsync(10, ct);
// Modify and save
await db.SaveChangesAsync(ct);
// Both transactions follow same order = no deadlock
