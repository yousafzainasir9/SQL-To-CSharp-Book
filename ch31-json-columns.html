Chapter 31

JSON Columns

You get a requirement: “Store flexible settings without creating extra tables.” JSON columns store semi-structured data right in a table column. A SQL developer uses JSON_VALUE() and JSON_MODIFY(). A C# developer uses value converters and owned types. Both store and query flexible data — here’s how.

01

Requirement: “Store User Preferences”

Business Requirement
Each user has different settings: theme, language, notification preferences, dashboard layout. The shape varies per user. Creating a column for each setting is impractical.
SQL Developer Thinks I’ll add an NVARCHAR(MAX) column and store JSON. SQL Server has JSON_VALUE() and JSON_QUERY() to read into it. No extra tables needed.
C# Developer Thinks I’ll create a UserPreferences class and let EF Core serialize/deserialize it automatically. I work with a strongly-typed object, not raw JSON strings.
SQL — Store JSON
CREATE TABLE Users (
    Id          INT IDENTITY(1,1) PRIMARY KEY,
    Name        NVARCHAR(100) NOT NULL,
    Email       NVARCHAR(255) NOT NULL,
    Preferences NVARCHAR(MAX) NULL
);

INSERT INTO Users (Name, Email, Preferences)
VALUES ('Alice', 'alice@example.com',
  '{"theme":"dark","lang":"en",
    "notifications":{"email":true,
    "sms":false},"pageSize":25}');
C# — Strongly-Typed Class
public class User
{
    public int Id { get; set; }
    public string Name { get; set; } = "";
    public string Email { get; set; } = "";
    public UserPreferences? Preferences
        { get; set; }
}

public class UserPreferences
{
    public string Theme { get; set; } = "light";
    public string Lang { get; set; } = "en";
    public NotificationSettings Notifications
        { get; set; } = new();
    public int PageSize { get; set; } = 20;
}

public class NotificationSettings
{
    public bool Email { get; set; } = true;
    public bool Sms { get; set; } = false;
}
02

Requirement: “Map C# Object to JSON Column”

Business Requirement
When I save a User, Preferences should auto-serialize to JSON. When I load, it auto-deserializes. No manual JSON parsing.
SQL Developer Thinks The column stores text. JSON is just a string to the database. I use JSON functions to query it when needed.
C# Developer Thinks EF Core 7+ supports OwnsOne with ToJson() — maps a nested C# object to a JSON column automatically. For older versions, HasConversion works.
C# — EF Core 7+ OwnsOne + ToJson()
modelBuilder.Entity<User>(e =>
{
    e.HasKey(u => u.Id);
    e.Property(u => u.Name).IsRequired().HasMaxLength(100);
    e.Property(u => u.Email).IsRequired().HasMaxLength(255);

    // Map Preferences to a JSON column
    e.OwnsOne(u => u.Preferences, prefs =>
    {
        prefs.ToJson();
        prefs.OwnsOne(p => p.Notifications);
    });
});
C# — HasConversion (EF Core 6 and earlier)
modelBuilder.Entity<User>(e =>
{
    e.Property(u => u.Preferences)
        .HasConversion(
            v => JsonSerializer.Serialize(v, (JsonSerializerOptions?)null),
            v => JsonSerializer.Deserialize<UserPreferences>(v,
                (JsonSerializerOptions?)null))
        .HasColumnType("nvarchar(max)");
});
C# — Usage (Auto Serialization)
// Create — object serializes to JSON automatically
var user = new User
{
    Name = "Alice", Email = "alice@example.com",
    Preferences = new UserPreferences
    {
        Theme = "dark", Lang = "en", PageSize = 25,
        Notifications = new() { Email = true, Sms = false }
    }
};
context.Users.Add(user);
await context.SaveChangesAsync(ct);

// Read — JSON deserializes automatically
var loaded = await context.Users
    .AsNoTracking()
    .FirstOrDefaultAsync(u => u.Id == user.Id, ct);
var theme = loaded?.Preferences?.Theme; // "dark"
03

Requirement: “Query Inside the JSON”

Business Requirement
Find all users who prefer dark mode. Find users with email notifications enabled. Filter and sort by JSON properties.
SQL — JSON_VALUE()
-- Find dark mode users
SELECT Id, Name
FROM Users
WHERE JSON_VALUE(Preferences, '$.theme')
    = 'dark';

-- Sort by page size
SELECT Id, Name,
    JSON_VALUE(Preferences, '$.pageSize')
        AS PageSize
FROM Users
ORDER BY CAST(JSON_VALUE(Preferences,
    '$.pageSize') AS INT) DESC;

-- Query nested value
SELECT Id, Name
FROM Users
WHERE JSON_VALUE(Preferences,
    '$.notifications.email') = 'true';
C# — LINQ with Owned Types (EF 7+)
// Find dark mode users
var darkUsers = await context.Users
    .AsNoTracking()
    .Where(u => u.Preferences!.Theme
        == "dark")
    .ToListAsync(ct);

// Sort by page size
var sorted = await context.Users
    .AsNoTracking()
    .OrderByDescending(u =>
        u.Preferences!.PageSize)
    .ToListAsync(ct);

// Query nested value
var emailNotif = await context.Users
    .AsNoTracking()
    .Where(u => u.Preferences!
        .Notifications.Email == true)
    .ToListAsync(ct);
Dapper — Direct JSON Queries
await using var conn = new SqlConnection(connectionString);
var darkUsers = await conn.QueryAsync<User>(
    @"SELECT Id, Name, Email, Preferences
      FROM Users
      WHERE JSON_VALUE(Preferences, '$.theme') = @Theme",
    new { Theme = "dark" });
04

Requirement: “Update Part of the JSON”

Business Requirement
User clicks “Switch to dark mode.” Update only the theme inside their preferences — don’t replace the entire object.
SQL — JSON_MODIFY()
-- Update only the theme
UPDATE Users
SET Preferences = JSON_MODIFY(
    Preferences, '$.theme', 'dark')
WHERE Id = 42;

-- Update nested value
UPDATE Users
SET Preferences = JSON_MODIFY(
    Preferences,
    '$.notifications.sms',
    CAST(1 AS BIT))
WHERE Id = 42;

-- Add a new key
UPDATE Users
SET Preferences = JSON_MODIFY(
    Preferences, '$.sidebar', 'collapsed')
WHERE Id = 42;
C# — Two Approaches
// Approach 1: Load, modify, save
// (EF re-serializes entire JSON)
var user = await context.Users
    .FirstOrDefaultAsync(
        u => u.Id == 42, ct);

if (user?.Preferences is not null)
{
    user.Preferences.Theme = "dark";
    await context.SaveChangesAsync(ct);
}

// Approach 2: Surgical raw SQL
await context.Database
    .ExecuteSqlInterpolatedAsync(
    $@"UPDATE Users
       SET Preferences = JSON_MODIFY(
           Preferences, '$.theme', {theme})
       WHERE Id = {userId}", ct);
05

“Should This Be JSON or a Separate Table?”

The Question
Product has attributes that vary by category (shoes have size/color, laptops have RAM/CPU). JSON column or separate table?
SQL Developer Thinks If I need to JOIN, GROUP BY, or index the data — use tables. If it’s read-mostly and schema varies — JSON is fine. JSON queries are slower than indexed columns.
C# Developer Thinks If the data has a fixed shape — use a table with proper entities. If it’s flexible, user-defined, or rarely queried — JSON keeps the model simple.
ScenarioUse JSONUse Separate Table
User preferences / settingsYes — flexible, rarely queried—
Product attributes (vary by type)Yes — schema varies per productIf you filter/sort by attributes
Audit log metadataYes — read-only, archival—
Order line items—Yes — need JOINs, aggregates, FKs
Address fields—Yes — queried frequently, indexed
API request/response logYes — blob storage, rarely parsed—
Permissions / roles—Yes — relational queries needed
Form builder dynamic fieldsYes — user-defined schema—
