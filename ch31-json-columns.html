Working with JSON Columns
Modern Data Patterns

Modern databases support JSON columns, blending structured and semi-structured data. Learn how SQL Server and PostgreSQL handle JSON, how to query it, and when JSON is the right choice over normalized relational tables.

In This Chapter

  • SQL Server's NVARCHAR(MAX) with JSON functions and PostgreSQL's native jsonb type
  • Querying JSON with JSON_VALUE, JSON_QUERY, OPENJSON, and PostgreSQL operators
  • EF Core value conversions and the .ToJson() mapping for owned entities
  • Querying JSON properties with EF Core and understanding generated SQL
  • Dapper custom type handlers for automatic JSON deserialization
  • Choosing between JSON columns and relational tables based on your use case
01 — JSON in SQL Server and PostgreSQL

Native JSON Support in Modern Databases

SQL Server and PostgreSQL both offer JSON support, but with different approaches. SQL Server stores JSON as NVARCHAR(MAX) and provides T-SQL functions to query it. PostgreSQL has a native jsonb type with operators optimized for JSON access. Both are powerful for flexible attributes, configuration data, and audit logs.

SQL Server -- Create table with JSON column CREATE TABLE Products ( Id INT PRIMARY KEY IDENTITY, Name NVARCHAR(100) NOT NULL, Price DECIMAL(10,2), -- JSON as text, validated Metadata NVARCHAR(MAX) CHECK (ISJSON(Metadata) = 1) ); -- Example JSON in Metadata: -- { -- "color": "Red", -- "weight": 500, -- "dimensions": { -- "length": 10, -- "width": 8 -- }, -- "tags": ["electronics", "new"] -- } -- SQL Server validates JSON on insert INSERT INTO Products VALUES ( 1, 'Smartphone', 799.99, '{"color":"Black","storage":256}' );
C# Domain Model public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } // Metadata as C# object // Will be serialized to JSON // when stored in database public ProductMetadata Metadata { get; set; } } public class ProductMetadata { public string Color { get; set; } public int? Weight { get; set; } public Dimensions Dimensions { get; set; } public List Tags { get; set; } } public class Dimensions { public int Length { get; set; } public int Width { get; set; } }
PostgreSQL -- Create table with native jsonb CREATE TABLE products ( id SERIAL PRIMARY KEY, name VARCHAR(100) NOT NULL, price NUMERIC(10,2), -- jsonb is native type, indexed metadata jsonb NOT NULL DEFAULT '{}'::jsonb ); -- Insert JSON directly INSERT INTO products VALUES ( 1, 'Smartphone', 799.99, '{ "color": "Black", "storage": 256, "tags": ["electronics"] }'::jsonb ); -- jsonb allows indexing on -- nested properties (GIN indexes) CREATE INDEX idx_metadata ON products USING GIN (metadata);
C# with Npgsql (PostgreSQL) using Npgsql; using NpgsqlTypes; public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } // JsonDocument or custom class // Npgsql handles mapping automatically [Column(TypeName = "jsonb")] public ProductMetadata Metadata { get; set; } } // Configure EF Core for PostgreSQL protected override void OnModelCreating( ModelBuilder modelBuilder) { modelBuilder.Entity() .Property(p => p.Metadata) .HasColumnType("jsonb"); }
Key Insight: Storage Format Differences

SQL Server stores JSON as text (NVARCHAR(MAX)) with validation but no special indexing. PostgreSQL's jsonb is a binary format that's compressed, indexed, and optimized for queries. This makes PostgreSQL better for JSON-heavy workloads, while SQL Server's approach is simpler for occasional JSON use.

02 — Querying JSON with SQL

Extracting Data from JSON Columns

Both databases provide functions to extract JSON values. SQL Server uses JSON_VALUE (single scalar), JSON_QUERY (objects/arrays), and OPENJSON (table-like access). PostgreSQL uses operators like ->> (text), -> (object), and #> (nested paths). Understanding these is crucial for filtering and projection in raw SQL.

SQL Server JSON Functions -- JSON_VALUE: Extract scalar value SELECT Id, Name, JSON_VALUE(Metadata, '$.color') AS Color FROM Products WHERE JSON_VALUE(Metadata, '$.color') = 'Red'; -- JSON_QUERY: Extract object/array SELECT Id, JSON_QUERY(Metadata, '$.dimensions') AS DimensionData FROM Products; -- OPENJSON: Treat JSON array as table SELECT p.Id, p.Name, tags.value AS Tag FROM Products p CROSS APPLY OPENJSON( JSON_QUERY(p.Metadata, '$.tags') ) AS tags; -- Filtering on numeric JSON values SELECT Id, Name FROM Products WHERE CAST( JSON_VALUE(Metadata, '$.weight') AS INT) > 100; -- Array operations SELECT Id, Name FROM Products WHERE JSON_VALUE(Metadata, '$.tags[0]') = 'electronics';
C# Raw SQL with Dapper public class ProductQueryService { private readonly IDbConnection _conn; public async Task> GetProductsByColorAsync(string color) { const string sql = @" SELECT Id, Name, JSON_VALUE(Metadata, '$.color') AS Color, JSON_VALUE(Metadata, '$.weight') AS Weight FROM Products WHERE JSON_VALUE(Metadata, '$.color') = @color "; return (await _conn.QueryAsync( sql, new { color }) ).ToList(); } public async Task> GetProductsWithTagsAsync() { const string sql = @" SELECT p.Id, p.Name, tags.value AS Tag FROM Products p CROSS APPLY OPENJSON( JSON_QUERY(p.Metadata, '$.tags') ) AS tags "; var data = await _conn.QueryAsync(sql); // Group manually since Dapper // can't auto-group OPENJSON results return data .GroupBy(d => (int)d.Id) .Select(g => new ProductWithTags { Id = g.Key, Name = g.First().Name, Tags = g.Select(x => (string)x.Tag) .ToList() }) .ToList(); } }
PostgreSQL JSON Operators -- -> : Access object value (returns jsonb) SELECT id, name, metadata -> 'color' AS color_json FROM products; -- ->> : Access value as text SELECT id, name, metadata ->> 'color' AS color FROM products WHERE metadata ->> 'color' = 'Red'; -- #> : Access nested path (jsonb result) SELECT id, metadata #> '{dimensions,length}' AS length FROM products; -- @> : Contains (for array filtering) SELECT id, name FROM products WHERE metadata @> '{"tags":["electronics"]}'::jsonb; -- ? : Key exists SELECT id, name FROM products WHERE metadata ? 'weight'; -- @@ : Full-text search in JSON SELECT id, name FROM products WHERE metadata @@ 'color:Red';
C# with Npgsql (PostgreSQL) using Npgsql.PluginTests; public class PostgresProductService { private readonly OrderContext _context; // PostgreSQL supports JSON in EF Core // but with limited LINQ translation public async Task> GetProductsByColorAsync(string color) { // Some LINQ works, but many operations // require raw SQL for jsonb var sql = @" SELECT id, name, metadata ->> 'color' as color FROM products WHERE metadata ->> 'color' = {0} "; return await _context.Products .FromSqlInterpolated(sql, color) .Select(p => new ProductDto { Id = p.Id, Name = p.Name }) .ToListAsync(); } // For complex JSON queries, // prefer Dapper public async Task> GetElectronicsAsync() { const string sql = @" SELECT id, name FROM products WHERE metadata @> '{"tags": ["electronics"]}'::jsonb "; using var conn = new NpgsqlConnection(_connectionString); return (await conn.QueryAsync( sql )).ToList(); } }
Key Insight: OPENJSON vs @> Operator

SQL Server's OPENJSON is useful when you need to expand JSON arrays into rows (like a pivot). PostgreSQL's @> operator is better for containment checks (does the JSON include this value?). Choose the right tool based on whether you're expanding or filtering.

03 — EF Core Value Conversions for JSON

Mapping C# Objects to JSON Columns

EF Core's value conversions allow you to serialize complex C# objects to JSON when storing in the database. This keeps your domain model clean while leveraging JSON storage. EF Core 7+ adds .ToJson() for owned entity types, which automatically handles the conversion.

Value Conversion Pattern using System.Text.Json; using Microsoft.EntityFrameworkCore; public class OrderContext : DbContext { public DbSet Products { get; set; } protected override void OnModelCreating( ModelBuilder modelBuilder) { // ═══════════════════════════════════════ // PATTERN 1: Manual Value Conversion // ═══════════════════════════════════════ modelBuilder.Entity() .Property(p => p.Metadata) .HasConversion( // To database: serialize to JSON string v => JsonSerializer.Serialize(v), // From database: deserialize from string v => JsonSerializer.Deserialize(v) ?? new ProductMetadata() ) .HasColumnType("nvarchar(max)"); // SQL Server specific // ═══════════════════════════════════════ // PATTERN 2: EF Core 7+ .ToJson() // (Recommended) // ═══════════════════════════════════════ modelBuilder.Entity() .OwnsOne(p => p.Metadata, nav => { // Automatically maps to JSON column // Handles serialization/deserialization nav.ToJson(); // Still configure nested properties nav.OwnsOne(m => m.Dimensions); }); // ═══════════════════════════════════════ // PATTERN 3: With Custom JSON Options // ═══════════════════════════════════════ var options = new JsonSerializerOptions { PropertyNamingPolicy = JsonNamingPolicy.CamelCase, WriteIndented = false }; modelBuilder.Entity() .Property(p => p.Metadata) .HasConversion( v => JsonSerializer.Serialize( v, options), v => JsonSerializer.Deserialize( v, options) ?? new ProductMetadata() ) .HasColumnType("nvarchar(max)"); } }
C# Domain Model with Owned Entities public class Product { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } // Owned entity type // Maps to JSON when configured with ToJson() public ProductMetadata Metadata { get; set; } } // Owned entity type public class ProductMetadata { public string Color { get; set; } public int? Weight { get; set; } public string Material { get; set; } public Dimensions Dimensions { get; set; } public List Tags { get; set; } = new(); } public class Dimensions { public int Length { get; set; } public int Width { get; set; } public int Height { get; set; } } // ============================================ // Usage in application // ============================================ var product = new Product { Id = 1, Name = "Smartphone", Price = 799.99m, Metadata = new ProductMetadata { Color = "Black", Weight = 200, Material = "Aluminum", Dimensions = new Dimensions { Length = 160, Width = 75, Height = 8 }, Tags = new() { "electronics", "premium" } } }; await _context.Products.AddAsync(product); await _context.SaveChangesAsync(); // In database, Metadata column contains: // {"color":"Black","weight":200, // "material":"Aluminum", // "dimensions": // {"length":160,"width":75,"height":8}, // "tags":["electronics","premium"]}
Generated SQL (EF Core ToJson) -- When saving Product with Metadata INSERT INTO Products (Name, Price, Metadata) VALUES ('Smartphone', 799.99, '{"color":"Black","weight":200,...}'); -- EF Core automatically serializes -- the ProductMetadata object to JSON -- Generated COLUMN constraint -- (SQL Server 2017+) CREATE TABLE Products ( Id INT PRIMARY KEY IDENTITY, Name NVARCHAR(100), Price DECIMAL(10,2), Metadata NVARCHAR(MAX) CONSTRAINT DF_Metadata DEFAULT (JSON_OBJECT()), CONSTRAINT CK_Metadata_JSON CHECK (ISJSON(Metadata) = 1) ); -- Retrieved data is automatically -- deserialized back to C# objects
Key Insight: ToJson() vs Value Conversion

.ToJson() was introduced in EF Core 7 and provides automatic bidirectional conversion for owned entity types. It's cleaner than manual value conversions and better integrates with EF's change tracking. Use .ToJson() for new projects; value conversions are useful when retrofitting existing code.

04 — Querying JSON with EF Core

LINQ on JSON Properties and Generated SQL

EF Core's LINQ can query JSON properties, though support varies by database. For SQL Server, property access on JSON columns must be done carefully. With .ToJson() mapped entities, EF generates appropriate SQL based on your predicates. Understanding the generated SQL is critical for performance.

LINQ Queries on JSON using Microsoft.EntityFrameworkCore; public class ProductQueryService { private readonly OrderContext _context; // ═══════════════════════════════════════ // PATTERN 1: Simple Property Access // ═══════════════════════════════════════ public async Task> GetRedProductsAsync() { // LINQ: Access JSON property return await _context.Products .Where(p => p.Metadata.Color == "Red") .ToListAsync(); // Generated SQL (SQL Server): // SELECT * FROM Products // WHERE JSON_VALUE(Metadata, '$.color') = 'Red' } // ═══════════════════════════════════════ // PATTERN 2: Numeric Comparisons // ═══════════════════════════════════════ public async Task> GetLightProductsAsync(int maxWeight) { return await _context.Products .Where(p => p.Metadata.Weight < maxWeight) .ToListAsync(); // Generated SQL: // WHERE CAST( // JSON_VALUE(Metadata, '$.weight') // AS INT) < @p0 } // ═══════════════════════════════════════ // PATTERN 3: Nested Property Access // ═══════════════════════════════════════ public async Task> GetLongProductsAsync(int minLength) { return await _context.Products .Where(p => p.Metadata.Dimensions.Length > minLength) .ToListAsync(); // Generated SQL: // WHERE CAST( // JSON_VALUE(Metadata, // '$.dimensions.length') // AS INT) > @p0 } // ═══════════════════════════════════════ // PATTERN 4: Array Contains (Limited) // ═══════════════════════════════════════ public async Task> GetTaggedProductsAsync(string tag) { // This may NOT translate well to SQL // EF's array support in JSON is limited return await _context.Products .Where(p => p.Metadata.Tags.Contains(tag)) .ToListAsync(); // Might require: // .AsEnumerable().Where(...) // if EF can't translate to SQL // For reliable behavior, use raw SQL const string sql = @" SELECT * FROM Products WHERE JSON_VALUE(Metadata, '$.tags[0]') = {0} OR JSON_VALUE(Metadata, '$.tags[1]') = {0} OR JSON_VALUE(Metadata, '$.tags[2]') = {0} "; return await _context.Products .FromSqlInterpolated(sql, tag) .ToListAsync(); } // ═══════════════════════════════════════ // PATTERN 5: Complex Filtering // ═══════════════════════════════════════ public async Task> GetPremiumBlackProductsAsync() { return await _context.Products .Where(p => p.Metadata.Color == "Black" && p.Metadata.Tags.Any(t => t == "premium") && p.Price > 500) .OrderByDescending(p => p.Price) .ToListAsync(); // EF generates appropriate WHERE clauses // Combining JSON extraction with // regular columns } // ═══════════════════════════════════════ // PATTERN 6: Projection from JSON // ═══════════════════════════════════════ public async Task> GetProductSummariesAsync() { return await _context.Products .Select(p => new ProductSummary { Id = p.Id, Name = p.Name, Color = p.Metadata.Color, Weight = p.Metadata.Weight, Dimensions = p.Metadata.Dimensions }) .ToListAsync(); // EF projects directly from JSON // Extracting only needed properties } }
Key Insight: LINQ Translation Limits

EF Core translates simple property access and comparisons well, but complex operations (like filtering on array elements) may not translate. Check the generated SQL by using .ToQueryString(). If EF can't translate, fall back to .FromSqlRaw() or .FromSqlInterpolated() for control and clarity.

05 — Dapper with JSON Columns

Custom Type Handlers for Automatic JSON Deserialization

Dapper doesn't automatically deserialize JSON columns. You need custom type handlers that convert JSON strings to C# objects. This pattern is essential when using Dapper for queries that include JSON data, especially on the read side of a CQRS architecture.

Dapper Custom Type Handler for JSON using System.Data; using System.Text.Json; using Dapper; // ════════════════════════════════════════════ // STEP 1: Create Generic Type Handler // ════════════════════════════════════════════ public class JsonTypeHandler : SqlMapper.TypeHandler where T : class { public override void SetValue( IDbDataParameter parameter, T value) { if (value == null) { parameter.Value = DBNull.Value; } else { parameter.Value = JsonSerializer.Serialize(value); parameter.DbType = DbType.String; } } public override T Parse(object value) { if (value == null || value is DBNull) return null; var json = value.ToString(); return JsonSerializer.Deserialize(json); } } // ════════════════════════════════════════════ // STEP 2: Register Handler at Startup // ════════════════════════════════════════════ public static class ServiceConfiguration { public static void ConfigureDapper() { // Register handler for ProductMetadata SqlMapper.AddTypeHandler( new JsonTypeHandler()); SqlMapper.AddTypeHandler( new JsonTypeHandler()); SqlMapper.AddTypeHandler( new JsonTypeHandler>()); } } // ════════════════════════════════════════════ // STEP 3: Use in Queries // ════════════════════════════════════════════ public class ProductQueryRepository { private readonly IDbConnection _connection; public ProductQueryRepository( IDbConnection connection) { _connection = connection; } public async Task> GetAllProductsAsync() { const string sql = @" SELECT Id, Name, Price, Metadata FROM Products ORDER BY Name "; // Dapper automatically uses JsonTypeHandler // to deserialize Metadata string to // ProductMetadata object var products = await _connection .QueryAsync(sql); return products.ToList(); } public async Task> GetProductsByColorAsync(string color) { const string sql = @" SELECT Id, Name, Price, Metadata FROM Products WHERE JSON_VALUE(Metadata, '$.color') = @Color ORDER BY Name "; var products = await _connection .QueryAsync( sql, new { Color = color }); return products.ToList(); } // ════════════════════════════════════════ // Multi-mapping with JSON (useful pattern) // ════════════════════════════════════════ public async Task> GetProductsWithReviewsAsync() { const string sql = @" SELECT p.Id, p.Name, p.Price, p.Metadata, r.Id as ReviewId, r.Rating, r.Comment FROM Products p LEFT JOIN Reviews r ON p.Id = r.ProductId ORDER BY p.Id, r.Id "; var productDict = new Dictionary(); await _connection.QueryAsync< ProductWithReviews, Review, ProductWithReviews>( sql, (product, review) => { if (!productDict.TryGetValue( product.Id, out var p)) { p = product; productDict[p.Id] = p; } if (review != null) { p.Reviews.Add(review); } return p; }, splitOn: "ReviewId"); return productDict.Values.ToList(); } } // ════════════════════════════════════════════ // Usage in application code // ════════════════════════════════════════════ // Configure at startup public static void Main(string[] args) { ServiceConfiguration.ConfigureDapper(); // Now queries work seamlessly: var products = await repository .GetAllProductsAsync(); // Metadata is fully deserialized var blackProducts = products .Where(p => p.Metadata.Color == "Black") .ToList(); }
Key Insight: Registering Handlers Once

Register type handlers once at application startup, not per query. This is a static, application-wide configuration that Dapper caches. Forgetting to register is a common mistake that results in JSON strings instead of deserialized objects.

06 — When to Use JSON vs Relational Tables

Choosing the Right Storage Pattern

JSON columns are powerful but not always the right choice. Use JSON for flexible attributes, configuration data, and audit logs. Use relational tables for frequently queried data, relationships, and aggregations. Often the best design combines both: relational core with JSON for extended attributes.

Decision Matrix: JSON vs Relational // ════════════════════════════════════════════ // USE JSON COLUMNS WHEN: // ════════════════════════════════════════════ // 1. Flexible, semi-structured attributes // Products have different attributes public class Product { public int Id { get; set; } public string Name { get; set; } // Phone: {storage, color, os} // Laptop: {cpu, ram, screen-size} // Shoes: {size, material, brand} // Store variable data structure public Dictionary Attributes { get; set; } } // Query: GetByAttribute("storage", "256GB") // Hard to do in relational (many joins) // Easy in JSON (JSON_VALUE) // 2. Configuration and settings public class AppConfiguration { public int Id { get; set; } public string AppName { get; set; } // Config changes frequently // Not queried often // Just serialized/deserialized public AppSettings Settings { get; set; } } // 3. Audit logs and event data public class AuditLog { public int Id { get; set; } public DateTime Timestamp { get; set; } public string EntityType { get; set; } public int EntityId { get; set; } // What changed: variable per entity // Queried by EntityId, not by fields public Dictionary Changes { get; set; } } public class FieldChange { public string OldValue { get; set; } public string NewValue { get; set; } } // ════════════════════════════════════════════ // USE RELATIONAL TABLES WHEN: // ════════════════════════════════════════════ // 1. Frequently queried fields // Customers always filter by: // Name, Email, Country, City, Status public class Customer { public int Id { get; set; } public string Name { get; set; } public string Email { get; set; } public string Country { get; set; } public string City { get; set; } public string Status { get; set; } // Multiple indexes, fast lookups // Many queries like: // WHERE Country = 'USA' AND Status = 'Active' } // 2. Relationships and joins public class Order { public int Id { get; set; } public int CustomerId { get; set; } // FK public DateTime OrderDate { get; set; } public decimal Total { get; set; } } public class OrderItem { public int Id { get; set; } public int OrderId { get; set; } // FK public int ProductId { get; set; } // FK public int Quantity { get; set; } } // Query: GetOrdersWithItems() // Need JOIN, FOREIGN KEYS // JSON can't express relationships // 3. Aggregations // SELECT COUNT(*) FROM Orders // WHERE CustomerId = 5 // SELECT SUM(Total) FROM Orders // WHERE Status = 'Completed' // SELECT AVG(OrderValue) FROM Orders // WHERE OrderDate >= '2024-01-01' // JSON doesn't optimize aggregations // ════════════════════════════════════════════ // HYBRID APPROACH (RECOMMENDED) // ════════════════════════════════════════════ public class Product { // Core fields (normalized, indexed) public int Id { get; set; } public string Name { get; set; } public int CategoryId { get; set; } // FK public decimal Price { get; set; } public DateTime CreatedDate { get; set; } public int StockLevel { get; set; } // Category relationship public Category Category { get; set; } // Extended attributes (semi-structured) // Phone: {color, storage, os} // Laptop: {cpu, ram, screen} // Not queried in WHERE, only viewed [Column(TypeName = "jsonb")] public Dictionary ExtendedAttributes { get; set; } // Configuration (never queried) public ProductConfiguration Configuration { get; set; } } // Query pattern: // WHERE CategoryId = 5 AND Price < 1000 // (fast, indexed, relational) // Then in app: var product = await _context.Products .Where(p => p.CategoryId == 5 && p.Price < 1000) .FirstAsync(); // Access extended data: var color = product .ExtendedAttributes["color"]; // ════════════════════════════════════════════ // MIGRATION EXAMPLE: EAV to JSON // ════════════════════════════════════════════ // OLD APPROACH (Entity-Attribute-Value) public class Product { public int Id { get; set; } public string Name { get; set; } } public class ProductAttribute { public int Id { get; set; } public int ProductId { get; set; } public string AttributeName { get; set; } public string AttributeValue { get; set; } } // Query: Find products with storage=256GB // SELECT DISTINCT p.* FROM Products p // JOIN ProductAttributes pa // ON p.Id = pa.ProductId // WHERE pa.AttributeName = 'storage' // AND pa.AttributeValue = '256GB' // (multiple joins, slow) // NEW APPROACH (JSON column) public class Product { public int Id { get; set; } public string Name { get; set; } public Dictionary Attributes { get; set; } } // Query: Find products with storage=256GB // SELECT * FROM Products // WHERE JSON_VALUE(Attributes, '$.storage') // = '256GB' // (single table scan, single index)
Key Insight: The Hybrid Approach Wins

In practice, the best schema combines both patterns. Use relational tables for the data you query frequently and need relationships on. Use JSON columns for optional, semi-structured, or configuration data. This gives you the performance of normalization where it matters and the flexibility of JSON where it doesn't.

Key Takeaways

  • SQL Server stores JSON as NVARCHAR(MAX) with validation; PostgreSQL uses native jsonb type with built-in indexing and optimization
  • SQL Server provides JSON_VALUE (scalar), JSON_QUERY (objects/arrays), and OPENJSON (table conversion); PostgreSQL uses ->, ->>, @>, and other operators
  • EF Core's .ToJson() mapping (EF 7+) automatically handles serialization/deserialization for owned entity types mapped to JSON columns
  • LINQ on JSON properties translates to JSON_VALUE or PostgreSQL operators; check generated SQL for performance, and use raw SQL for complex array operations
  • Dapper requires custom SqlMapper.TypeHandler to deserialize JSON; register once at startup and use in all queries
  • Use JSON for flexible attributes, configuration, audit logs, and EAV replacement; use relational tables for frequently queried, related, aggregated data
  • The hybrid approach—relational core with JSON for extended attributes—combines normalization performance with JSON flexibility
