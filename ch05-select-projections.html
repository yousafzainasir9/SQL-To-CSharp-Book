01

What Are Projections?

In relational algebra, projection is a fundamental operation that selects a subset of attributes (columns) from a relation (table). Rather than retrieving entire rows with all their columns, projection reduces the dataset to only the columns you need. This isn't just a convenience—it's a core principle of efficient database design and query optimization.

When you write a SQL SELECT statement, you're performing a projection. You specify which columns should appear in the result set, and the database engine executes exactly that operation. In LINQ, the .Select() method performs the same relational algebra operation, transforming each element in a sequence into a new shape. Both approaches reduce the volume of data transferred from the database to your application, lower memory consumption, and improve query performance.

The theoretical foundation is this: given a table with 50 columns, fetching all 50 columns for every row (SELECT *) is dramatically slower than fetching only the 3 columns you actually need. The database must:

1. Read more pages from disk (each page has limited space)
2. Transfer more data over the network (in networked databases)
3. Allocate more memory in your application
4. Process unnecessary data in subsequent operations

Projections solve this by pushing column selection down to the database level, where the filtering happens before data leaves the database.

Relational Algebra Foundation

Projection (π in formal notation) is a unary operation: it takes one relation and produces one relation with fewer attributes. Both SQL and LINQ implement projection the same way. The difference is syntax: SQL uses explicit column lists, while LINQ uses lambda expressions and anonymous types. The result is identical.

02

Basic Column Selection

The most fundamental projection is selecting a few columns from a table. In SQL, you list the columns in the SELECT clause. In LINQ, you use .Select() with a lambda that creates an anonymous type containing only those properties.

SQL 
SELECT Name, Email, Phone
FROM Customers;
Method Syntax 
var result = customers
  .Select(c => new {
    c.Name,
    c.Email,
    c.Phone
  })
  .ToList();
Query Syntax 
var result = (from c in customers
              select new {
                c.Name,
                c.Email,
                c.Phone
              })
  .ToList();

Both C# approaches create anonymous types—compiler-generated classes with the specified properties. When you use property shorthand (c.Name instead of Name = c.Name), the property name is automatically inferred from the expression. The query syntax reads more like SQL, while method syntax is more flexible and chainable. Entity Framework translates both to identical SQL.

Here's what happens under the hood:

Generated Anonymous Type (compiler creates this) 
// Compiler generates this (approximate):
internal sealed class <>f__AnonymousType0 {
  public string Name { get; }
  public string Email { get; }
  public string Phone { get; }
}
Generated SQL by EF 
SELECT
  c.Name,
  c.Email,
  c.Phone
FROM Customers AS c;

Notice the generated SQL selects only the three columns you need, not SELECT *. This is the performance win: reduced I/O, reduced network traffic, and reduced memory footprint.

Why Projections Matter

On a table with 20 columns where you only need 3, a projection reduces the result set volume by ~85%. This has compounding effects: fewer database pages read, smaller network packets, less memory allocation, faster application processing. Always project to only the columns you need.

03

Computed Columns and Aliases

Beyond selecting existing columns, projections can transform data. SQL uses the AS keyword to create column aliases and computed columns. LINQ uses property assignment in anonymous types. Both approaches compute new values at query time, meaning the database does the work, not your C# code.

SQL 
SELECT
  OrderId,
  Quantity,
  UnitPrice,
  Quantity * UnitPrice AS Total,
  Quantity * UnitPrice * 0.9 AS Discounted
FROM OrderItems;
Method Syntax 
var result = items
  .Select(i => new {
    i.OrderId,
    i.Quantity,
    i.UnitPrice,
    Total = i.Quantity * i.UnitPrice,
    Discounted = i.Quantity
      * i.UnitPrice * 0.9m
  })
  .ToList();
Query Syntax 
var result = (from i in items
              select new {
                i.OrderId,
                i.Quantity,
                i.UnitPrice,
                Total = i.Quantity * i.UnitPrice,
                Discounted = i.Quantity
                  * i.UnitPrice * 0.9m
              })
  .ToList();

In SQL, the AS keyword creates an alias that names the computed expression. In LINQ, you assign the computed value to a property. The key principle is the same: the computation happens at the database, not in your application, which is far more efficient than fetching raw values and computing in C#.

String operations work similarly:

SQL 
SELECT
  FirstName,
  LastName,
  CONCAT(FirstName, ' ', LastName)
    AS FullName,
  LEN(Email) AS EmailLength
FROM Users;
Method Syntax 
var result = users
  .Select(u => new {
    u.FirstName,
    u.LastName,
    FullName = $"{u.FirstName} "
      + u.LastName,
    EmailLength = u.Email.Length
  })
  .ToList();
Query Syntax 
var result = (from u in users
              select new {
                u.FirstName,
                u.LastName,
                FullName = $"{u.FirstName} "
                  + u.LastName,
                EmailLength = u.Email.Length
              })
  .ToList();

COALESCE and ISNULL in SQL handle NULL values. In C#, use the null-coalescing operator ?? to provide defaults:

SQL NULL Handling 
SELECT
  CustomerId,
  COALESCE(MiddleName, '') AS Middle,
  ISNULL(Phone, 'No phone') AS ContactPhone
FROM Customers;
C# NULL Handling 
var result = customers
  .Select(c => new {
    c.CustomerId,
    Middle = c.MiddleName ?? "",
    ContactPhone = c.Phone ?? "No phone"
  })
  .ToList();
Computation Location Matters

Computing in SQL means the database processes millions of rows in parallel with optimized code. Computing in C# means serializing millions of rows to your app, then processing one at a time. Always push computation to the database with .Select(). Entity Framework translates C# string concatenation, arithmetic, and null coalescing to SQL equivalents.

04

SELECT DISTINCT — Removing Duplicates

When you need unique values from a column, SQL's DISTINCT keyword eliminates duplicates from the result set. LINQ provides the .Distinct() method, which removes duplicate items based on object equality. Both are set-theoretic operations that require comparing all rows, which is computationally expensive but essential for certain queries.

SQL 
SELECT DISTINCT City
FROM Customers
ORDER BY City;
Method Syntax 
var result = customers
  .Select(c => c.City)
  .Distinct()
  .OrderBy(city => city)
  .ToList();
Query Syntax 
var result = (from c in customers
              select c.City)
  .Distinct()
  .OrderBy(city => city)
  .ToList();

DISTINCT works on multiple columns too:

SQL 
SELECT DISTINCT City, Country
FROM Customers
ORDER BY Country, City;
Method Syntax 
var result = customers
  .Select(c => new { c.City, c.Country })
  .Distinct()
  .OrderBy(x => x.Country)
  .ThenBy(x => x.City)
  .ToList();
Query Syntax 
var result = (from c in customers
              select new { c.City, c.Country })
  .Distinct()
  .OrderBy(x => x.Country)
  .ThenBy(x => x.City)
  .ToList();

When using DISTINCT on anonymous types, .Distinct() compares all properties. Two anonymous types are considered equal only if all properties match exactly.

DISTINCT Performance Impact

DISTINCT requires the database to sort or hash every result row to identify duplicates. On large result sets, this is expensive. Without an index, the database uses a hash aggregation or tempdb sort. Always apply DISTINCT at the database level (in .Select().Distinct()), not in C#. Fetching 10,000 rows then calling .Distinct() in-memory is far slower than SELECT DISTINCT with 500 rows returned.

05

CASE Expressions and Conditional Projections

SQL's CASE expression evaluates multiple conditions and returns different values accordingly. LINQ uses the ternary operator (? :) for the same purpose. Both are computed at the database level, adding a conditional branch to your projection. This is powerful for categorizing data without additional application-level processing.

SQL 
SELECT
  Name,
  Age,
  CASE
    WHEN Age >= 18 THEN 'Adult'
    WHEN Age >= 13 THEN 'Teenager'
    ELSE 'Child'
  END AS AgeCategory
FROM Users;
Method Syntax 
var result = users
  .Select(u => new {
    u.Name,
    u.Age,
    AgeCategory = u.Age >= 18
      ? "Adult"
      : u.Age >= 13
        ? "Teenager"
        : "Child"
  })
  .ToList();
Query Syntax 
var result = (from u in users
              select new {
                u.Name,
                u.Age,
                AgeCategory = u.Age >= 18
                  ? "Adult"
                  : u.Age >= 13
                    ? "Teenager"
                    : "Child"
              })
  .ToList();

For more complex categorization:

SQL CASE (Multiple Conditions) 
SELECT
  OrderId,
  OrderTotal,
  CASE
    WHEN OrderTotal > 1000 THEN 'Premium'
    WHEN OrderTotal > 500 THEN 'Standard'
    WHEN OrderTotal > 100 THEN 'Small'
    ELSE 'Minimal'
  END AS OrderCategory,
  CASE
    WHEN OrderDate >= DATEADD(day,-30,GETDATE())
      THEN 'Recent'
    ELSE 'Old'
  END AS OrderAge
FROM Orders;
C# Ternary (Multiple Conditions) 
var result = context.Orders
  .Select(o => new {
    o.OrderId,
    o.OrderTotal,
    Category = o.OrderTotal > 1000
      ? "Premium"
      : o.OrderTotal > 500
        ? "Standard"
        : o.OrderTotal > 100
          ? "Small"
          : "Minimal",
    Age = o.OrderDate >=
      DateTime.UtcNow.AddDays(-30)
      ? "Recent"
      : "Old"
  })
  .ToList();

Entity Framework translates these nested ternaries to SQL CASE statements:

Generated SQL (EF Translation) 
SELECT
  o.OrderId,
  o.OrderTotal,
  CASE
    WHEN o.OrderTotal > 1000 THEN N'Premium'
    WHEN o.OrderTotal > 500 THEN N'Standard'
    WHEN o.OrderTotal > 100 THEN N'Small'
    ELSE N'Minimal'
  END AS Category,
  CASE
    WHEN o.OrderDate >= @p0 THEN N'Recent'
    ELSE N'Old'
  END AS Age
FROM Orders AS o;
Translation Boundaries

EF translates simple ternaries and comparisons to CASE. If you call a custom C# method inside .Select(), translation fails and the query throws an exception. Keep your .Select() expressions simple: comparisons, arithmetic, string methods, null coalescing. Move complex logic after .ToList().

06

Projecting to DTOs, Records, and Anonymous Types

Projections can target different C# types: anonymous types (compiler-generated, single-use), DTOs (named classes, reusable), or records (immutable value types). The choice depends on where you need the data and how it flows through your application. All three generate the same SQL, but have different lifecycle and reusability characteristics.

Anonymous types are perfect for one-off transformations:

Anonymous Type (Local Scope) 
// Anonymous type: compiler-generated, scoped to this method
var customerSummaries = context.Customers
  .Select(c => new {
    c.Id,
    c.Name,
    OrderCount = c.Orders.Count()
  })
  .ToList();

// Type is not reusable outside this scope
// Generated type has auto-generated name like <>f__AnonymousType0

DTOs are reusable classes for data transfer:

DTO Class Definition 
// Define once, reuse everywhere
public class CustomerSummary
{
  public int Id { get; set; }
  public string Name { get; set; }
  public int OrderCount { get; set; }

  // Optional: method for business logic
  public string Status => OrderCount > 10
    ? "VIP"
    : "Regular";
}
Project to DTO 
// Reuse the CustomerSummary type
public List GetTopCustomers()
{
  return context.Customers
    .Where(c => c.Orders.Count() > 5)
    .Select(c => new CustomerSummary {
      Id = c.Id,
      Name = c.Name,
      OrderCount = c.Orders.Count()
    })
    .ToList();
}

Records are newer syntax for immutable data transfer objects:

Record Definition (Immutable) 
// Record syntax: immutable, concise
public record CustomerSummary(
  int Id,
  string Name,
  int OrderCount
);

// Can also use init-only properties:
public record CustomerSummary2
{
  public int Id { get; init; }
  public string Name { get; init; }
  public int OrderCount { get; init; }
}
Project to Record 
// Positional projection (positional record)
var result1 = context.Customers
  .Select(c => new CustomerSummary(
    c.Id,
    c.Name,
    c.Orders.Count()
  ))
  .ToList();

// Property-based projection (works with both)
var result2 = context.Customers
  .Select(c => new CustomerSummary {
    Id = c.Id,
    Name = c.Name,
    OrderCount = c.Orders.Count()
  })
  .ToList();

Comparing approaches:

Comparison: Anonymous vs DTO vs Record 
// Anonymous: Quick, local, single use
// Pros: No class definition, concise
// Cons: Not reusable, harder to test, no IntelliSense across files
var anonResult = customers.Select(c => new { c.Id, c.Name }).ToList();

// DTO: Explicit, reusable, testable
// Pros: Named type, can add methods, works everywhere
// Cons: Requires class definition, verbose
var dtoResult = customers.Select(c => new CustomerDTO {
  Id = c.Id,
  Name = c.Name
}).ToList();

// Record: Immutable, concise, reusable
// Pros: Immutability, pattern matching, equality by value
// Cons: Newer syntax, slightly less familiar to some devs
var recordResult = customers.Select(c => new CustomerRecord(c.Id, c.Name)).ToList();
Performance: All Three Are Identical

Entity Framework generates the same SQL for all three projection targets. The performance difference is negligible. Choose based on reusability: anonymous for local one-offs, DTOs for API responses and cross-layer data transfer, records for modern immutable data structures.

07

Nested Projections and Flattening

Projections can include aggregations over related data (counts, sums, nested objects). LINQ query syntax provides the let keyword for intermediate calculations, making complex projections more readable.

Simple nested aggregation:

SQL 
SELECT
  c.Id,
  c.Name,
  COUNT(o.Id) AS OrderCount,
  COALESCE(SUM(o.Total), 0)
    AS TotalSpent
FROM Customers c
LEFT JOIN Orders o
  ON c.Id = o.CustomerId
GROUP BY c.Id, c.Name;
Method Syntax 
var result = context.Customers
  .Select(c => new {
    c.Id,
    c.Name,
    OrderCount = c.Orders.Count(),
    TotalSpent = c.Orders
      .Sum(o => o.Total ?? 0)
  })
  .ToList();
Query Syntax (let) 
var result = (from c in context.Customers
              let count = c.Orders.Count()
              let total = c.Orders
                .Sum(o => o.Total ?? 0)
              select new {
                c.Id,
                c.Name,
                OrderCount = count,
                TotalSpent = total
              })
  .ToList();

The let keyword simplifies calculations by assigning intermediate values. This makes the final select clause more readable when projections are complex.

Nested object projection (be careful: this can cause N+1):

Simple Nested Projection 
// Good: Flattened projection
var result = context.Customers
  .Select(c => new {
    c.Id,
    c.Name,
    RecentOrderDates = c.Orders
      .Where(o => o.OrderDate >=
        DateTime.UtcNow.AddDays(-30))
      .Select(o => o.OrderDate)
      .ToList()
  })
  .ToList();

// This projects customer data + nested order dates
// Entity Framework can optimize this reasonably well
Avoid: Complex Nested Structures 
// Problematic: Nested collections in projection
var result = context.Customers
  .Select(c => new {
    c.Id,
    c.Name,
    Orders = c.Orders.Select(o => new {
      o.Id,
      o.Total,
      Items = o.OrderItems.Select(oi => new {
        oi.ProductId,
        oi.Quantity
      }).ToList()
    }).ToList()
  })
  .ToList();

// This can cause multiple round-trips or
// generate massive SQL with cartesian products
Projection Depth Trade-off

Shallow projections (select specific columns) are always efficient. Nested projections with .Select() on navigation properties require care: too many levels of nesting can generate N+1 queries or massive cartesian joins. When projecting nested collections, keep it to one level deep and flatten the structure when possible.

08

Understanding Entity Framework's Generated SQL

Entity Framework Core translates your LINQ expressions into SQL at query execution time. Understanding what it generates is crucial for debugging performance and ensuring your queries are efficient. The .ToQueryString() method (available in EF Core 5.0+) reveals the generated SQL before execution.

Simple projection:

LINQ Query 
var query = context.Products
  .Where(p => p.Category == "Electronics")
  .Select(p => new {
    p.Id,
    p.Name,
    p.Price
  });

string sql = query.ToQueryString();
Console.WriteLine(sql);
Generated SQL 
SELECT
  p.Id,
  p.Name,
  p.Price
FROM Products AS p
WHERE p.Category = N'Electronics';

Projection with computed columns:

LINQ with Computation 
var query = context.OrderItems
  .Select(oi => new {
    oi.OrderId,
    oi.ProductId,
    Subtotal = oi.Quantity * oi.UnitPrice,
    WithTax = (oi.Quantity * oi.UnitPrice)
      * 1.1m
  });

string sql = query.ToQueryString();
Generated SQL 
SELECT
  oi.OrderId,
  oi.ProductId,
  (oi.Quantity * oi.UnitPrice) AS Subtotal,
  ((oi.Quantity * oi.UnitPrice) * 1.1)
    AS WithTax
FROM OrderItems AS oi;

Projection with aggregation:

LINQ with Aggregation 
var query = context.Customers
  .Select(c => new {
    c.Id,
    c.Name,
    RecentOrders = c.Orders
      .Where(o => o.OrderDate >=
        DateTime.UtcNow.AddDays(-90))
      .Count(),
    TotalRevenue = c.Orders
      .Sum(o => o.Total ?? 0)
  });

string sql = query.ToQueryString();
Generated SQL 
SELECT
  c.Id,
  c.Name,
  COUNT(CASE
    WHEN o.OrderDate >=
      DATEADD(day, -90, GETDATE())
    THEN 1
    END) AS RecentOrders,
  COALESCE(SUM(o.Total), 0) AS TotalRevenue
FROM Customers AS c
LEFT JOIN Orders AS o
  ON c.Id = o.CustomerId
GROUP BY c.Id, c.Name;

Projection with conditional logic:

LINQ with Ternary 
var query = context.Products
  .Select(p => new {
    p.Id,
    p.Name,
    p.Price,
    PriceCategory = p.Price > 500
      ? "Premium"
      : p.Price > 100
        ? "Mid-Range"
        : "Budget"
  });

string sql = query.ToQueryString();
Generated SQL (CASE) 
SELECT
  p.Id,
  p.Name,
  p.Price,
  CASE
    WHEN p.Price > 500 THEN N'Premium'
    WHEN p.Price > 100 THEN N'Mid-Range'
    ELSE N'Budget'
  END AS PriceCategory
FROM Products AS p;

Detecting translation failures:

Monitoring EF Translation 
// Enable query logging to see SQL automatically
var options = new DbContextOptionsBuilder()
  .UseSqlServer(connectionString)
  .LogTo(
    message => Console.WriteLine(message),
    LogLevel.Information
  )
  .Build();

// Or use a logger factory
var loggerFactory = LoggerFactory.Create(
  builder => builder.AddConsole()
);

var options2 = new DbContextOptionsBuilder()
  .UseSqlServer(connectionString)
  .UseLoggerFactory(loggerFactory)
  .Build();

// Test your queries in development to spot translation issues early
using var context = new AppDbContext(options);
var problematicQuery = context.Products
  .Select(p => new {
    p.Id,
    Status = GetPriceStatus(p.Price) // WILL FAIL!
  });

// This throws: DbContext query compilation failed
// because GetPriceStatus can't be translated to SQL
Always Verify Generated SQL

Never assume your .Select() translates efficiently. Always use .ToQueryString() during development to inspect generated SQL. Look for: unnecessary SELECT *, missing WHERE clauses, unexpected JOINs, and GROUP BY clauses you didn't intend. These are signs your projection is not doing what you think.

Key Takeaways

  • Projections are a relational algebra operation: selecting a subset of attributes from a relation, reducing data volume at the database level.
  • Use .Select() to project only the columns you need. Fetching fewer columns dramatically reduces I/O, network traffic, and memory consumption.
  • Aliases (SQL AS) map to property assignment in LINQ anonymous types. Both rename and compute columns at query time.
  • DISTINCT and .Distinct() remove duplicates, but are expensive operations. Apply at the database level, not in C# after fetching.
  • CASE expressions (SQL) translate to ternary operators (C#). Entity Framework converts nested ternaries to SQL CASE statements.
  • Choose projection targets based on scope: anonymous types for local one-offs, DTOs for data transfer, records for immutable structures. All generate identical SQL.
  • Nested projections with aggregation (counts, sums) are powerful but can cause N+1 queries if not carefully structured.
  • Always verify generated SQL using .ToQueryString() or query logging. Never assume translation is efficient—verify it.
