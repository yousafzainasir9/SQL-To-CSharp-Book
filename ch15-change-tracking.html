Chapter 15

Change Tracking

Requirement: "Save only the fields that changed." In SQL, you write an explicit UPDATE with specific columns. In EF Core, change tracking automatically detects which properties changed and generates minimal UPDATE statements.

01

Requirement: "How Does EF Know What Changed?"

Business Requirement
Developer modifies customer.Name and calls SaveChanges. How does EF know to UPDATE only Name? How does it track changes?
SQL Developer Thinks I explicitly write UPDATE SET Name = @Name WHERE Id = @Id. I control exactly what gets updated. The database has no memory of the old value.
C# Developer Thinks EF takes a snapshot when loading an entity. On SaveChanges, it diffs current vs original. If only Name changed, it generates UPDATE SET Name = ... Nothing else touched.
SQL — Manual UPDATE
-- You write exactly what changed
UPDATE Customers
SET Name = @Name,
    ModifiedDate = GETDATE()
WHERE Id = @CustomerId;
C# — EF Detects Changes
var customer = await db.Customers
    .FirstOrDefaultAsync(c => c.Id == id, ct);

// Change only the name
customer.Name = "Jane Doe";

await db.SaveChangesAsync(ct);
// EF generates: UPDATE SET Name='Jane Doe'
Entity States Diagram
// The 5 states of an entity in EF:

Added       → New entity, not in DB yet
             SaveChanges generates INSERT

Unchanged   → Loaded from DB, no changes
             SaveChanges skips it

Modified    → Loaded from DB, properties changed
             SaveChanges generates UPDATE with changed columns only

Deleted     → Marked for deletion
             SaveChanges generates DELETE

Detached    → Not tracked (loaded with AsNoTracking
             or not attached to this DbContext)
C# — How EF Knows
var customer = await db.Customers.FindAsync(id, ct);
// EF snapshots: { Id=1, Name="John", Email="john@..." }

customer.Name = "Jane";
// Current state: { Id=1, Name="Jane", Email="john@..." }
// On SaveChanges, EF compares:
//   Name: "John" → "Jane" [CHANGED]
//   Email: "john@..." → "john@..." [NO CHANGE]
// Result: UPDATE SET Name='Jane' WHERE Id=1
02

Requirement: "See What Will Be Saved Before Saving"

Business Requirement
Audit log needs to capture all changes before they’re committed. What values changed? From what to what?
SQL Developer Thinks Use triggers (AFTER INSERT, AFTER UPDATE, AFTER DELETE) or temporal tables to track changes. The database records history.
C# Developer Thinks Loop through ChangeTracker.Entries() to inspect each Modified entity. Access original and current values. Log them, then SaveChanges.
SQL — Trigger Audit
CREATE TRIGGER tr_Customer_Audit
    ON Customers AFTER UPDATE
AS
BEGIN
    INSERT INTO AuditLog
        (EntityType, RecordId, OldValue, NewValue)
    SELECT 'Customer', INSERTED.Id,
           CONCAT(DELETED.Name, '→', INSERTED.Name)
    FROM INSERTED
    JOIN DELETED ON INSERTED.Id = DELETED.Id;
END;
C# — Inspect Changes
var customer = await db.Customers
    .FirstOrDefaultAsync(c => c.Id == 1, ct);
customer.Name = "Jane";

// BEFORE SaveChanges:
var entry = db.ChangeTracker
    .Entries<Customer>()
    .FirstOrDefault();

var original = entry.OriginalValues
    .GetValue<string>("Name"); // "John"
var current = entry.CurrentValues
    .GetValue<string>("Name");  // "Jane"

await LogAudit("Customer", 1, original, current, ct);
await db.SaveChangesAsync(ct);
C# — Full Audit Pattern
private async Task LogChangesAsync(CancellationToken ct)
{
    var entries = db.ChangeTracker
        .Entries()
        .Where(e => e.State == EntityState.Modified);

    foreach (var entry in entries)
    {
        var entityType = entry.Entity.GetType().Name;
        var recordId = entry.Property("Id").CurrentValue;

        var props = entry.Properties
            .Where(p => p.IsModified);

        foreach (var prop in props)
        {
            var oldValue = prop.OriginalValue;
            var newValue = prop.CurrentValue;

            await db.AuditLogs.AddAsync(new AuditLog
            {
                EntityType = entityType,
                RecordId = recordId?.ToString(),
                PropertyName = prop.Metadata.Name,
                OldValue = oldValue?.ToString(),
                NewValue = newValue?.ToString(),
                ChangedAt = DateTime.UtcNow
            }, ct);
        }
    }
}
03

Requirement: "Update a Record from an API Request"

Business Requirement
API receives a DTO with updated customer data. The entity isn’t currently tracked. Attach it, mark it as modified, save it.
SQL Developer Thinks Just run UPDATE with the values from the request. No need to load first. UPDATE Customer SET Name=@Name WHERE Id=@Id.
C# Developer Thinks Attach the untracked entity to tell EF it exists in the database. Mark it as Modified so EF knows to UPDATE it. SaveChanges generates the UPDATE.
SQL — Direct UPDATE
-- No need to SELECT first
UPDATE Customers
SET Name = @Name,
    Email = @Email,
    ModifiedDate = GETDATE()
WHERE Id = @CustomerId;
C# — Attach & Mark Modified
var dto = new UpdateCustomerDto
{
    Id = 1,
    Name = "Jane",
    Email = "jane@..."
};

var customer = new Customer
{
    Id = dto.Id,
    Name = dto.Name,
    Email = dto.Email
};

// Attach tells EF this entity exists
db.Customers.Attach(customer);
// Mark modified so EF knows to UPDATE
db.Entry(customer).State =
    EntityState.Modified;

await db.SaveChangesAsync(ct);
C# — Partial Updates (Only Changed Fields)
// Only update fields that were actually sent
var customer = await db.Customers
    .FirstOrDefaultAsync(c => c.Id == dto.Id, ct);

if (customer != null)
{
    // Mark entry as modified
    var entry = db.Entry(customer);

    // Update only the fields provided
    if (!string.IsNullOrEmpty(dto.Name))
        customer.Name = dto.Name;

    if (!string.IsNullOrEmpty(dto.Email))
        customer.Email = dto.Email;

    // Only modified properties will be in the UPDATE
    await db.SaveChangesAsync(ct);
}
04

Requirement: "Don't Track Read-Only Queries"

Business Requirement
Dashboard loads 1000 customers for display only. No edits needed. Tracking them costs memory and CPU.
SQL Developer Thinks SELECT is read-only by nature. The database doesn’t keep anything in memory. You get the data, done.
C# Developer Thinks AsNoTracking() skips snapshot creation. For read-only queries, this is faster and uses less memory. No change detection needed.
With Tracking
// EF creates snapshots for 1000 customers
var customers = await db.Customers
    .ToListAsync(ct); // Tracked

// Memory usage: HIGH
// Change detection: ACTIVE
// Use when: You'll edit them
Without Tracking
// No snapshots created
var customers = await db.Customers
    .AsNoTracking()
    .ToListAsync(ct); // Not tracked

// Memory usage: LOW
// Change detection: SKIPPED
// Use when: Display only
Performance Comparison
// TRACKING: Snapshot created for each entity
var trackedCustomers = await db.Customers
    .Where(c => c.IsActive)
    .ToListAsync(ct);
// ~100ms for 1000 records (includes snapshots)

// NO TRACKING: Raw data only
var untracked = await db.Customers
    .AsNoTracking()
    .Where(c => c.IsActive)
    .ToListAsync(ct);
// ~30ms for 1000 records (no snapshots)

// For read-only queries: Always use AsNoTracking()
// For queries you'll edit: Use with tracking
05

Requirement: "Delete Customer and All Their Orders Automatically"

Business Requirement
When a customer is deleted, their orders should be deleted too automatically. How do we enforce this?
SQL Developer Thinks ON DELETE CASCADE in the FOREIGN KEY. The database enforces it automatically. Delete the parent, children are gone.
C# Developer Thinks EF has cascade delete behavior. Remove the customer, SaveChanges propagates the delete to related orders. Configure in modelBuilder.
SQL — Cascade Delete
CREATE TABLE Orders (
    Id         INT IDENTITY(1,1) PRIMARY KEY,
    CustomerId INT NOT NULL,
    Total      DECIMAL(10,2) NOT NULL,
    FOREIGN KEY (CustomerId)
        REFERENCES Customers(Id)
        ON DELETE CASCADE
);

-- Delete customer → Orders auto-deleted
DELETE FROM Customers WHERE Id = 5;
C# — EF Cascade
var customer = await db.Customers
    .Include(c => c.Orders)
    .FirstOrDefaultAsync(c => c.Id == 5, ct);

// Remove customer
db.Customers.Remove(customer);

// SaveChanges ALSO removes all customer.Orders
await db.SaveChangesAsync(ct);
// Generates: DELETE FROM Orders WHERE CustomerId=5
//            DELETE FROM Customers WHERE Id=5
C# — Configure Cascade Behavior
// In OnModelCreating:
modelBuilder.Entity<Order>(e =>
{
    e.HasOne(o => o.Customer)
     .WithMany(c => c.Orders)
     .HasForeignKey(o => o.CustomerId)
     .OnDelete(DeleteBehavior.Cascade);
    // Options:
    // Cascade: Delete orders when customer deleted
    // SetNull: Set CustomerId=NULL (requires nullable FK)
    // Restrict: Throw if trying to delete customer with orders
    // NoAction: No automatic action (client-side delete)
});

// Or in the navigation property:
public class Order
{
    public int Id { get; set; }
    public int CustomerId { get; set; }

    [DeleteBehavior(DeleteBehavior.Cascade)]
    public Customer Customer { get; set; } = null!;
}
06

Change Tracking Quick Reference

State What Happens on SaveChanges How to Set
Added INSERT into database db.Customers.Add(customer)
Unchanged No action (skipped) Load from DB, no changes made
Modified UPDATE changed columns only Load entity, modify property, SaveChanges auto-detects
Deleted DELETE from database db.Customers.Remove(customer)
Detached No action (not tracked) .AsNoTracking() or new instance
C# — Detecting and Examining Changes
// Get all modified entries
var modified = db.ChangeTracker
    .Entries()
    .Where(e => e.State == EntityState.Modified);

foreach (var entry in modified)
{
    var entityName = entry.Entity.GetType().Name;
    Console.WriteLine($"Modified: {entityName}");

    // List changed properties
    foreach (var prop in entry.Properties
        .Where(p => p.IsModified))
    {
        var oldVal = prop.OriginalValue;
        var newVal = prop.CurrentValue;
        Console.WriteLine(
            $"  {prop.Metadata.Name}: " +
            $"{oldVal} → {newVal}");
    }
}

// Accept all changes (clears snapshots)
db.ChangeTracker.AcceptAllChanges();

// Clear all tracked entities
db.ChangeTracker.Clear();

Chapter 15

Change Tracking

EF watches entities for changes. Understand states (Added, Modified, Deleted, Unchanged) and ChangeTracker to audit what changed.

01

Entity States: The Lifecycle

Every tracked entity has a state: Detached, Added, Unchanged, Modified, or Deleted. SaveChanges() generates SQL based on state.

C#: Entity States 
// Detached (not tracked)
var customer = new Customer { Name = "John" };

// Added (will INSERT)
context.Customers.Add(customer);
// State: EntityState.Added

// Unchanged (just loaded)
var loaded = await context.Customers.FirstOrDefaultAsync();
// State: EntityState.Unchanged

// Modified (changed property)
loaded.Name = "Jane";
// State: EntityState.Modified

// Deleted (marked for deletion)
context.Customers.Remove(loaded);
// State: EntityState.Deleted

// SaveChangesAsync() generates INSERT, UPDATE,
// DELETE based on these states
States drive SQL generation. Detached = ignored. Added/Modified/Deleted = included in SaveChanges().
02

Detecting Changes: Snapshots

EF stores a snapshot of each entity when loaded. On SaveChanges(), it compares current values to snapshot to detect changes.

SQL Comparison 
-- Load snapshot
SELECT Id, Name, Email FROM Customers
WHERE Id = 1;
-- Snapshot: {1, "John", "john@ex"}

-- Later:
UPDATE Customers
SET Name = "Jane"
WHERE Id = 1;
-- Comparison: "John" != "Jane"
C#: Snapshot Check 
var customer = await db.Customers
  .FirstOrDefaultAsync(c => c.Id == 1);
// EF snapshot: {Id:1, Name:"John",
//               Email:"john@ex"}

customer.Name = "Jane";
// Comparison: snapshot["John"] !=
//             current["Jane"]
// State: EntityState.Modified

await context.SaveChangesAsync();
// Generates UPDATE
EF compares property values to snapshot. If any property differs, state becomes Modified.
03

ChangeTracker: Inspect All Tracked Entities

Access DbContext.ChangeTracker.Entries() to inspect all tracked entities, their states, and what changed.

C#: ChangeTracker Inspection 
var customer = await db.Customers.FirstOrDefaultAsync();
customer.Name = "Updated";

// Inspect all tracked entities
foreach (var entry in db.ChangeTracker.Entries())
{
  Console.WriteLine(
    $"Entity: {entry.Entity.GetType().Name}");
  Console.WriteLine(
    $"State: {entry.State}");

  // See property changes
  foreach (var prop in entry.Properties)
  {
    var original = prop.OriginalValue;
    var current = prop.CurrentValue;
    if (original is not null && !Equals(original, current))
    {
      Console.WriteLine(
        $"  {prop.Name}: " +
        $"{original} → {current}");
    }
  }
}

// Output:
// Entity: Customer
// State: Modified
//   Name: John → Updated
ChangeTracker.Entries() shows all tracked entities. Use to audit, validate, or implement custom logic before SaveChanges().
04

Original vs Current Values

Each property tracks both original (loaded) and current (changed) value. Use for validation or audit trails.

C#: Audit Trail Example 
public override int SaveChanges()
{
  var entries = ChangeTracker
    .Entries()
    .Where(e => e.State ==
      EntityState.Modified);

  foreach (var entry in entries)
  {
    var auditLog = new AuditLog
    {
      TableName = entry.Entity
        .GetType().Name,
      EntityId = (int)entry
        .Property("Id").CurrentValue,
      ChangedAt = DateTime.UtcNow,
      OldValues = string.Join("; ",
        entry.Properties
          .Where(p => !Equals(
            p.OriginalValue,
            p.CurrentValue))
          .Select(p =>
            $"{p.Name}: " +
            $"{p.OriginalValue}")),
      NewValues = string.Join("; ",
        entry.Properties
          .Where(p => !Equals(
            p.OriginalValue,
            p.CurrentValue))
          .Select(p =>
            $"{p.Name}: " +
            $"{p.CurrentValue}"))
    };
    AuditLogs.Add(auditLog);
  }

  return base.SaveChanges();
}
OriginalValue and CurrentValue track old/new. Perfect for audit logs, change notifications, or undo.
05

AcceptAllChanges: Commit Tracking

SaveChanges() automatically calls AcceptAllChanges(), updating snapshots. All entities become Unchanged.

Before SaveChanges 
customer.Name = "Jane";
// State: EntityState.Modified
// ChangeTracker sees difference

foreach (var entry in
  db.ChangeTracker.Entries())
  Console.WriteLine(entry.State);
  // Output: Modified
After SaveChanges 
await context.SaveChangesAsync();
// SQL UPDATE executes
// ChangeTracker.AcceptAllChanges()
//   updates snapshots
// All states reset to Unchanged

foreach (var entry in
  db.ChangeTracker.Entries())
  Console.WriteLine(entry.State);
  // Output: Unchanged
SaveChanges updates snapshots. Modified entities become Unchanged. Tracking resets after commit.
06

Cascade Delete: Automatic Deletion

Configure relationships with OnDelete(DeleteBehavior.Cascade). Removing parent auto-marks children for deletion.

C#: Cascade Delete Setup 
// OnModelCreating
e.HasMany(c => c.Orders)
  .WithOne(o => o.Customer)
  .HasForeignKey(o => o.CustomerId)
  .OnDelete(DeleteBehavior.Cascade);

// Later:
var customer = await db.Customers
  .Include(c => c.Orders)
  .FirstOrDefaultAsync(c => c.Id == 1);

db.Customers.Remove(customer);
// EF marks all customer.Orders
// as Deleted too

await context.SaveChangesAsync();
// SQL: DELETE FROM Orders WHERE CustomerId = 1
// SQL: DELETE FROM Customers WHERE Id = 1
Cascade deletes propagate state changes. Remove parent, children auto-deleted. Set via OnDelete() in Fluent API.
07

Attach and Reattach: Bring Detached Into Scope

Entities created outside DbContext are Detached. Use Attach() or Update() to reattach and track them.

Detached Entity 
var customer = new Customer
{ Id = 1, Name = "John" };
// Detached: not tracked

db.Customers.Add(customer);
// Now tracked, marked Added
Reattach Existing 
var customer = new Customer
{ Id = 1, Name = "Jane" };
// Detached: not tracked

db.Customers.Update(customer);
// Now tracked, marked Modified

await context.SaveChangesAsync();
// SQL: UPDATE Customers WHERE Id = 1
Attach() marks Unchanged. Update() marks Modified. Use to reattach entities from API requests or serialization.

Key Takeaways

  • Entity states: Detached, Added, Unchanged, Modified, Deleted.
  • EF snapshots entities on load. Compares current to snapshot on SaveChanges().
  • ChangeTracker.Entries() inspects all tracked entities and their states.
  • OriginalValue vs CurrentValue for audit logs and change detection.
  • SaveChanges() generates SQL based on states, then resets to Unchanged.
  • Cascade deletes propagate state changes (parent deleted = children deleted).
  • Update() or Attach() to retrack detached entities from API requests.
