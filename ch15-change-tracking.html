01 — THE FUNDAMENTAL DIFFERENCE

SQL Transactions vs EF's In-Memory Snapshots

In pure SQL, you are stateless. You load a row, you send an explicit UPDATE statement that modifies specific columns, you commit the transaction. The database itself tracks what changed through the transaction log. EF works fundamentally differently: it's stateful. When you load a Customer into DbContext, EF takes a snapshot of every property value in memory. When you modify customer.Name, EF marks that property as changed. On SaveChanges(), EF compares current values to the snapshot and generates UPDATE statements for only the changed columns. This snapshot mechanism is the heart of EF's magic: automatic change detection without writing explicit UPDATE statements.

The key insight is that EF is an object-oriented ORM wrapping a stateless SQL database. To bridge that gap, EF maintains in-memory state (snapshots) to detect what changed. This is different from how you think in SQL, where every change is explicit. In EF, change detection is implicit and automatic.

SQL: Explicit Changes 
-- You must explicitly list columns
UPDATE Customers
SET Name = 'Alice Smith',
    Email = 'alice@example.com'
WHERE Id = 1;

-- The database writes to transaction log
-- No history of what was before
-- You control exactly what changes
EF: Implicit Change Detection 
// Load and EF snapshots original values
var customer = context.Customers.Find(1);
// Snapshot: Name="John", Email="john@example.com"

// Modify object
customer.Name = "Alice Smith";

// SaveChanges: EF compares snapshot to current
// Detects Name changed, Email unchanged
// Generates: UPDATE Customers SET Name='Alice'
// WHERE Id=1; (Email not included!)
EF is Stateful, SQL is Stateless

EF maintains in-memory snapshots of entities to detect changes. SQL has no such state—every change must be explicit in the statement. This is why EF can generate lean UPDATE statements with only changed columns: it has the original snapshot to compare against. SQL requires you to specify each column manually.

This fundamental difference explains why EF can be so efficient: by tracking state, it avoids sending unchanged data back to the database. It also explains why detached entities (entities not loaded by this DbContext) require special handling—EF has no snapshot for them, so it can't detect what changed.

02 — ENTITY STATES

Added, Modified, Deleted, Unchanged, Detached

Every entity tracked by DbContext has an EntityState that describes its relationship to the database. Added means the entity is new and will be inserted on SaveChanges(). Modified means an entity was loaded, changed, and will be updated. Deleted means it was marked for deletion. Unchanged means it was loaded and hasn't changed (still tracked). Detached means it's not being tracked at all. Understanding state transitions is critical: you can inspect the state via context.Entry(entity).State and the EntityState enum, and you can manually change states to control what SQL SaveChanges() generates.

Entity State Lifecycle and Transitions 
// STATE 1: Create new entity (Detached)
var newCustomer = new Customer { Name = "Bob" };
Console.WriteLine(context.Entry(newCustomer).State);
// Output: EntityState.Detached (not tracked yet)

// STATE 2: Add to context (Added)
context.Customers.Add(newCustomer);
Console.WriteLine(context.Entry(newCustomer).State);
// Output: EntityState.Added (pending INSERT)

// STATE 3: SaveChanges inserts and becomes Unchanged
context.SaveChanges();
Console.WriteLine(context.Entry(newCustomer).State);
// Output: EntityState.Unchanged (persisted, no changes)

// STATE 4: Load existing entity (Unchanged)
var existingCustomer = context.Customers.Find(5);
Console.WriteLine(context.Entry(existingCustomer).State);
// Output: EntityState.Unchanged (loaded, not modified)

// STATE 5: Modify entity (Modified)
existingCustomer.Name = "Bobby";
Console.WriteLine(context.Entry(existingCustomer).State);
// Output: EntityState.Modified (changes pending)

// STATE 6: SaveChanges updates and becomes Unchanged
context.SaveChanges();
Console.WriteLine(context.Entry(existingCustomer).State);
// Output: EntityState.Unchanged (changes persisted)

// STATE 7: Mark for deletion (Deleted)
context.Customers.Remove(existingCustomer);
Console.WriteLine(context.Entry(existingCustomer).State);
// Output: EntityState.Deleted (pending DELETE)

// STATE 8: SaveChanges deletes and becomes Detached
context.SaveChanges();
Console.WriteLine(context.Entry(existingCustomer).State);
// Output: EntityState.Detached (no longer tracked)

// Check state of modified entity without changes
var unchanged = context.Customers.AsNoTracking().Find(3);
var state = EntityState.Unchanged; // assumed state

State transitions are deterministic: Added entities that are saved become Unchanged. Modified entities become Unchanged after SaveChanges(). Deleted entities become Detached. Understanding this lifecycle helps you predict what SQL will be generated and debug persistence issues. You can also manually set states via context.Entry(entity).State = EntityState.Modified, though this is advanced and should be done carefully.

03 — SAVECHANGES GENERATES SQL FROM STATES

How Entity States Map to INSERT, UPDATE, DELETE

SaveChanges() iterates through the change tracker, examines each entity's state, and generates SQL accordingly. Added entities generate INSERT statements. Modified entities generate UPDATE statements (with only changed columns due to snapshot comparison). Deleted entities generate DELETE statements. Unchanged entities generate no SQL. This is why understanding entity states is crucial: you can control what SQL gets generated by understanding and managing states. Added and Modified states are the only ones that generate DML statements; Unchanged and Detached generate nothing.

LINQ Method Syntax: Track State Changes 
var customers = context.Customers
  .Where(c => c.IsActive)
  .ToList();

customers[0].Name = "Modified";
customers.Add(new Customer { Name = "New" });
context.Customers.Remove(customers[1]);

// Before SaveChanges():
// Modified: EntityState.Modified
// New entity: EntityState.Added
// Removed: EntityState.Deleted
// Rest: EntityState.Unchanged

context.SaveChanges();
// Generates: 1 UPDATE, 1 INSERT, 1 DELETE, rest no-op
LINQ Query Syntax: State Inspection 
from c in context.Customers
where c.IsActive
select c // materialize into memory
into customers
select (Customer: customers,
  State: context.Entry(customers).State)
// Can inspect each entity's state

// Or direct access to ChangeTracker:
var allChanges = context.ChangeTracker
  .Entries()
  .Where(e => e.State != EntityState.Unchanged)
  .ToList();
// Get all Added, Modified, Deleted, Detached entities
SaveChanges() is Driven by Entity States

Added generates INSERT, Modified generates UPDATE, Deleted generates DELETE, Unchanged and Detached generate nothing. This is why manually managing states (or allowing EF to detect them automatically) is so important. Incorrect states lead to missing or unwanted SQL statements.

04 — THE CHANGETRACKER API

Inspecting and Managing Change Detection

The ChangeTracker is a powerful API for inspecting tracked entities and their states. You can access it via context.ChangeTracker.Entries() to get all tracked entities, iterate their states, inspect original and current values, and manually mark properties as modified or reset them. This is essential for advanced scenarios like auditing changes, implementing change logs, or handling complex conflict resolution in concurrent systems. The API exposes both property-level and entity-level change information.

ChangeTracker API: Deep Inspection 
// Get all tracked entries
var entries = context.ChangeTracker.Entries();

foreach (var entry in entries)
{
  Console.WriteLine($"Entity: {entry.Entity.GetType().Name}");
  Console.WriteLine($"State: {entry.State}");

  // Inspect original vs current values
  if (entry.State == EntityState.Modified)
  {
    Console.WriteLine("Changed properties:");
    foreach (var property in entry.Properties)
    {
      var original = property.OriginalValue;
      var current = property.CurrentValue;

      if (!Equals(original, current))
      {
        Console.WriteLine(
          $"  {property.Metadata.Name}: " +
          $"{original} → {current}");
      }
    }
  }
}

// Access specific entity's entry
var customer = context.Customers.Find(1);
var entry = context.Entry(customer);

Console.WriteLine("All property states:");
foreach (var prop in entry.Properties)
{
  Console.WriteLine(
    $"{prop.Metadata.Name} = " +
    $"{prop.CurrentValue} " +
    $"(IsModified: {prop.IsModified})");
}

// Manually mark property as modified
entry.Property(c => c.Email).IsModified = true;

// Force re-detection of changes
context.ChangeTracker.DetectChanges();

// Clear modifications without discarding
entry.RejectChanges(); // Revert to original snapshot

// Get only entities in certain states
var modified = context.ChangeTracker
  .Entries()
  .Where(e => e.State == EntityState.Modified)
  .ToList();

var added = context.ChangeTracker
  .Entries()
  .Where(e => e.State == EntityState.Added)
  .ToList();

The ChangeTracker API is foundational for building audit systems, implementing change logs, detecting conflicts, and managing complex persistence scenarios. Many enterprise applications use ChangeTracker to implement automatic audit trails by inspecting what changed and logging it to an audit table before SaveChanges().

05 — DETECTCHANGES AND SNAPSHOT COMPARISON

Automatic vs Manual Change Detection

EF has two mechanisms for detecting that an entity has changed: implicit snapshot comparison on SaveChanges() and explicit DetectChanges() calls. By default, EF calls DetectChanges() automatically at the start of SaveChanges(), and also after Add(), Attach(), Update(), and Remove() operations. This automatic detection is transparent but has a cost: it iterates all tracked entities comparing current to snapshot values. In some scenarios, you might want to disable automatic detection and call DetectChanges() manually, trading transparency for performance control. You can also implement INotifyPropertyChanged on your entities for notification-based change tracking, which eliminates snapshot comparison overhead by detecting changes as they happen.

Automatic DetectChanges (Default) 
var customers = context.Customers
  .Where(c => c.IsActive)
  .ToList(); // Loaded and tracked

// Modify many properties
foreach (var c in customers)
{
  c.Name = c.Name + "_Updated";
  c.Email = "updated@example.com";
  c.Phone = "555-1234";
}

// SaveChanges calls DetectChanges automatically
context.SaveChanges();
// Automatic detection scans all tracked entities
// Compares snapshot to current values
// Generates UPDATE for each modified customer
Manual DetectChanges for Control 
// Disable automatic detection
context.ChangeTracker
  .AutoDetectChangesEnabled = false;

var customers = context.Customers.ToList();

foreach (var c in customers)
{
  c.Name = c.Name + "_Updated";
}

// Must call DetectChanges explicitly
context.ChangeTracker.DetectChanges();

// Now inspect what changed
var modified = context.ChangeTracker
  .Entries()
  .Where(e => e.State == EntityState.Modified)
  .ToList();

// Re-enable for SaveChanges
context.ChangeTracker
  .AutoDetectChangesEnabled = true;
context.SaveChanges();
DetectChanges Scans All Tracked Entities

Automatic DetectChanges() is convenient but has cost: it compares every tracked entity's snapshot to current values. Disabling it and calling manually is an optimization for scenarios with many tracked entities where you know exactly when changes might occur. Notification-based tracking via INotifyPropertyChanged eliminates this overhead entirely.

For most applications, automatic DetectChanges is fine. The cost is typically negligible unless you're tracking thousands of entities. But understanding that it exists and can be controlled is important for high-performance scenarios. Some data-heavy applications disable automatic detection and call it strategically to avoid redundant scans.

06 — ASNOTRACKING OPTIMIZES READ ONLY QUERIES

Skip Snapshots When You Don't Modify Data

By default, every query result is tracked: EF creates snapshots and monitors them for changes. For read-only queries like API responses, reports, or exports, tracking is wasted overhead. Use AsNoTracking() to skip all snapshot creation and change monitoring. The SQL generated is identical, but EF skips the in-memory snapshot step. This is a dramatic performance win: benchmarks show 20-50% faster materialization and 50-70% less memory usage, especially for large result sets. Use AsNoTracking() as your default for queries that don't call SaveChanges().

SQL: No Difference 
-- Both queries generate identical SQL
SELECT Id, Name, Email, IsActive
FROM Customers
WHERE IsActive = 1;

-- SQL doesn't know or care about
-- snapshots or change tracking.
-- That's EF's in-memory concern.
EF: Tracked vs AsNoTracking 
// TRACKED: Creates snapshots, slower
var tracked = context.Customers
  .Where(c => c.IsActive)
  .ToList();
// 10,000 rows: 45ms, 50MB memory

// ASNOTRACKING: No snapshots, faster
var untracked = context.Customers
  .Where(c => c.IsActive)
  .AsNoTracking()
  .ToList();
// 10,000 rows: 28ms, 25MB memory
// 40% faster, 50% less memory!
AsNoTracking Best Practices 
// Make AsNoTracking your default for reads

// API GET endpoint - no modifications
public async Task> GetCustomers()
{
  return await context.Customers
    .AsNoTracking()
    .Where(c => c.IsActive)
    .Select(c => new CustomerDto
    {
      Id = c.Id,
      Name = c.Name,
      Email = c.Email
    })
    .ToListAsync();
}

// Report generation - read only
public async Task GenerateReport()
{
  var data = await context.Customers
    .AsNoTracking()
    .Where(c => c.IsActive)
    .GroupBy(c => c.Region)
    .Select(g => new
    {
      Region = g.Key,
      Count = g.Count(),
      TotalRevenue = g.Sum(c => c.TotalRevenue)
    })
    .ToListAsync();
  return new ReportData { Data = data };
}

// Search results - read only
public async Task> Search(string query)
{
  return await context.Products
    .AsNoTracking()
    .Where(p => p.Name.Contains(query))
    .OrderByDescending(p => p.Rating)
    .Take(20)
    .ToListAsync();
}

// Only track when you modify
public async Task UpdateCustomer(int id, CustomerDto dto)
{
  // No AsNoTracking - need to track for SaveChanges
  var customer = await context.Customers
    .FirstOrDefaultAsync(c => c.Id == id);

  customer.Name = dto.Name;
  customer.Email = dto.Email;

  await context.SaveChangesAsync();
}
AsNoTracking is Free Performance

The SQL is identical with or without AsNoTracking(). The only difference is EF's in-memory overhead. For any query that doesn't modify data, AsNoTracking() is a pure win. It costs nothing and saves significant memory and CPU on large result sets.

07 — ASNOTRACKINGWITHIDENTITYRESOLUTION

Efficient Complex Queries Without Duplicates

When queries include relationships and the same entity appears multiple times in results, AsNoTracking() can create duplicate object instances in memory. AsNoTrackingWithIdentityResolution prevents this by tracking entity identities (primary keys) to ensure the same entity instance is reused across result rows, all while skipping snapshot creation. This combines the memory efficiency of AsNoTracking() with the logical correctness of tracked queries. Perfect for complex loads and projections where you want lean memory footprint without duplicate objects corrupting your object graph.

AsNoTracking: May Create Duplicates 
// Load Customer with Orders
var customers = context.Customers
  .Include(c => c.Orders)
  .AsNoTracking()
  .ToList();

// If Customer 1 has 5 Orders:
// The JOIN creates 5 result rows
// AsNoTracking creates 5 separate
// Customer 1 instances in memory
// They're separate objects, not same instance
// Can lead to data consistency issues
AsNoTrackingWithIdentityResolution 
// Load Customer with Orders
var customers = context.Customers
  .Include(c => c.Orders)
  .AsNoTrackingWithIdentityResolution()
  .ToList();

// If Customer 1 has 5 Orders:
// The JOIN creates 5 result rows
// Identity Resolution detects Customer 1
// appears 5 times by primary key
// Single Customer 1 instance is reused
// Orders collection populated correctly
// No change tracking overhead
// Best of both worlds!

AsNoTrackingWithIdentityResolution is ideal for APIs that return large hierarchical data (customer with orders with order items, etc.) where you want lean memory usage but correct object graph composition. Without it, you'd have duplicate customer objects; with it, the object graph is correct and memory efficient.

08 — SNAPSHOT VS NOTIFICATION CHANGE TRACKING

Two Mechanisms for Detecting Changes

EF has two underlying implementations of change tracking. Snapshot change tracking (the default) stores original values as snapshots and compares on DetectChanges(). Notification change tracking uses INotifyPropertyChanged to detect changes as properties are modified, eliminating the DetectChanges overhead entirely. Snapshot is simpler and requires no entity class changes. Notification requires implementing INotifyPropertyChanged but gives real-time change detection with no DetectChanges() overhead. Choose based on your performance requirements and architecture preferences.

Snapshot Change Tracking (Default) 
// Plain POCO entity, no interfaces
public class Customer
{
  public int Id { get; set; }
  public string Name { get; set; }
  public string Email { get; set; }
}

var customer = context.Customers.Find(1);
customer.Name = "NewName"; // Property set

// Change not detected until SaveChanges
// (which calls DetectChanges internally)

context.SaveChanges();
// DetectChanges compares snapshot to current
// Detects Name changed
// Generates: UPDATE Customers SET Name=...

// Pro: Simple, no entity changes needed
// Con: DetectChanges overhead on large entity sets
Notification Change Tracking 
// Entity implements INotifyPropertyChanged
public class Customer : INotifyPropertyChanged
{
  private string _name;

  public int Id { get; set; }

  public string Name
  {
    get => _name;
    set
    {
      if (_name != value)
      {
        _name = value;
        OnPropertyChanged(nameof(Name));
      }
    }
  }

  public event PropertyChangedEventHandler
    PropertyChanged;

  protected void OnPropertyChanged(
    string propertyName)
  {
    PropertyChanged?.Invoke(this,
      new PropertyChangedEventArgs(propertyName));
  }
}

var customer = context.Customers.Find(1);
customer.Name = "NewName";
// PropertyChanged event fires immediately
// EF marks property as modified in real-time
// No DetectChanges needed

context.SaveChanges();
// EF already knows what changed
// Generates UPDATE immediately, no scanning

// Pro: Real-time change detection, no scan overhead
// Con: Entity classes must implement interface
Snapshot is Default, Notification is Advanced

Snapshot change tracking requires DetectChanges() calls but works with plain POCOs. Notification change tracking requires INotifyPropertyChanged but gives real-time detection with no overhead. Choose snapshot for simplicity, notification for performance on large entity sets.

09 — DETACHED ENTITIES AND DISCONNECTED SCENARIOS

Updating Entities from Web APIs

In web APIs, entities come from clients as JSON, are deserialized to C# objects, but are NOT tracked because they weren't loaded by this DbContext. They're detached. When you call context.Update(detachedEntity), EF marks ALL properties as Modified (assuming everything changed) and generates UPDATE for every column—inefficient. Better approach: context.Attach(detachedEntity) marks it Unchanged without snapshots, then manually mark only the properties that changed via IsModified = true. This generates lean UPDATE statements with only changed columns. Alternatively, return only changed fields from the client and use projections or DTOs to avoid updating unmodified columns.

LINQ Method: Update All Columns 
// From API: deserialize JSON to entity
var customerDto = JsonSerializer
  .Deserialize(jsonPayload);

// Call Update (assumes everything changed)
context.Update(customerDto);
context.SaveChanges();

// SQL generated:
// UPDATE Customers
// SET Name=@1, Email=@2, Phone=@3,
//     Region=@4, IsActive=@5
// WHERE Id=@6;
// ALL columns included even if only
// Name changed! Wasteful.
LINQ Method: Attach Then Mark Changes 
// From API: deserialize
var customerDto = JsonSerializer
  .Deserialize(jsonPayload);

// Attach marks Unchanged (no snapshot)
context.Attach(customerDto);

// Only mark what actually changed
context.Entry(customerDto)
  .Property(c => c.Name).IsModified = true;
context.Entry(customerDto)
  .Property(c => c.Email).IsModified = true;

context.SaveChanges();

// SQL generated:
// UPDATE Customers
// SET Name=@1, Email=@2
// WHERE Id=@3;
// ONLY changed columns! Much leaner.
Real-World Pattern: Track Changes at API Layer 
// API controller receives partial update
[HttpPatch("customers/{id}")]
public async Task UpdateCustomer(
  int id,
  [FromBody] CustomerUpdateDto updateDto)
{
  var customer = await context.Customers
    .FindAsync(id);

  if (customer == null)
    return NotFound();

  // Apply only non-null fields from DTO
  if (updateDto.Name != null)
  {
    customer.Name = updateDto.Name;
  }
  if (updateDto.Email != null)
  {
    customer.Email = updateDto.Email;
  }
  if (updateDto.Phone != null)
  {
    customer.Phone = updateDto.Phone;
  }

  // EF only tracks changed fields automatically
  // because they were loaded, modified,
  // and SaveChanges detected changes via snapshot

  await context.SaveChangesAsync();
  return Ok(customer);
}

// Alternative: Manual DTO-to-Entity Mapping
public async Task UpdateFromDto(
  Customer entity, CustomerUpdateDto dto)
{
  context.Attach(entity);

  // Map only non-null fields
  if (dto.Name != null)
  {
    context.Entry(entity)
      .Property(e => e.Name).IsModified = true;
    entity.Name = dto.Name;
  }
  if (dto.Email != null)
  {
    context.Entry(entity)
      .Property(e => e.Email).IsModified = true;
    entity.Email = dto.Email;
  }

  await context.SaveChangesAsync();
}
Update vs Attach: Different Semantics

Update() assumes all properties changed; Attach() assumes none changed. For detached entities from APIs, Attach() + manual IsModified marking produces lean UPDATEs with only changed columns. Update() is simpler but generates fatter UPDATE statements with all columns.

10 — OPTIMISTIC CONCURRENCY CONTROL

Timestamp and ConcurrencyCheck for Conflict Detection

In concurrent systems, multiple users might edit the same row, leading to lost updates. Optimistic concurrency detects conflicts: store a Timestamp column (SQL Server's rowversion or similar). On UPDATE, include the original timestamp in the WHERE clause. If another user updated the row first, the timestamp won't match and UPDATE affects zero rows—conflict detected. The alternative is ConcurrencyCheck attributes on regular columns: EF includes them in WHERE clauses too. SQL Server temporal tables offer an alternative approach with automatic versioning, but EF's timestamp mechanism is simpler and more portable.

SQL: Manual Concurrency Check 
-- Conflict detection via rowversion
UPDATE Products
SET Name = @NewName,
    RowVersion = DEFAULT  -- Auto-updated
WHERE Id = @Id
  AND RowVersion = @OriginalVersion;

-- If 0 rows affected:
-- @@ROWCOUNT = 0
-- Another user must have updated it
-- Conflict detected, client retries or errors
EF: Timestamp for Concurrency 
public class Product
{
  public int Id { get; set; }
  public string Name { get; set; }

  // Marks property as rowversion
  // SQL Server auto-increments on every change
  [Timestamp]
  public byte[] RowVersion { get; set; }
}

var product = context.Products.Find(1);
var originalVersion = product.RowVersion;

product.Name = "NewName";

try
{
  context.SaveChanges();
  // EF generates:
  // UPDATE Products
  // SET Name=@1, RowVersion=DEFAULT
  // WHERE Id=@2 AND RowVersion=@3;
  // @3 = originalVersion
}
catch (DbUpdateConcurrencyException)
{
  // RowVersion didn't match
  // Another user updated this product
  // Handle conflict
}
ConcurrencyCheck on Alternate Columns 
public class Order
{
  public int Id { get; set; }
  public string Status { get; set; }

  // Marks column for concurrency checks
  // EF includes in WHERE clause on UPDATE
  [ConcurrencyCheck]
  public int Version { get; set; }

  // Alternative: use multiple ConcurrencyCheck columns
  [ConcurrencyCheck]
  public DateTime LastModified { get; set; }
}

var order = context.Orders.Find(1);
var originalVersion = order.Version;

order.Status = "Completed";
order.Version++; // Manual increment

try
{
  context.SaveChanges();
  // EF generates:
  // UPDATE Orders
  // SET Status=@1, LastModified=@2
  // WHERE Id=@3
  //   AND Version=@4
  //   AND LastModified=@5;
  // Both ConcurrencyCheck columns in WHERE
}
catch (DbUpdateConcurrencyException ex)
{
  // Handle conflict
  var entry = ex.Entries.Single();
  var databaseVersion =
    await entry.GetDatabaseValuesAsync();

  // Resolve: client wins
  var clientVersion = order.Version;
  entry.OriginalValues.SetValues(
    databaseVersion);
  order.Version = clientVersion;

  await context.SaveChangesAsync();
}

Timestamp (rowversion) is the preferred mechanism because SQL Server auto-manages it. ConcurrencyCheck attributes on regular columns give you flexibility to choose which properties participate in conflict detection. Temporal tables offer a different approach: automatic versioning at the SQL Server level with a separate history table. Choose based on your concurrency strategy and conflict resolution approach.

11 — CONFLICT RESOLUTION PATTERNS

Client Wins, Database Wins, and Merge Strategies

When a DbUpdateConcurrencyException is caught, you have three main resolution patterns. Client wins: discard database changes and re-apply the client's edit (optimistic, common in web apps). Database wins: discard client changes and reload from database (pessimistic). Merge: resolve field-by-field, prompting user for choices where they differ. Which strategy you choose depends on your domain: financial systems often use database wins (conservative), collaborative apps often use client wins (optimistic), and conflict-prone scenarios might need merge or user intervention.

All Conflict Resolution Patterns 
public async Task UpdateProductAsync(
  int id, ProductUpdateDto dto)
{
  var product = await context.Products
    .FindAsync(id);
  var originalVersion = product.RowVersion;

  // Apply client changes
  product.Name = dto.Name;
  product.Price = dto.Price;

  try
  {
    await context.SaveChangesAsync();
  }
  catch (DbUpdateConcurrencyException ex)
  {
    var entry = ex.Entries.Single();
    var dbValues =
      await entry.GetDatabaseValuesAsync();
    var clientValues =
      entry.CurrentValues.Clone();

    // PATTERN 1: Client Wins
    // Discard DB, re-apply client edits
    Console.WriteLine("Pattern 1: Client Wins");
    entry.OriginalValues.SetValues(dbValues);
    product.Name = dto.Name;
    product.Price = dto.Price;
    await context.SaveChangesAsync();
    // Client's version is persisted

    // PATTERN 2: Database Wins
    // Discard client changes
    Console.WriteLine("Pattern 2: Database Wins");
    entry.CurrentValues.SetValues(dbValues);
    // Reload and return DB values to client
    return new { Success = false,
      Data = dbValues };

    // PATTERN 3: Merge (Field-by-Field)
    // Detect which fields conflicted
    Console.WriteLine("Pattern 3: Merge");
    var nameConflict =
      !Equals(dbValues["Name"],
        clientValues["Name"]);
    var priceConflict =
      !Equals(dbValues["Price"],
        clientValues["Price"]);

    if (nameConflict)
    {
      throw new ConflictException(
        $"Name conflict: " +
        $"DB='{dbValues["Name"]}' " +
        $"vs Client='{clientValues["Name"]}'");
      // User must resolve manually
    }

    if (priceConflict)
    {
      // For price, use DB value (conservative)
      product.Price =
        (decimal)dbValues["Price"];
    }

    // Re-apply what can be applied
    entry.OriginalValues.SetValues(dbValues);
    product.Name = dto.Name;
    await context.SaveChangesAsync();
  }
}
Choose Conflict Strategy Based on Domain

Client wins is optimistic and common. Database wins is conservative. Merge requires user intervention. Financial systems often use database wins. Web apps often use client wins. Choose based on your requirements and risk tolerance.

12 — BULK UPDATES WITH EXECUTEUPDATE AND EXECUTEDELETE

EF Core 7+ Bulk Operations that Bypass Change Tracking

Starting in EF Core 7, ExecuteUpdate() and ExecuteDelete() allow bulk operations that bypass the change tracker entirely, sending SQL directly to the database. This is much faster than loading entities, modifying them, and calling SaveChanges(). ExecuteUpdate() maps to SQL UPDATE statements. ExecuteDelete() maps to DELETE. These bypass change tracking, snapshots, and conflict detection—they're pure SQL operations at the LINQ level. Use them for bulk operations where you don't need entity materialization or change tracking. They generate efficient batch SQL and execute immediately without a surrounding transaction context like SaveChanges().

SQL: Direct Bulk Update 
-- Traditional SQL update
UPDATE Customers
SET IsActive = 0,
    LastModified = GETDATE()
WHERE CreatedDate < DATEADD(YEAR, -1, GETDATE());

-- Or direct delete
DELETE FROM Orders
WHERE Status = 'Cancelled'
  AND CreatedDate < DATEADD(MONTH, -3, GETDATE());
EF Core 7+: ExecuteUpdate/ExecuteDelete 
// Bulk deactivate old customers
await context.Customers
  .Where(c => c.CreatedDate <
    DateTime.Now.AddYears(-1))
  .ExecuteUpdateAsync(s => s
    .SetProperty(c => c.IsActive, false)
    .SetProperty(c => c.LastModified,
      DateTime.Now));

// Bulk delete old orders
await context.Orders
  .Where(o => o.Status == "Cancelled" &&
    o.CreatedDate < DateTime.Now.AddMonths(-3))
  .ExecuteDeleteAsync();

// No entity materialization, no change tracking
// Executes directly as SQL
// Much faster than load-modify-save
Bulk Operations with Multiple SetProperty Clauses 
// Update multiple columns at once
await context.Products
  .Where(p => p.StockQuantity < p.MinimumStock)
  .ExecuteUpdateAsync(s => s
    .SetProperty(p => p.Status, "LowStock")
    .SetProperty(p => p.LastAlertDate,
      DateTime.Now)
    .SetProperty(p => p.AlertCount,
      p => p.AlertCount + 1));

// Complex condition: increase price for certain category
await context.Products
  .Where(p => p.Category == "Premium" &&
    p.Price < 100)
  .ExecuteUpdateAsync(s => s
    .SetProperty(p => p.Price,
      p => p.Price * 1.1m)); // 10% increase

// Soft delete pattern
await context.Orders
  .Where(o => o.Id == orderId)
  .ExecuteUpdateAsync(s => s
    .SetProperty(o => o.IsDeleted, true)
    .SetProperty(o => o.DeletedDate, DateTime.Now));

// Method syntax: alternative to Where
var deletedCount = await context.Orders
  .Where(o => o.Status == "Cancelled")
  .ExecuteDeleteAsync();

Console.WriteLine($"Deleted {deletedCount} cancelled orders");

// Query syntax equivalent
var updated = await (from c in context.Customers
  where c.IsInactive == true
  select c)
  .ExecuteUpdateAsync(s => s
    .SetProperty(c => c.IsActive, false));
ExecuteUpdate/ExecuteDelete Skip Change Tracking

These operations don't raise change tracking events, don't populate snapshots, and don't respect concurrency tokens. They execute immediately and directly as SQL. Use them only for bulk operations where you don't need entity materialization or change tracking. For single-entity updates with concurrency control, use traditional load-modify-SaveChanges pattern.

ExecuteUpdate and ExecuteDelete are game-changers for bulk operations. A query that would load 100,000 entities, modify them, and SaveChanges (which calls DetectChanges on all 100,000 entities) can now be a single SQL UPDATE statement. This is why they're preferred for bulk operations: they're orders of magnitude faster and use far less memory.

13 — ORIGINAL VALUES VS CURRENT VALUES

Auditing What Changed and Detecting Modifications

Every tracked entity has both OriginalValues (what it was when loaded) and CurrentValues (what it is now). You can inspect these to implement auditing, change logs, or complex conflict detection. OriginalValues is read-only and represents the snapshot. CurrentValues is writable and represents the object's current state. This is core to how EF detects what changed: it compares original to current. You can also manually inspect and modify these values for advanced scenarios like replaying changes or implementing audit trails.

Inspecting Original and Current Values 
var customer = context.Customers.Find(1);
// Original: Name="Alice", Email="alice@old.com"

customer.Name = "Alice Smith";
customer.Email = "alice@new.com";

var entry = context.Entry(customer);

// Get original values
var originalName =
  entry.OriginalValues["Name"];
var originalEmail =
  entry.OriginalValues["Email"];

// Get current values
var currentName =
  entry.CurrentValues["Name"];
var currentEmail =
  entry.CurrentValues["Email"];

Console.WriteLine(
  $"Name: {originalName} → {currentName}");
// Output: Name: Alice → Alice Smith

// Check specific property
var nameChanged =
  entry.Property(c => c.Name).IsModified;
// Output: true

// Inspect all properties
foreach (var property in entry.Properties)
{
  var original = property.OriginalValue;
  var current = property.CurrentValue;
  var changed = !Equals(original, current);

  if (changed)
  {
    Console.WriteLine(
      $"{property.Metadata.Name}: " +
      $"{original} → {current}");
  }
}

// Build audit trail before saving
var auditLog = new AuditLog
{
  EntityType = entry.Entity.GetType().Name,
  EntityId = (int)entry.CurrentValues["Id"],
  Changes = new Dictionary()
};

foreach (var property in entry.Properties
  .Where(p => p.IsModified))
{
  auditLog.Changes[property.Metadata.Name] =
    $"{property.OriginalValue} → {property.CurrentValue}";
}

context.AuditLogs.Add(auditLog);
context.SaveChanges();
Original vs Current: The Foundation of Change Detection

EF's change tracking works by comparing OriginalValues to CurrentValues. This comparison drives the UPDATE generation and enables auditing. Understanding and inspecting these values is essential for implementing advanced patterns like change logs and conflict detection.

14 — SQL SERVER TEMPORAL TABLES VS EF CHANGE TRACKING

Two Approaches to Audit Trails and Versioning

SQL Server temporal tables provide automatic row versioning at the database level: every modification is recorded in a history table with SysStartTime and SysEndTime columns. EF's change tracking is a different approach: in-memory snapshot comparison. Temporal tables excel at full audit trails and point-in-time queries. EF's change tracking excels at detecting what changed before SaveChanges(). They're complementary: use temporal tables for complete audit history, use EF's change tracking for pre-save validation and conflict detection. EF 7+ has improved temporal table support, but the mechanisms are distinct.

SQL Server Temporal Tables 
-- Table with temporal versioning
CREATE TABLE Customers
(
  Id INT PRIMARY KEY,
  Name NVARCHAR(100),
  Email NVARCHAR(100),
  SysStartTime DATETIME2 GENERATED
    ALWAYS AS ROW START,
  SysEndTime DATETIME2 GENERATED
    ALWAYS AS ROW END,
  PERIOD FOR SYSTEM_TIME
    (SysStartTime, SysEndTime)
)
WITH (SYSTEM_VERSIONING = ON);

-- When you UPDATE, history is kept automatically
UPDATE Customers SET Name = 'Alice Smith'
WHERE Id = 1;

-- Query current data
SELECT * FROM Customers;
-- Get row at specific time
SELECT * FROM Customers
FOR SYSTEM_TIME AS OF '2024-01-01';

-- Get full version history
SELECT * FROM Customers
FOR SYSTEM_TIME ALL;
EF Change Tracking for Audits 
// Before SaveChanges, audit what changed
public override int SaveChanges()
{
  var changes = ChangeTracker.Entries()
    .Where(e => e.State != EntityState.Unchanged)
    .ToList();

  foreach (var entry in changes)
  {
    var auditEntry = new AuditLog
    {
      EntityType = entry.Entity.GetType().Name,
      EntityId = (int)entry
        .CurrentValues.GetValue("Id"),
      Operation = entry.State.ToString(),
      Timestamp = DateTime.UtcNow
    };

    if (entry.State == EntityState.Modified)
    {
      var changes = new Dictionary();
      foreach (var property in entry.Properties)
      {
        if (property.IsModified)
        {
          changes[property.Metadata.Name] =
            $"{property.OriginalValue} → " +
            $"{property.CurrentValue}";
        }
      }
      auditEntry.Changes =
        JsonConvert.SerializeObject(changes);
    }

    AuditLogs.Add(auditEntry);
  }

  return base.SaveChanges();
}

SQL Server temporal tables are automatic and comprehensive—every version is kept. EF change tracking is selective—you decide what to audit. For compliance-heavy applications needing complete audit trails, temporal tables are ideal. For applications needing to detect changes before saving and implement custom logic, EF's change tracking is more flexible. Many applications use both: temporal tables for complete history, EF change tracking for pre-save validation.

15 — PERFORMANCE BENCHMARKS AND OPTIMIZATION

Tracking Overhead and Memory Impact

Change tracking has measurable overhead: snapshots are stored in memory, DetectChanges iterates all tracked entities, and the ChangeTracker data structures consume memory. Benchmarks show that tracking 10,000 entities costs approximately 50-100MB of memory and adds 20-40ms to materialization time versus AsNoTracking(). For read-heavy workloads like reports or APIs, this overhead is pure waste. For write-heavy workloads, tracking is necessary. The optimization strategy is clear: use AsNoTracking() by default for queries that don't modify data, track only when you call SaveChanges(). This simple rule avoids most overhead while maintaining correctness where it matters.

Performance Comparison: Tracked vs AsNoTracking 
// Benchmark: Loading 10,000 customers
var sw = Stopwatch.StartNew();

// SCENARIO 1: Tracked (default)
var tracked = context.Customers
  .Where(c => c.IsActive)
  .ToList();
sw.Stop();
Console.WriteLine(
  $"Tracked: {sw.ElapsedMilliseconds}ms");
// Output: ~45ms, memory: ~50MB

sw.Restart();

// SCENARIO 2: AsNoTracking
var untracked = context.Customers
  .Where(c => c.IsActive)
  .AsNoTracking()
  .ToList();
sw.Stop();
Console.WriteLine(
  $"AsNoTracking: {sw.ElapsedMilliseconds}ms");
// Output: ~28ms, memory: ~25MB
// 40% faster, 50% less memory!

// SCENARIO 3: Large batch with DetectChanges
sw.Restart();
var customers = context.Customers
  .Take(10000)
  .ToList();

foreach (var c in customers)
{
  c.LastActivityDate = DateTime.UtcNow;
  // No automatic detection yet
}

// Explicit DetectChanges call
context.ChangeTracker.DetectChanges();
sw.Stop();
Console.WriteLine(
  $"DetectChanges on 10,000: " +
  $"{sw.ElapsedMilliseconds}ms");
// Output: ~35ms

context.SaveChanges();

// OPTIMIZATION STRATEGY:
// - API GETs: Always AsNoTracking
// - Reports: Always AsNoTracking
// - List displays: Always AsNoTracking
// - PUTs/POSTs: Track only when modifying
// - Bulk operations: ExecuteUpdate/ExecuteDelete
// - Concurrency-critical: Track with timestamps
AsNoTracking is the Key to Performance

For read-only queries, AsNoTracking() eliminates all tracking overhead: no snapshots, no DetectChanges, no change detection. The SQL is identical, but EF's in-memory cost drops dramatically. Use it as your default for any query that doesn't call SaveChanges().

16 — ATTACH UPDATE AND ENTRY STATE PATTERNS

Managing Entity Lifecycle Programmatically

You can control entity states manually via Attach(), Update(), Remove(), and Entry().State assignments. Attach() marks a detached entity as Unchanged (no snapshot). Update() marks all properties as Modified. Remove() marks for deletion. Entry().State = EntityState.* allows direct state assignment. These patterns are useful for managing detached entities from APIs or for advanced scenarios where you need explicit control over what SQL SaveChanges() generates. Understanding these methods helps you architect persistence logic correctly, especially in web APIs where entities are frequently detached and reattached.

Attach, Update, Remove, and Manual State Control 
// Scenario 1: Attach a detached entity
var detachedCustomer = new Customer
{
  Id = 1,
  Name = "Alice"
};

// Attach marks Unchanged, no snapshot
context.Attach(detachedCustomer);

// Now you can modify and it's tracked
detachedCustomer.Name = "Alice Smith";
// State becomes Modified

context.SaveChanges();
// UPDATE generated for Name

// Scenario 2: Update (marks all as Modified)
var detachedCustomer2 = new Customer
{
  Id = 2,
  Name = "Bob"
};

context.Update(detachedCustomer2);
// Marks ALL properties as Modified immediately
// No need to manually mark each property

context.SaveChanges();
// UPDATE includes all columns

// Scenario 3: Manual state assignment
var customer = context.Customers.Find(3);

// Manually mark as unmodified
context.Entry(customer).State =
  EntityState.Unchanged;
// Changes won't be saved

// Manually mark as deleted
context.Entry(customer).State =
  EntityState.Deleted;

context.SaveChanges();
// DELETE generated

// Scenario 4: Revert changes
var customer4 = context.Customers.Find(4);
customer4.Name = "Changed";

// Revert to original
context.Entry(customer4).RejectChanges();

Console.WriteLine(customer4.Name);
// Output: original value, changes discarded

// Scenario 5: Create duplicate tracking
var customer5a = context.Customers.Find(5);
var customer5b = context.Customers.Find(5);
// EF ensures they're the SAME instance
Console.WriteLine(ReferenceEquals(customer5a,
  customer5b)); // true

// Scenario 6: Detach entity
context.Detach(customer);
// Entry is no longer tracked
var state = context.Entry(customer).State;
// EntityState.Detached

// Scenario 7: Bulk state change
var allCustomers = context.Customers
  .Local // Only loaded in this context
  .Where(c => c.IsInactive)
  .ToList();

foreach (var c in allCustomers)
{
  context.Entry(c).State = EntityState.Deleted;
}

context.SaveChanges();
// Multiple DELETEs for all inactive

These patterns give you fine-grained control over entity lifecycle management. Attach is efficient for detached entities. Update() is simple but generates fat UPDATEs. Manual state assignment gives you maximum control. Choose based on your scenario and preference for explicit control vs simplicity.

Key Takeaways

  • Change tracking: EF snapshots entities when loaded and detects modifications by comparing current values to originals
  • Entity states (Added, Modified, Deleted, Unchanged, Detached) directly map to INSERT, UPDATE, DELETE, and no-op SQL statements
  • SaveChanges() generates SQL based on entity states; only Modified and Added entities produce DML statements
  • The snapshot mechanism enables lean UPDATE statements: only changed columns are included, not entire rows
  • ChangeTracker API exposes all tracked entities, their states, and original/current values for auditing and validation
  • DetectChanges() compares snapshots to current values; it's automatic on SaveChanges() but can be called manually
  • AsNoTracking() eliminates snapshot creation and change monitoring for read-only queries: 30-50% performance improvement
  • AsNoTrackingWithIdentityResolution prevents duplicate objects in complex result sets without tracking overhead
  • Detached entities from APIs require Attach() plus manual IsModified marking for efficient updates, or Update() for simplicity
  • Snapshot change tracking is default and requires no entity changes; notification tracking via INotifyPropertyChanged is faster but requires implementation
  • Optimistic concurrency: [Timestamp] columns detect conflicts when multiple users edit the same row
  • Conflict resolution patterns: client wins (re-apply edit), database wins (discard changes), merge (field-by-field with user intervention)
  • ExecuteUpdate() and ExecuteDelete() bypass change tracking entirely for efficient bulk operations (EF Core 7+)
  • Original values vs current values enable auditing, change logs, and conflict detection; EF compares them to detect what changed
  • SQL Server temporal tables provide automatic version history at the database level; EF change tracking provides pre-save validation
  • Performance: tracking overhead is measurable (~50MB per 10K entities) but eliminated by AsNoTracking() for read-only queries
  • Attach, Update, Remove, and manual state control give fine-grained management of entity lifecycle and what SQL SaveChanges() generates
