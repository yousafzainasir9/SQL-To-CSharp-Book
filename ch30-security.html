Chapter 30

Security

Prevent SQL injection. Protect connection strings. Validate input. Use parameterized queries. Implement least privilege. Defense in depth.

01

Requirement: "Prevent SQL Injection"

Business Requirement
Users can search customers by email. Attackers could inject SQL. How do we prevent it?
SQL Developer Thinks
Never concatenate user input into SQL. Use parameterized queries where values are passed separately from SQL text.
C# Developer Thinks
LINQ automatically uses parameters. FromSqlInterpolated uses parameters. Never use FromSqlRaw with string concatenation.
VULNERABLE - String Concat
string email = GetUserInput();  // "alice' OR '1'='1"
var sql = "SELECT * FROM Users WHERE Email = '" + email + "'";
// Executes: ... WHERE Email = 'alice' OR '1'='1'
// Returns ALL users!
SAFE - Parameterized Query
string email = GetUserInput();
var user = await db.Users
    .AsNoTracking()
    .FromSqlInterpolated($"SELECT * FROM Users WHERE Email = {email}")
    .FirstOrDefaultAsync(ct);
// Parameter passed separately, never interpreted as SQL
02

Safe Query Methods

Business Requirement
Different ways to query. Which are safe?
SQL Developer Thinks
Always use parameters. Never concatenate user input.
C# Developer Thinks
LINQ is safest. FromSqlInterpolated is safe. FromSqlRaw needs careful handling.
MethodSafetyNotes
LINQ WhereSafeAuto-parameterized
FromSqlInterpolatedSafeDollar-sign parameterizes {}
FromSqlRaw + paramsSafePass SqlParameter separately
String concatUNSAFENever use
LINQ (Safest)
var email = userInput;
var user = await db.Users
    .AsNoTracking()
    .Where(u => u.Email == email)
    .FirstOrDefaultAsync(ct);
// LINQ parameterizes automatically
FromSqlRaw (Needs params)
var user = await db.Users
    .AsNoTracking()
    .FromSqlRaw("SELECT * FROM Users WHERE Email = @Email",
        new SqlParameter("@Email", email))
    .FirstOrDefaultAsync(ct);
03

Connection String Security

Business Requirement
Never hardcode passwords. Where do credentials belong?
SQL Developer Thinks
Credentials in connection string. Never in source code.
C# Developer Thinks
Use configuration (appsettings.json). Inject secrets at runtime from Azure Key Vault or AWS Secrets Manager.
WRONG - Hardcoded
const string connStr = "Server=srv;User=admin;Password=Passw0rd123!";
// NEVER commit secrets to source code!
RIGHT - Configured
// appsettings.json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=srv;User=admin;Password=???"
  }
}

// Startup
var connStr = configuration.GetConnectionString("DefaultConnection");
// Secrets injected at runtime from Azure Key Vault
04

Least Privilege Principle

Business Requirement
Application user should only have permissions it needs. Separate app user from admin user.
SQL Developer Thinks
Create AppUser with only SELECT, INSERT, UPDATE on specific tables. No DELETE, ALTER, CREATE. Admin user for migrations only.
C# Developer Thinks
Connection string uses AppUser. Migrations run with AdminUser or separate script.
SQL Server Least Privilege
-- Application user (limited)
CREATE USER AppUser WITH PASSWORD = 'Strong@Password1';
GRANT SELECT, INSERT, UPDATE ON Products TO AppUser;
GRANT SELECT, INSERT, UPDATE ON Orders TO AppUser;
-- No DELETE, ALTER, CREATE

-- Admin user (for migrations)
CREATE USER AdminUser WITH PASSWORD = 'Strong@Password2';
GRANT ALL PRIVILEGES TO AdminUser;

-- Connection string uses AppUser
Server=myserver;User Id=AppUser;Password=Strong@Password1;
