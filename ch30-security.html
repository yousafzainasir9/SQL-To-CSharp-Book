Security — SQL Injection Prevention

Master the most critical security vulnerability in data access code: SQL injection. Learn how parameterized queries prevent attacks, identify dangerous patterns in EF Core and Dapper, and implement defense-in-depth strategies with stored procedures and connection string security.

01 — What Is SQL Injection?

Understanding the Attack and Its Consequences

SQL injection is an attack where user input is concatenated directly into SQL commands, allowing attackers to insert arbitrary SQL code. Instead of using your intended SQL, the database executes the attacker's malicious SQL. The consequences can be catastrophic: data theft, deletion, modification, or complete database compromise.

The fundamental problem is treating SQL as a string to be built rather than a structured command with separate code and data. When code and data are mixed, the parser can't distinguish which is which, and an attacker exploiting this ambiguity can inject commands.

SQL — Innocent Query vs Injection Attack -- INNOCENT: Looking for user by username -- Code: SELECT * FROM Users WHERE Username = '...' -- User input: "alice" -- Result: SELECT * FROM Users WHERE Username = 'alice' -- Returns: Alice's record -- ATTACK: Attacker inputs a different string -- User input: "'; DROP TABLE Users; --" -- Result: SELECT * FROM Users WHERE Username = ''; // DROP TABLE Users; --' -- Executes 3 statements: // 1. SELECT * FROM Users WHERE Username = '' (empty result) // 2. DROP TABLE Users; (DESTROYS TABLE!) // 3. -- ' (comment, rest ignored) -- CONSEQUENCES: // - Users table is deleted // - All user data lost // - Application breaks // - Data recovery needed (backup restore) -- Another attack example: data extraction -- User input: "' OR '1'='1" -- Result: SELECT * FROM Users WHERE Username = '' OR '1'='1' -- WHERE condition is always true // Returns all users (password hash exposure) -- More sophisticated: time-based inference // Attacker can extract data character-by-character // By causing delays on true conditions // This is called "blind SQL injection"
C# — The Vulnerable Pattern // VULNERABLE CODE - DO NOT USE public class UserRepository { private readonly SqlConnection _connection; // NEVER do this: public User GetUserVulnerable(string username) { // String concatenation = code + data mixed string query = "SELECT * FROM Users WHERE Username = '" + username + "'"; using (var command = new SqlCommand(query, _connection)) { using (var reader = command.ExecuteReader()) { if (reader.Read()) { return new User { Id = (int)reader["Id"], Username = (string)reader["Username"], // ... other fields }; } } } return null; } // If attacker passes username = "'; DROP TABLE Users; --" // SQL becomes: SELECT * FROM Users WHERE Username = ''; // DROP TABLE Users; --' // Two commands execute: // 1. SELECT (finds nothing) // 2. DROP TABLE (DELETES USERS TABLE!) // LINQ in EF Core is safe automatically: public User GetUserSafe(string username) { // EF Core AUTOMATICALLY parameterizes this var user = dbContext.Users .FirstOrDefault(u => u.Username == username); // No SQL injection possible - LINQ handles parameterization return user; } // Dapper is safe with parameters: public User GetUserDapperSafe(string username) { const string query = "SELECT * FROM Users WHERE Username = @Username"; return _connection.QueryFirstOrDefault( query, new { Username = username } // @ parameter syntax ); // Dapper parameterizes automatically } }
Key Insight: Concatenation + User Input = Vulnerability

The root cause of SQL injection is simple: concatenating user input directly into SQL strings. When input is treated as code instead of data, attackers can insert malicious commands. The fix is equally simple: use parameterized queries where code and data are separate. The SQL engine then treats the parameter as data only, never as executable code.

02 — Parameterized Queries: The Universal Fix

Separating Code from Data to Eliminate Injection

Parameterized queries (also called prepared statements) are the universal solution. Instead of concatenating user input into the SQL string, you provide SQL with placeholders (parameters) and then bind the data separately. The SQL engine treats the data as data, never as code. This is supported everywhere: SQL Server, MySQL, PostgreSQL, SQLite—and all data access frameworks.

SQL — Parameterized Query Syntax -- Parameterized query uses @ placeholders -- SQL code: "SELECT * FROM Users WHERE Username = @Username" -- Data: { Username = "alice" } -- The database receives the query and parameters SEPARATELY SELECT * FROM Users WHERE Username = @Username; -- @Username is a PLACEHOLDER -- Value is provided separately by the client -- Database never interprets the value as SQL code -- Example with multiple parameters: SELECT * FROM Orders WHERE CustomerId = @CustomerId AND OrderDate >= @StartDate AND Status = @Status; -- Three parameters: CustomerId, StartDate, Status // Database receives SQL structure + separate data values // Data cannot contain SQL code because it's not in the query -- Attack attempt with parameterized query: // Username parameter = "'; DROP TABLE Users; --" // SQL sent to server: SELECT * FROM Users WHERE Username = @Username // Parameter value sent: "'; DROP TABLE Users; --" // Database interprets as string literal, looks for username containing '; DROP... // No execution of DROP TABLE - it's just part of the data value
C# — Parameterized Queries in All Frameworks // PATTERN 1: Dapper with parameters public class DapperUserRepository { private readonly IDbConnection _connection; public User GetByUsername(string username) { const string query = "SELECT * FROM Users WHERE Username = @Username"; return _connection.QueryFirstOrDefault( query, new { Username = username } // Parameter object ); // Dapper automatically: // 1. Parameterizes the query // 2. Sends SQL and parameters to server separately // 3. Server never treats parameter as code } public List GetOrdersByDateRange( DateTime startDate, DateTime endDate, int customerId) { const string query = @" SELECT * FROM Orders WHERE CustomerId = @CustomerId AND OrderDate >= @StartDate AND OrderDate <= @EndDate ORDER BY OrderDate DESC"; return _connection.Query( query, new { CustomerId = customerId, StartDate = startDate, EndDate = endDate } ).ToList(); } } // PATTERN 2: Raw ADO.NET with SqlCommand public class AdoNetUserRepository { private readonly string _connectionString; public User GetByUsername(string username) { using (var connection = new SqlConnection(_connectionString)) using (var command = new SqlCommand( "SELECT * FROM Users WHERE Username = @Username", connection)) { // Add parameter - separated from SQL command.Parameters.AddWithValue("@Username", username); connection.Open(); using (var reader = command.ExecuteReader()) { if (reader.Read()) { return new User { Id = (int)reader["Id"], Username = (string)reader["Username"] }; } } } return null; } } // PATTERN 3: EF Core LINQ (safest - automatic parameterization) public class EFCoreUserRepository { private readonly UserContext _context; public User GetByUsername(string username) { // EF Core AUTOMATICALLY parameterizes this // Behind the scenes: // SQL: "SELECT ... FROM Users WHERE Username = @p0" // Parameter: { @p0 = username } return _context.Users .FirstOrDefault(u => u.Username == username); // No chance of SQL injection - parameterization is automatic } public List GetOrdersByDateRange( DateTime startDate, DateTime endDate, int customerId) { // Multiple parameters, automatic parameterization return _context.Orders .Where(o => o.CustomerId == customerId && o.OrderDate >= startDate && o.OrderDate <= endDate) .OrderByDescending(o => o.OrderDate) .ToList(); } }

The key principle is: SQL code and data are never mixed. SQL is sent to the database, and data is sent separately. The database engine uses the SQL as the template and inserts the data without interpreting it as code.

03 — Dangerous Patterns to Avoid

Identifying Vulnerable Code and Fixing It

Even experienced developers sometimes write vulnerable SQL. The most common mistake is using raw SQL in EF Core, which looks safe but isn't. Other mistakes include string interpolation, dynamic table names, and trying to "sanitize" user input (a losing battle—parameterization is the only reliable defense).

SQL — Scenarios That Need Parameters -- Scenario 1: Search query -- VULNERABLE: string concatenation SELECT * FROM Products WHERE Name LIKE '%' + @SearchTerm + '%' -- SAFE: parameterization (with LIKE wildcard in parameter) SELECT * FROM Products WHERE Name LIKE @SearchTerm -- Pass parameter as "%searchterm%" -- Scenario 2: Dynamic column names (can't be parameterized) -- VULNERABLE: SELECT * FROM Users WHERE @ColumnName = @Value -- This doesn't work - @ColumnName is treated as data, not column name -- SAFE: Use stored procedures or builder pattern -- Stored procedure checks input against whitelist EXEC sp_SearchColumn @ColumnName, @Value -- Scenario 3: IN clause with multiple values -- VULNERABLE: SELECT * FROM Orders WHERE Id IN (123,456,789) -- If building the list from user input: // string ids = "123,456," + userInput -- concatenation! -- SAFE: Use Table-Valued Parameters or multiple parameters SELECT * FROM Orders WHERE Id IN (@Id1, @Id2, @Id3) -- or use SQL Server's Table-Valued Parameters
C# — Dangerous Code Patterns and Fixes // ══════════════════════════════════════════════════ // PATTERN 1: String Concatenation (NEVER DO THIS) // ══════════════════════════════════════════════════ // VULNERABLE: public List SearchProductsVulnerable(string searchTerm) { string query = $"SELECT * FROM Products WHERE Name LIKE '%{searchTerm}%'"; // Attacker passes: "%' OR 1=1 --" // Results in: SELECT * FROM Products WHERE Name LIKE '%%' OR 1=1 --%' // Returns all products! using (var command = new SqlCommand(query, _connection)) { // VULNERABLE CODE - NEVER USE } } // SAFE: public List SearchProductsSafe(string searchTerm) { const string query = "SELECT * FROM Products WHERE Name LIKE @SearchTerm"; using (var command = new SqlCommand(query, _connection)) { command.Parameters.AddWithValue("@SearchTerm", "%" + searchTerm + "%"); // String concatenation AFTER parameterization is safe // The concatenated value is treated as a single parameter } } // ══════════════════════════════════════════════════ // PATTERN 2: EF Core FromSqlRaw with Interpolation (DANGEROUS!) // ══════════════════════════════════════════════════ public class ProductRepository { private readonly ProductContext _context; // VULNERABLE: FromSqlRaw with string interpolation public List GetByNameVulnerable(string name) { // This LOOKS like it might be safe, but it's not! return _context.Products .FromSqlRaw($"SELECT * FROM Products WHERE Name = {name}") .ToList(); // String interpolation happens BEFORE SQL is sent // Equivalent to: "SELECT * FROM Products WHERE Name = " + name // SQL INJECTION VULNERABILITY! } // SAFE: FromSqlInterpolated (auto-parameterizes) public List GetByNameSafe(string name) { return _context.Products .FromSqlInterpolated($"SELECT * FROM Products WHERE Name = {name}") .ToList(); // FromSqlInterpolated automatically parameterizes // The interpolation creates parameters, not concatenation // Safe from SQL injection! } // Also SAFE: FromSqlRaw with parameters public List GetByNameAlt(string name) { return _context.Products .FromSqlRaw( "SELECT * FROM Products WHERE Name = @Name", new SqlParameter("@Name", name)) .ToList(); } } // KEY DIFFERENCE: // FromSqlRaw($"...{var}...") = VULNERABLE (concatenation) // FromSqlInterpolated($"...{var}...") = SAFE (parameterization) // FromSqlRaw("...", params) = SAFE (explicit parameters) // ══════════════════════════════════════════════════ // PATTERN 3: Dynamic Table/Column Names (CAN'T parameterize) // ══════════════════════════════════════════════════ // VULNERABLE: public List SearchDynamicColumn( string tableName, string columnName, string value) { string query = $"SELECT * FROM {tableName} WHERE {columnName} = '{value}'"; // All three inputs could be injection vectors! } // SAFE: Validate against whitelist public List SearchDynamic( string tableName, string columnName, string value) { // Whitelist of allowed tables and columns var allowedTables = new[] { "Products", "Orders", "Customers" }; var allowedColumns = new[] { "Id", "Name", "Email" }; if (!allowedTables.Contains(tableName)) throw new ArgumentException("Invalid table"); if (!allowedColumns.Contains(columnName)) throw new ArgumentException("Invalid column"); // Now safe to build query (table/column are whitelisted) string query = $"SELECT * FROM {tableName} WHERE {columnName} = @Value"; // Value is still parameterized! using (var command = new SqlCommand(query, _connection)) { command.Parameters.AddWithValue("@Value", value); // Execute safely } } // ══════════════════════════════════════════════════ // PATTERN 4: "Sanitizing" Input (WRONG APPROACH) // ══════════════════════════════════════════════════ // NEVER rely on sanitization: public bool IsSafeInput(string input) { // Try to remove dangerous characters return !input.Contains("'") && !input.Contains(";") && !input.Contains("--"); // This is ineffective - many bypasses exist // Use parameterization, not sanitization! }
Key Insight: FromSqlRaw vs FromSqlInterpolated

In EF Core, FromSqlRaw($"...{var}...") is vulnerable because the $ happens before FromSqlRaw sees it. FromSqlInterpolated($"...{var}...") is safe because it converts the interpolation into parameters. This subtle difference trips up experienced developers. Always use FromSqlInterpolated unless you're explicitly providing SqlParameter objects to FromSqlRaw.

04 — EF Core SQL Injection Risks

Safe vs Unsafe Patterns in Entity Framework

EF Core's LINQ is safe by default—automatic parameterization prevents injection. However, raw SQL methods (FromSqlRaw, ExecuteSqlRaw) can be dangerous if misused. The pattern FromSqlRaw($"...") with string interpolation is the most common mistake. Understanding the difference between FromSqlRaw and FromSqlInterpolated is critical.

SQL — What Gets Sent to Database -- When using FromSqlInterpolated: -- EF generates: "SELECT * FROM Products WHERE Name = @p0" -- Parameter: { @p0 = userInput } -- Safe: parameter is data, not code -- When using FromSqlRaw($"...{input}..."): -- C# string interpolation happens first -- Result: "SELECT * FROM Products WHERE Name = " + input -- If input = "'; DROP TABLE Products; --" // Result: SELECT * FROM Products WHERE Name = ''; DROP TABLE Products; --' // Multiple statements! -- When using FromSqlRaw("...", params): -- EF generates: "SELECT * FROM Products WHERE Name = @p0" // Parameter: { @p0 = userInput } // Safe: parameters provided explicitly
C# — EF Core SQL Safety Patterns public class ProductService { private readonly ProductContext _context; // ✓ SAFE: LINQ (automatic parameterization) public async Task> GetByNameLinqAsync(string name) { return await _context.Products .Where(p => p.Name == name) .ToListAsync(); // EF automatically parameterizes the WHERE clause // No SQL injection possible } // ✓ SAFE: FromSqlInterpolated public async Task> GetByNameInterpolatedAsync(string name) { return await _context.Products .FromSqlInterpolated($@" SELECT * FROM Products WHERE Name = {name}") .ToListAsync(); // FromSqlInterpolated converts the interpolation to parameters // Behind the scenes: SELECT * FROM Products WHERE Name = @p0 // Parameter { @p0 = name } } // ✓ SAFE: FromSqlRaw with explicit parameters public async Task> GetByNameRawParametersAsync(string name) { return await _context.Products .FromSqlRaw( "SELECT * FROM Products WHERE Name = @name", new SqlParameter("@name", name)) .ToListAsync(); } // ✗ DANGEROUS: FromSqlRaw with string interpolation public async Task> GetByNameRawInterpolationAsync(string name) { // DO NOT DO THIS! return await _context.Products .FromSqlRaw($"SELECT * FROM Products WHERE Name = {name}") .ToListAsync(); // String interpolation happens in C# BEFORE FromSqlRaw // Result is concatenation, not parameterization // Vulnerable to SQL injection! } // ✗ DANGEROUS: String concatenation with ExecuteSqlRaw public async Task ExecuteUpdateVulnerableAsync( int productId, string newName) { // DO NOT DO THIS! string sql = $"UPDATE Products SET Name = '{newName}' WHERE Id = {productId}"; return await _context.Database.ExecuteSqlRawAsync(sql); // Pure string concatenation = vulnerable! } // ✓ SAFE: ExecuteSqlInterpolated public async Task ExecuteUpdateSafeAsync( int productId, string newName) { return await _context.Database.ExecuteSqlInterpolatedAsync( $"UPDATE Products SET Name = {newName} WHERE Id = {productId}"); // Parameterized automatically } // ✓ SAFE: ExecuteSqlRaw with parameters public async Task ExecuteUpdateSafeAltAsync( int productId, string newName) { return await _context.Database.ExecuteSqlRawAsync( "UPDATE Products SET Name = @name WHERE Id = @id", new SqlParameter("@name", newName), new SqlParameter("@id", productId)); } // Real-world example: Complex filtering public async Task> GetOrdersComplexAsync( DateTime startDate, DateTime endDate, string status, int? customerId) { // Safe: All parameters are automatic with LINQ var query = _context.Orders .Where(o => o.OrderDate >= startDate && o.OrderDate <= endDate && o.Status == status); if (customerId.HasValue) query = query.Where(o => o.CustomerId == customerId.Value); return await query.ToListAsync(); // Even complex queries are safe with LINQ // EF parameterizes every condition automatically } }
05 — Stored Procedures as a Security Layer

Defense in Depth: Application + Database Security

While parameterized queries at the application level prevent SQL injection, stored procedures add a second layer of defense. When using stored procedures, the application doesn't send raw SQL at all—only the procedure name and parameters. The database has full control over what SQL executes. This also enables you to grant database permissions to the application user at the procedure level, denying direct table access.

SQL — Stored Procedures for Security -- Create stored procedure with parameterized query CREATE PROCEDURE sp_GetOrdersByCustomer @CustomerId INT, @StartDate DATETIME AS BEGIN SELECT * FROM Orders WHERE CustomerId = @CustomerId AND OrderDate >= @StartDate ORDER BY OrderDate DESC; END; -- Grant procedure execute permission (not table access) GRANT EXECUTE ON sp_GetOrdersByCustomer TO AppUser; -- DENY direct table access DENY SELECT, INSERT, UPDATE, DELETE ON Orders TO AppUser; -- Now: // 1. App can call the procedure (EXECUTE permission) // 2. App cannot query Orders table directly (DENY SELECT) // 3. Attack must go through procedure (parameterized) // 4. Even if SQL injection happens in C# code, // the application user can't bypass the procedure
C# — Calling Stored Procedures from C# // PATTERN 1: Dapper (simple) public class OrderRepository { private readonly IDbConnection _connection; public async Task> GetOrdersByCustomer( int customerId, DateTime startDate) { const string procedureName = "sp_GetOrdersByCustomer"; var orders = await _connection.QueryAsync( procedureName, new { CustomerId = customerId, StartDate = startDate }, commandType: CommandType.StoredProcedure); return orders.ToList(); // CommandType.StoredProcedure tells ADO.NET: // - This is a procedure, not raw SQL // - Always parameterized (can't be SQL injection) } } // PATTERN 2: EF Core with FromSqlRaw (special syntax) public class OrderRepositoryEF { private readonly OrderContext _context; public async Task> GetOrdersByCustomer( int customerId, DateTime startDate) { return await _context.Orders .FromSqlRaw( "EXECUTE sp_GetOrdersByCustomer @CustomerId, @StartDate", new SqlParameter("@CustomerId", customerId), new SqlParameter("@StartDate", startDate)) .ToListAsync(); // EXECUTE sp_... with parameters is safe // Parameters are always respected by SQL Server } } // PATTERN 3: Raw ADO.NET with SqlCommand public class OrderRepositoryADO { private readonly string _connectionString; public async Task> GetOrdersByCustomer( int customerId, DateTime startDate) { var orders = new List(); using (var connection = new SqlConnection(_connectionString)) using (var command = new SqlCommand( "sp_GetOrdersByCustomer", connection)) { command.CommandType = CommandType.StoredProcedure; command.Parameters.AddWithValue("@CustomerId", customerId); command.Parameters.AddWithValue("@StartDate", startDate); await connection.OpenAsync(); using (var reader = await command.ExecuteReaderAsync()) { while (await reader.ReadAsync()) { orders.Add(new Order { Id = (int)reader["Id"], CustomerId = (int)reader["CustomerId"], OrderDate = (DateTime)reader["OrderDate"] }); } } } return orders; } } // Security benefits of stored procedures: // 1. Application never sends raw SQL // 2. Database can restrict procedure-only access // 3. Stored procedure does the querying (parameterized SQL) // 4. Defense in depth: multiple layers of protection
06 — Beyond SQL Injection: Connection String Security

Protecting Credentials and Secrets

While SQL injection is the primary concern, connection strings themselves are sensitive: they contain database server addresses, usernames, and sometimes passwords. Never hardcode connection strings in your code. Use environment variables, configuration files excluded from source control, or secret management services like Azure Key Vault.

SQL — Connection String Components -- Connection string example: Server=myserver.database.windows.net; Database=mydb; User Id=appuser; Password=SuperSecretPassword123!; Encrypt=true; TrustServerCertificate=false; Connection Timeout=30; -- Sensitive components: // - Server: database location (could be exploited) // - Database: schema information leak // - User Id: application database user (principle of least privilege) // - Password: absolutely must not be hardcoded! -- Best practice: // 1. Use Azure SQL authentication (no passwords) // 2. Use managed identities (app automatically authenticated) // 3. Use environment-specific connection strings // 4. Never commit passwords to source control
C# — Secure Connection String Management // ══════════════════════════════════════════════════ // WRONG: Hardcoded in code // ══════════════════════════════════════════════════ public class OrderContext : DbContext { protected override void OnConfiguring( DbContextOptionsBuilder optionsBuilder) { // NEVER DO THIS! optionsBuilder.UseSqlServer( "Server=myserver.database.windows.net;" + "Database=mydb;" + "User Id=appuser;" + "Password=SuperSecretPassword123!;"); } } // Password visible in source code - SECURITY DISASTER! // ══════════════════════════════════════════════════ // RIGHT: From appsettings.json (not in source control) // ══════════════════════════════════════════════════ public class OrderContext : DbContext { private readonly string _connectionString; public OrderContext(IConfiguration config) { _connectionString = config.GetConnectionString("DefaultConnection"); // appsettings.json has connection string // appsettings.json is in .gitignore (not committed) } protected override void OnConfiguring( DbContextOptionsBuilder optionsBuilder) { optionsBuilder.UseSqlServer(_connectionString); } } // appsettings.json (committed to repo, no secrets): { "ConnectionStrings": { "DefaultConnection": "Server=localhost;Database=mydb;" } } // appsettings.Production.json (NOT committed, has secrets): { "ConnectionStrings": { "DefaultConnection": "Server=prod-server;Database=mydb;User Id=appuser;Password=..." } } // ══════════════════════════════════════════════════ // BETTER: User Secrets for development // ══════════════════════════════════════════════════ // dotnet user-secrets init // dotnet user-secrets set "ConnectionStrings:DefaultConnection" "..." // Stored in ~/.microsoft/usersecrets (not in project) // Loaded automatically in development // ══════════════════════════════════════════════════ // BEST: Azure Key Vault for production // ══════════════════════════════════════════════════ public class OrderContext : DbContext { private readonly string _connectionString; public OrderContext(IConfiguration config) { // Retrieve connection string from Key Vault _connectionString = config["ConnectionStrings:DefaultConnection"]; // Accessed via Azure.Identity managed identity // No passwords in code or config files! } } // Startup configuration: public static IHostBuilder CreateHostBuilder(string[] args) => Host.CreateDefaultBuilder(args) .ConfigureAppConfiguration((context, config) => { var keyVaultUrl = "https://myvault.vault.azure.net/"; var credential = new DefaultAzureCredential(); config.AddAzureKeyVault( new Uri(keyVaultUrl), credential); }) .ConfigureServices((context, services) => { services.AddDbContext(); }); // ══════════════════════════════════════════════════ // Environment variables (for containers) // ══════════════════════════════════════════════════ var connectionString = Environment.GetEnvironmentVariable( "ConnectionStrings__DefaultConnection") ?? throw new InvalidOperationException("Connection string not configured"); var optionsBuilder = new DbContextOptionsBuilder(); optionsBuilder.UseSqlServer(connectionString); // Docker or CI/CD sets environment variable (not visible in logs) // Container runtime: docker run -e ConnectionStrings__DefaultConnection=... myapp // SUMMARY: // ✓ Development: User Secrets (dotnet user-secrets) // ✓ Staging/Production: Azure Key Vault or environment variables // ✗ Never: hardcoded passwords in source code // ✗ Never: committed appsettings with production secrets
Key Insight: Defense in Depth

Security requires multiple layers: parameterized queries prevent injection, stored procedures add application control, proper permissions limit damage, and secret management protects credentials. No single measure is foolproof. Combine all of them for a secure system.

Key Takeaways

  • SQL injection occurs when user input is concatenated into SQL strings; the database can't distinguish code from data
  • Parameterized queries are the universal fix: separate SQL code from data parameters, preventing injection attacks
  • LINQ in EF Core is automatically safe; parameterization happens behind the scenes
  • FromSqlInterpolated is safe; FromSqlRaw with string interpolation is dangerous—always use explicit parameters with FromSqlRaw
  • Dangerous patterns include string concatenation, dynamic table/column names (use whitelists), and relying on "sanitization"
  • Stored procedures add defense-in-depth: procedures are always parameterized, and database permissions can restrict direct table access
  • Connection strings contain sensitive data—never hardcode passwords; use environment variables, user secrets, or Azure Key Vault
