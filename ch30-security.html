Chapter 30

Security

Prevent SQL injection, secure connection strings, validate inputs, and follow defense-in-depth.

In This Chapter

  • SQL injection attacks and prevention
  • Parameterized queries
  • Secure connection strings
  • Input validation
  • Least privilege principle
01

SQL Injection

SQL injection occurs when untrusted input is concatenated into SQL queries. Use parameterized queries always.

VULNERABLE: String Concatenation string email = GetUserInput(); // "alice' OR '1'='1" string sql = $"SELECT * FROM Users WHERE Email = '{email}'"; // Executed as: ... WHERE Email = 'alice' OR '1'='1' // Returns ALL users!
SAFE: Parameterized Query string email = GetUserInput(); var user = await db.Users .AsNoTracking() .FromSqlInterpolated( $"SELECT * FROM Users WHERE Email = {email}") .FirstOrDefaultAsync(); // Parameter is passed separately, never interpreted as SQL
Golden Rule:

Never concatenate user input into SQL. Always use parameterized queries or LINQ.

02

Safe Query Methods

Different approaches, all safe when done correctly.

LINQ (Safest) var email = userInput; // "alice@example.com" var user = await db.Users .AsNoTracking() .Where(u => u.Email == email) .FirstOrDefaultAsync(); // LINQ translates to parameterized SQL automatically
FromSqlInterpolated (Safe) var result = await db.Users .AsNoTracking() .FromSqlInterpolated( $"SELECT * FROM Users WHERE Email = {email}") .ToListAsync(); // $ interpolation parameterizes {email} automatically
FromSqlRaw (Dangerous!) // NEVER DO THIS: string sql = "SELECT * FROM Users WHERE Email = '" + email + "'"; var users = db.Users.FromSqlRaw(sql).ToList();
FromSqlRaw (Correct) // Do this instead: var users = await db.Users .AsNoTracking() .FromSqlRaw("SELECT * FROM Users WHERE Email = @Email", new SqlParameter("@Email", email)) .ToListAsync(); // Parameter passed separately
Order of Safety:

LINQ > FromSqlInterpolated > FromSqlRaw with parameters > (never) string concatenation.

03

Connection String Security

Never hardcode credentials. Use configuration and secrets management.

NEVER: Hardcoded Secrets var connStr = "Server=myserver;User Id=admin;Password=Passw0rd123!"; var options = new DbContextOptionsBuilder<AppDbContext>() .UseSqlServer(connStr) .Options;
SAFE: Configuration // appsettings.json { "ConnectionStrings": { "DefaultConnection": "Server=myserver;User Id=admin;Password=???" } } // Startup services.AddDbContext<AppDbContext>(options => options.UseSqlServer( configuration.GetConnectionString("DefaultConnection"))); // Secrets injected at runtime (e.g., from Azure Key Vault)
Best Practice:

Use Azure Key Vault, AWS Secrets Manager, or HashiCorp Vault. Never commit secrets to source control.

04

Input Validation

Validate early. Even with parameterized queries, reject invalid input.

Validate Before Query public async Task<User> GetUserAsync(string email, CancellationToken cancellationToken = default) { // Validate input if (string.IsNullOrWhiteSpace(email)) throw new ArgumentException("Email required"); if (!email.Contains("@") || email.Length > 255) throw new ArgumentException("Invalid email format"); // Query return await db.Users .AsNoTracking() .FirstOrDefaultAsync(u => u.Email == email, cancellationToken); }
05

Least Privilege

Database user should have minimal required permissions.

Principle of Least Privilege -- Application user (limited permissions) CREATE USER AppUser WITH PASSWORD = '...'; GRANT SELECT, INSERT, UPDATE ON Products TO AppUser; GRANT SELECT, INSERT, UPDATE ON Orders TO AppUser; -- No DELETE, ALTER, CREATE permissions -- Admin user (full permissions, for migrations only) CREATE USER AdminUser WITH PASSWORD = '...'; GRANT ALL PRIVILEGES TO AdminUser; -- Connection string uses AppUser Server=...;User Id=AppUser;Password=...;
Defense in Depth:

Parameterized queries + input validation + proper DB permissions = secure system.
