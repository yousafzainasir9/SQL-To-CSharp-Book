Chapter 26

Choosing the Right Tool

EF Core, Dapper, and ADO.NET each have strengths. Choose based on your needs.

01

EF Core vs Dapper vs ADO.NET

Three main data access tools in .NET: ORM (EF Core), micro-ORM (Dapper), and raw SQL (ADO.NET).

Aspect EF Core Dapper ADO.NET
Abstraction Full ORM Micro-ORM None (raw SQL)
SQL Generated? Yes (automatic) No (you write it) No (you write it)
Relationships Full support Manual joins Manual joins
Performance Good (N+1 issues) Excellent (minimal overhead) Fastest (no abstraction)
Learning Curve Moderate Low Low
Best For CRUD, rapid development Queries, reporting Complex SQL, stored procedures
Hybrid Approach:

Most teams use EF Core for writes, Dapper/ADO.NET for complex reads. Use the best tool for each job.

02

EF Core: Full ORM

Generates SQL, handles relationships, tracks changes. Best for CRUD and rapid development.

EF Core Example
// Write
var customer = new Customer { Name = "Alice" };
db.Customers.Add(customer);
await db.SaveChangesAsync();

// Read with relationships
var customers = await db.Customers
    .AsNoTracking()
    .Include(c => c.Orders)
    .ThenInclude(o => o.Items)
    .Where(c => c.IsActive)
    .ToListAsync();

// Automatic SQL generation
// SELECT c.* FROM Customers c
// LEFT JOIN Orders o ON c.Id = o.CustomerId
// LEFT JOIN OrderItems oi ON o.Id = oi.OrderId
// WHERE c.IsActive
Pros:

Fast to code, handles relationships elegantly, automatic change tracking.

Cons:

Generated SQL can be inefficient; N+1 queries if not careful; learning curve for advanced scenarios.

03

Dapper: Micro-ORM

You write SQL; Dapper maps results to objects. Minimal overhead, excellent for reporting and complex queries.

Dapper Example
await using var conn = new SqlConnection(connString);

// Explicit SQL with parameters
var customers = await conn.QueryAsync<CustomerDto>(
    @"SELECT c.Id, c.Name, c.Email,
             COUNT(o.Id) AS OrderCount,
             SUM(oi.Price * oi.Quantity) AS TotalSpent
      FROM Customers c
      LEFT JOIN Orders o ON c.Id = o.CustomerId
      LEFT JOIN OrderItems oi ON o.Id = oi.OrderId
      WHERE c.IsActive = 1
      GROUP BY c.Id, c.Name, c.Email",
    new { }
);

// Insert
await conn.ExecuteAsync(
    "INSERT INTO Customers (Name, Email) VALUES (@Name, @Email)",
    new { Name = "Alice", Email = "alice@example.com" }
);
Pros:

Fast, lightweight, explicit SQL, perfect for complex queries and reporting.

Cons:

No relationship loading, no change tracking, must manage connections.

04

ADO.NET: Raw SQL

No abstraction. Complete control, maximum performance for edge cases.

ADO.NET Example
using var conn = new SqlConnection(connString);
await conn.OpenAsync();

using var cmd = new SqlCommand(
    "SELECT * FROM Customers WHERE IsActive = 1", conn);

using var reader = await cmd.ExecuteReaderAsync();
var customers = new List<Customer>();

while (await reader.ReadAsync())
{
    customers.Add(new Customer
    {
        Id = (int)reader["Id"],
        Name = (string)reader["Name"],
        Email = (string)reader["Email"]
    });
}

// Explicit transaction control
using var transaction = conn.BeginTransaction();
try
{
    await cmd.ExecuteNonQueryAsync();
    await transaction.CommitAsync();
}
catch
{
    await transaction.RollbackAsync();
    throw;
}
When to Use:

Stored procedures, extreme performance requirements, legacy systems. Rarely needed in modern apps.

05

Decision Framework

Choose based on query complexity, performance requirements, and team experience.

Scenario Recommendation
Standard CRUD on a table EF Core. Fast to code, relationships work.
Complex join with aggregation Dapper. Write explicit SQL for clarity and control.
Real-time reporting/dashboards Dapper + read-only replica. Explicit SQL for speed.
Bulk inserts/updates Dapper or ADO.NET with batch operations.
API with read/write split EF Core for writes, Dapper for complex reads.
Stored procedure calls ADO.NET or EF Core .FromSqlInterpolated().
06

Hybrid Approach (Best Practice)

Combine EF Core (writes) and Dapper (reads) for optimal performance and code clarity.

Service Using Both
public class OrderService
{
    private readonly DbContext _db;
    private readonly IDbConnection _readConn;

    // Write operation with EF Core
    public async Task CreateOrderAsync(CreateOrderCommand cmd)
    {
        var order = new Order { CustomerId = cmd.CustomerId };
        _db.Orders.Add(order);

        foreach (var item in cmd.Items)
        {
            order.Items.Add(new OrderItem
            {
                ProductId = item.ProductId,
                Quantity = item.Quantity
            });
        }

        await _db.SaveChangesAsync();
    }

    // Complex read with Dapper
    public async Task<OrderReportDto> GetOrderReportAsync(int customerId, CancellationToken cancellationToken = default)
    {
        await using var conn = new SqlConnection(_readConn.ConnectionString);

        var report = await conn.QueryFirstOrDefaultAsync<OrderReportDto>(
            @"SELECT c.Id, c.Name,
                     COUNT(DISTINCT o.Id) AS OrderCount,
                     SUM(oi.Price * oi.Quantity) AS TotalSpent,
                     MAX(o.CreatedAt) AS LastOrder
              FROM Customers c
              LEFT JOIN Orders o ON c.Id = o.CustomerId
              LEFT JOIN OrderItems oi ON o.Id = oi.OrderId
              WHERE c.Id = @CustomerId
              GROUP BY c.Id, c.Name",
            new { CustomerId = customerId }
        );

        return report;
    }
}
Best of Both Worlds:

EF's simplicity for writes, Dapper's control for reads. Most scalable .NET data access pattern.

Key Takeaways

  • EF Core: Full ORM, generates SQL, great for CRUD and rapid development.
  • Dapper: Micro-ORM, explicit SQL, perfect for complex queries and reporting.
  • ADO.NET: Raw SQL, rare in modern apps, maximum control.
  • Hybrid approach: EF Core for writes, Dapper for complex reads.
  • Always parameterize queries to prevent SQL injection.
  • Profile and measure; don't optimize prematurely.
