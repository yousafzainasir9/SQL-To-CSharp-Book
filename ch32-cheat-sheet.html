01 — CRUD Operations Cheat Sheet

Create, Read, Update, Delete Patterns

The four fundamental database operations: Create (INSERT), Read (SELECT), Update (UPDATE), and Delete (DELETE). This table shows how to perform each in raw SQL, EF Core, and Dapper. Understanding these patterns is foundational to all data access work.

Operation Raw SQL EF Core Dapper
SELECT ALL SELECT * FROM Products context.Products.ToList() conn.Query("SELECT * FROM Products")
SELECT BY ID SELECT * FROM Products WHERE Id = @id context.Products.Find(id) conn.QueryFirstOrDefault("SELECT * FROM Products WHERE Id = @Id", new { Id = id })
SELECT WITH FILTER SELECT * FROM Products WHERE Price < @price context.Products.Where(p => p.Price < price).ToList() conn.Query("SELECT * FROM Products WHERE Price < @Price", new { Price = price })
INSERT INSERT INTO Products VALUES (@name, @price) context.Products.Add(product); context.SaveChanges() conn.Execute("INSERT INTO Products (Name, Price) VALUES (@Name, @Price)", product)
INSERT RETURNING ID INSERT ... OUTPUT INSERTED.Id context.Products.Add(product); context.SaveChanges(); return product.Id conn.QuerySingle("INSERT INTO Products ... RETURNING Id", product)
UPDATE UPDATE Products SET Name = @name WHERE Id = @id product.Name = "new"; context.SaveChanges() conn.Execute("UPDATE Products SET Name = @Name WHERE Id = @Id", product)
DELETE DELETE FROM Products WHERE Id = @id context.Products.Remove(product); context.SaveChanges() conn.Execute("DELETE FROM Products WHERE Id = @Id", new { Id = id })
BULK INSERT (1000+ rows) INSERT INTO ... SELECT ... (multi-row) context.Products.AddRange(products); context.SaveChanges() (Slow!) conn.Execute("INSERT ...", products) in loop, or use SqlBulkCopy
Key Insight: EF Change Tracking vs Explicit Updates

EF Core tracks changes automatically—fetch an entity, modify it in memory, and SaveChanges() generates the UPDATE. Dapper is explicit—you write the UPDATE SQL. This makes EF easier for simple scenarios but requires understanding change tracking behavior.

02 — Query Operations Translation

SQL Clauses to LINQ Methods Mapping

SQL and LINQ both express queries, but with different syntax. This reference maps SQL clauses to LINQ methods, showing how to express the same operations in both languages. Mastering this translation is essential for switching between raw SQL and LINQ.

SQL Clause LINQ Method Example

WHERE .Where() .Where(p => p.Price > 100)
ORDER BY .OrderBy() .OrderBy(p => p.Name)
ORDER BY DESC .OrderByDescending() .OrderByDescending(p => p.Price)
GROUP BY .GroupBy() .GroupBy(p => p.Category)
HAVING .Where() after GroupBy .GroupBy(...).Where(g => g.Count() > 5)
INNER JOIN .Join() .Join(ctx.Categories, p => p.CategoryId, c => c.Id, (p,c) => ...)
LEFT JOIN .GroupJoin() + .DefaultIfEmpty() .GroupJoin(...).SelectMany(g => g.DefaultIfEmpty())
DISTINCT .Distinct() .Select(p => p.Category).Distinct()
TOP / LIMIT .Take() .Take(10)
OFFSET .Skip() .Skip(20).Take(10)
EXISTS .Any() .Any(p => p.Price > 100)
IN (@list) .Contains() .Where(p => ids.Contains(p.Id))
COUNT(*) .Count() .Count() or .Count(p => p.Price > 100)
SUM() .Sum() .Sum(p => p.Price)
AVG() .Average() .Average(p => p.Price)
MIN() / MAX() .Min() / .Max() .Min(p => p.Price) or .Max(p => p.Price)

Key Insight: Some SQL Concepts Don't Translate

Not all SQL features have LINQ equivalents. Window functions (ROW_NUMBER, DENSE_RANK) have limited LINQ support. Set operations (UNION, INTERSECT) are available but cumbersome. For complex SQL, raw SQL with .FromSqlRaw() is clearer than forcing it into LINQ.

Key Insight: Fluent API Chains

The Fluent API uses method chaining. Each method returns an object allowing you to call the next method. This creates readable, declarative configurations. Break chains at logical points for readability: property config, then relationship config, then table config.

05 — Dapper Method Reference

Query, Execute, and QueryMultiple Methods

Dapper provides a small set of extension methods on IDbConnection. These six methods cover all data access scenarios. Understanding their return types and when to use each is essential for effective Dapper usage.

Method Returns Use For
Query IEnumerable Multiple rows, eager evaluation
QueryAsync Task> Multiple rows (async)
QueryFirst T First row, throws if empty
QueryFirstOrDefault T or null First row or null
QuerySingle T Exactly one row, throws otherwise
QuerySingleOrDefault T or null Exactly one row or null
Execute int (rows affected) INSERT, UPDATE, DELETE
ExecuteAsync Task INSERT, UPDATE, DELETE (async)
ExecuteScalar T Single value (COUNT, SUM, etc)
ExecuteScalarAsync Task Single value (async)
QueryMultiple GridReader Multiple result sets in one call
QueryMultipleAsync Task Multiple result sets (async)
Key Insight: Query vs QueryFirst Distinction

Query() returns IEnumerable and is lazy—the database isn't queried until you iterate. QueryFirst() executes immediately and returns the first row. Use Query() for filtering/projecting in LINQ; use QueryFirst() when you know you want only one row.

06 — Common Patterns Quick Reference

Repository, Unit of Work, CQRS, and Bulk Operations

Beyond CRUD, architectural patterns guide how you organize data access code. This table shows the four most common patterns, when to use them, and which tools fit each best.

Pattern When to Use Best Tool
Repository
Abstract data access behind interfaces		 Always. Decouples domain logic from data access. Easier testing. EF Core or Dapper. Both work with Repository pattern.
Unit of Work
Atomic transactions across multiple aggregates		 When multiple entities must change together, or changes must succeed/fail as a unit. EF Core has built-in UoW (DbContext). Dapper needs manual transaction management.
CQRS
Separate read and write paths		 When reads and writes have different performance requirements, or different models fit each. EF Core for writes (change tracking). Dapper for reads (no overhead).
Bulk Insert
Insert 1000+ rows efficiently		 Data imports, batch operations, ETL pipelines. SqlBulkCopy (fastest). Dapper batch (good balance). EF SaveChanges on collections (slowest).
Specification
Encapsulate query logic in objects		 Complex filtering logic that repeats. Domain-driven design. EF Core with LINQ. Build Specification object that returns IQueryable.
Cache-Aside
Query cache before database		 Frequently read, rarely changed data (reference data, configs). Either EF or Dapper. Cache layer is independent of data access tool.
Eventual Consistency
Multiple databases, async updates		 Microservices, event sourcing, distributed systems. Either. Combine with message queues (RabbitMQ, Azure Service Bus).
Pagination
Load large result sets in chunks		 Always. Never load 1M rows at once. EF: .Skip(pageSize * (pageNum-1)).Take(pageSize)
Dapper: Same SQL pattern
Key Insight: CQRS is Not Overengineering

Many projects benefit from CQRS without being "enterprise." Use EF Core for writes (which benefit from change tracking and validations) and Dapper for reads (which are often complex projections). The tools fit naturally into their domains.

07 — LINQ Query Syntax vs Method Syntax

from/where/select Style Alongside Method Chaining

LINQ supports two syntaxes: fluent method syntax (familiar to LINQ veterans) and query syntax (resembles SQL). Both compile to the same code, but query syntax is often more readable for complex filtering and joining. This section shows how to write the same query in both styles.

Scenario Method Syntax Query Syntax
Simple Filter ctx.Products.Where(p => p.Price > 100).ToList() from p in ctx.Products where p.Price > 100 select p
Filter + Project ctx.Products.Where(p => p.Active).Select(p => p.Name).ToList() from p in ctx.Products where p.Active select p.Name
Order + Take ctx.Products.OrderByDescending(p => p.Price).Take(5).ToList() from p in ctx.Products orderby p.Price descending select p) into top5 select top5
Inner Join ctx.Products.Join(ctx.Categories, p => p.CategoryId, c => c.Id, (p, c) => new { p.Name, c.Title }) from p in ctx.Products join c in ctx.Categories on p.CategoryId equals c.Id select new { p.Name, c.Title }
Left Join (DefaultIfEmpty) ctx.Products.GroupJoin(ctx.Categories, p => p.CategoryId, c => c.Id, (p, cs) => new { p, cs }).SelectMany(x => x.cs.DefaultIfEmpty(), (p, c) => new { p.Name, c.Title }) from p in ctx.Products join c in ctx.Categories on p.CategoryId equals c.Id into cs from c2 in cs.DefaultIfEmpty() select new { p.Name, c2.Title }
Group By Count ctx.Products.GroupBy(p => p.Category).Select(g => new { Category = g.Key, Count = g.Count() }).ToList() from p in ctx.Products group p by p.Category into g select new { Category = g.Key, Count = g.Count() }
Multiple Conditions ctx.Products.Where(p => p.Active && p.Price > 50 && p.Stock > 0).ToList() from p in ctx.Products where p.Active && p.Price > 50 && p.Stock > 0 select p
Key Insight: Query Syntax Shines with Joins

Query syntax (from/join/select) reads more naturally for complex joins than method syntax. However, query syntax is just syntactic sugar—the compiler converts it to method calls. Use whichever is more readable for your team. If using a tool like Entity Framework, both generate the same SQL.

09 — Transaction Patterns Quick Reference

Managing Atomicity and Isolation in Data Access

Transactions ensure that multiple operations succeed or fail as a unit. This reference shows transaction patterns across EF Core and Dapper, plus common isolation levels and when to use each.

Pattern EF Core Dapper Use Case
Implicit Transaction (SaveChanges) context.Products.Add(p); context.SaveChanges(); N/A (explicit required) Single operation, EF handles atomicity
Explicit Transaction Scope using (var txn = context.Database.BeginTransaction()) { /* ops */ txn.Commit(); } using (var txn = conn.BeginTransaction()) { /* ops */ txn.Commit(); } Multiple operations must succeed/fail together
Async Transaction using (var txn = await context.Database.BeginTransactionAsync()) { /* ops */ await txn.CommitAsync(); } using (var txn = conn.BeginTransaction()) { /* async ops */ txn.Commit(); } Non-blocking transactions in async methods
Isolation Level: Serializable context.Database.BeginTransaction(System.Data.IsolationLevel.Serializable) conn.BeginTransaction(System.Data.IsolationLevel.Serializable) Strongest isolation, slowest. For critical operations.
Isolation Level: RepeatableRead BeginTransaction(System.Data.IsolationLevel.RepeatableRead) BeginTransaction(System.Data.IsolationLevel.RepeatableRead) Prevent dirty reads and lost updates
Isolation Level: ReadCommitted BeginTransaction(System.Data.IsolationLevel.ReadCommitted) (default) BeginTransaction(System.Data.IsolationLevel.ReadCommitted) (default) Default. Prevents dirty reads.
Isolation Level: ReadUncommitted BeginTransaction(System.Data.IsolationLevel.ReadUncommitted) BeginTransaction(System.Data.IsolationLevel.ReadUncommitted) Fastest, allows dirty reads. For analytics only.
Savepoints (Nested Rollback) txn.CreateSavepoint("sp1"); /* ops */ txn.RollbackToSavepoint("sp1"); conn.Execute("SAVE TRANSACTION sp1"); /* ops */ conn.Execute("ROLLBACK TRANSACTION sp1"); Rollback part of a transaction, keep rest
Key Insight: Default Isolation Levels are Usually Fine

ReadCommitted (the default) prevents dirty reads and works for 95% of applications. Serializable guarantees consistency but kills concurrency. RepeatableRead is the middle ground. Only change isolation levels if you've identified specific concurrency bugs.

10 — Common Gotchas and Pitfalls

Avoid These Data Access Mistakes

Even experienced developers fall into these traps. Understanding common pitfalls helps you avoid subtle bugs that appear under load or in production.

Gotcha What Happens Solution
Lazy Evaluation in LINQ
Modifying query after materialization		 var q = ctx.Products.Where(p => p.Active); (not executed yet) q = q.Where(p => p.Price > 100); (compiled as one query) Call .ToList() to materialize immediately if query references change
Change Tracking Overhead
EF Core tracks all fetched entities		 Fetching 100,000 rows with EF slows to a crawl; memory usage explodes Use .AsNoTracking() for read-only queries. Use Dapper for large result sets.
SELECT N+1
Query loop executes separate queries		 foreach (var p in products) { var cat = ctx.Categories.Find(p.CategoryId); } (1 + N queries) Use .Include(p => p.Category) to fetch related data in one query
Identity Column Not Returned
Insert doesn't return new ID		 ctx.Products.Add(p); ctx.SaveChanges(); Then p.Id is still 0 EF automatically populates p.Id if configured with .ValueGeneratedOnAdd()
Connection String Leaks
Sensitive data in code		 Connection string with password committed to GitHub Use appsettings.json, user secrets, or environment variables. Never hardcode.
DateTime.Now vs UTC
Timezone bugs		 Database stores UTC, C# uses local time; comparisons fail Always use DateTime.UtcNow in C#. Configure database columns as UTC.
Missing .ToList() on Async
Forgetting to materialize async queries		 var items = await ctx.Products.ToListAsync(); (correct) vs var items = ctx.Products.ToListAsync(); (wrong—returns Task, not list) Always await async methods. Use .ToListAsync(), .FirstOrDefaultAsync(), etc.
Null Reference in LINQ
Comparing null in query		 .Where(p => p.Category.Name == "Electronics") throws if Category is null Check for null: .Where(p => p.Category != null && p.Category.Name == "Electronics")
DbContext Scope Issues
Using DbContext outside its scope		 DbContext disposed in one method, used in another; throws "DbContext has been disposed" Always scope DbContext to the operation. Use dependency injection with AddScoped() or AddTransient().
Dapper Parameter Case Sensitivity
SQL @Name doesn't match object property Name		 conn.Execute("INSERT ... @name", new { Name = "test" }); fails if SQL uses @Name Dapper matches parameter names case-insensitively by default, but be explicit for clarity
Key Insight: Profile Before Optimizing

Don't guess about performance. Use Entity Framework's logging to see generated SQL. Use SQL Profiler to capture actual queries. Measure memory and CPU before and after changes. The biggest wins often come from fixing SELECT N+1, not micro-optimizations.

11 — Performance Tips Quick Reference

Optimization Strategies for Data Access Across Tools

Database performance is often the bottleneck in applications. This reference covers the highest-impact optimizations for EF Core and Dapper, ranked by impact and ease of implementation.

Optimization Impact Effort Details
Use Indexes
CREATE INDEX on frequently queried columns		 Very High (10-100x speedup) Low Index WHERE, ORDER BY, and JOIN columns. In EF Core: .HasIndex(p => p.CategoryId)
AsNoTracking for Reads
Disable change tracking on queries		 High Very Low EF: ctx.Products.AsNoTracking().ToList() for read-only queries. Skip this for objects you'll modify.
Use Dapper for Complex Reads
Skip ORM overhead when reading		 High Low For reports, analytics, or complex joins: Dapper is 2-5x faster than EF on large result sets
Avoid SELECT *
Project only needed columns		 Medium-High Low EF: .Select(p => new { p.Id, p.Name }) Dapper: List columns in SELECT
Batch Updates
Update multiple rows in one statement		 Very High (for bulk ops) Medium Use .ExecuteUpdate() in EF 7+ or SqlBulkCopy. Avoid looping and SaveChanges().
Enable IDENTITY_INSERT
Speed up bulk inserts when ID is provided		 High (for bulk only) Medium SqlBulkCopy with KeepIdentity = true
Use Parameterized Queries Always
Prevent SQL injection and aid caching		 Medium (caching) Very Low EF and Dapper use parameters by default. Avoid string concatenation in SQL.
Include Only Needed Related Data
Use .Include() selectively		 Medium Low EF: .Include(p => p.Category) only when needed. Avoid .Include() in loops.
Use Query Caching
Cache frequently accessed data		 High (for read-heavy) Medium MemoryCache or Redis for reference data, counts, or reports. Use cache invalidation strategy.
Monitor Query Plans
Use EXPLAIN or SQL Profiler		 High (prevents regression) Low-Medium SQL Profiler shows actual execution time. EXPLAIN shows plan before execution. Look for Table Scans.
Key Insight: The 80/20 Rule

80% of performance gains come from 20% of changes. Indexes, AsNoTracking(), and avoiding SELECT N+1 solve most problems. Micro-optimizations (like expression caching) have minimal impact unless you're in a hot path serving millions of requests.

12 — SQL vs LINQ vs Query Syntax: Additional Examples

More Complex Comparisons Across All Three Approaches

Here are two additional complex scenarios showing how SQL, LINQ method syntax, and LINQ query syntax express the same logic differently. These highlight when each approach shines.

Scenario Raw SQL LINQ Method Syntax LINQ Query Syntax
Top Products by Category
Get the 3 most expensive items per category		 SELECT TOP 3 p.Id, p.Name, p.Price, c.Name FROM Products p JOIN Categories c ON p.CategoryId = c.Id ORDER BY c.Name, p.Price DESC ctx.Products.Join(ctx.Categories, p => p.CategoryId, c => c.Id, (p, c) => new { p, c }).GroupBy(x => x.c.Id).SelectMany(g => g.OrderByDescending(x => x.p.Price).Take(3)).ToList() from p in ctx.Products join c in ctx.Categories on p.CategoryId equals c.Id group p by c.Id into g from top3 in (from x in g orderby x.Price descending select x).Take(3) select top3
Orders with Item Count
Return orders and count items per order, excluding cancelled orders		 SELECT o.Id, o.OrderDate, COUNT(oi.Id) as ItemCount FROM Orders o LEFT JOIN OrderItems oi ON o.Id = oi.OrderId WHERE o.Status != 'Cancelled' GROUP BY o.Id, o.OrderDate ctx.Orders.Where(o => o.Status != "Cancelled").GroupJoin(ctx.OrderItems, o => o.Id, oi => oi.OrderId, (o, items) => new { o.Id, o.OrderDate, ItemCount = items.Count() }).ToList() from o in ctx.Orders where o.Status != "Cancelled" join oi in ctx.OrderItems on o.Id equals oi.OrderId into items select new { o.Id, o.OrderDate, ItemCount = items.Count() }
Key Insight: Method Syntax Works Better for Intermediate Operations

When you need to build queries dynamically (applying filters conditionally), method syntax shines because each method returns IQueryable. Query syntax requires the entire query in one statement, making conditional construction harder. For complex aggregations, raw SQL often wins in clarity.

Nullable Handling in EF Core and Dapper

Nullable columns in SQL Server require corresponding nullable C# types or special handling:

// EF Core: Using nullable reference types
public class Product
{
  public int Id { get; set; }
  public string Name { get; set; } // NOT nullable
  public string? Description { get; set; } // Nullable (allows NULL)
  public int? CategoryId { get; set; } // Nullable foreign key
  public decimal Price { get; set; } // NOT nullable
  public decimal? DiscountPrice { get; set; } // Nullable decimal
}

// Dapper: Read from NULL-able column
var products = connection.Query<Product>("SELECT Id, Name, CategoryId FROM Products");
// If CategoryId is NULL in database, property gets null value
// CategoryId must be int? to handle NULL

// EF Core Configuration for NULL handling
modelBuilder.Entity<Product>()
  .Property(p => p.Description)
  .IsRequired(false); // Explicitly allows NULL

modelBuilder.Entity<Product>()
  .Property(p => p.CategoryId)
  .IsRequired(false); // Nullable foreign key

Key Takeaways

  • CRUD operations map across SQL, EF Core, and Dapper; understand the differences in how each handles inserts with identity return values
  • SQL clauses (WHERE, ORDER BY, GROUP BY) have direct LINQ equivalents (.Where, .OrderBy, .GroupBy); learn the mapping for fluent translation
  • SQL types map to C# types predictably; use nullable types (int?, DateTime?) when columns allow NULL
  • EF Core's Fluent API provides declarative configuration for keys, relationships, indexes, and constraints; master .HasKey(), .HasOne(), .HasMany()
  • Dapper methods are simple: Query() for multiple rows, QueryFirstOrDefault() for one, Execute() for modifications, QueryMultiple() for multiple result sets
  • Choose patterns based on your needs: Repository for abstraction, Unit of Work for transactions, CQRS for read/write separation, Bulk Insert for large imports
  • EF Core excels at writes with change tracking; Dapper excels at complex reads; the best architectures use both
