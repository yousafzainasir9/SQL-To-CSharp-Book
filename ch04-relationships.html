01

One-to-One Relationships

A one-to-one relationship means each row in table A relates to at most one row in table B, and vice versa. Examples: each Employee has one UserAccount; each Customer has one Address; each User has one Profile. In SQL, you implement 1:1 with either a shared primary key (less common) or a unique foreign key (more common). In C#, both sides have nullable or non-nullable navigation properties pointing to a single object.

One-to-one relationships are less common than one-to-many, but they solve specific problems. Sometimes you have a large table with optional data that you want to separate: Customer with OrderHistory, where OrderHistory is a separate table you only load when needed. Or you have a base table and derived tables: Employee and EmployeeContractDetails. This separation improves query performance and data organization.

SQL: Unique FK (Preferred) -- Customer table CREATE TABLE Customers ( Id INT PRIMARY KEY IDENTITY(1,1), Name NVARCHAR(100) NOT NULL ); -- Address with unique FK to Customer -- Each customer has exactly one address CREATE TABLE Addresses ( Id INT PRIMARY KEY IDENTITY(1,1), CustomerId INT NOT NULL UNIQUE, Street NVARCHAR(100), City NVARCHAR(50), Country NVARCHAR(50), FOREIGN KEY (CustomerId) REFERENCES Customers(Id) ON DELETE CASCADE ); -- Query: customer with their address SELECT c.Name, a.Street, a.City FROM Customers c LEFT JOIN Addresses a ON c.Id = a.CustomerId WHERE c.Id = 5;

Key Insight:

Use one-to-one relationships to separate optional large data or logically distinct concerns. The unique constraint ensures true 1:1 cardinality. Always decide which side holds the foreign key: the dependent table references the principal. Include both navigations (optional on the principal, non-nullable on the dependent) for natural access from both directions.

02

One-to-Many Relationships

One-to-many is the most common relationship type. One customer has many orders; one category has many products; one blog has many posts. In SQL, the child table has a foreign key column. In C#, the parent has an ICollection navigation, and the child has a single-object navigation back to the parent. This is the pattern you've seen most in this book so far.

Querying one-to-many relationships is where LINQ shines. You can filter parents by child properties, navigate collections with Include(), and project nested data into DTOs. This is also where the N+1 problem appears: loading 100 customers and then loading each customer's orders one-by-one results in 101 database queries. Using Include() or explicit JOINs solves this.

SQL: One-to-Many with JOINs -- Parent table CREATE TABLE Customers ( Id INT PRIMARY KEY IDENTITY(1,1), Name NVARCHAR(100) NOT NULL ); -- Child table with FK CREATE TABLE Orders ( Id INT PRIMARY KEY IDENTITY(1,1), CustomerId INT NOT NULL, OrderDate DATETIME2, Total DECIMAL(10,2), FOREIGN KEY (CustomerId) REFERENCES Customers(Id) ON DELETE CASCADE ); -- Query: customer with their orders SELECT c.Name, o.Id, o.OrderDate, o.Total FROM Customers c INNER JOIN Orders o ON c.Id = o.CustomerId WHERE c.Id = 5 ORDER BY o.OrderDate DESC; -- Query: customer even if they have no orders SELECT c.Name, o.Id, o.OrderDate, o.Total FROM Customers c LEFT JOIN Orders o ON c.Id = o.CustomerId ORDER BY c.Name, o.OrderDate DESC;
