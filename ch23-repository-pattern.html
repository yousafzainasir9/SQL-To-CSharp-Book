01

What is the Repository Pattern?

The Repository pattern is an abstraction layer that sits between your business logic and your data access logic. Think of it as a "data service" that encapsulates how you fetch, store, update, and delete data. Instead of your application directly querying the database, it asks the repository "give me the customer with ID 5" or "save this order," and the repository handles the details.

In SQL terms, this is analogous to creating Views or Stored Procedures. A view hides the complexity of joins and filters; a repository hides the complexity of database queries. Both are abstractions that make the underlying data access details invisible to the consumer.

Martin Fowler's Definition: In his seminal book Patterns of Enterprise Application Architecture, Fowler described the Repository pattern as "mediating between the domain and mapping layers, acting like an in-memory collection of entities." The key idea: repositories should feel like collections you're operating on, not like database access code. You shouldn't think about SQL or EF Core—you should think about domain objects.

The Core Idea
A repository is a contract (interface) that says "I can do these data operations." The actual implementation details—whether you use EF Core, Dapper, raw SQL, or even in-memory lists—are hidden from the caller. This means you can swap implementations without breaking your business logic.
02

SQL Views and Stored Procedures as Repositories

Before diving into C# repositories, let's understand the concept at the database level. In SQL, the same abstraction principle exists: you define views and stored procedures to hide query complexity.

A database view is like a repository's "query method"—it encapsulates complex joins and filters:

SQL - Complex Query (Raw)
SELECT c.Id, c.Name, c.Email, COUNT(o.Id) AS OrderCount, SUM(o.Total) AS TotalSpent FROM Customers c LEFT JOIN Orders o ON c.Id = o.CustomerId WHERE c.IsActive = 1 GROUP BY c.Id, c.Name, c.Email ORDER BY TotalSpent DESC;
SQL - Same Logic Encapsulated as View
CREATE VIEW vw_ActiveCustomerStats AS SELECT c.Id, c.Name, c.Email, COUNT(o.Id) AS OrderCount, SUM(o.Total) AS TotalSpent FROM Customers c LEFT JOIN Orders o ON c.Id = o.CustomerId WHERE c.IsActive = 1 GROUP BY c.Id, c.Name, c.Email; -- Now any consumer uses the abstraction: SELECT * FROM vw_ActiveCustomerStats ORDER BY TotalSpent DESC;

The same principle applies in C#. A repository method "GetActiveCustomersWithStats()" is the OOP equivalent of the view. Both hide complexity and present a simpler interface.

Database vs. Application Repositories
SQL repositories (views/stored procedures) operate at the database level and return data. C# repositories operate at the application level and return domain objects. The concepts are identical: both abstract away complexity. In legacy systems with mature SQL code, you might use stored procedures directly from C#. In modern systems, C# repositories wrap EF Core or Dapper.
03

Generic Repository with EF Core

A generic repository is a single repository class that works with any entity type. It provides standard CRUD operations (Create, Read, Update, Delete) that apply to all entities. Let's start with the interface:

C# - Generic Repository Interface
public interface IRepository<T> where T : class
{
    // Read operations
    Task<T?> GetByIdAsync(int id);
    Task<IEnumerable<T>> GetAllAsync();
    Task<IEnumerable<T>> FindAsync(
        Expression<Func<T, bool>> predicate);

    // Write operations
    Task AddAsync(T entity);
    Task AddRangeAsync(IEnumerable<T> entities);
    void Update(T entity);
    void Delete(T entity);
    void DeleteRange(IEnumerable<T> entities);

    // Persistence
    Task<int> SaveChangesAsync();
}

Now, here's the EF Core implementation using DbContext:

C# - EF Core Generic Repository Implementation
public class Repository<T> : IRepository<T>
    where T : class
{
    private readonly DbContext _context;
    private readonly DbSet<T> _dbSet;

    public Repository(DbContext context)
    {
        _context = context;
        _dbSet = context.Set<T>();
    }

    public async Task<T?> GetByIdAsync(int id)
    {
        return await _dbSet.FindAsync(id);
    }

    public async Task<IEnumerable<T>> GetAllAsync()
    {
        return await _dbSet.ToListAsync();
    }

    public async Task<IEnumerable<T>> FindAsync(
        Expression<Func<T, bool>> predicate)
    {
        return await _dbSet.Where(predicate).ToListAsync();
    }

    public async Task AddAsync(T entity)
    {
        await _dbSet.AddAsync(entity);
    }

    public async Task AddRangeAsync(IEnumerable<T> entities)
    {
        await _dbSet.AddRangeAsync(entities);
    }

    public void Update(T entity)
    {
        _dbSet.Update(entity);
    }

    public void Delete(T entity)
    {
        _dbSet.Remove(entity);
    }

    public void DeleteRange(IEnumerable<T> entities)
    {
        _dbSet.RemoveRange(entities);
    }

    public async Task<int> SaveChangesAsync()
    {
        return await _context.SaveChangesAsync();
    }
}

Usage is straightforward. You inject the generic repository and use it with any entity:

C# - Using Generic Repository
public class OrderService
{
    private readonly IRepository<Order> _orderRepo;

    public OrderService(IRepository<Order> orderRepo)
    {
        _orderRepo = orderRepo;
    }

    public async Task PlaceOrderAsync(Order order)
    {
        await _orderRepo.AddAsync(order);
        await _orderRepo.SaveChangesAsync();
    }

    public async Task<Order?> GetOrderAsync(int orderId)
    {
        return await _orderRepo.GetByIdAsync(orderId);
    }
}
Dependency Injection Setup
In your DI container (Startup.cs or Program.cs), register the generic repository like this: services.AddScoped(typeof(IRepository<>), typeof(Repository<>)); This allows you to inject IRepository<Customer>, IRepository<Order>, etc., automatically.

Generic Repository with Dapper

The same interface works with Dapper, but the implementation is quite different because Dapper requires explicit SQL queries. This shows the power of the repository pattern—the interface is identical, but the implementation changes:

C# - Dapper Generic Repository Implementation
public class DapperRepository<T> : IRepository<T>
    where T : class
{
    private readonly IDbConnection _connection;
    private readonly string _tableName;

    public DapperRepository(IDbConnection connection)
    {
        _connection = connection;
        // Assumes table name matches entity name
        _tableName = typeof(T).Name + "s";
    }

    public async Task<T?> GetByIdAsync(int id)
    {
        var sql = $"SELECT * FROM {_tableName} WHERE Id = @Id";
        return await _connection.QueryFirstOrDefaultAsync<T>(
            sql,
            new { Id = id }
        );
    }

    public async Task<IEnumerable<T>> GetAllAsync()
    {
        var sql = $"SELECT * FROM {_tableName}";
        return await _connection.QueryAsync<T>(sql);
    }

    public async Task AddAsync(T entity)
    {
        var insertSql = @"INSERT INTO " + _tableName +
            " (Name, Email) VALUES (@Name, @Email)";
        await _connection.ExecuteAsync(insertSql, entity);
    }

    public void Update(T entity)
    {
        var updateSql = @"UPDATE " + _tableName +
            " SET Name = @Name WHERE Id = @Id";
        _connection.Execute(updateSql, entity);
    }

    public void Delete(T entity)
    {
        var deleteSql = $"DELETE FROM {_tableName} WHERE Id = @Id";
        _connection.Execute(deleteSql, entity);
    }

    // Other methods omitted for brevity
}

Notice how Dapper's implementation is more verbose because we need explicit SQL. With EF Core, the framework generated the SQL for us. This is a key difference: EF Core abstracts SQL away, but you lose control. Dapper lets you write SQL explicitly, so you gain control but write more code.

Specific Repositories (Better Approach)

While generic repositories seem elegant, they often hide important business logic. A specific repository is designed for a single entity type and includes domain-specific methods that express business intent. Consider:

C# - Specific Customer Repository Interface
public interface ICustomerRepository
{
    // Generic CRUD
    Task<Customer?> GetByIdAsync(int id);
    Task AddAsync(Customer customer);
    Task SaveAsync();

    // Domain-specific queries
    Task<IEnumerable<Customer>> GetActiveCustomersAsync();
    Task<IEnumerable<Customer>> GetByCity(string city);
    Task<IEnumerable<Customer>> GetTopSpendersAsync(int count);
    Task<bool> IsEmailUnique(string email);
}

Notice how the method names tell a story: "GetActiveCustomers" immediately tells a developer what business rule is being applied. Here's the EF Core implementation:

C# - Specific Customer Repository with EF Core
public class CustomerRepository : ICustomerRepository
{
    private readonly DbContext _context;

    public CustomerRepository(DbContext context)
    {
        _context = context;
    }

    public async Task<Customer?> GetByIdAsync(int id)
    {
        return await _context.Customers.FindAsync(id);
    }

    public async Task<IEnumerable<Customer>> GetActiveCustomersAsync()
    {
        return await _context.Customers
            .Where(c => c.IsActive && c.LastOrderDate > DateTime.Now.AddMonths(-6))
            .OrderByDescending(c => c.LastOrderDate)
            .ToListAsync();
    }

    public async Task<IEnumerable<Customer>> GetByCity(string city)
    {
        return await _context.Customers
            .Where(c => c.Address.City == city)
            .OrderBy(c => c.Name)
            .ToListAsync();
    }

    public async Task<IEnumerable<Customer>> GetTopSpendersAsync(int count)
    {
        return await _context.Customers
            .Include(c => c.Orders)
            .OrderByDescending(c => c.Orders.Sum(o => o.Total))
            .Take(count)
            .ToListAsync();
    }

    public async Task<bool> IsEmailUnique(string email)
    {
        return await _context.Customers
            .AnyAsync(c => c.Email == email) == false;
    }

    public async Task AddAsync(Customer customer)
    {
        await _context.Customers.AddAsync(customer);
    }

    public async Task SaveAsync()
    {
        await _context.SaveChangesAsync();
    }
}
Why Specific Repositories Win
Specific repositories act as a ubiquitous language between your code and your business domain. When a developer reads "GetTopSpenders()", they immediately understand the business rule being applied. With a generic repository and LINQ scattered throughout your code, that intent is buried in query details. Specific repositories also make testing easier—you can mock the interface with domain-specific test data.

Repository with Pagination, Filtering, and Sorting

Real applications need to handle pagination (splitting results into pages), filtering (applying criteria), and sorting (ordering results). Let's extend the repository pattern to support these:

C# - Repository with Pagination and Filtering
public class PaginationParams
{
    public int PageNumber { get; set; } = 1;
    public int PageSize { get; set; } = 10;
    public string SortBy { get; set; } = "Id";
    public bool Ascending { get; set; } = true;
}

public class PagedResult<T>
{
    public List<T> Items { get; set; }
    public int TotalCount { get; set; }
    public int PageNumber { get; set; }
    public int PageSize { get; set; }
    public int TotalPages =>
        (TotalCount + PageSize - 1) / PageSize;
}

public interface ICustomerRepository
{
    Task<PagedResult<Customer>>
        GetCustomersPagedAsync(
            PaginationParams pagination,
            string searchTerm = "",
            CustomerFilter filter = null);
}

public class CustomerRepository : ICustomerRepository
{
    private readonly DbContext _context;

    public async Task<PagedResult<Customer>>
        GetCustomersPagedAsync(
            PaginationParams pagination,
            string searchTerm = "",
            CustomerFilter filter = null)
    {
        var query = _context.Customers.AsQueryable();

        // Apply filters
        if (!string.IsNullOrEmpty(searchTerm))
            query = query.Where(c =>
                c.Name.Contains(searchTerm) || c.Email.Contains(searchTerm));

        if (filter?.IsActive.HasValue == true)
            query = query.Where(c => c.IsActive == filter.IsActive.Value);

        // Total count before pagination
        var totalCount = await query.CountAsync();

        // Apply sorting
        query = pagination.Ascending
            ? query.OrderBy(c => EF.Property<object>(c, pagination.SortBy))
            : query.OrderByDescending(c =>
                EF.Property<object>(c, pagination.SortBy));

        // Apply pagination
        var items = await query
            .Skip((pagination.PageNumber - 1) * pagination.PageSize)
            .Take(pagination.PageSize)
            .ToListAsync();

        return new PagedResult<Customer>
        {
            Items = items,
            TotalCount = totalCount,
            PageNumber = pagination.PageNumber,
            PageSize = pagination.PageSize
        };
    }
}

Now let's compare the LINQ query syntax version:

C# - LINQ Query Syntax Alternative
public async Task<PagedResult<Customer>>
    GetCustomersPagedAsync(
        PaginationParams pagination,
        string searchTerm = "",
        CustomerFilter filter = null)
{
    // Query syntax version
    var query = from c in _context.Customers
                 where (string.IsNullOrEmpty(searchTerm) ||
                        c.Name.Contains(searchTerm) ||
                        c.Email.Contains(searchTerm))
                 where (filter?.IsActive == null ||
                        c.IsActive == filter.IsActive.Value)
                 orderby c.Name ascending
                 select c;

    var totalCount = await query.CountAsync();

    var items = await query
        .Skip((pagination.PageNumber - 1) * pagination.PageSize)
        .Take(pagination.PageSize)
        .ToListAsync();

    return new PagedResult<Customer> { ... };
}
Query Syntax vs Method Syntax
Both LINQ syntaxes produce identical SQL. Use query syntax when you have multiple where clauses and joins—it's often more readable. Use method syntax for simple queries or when combining complex operations. Both work seamlessly with repositories.

Repository with the Specification Pattern

For complex filtering logic, the Specification pattern pairs beautifully with repositories. A specification encapsulates a query predicate and related data (includes, ordering) in a reusable object:

C# - Specification Pattern
public abstract class Specification<T>
{
    public Expression<Func<T, bool>> Criteria { get; set; }
    public List<Expression<Func<T, object>>>
        Includes { get; } = new List<Expression<Func<T, object>>>();
    public Expression<Func<T, object>> OrderBy { get; set; }
}

// Concrete specification for active customers
public class ActiveCustomersSpec : Specification<Customer>
{
    public ActiveCustomersSpec(string city = null)
    {
        Criteria = c => c.IsActive;

        if (!string.IsNullOrEmpty(city))
            Criteria = c => c.IsActive && c.Address.City == city;

        Includes.Add(c => c.Orders);
        OrderBy = c => c.Name;
    }
}

// Repository accepts specs
public interface ISpecificationRepository<T>
{
    Task<IEnumerable<T>>
        GetAsync(Specification<T> spec);
}

public class SpecificationRepository<T> :
    ISpecificationRepository<T>
    where T : class
{
    private readonly DbContext _context;

    public async Task<IEnumerable<T>>
        GetAsync(Specification<T> spec)
    {
        var query = _context.Set<T>().AsQueryable();

        // Apply criteria
        if (spec.Criteria != null)
            query = query.Where(spec.Criteria);

        // Apply includes
        query = spec.Includes.Aggregate(query,
            (current, include) => current.Include(include));

        // Apply ordering
        if (spec.OrderBy != null)
            query = query.OrderBy(spec.OrderBy);

        return await query.ToListAsync();
    }
}

Usage is clean and expressive:

C# - Using Specifications
public class CustomerService
{
    private readonly ISpecificationRepository<Customer> _repo;

    public async Task GetActiveInSeattle()
    {
        // Create spec with business rules
        var spec = new ActiveCustomersSpec("Seattle");

        // Repository applies it
        var customers = await _repo.GetAsync(spec);
    }
}
Specifications Encapsulate Complexity
Specifications move complex filtering logic out of service classes and into reusable, testable objects. The repository doesn't need to know about business rules—it just applies the specification. This is especially powerful when the same criteria are used in multiple places.

Testing Repositories: Mocking and In-Memory Implementations

One of the biggest advantages of the repository pattern is testability. Because the interface is abstracted from the implementation, you can create mock or in-memory repositories for testing:

C# - In-Memory Test Repository
public class InMemoryCustomerRepository :
    ICustomerRepository
{
    private readonly List<Customer> _customers =
        new List<Customer>();

    public async Task<Customer?> GetByIdAsync(int id)
    {
        return await Task.FromResult(
            _customers.FirstOrDefault(c => c.Id == id));
    }

    public async Task<IEnumerable<Customer>>
        GetActiveCustomersAsync()
    {
        return await Task.FromResult(
            _customers.Where(c => c.IsActive).ToList());
    }

    public async Task AddAsync(Customer customer)
    {
        _customers.Add(customer);
        return await Task.CompletedTask;
    }

    public async Task SaveAsync()
    {
        return await Task.CompletedTask;
    }
}

// Unit test using in-memory repository
[Fact]
public async Task GetActiveCustomersAsync_ReturnsOnlyActive()
{
    // Arrange
    var repo = new InMemoryCustomerRepository();
    var customer1 = new Customer { Id = 1, IsActive = true };
    var customer2 = new Customer { Id = 2, IsActive = false };
    await repo.AddAsync(customer1);
    await repo.AddAsync(customer2);

    // Act
    var result = await repo.GetActiveCustomersAsync();

    // Assert
    Assert.Single(result);
    Assert.Equal(customer1.Id, result.First().Id);
}

Alternatively, use Moq for more complex scenarios:

C# - Mocking Repositories with Moq
[Fact]
public async Task ApplyDiscountAsync_CallsRepository()
{
    // Arrange
    var mockRepo = new Mock<ICustomerRepository>();
    var customer = new Customer
    {
        Id = 1,
        Email = "test@example.com",
        IsActive = true
    };

    mockRepo
        .Setup(r => r.GetByIdAsync(1))
        .ReturnsAsync(customer);

    var service = new CustomerService(mockRepo.Object);

    // Act
    await service.ApplyDiscountAsync(1);

    // Assert
    mockRepo.Verify(r => r.GetByIdAsync(1),
                    Times.Once);
    mockRepo.Verify(r => r.SaveAsync(),
                    Times.Once);
}
Testing Without a Database
In-memory repositories let you test business logic without touching the database, making tests fast and reliable. Mocks are useful when you want to verify that your service called the repository with specific parameters. Together, they eliminate the need for integration tests during development—unit tests are enough for the service layer.

The Repository Anti-Pattern: Leaky Abstractions

The Repository pattern can become problematic when it becomes a leaky abstraction—when implementation details seep through and break the abstraction. Here are common anti-patterns:

C# - ANTI-PATTERN: Exposing IQueryable
// BAD: Exposes EF Core's query provider
public interface IBadRepository<T>
{
    IQueryable<T> Query(); // Oops!
}

// Now the caller is writing LINQ against EF Core!
// They're tightly coupled to EF Core, defeating the purpose
var customers = await _repository
    .Query()
    .Where(c => c.IsActive)
    .ToListAsync();

// If you swap to Dapper, this code breaks!
// The abstraction is worthless now.

The problem: IQueryable is an EF Core implementation detail. By exposing it, you've lost the abstraction—callers are now writing EF LINQ directly. This means you can't swap implementations.

Another anti-pattern is making the repository too thin:

C# - ANTI-PATTERN: Repositories That Are Just CRUD
// BAD: Repository is just a thin wrapper
public class OrderService
{
    private readonly IRepository<Order> _orderRepo;

    public async Task ProcessOrderAsync(Order order)
    {
        await _orderRepo.AddAsync(order); // OK

        // But all the real logic is here in the service:
        foreach (var item in order.Items)
        {
            item.Discount = CalculateDiscount(item);
        }

        if (order.Total > 1000)
        {
            // Manually fetch related data
            var customer = await _orderRepo.GetByIdAsync(
                order.CustomerId);
            var rewards = await _rewardRepo
                .GetByCustomerAsync(customer.Id);
            customer.RewardPoints += 100;
        }

        await _orderRepo.SaveChangesAsync();
    }
}

Here, the repository is useless—it's just calling DbContext methods. The real logic is in the service. This is a sign that your repository should have domain-specific methods like ProcessOrderWithRewardsAsync().

When NOT to use the Repository pattern: In simple CRUD applications where you're just wrapping entity operations, the repository adds boilerplate without value. For example, if your entire application is "get user, update user, save," using DbContext directly might be simpler. Also, in CQRS architectures, you often don't use repositories at all—you use query handlers and command handlers instead.

Rule of Thumb
Use the Repository pattern when: (1) You have complex query logic that you want to encapsulate, (2) You might swap implementations, or (3) You want to express business intent in method names. Don't use it when your application is simple CRUD and the repository would just be thin wrappers around DbSet methods.

Deep Theory: The Cost of Abstraction

Every abstraction comes with a cost. Let's examine the trade-offs:

Benefits of Repository Pattern: (1) Swappable implementations—you can test with in-memory, change to Dapper later, (2) Domain language—method names express intent, (3) Centralized query logic—easier to optimize later, (4) Easier testing—no database required.

Costs of Repository Pattern: (1) Boilerplate—every method needs an interface and implementation, (2) Leaky abstractions—IQueryable exposes details anyway, (3) Unnecessary for simple apps—over-engineering for CRUD, (4) Hidden query cost—developers don't see SQL, may write N+1 queries unknowingly.

The real insight: a repository is a deliberate trade-off. In large systems with many developers and complex queries, repositories pay for themselves. In small systems, they're overhead. Be honest about whether you need the abstraction.

DbSet IS a Repository
This point bears repeating: EF Core's DbSet already implements the repository pattern. When you do `_context.Customers.Where(...).ToListAsync()`, you're using a repository. The question isn't "should I use repositories?" but "should I wrap this in a custom interface?" For simple CRUD, probably not. For complex business logic, yes.

Repository Pattern in Clean Architecture

In Clean Architecture, the repository pattern plays a crucial role in separating concerns across layers:

C# - Clean Architecture Repository Placement
// DOMAIN LAYER (no external dependencies)
// Contains business rules and entities
public class Customer
{
    public int Id { get; set; }
    public string Name { get; set; }
    public bool IsActive { get; set; }

    // Domain method - no data access
    public bool IsEligibleForDiscount()
    {
        return IsActive && TotalOrders > 10;
    }
}

// APPLICATION LAYER (high-level rules, no DB details)
// Interface only - no implementation!
namespace MyApp.Application.Repositories
{
    public interface ICustomerRepository
    {
        Task<Customer?> GetByIdAsync(int id);
        Task SaveAsync(Customer customer);
    }
}

// INFRASTRUCTURE LAYER (DB-specific details)
// Implementation here - uses EF Core, Dapper, etc.
namespace MyApp.Infrastructure.Repositories
{
    public class CustomerRepository : ICustomerRepository
    {
        private readonly ApplicationDbContext _context;

        public CustomerRepository(ApplicationDbContext context)
        {
            _context = context;
        }

        public async Task<Customer?> GetByIdAsync(int id)
        {
            return await _context.Customers.FindAsync(id);
        }

        public async Task SaveAsync(Customer customer)
        {
            _context.Customers.Update(customer);
            await _context.SaveChangesAsync();
        }
    }
}

// APPLICATION SERVICE (business logic orchestration)
public class ApplyDiscountService
{
    private readonly ICustomerRepository _repository;

    public ApplyDiscountService(ICustomerRepository repository)
    {
        _repository = repository;
    }

    public async Task ApplyEligibleDiscountAsync(int customerId)
    {
        var customer = await _repository.GetByIdAsync(customerId);

        if (customer?.IsEligibleForDiscount() == true)
        {
            customer.ApplyDiscount(0.1);
            await _repository.SaveAsync(customer);
        }
    }
}

The key insight: The Application layer defines the interface, the Infrastructure layer implements it. The Domain layer knows nothing about data access. This is the dependency inversion principle—high-level modules don't depend on low-level modules; they both depend on abstractions.

Architecture Benefits
By placing the repository interface in the Application layer and the implementation in Infrastructure, you ensure that your business logic is independent of database technology. You can test your services with a mock repository, and you can swap database implementations without touching the business logic—that's the power of Clean Architecture.

Real-World Scenarios: When Repositories Shine

Scenario 1: Multi-Tenant SaaS Applications Repositories encapsulate the complexity of filtering by tenant. Every query must include a tenant filter. With a repository, this logic is in one place:

C# - Multi-Tenant Repository
public class TenantAwareRepository<T>
    IRepository<T>
    where T : ITenantEntity
{
    private readonly DbContext _context;
    private readonly ITenantProvider _tenantProvider;

    public async Task<IEnumerable<T>> GetAllAsync()
    {
        var tenantId = _tenantProvider.GetTenantId();
        return await _context.Set<T>()
            .Where(e => e.TenantId == tenantId)
            .ToListAsync();
    }

    // EVERY query automatically filters by tenant
    // Developers can't forget it
}

Scenario 2: Legacy Database with Complex Stored Procedures You have years of SQL logic in stored procedures. Repositories wrap these calls:

C# - Repository Wrapping Stored Procedures
public class LegacyOrderRepository : IOrderRepository
{
    private readonly DbContext _context;

    public async Task<OrderSummary> GetOrderSummaryAsync(
        int orderId)
    {
        // Call stored procedure directly
        var result = await _context
            .OrderSummaries
            .FromSql($"EXEC sp_GetOrderSummary {orderId}")
            .FirstOrDefaultAsync();

        return result;
    }

    // Encapsulates legacy SQL complexity
    // New code uses clean repository interface
}

Scenario 3: Complex Analytical Queries with Caching Some queries are expensive. Repositories cache results:

C# - Repository with Caching
public class CachedAnalyticsRepository :
    IAnalyticsRepository
{
    private readonly DbContext _context;
    private readonly IMemoryCache _cache;

    public async Task<RevenueStats>
        GetMonthlyRevenueAsync(int month)
    {
        var cacheKey = $"revenue_{month}";

        if (!_cache.TryGetValue(cacheKey, out
            RevenueStats stats))
        {
            // Expensive query
            stats = await _context.Orders
                .Where(o => o.Date.Month == month)
                .GroupBy(o => o.Date.Date)
                .Select(g => new RevenueStats { ... })
                .FirstOrDefaultAsync();

            _cache.Set(cacheKey, stats,
                TimeSpan.FromHours(1));
        }

        return stats;
    }

    // Callers don't know about caching
    // Repository can optimize transparently
}
Repositories Enable Optimization Without Refactoring
The real power of repositories: you can add caching, logging, performance optimizations, or migrate to a different database without changing a single line of application code. The interface stays the same; the implementation evolves.

Key Takeaways

  • Repository pattern abstracts data access, separating business logic from database details—like SQL views/stored procedures for OOP. Martin Fowler defined it as "mediating between domain and mapping layers."
  • SQL repositories exist too—views and stored procedures serve the same purpose at the database level. C# repositories are the application-level equivalent.
  • Generic repositories work with any entity type but often hide important business logic and are better as a foundation than a primary pattern.
  • Specific repositories express domain intent through method names like GetActiveCustomers() and are usually the better approach in real applications.
  • Pagination, filtering, sorting belong in repositories, not scattered through service code. Use both LINQ method syntax and query syntax depending on readability.
  • Specification pattern pairs beautifully with repositories, encapsulating complex filtering logic into reusable objects.
  • Testing is easier with repositories—use in-memory implementations or mocks to test business logic without a database.
  • Avoid leaky abstractions—don't expose IQueryable or let implementation details (EF Core, Dapper) leak into the caller's code.
  • DbSet IS a repository—the real question is whether you need a custom wrapper, not whether to use repositories at all.
  • In Clean Architecture, the interface lives in the Application layer, the implementation in Infrastructure—this keeps your business logic independent of database tech.
  • Abstraction costs—repositories add boilerplate but pay off in large systems with complex logic and multiple developers. In small CRUD apps, they're overhead.
  • Real-world wins: multi-tenant filtering, legacy stored procedures, caching, and performance optimization all become transparent with repositories.
