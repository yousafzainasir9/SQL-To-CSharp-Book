Chapter 09

JOINs

You get a requirement: "Show each order with the customer who placed it." SQL developers write JOIN. C# developers use .Include() (navigation properties) or .Join() (explicit). Both combine data from related tables.

01

Requirement: "Show Orders with Customer Names"

Business Requirement
Order list needs customer name next to each order. Users must see "Order #1001 from Alice" not "Order #1001 from CustomerId 5".
SQL Developer Thinks INNER JOIN: keep only orders that have a matching customer. SELECT o.*, c.Name FROM Orders o INNER JOIN Customers c ON o.CustomerId = c.Id. Only rows where both sides match.
C# Developer Thinks Either use .Include(o => o.Customer) to load the navigation property, or explicit .Join() for more control. Include is simpler when FK relationships exist.
SQL INNER JOIN
SELECT o.Id, o.OrderDate, o.Total,
       c.Name, c.Email
FROM Orders o
INNER JOIN Customers c
  ON o.CustomerId = c.Id
ORDER BY o.OrderDate DESC;
LINQ .Include()
var orders = await db.Orders
    .AsNoTracking()
    .Include(o => o.Customer)
    .OrderByDescending(o => o.OrderDate)
    .ToListAsync(ct);

foreach (var o in orders)
    Console.WriteLine(
        $"{o.Id} - {o.Customer.Name}");
LINQ .Join()
var orders = await (
    from o in db.Orders.AsNoTracking()
    join c in db.Customers
        on o.CustomerId equals c.Id
    orderby o.OrderDate descending
    select new {
        o.Id, o.OrderDate, o.Total,
        c.Name, c.Email
    }
).ToListAsync(ct);
Dapper — Multi-mapping with splitOn
await using var conn = new SqlConnection(cs);
var sql = @"
    SELECT o.Id, o.OrderDate, o.Total,
           c.Id, c.Name, c.Email
    FROM Orders o
    INNER JOIN Customers c ON o.CustomerId = c.Id
    ORDER BY o.OrderDate DESC";

var orders = await conn.QueryAsync<OrderDto, CustomerDto, OrderDto>(
    sql,
    (order, customer) => { order.Customer = customer; return order; },
    splitOn: "Id");
02

Requirement: "Show All Customers, Even Without Orders"

Business Requirement
Customer report includes everyone — customers with 0 orders show as "No orders" instead of disappearing entirely.
SQL Developer Thinks LEFT JOIN: keep all left rows even without a match. SELECT c.Name, COUNT(o.Id) FROM Customers c LEFT JOIN Orders o ON c.Id = o.CustomerId GROUP BY c.Id, c.Name. The COUNT gives 0 for no orders.
C# Developer Thinks LINQ doesn't have Left Join natively. Use GroupJoin instead: group orders by customer, then SelectMany with DefaultIfEmpty to keep customers with no matches.
SQL LEFT JOIN
SELECT c.Id, c.Name,
       COUNT(o.Id) AS OrderCount
FROM Customers c
LEFT JOIN Orders o
  ON c.Id = o.CustomerId
GROUP BY c.Id, c.Name
ORDER BY c.Name;
LINQ GroupJoin
var customers = await (
    from c in db.Customers.AsNoTracking()
    join o in db.Orders on c.Id equals o.CustomerId
        into orderGroup
    select new {
        c.Id, c.Name,
        OrderCount = orderGroup.Count()
    }
    orderby c.Name
).ToListAsync(ct);
LINQ Query join...into
var customers = await (
    from c in db.Customers.AsNoTracking()
    join o in db.Orders
        on c.Id equals o.CustomerId into grp
    select new {
        c.Id, c.Name,
        OrderCount = grp.Count(),
        Orders = grp.ToList()
    }
    orderby c.Name
).ToListAsync(ct);
03

Requirement: "Compare Every Product Against Every Category"

Business Requirement
Generate a matrix of all products times all categories for a comparison report. We need all possible combinations even if product-category pairs don't normally match.
SQL Developer Thinks CROSS JOIN: every row from A paired with every row from B. No ON clause. SELECT p.Name, cat.Name FROM Products p CROSS JOIN Categories cat. If 100 products and 10 categories, result is 1000 rows.
C# Developer Thinks .SelectMany() with two collections creates a Cartesian product. For each product, select it paired with every category.
SQL CROSS JOIN
SELECT p.Id, p.Name,
       cat.Id, cat.Name
FROM Products p
CROSS JOIN Categories cat
ORDER BY p.Name, cat.Name;
LINQ SelectMany
var matrix = await db.Products
    .AsNoTracking()
    .SelectMany(
        p => db.Categories,
        (p, cat) => new {
            p.Id, p.Name,
            cat.Id, cat.Name
        }
    )
    .OrderBy(x => x.Name)
    .ThenBy(x => x.Name)
    .ToListAsync(ct);
LINQ Query from...from
var matrix = await (
    from p in db.Products.AsNoTracking()
    from cat in db.Categories.AsNoTracking()
    orderby p.Name, cat.Name
    select new {
        p.Id, p.Name,
        cat.Id, cat.Name
    }
).ToListAsync(ct);
04

Requirement: "Show Order Details with Product and Category"

Business Requirement
Invoice page needs Order → OrderItems → Product → Category all in one query. Show: "Order #1001: Widget (Electronics) $50".
SQL Developer Thinks Chain multiple JOINs. Orders JOIN OrderItems JOIN Products JOIN Categories. Four tables connected sequentially. Each ON links one FK to the next PK.
C# Developer Thinks .Include().ThenInclude() for nested loading. First load Order with Customer, then for each OrderItem load Product, then for Product load Category.
SQL — Multiple JOINs
SELECT o.Id, o.OrderDate,
       oi.Quantity, oi.UnitPrice,
       p.Name AS ProductName,
       cat.Name AS CategoryName
FROM Orders o
INNER JOIN OrderItems oi
  ON o.Id = oi.OrderId
INNER JOIN Products p
  ON oi.ProductId = p.Id
INNER JOIN Categories cat
  ON p.CategoryId = cat.Id
ORDER BY o.OrderDate DESC;
C# — Include Chain
var orders = await db.Orders
    .AsNoTracking()
    .Include(o => o.OrderItems)
        .ThenInclude(oi => oi.Product)
            .ThenInclude(p => p.Category)
    .OrderByDescending(o => o.OrderDate)
    .ToListAsync(ct);

foreach (var order in orders)
  foreach (var item in order.OrderItems)
    Console.WriteLine(
      $"{item.Product.Name} " +
      $"({item.Product.Category.Name})");
Dapper — Multi-mapping with Multiple splitOn
await using var conn = new SqlConnection(cs);
var sql = @"
    SELECT o.Id, o.OrderDate,
           oi.Id, oi.OrderId, oi.ProductId, oi.Quantity, oi.UnitPrice,
           p.Id, p.Name, p.CategoryId,
           cat.Id, cat.Name
    FROM Orders o
    INNER JOIN OrderItems oi ON o.Id = oi.OrderId
    INNER JOIN Products p ON oi.ProductId = p.Id
    INNER JOIN Categories cat ON p.CategoryId = cat.Id
    ORDER BY o.OrderDate DESC";

var orders = await conn.QueryAsync<OrderDto, OrderItemDto, ProductDto, CategoryDto, OrderDto>(
    sql,
    (order, item, product, category) =>
    {
        product.Category = category;
        item.Product = product;
        order.OrderItems.Add(item);
        return order;
    },
    splitOn: "Id");
05

Requirement: "Show Each Employee with Their Manager"

Business Requirement
Org chart — each employee shows their manager's name. "Alice reports to Bob" where Bob is also an Employee.
SQL Developer Thinks Self JOIN: same table twice with aliases. Employees e JOIN Employees m ON e.ManagerId = m.Id. The left side is the employee, right side is their manager (also an employee).
C# Developer Thinks Self-referencing navigation property: Employee has Manager property pointing to another Employee. .Include(e => e.Manager) loads it. Or use explicit .Join() with same table twice.
SQL Self-JOIN
SELECT e.Id, e.Name,
       m.Id AS ManagerId,
       m.Name AS ManagerName
FROM Employees e
LEFT JOIN Employees m
  ON e.ManagerId = m.Id
ORDER BY e.Name;
LINQ .Include()
var employees = await db.Employees
    .AsNoTracking()
    .Include(e => e.Manager)
    .OrderBy(e => e.Name)
    .ToListAsync(ct);

foreach (var emp in employees)
{
    var mgr = emp.Manager?.Name ?? "No manager";
    Console.WriteLine($"{emp.Name} → {mgr}");
}
LINQ Explicit .Join()
var employees = await (
    from e in db.Employees.AsNoTracking()
    join m in db.Employees
        on e.ManagerId equals m.Id
        into mgr
    select new {
        e.Id, e.Name,
        ManagerName = mgr
            .FirstOrDefault()!.Name
    }
    orderby e.Name
).ToListAsync(ct);
06

Requirement: "Join on Multiple Columns"

Business Requirement
Multi-tenant system: match records by both TenantId AND ProductId. A product belongs to a tenant and must match both columns to avoid leaking data across tenants.
SQL Developer Thinks ON clause with AND: ON a.TenantId = b.TenantId AND a.ProductId = b.ProductId. Both conditions must match simultaneously.
C# Developer Thinks .Join() with anonymous key object: new { o.TenantId, o.ProductId } on both sides. Or in query syntax: join...on multiple properties with equals.
SQL — Composite Key
SELECT o.Id, o.Quantity,
       pi.Price
FROM Orders o
INNER JOIN PriceInventory pi
  ON o.TenantId = pi.TenantId
  AND o.ProductId = pi.ProductId
ORDER BY o.Id;
LINQ Method — Composite
var orders = await (
    from o in db.Orders.AsNoTracking()
    join pi in db.PriceInventory
        on new { o.TenantId, o.ProductId }
        equals new { pi.TenantId, pi.ProductId }
    select new {
        o.Id, o.Quantity, pi.Price
    }
    orderby o.Id
).ToListAsync(ct);
LINQ .Join() — Composite
var orders = await db.Orders
    .AsNoTracking()
    .Join(
        db.PriceInventory,
        o => new { o.TenantId, o.ProductId },
        pi => new { pi.TenantId, pi.ProductId },
        (o, pi) => new {
            o.Id, o.Quantity, pi.Price
        }
    )
    .OrderBy(x => x.Id)
    .ToListAsync(ct);
07

Navigation Properties vs Explicit Joins

The Question
Should I use .Include() or .Join()? When should I use Dapper instead?
Scenario Use Navigation (.Include) Use Explicit (.Join) Use Dapper
FK relationship exists in model ✓ Yes — simplest — Not needed — Overkill
Joining unrelated tables ✗ No relationship ✓ Yes — only option in EF ✓ Preferred
Complex join conditions ✗ Can't express ✓ Full LINQ power ✓ Full SQL power
Performance critical — May load too much ✓ Project only needed columns ✓ Direct SQL control
Need raw SQL control ✗ EF translates ✗ EF translates ✓ You write SQL
Left/Right/Full outer join — .Include() is inner ✓ Use GroupJoin + DefaultIfEmpty ✓ Any join type
Large result set — EF translates poorly ✓ Better translation ✓ Best control
Already have SQL — Must rewrite ✗ Must rewrite ✓ Use as-is
SQL Rule Use the JOIN that matches your requirement: INNER (both match), LEFT (keep left), RIGHT (keep right), CROSS (all combinations). Write custom SQL when EF/LINQ falls short.
C# Rule Start with .Include() if relationships exist — it's readable and usually correct. Move to .Join() for control. Use Dapper when you need raw SQL or performance optimization.
Example: When to Reach for Dapper
// Complex SQL that EF can't translate well
await using var conn = new SqlConnection(cs);
var results = await conn.QueryAsync<OrderSummaryDto>(@"
    SELECT o.Id, o.OrderDate,
           STRING_AGG(p.Name, ', ') AS ProductNames,
           SUM(oi.Quantity * oi.UnitPrice) AS Total
    FROM Orders o
    INNER JOIN OrderItems oi ON o.Id = oi.OrderId
    INNER JOIN Products p ON oi.ProductId = p.Id
    GROUP BY o.Id, o.OrderDate
    HAVING SUM(oi.Quantity * oi.UnitPrice) > @MinTotal
    ORDER BY o.OrderDate DESC",
    new { MinTotal = 1000m },
    commandTimeout: 30);

Chapter 09

JOINs → .Join() & Navigation Properties

Master SQL JOINs by understanding how LINQ implements them through method syntax, query syntax, and Entity Framework navigation properties. Learn when to use explicit joins versus navigation properties, and how join semantics differ across SQL and C#.

In This Chapter

  • INNER JOIN with explicit .Join() method syntax and query syntax
  • LEFT JOIN using GroupJoin and query syntax with DefaultIfEmpty()
  • CROSS JOIN with SelectMany cartesian products
  • Multi-table JOINs with chained joins and Include().ThenInclude()
  • Self-joins comparing employee-manager hierarchies
  • Composite key joins on multiple columns
  • Navigation properties versus explicit joins: decision framework
01 — INNER JOIN

INNER JOIN: .Join() & Explicit Relationships

An INNER JOIN returns only rows where both tables have matching values. In LINQ, this is the most explicit join type.

SQL 
SELECT
  c.CustomerId,
  c.Name,
  o.OrderId,
  o.OrderDate
FROM Customers c
INNER JOIN Orders o
  ON c.CustomerId = o.CustomerId
ORDER BY c.CustomerId;
Method Syntax 
var result = customers
  .Join(
    orders,
    c => c.CustomerId,
    o => o.CustomerId,
    (c, o) => new {
      c.CustomerId,
      c.Name,
      o.OrderId,
      o.OrderDate
    }
  )
  .OrderBy(x => x.CustomerId)
  .ToList();
Query Syntax 
var result = (
  from c in customers
  join o in orders
    on c.CustomerId equals o.CustomerId
  orderby c.CustomerId
  select new {
    c.CustomerId,
    c.Name,
    o.OrderId,
    o.OrderDate
  }
).ToList();
Key Point

Query syntax's join clause with equals keyword is strongly typed and leverages C#'s type system. It prevents the subtle bugs that arise from swapped selectors in .Join(). The compiler ensures both sides of equals have compatible types.

In Entity Framework, navigation properties make explicit joins unnecessary. If Customer has an Orders collection, you can simply iterate or filter through it.

EF with Navigation Property 
// When Customers has a virtual Orders property
var result = customers
  .SelectMany(c => c.Orders, (c, o) => new {
    c.CustomerId,
    c.Name,
    o.OrderId,
    o.OrderDate
  })
  .OrderBy(x => x.CustomerId)
  .ToList();

// Or using Include for better control and clarity
var result = context.Customers
  .Include(c => c.Orders)
  .SelectMany(c => c.Orders, (c, o) => new {
    c.CustomerId,
    c.Name,
    o.OrderId,
    o.OrderDate
  })
  .OrderBy(x => x.CustomerId)
  .ToList();

// Navigation properties in query syntax:
var result = (
  from c in context.Customers
  from o in c.Orders
  orderby c.CustomerId
  select new {
    c.CustomerId,
    c.Name,
    o.OrderId,
    o.OrderDate
  }
).ToList();

When using Entity Framework, prefer navigation properties and Include() over explicit .Join() calls.

02 — LEFT JOIN

LEFT JOIN: GroupJoin & Query Syntax Superiority

A LEFT JOIN returns all rows from the left table plus matching rows from the right table. In SQL, this is straightforward with the LEFT JOIN keyword.

SQL LEFT JOIN 
SELECT
  c.CustomerId,
  c.Name,
  o.OrderId,
  o.OrderDate
FROM Customers c
LEFT JOIN Orders o
  ON c.CustomerId = o.CustomerId
ORDER BY c.CustomerId;
Query Syntax Advantage

LEFT JOIN is where LINQ query syntax significantly outperforms method syntax. The method syntax equivalent requires three operations (GroupJoin, SelectMany, DefaultIfEmpty) whereas query syntax needs a single into clause followed by DefaultIfEmpty. Always prefer query syntax for LEFT JOINs unless you have a specific reason to use method chaining.

Method syntax for LEFT JOIN requires careful composition. First, GroupJoin collects all matching orders for each customer into a grouped collection.

Method Syntax (Complex) 
var result = customers
  .GroupJoin(
    orders,
    c => c.CustomerId,
    o => o.CustomerId,
    (c, ordersGroup) => new { c, ordersGroup }
  )
  .SelectMany(
    x => x.ordersGroup.DefaultIfEmpty(),
    (x, o) => new {
      x.c.CustomerId,
      x.c.Name,
      OrderId = o?.OrderId,
      OrderDate = o?.OrderDate
    }
  )
  .OrderBy(x => x.CustomerId)
  .ToList();

// This is equivalent to the SQL but harder to read and maintain

Compare that to query syntax, which is substantially cleaner and more similar to SQL:

Query Syntax (Clean) 
var result = (
  from c in customers
  join o in orders
    on c.CustomerId equals o.CustomerId
    into customerOrders
  from co in customerOrders.DefaultIfEmpty()
  orderby c.CustomerId
  select new {
    c.CustomerId,
    c.Name,
    OrderId = co?.OrderId,
    OrderDate = co?.OrderDate
  }
).ToList();

// The 'into' clause groups matches. DefaultIfEmpty adds empty
// when no matches. Very similar to SQL LEFT JOIN semantics.

In Entity Framework with navigation properties, LEFT JOIN behavior comes naturally. The Include method loads related data, and then you project or filter as needed.

EF LEFT JOIN Patterns 
// Pattern 1: Include then SelectMany with DefaultIfEmpty
var result = await context.Customers
  .Include(c => c.Orders)
  .AsNoTracking()
  .ToListAsync();

var finalResult = result
  .SelectMany(
    c => c.Orders.DefaultIfEmpty(),
    (c, o) => new {
      c.CustomerId,
      c.Name,
      OrderId = o?.OrderId,
      OrderDate = o?.OrderDate
    }
  )
  .OrderBy(x => x.CustomerId)
  .ToList();

// Pattern 2: Query syntax after Include
var result = await (
  from c in context.Customers.Include(c => c.Orders).AsNoTracking()
  from o in c.Orders.DefaultIfEmpty()
  orderby c.CustomerId
  select new {
    c.CustomerId,
    c.Name,
    OrderId = o?.OrderId,
    OrderDate = o?.OrderDate
  }
).ToListAsync();

// Pattern 3: Using explicit LEFT JOIN syntax (still on DbSet)
var result = await (
  from c in context.Customers.AsNoTracking()
  join o in context.Orders.AsNoTracking()
    on c.CustomerId equals o.CustomerId
    into customerOrders
  from co in customerOrders.DefaultIfEmpty()
  orderby c.CustomerId
  select new {
    c.CustomerId,
    c.Name,
    OrderId = co?.OrderId,
    OrderDate = co?.OrderDate
  }
).ToListAsync();

Note the AsEnumerable() call in Pattern 1. Entity Framework's query provider cannot translate DefaultIfEmpty() to SQL directly when used with navigation property flattening.

03 — CROSS JOIN

CROSS JOIN: Cartesian Products with SelectMany

A CROSS JOIN produces a cartesian product: every row from the first table combined with every row from the second table.

SQL CROSS JOIN 
SELECT
  p.ProductId,
  p.Name,
  c.ColorId,
  c.ColorName
FROM Products p
CROSS JOIN Colors c;
Method Syntax 
var result = products
  .SelectMany(
    p => colors,
    (p, c) => new {
      p.ProductId,
      p.Name,
      c.ColorId,
      c.ColorName
    }
  )
  .ToList();
Query Syntax 
var result = (
  from p in products
  from c in colors
  select new {
    p.ProductId,
    p.Name,
    c.ColorId,
    c.ColorName
  }
).ToList();
SelectMany Mechanics

SelectMany flattens nested collections into a single stream. When you provide a second selector, it combines elements from both collections. This is precisely how CROSS JOIN works in SQL. The dual nested from syntax in query syntax is syntactic sugar for SelectMany.

A practical example: imagine you have sizes (S, M, L, XL) and colors (Red, Blue, Green, Black). A CROSS JOIN creates all 16 combinations: S+Red, S+Blue, M+Red, M+Blue, and so on.

Practical Example: Product Variants 
var sizes = new[] { "S", "M", "L", "XL" };
var colors = new[] { "Red", "Blue", "Green", "Black" };

// Method syntax
var variants = sizes
  .SelectMany(size => colors, (size, color) => new {
    SKU = $"{size}-{color}",
    Size = size,
    Color = color
  })
  .ToList();

// Query syntax (more readable)
var variants = (
  from size in sizes
  from color in colors
  select new {
    SKU = $"{size}-{color}",
    Size = size,
    Color = color
  }
).ToList();

// Result: 16 combinations (4 sizes × 4 colors)
// S-Red, S-Blue, S-Green, S-Black,
// M-Red, M-Blue, M-Green, M-Black, ... etc

Be cautious with CROSS JOINs on large tables. The result set grows exponentially.

04 — MULTI-TABLE JOINS

Multi-Table JOINs: Chaining & ThenInclude

Real-world queries often join three, four, or more tables. Joining Orders to Customers is simple, but joining Orders to OrderItems to Products requires chaining joins.

Consider a query: get all order items and their products for customers in Seattle. You need to join Customers → Orders → OrderItems → Products, filtering by city.

SQL Three-Table Join 
SELECT
  c.CustomerId,
  c.Name,
  o.OrderId,
  o.OrderDate,
  oi.OrderItemId,
  oi.Quantity,
  p.ProductId,
  p.ProductName
FROM Customers c
INNER JOIN Orders o ON c.CustomerId = o.CustomerId
INNER JOIN OrderItems oi ON o.OrderId = oi.OrderId
INNER JOIN Products p ON oi.ProductId = p.ProductId
WHERE c.City = 'Seattle'
ORDER BY o.OrderDate DESC;

Method syntax becomes verbose with multiple joins. Each additional join requires more nesting and intermediate variable tracking:

Method Syntax (Chained & Complex) 
var result = customers
  .Where(c => c.City == "Seattle")
  .Join(
    orders,
    c => c.CustomerId,
    o => o.CustomerId,
    (c, o) => new { c, o }
  )
  .Join(
    orderItems,
    co => co.o.OrderId,
    oi => oi.OrderId,
    (co, oi) => new { co.c, co.o, oi }
  )
  .Join(
    products,
    cooi => cooi.oi.ProductId,
    p => p.ProductId,
    (cooi, p) => new {
      cooi.c.CustomerId,
      cooi.c.Name,
      cooi.o.OrderId,
      cooi.o.OrderDate,
      cooi.oi.OrderItemId,
      cooi.oi.Quantity,
      p.ProductId,
      p.ProductName
    }
  )
  .OrderByDescending(x => x.OrderDate)
  .ToList();
Query Syntax for Multi-Joins

Query syntax scales much better than method syntax for multi-table joins. Each additional table is a single join clause, not nested selectors. The variable names remain readable, and the structure mirrors SQL almost exactly.

Query Syntax (Cleaner & More Maintainable) 
var result = (
  from c in customers
  where c.City == "Seattle"
  join o in orders on c.CustomerId equals o.CustomerId
  join oi in orderItems on o.OrderId equals oi.OrderId
  join p in products on oi.ProductId equals p.ProductId
  orderby o.OrderDate descending
  select new {
    c.CustomerId,
    c.Name,
    o.OrderId,
    o.OrderDate,
    oi.OrderItemId,
    oi.Quantity,
    p.ProductId,
    p.ProductName
  }
).ToList();

In Entity Framework with navigation properties, the code becomes even simpler and more efficient.

EF with Include().ThenInclude() (Best Approach) 
// Load the full object graph with Include chains
var customerData = context.Customers
  .Where(c => c.City == "Seattle")
  .Include(c => c.Orders)
    .ThenInclude(o => o.OrderItems)
    .ThenInclude(oi => oi.Product)
  .ToList();

// Then project to DTO in memory
var result = customerData
  .SelectMany(c => c.Orders)
  .SelectMany(o => o.OrderItems, (o, oi) => new {
    CustomerId = o.Customer.CustomerId,
    CustomerName = o.Customer.Name,
    o.OrderId,
    o.OrderDate,
    oi.OrderItemId,
    oi.Quantity,
    ProductId = oi.Product.ProductId,
    ProductName = oi.Product.ProductName
  })
  .OrderByDescending(x => x.OrderDate)
  .ToList();

// Or using query syntax on DbSets (runs on server)
var result = (
  from c in context.Customers
  where c.City == "Seattle"
  from o in c.Orders
  from oi in o.OrderItems
  join p in context.Products on oi.ProductId equals p.ProductId
  orderby o.OrderDate descending
  select new {
    c.CustomerId,
    c.Name,
    o.OrderId,
    o.OrderDate,
    oi.OrderItemId,
    oi.Quantity,
    p.ProductId,
    p.ProductName
  }
).ToList();

The Include().ThenInclude() approach tells Entity Framework exactly which related data to load, reducing the "N+1 query" problem.

05 — SELF-JOIN

Self-Join: Employee-Manager Hierarchies

A self-join connects a table to itself. The classic example is an Employee table where each employee has a ManagerId pointing to another employee row.

SQL Self-Join 
SELECT
  e.EmployeeId,
  e.Name AS EmployeeName,
  m.EmployeeId AS ManagerId,
  m.Name AS ManagerName
FROM Employees e
LEFT JOIN Employees m
  ON e.ManagerId = m.EmployeeId;
Method Syntax 
var result = employees
  .GroupJoin(
    employees,
    e => e.ManagerId,
    m => m.EmployeeId,
    (e, managers) => new { e, managers }
  )
  .SelectMany(
    x => x.managers.DefaultIfEmpty(),
    (x, m) => new {
      x.e.EmployeeId,
      EmployeeName = x.e.Name,
      ManagerId = m?.EmployeeId,
      ManagerName = m?.Name
    }
  )
  .ToList();
Query Syntax 
var result = (
  from e in employees
  join m in employees
    on e.ManagerId equals m.EmployeeId
    into managersGroup
  from manager in managersGroup.DefaultIfEmpty()
  select new {
    e.EmployeeId,
    EmployeeName = e.Name,
    ManagerId = manager?.EmployeeId,
    ManagerName = manager?.Name
  }
).ToList();
Navigation Property Self-Join

In Entity Framework, if Employee has a Manager navigation property (a reference back to another Employee), you can access it directly without a join. No explicit join syntax is needed—you simply follow the navigation property through the object graph.

EF Self-Join with Navigation Properties 
// Entity model with self-referential relationship:
public class Employee {
  public int EmployeeId { get; set; }
  public string Name { get; set; }
  public int? ManagerId { get; set; }

  // Navigation back to manager
  public virtual Employee Manager { get; set; }

  // Navigation to employees reporting to this person
  public virtual ICollection Reports { get; set; }
}

// Query using navigation properties:
var result = context.Employees
  .Include(e => e.Manager)
  .Select(e => new {
    e.EmployeeId,
    EmployeeName = e.Name,
    ManagerId = e.Manager.EmployeeId,
    ManagerName = e.Manager.Name
  })
  .ToList();

// With DefaultIfEmpty for true LEFT JOIN (null managers):
var result = (
  from e in context.Employees.Include(e => e.Manager)
  from m in new[] { e.Manager }.DefaultIfEmpty()
  select new {
    e.EmployeeId,
    EmployeeName = e.Name,
    ManagerId = m?.EmployeeId,
    ManagerName = m?.Name
  }
).ToList();

// Multi-level hierarchy (employee, manager, manager's manager):
var result = context.Employees
  .Include(e => e.Manager)
    .ThenInclude(m => m.Manager)
  .Select(e => new {
    e.EmployeeId,
    EmployeeName = e.Name,
    ManagerId = e.Manager?.EmployeeId,
    ManagerName = e.Manager?.Name,
    GrandManagerId = e.Manager?.Manager?.EmployeeId,
    GrandManagerName = e.Manager?.Manager?.Name
  })
  .ToList();

Self-joins become complex when you need multi-level hierarchies (employees, their managers, and their managers' managers).

06 — COMPOSITE KEY JOINS

Composite Key JOINs: Multiple Join Conditions

Some tables are identified by multiple columns, called a composite key (or compound key). When joining on composite keys, you must match all key columns.

SQL Composite Join 
SELECT
  o.OrderId,
  o.CustomerId,
  oi.OrderItemId,
  oi.Quantity
FROM Orders o
INNER JOIN OrderItems oi
  ON o.OrderId = oi.OrderId
  AND o.CustomerId = oi.CustomerId;
Method Syntax 
var result = orders
  .Join(
    orderItems,
    o => new { o.OrderId, o.CustomerId },
    oi => new { oi.OrderId, oi.CustomerId },
    (o, oi) => new {
      o.OrderId,
      o.CustomerId,
      oi.OrderItemId,
      oi.Quantity
    }
  )
  .ToList();
Query Syntax 
var result = (
  from o in orders
  join oi in orderItems
    on new { o.OrderId, o.CustomerId }
    equals new { oi.OrderId, oi.CustomerId }
  select new {
    o.OrderId,
    o.CustomerId,
    oi.OrderItemId,
    oi.Quantity
  }
).ToList();
Anonymous Object Keys

C# requires anonymous objects with matching property names and types for the equals operator to work correctly. If your keys are named differently (OrderId vs OId), you must explicitly create the anonymous objects with consistent property names. The compiler compares the entire object for equality, not individual fields.

A practical example: a StudentGrades table keyed by (StudentId, CourseId, Semester). Joining to a Courses table requires matching all three components.

Three-Component Composite Key Example 
// Entities:
public class StudentGrade {
  public int StudentId { get; set; }
  public int CourseId { get; set; }
  public int Semester { get; set; }
  public decimal Grade { get; set; }
}

public class Course {
  public int CourseId { get; set; }
  public int Semester { get; set; }
  public string CourseName { get; set; }
  public string Instructor { get; set; }
}

// Query with composite key joining 3 columns:
var result = (
  from sg in studentGrades
  join c in courses
    on new { sg.CourseId, sg.Semester }
    equals new { c.CourseId, c.Semester }
  select new {
    sg.StudentId,
    sg.CourseId,
    sg.Semester,
    sg.Grade,
    c.CourseName,
    c.Instructor
  }
).ToList();

// Method syntax (less readable but equivalent):
var result = studentGrades
  .Join(
    courses,
    sg => new { sg.CourseId, sg.Semester },
    c => new { c.CourseId, c.Semester },
    (sg, c) => new {
      sg.StudentId,
      sg.CourseId,
      sg.Semester,
      sg.Grade,
      c.CourseName,
      c.Instructor
    }
  )
  .ToList();

Composite keys are straightforward once you remember to create anonymous objects with matching property names and types.

07 — DECISION FRAMEWORK

Navigation Properties vs. Explicit Joins: When to Use Each

You now have multiple tools for joining: explicit .Join(), query syntax join, navigation properties with Include(), and SelectMany with navigation.

Approach Readability Performance When to Use
Navigation Property + SelectMany Excellent Best (Include controls loading) EF Core, simple hierarchies, one-to-many relationships
Query Syntax join Excellent Good (translates to SQL) Multi-table joins, explicit conditions, LINQ to Objects, disconnected data
Method .Join() Fair Good (translates to SQL) When method chaining is required, complex lambda expressions, dynamic joins
Left Join with GroupJoin Poor Good Never use method syntax for LEFT—use query syntax instead
Raw SQL (Dapper, FromSqlRaw) Excellent Best (exact SQL control) Complex joins, window functions, business logic in DB, performance-critical
EF Core Preference

In Entity Framework Core, prefer navigation properties and Include().ThenInclude() over explicit joins. EF optimizes Include operations and generates efficient SQL. Explicit joins in EF often result in unexpected CROSS JOINs or Cartesian products when dealing with collections.

Decision Framework: Example Queries 
// Scenario 1: Simple parent-child in EF
// Decision: Navigation property + Include
var customers = await context.Customers
  .Include(c => c.Orders)
  .AsNoTracking()
  .ToListAsync();

// Scenario 2: Three-table join, in-memory data
// Decision: Query syntax (readable and efficient)
var result = (
  from o in orders
  join c in customers on o.CustomerId equals c.CustomerId
  join p in products on o.ProductId equals p.ProductId
  select new { o, c, p }
).ToList();

// Scenario 3: LEFT JOIN, all customers with optional orders
// Decision: Query syntax with into/DefaultIfEmpty
var result = (
  from c in customers
  join o in orders on c.CustomerId equals o.CustomerId into customerOrders
  from co in customerOrders.DefaultIfEmpty()
  select new { c, co }
).ToList();

// Scenario 4: Dynamic join condition from user input
// Decision: Method .Join() for parameterized conditions
Func customerKeySelector = c => c.CustomerId;
Func orderKeySelector = o => o.CustomerId;
var result = customers
  .Join(orders, customerKeySelector, orderKeySelector, (c, o) => new { c, o })
  .ToList();

// Scenario 5: Complex SQL needed, window functions, CTEs
// Decision: Raw SQL or Dapper
var result = await context.Orders
  .FromSqlRaw(
    "SELECT * FROM Orders WHERE CustomerId = @id AND OrderDate > @date",
    new SqlParameter("@id", customerId),
    new SqlParameter("@date", DateTime.Now.AddYears(-1))
  )
  .AsNoTracking()
  .ToListAsync();

// Scenario 6: Avoiding N+1 with deep include chains
// Decision: Include().ThenInclude() chains
var result = await context.Customers
  .AsNoTracking()
  .Where(c => c.City == "Seattle")
  .Include(c => c.Orders)
    .ThenInclude(o => o.OrderItems)
    .ThenInclude(oi => oi.Product)
  .Include(c => c.Addresses)
  .ToListAsync();
Performance Measurement

Navigation properties with Include() allow Entity Framework to generate a single, optimized query. Explicit joins in EF may result in multiple queries or inefficient SQL. Always measure with SQL logging enabled to see what SQL is generated. Use the Database.Log property or enable query logging in your DbContext to inspect generated queries.

Key Takeaways

  • INNER JOINs are explicit with .Join() method syntax, but query syntax is more readable for complex joins
  • LEFT JOINs shine with query syntax using into/DefaultIfEmpty(). Avoid method syntax GroupJoin for readability
  • CROSS JOINs use SelectMany or multiple from clauses for cartesian products—be cautious with large datasets
  • Multi-table joins benefit from query syntax over chained .Join() calls in terms of readability
  • Self-joins and composite key joins use the same patterns as regular joins but with careful variable management
  • Prefer EF navigation properties and Include() over explicit joins in Entity Framework for optimal SQL generation
  • Query syntax scales better than method syntax as the number of joins increases
  • LEFT JOIN is where query syntax significantly outperforms method syntax due to the into/DefaultIfEmpty pattern
