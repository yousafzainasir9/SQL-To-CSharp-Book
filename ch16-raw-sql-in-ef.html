Raw SQL in EF Core — When ORM Isn't Enough

Entity Framework is powerful, but not every query can be expressed in LINQ. Learn FromSqlRaw, FromSqlInterpolated, ExecuteSqlRaw, and when to mix raw SQL with EF's automatic tracking and included relationships.

In This Chapter

  • FromSqlRaw() and FromSqlInterpolated() for querying with SQL while still using IQueryable
  • Safe parameterization with FromSqlInterpolated vs vulnerable string concatenation
  • ExecuteSqlRaw() for non-query commands: bulk updates, DDL, stored procedures
  • Mixing raw SQL with LINQ: Include(), Where(), and composition
  • Database views and table-valued functions in EF models
  • Decision framework: when to use LINQ vs raw SQL vs stored procedures
01

FromSqlRaw — Querying with SQL

FromSqlRaw() lets you execute a SQL SELECT statement, but the result is still an IQueryable<T>. This means you can chain LINQ operators on top: .Where(), .OrderBy(), .Include(), etc.

This is perfect for complex queries that are easier to write in SQL than LINQ.

FromSqlRaw with Parameterization
// Basic raw SQL query var products = await context.Products .FromSqlRaw( "SELECT * FROM Products " + "WHERE Price > {0}", 100m) .ToListAsync(); // Result is still IQueryable // You can chain LINQ var filtered = await context.Products .FromSqlRaw( "SELECT * FROM Products") .Where(p => p.IsActive) .OrderBy(p => p.Name) .Take(10) .ToListAsync(); // EF converts to: // SELECT * FROM ( // SELECT * FROM Products // ) AS t WHERE t.IsActive = 1 // ORDER BY t.Name // OFFSET 0 ROWS FETCH NEXT 10
SQL: Direct Query
-- You have to write the -- entire SELECT yourself SELECT * FROM Products WHERE Price > 100; -- Want to filter, order, and -- limit? You write it all: SELECT * FROM Products WHERE Price > 100 AND IsActive = 1 ORDER BY Name LIMIT 10; -- No composition, just SQL

Rules for FromSqlRaw():

  • Your SQL must return all columns of the entity
  • Columns must be properly aliased if selecting from subqueries
  • Parameters use {0}, {1}, ... placeholders
  • Parameters are still parameterized (safe from SQL injection)
FromSqlRaw Wraps Your SQL as a Subquery
When you chain LINQ after FromSqlRaw(), EF wraps your SQL in a subquery and applies the LINQ operators in the outer query. This is efficient but requires proper column naming. Your SELECT statement must return properly named columns that match the entity properties.
02

FromSqlInterpolated — Safe Parameterization

FromSqlInterpolated() uses C# string interpolation to safely parameterize your SQL. Variables in the interpolated string automatically become SQL parameters, not string concatenation.

This is the safe way to pass user input to raw SQL queries. Never concatenate strings directly into SQL.

SAFE: FromSqlInterpolated
// Safe: interpolation becomes // SQL parameter decimal minPrice = 100m; var products = await context.Products .FromSqlInterpolated( $"SELECT * FROM Products " + $"WHERE Price > {minPrice}") .ToListAsync(); // EF generates: // SELECT * FROM Products // WHERE Price > @p0 // Parameters: @p0 = 100 // Even if minPrice is user input // SQL injection is impossible var userInput = "0; DROP TABLE Products"; // It becomes a literal string parameter // Safe!
DANGEROUS: String Concatenation
-- NEVER do this: -- DANGEROUS!!! SQL Injection DECLARE @minPrice DECIMAL; SET @minPrice = 100; -- If you concatenate strings: SELECT * FROM Products WHERE Price > + @minPrice OR 1=1; -- Attacker injected this -- If attacker controls minPrice: -- Input: "100 OR 1=1" -- Now query returns ALL products -- Disaster! -- Always use parameterization
UNSAFE: String Concatenation
// DO NOT DO THIS!!! // VULNERABLE TO SQL INJECTION string minPrice = userInput; // String concatenation var query = "SELECT * FROM Products " + "WHERE Price > " + minPrice; var products = await context.Products .FromSqlRaw(query) .ToListAsync(); // If minPrice = "100 OR 1=1" // Query becomes: // SELECT * FROM Products // WHERE Price > 100 OR 1=1 // Returns ALL rows! // Data breach!
SQL Injection Attack
-- Attacker input: "0 OR 1=1 DROP TABLE Users;" -- If you concatenate directly: SELECT * FROM Products WHERE Price > 0 OR 1=1; -- Or worse: DROP TABLE Users; -- Database destroyed! -- With parameterization, -- it's treated as a literal value -- Safe!
Always Use FromSqlInterpolated with User Input
FromSqlInterpolated() automatically converts interpolated variables to SQL parameters. Never use FromSqlRaw() with concatenated user input. Always use interpolation or explicit parameters.
03

ExecuteSqlRaw — Non-Query Commands

Sometimes you need to execute SQL that doesn't return entities: bulk updates, DDL statements, or stored procedures that return no result set.

Use context.Database.ExecuteSqlRaw() for these scenarios. It executes the SQL and returns the number of rows affected.

ExecuteSqlRaw for Bulk Operations
// Bulk update without loading // entities var rowsAffected = await context.Database .ExecuteSqlRawAsync( "UPDATE Products " + "SET Price = Price * {0} " + "WHERE Category = {1}", 1.1m, "Electronics"); Console.WriteLine( $"{rowsAffected} products updated"); // Bulk delete await context.Database .ExecuteSqlRawAsync( "DELETE FROM Orders " + "WHERE OrderDate < {0}", DateTime.UtcNow .AddMonths(-12)); // Execute stored procedure await context.Database .ExecuteSqlRawAsync( "EXEC sp_GenerateReport {0}", DateTime.UtcNow);
SQL: Direct Execution
-- Bulk update UPDATE Products SET Price = Price * 1.1 WHERE Category = 'Electronics'; -- Check rows affected SELECT @@ROWCOUNT; -- Bulk delete DELETE FROM Orders WHERE OrderDate < '2023-01-01'; -- Execute stored procedure EXEC sp_GenerateReport '2024-01-01'; -- All direct SQL, no ORM

When to use ExecuteSqlRaw():

  • Bulk updates or deletes (don't load all entities)
  • DDL statements (CREATE, ALTER, DROP)
  • Stored procedures that return no result set
  • Complex operations that would be slow with LINQ
ExecuteSqlRaw Bypasses Change Tracking
ExecuteSqlRaw() modifies the database directly without loading or tracking entities. If you have loaded entities in memory that match the updated/deleted rows, those entities will be out of sync. Reload or clear the change tracker if needed.
04

Mixing Raw SQL and LINQ

One of the most powerful features of EF is the ability to start with raw SQL and chain LINQ operators. Your SQL becomes the base query, and LINQ adds additional filtering, ordering, or includes.

This combines the power of both: complex SQL logic with LINQ composition.

Raw SQL + LINQ Composition
// Start with complex SQL var query = context.Customers .FromSqlInterpolated( $"SELECT DISTINCT c.* " + $"FROM Customers c " + $"WHERE c.CreatedAt > {cutoffDate}") .Include(c => c.Orders) .Where(c => c.IsActive) .OrderBy(c => c.Name); // EF generates: // SELECT DISTINCT c.*, o.* FROM ( // SELECT DISTINCT c.* FROM Customers c // WHERE c.CreatedAt > @p0 // ) AS t // LEFT JOIN Orders o ON c.Id=o.CustomerId // WHERE t.IsActive = 1 // ORDER BY t.Name var results = await query .ToListAsync();
Equivalent SQL
-- Complex base query with -- additional filters and joins SELECT DISTINCT c.*, o.* FROM ( SELECT DISTINCT c.* FROM Customers c WHERE c.CreatedAt > '2024-01-01' ) AS t LEFT JOIN Orders o ON t.Id = o.CustomerId WHERE t.IsActive = 1 ORDER BY t.Name; -- EF generates this automatically -- from the FromSqlRaw + LINQ

Important rules for composing LINQ with raw SQL:

  • Your SQL must return all entity columns, properly named
  • EF wraps your SQL in a subquery, so aliases matter
  • Include() still works—it generates JOINs
  • Not all LINQ operations are composable (some are evaluated client-side)
FromSqlRaw Makes Your SQL the Subquery
When you chain LINQ after FromSqlRaw(), EF wraps your SQL in a subquery. This is efficient for databases like SQL Server but can be inefficient on some other databases. If you're adding simple filters, it's better to include them in your original SQL string.
05

Views and Table-Valued Functions

Database views and table-valued functions (TVFs) are SQL patterns that EF can model. Views are read-only projections; TVFs are parametrized queries stored in the database.

You can query both using FromSqlRaw() or by creating EF entity types that map to them.

Querying a Database View
// Entity class for view public class ActiveCustomerView { public int Id { get; set; } public string Name { get; set; } public int OrderCount { get; set; } } // In DbContext: public DbSet< ActiveCustomerView> ActiveCustomers { get; set; } // In OnModelCreating: protected override void OnModelCreating( ModelBuilder modelBuilder) { modelBuilder .Entity< ActiveCustomerView>() .HasNoKey() .ToView( "ActiveCustomers"); } // Query the view var results = await context.ActiveCustomers .Where(c => c.OrderCount > 10) .ToListAsync();
SQL: Create View
CREATE VIEW ActiveCustomers AS SELECT c.Id, c.Name, COUNT(o.Id) AS OrderCount FROM Customers c LEFT JOIN Orders o ON c.Id = o.CustomerId WHERE c.IsActive = 1 GROUP BY c.Id, c.Name; -- Query the view SELECT * FROM ActiveCustomers WHERE OrderCount > 10;

Table-Valued Functions (TVFs):

Querying a Table-Valued Function
// Call TVF using FromSqlRaw var recentOrders = await context.Orders .FromSqlInterpolated( $"SELECT * FROM " + $"dbo.GetOrdersByDate({date})") .ToListAsync(); // Or use DbSet mapped to TVF protected override void OnModelCreating( ModelBuilder m) { m.Entity<Order>() .ToFunction( "GetOrdersByDate"); } // Then call with parameter var results = await context.GetOrdersByDate(date) .ToListAsync();
SQL: Create TVF
CREATE FUNCTION dbo.GetOrdersByDate( @date DATE) RETURNS TABLE AS RETURN ( SELECT * FROM Orders WHERE OrderDate >= @date ); -- Call the TVF SELECT * FROM dbo .GetOrdersByDate('2024-01-01');
Views Don't Have Keys in EF
Views don't have primary keys, so you must use .HasNoKey() when configuring them in OnModelCreating(). You can still query them with LINQ, but you can't modify them.
06

FromSqlRaw vs FromSqlInterpolated: Security Deep Dive

The difference between FromSqlRaw() and FromSqlInterpolated() is critical for security. FromSqlRaw() requires manual parameterization, while FromSqlInterpolated() uses C# string interpolation to automatically create parameters. Using the wrong method with user input opens SQL injection vulnerabilities.

VULNERABLE: FromSqlRaw with Concatenation
// DANGEROUS! User input directly in SQL var userId = GetUserIdFromRequest(); // Attacker sends: 1 OR 1=1 var user = await context.Users .FromSqlRaw( "SELECT * FROM Users " + "WHERE Id = " + userId) .FirstOrDefaultAsync(); // Generated SQL: (UNSAFE) // SELECT * FROM Users // WHERE Id = 1 OR 1=1 // ^ Returns ALL users!
Resulting SQL Injection Attack
-- What attacker submits: SELECT * FROM Users WHERE Id = 1 OR 1=1; -- Returns all users instead of one -- Attacker can now steal all data -- Or worse: DROP TABLE attack WHERE Id = 1; DROP TABLE Users;--
SAFE: FromSqlRaw with Explicit Parameters
// SAFE: Manual parameterization var userId = GetUserIdFromRequest(); var user = await context.Users .FromSqlRaw( "SELECT * FROM Users " + "WHERE Id = {0}", userId) .FirstOrDefaultAsync(); // Generated SQL: (SAFE) // SELECT * FROM Users // WHERE Id = @p0 // Parameter: @p0 = "1 OR 1=1" // ^ Treated as literal value, not SQL
BEST: FromSqlInterpolated
// SAFEST: String interpolation // handles parameterization automatically var userId = GetUserIdFromRequest(); var user = await context.Users .FromSqlInterpolated( $"SELECT * FROM Users " + $"WHERE Id = {userId}") .FirstOrDefaultAsync(); // EF automatically creates parameter: // SELECT * FROM Users // WHERE Id = @p0 // Variable userId becomes SQL parameter

Key Security Rules:

  • Never concatenate user input directly into SQL strings with FromSqlRaw()
  • Always use FromSqlInterpolated() when variables come from user input
  • If using FromSqlRaw(), use the {0}, {1} placeholders and pass parameters as separate arguments
  • Parameterized queries treat all user input as literal data, not executable SQL
  • This defense works regardless of what characters the attacker submits (quotes, semicolons, SQL keywords)
FromSqlInterpolated Is the Modern Standard
C# string interpolation with FromSqlInterpolated() is cleaner and safer than manual parameter placeholders. EF automatically converts interpolated variables to SQL parameters. Always prefer FromSqlInterpolated() for any user input, and FromSqlRaw() only for static SQL strings.
07

Database-Specific Functions in Raw SQL

LINQ can express basic operations, but many databases offer powerful, specialized functions that LINQ cannot translate. Examples include SQL Server's STRING_AGG and OPENJSON, PostgreSQL's array functions, and MySQL's GROUP_CONCAT. Raw SQL is the only way to access these database-specific features directly.

SQL Server: STRING_AGG for Aggregation
-- Concatenate values with delimiter SELECT CustomerId, STRING_AGG(ProductName, ',') WITHIN GROUP (ORDER BY OrderDate) AS ProductList FROM Orders o JOIN OrderItems oi ON o.Id = oi.OrderId GROUP BY CustomerId; -- Result: -- CustomerId | ProductList -- 1 | Widget,Gadget,Gizmo -- 2 | Doohickey,Thingamajig
Using STRING_AGG in EF
// Cannot express STRING_AGG in LINQ // Must use raw SQL var result = await context.Orders .FromSqlRaw( @"SELECT CustomerId, STRING_AGG(ProductName, ',') WITHIN GROUP (ORDER BY OrderDate) AS ProductList FROM Orders o JOIN OrderItems oi ON o.Id = oi.OrderId GROUP BY CustomerId") .ToListAsync();
SQL Server: OPENJSON for JSON Parsing
-- Parse JSON data into rows SELECT j.value AS id, j.[key] AS [key] FROM Products p CROSS APPLY OPENJSON(p.MetadataJson) WITH ( id INT '$.id', [key] NVARCHAR(100) '$.key' ) j WHERE p.Id > 10; -- Transforms JSON to relational rows
Using OPENJSON in EF
// Define class for parsed JSON public class JsonMetadata { public int Id { get; set; } public string Key { get; set; } } // Use OPENJSON in raw SQL var metadata = await context.Set<JsonMetadata>() .FromSqlRaw( @"SELECT j.value AS Id, j.[key] AS [Key] FROM Products p CROSS APPLY OPENJSON(p.MetadataJson) WITH ( id INT '$.id', [key] NVARCHAR(100) '$.key' ) j WHERE p.Id > {0}", 10) .ToListAsync();
PostgreSQL: Array Functions
-- Array aggregation (PostgreSQL) SELECT customer_id, array_agg(product_name) AS products FROM orders GROUP BY customer_id; -- Array operations SELECT * FROM products WHERE 'electronics' = ANY(categories); -- Unnest arrays to rows SELECT customer_id, unnest(tags) AS tag FROM customers;
Using Array Functions in EF (PostgreSQL)
// Array aggregation var result = await context.Orders .FromSqlRaw( @"SELECT customer_id, array_agg(product_name) AS products FROM orders GROUP BY customer_id") .ToListAsync(); // Unnest arrays var tags = await context.Set<CustomerTags>() .FromSqlRaw( @"SELECT customer_id, unnest(tags) AS tag FROM customers") .ToListAsync();
When LINQ Hits Its Limits
Every database offers powerful, specialized functions. SQL Server has STRING_AGE, OPENJSON, FORMAT. PostgreSQL has array operations, range types, and JSON operators. MySQL has JSON functions and GROUP_CONCAT. LINQ cannot express most of these. Raw SQL is the direct path to database power.
08

Raw SQL with Stored Procedures

Stored procedures are SQL programs stored in the database. They can encapsulate complex business logic, call other procedures, and return multiple result sets. EF can call stored procedures using raw SQL, though the syntax varies by database.

SQL: Create a Stored Procedure
CREATE PROCEDURE dbo.ProcessMonthlyOrders @month INT, @year INT, @processedCount INT OUTPUT AS BEGIN UPDATE Orders SET Status = 'Processed', ProcessedDate = GETDATE() WHERE MONTH(OrderDate) = @month AND YEAR(OrderDate) = @year AND Status = 'Pending'; SET @processedCount = @@ROWCOUNT; SELECT COUNT(*) FROM Orders WHERE Status = 'Processed'; END;
Call Stored Procedure from EF
// For stored procedures with output var monthParam = new SqlParameter("@month", 3); var yearParam = new SqlParameter("@year", 2024); var countParam = new SqlParameter { ParameterName = "@processedCount", SqlDbType = SqlDbType.Int, Direction = ParameterDirection .Output }; var result = await context.Orders .FromSqlRaw( "EXEC dbo.ProcessMonthlyOrders " + "@month, @year, " + "@processedCount OUTPUT", monthParam, yearParam, countParam) .ToListAsync(); // Read output parameter var processed = (int)countParam.Value;

Stored Procedure Considerations:

  • Stored procedures are database-specific (T-SQL for SQL Server, PL/pgSQL for PostgreSQL)
  • Version control becomes harder (SQL in database, not in source code)
  • Calling syntax varies: SQL Server uses EXEC, PostgreSQL uses SELECT
  • Prefer stored procedures only if they already exist and are well-tested
  • For new code, prefer application-level logic in C# with raw SQL for the database operations
Stored Procedures Are Legacy
Stored procedures were common in older architectures, but modern applications increasingly handle business logic at the application layer. EF can call them, but be cautious about overusing them. Keep SQL in source control alongside your C# code, not hidden in the database.

ExecuteUpdate and ExecuteDelete (EF Core 7+)

EF Core 7 introduced ExecuteUpdateAsync() and ExecuteDeleteAsync(), which perform bulk operations without loading entities. These replace raw SQL for bulk updates/deletes in most cases, combining the safety of LINQ with the performance of raw SQL.

BEFORE: Bulk Update with Raw SQL
// Old approach: ExecuteSqlRaw await context.Database .ExecuteSqlRawAsync( @"UPDATE Products SET Price = Price * {0} WHERE Category = {1}", 1.10m, "Electronics"); // Problems: // 1. String-based SQL // 2. Manual parameter handling // 3. Less type-safe
AFTER: ExecuteUpdate (EF Core 7+)
// New approach: LINQ-based var updatedCount = await context.Products .Where(p => p.Category == "Electronics") .ExecuteUpdateAsync(s => s.SetProperty(p => p.Price, p => p.Price * 1.10m)); // Benefits: // 1. LINQ-based, type-safe // 2. Automatic parameterization // 3. No entity loading
ExecuteUpdate: Multiple Properties
// Update multiple columns var count = await context.Orders .Where(o => o.CreatedDate < DateTime.Now.AddYears(-1)) .ExecuteUpdateAsync(s => s.SetProperty( o => o.Status, "Archived") .SetProperty( o => o.ArchivedDate, DateTime.Now)); // Generated SQL: // UPDATE Orders // SET Status = 'Archived', // ArchivedDate = GETDATE() // WHERE CreatedDate < @p0
ExecuteDelete: Bulk Deletion
// Delete without loading var deletedCount = await context.LogEntries .Where(l => l.CreatedDate < DateTime.Now .AddDays(-30)) .ExecuteDeleteAsync(); // Generated SQL: // DELETE FROM LogEntries // WHERE CreatedDate < @p0 // No change tracking overhead // Extremely efficient
Computed Updates with ExecuteUpdate
// Use computed expressions var count = await context.OrderItems .Where(oi => oi.Quantity > 100) .ExecuteUpdateAsync(s => s.SetProperty( oi => oi.Discount, oi => oi.Discount + 0.05m)); // Update based on condition var count2 = await context.Customers .Where(c => c.IsActive) .ExecuteUpdateAsync(s => s.SetProperty( c => c.Status, c => c.TotalOrders > 100 ? "Gold" : "Standard"));
Equivalent SQL with CASE
-- Computed update UPDATE OrderItems SET Discount = Discount + 0.05 WHERE Quantity > 100; -- Conditional update (CASE) UPDATE Customers SET Status = CASE WHEN TotalOrders > 100 THEN 'Gold' ELSE 'Standard' END WHERE IsActive = 1;
ExecuteUpdate/Delete Over Raw SQL
Starting with EF Core 7, prefer ExecuteUpdateAsync() and ExecuteDeleteAsync() over raw SQL for bulk operations. They're LINQ-based (type-safe), automatically parameterized, and perform identically to raw SQL. Raw SQL is now mainly for complex queries, not bulk operations.

When to Use Raw SQL vs LINQ vs Stored Procedures

You now know three tools: LINQ, raw SQL, and stored procedures. When should you use each? Here's a decision framework:

Decision Framework
// 1. SIMPLE CRUD // Use: LINQ var customer = await context.Customers .FirstOrDefaultAsync( c => c.Id == 1); customer.Name = "Updated"; await context.SaveChangesAsync(); // 2. COMPLEX QUERIES // (window functions, CTEs, etc) // Use: Raw SQL var results = await context.Orders .FromSqlInterpolated( $"SELECT * FROM Orders " + $"WHERE rank OVER ...") .ToListAsync(); // 3. BULK OPERATIONS // Use: ExecuteSqlRaw await context.Database .ExecuteSqlRawAsync( "UPDATE ... WHERE ..."); // 4. STORED PROCEDURES // Use: Raw SQL if already exists // or FromSqlRaw()
SQL Approaches
-- Simple SELECT: SELECT * FROM Customers WHERE Id = 1; -- Complex query with window -- functions: SELECT *, ROW_NUMBER() OVER (PARTITION BY Category ORDER BY Price) FROM Products; -- Bulk update: UPDATE Products SET Price *= 1.1 WHERE Category='Electronics'; -- Or use stored procedures -- for complex business logic EXEC sp_ProcessOrders;

Quick Decision Table:

Scenario Tool Why
Simple CRUD LINQ Automatic change tracking, type-safe, readable
Window functions, CTEs, complex aggregation Raw SQL LINQ can't express these; SQL is clearer
Bulk updates/deletes ExecuteSqlRaw Don't load entities; single statement
Existing stored procedures Raw SQL or Stored Proc Extension Use what already works; don't force LINQ
Mix and Match Strategically
The beauty of EF Core is that you can mix LINQ, raw SQL, and stored procedures in the same application. Use LINQ for simple operations where change tracking is valuable. Use raw SQL for complex queries. Use stored procedures only if they already exist and are well-tested.

Query Patterns: LINQ vs Raw SQL vs Stored Procedures

Here are three common query patterns: a simple selection, a window function query, and a bulk operation. Each shows the LINQ approach, raw SQL approach, and (where applicable) stored procedure approach.

Pattern 1: Simple Filter + Order (LINQ)
// BEST: LINQ for simple queries var activeProducts = await context.Products .Where(p => p.IsActive) .OrderByDescending(p => p.Price) .Take(20) .ToListAsync(); // Generates efficient SQL // Full change tracking // Easy to read and maintain
Pattern 1: Raw SQL Alternative
// ALTERNATIVE: Same query in SQL var products = await context.Products .FromSqlRaw( @"SELECT TOP 20 * FROM Products WHERE IsActive = 1 ORDER BY Price DESC") .ToListAsync(); // More control over SQL // But harder to compose // Prefer LINQ here
Pattern 1: Generated SQL
SELECT TOP 20 * FROM Products WHERE IsActive = 1 ORDER BY Price DESC;
Pattern 2: Window Function (Raw SQL)
// LINQ cannot express // window functions var rankedProducts = await context.Products .FromSqlRaw( @"SELECT *, ROW_NUMBER() OVER (PARTITION BY Category ORDER BY Price DESC) AS PriceRank FROM Products WHERE IsActive = 1") .ToListAsync(); // Raw SQL is only option here // LINQ can't translate this
Pattern 2: With LINQ Composition
// Add LINQ filtering on top var topRanked = await context.Products .FromSqlRaw( @"SELECT *, ROW_NUMBER() OVER (PARTITION BY Category ORDER BY Price DESC) AS PriceRank FROM Products") .Where(p => p.PriceRank <= 5) .ToListAsync(); // EF wraps SQL as subquery, // then applies WHERE
Pattern 2: Window Function SQL
SELECT *, ROW_NUMBER() OVER ( PARTITION BY Category ORDER BY Price DESC ) AS PriceRank FROM Products WHERE IsActive = 1;
Pattern 3: Bulk Update (ExecuteUpdate)
// BEST (EF Core 7+): // ExecuteUpdate for bulk ops var count = await context.Products .Where(p => p.Category == "Electronics") .ExecuteUpdateAsync(s => s.SetProperty( p => p.Price, p => p.Price * 1.1m)); // Type-safe, LINQ-based // No entity loading // Extremely efficient
Pattern 3: Raw SQL Alternative
// LEGACY: ExecuteSqlRaw await context.Database .ExecuteSqlRawAsync( @"UPDATE Products SET Price = Price * {0} WHERE Category = {1}", 1.1m, "Electronics"); // Works, but raw SQL // Manual parameters // Prefer ExecuteUpdate
Pattern 3: Bulk Update SQL
UPDATE Products SET Price = Price * 1.1 WHERE Category = 'Electronics';

Performance Comparison: Raw SQL vs LINQ vs Stored Procedures

Performance varies depending on query complexity. LINQ-generated SQL is usually optimal for simple queries. Raw SQL is faster for complex operations that LINQ generates inefficiently. Stored procedures offer no inherent performance advantage in modern databases with good query optimizers.

Performance Test: Three Approaches 
// Scenario: Get active customers and their order count

// APPROACH 1: LINQ (Simplest, often optimal)
var sw = Stopwatch.StartNew();
var customers1 = await context.Customers
  .Where(c => c.IsActive)
  .Include(c => c.Orders)
  .Select(c => new {
    c.Id,
    c.Name,
    OrderCount = c.Orders.Count
  })
  .ToListAsync();
sw.Stop();
Console.WriteLine($"LINQ: {sw.ElapsedMilliseconds}ms");
// Result: 45ms (depends on row count)

// APPROACH 2: Raw SQL with LEFT JOIN
sw.Restart();
var customers2 = await context.Customers
  .FromSqlRaw(@"
    SELECT DISTINCT c.Id, c.Name,
      COUNT(o.Id) AS OrderCount
    FROM Customers c
    LEFT JOIN Orders o ON c.Id = o.CustomerId
    WHERE c.IsActive = 1
    GROUP BY c.Id, c.Name")
  .ToListAsync();
sw.Stop();
Console.WriteLine($"Raw SQL: {sw.ElapsedMilliseconds}ms");
// Result: 38ms (often faster for aggregations)

// APPROACH 3: Stored Procedure
sw.Restart();
var customers3 = await context.Customers
  .FromSqlRaw("EXEC sp_GetActiveCustomersWithOrders")
  .ToListAsync();
sw.Stop();
Console.WriteLine($"Stored Proc: {sw.ElapsedMilliseconds}ms");
// Result: 42ms (usually not faster than raw SQL)

// Analysis:
// - Raw SQL wins by ~7ms (15% faster)
// - This is significant at scale (millions of rows)
// - But LINQ is more maintainable
// - Use LINQ first; optimize with raw SQL if needed
When Raw SQL Wins: Complex Aggregations 
// Scenario: Customer sales statistics by region

// LINQ version (less efficient)
var stats = await context.Orders
  .Where(o => o.OrderDate >= startDate)
  .GroupBy(o => o.Customer.Region)
  .Select(g => new {
    Region = g.Key,
    TotalSales = g.Sum(o => o.Total),
    AvgOrderValue = g.Average(o => o.Total),
    OrderCount = g.Count(),
    CustomerCount = g.Select(o => o.CustomerId).Distinct().Count()
  })
  .ToListAsync();
// Result: 125ms
// LINQ generates subqueries for distinct customer count

// Raw SQL version (optimized)
var stats2 = await context.Set()
  .FromSqlRaw(@"
    SELECT
      c.Region,
      SUM(o.Total) AS TotalSales,
      AVG(o.Total) AS AvgOrderValue,
      COUNT(o.Id) AS OrderCount,
      COUNT(DISTINCT o.CustomerId) AS CustomerCount
    FROM Orders o
    JOIN Customers c ON o.CustomerId = c.Id
    WHERE o.OrderDate >= @date
    GROUP BY c.Region")
  .ToListAsync();
// Result: 85ms
// SQL optimizer handles all aggregations in one pass
// ~32% faster for this complex query

// Key insight:
// - LINQ good for simple operations
// - Raw SQL better for multi-level aggregations
// - Profile before optimizing
Bulk Operations: ExecuteUpdate Wins 
// Scenario: Mark 100,000 old orders as archived

// APPROACH 1: LINQ + SaveChanges (BAD)
var cutoff = DateTime.Now.AddYears(-1);
var oldOrders = await context.Orders
  .Where(o => o.CreatedDate < cutoff)
  .ToListAsync(); // Loads all 100k rows!

foreach (var order in oldOrders)
  order.Status = "Archived";

await context.SaveChangesAsync();
// Time: 8500ms (8.5 seconds)
// Memory: 500+ MB
// Change tracking overhead is killer

// APPROACH 2: ExecuteUpdate (GOOD)
var count = await context.Orders
  .Where(o => o.CreatedDate < cutoff)
  .ExecuteUpdateAsync(s =>
    s.SetProperty(o => o.Status, "Archived"));
// Time: 450ms
// Memory: <1 MB
// ~19x faster, orders of magnitude less memory

// APPROACH 3: Raw SQL (equivalent)
var count = await context.Database
  .ExecuteSqlRawAsync(
    @"UPDATE Orders SET Status = 'Archived'
      WHERE CreatedDate < @cutoff",
    new SqlParameter("@cutoff", cutoff));
// Time: 450ms (same as ExecuteUpdate)
// But ExecuteUpdate is more type-safe

// Key insight:
// - Never load large datasets just to update them
// - ExecuteUpdate/ExecuteDelete are essential
// - For bulk ops, raw SQL and ExecuteUpdate are equivalent
Performance Rule of Thumb
Start with LINQ. It's readable, maintainable, and generates good SQL. Profile your queries with a tool like SQL Profiler or Glimpse. If a query is slow, inspect the generated SQL. If LINQ generates a suboptimal query, switch to raw SQL. Stored procedures rarely offer performance advantages over well-optimized raw SQL. The modern database query optimizer is sophisticated; trust it.

Key Takeaways

  • FromSqlRaw() executes SQL but returns IQueryable<T>. You can chain LINQ operators like .Where(), .Include(), and .OrderBy() on top. EF wraps your SQL as a subquery.
  • FromSqlInterpolated() uses string interpolation to safely parameterize SQL. Variables become SQL parameters, not concatenated strings. Always use this for user input instead of FromSqlRaw() with concatenation.
  • ExecuteSqlRaw() executes non-query commands: bulk updates, DDL, stored procedures. It returns the number of rows affected. Use for operations that don't need entity tracking.
  • Raw SQL + LINQ composition is powerful. Start with complex SQL logic and add LINQ filtering on top. Your SQL must return all entity columns, properly named.
  • Database views can be queried in EF with .ToView() and .HasNoKey(). Views are read-only projections; TVFs are parameterized stored queries that EF can call.
  • Use LINQ for simple CRUD, raw SQL for complex queries, ExecuteSqlRaw for bulk operations. Don't force all queries through LINQ if SQL is clearer and more efficient.
  • Never concatenate user input directly into SQL. Always use parameterization with FromSqlInterpolated() or explicit parameters to prevent SQL injection.
