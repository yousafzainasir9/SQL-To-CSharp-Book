Chapter 16

Raw SQL in EF

Requirement: "This query is too complex for LINQ." Sometimes you need raw SQL power inside EF Core. FromSqlInterpolated for queries, ExecuteSqlInterpolated for commands. You get full SQL control while keeping EF's entity mapping.

01

Requirement: "Run a Complex Query That LINQ Can't Express"

Business Requirement
Calculate each customer's total spend and rank them by spend within their region using a window function. LINQ doesn't support window functions natively.
SQL Developer Thinks This is perfect for a window function. ROW_NUMBER() OVER (PARTITION BY Region ORDER BY Total DESC) ranks customers by spend within each region. Write the SQL and let the database do it fast.
C# Developer Thinks LINQ can't express window functions, so I'll use FromSqlInterpolated() to pass raw SQL directly. EF still maps the result to Customer entities. Always use interpolated (parameterized) queries, never raw string concat.
SQL — Window Function
SELECT
    Id, Name, Email, Region,
    SUM(Amount) AS TotalSpend,
    ROW_NUMBER() OVER (
        PARTITION BY Region
        ORDER BY SUM(Amount) DESC
    ) AS RankInRegion
FROM Customers
GROUP BY Id, Name, Email, Region
ORDER BY Region, RankInRegion;
C# — FromSqlInterpolated
var results = await db.Customers
    .FromSqlInterpolated($@"
        SELECT Id, Name, Email,
               Region, SUM(Amount)
               AS TotalSpend,
               ROW_NUMBER() OVER (
                   PARTITION BY Region
                   ORDER BY SUM(Amount) DESC
               ) AS RankInRegion
        FROM Customers
        GROUP BY Id, Name, Email,
                 Region")
    .ToListAsync(ct);
Key Point: Always use FromSqlInterpolated() or ExecuteSqlInterpolated() with C# string interpolation. Never concatenate SQL strings manually—parameterization prevents SQL injection.
02

Requirement: "Add LINQ Filters on Top of Raw SQL"

Business Requirement
Run a complex raw SQL base query, then dynamically add filters like .Where(), .OrderBy(), and pagination on top of it.
SQL Developer Thinks Build a WHERE clause dynamically to add extra filters. Could use subqueries or CTEs if it gets complex, but EF will wrap my raw SQL in a subquery anyway.
C# Developer Thinks Chain LINQ operators after FromSqlInterpolated(). EF wraps the raw SQL in a subquery and applies the LINQ filters on top. This gives me both SQL power and C# flexibility.
C# — FromSql + LINQ Chaining
var customers = await db.Customers
    .FromSqlInterpolated($@"
        SELECT Id, Name, Email, Region,
               ISNULL(LastPurchaseDate,
                      CAST('1900-01-01' AS DATETIME2))
               AS LastPurchaseDate
        FROM Customers")
    .Where(c => c.Region == "West")
    .Where(c => c.LastPurchaseDate > DateTime.Now.AddMonths(-6))
    .OrderByDescending(c => c.LastPurchaseDate)
    .Skip(skip)
    .Take(pageSize)
    .ToListAsync(ct);
What EF Generates EF wraps your raw SQL in a subquery, then applies WHERE and ORDER BY on top. The database gets a single, optimized query.
Best Practice Use this pattern when you need a complex base query (window functions, CTEs, subqueries) but also need dynamic filtering and sorting from C#.
03

Requirement: "Run an UPDATE or DELETE Without Loading Entities"

Business Requirement
Deactivate 10,000 stale customer accounts based on their last login date. Loading all entities into memory is wasteful.
SQL Developer Thinks UPDATE table SET column = value WHERE condition. The database does it in one shot, no round trips.
C# Developer Thinks Use ExecuteSqlInterpolatedAsync() instead of FromSqlInterpolated(). It's a non-query command. It returns the number of affected rows, not data.
SQL — Update Command
UPDATE Customers
SET Status = 'Inactive'
WHERE LastLogin < @StaleSinceDate
  AND Status = 'Active';
C# — ExecuteSqlInterpolated
var staleSinceDate = DateTime.Now
    .AddMonths(-6);

int affectedRows = await db.Database
    .ExecuteSqlInterpolatedAsync(
        $@"UPDATE Customers
           SET Status = 'Inactive'
           WHERE LastLogin < {staleSinceDate}
             AND Status = 'Active'",
        ct);
C# — Delete Command
int deletedRows = await db.Database
    .ExecuteSqlInterpolatedAsync(
        $@"DELETE FROM AuditLogs
           WHERE CreatedAt < {oldCutoffDate}",
        ct);
Performance Win No SELECT, no round trips, no entity tracking. The database processes millions of rows instantly.
Return Value ExecuteSqlInterpolatedAsync() returns int—the count of affected rows. Use it for validation: if you expected 10,000 rows to update but only 5,000 were, something's wrong.
04

Requirement: "Call a Stored Procedure"

Business Requirement
DBA wrote a complex stored procedure for a monthly customer report. We need to call it from C# and map the result to entities.
SQL Developer Thinks EXEC sp_GetCustomerReport @StartDate, @EndDate. The stored procedure handles complex logic, joins, aggregations. C# just needs to invoke it.
C# Developer Thinks FromSqlInterpolated() for stored procs that return results. ExecuteSqlInterpolated() for stored procs that don't. EF handles the parameter binding.
SQL — Stored Procedure
CREATE PROCEDURE sp_GetCustomerReport
    @StartDate DATETIME2,
    @EndDate DATETIME2
AS
SELECT
    CustomerId, Name, Email,
    SUM(Amount) AS TotalSpent,
    COUNT(*) AS OrderCount
FROM Orders
WHERE OrderDate BETWEEN @StartDate AND @EndDate
GROUP BY CustomerId, Name, Email
ORDER BY TotalSpent DESC;
C# — Call Stored Proc
var start = new DateTime(2024, 1, 1);
var end = new DateTime(2024, 12, 31);

var results = await db.Customers
    .FromSqlInterpolated(
        $@"EXEC sp_GetCustomerReport
              @StartDate = {start},
              @EndDate = {end}")
    .ToListAsync(ct);
C# — Stored Proc Returning Nothing
// Stored proc that just does work
// (archival, cleanup, etc.)
int rowsAffected = await db.Database
    .ExecuteSqlInterpolatedAsync(
        $@"EXEC sp_ArchiveOldOrders
              @CutoffDate = {cutoffDate}",
        ct);
05

Requirement: "Map Raw SQL Results to a Custom DTO"

Business Requirement
Report query returns custom computed columns that don't match any entity. Map results to a read-only DTO instead.
SQL Developer Thinks SELECT computed columns: employee name, department, salary, bonus, tax-adjusted income. These don't map to a single entity.
C# Developer Thinks Create a [Keyless] entity (or use .HasNoKey() in config) as a DTO. Register it in DbContext. Use FromSqlInterpolated() just like a normal entity.
C# — Keyless DTO Entity
[Keyless]
public class EmployeeReport
{
    public string EmployeeName { get; set; } = "";
    public string DepartmentName { get; set; } = "";
    public decimal BaseSalary { get; set; }
    public decimal BonusAmount { get; set; }
    public decimal TaxAdjustedIncome { get; set; }
}
C# — Register in DbContext
public class AppDbContext : DbContext
{
    public DbSet<Employee> Employees { get; set; }
    public DbSet<Department> Departments { get; set; }

    // Register the keyless entity
    public DbSet<EmployeeReport> EmployeeReports { get; set; }

    protected override void OnModelCreating(
        ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<EmployeeReport>()
            .HasNoKey();
    }
}
C# — Query with FromSql
var reports = await db.EmployeeReports
    .FromSqlInterpolated($@"
        SELECT
            e.Name AS EmployeeName,
            d.Name AS DepartmentName,
            e.Salary AS BaseSalary,
            ISNULL(b.Amount, 0) AS BonusAmount,
            e.Salary + ISNULL(b.Amount, 0)
                - (e.Salary * 0.25) AS TaxAdjustedIncome
        FROM Employees e
        JOIN Departments d ON e.DepartmentId = d.Id
        LEFT JOIN Bonuses b ON e.Id = b.EmployeeId
        WHERE d.IsActive = 1")
    .ToListAsync(ct);
Column Name Matching SQL column names must match C# property names (case-insensitive). Use AS aliases if needed.
Read-Only Pattern [Keyless] entities are ideal for reports and projections. No primary key, no change tracking, no insert/update/delete.
06

Requirement: "Debug What SQL EF Is Generating"

Business Requirement
Query is slow. Need to see the actual SQL EF generates, not just the LINQ code.
SQL Developer Thinks Use SQL Profiler to capture what the app sends to the database. See the exact query, parameters, and execution time.
C# Developer Thinks Enable EF logging to intercept and print SQL. Use .ToQueryString() to inspect a single query. This reveals what EF actually sends to the database.
C# — ToQueryString() Method
var query = db.Customers
    .Where(c => c.Status == "Active")
    .OrderByDescending(c => c.CreatedAt)
    .Select(c => new { c.Id, c.Name });

// See the SQL without executing it
string generatedSql = query.ToQueryString();
Console.WriteLine(generatedSql);

// Then execute
var results = await query.ToListAsync(ct);
C# — Enable EF Logging in ConfigureServices
services.AddDbContext<AppDbContext>(options =>
{
    options
        .UseSqlServer(connectionString)
        .LogTo(
            Console.WriteLine,
            LogLevel.Information,
            DbContextLoggerOptions.DefaultWithQueryBuilder);
});
C# — Custom Logging Handler
services.AddDbContext<AppDbContext>(options =>
{
    options
        .UseSqlServer(connectionString)
        .LogTo(sql =>
        {
            // Log to file, monitoring system, console
            File.AppendAllText("sql.log",
                $"{DateTime.Now:G} | {sql}\n");
            Console.WriteLine($"SQL: {sql}");
        },
        LogLevel.Information);
});
Parameter Values Logging shows the SQL query structure. To see actual parameter values, enable DbContextLoggerOptions.UtcTime or use SQL Profiler on the server side.
Performance Insight ToQueryString() is zero-cost—it doesn't execute. Perfect for debugging before hitting the database. Logging is production-safe if you filter by LogLevel.
