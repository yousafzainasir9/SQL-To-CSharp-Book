Chapter 08

ORDER BY & Sorting

You need results in a specific sequence. SQL uses ORDER BY, C# uses .OrderBy() and .OrderByDescending(). Both control the sequence of results — whether you're showing customers alphabetically or orders newest-first.

01

Requirement: "Show Customers A to Z"

Business Requirement
Display all customers sorted alphabetically by name. Also need newest orders first, sorted by order date descending.
SQL Developer Thinks ORDER BY sorts results. Default is ASC (ascending: A-Z, low-to-high). Add DESC for descending (Z-A, high-to-low). ORDER BY Name ASC sorts names alphabetically. ORDER BY OrderDate DESC shows newest first.
C# Developer Thinks In LINQ, I use .OrderBy() for ascending and .OrderByDescending() for descending. Both take a lambda selecting the column to sort by. EF translates to SQL ORDER BY automatically.
SQL
-- Alphabetically A to Z
SELECT * FROM Customers
ORDER BY Name ASC;

-- Newest orders first
SELECT * FROM Orders
ORDER BY OrderDate DESC;
LINQ Method
// Alphabetically
var customers = await db.Customers
    .AsNoTracking()
    .OrderBy(c => c.Name)
    .ToListAsync(ct);

// Newest first
var orders = await db.Orders
    .AsNoTracking()
    .OrderByDescending(o => o.OrderDate)
    .ToListAsync(ct);
LINQ Query
// Alphabetically
var customers = await (
    from c in db.Customers.AsNoTracking()
    orderby c.Name
    select c
).ToListAsync(ct);

// Newest first
var orders = await (
    from o in db.Orders.AsNoTracking()
    orderby o.OrderDate descending
    select o
).ToListAsync(ct);
02

Requirement: "Sort by Category, Then by Price"

Business Requirement
Product catalog should group by category name, then sorted by price within each category (highest price first).
SQL Developer Thinks ORDER BY with multiple columns: first column is primary sort, second is secondary (tie-breaker). ORDER BY Category ASC, Price DESC groups by category, then sorts price high-to-low within each.
C# Developer Thinks Chain .OrderBy() with .ThenBy() or .ThenByDescending(). First call is primary, each Then* is a tie-breaker. The order of method calls determines sort priority.
SQL
SELECT * FROM Products
ORDER BY Category ASC,
         Price DESC;
LINQ Method
var products = await db.Products
    .AsNoTracking()
    .OrderBy(p => p.Category)
    .ThenByDescending(p => p.Price)
    .ToListAsync(ct);
LINQ Query
var products = await (
    from p in db.Products
           .AsNoTracking()
    orderby p.Category,
            p.Price descending
    select p
).ToListAsync(ct);
03

Requirement: "Show Computed Rankings"

Business Requirement
Sort customers by total spending (computed from their orders), not a stored column. Highest spenders first.
SQL Developer Thinks ORDER BY can sort by computed expressions. Use (SELECT SUM(Total) FROM Orders WHERE CustomerId = c.Id) DESC to sort by computed total. This is evaluated at query time.
C# Developer Thinks After loading customers, I can sort by a nested aggregate: .OrderByDescending(c => c.Orders.Sum(o => o.Total)). EF translates the sum to a subquery in SQL.
SQL — Correlated Subquery
SELECT c.*
FROM Customers c
ORDER BY (
    SELECT SUM(o.Total)
    FROM Orders o
    WHERE o.CustomerId = c.Id
) DESC;
LINQ Method
var customers = await db.Customers
    .AsNoTracking()
    .OrderByDescending(c =>
        c.Orders.Sum(o => o.Total)
    )
    .ToListAsync(ct);
04

Requirement: "Put Unassigned Items at the End"

Business Requirement
Products without a category should appear last in the sorted list, not first. Category assignments first, unassigned last.
SQL Developer Thinks NULL sorts first (ascending) or last (descending) depending on dialect. To force control, use CASE WHEN: CASE WHEN CategoryId IS NULL THEN 1 ELSE 0 END sorts assigned (0) first, NULL (1) last.
C# Developer Thinks I can use .OrderBy(p => p.CategoryId == null) which sorts false (assigned) first, then true (null) last. Then chain .ThenBy(p => p.CategoryId) for the actual sort.
SQL — CASE for NULL Control
SELECT * FROM Products
ORDER BY CASE
    WHEN CategoryId IS NULL THEN 1
    ELSE 0
    END,
    CategoryId;
LINQ — Null-Aware Sort
var products = await db.Products
    .AsNoTracking()
    .OrderBy(p =>
        p.CategoryId == null)
    .ThenBy(p => p.CategoryId)
    .ToListAsync(ct);
05

Requirement: "Show Page 3 of Results"

Business Requirement
Product listing displays 20 items per page. User clicks page 3, so skip the first 40 items and fetch the next 20.
SQL Developer Thinks OFFSET/FETCH: OFFSET 40 ROWS FETCH NEXT 20 ROWS ONLY. Or in MySQL/PostgreSQL: LIMIT 20 OFFSET 40. For better performance on large offsets, keyset pagination (WHERE Id > @lastId) is faster than scanning.
C# Developer Thinks Use .Skip(40).Take(20) — EF translates to OFFSET/FETCH. For pagination, skip = (pageNumber - 1) * pageSize. For large offsets, implement keyset (cursor-based) pagination with WHERE Id > lastId.
SQL — OFFSET/FETCH
-- Page 3: skip 40, take 20
SELECT * FROM Products
ORDER BY CreatedAt DESC
OFFSET 40 ROWS
FETCH NEXT 20 ROWS ONLY;
LINQ Method
var pageSize = 20;
var pageNum = 3;
var skip = (pageNum - 1) * pageSize;

var products = await db.Products
    .AsNoTracking()
    .OrderByDescending(p =>
        p.CreatedAt)
    .Skip(skip)
    .Take(pageSize)
    .ToListAsync(ct);
LINQ Query
var pageSize = 20;
var pageNum = 3;
var skip = (pageNum - 1) * pageSize;

var products = await (
    from p in db.Products
           .AsNoTracking()
    orderby p.CreatedAt descending
    select p
).Skip(skip)
 .Take(pageSize)
 .ToListAsync(ct);
Keyset Pagination (Faster for Large Offsets)
public async Task<List<ProductDTO>> GetProductsKeyset(
    int? lastProductId = null,
    int pageSize = 20,
    CancellationToken ct = default)
{
    IQueryable<Product> query = db.Products
        .AsNoTracking()
        .OrderByDescending(p => p.CreatedAt);

    if (lastProductId.HasValue)
        query = query.Where(p => p.Id > lastProductId.Value);

    return await query
        .Take(pageSize)
        .Select(p => new ProductDTO {
            Id = p.Id,
            Name = p.Name,
            CreatedAt = p.CreatedAt
        })
        .ToListAsync(ct);
}
06

Sort Performance and Indexes

Business Requirement
Some sorts are lightning-fast, others are slow. Understand which columns to index for common sort patterns.
Sort TypePerformanceReason
ORDER BY Id✓ FastPrimary key always indexed
ORDER BY CreatedAt✓ FastIf index exists on CreatedAt
ORDER BY Status, CreatedAt✓ FastComposite index on (Status, CreatedAt)
ORDER BY Name.ToUpper()✗ SlowFunction prevents index use; full scan
ORDER BY Price * Quantity✗ SlowComputed expression; in-memory sort
ORDER BY (SELECT SUM(...))✗ SlowSubquery evaluated per row
SQL — Create Indexes
-- Single-column index
CREATE INDEX IX_Products_CreatedAt
    ON Products(CreatedAt);

-- Composite index
CREATE INDEX IX_Orders_StatusDate
    ON Orders(Status, CreatedAt DESC);

-- Filtered index (only active)
CREATE INDEX IX_Products_Active_Name
    ON Products(Name)
    WHERE IsActive = 1;
EF — Fluent API Configuration
modelBuilder.Entity<Product>(e =>
{
    e.HasIndex(p => p.CreatedAt);

    e.HasIndex(p => new {
        p.Status,
        p.CreatedAt
    }).IsDescending(false, true);

    e.HasIndex(p => p.Name)
     .HasFilter("[IsActive] = 1");
});
