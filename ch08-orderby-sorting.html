01

Basic Sorting with ASC and DESC

ORDER BY specifies the sort order of result rows. By default, sorting is ascending (smallest to largest, A to Z). DESC reverses the order to descending. Both SQL and LINQ provide identical functionality with slightly different syntax. The generated SQL is indistinguishable.

SQL ORDER BY 
-- Ascending (default)
SELECT * FROM Products
ORDER BY Price;

-- Descending
SELECT * FROM Products
ORDER BY Price DESC;
Method Syntax 
// Ascending (default)
var ascending = products
  .OrderBy(p => p.Price)
  .ToList();

// Descending
var descending = products
  .OrderByDescending(p => p.Price)
  .ToList();
Query Syntax 
// Ascending
var ascending = (from p in products
                 orderby p.Price
                 select p)
  .ToList();

// Descending
var descending = (from p in products
                  orderby p.Price descending
                  select p)
  .ToList();

Sorting different data types:

SQL Various Types 
-- Alphabetical (strings)
SELECT * FROM Customers
ORDER BY Name;

-- Numeric (numbers)
SELECT * FROM Orders
ORDER BY Total DESC;

-- Chronological (dates)
SELECT * FROM Events
ORDER BY EventDate ASC;
C# Various Types 
// Alphabetical
var byName = customers
  .OrderBy(c => c.Name)
  .ToList();

// Numeric descending
var byTotal = orders
  .OrderByDescending(o => o.Total)
  .ToList();

// Chronological ascending
var byDate = events
  .OrderBy(e => e.EventDate)
  .ToList();
Default Order Matters

Without ORDER BY, result order is undefined. The database may return rows in any order. Always specify ORDER BY when result order matters. For consistency, prefer ascending order (the SQL default) unless descending is explicitly required.

02

Multi-Column Sorting with ThenBy

Often you sort by multiple columns: first by category, then by price within each category. SQL uses multiple comma-separated columns. LINQ uses .OrderBy() followed by .ThenBy() for each additional sort level. A critical mistake in LINQ: calling .OrderBy() multiple times replaces the previous sort, not adds to it. Always use .ThenBy() for additional sort criteria.

SQL 
SELECT *
FROM Products
ORDER BY Category ASC, Price DESC,
  Name ASC;
Method Syntax (CORRECT) 
var result = products
  .OrderBy(p => p.Category)
  .ThenByDescending(p => p.Price)
  .ThenBy(p => p.Name)
  .ToList();

// Key: use ThenBy, not multiple OrderBy
Query Syntax 
var result = (from p in products
              orderby p.Category,
                      p.Price descending,
                      p.Name
              select p)
  .ToList();

Common mistake: multiple OrderBy calls (WRONG):

INCORRECT: Multiple OrderBy (Replaces) 
// BAD: Second OrderBy replaces the first!
var result = products
  .OrderBy(p => p.Category)  // Sort by category
  .OrderBy(p => p.Price)     // REPLACES category sort!
  .ToList();

// Final result: sorted ONLY by Price
// Category sort is lost
CORRECT: OrderBy + ThenBy (Chains) 
// GOOD: ThenBy adds a secondary sort
var result = products
  .OrderBy(p => p.Category)    // First sort
  .ThenBy(p => p.Price)        // Then sort within category
  .ToList();

// Final result: sorted by category, then price
// Both sorts are applied

Complex multi-column scenarios:

Multi-Level Sorting 
var result = context.Orders
  .OrderBy(o => o.CustomerId)
  .ThenByDescending(o => o.OrderDate)
  .ThenBy(o => o.Id)
  .ToList();

// SQL equivalent:
// ORDER BY CustomerId ASC, OrderDate DESC, Id ASC

// Real-world example: sales report
var salesReport = context.Transactions
  .Where(t => t.Date >= startDate)
  .OrderBy(t => t.SalesRep)      // Salesperson
  .ThenByDescending(t => t.Amount) // Largest first
  .ThenBy(t => t.Date)            // Then chronological
  .ToList();
ThenBy, Not OrderBy (Critical)

This is a frequent source of bugs: using .OrderBy() twice replaces the first sort. Always use .ThenBy() for additional sort criteria. Method chaining is order-sensitive: each .ThenBy() adds a new sort tier.

03

Sorting by Computed Expressions

You often need to sort by a computed value, not a stored column. SQL uses expressions in ORDER BY. LINQ accepts any lambda expression. Entity Framework translates these to SQL equivalents at the database level, which is far more efficient than sorting in C#.

SQL 
SELECT *
FROM OrderItems
ORDER BY Quantity * UnitPrice DESC;
Method Syntax 
var result = items
  .OrderByDescending(oi =>
    oi.Quantity * oi.UnitPrice)
  .ToList();
Query Syntax 
var result = (from oi in items
              orderby oi.Quantity
                * oi.UnitPrice descending
              select oi)
  .ToList();

Complex computed sorts:

SQL Complex Expression 
SELECT *
FROM Customers
ORDER BY
  CASE
    WHEN Status = 'VIP' THEN 1
    WHEN Status = 'Premium' THEN 2
    ELSE 3
  END,
  TotalSpent DESC;
C# Complex Expression 
var result = customers
  .OrderBy(c => c.Status == "VIP" ? 1
    : c.Status == "Premium" ? 2 : 3)
  .ThenByDescending(c => c.TotalSpent)
  .ToList();

// EF translates the ternary to CASE in SQL
Computed Sorts at the Database

Entity Framework translates computed sort expressions to SQL, meaning the database performs the sorting. This is far faster than fetching all rows and sorting in C#. Keep your lambda expressions simple (comparisons, arithmetic, string methods) to ensure they translate to SQL. Test with .ToQueryString() to verify.

04

NULL Handling in Sorting

NULL values complicate sorting. Different databases handle NULLs differently: SQL Server puts NULLs first in ascending order; PostgreSQL puts them last. This inconsistency can lead to surprising result variations across databases. C# can handle NULL explicitly using null coalescing and conditional sorting.

Database-specific NULL behavior:

SQL NULL Ordering (Database Dependent) 
-- SQL Server: NULLs first in ascending
SELECT * FROM Customers
ORDER BY MiddleName ASC;
-- Result: [NULL, NULL, ..., "Alice", "Bob", ...]

-- PostgreSQL: NULLs last in ascending
SELECT * FROM Customers
ORDER BY MiddleName ASC;
-- Result: ["Alice", "Bob", ..., NULL, NULL, ...]

-- To force consistent behavior:
-- SQL Server: make NULLs last
SELECT * FROM Customers
ORDER BY CASE WHEN MiddleName IS NULL THEN 1
             ELSE 0 END, MiddleName ASC;

-- PostgreSQL: make NULLs first
SELECT * FROM Customers
ORDER BY MiddleName IS NULL DESC, MiddleName ASC;

Handling NULLs in C#:

C# NULL Handling (Database Independent) 
// Option 1: Sort NULLs to the end
var result = customers
  .OrderBy(c => c.MiddleName == null ? 1 : 0)
  .ThenBy(c => c.MiddleName)
  .ToList();

// Option 2: Sort NULLs to the beginning
var result2 = customers
  .OrderBy(c => c.MiddleName == null ? 0 : 1)
  .ThenBy(c => c.MiddleName)
  .ToList();

// Option 3: Replace NULL with default value
var result3 = customers
  .OrderBy(c => c.MiddleName ?? "")
  .ToList();
Generated SQL (Option 1) 
-- EF generates CASE to handle NULL consistently
SELECT *
FROM Customers AS c
ORDER BY
  CASE WHEN c.MiddleName IS NULL THEN 1
       ELSE 0 END,
  c.MiddleName ASC;
NULL Sorting Is Database-Specific

If result order consistency matters across databases (SQL Server, PostgreSQL, MySQL), handle NULLs explicitly in C#. Use conditional sorting to place NULLs first or last as desired. Test on your target database to ensure expected behavior.

05

Pagination: OFFSET/FETCH and Skip/Take

Large result sets require pagination. SQL's OFFSET and FETCH (or LIMIT in other databases) retrieve a specific range of rows. LINQ provides .Skip() and .Take() for the same purpose. Both must follow ORDER BY for consistent results. However, both have performance implications on large datasets that must be understood.

SQL OFFSET/FETCH 
-- SQL Server 2012+
SELECT *
FROM Products
ORDER BY Id
OFFSET 20 ROWS FETCH NEXT 10 ROWS ONLY;

-- MySQL/PostgreSQL
SELECT *
FROM Products
ORDER BY Id
LIMIT 10 OFFSET 20;
Method Syntax 
var pageSize = 10;
var pageNumber = 2; // 0-indexed

var result = products
  .OrderBy(p => p.Id)
  .Skip((pageNumber - 1) * pageSize)
  .Take(pageSize)
  .ToList();
Query Syntax 
var pageSize = 10;
var pageNumber = 2;

var result = (from p in products
              orderby p.Id
              select p)
  .Skip((pageNumber - 1) * pageSize)
  .Take(pageSize)
  .ToList();

Real-world pagination implementation:

Pagination Pattern 
public class PagedResult
{
  public List Items { get; set; }
  public int TotalCount { get; set; }
  public int PageNumber { get; set; }
  public int PageSize { get; set; }
  public int TotalPages =>
    (TotalCount + PageSize - 1) / PageSize;
}

public PagedResult GetProducts(
  int pageNumber = 1, int pageSize = 10)
{
  var query = context.Products.AsQueryable();

  // Get total before pagination
  var totalCount = query.Count();

  // Apply pagination
  var items = query
    .OrderBy(p => p.Id)
    .Skip((pageNumber - 1) * pageSize)
    .Take(pageSize)
    .Select(p => new ProductDto {
      Id = p.Id,
      Name = p.Name,
      Price = p.Price
    })
    .ToList();

  return new PagedResult {
    Items = items,
    TotalCount = totalCount,
    PageNumber = pageNumber,
    PageSize = pageSize
  };
}

Performance problem: OFFSET on large pages:

Performance Problem with OFFSET 
// Problem: Getting page 1000 of 100 items per page
// OFFSET 99,900 ROWS
var items = products
  .OrderBy(p => p.Id)
  .Skip(99_900)
  .Take(100)
  .ToList();

// The database must:
// 1. Sort all rows
// 2. Skip first 99,900 (still reads them!)
// 3. Return next 100
// This is slow even with indexes

// Solution: Keyset pagination (see next section)
OFFSET/FETCH Performance Degradation

OFFSET must skip N rows, which the database must still read/evaluate. Page 1000 of 100 items requires skipping 99,900 rows. This gets exponentially slower as you go deeper. For large datasets and user pagination, use keyset pagination instead.

06

Keyset Pagination for Large Datasets

For large result sets, keyset (or cursor) pagination is far more efficient than OFFSET. Instead of skipping rows, you remember the last key from the previous page and request rows after that key. This uses indexes efficiently and maintains consistent performance regardless of page depth.

Keyset pagination pattern:

Keyset Pagination Implementation 
public class KeysetPagedResult
{
  public List Items { get; set; }
  public int LastId { get; set; } // Cursor for next page
  public bool HasMore { get; set; }
}

// First page
public KeysetPagedResult GetProducts(
  int pageSize = 10,
  int? lastId = null)
{
  var query = context.Products.AsQueryable();

  // If we have a cursor, start after it
  if (lastId.HasValue)
  {
    query = query.Where(p => p.Id > lastId.Value);
  }

  // Fetch pageSize + 1 to detect if there are more
  var items = query
    .OrderBy(p => p.Id)
    .Take(pageSize + 1)
    .Select(p => new { p.Id, p.Name, p.Price })
    .ToList();

  var hasMore = items.Count > pageSize;
  if (hasMore)
  {
    items = items.Take(pageSize).ToList();
  }

  return new KeysetPagedResult {
    Items = items.Select(i => new ProductDto {
      Id = i.Id,
      Name = i.Name,
      Price = i.Price
    }).ToList(),
    LastId = items.Last().Id,
    HasMore = hasMore
  };
}

// Usage:
var page1 = GetProducts(pageSize: 10);
// Send user to page 2 with LastId = page1.LastId
var page2 = GetProducts(pageSize: 10, lastId: page1.LastId);

Comparing approaches:

OFFSET vs Keyset: Performance Comparison 
// OFFSET approach: Gets slower with each page
var page = 100;
var pageSize = 10;
var items = products
  .OrderBy(p => p.Id)
  .Skip((page - 1) * pageSize)  // Skip 990 rows
  .Take(pageSize)
  .ToList();
// Cost: O(skip + take) = O(1000) per request

// Keyset approach: Consistent performance
var items = products
  .Where(p => p.Id > lastId)
  .OrderBy(p => p.Id)
  .Take(pageSize)
  .ToList();
// Cost: O(index seek + take) = O(10) always
// Uses index efficiently, no skip needed
Keyset Pagination > OFFSET for Scale

For user-facing pagination on large tables (millions of rows), use keyset pagination. It maintains constant performance regardless of page depth. OFFSET is fine for small datasets or admin interfaces with limited page depth. For infinite scroll patterns, keyset is essential.

Index Usage and Sort Performance

ORDER BY performance depends critically on indexes. A sort on an indexed column uses the index to retrieve rows in sorted order (cost: O(log N) seek + O(rows) scan). A sort on a non-indexed column requires a full table scan and in-memory sort using tempdb (cost: O(N log N) sort). Understanding this difference is essential for writing performant queries.

Index-assisted sort (fast):

Index-Assisted Sort Performance 
// Assuming index on Products.Category
var result = context.Products
  .OrderBy(p => p.Category)
  .ToList();

// Generated SQL:
// SELECT *
// FROM Products
// ORDER BY Category
//
// Execution plan:
// 1. Index seek on Category index (fast O(log N))
// 2. Ordered scan of index (returns rows in order)
// 3. Total cost: O(N) - very efficient
//
// This is fast even with millions of rows
// because the index already contains sorted keys

Non-indexed sort (slow):

Non-Indexed Sort (Full Scan Required) 
// No index on Products.Description
var result = context.Products
  .OrderBy(p => p.Description)
  .ToList();

// Generated SQL:
// SELECT *
// FROM Products
// ORDER BY Description
//
// Execution plan:
// 1. Full table scan (read all rows into memory)
// 2. Sort in memory/tempdb (O(N log N))
// 3. Return sorted results
//
// This is slow with millions of rows
// because no index exists to help ordering
Index Impact on Sort Performance 
-- Create indexes to optimize sorts
CREATE INDEX idx_product_category
ON Products(Category);

CREATE INDEX idx_order_date
ON Orders(OrderDate DESC);

-- These indexes make sorting by Category
-- and OrderDate extremely fast

-- Complex index for multi-column sort
CREATE INDEX idx_customer_status_spent
ON Customers(Status, TotalSpent DESC);

-- This index accelerates:
-- ORDER BY Status, TotalSpent DESC
-- Because it's already ordered that way
Sort Performance Depends on Indexes

Sorts on indexed columns are extremely fast (index provides pre-sorted data). Sorts on non-indexed columns require full scans and expensive in-memory sorting. When a query consistently sorts by a column, create an index on that column. For multi-column sorts, create a composite index matching the sort order. Query performance analysis tools (SQL Profiler, Query Execution Plans) reveal whether an index-assisted sort is being used.

08

Database Sort Algorithms and Optimization

Modern databases use sophisticated algorithms to sort data efficiently. The two main approaches are in-memory sorts (quicksort, merge sort) and index-assisted sorts. Understanding which algorithm your database uses helps explain performance characteristics and guides optimization decisions.

For in-memory sorting on non-indexed columns, databases typically use merge sort, which guarantees O(N log N) performance. Quicksort is faster on average but has O(N²) worst-case behavior, which databases avoid. When sorting on indexed columns, the database simply traverses the index in order, achieving O(N) performance because the data is already sorted.

Understanding Sort Strategy Selection 
// Query 1: Indexed column sort (FAST - Index Scan)
var sorted = context.Products
  .OrderBy(p => p.Id)  // Id has primary key index
  .ToList();
// Execution plan: Index Seek → Index Scan
// Cost: O(N) - traverses index in order
// Time: ~10ms for 100k rows

// Query 2: Non-indexed column sort (SLOW - Full Scan + Sort)
var sorted2 = context.Products
  .OrderBy(p => p.Description)  // No index
  .ToList();
// Execution plan: Table Scan → Sort (Merge Sort in tempdb)
// Cost: O(N log N) merge sort in memory/tempdb
// Time: ~500ms for 100k rows
// ~50x slower because of full table scan + sorting

// Query 3: Composite index sort (VERY FAST)
var sorted3 = context.Orders
  .OrderBy(o => o.CustomerId)
  .ThenBy(o => o.OrderDate)
  .ToList();
// If index on (CustomerId, OrderDate) exists:
// Execution plan: Index Seek → Index Scan (already in order)
// Cost: O(N) - index naturally provides both sort orders
// Time: ~5ms for 100k rows

// Rule: Index order matters!
// Index on (A, B) helps:
// - ORDER BY A, B (matches exactly)
// - ORDER BY A (prefix matches)
// But NOT:
// - ORDER BY B (wrong order)
// - ORDER BY A DESC, B ASC (mixed direction)
SQL: Analyze Execution Plan 
-- Check if sort uses index or tempdb
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

-- This uses index (fast)
SELECT * FROM Products
ORDER BY Id;
-- Output: 'Scan count 1' = index scan only

-- This uses tempdb (slow)
SELECT * FROM Products
ORDER BY Description;
-- Output: 'Sort with Advanced Scan'
-- Shows merge sort happening in tempdb

-- Create index to avoid sort
CREATE INDEX idx_description
ON Products(Description);

-- Now this query is fast (index scan instead of sort)
SELECT * FROM Products
ORDER BY Description;
C#: Verify Generated SQL Strategy 
// See the generated SQL execution strategy
var query = context.Products
  .OrderBy(p => p.Description);

// In EF Core, call ToQueryString() to see SQL
string sql = query.ToQueryString();
Console.WriteLine(sql);
// Outputs:
// SELECT * FROM [Products] AS [p]
// ORDER BY [p].[Description]

// Then run in SQL Profiler to see execution plan
// You'll notice:
// - Full Table Scan (no index)
// - Sort operator using tempdb
// - Performance: slow for large tables

// Solution: Create the index, then re-run
// Query will now use Index Scan instead
// Same LINQ code, but vastly faster
The Sort Algorithm Trade-off

Indexes avoid sorting algorithms entirely (data is pre-sorted). Without indexes, databases must use merge sort in tempdb, which works but is slow. For frequently sorted columns, create indexes. The index creation cost (one-time, offline) is worth the query performance gain (every execution, online).

Multi-Column Sorting with Mixed Directions

Sorting by multiple columns with different directions (ascending on one, descending on another) is common. SQL handles this naturally, but LINQ requires careful method chaining. Additionally, indexes can only assist if the index itself has matching sort directions.

SQL Mixed Directions 
-- Sort ascending on one column,
-- descending on another
SELECT *
FROM Orders
ORDER BY CustomerId ASC,
         OrderDate DESC,
         Total ASC;
Method Syntax 
var orders = context.Orders
  .OrderBy(o => o.CustomerId)      // ASC
  .ThenByDescending(o => o.OrderDate)  // DESC
  .ThenBy(o => o.Total)            // ASC
  .ToList();

// Chaining is crucial!
// Each ThenBy() adds a tier
// Order matters
Query Syntax 
var orders = (from o in context.Orders
              orderby o.CustomerId,
                      o.OrderDate descending,
                      o.Total
              select o)
  .ToList();

// Query syntax is more readable
// for complex multi-column sorts

To optimize multi-column sorts with mixed directions, create a composite index with matching directions:

SQL: Composite Index with Directions 
-- Create index matching sort order
CREATE INDEX idx_order_sort
ON Orders(CustomerId ASC, OrderDate DESC, Total ASC);

-- This index makes the query extremely fast
-- The index naturally provides the sort order
-- No merge sort needed
SELECT *
FROM Orders
ORDER BY CustomerId ASC,
         OrderDate DESC,
         Total ASC;

-- Alternative (SQL Server):
-- Use DESC direction in index creation
CREATE INDEX idx_order_sort
ON Orders(CustomerId, OrderDate DESC, Total);
C#: When Index Helps 
// With the composite index above:
var orders = context.Orders
  .OrderBy(o => o.CustomerId)
  .ThenByDescending(o => o.OrderDate)
  .ThenBy(o => o.Total)
  .ToList();

// Execution plan:
// Index Seek (fast, no sort!)
// Time: ~5ms for 100k rows

// Without the index:
// Table Scan → Sort (merge sort)
// Time: ~500ms
// ~100x slower

// Lesson: Match index to sort order exactly
Index Direction Matters for Multi-Column Sorts

A composite index on (A ASC, B DESC) helps ORDER BY A, B DESC but NOT ORDER BY A DESC, B ASC. SQL databases respect index direction. When you consistently sort with mixed directions, create a composite index with matching directions. This is a database-specific optimization (different syntax for SQL Server, PostgreSQL, MySQL).

Custom Sorting with IComparer

Sometimes you need sorting logic that's too complex for SQL. LINQ in C# allows custom IComparer implementations for in-memory sorting. Custom comparers let you define business-specific sort rules that databases don't understand. However, custom sorting happens in C#, not at the database level, so it's slower for large datasets.

Use custom IComparer only after fetching from the database, never on IQueryable. Applying custom sorts to IQueryable will throw an error because the database can't execute C# logic.

Custom IComparer Example 
// Custom comparer: Sort by length of name, then alphabetically
public class CustomNameComparer : IComparer
{
  public int Compare(Product a, Product b)
  {
    // First: compare by name length
    int lengthCompare = a.Name.Length.CompareTo(b.Name.Length);
    if (lengthCompare != 0)
      return lengthCompare;

    // If same length: compare alphabetically
    return a.Name.CompareTo(b.Name);
  }
}

// Use the custom comparer
var products = await context.Products
  .Where(p => p.IsActive)
  .ToListAsync(); // Fetch from DB first!

// NOW apply custom sort (in memory)
var sorted = products
  .OrderBy(p => p, new CustomNameComparer())
  .ToList();

// Result: Products sorted by name length,
// then alphabetically within each length

// Example:
// "Cat"       (3 chars)
// "Dog"       (3 chars, alphabetically after Cat)
// "Elephant"  (8 chars)
// "Tiger"     (5 chars)

Another common use: business-specific sort orders (e.g., status priorities):

Status Priority Comparer 
// Custom: Sort orders by status priority
public class StatusPriorityComparer
  : IComparer
{
  private static readonly Dictionary
    StatusPriority = new()
    {
      { "Urgent", 1 },
      { "High", 2 },
      { "Normal", 3 },
      { "Low", 4 }
    };

  public int Compare(Order a, Order b)
  {
    var priorityA =
      StatusPriority.GetValueOrDefault(
        a.Status, 99);
    var priorityB =
      StatusPriority.GetValueOrDefault(
        b.Status, 99);

    return priorityA.CompareTo(priorityB);
  }
}

// Usage
var orders = await context.Orders
  .ToListAsync();
var prioritized = orders
  .OrderBy(o => o,
    new StatusPriorityComparer())
  .ToList();
Alternative: LINQ OrderBy with Priority 
// Alternative without IComparer:
// Use OrderBy with a computable value
var orders = await context.Orders
  .ToListAsync();

var prioritized = orders
  .OrderBy(o => o.Status switch
  {
    "Urgent" => 1,
    "High" => 2,
    "Normal" => 3,
    "Low" => 4,
    _ => 99
  })
  .ToList();

// Simpler for most cases
// Avoids IComparer class
Custom Comparers: Last Resort

Custom IComparer implementations are powerful but slow—they evaluate entirely in C# after data is fetched. Use them only when SQL cannot express the sort logic. For business priorities or custom rules, use .OrderBy() with a switch expression or computed property. Reserve IComparer for truly complex logic that requires state or recursive evaluation.

NULL Handling: NULLS FIRST/LAST in SQL vs C#

Different databases handle NULLs in sorting differently. SQL Server puts NULLs first in ascending order. PostgreSQL puts them last. This database-specific behavior can surprise you when switching databases. C# offers explicit control over NULL placement using conditional sorting.

SQL: Explicit NULL Control 
-- SQL Server: NULLs appear first by default
SELECT * FROM Customers
ORDER BY MiddleName ASC;
-- Result: [NULL, NULL, Alice, Bob]

-- Force NULLs last in SQL Server
SELECT * FROM Customers
ORDER BY CASE WHEN MiddleName IS NULL
             THEN 1 ELSE 0 END,
         MiddleName ASC;
-- Result: [Alice, Bob, NULL, NULL]

-- PostgreSQL: NULLS LAST is explicit
SELECT * FROM Customers
ORDER BY MiddleName ASC NULLS LAST;
-- Result: [Alice, Bob, NULL, NULL]
Method Syntax: NULL Control 
// C#: Make NULLs explicit
// Option 1: NULLs to end
var result = customers
  .OrderBy(c =>
    c.MiddleName == null ? 1 : 0)
  .ThenBy(c => c.MiddleName)
  .ToList();

// Option 2: NULLs to beginning
var result2 = customers
  .OrderBy(c =>
    c.MiddleName == null ? 0 : 1)
  .ThenBy(c => c.MiddleName)
  .ToList();

// Option 3: Treat NULL as default
var result3 = customers
  .OrderBy(c =>
    c.MiddleName ?? "")
  .ToList();
Query Syntax: NULL Control 
// Query syntax with explicit NULL handling
var result = (from c in customers
              orderby (c.MiddleName == null
                ? 1 : 0),
                      c.MiddleName
              select c)
  .ToList();

// This generates:
// ORDER BY CASE
//   WHEN MiddleName IS NULL THEN 1
//   ELSE 0 END,
//   MiddleName ASC

For cross-database consistency, always handle NULLs explicitly in C# rather than relying on database defaults:

Database-Independent NULL Handling 
// Create a reusable extension method
public static class SortExtensions
{
  public static IOrderedQueryable
    OrderByNullSafe(
      this IQueryable source,
      Expression> keySelector,
      bool nullsLast = true)
    where TKey : struct
  {
    return nullsLast
      ? source
        .OrderBy(x => keySelector.Compile()(x)
          == null ? 1 : 0)
        .ThenBy(keySelector)
      : source
        .OrderBy(x => keySelector.Compile()(x)
          == null ? 0 : 1)
        .ThenBy(keySelector);
  }
}

// Usage - works identically on SQL Server, PostgreSQL
var customers = context.Customers
  .OrderByNullSafe(c => c.BirthDate, nullsLast: true)
  .ToList();

// Result is consistent regardless of database
Database NULL Comparison 
-- SQL Server (default: NULLS first)
SELECT * FROM Customers
ORDER BY BirthDate ASC;
-- NULLs appear first

-- PostgreSQL (default: NULLS last)
SELECT * FROM Customers
ORDER BY BirthDate ASC;
-- NULLs appear last

-- MySQL (default: NULLS first)
SELECT * FROM Customers
ORDER BY BirthDate ASC;
-- NULLs appear first

-- For consistency, explicitly specify:
-- SQL Server
ORDER BY CASE WHEN BirthDate IS NULL THEN 1
             ELSE 0 END, BirthDate ASC

-- PostgreSQL
ORDER BY BirthDate ASC NULLS LAST

-- MySQL (use ISNULL or IF)
ORDER BY IF(BirthDate IS NULL, 1, 0),
         BirthDate ASC
Never Rely on NULL Defaults

Database NULL behavior is inconsistent across vendors. Always handle NULLs explicitly in C# using conditional ordering or coalescing. This ensures result consistency across SQL Server, PostgreSQL, MySQL, and other databases. It's a small addition to your code that prevents surprises in production.

Natural Sorting Patterns

Standard alphabetical sorting (A-Z) is called "lexicographic" sorting. It works for most cases but fails for mixed alphanumeric strings. For example, lexicographic sorting produces: ["File1", "File10", "File2"], which is wrong. "Natural" sorting recognizes numeric portions and produces: ["File1", "File2", "File10"]. SQL databases don't have built-in natural sort support, so this logic lives in C#.

Natural sorting requires fetching data into C# and applying custom logic (IComparer or regex-based sorting). It's not database-efficient for large datasets, so consider whether the client needs it or if the display layer can handle it.

Natural Sort Implementation with IComparer 
// Natural sort comparer for human-friendly ordering
public class NaturalComparer : IComparer
{
  private static readonly Regex NumberRegex =
    new Regex(@"(\d+)", RegexOptions.Compiled);

  public int Compare(string a, string b)
  {
    if (a == b) return 0;
    if (a == null) return -1;
    if (b == null) return 1;

    var aParts = NumberRegex.Split(a);
    var bParts = NumberRegex.Split(b);

    int minLength = Math.Min(
      aParts.Length, bParts.Length);

    for (int i = 0; i < minLength; i++)
    {
      // Try parsing as number
      if (int.TryParse(aParts[i], out int aNum) &&
          int.TryParse(bParts[i], out int bNum))
      {
        int numCompare = aNum.CompareTo(bNum);
        if (numCompare != 0)
          return numCompare;
      }
      else
      {
        // Lexicographic comparison
        int lexCompare = aParts[i]
          .CompareTo(bParts[i]);
        if (lexCompare != 0)
          return lexCompare;
      }
    }

    return aParts.Length.CompareTo(
      bParts.Length);
  }
}

// Usage
var filenames = new[] {
  "File10.txt",
  "File2.txt",
  "File1.txt",
  "Document5.docx",
  "Document20.docx"
};

var sorted = filenames
  .OrderBy(f => f,
    new NaturalComparer())
  .ToArray();

// Result:
// Document5.docx
// Document20.docx
// File1.txt
// File2.txt
// File10.txt

For database queries, natural sorting is expensive. Consider loading a subset of data:

Natural Sort on Query Results (Small Dataset) 
// Good: Natural sort on small results
var products = await context.Products
  .Where(p => p.Category == "Electronics")
  .Take(100)  // Limit to 100 items
  .ToListAsync();

// Now safe to apply natural sort in C#
var sorted = products
  .OrderBy(p => p.Name,
    new NaturalComparer())
  .ToList();

// Time: Fast (only 100 items sorted)
AVOID: Natural Sort on Large Dataset 
// Bad: Loading millions for natural sort
var allProducts = await context.Products
  .ToListAsync(); // Loads ALL rows!

var sorted = allProducts
  .OrderBy(p => p.Name,
    new NaturalComparer())
  .ToList();

// Time: Very slow (millions sorted in C#)
// Memory: Extremely high
// AVOID THIS PATTERN
Natural Sorting Is Client-Side

Databases have no natural sort concept. Natural sorting must happen in C# using custom IComparer logic. This is fine for small result sets (pagination, autocomplete) but not for large datasets. If you need natural sorting on millions of rows, consider whether it's necessary or if display-layer formatting suffices.

Sorting with Include and Query Bloat

When using Include() to eagerly load related entities, ORDER BY can cause unexpected query bloat. The database must return related data alongside sorted rows, which can multiply the result set size. For example, sorting customers with orders includes all orders in the result, potentially returning the same customer data multiple times.

Understanding the interaction between sorting, including, and result multiplication is crucial for performance optimization. Sometimes, it's faster to sort before including, or to use explicit loading instead of eager loading.

Inefficient: Sort After Include 
// Problem: Include multiplies rows
var customers = await context.Customers
  .Include(c => c.Orders)  // Eager load orders
  .OrderBy(c => c.Name)    // Sort by customer name
  .ToListAsync();

// Generated SQL:
// SELECT c.*, o.*
// FROM Customers c
// LEFT JOIN Orders o ON c.Id = o.CustomerId
// ORDER BY c.Name
//
// Problem:
// If customer has 5 orders, that customer
// appears 5 times in the result set
// The database returns 5 rows for that 1 customer
// Then EF client-side deduplicates
// Wasteful!

// Example:
// 1000 customers, 2 orders each
// Database returns 2000 rows
// EF collapses to 1000 customer objects
// ~2x data transfer
Better: Sort Before Include 
// Solution: Defer Include
var customerIds = await context.Customers
  .OrderBy(c => c.Name)
  .Select(c => c.Id)
  .ToListAsync();

var customers = await context.Customers
  .Where(c => customerIds.Contains(c.Id))
  .Include(c => c.Orders)
  .ToListAsync();

// Benefits:
// - First query sorts only customer rows
// - Second query includes orders
// - No result set multiplication during sort
// - Cleaner SQL execution

// OR, if you don't need orders sorted:
var customers = await context.Customers
  .OrderBy(c => c.Name)
  .ToListAsync();

// Lazy load orders later if needed
foreach (var customer in customers)
{
  var orders = await context.Entry(customer)
    .Collection(c => c.Orders)
    .LoadAsync();
}
SQL: Order by Customer 
-- Multiplies rows:
-- 1000 customers * 2 orders = 2000 rows
SELECT c.*, o.*
FROM Customers c
LEFT JOIN Orders o
  ON c.Id = o.CustomerId
ORDER BY c.Name;

-- Result set: 2000 rows
EF: Efficient Approach 
// Load and sort customers first
var customers = await context.Customers
  .OrderBy(c => c.Name)
  .ToListAsync();

// Then load orders
await context.Orders
  .Where(o => customers
    .Select(c => c.Id)
    .Contains(o.CustomerId))
  .LoadAsync();

// Two queries, clean separation
SQL: Generated (Optimized) 
-- Query 1: Sort and load customers
SELECT c.*
FROM Customers c
ORDER BY c.Name;
-- Result: 1000 rows

-- Query 2: Load orders for those customers
SELECT o.*
FROM Orders o
WHERE o.CustomerId IN (...)
-- Result: 2000 rows
-- Total: 3000 rows instead of 2000
Performance Comparison: Include vs Lazy Load 
// Scenario: Get 1000 customers with their orders, sorted by name

// APPROACH 1: Eager load with Include (BAD for sort)
var sw = Stopwatch.StartNew();
var customers1 = await context.Customers
  .Include(c => c.Orders)
  .OrderBy(c => c.Name)
  .ToListAsync();
sw.Stop();
Console.WriteLine($"Eager + Sort: {sw.ElapsedMilliseconds}ms");
// Result: 850ms
// SQL: JOIN multiplies rows, large result set

// APPROACH 2: Sort first, then include (BETTER)
sw.Restart();
var customerIds = await context.Customers
  .OrderBy(c => c.Name)
  .Select(c => c.Id)
  .ToListAsync();

var customers2 = await context.Customers
  .Where(c => customerIds.Contains(c.Id))
  .Include(c => c.Orders)
  .ToListAsync();
sw.Stop();
Console.WriteLine($"Sort + Include: {sw.ElapsedMilliseconds}ms");
// Result: 320ms (~2.6x faster)
// Two clean queries, no row multiplication

// APPROACH 3: Lazy load (simple but slower iteration)
sw.Restart();
var customers3 = await context.Customers
  .OrderBy(c => c.Name)
  .ToListAsync();

foreach (var customer in customers3)
{
  await context.Entry(customer)
    .Collection(c => c.Orders)
    .LoadAsync();
}
sw.Stop();
Console.WriteLine($"Sort + Lazy Load: {sw.ElapsedMilliseconds}ms");
// Result: 1200ms (N+1 queries)
// Simple but slower for iteration

// Key insight:
// - Eager load + sort = multiplied rows
// - Sort + eager load = cleaner queries
// - Structure queries carefully around sorting
Sort Before Including Related Data

When you Include() related entities, the database returns rows for each combination of parent and child. Sorting then multiplies the result set. Sort at the parent level first, then include related data. This avoids unnecessary data transfer and lets the database optimizer use better execution plans.

Key Takeaways

  • ORDER BY specifies result order. Without it, order is undefined. Always use ORDER BY when result order matters.
  • Ascending is the default; DESC reverses order. LINQ methods: .OrderBy() for ascending, .OrderByDescending() for descending.
  • For multiple sort criteria, chain .ThenBy() and .ThenByDescending(). Multiple .OrderBy() calls replace rather than chain.
  • Sort by computed expressions (Price * Quantity), not just columns. Entity Framework translates these to SQL equivalents at the database.
  • NULL handling in sorting varies by database. SQL Server puts NULLs first, PostgreSQL puts them last. Use conditional sorting in C# for consistency.
  • OFFSET/FETCH (Skip/Take) retrieves a range of rows. Must follow ORDER BY for consistent results. Always specify ORDER BY in pagination queries.
  • OFFSET performance degrades with large skip values. Page 1000 requires skipping 99,900 rows, which is slow. Use keyset pagination for large datasets.
  • Keyset pagination is more efficient than OFFSET for large result sets. Use a cursor (last key) to request rows after it, maintaining constant performance regardless of page depth.
  • For small datasets or admin interfaces, OFFSET is fine. For user-facing pagination on millions of rows, use keyset pagination.
  • Always apply ORDER BY before OFFSET/FETCH or Skip/Take to ensure consistent, predictable results.
